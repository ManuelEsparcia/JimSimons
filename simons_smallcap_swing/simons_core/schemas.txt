Especificación Cuantitativa: simons_core/schemas.py

0. Propósito y papel dentro del stack

simons_core/schemas.py define la capa institucional de contratos estructurales de datos del sistema. Su función no es limpiar datos, imputarlos, transformarlos ni inferir reglas de negocio, sino fijar de manera explícita qué forma, tipos, claves, restricciones y semántica mínima deben tener los datasets que circulan entre DataProvider, FeatureBuilder, ModelTrainer, inferencia y portfolio construction. En un framework cuantitativo serio, este módulo existe para impedir incompatibilidades silenciosas entre etapas, eliminar la ambigüedad inherente de trabajar con DataFrames “duck-typed” y proporcionar una fuente única de verdad sobre qué significa que un dataset sea válido. :contentReference[oaicite:0]{index=0}

La regla de diseño es:

    Dataset válido = shape + types + keys + domains + semantics

Es decir, un schema no es solo una lista de columnas. También incluye:
- nombres y orden lógico de columnas;
- tipos esperados;
- nullabilidad;
- claves primarias o unicidad;
- restricciones de dominio;
- semántica temporal;
- versión del contrato.

Si interfaces.py fija contratos entre componentes, schemas.py fija contratos sobre los artefactos de datos que esos componentes producen y consumen.

1. Responsabilidad exacta

El módulo debe encargarse de:

- declarar schemas canónicos del pipeline;
- exponer estructuras formales como ColumnSpec, DataSchema, ValidationIssue y ValidationResult;
- validar columnas, tipos, nullabilidad, claves primarias y dominios;
- ofrecer una API estable de validación y aserción;
- gobernar la evolución y versionado de los contratos de datos.

No debe encargarse de:

- imputar nulos automáticamente;
- corregir silenciosamente datasets inválidos;
- transformar features;
- entrenar modelos;
- decidir lógica de negocio más allá del contrato declarado.

La validación debe ser explícita; la reparación automática, si existe, debe vivir en otro módulo.

2. Modelo formal

Sea un dataset D representado como tabla con columnas C = {c_1, ..., c_n}. Un schema S puede modelarse como la tupla:

    S = (C, T, N, K, R, V)

donde:
- C: conjunto ordenado de columnas esperadas;
- T: función de tipos T(c_i);
- N: función de nullabilidad;
- K: conjunto de claves e invariantes de unicidad;
- R: restricciones de dominio o rango;
- V: versión del contrato.

Diremos que un dataset D satisface un schema S, y lo denotamos D ⊨ S, si y solo si cumple simultáneamente:

- todas las columnas requeridas existen;
- los tipos son compatibles con T;
- las columnas no-nullable no contienen nulos;
- las claves K satisfacen unicidad;
- los valores pertenecen a los dominios R;
- la semántica temporal declarada no se viola.

3. Principios no negociables

P1. Contract-first.
Todo dataset importante del pipeline debe validarse contra un schema explícito.

P2. Structural clarity.
Los schemas deben declarar columnas, tipos, nullabilidad y PK sin depender de inferencias implícitas.

P3. Fail-fast.
Los datasets estructuralmente inválidos deben fallar antes de contaminar etapas posteriores.

P4. Single source of truth.
El catálogo de schemas debe vivir en un único lugar, no repartido por módulos.

P5. Versioned evolution.
Los cambios incompatibles deben versionarse explícitamente.

P6. DataFrame discipline.
Un DataFrame no debe circular como “tabla cualquiera”; debe obedecer un contrato verificable.

P7. Semantics matter.
Un schema correcto no es sólo dtype alignment; también incluye key semantics, domain semantics y, cuando corresponda, semántica temporal.

4. Ámbito y no-objetivos

El núcleo mínimo del módulo debe cubrir, como mínimo, los schemas canónicos:

- prices_adjusted
- features
- labels
- predictions

Opcionalmente pueden añadirse más adelante:
- universe
- positions
- orders
- model_metadata
- validation_report

Pero el foco principal debe estar en los artefactos transversales del pipeline, no en estructuras efímeras o internas.

5. Claves primarias e invariantes estructurales

Para la mayoría de datasets longitudinales del sistema, la clave primaria natural es:

    (date, symbol)

Formalmente, la unicidad puede expresarse como:

    ∀ (d, s),
    | { r ∈ D : r.date = d ∧ r.symbol = s } | ≤ 1

Si el schema exige unicidad estricta, cualquier violación debe producir:

    PRIMARY_KEY_VIOLATION

No debe aceptarse que haya duplicados en la PK bajo el pretexto de “ya se agregará luego”. El contrato debe fijarse donde nace el dataset.

6. Estructuras formales recomendadas

Una base sobria y suficientemente fuerte es:

~~~python
from __future__ import annotations

from dataclasses import dataclass
from typing import Any

@dataclass(frozen=True)
class ColumnSpec:
    name: str
    dtype: str
    nullable: bool = False
    allowed_values: tuple[Any, ...] | None = None

@dataclass(frozen=True)
class DataSchema:
    name: str
    version: str
    columns: tuple[ColumnSpec, ...]
    primary_key: tuple[str, ...] = ()
    allow_extra_columns: bool = False

    @property
    def column_names(self) -> tuple[str, ...]:
        return tuple(c.name for c in self.columns)

@dataclass(frozen=True)
class ValidationIssue:
    code: str
    message: str
    column: str | None = None

@dataclass(frozen=True)
class ValidationResult:
    passed: bool
    issues: tuple[ValidationIssue, ...] = ()
~~~

Esta estructura ya fija los pilares correctos:
- contrato declarativo;
- errores estructurados;
- versionado explícito;
- clave primaria formal.

7. Schema canónico: prices_adjusted

Este schema representa precios ajustados y volumen por fecha y símbolo.

Columnas mínimas recomendadas:

- date        : date / normalized timestamp, no nullable
- symbol      : string, no nullable
- open        : float64, no nullable
- high        : float64, no nullable
- low         : float64, no nullable
- close       : float64, no nullable
- volume      : int64 o float64 según proveedor, nullable según política

Ejemplo:

~~~python
PRICE_ADJUSTED_SCHEMA = DataSchema(
    name="prices_adjusted",
    version="1.0.0",
    columns=(
        ColumnSpec("date", "date", nullable=False),
        ColumnSpec("symbol", "string", nullable=False),
        ColumnSpec("open", "float64", nullable=False),
        ColumnSpec("high", "float64", nullable=False),
        ColumnSpec("low", "float64", nullable=False),
        ColumnSpec("close", "float64", nullable=False),
        ColumnSpec("volume", "int64", nullable=True),
    ),
    primary_key=("date", "symbol"),
    allow_extra_columns=False,
)
~~~

Invariantes recomendados:
- high >= low
- open > 0
- high > 0
- low > 0
- close > 0
- volume >= 0 si existe
- unicidad de (date, symbol)

8. Schema canónico: features

Representa la matriz de features utilizada por modelos o reglas aguas abajo.

Columnas mínimas:
- date
- symbol
- feature_*

Invariantes recomendados:
- no duplicados de columnas
- PK=(date, symbol)
- features numéricas con dtype compatible
- nullabilidad declarada explícitamente
- semántica temporal compatible con la fecha de decisión
- opcionalmente: restricciones ligeras de calidad como fracción de nulos por columna

Estructuralmente, el schema de features debe resolver la ambigüedad típica de “un DataFrame con columnas varias”. Debe quedar claro:
- qué columnas son identificadores;
- cuáles son features reales;
- cuál es la versión del conjunto de features;
- si se permiten columnas extra.

9. Schema canónico: labels

Representa objetivos supervisados o etiquetas futuras.

Columnas mínimas típicas:
- date
- symbol
- future_ret_5d
- binary_swing

Invariantes recomendados:
- PK=(date, symbol)
- dominio claro para labels continuas y discretas
- semántica temporal explícita: la etiqueta pertenece a una observación con fecha base date, aunque su valor se construya con horizonte futuro fuera del módulo

Ejemplo de dominio:
- future_ret_5d ∈ [-1, 1] si se usa normalización explícita
- binary_swing ∈ {0,1} o bool

Importante:
schemas.py no prueba por sí solo ausencia de leakage, pero sí debe fijar con claridad qué columnas son labels y qué semántica temporal tienen.

10. Schema canónico: predictions

Representa la salida del modelo o del bloque de inferencia.

Columnas mínimas típicas:
- date
- symbol
- score
- prob_long opcional
- confidence opcional

Invariantes recomendados:
- PK=(date, symbol)
- score no nulo salvo política explícita
- prob_long ∈ [0,1] si existe
- confidence documentada si existe
- no mezclar score ordinal con probabilidad calibrada en la misma columna

11. Schema registry

El módulo debe exponer un catálogo canónico:

~~~python
SCHEMA_REGISTRY = {

~~~

Esto no es sólo ergonomía. Permite:
- descubrimiento uniforme de schemas;
- validación por nombre;
- versionado controlado;
- trazabilidad centralizada.

13. Lógica base de validación

Una implementación de referencia razonable es:

~~~python
def get_schema(name: str) -> DataSchema:
    return SCHEMA_REGISTRY[name]

def list_schemas() -> list[str]:
    return sorted(SCHEMA_REGISTRY.keys())

def validate_schema(df, schema: str | DataSchema) -> ValidationResult:
    spec = get_schema(schema) if isinstance(schema, str) else schema
    issues: list[ValidationIssue] = []

    actual_cols = set(df.columns)
    expected_cols = set(spec.column_names)

    missing = expected_cols - actual_cols
    extra = actual_cols - expected_cols

    for col in sorted(missing):
        issues.append(ValidationIssue(
            code="MISSING_COLUMN",
            message=f"Missing required column: {col}",
            column=col,
        ))

    if not spec.allow_extra_columns:
        for col in sorted(extra):
            issues.append(ValidationIssue(
                code="UNEXPECTED_COLUMN",
                message=f"Unexpected column: {col}",
                column=col,
            ))

    # dtype checks
    for colspec in spec.columns:
        if colspec.name not in df.columns:
            continue
        series = df[colspec.name]

        if not _dtype_compatible(series, colspec.dtype):
            issues.append(ValidationIssue(
                code="INVALID_DTYPE",
                message=f"Column {colspec.name} incompatible with dtype {colspec.dtype}",
                column=colspec.name,
            ))

        if not colspec.nullable and series.isna().any():
            issues.append(ValidationIssue(
                code="NULL_IN_NONNULLABLE",
                message=f"Column {colspec.name} contains nulls but is not nullable",
                column=colspec.name,
            ))

        if colspec.allowed_values is not None:
            valid = series.dropna().isin(colspec.allowed_values)
            if not valid.all():
                issues.append(ValidationIssue(
                    code="DOMAIN_VIOLATION",
                    message=f"Column {colspec.name} contains values outside allowed domain",
                    column=colspec.name,
                ))

    # primary key checks
    if spec.primary_key:
        missing_pk = [c for c in spec.primary_key if c not in df.columns]
        if not missing_pk:
            if df.duplicated(subset=list(spec.primary_key)).any():
                issues.append(ValidationIssue(
                    code="PRIMARY_KEY_VIOLATION",
                    message=f"Primary key {spec.primary_key} is not unique",
                    column=None,
                ))

    return ValidationResult(
        passed=(len(issues) == 0),
        issues=tuple(issues),
    )

def assert_schema(df, schema: str | DataSchema) -> None:
    result = validate_schema(df, schema)
    if not result.passed:
        rendered = "; ".join(f"{i.code}: {i.message}" for i in result.issues)
        raise ValueError(f"Schema validation failed: {rendered}")
~~~

14. Compatibilidad de tipos

La validación de dtype no debe basarse en igualdad textual ingenua. Debe existir una función auxiliar tipo:

~~~python
import pandas as pd
from pandas.api.types import (
    is_bool_dtype,
    is_datetime64_any_dtype,
    is_float_dtype,
    is_integer_dtype,
    is_object_dtype,
    is_string_dtype,
)

def _dtype_compatible(series: pd.Series, expected: str) -> bool:
    if expected == "float64":
        return is_float_dtype(series)
    if expected == "int64":
        return is_integer_dtype(series)
    if expected == "bool":
        return is_bool_dtype(series)
    if expected == "string":
        return is_string_dtype(series) or is_object_dtype(series)
    if expected == "date":
        return is_datetime64_any_dtype(series)
    return True
~~~

15. Restricciones de dominio

Más allá de tipos y columnas, el módulo debe soportar invariantes de dominio. Ejemplos críticos:

Para prices_adjusted:
- high >= low
- close > 0
- volume >= 0

Para predictions:
- prob_long ∈ [0,1]

Estas restricciones pueden integrarse como validadores específicos por schema o como callbacks asociados al DataSchema.

Ejemplo simple:

~~~python
def _validate_prices_adjusted_domains(df) -> list[ValidationIssue]:
    issues = []

    if {"high", "low"}.issubset(df.columns):
        bad = df["high"] < df["low"]
        if bad.any():
            issues.append(ValidationIssue(
                code="DOMAIN_VIOLATION",
                message="Found rows with high < low",
            ))

    for col in ("open", "high", "low", "close"):
        if col in df.columns:
            bad = df[col] <= 0
            if bad.any():
                issues.append(ValidationIssue(
                    code="DOMAIN_VIOLATION",
                    message=f"Found rows with {col} <= 0",
                    column=col,
                ))

    if "volume" in df.columns:
        bad = df["volume"].dropna() < 0
        if bad.any():
            issues.append(ValidationIssue(
                code="DOMAIN_VIOLATION",
                message="Found rows with volume < 0",
                column="volume",
            ))

    return issues
~~~

16. Semántica temporal

Un dataset puede ser estructuralmente correcto y temporalmente incorrecto. Por ello, schemas.py debe poder declarar, al menos conceptualmente:

- qué columna representa fecha efectiva;
- si la fecha debe ser monotónica;
- si existe un asof_date externo con respecto al cual validar;
- si el dataset puede contener timestamps posteriores al horizonte permitido.

Aunque el anti-look-ahead fuerte puede residir en interfaces.py o validadores específicos, aquí debe existir espacio para checks como:

- NON_MONOTONIC_DATE
- FUTURE_TIMESTAMP_VIOLATION

Una extensión razonable sería permitir:

~~~python
@dataclass(frozen=True)
class DataSchema:
    ...
    date_column: str | None = None
    require_monotonic_date: bool = False
~~~

17. Versionado y evolución

Cada schema debe tener version explícita. Política recomendada:

- Patch:
  ajustes internos sin cambiar contrato efectivo.

- Minor:
  cambios retrocompatibles, como añadir columna opcional.

- Major:
  eliminar columnas, endurecer nullabilidad o cambiar semántica.

Formalmente, si S_v y S_{v+1} son dos versiones:

- compatibilidad backward implica que datasets válidos bajo S_v siguen siendo legibles o migrables bajo S_{v+1};
- si esto no se cumple, debe existir estrategia de migración explícita.

Errores clásicos a evitar:
- añadir columna obligatoria sin cambiar versión;
- cambiar PK sin version bump;
- renombrar binary_swing o future_ret_5d silenciosamente;
- cambiar date de daily label a intraday timestamp sin gobernanza.

18. Política de errores

La validación no debe devolver mensajes vagos. Debe producir incidencias estructuradas como:

- MISSING_COLUMN
- UNEXPECTED_COLUMN
- INVALID_DTYPE
- NULL_IN_NONNULLABLE
- PRIMARY_KEY_VIOLATION
- DOMAIN_VIOLATION
- NON_MONOTONIC_DATE
- FUTURE_TIMESTAMP_VIOLATION

Esto permite a los consumidores:
- fallar inmediatamente;
- agregar issues en un informe;
- separar errores estructurales de warnings de calidad.

19. Casos límite que deben quedar resueltos

El módulo debe contemplar y testear al menos:

- columnas presentes pero mal nombradas;
- columnas con capitalización distinta si el contrato es case-sensitive;
- duplicados en (date, symbol);
- nulos en columnas no-nullable;
- columnas extra en schemas estrictos;
- mezcla de strings y floats en una columna numérica;
- probabilidades fuera de [0,1];
- datasets vacíos, si se decide permitirlos o no;
- fechas como string sin normalización previa cuando el contrato exige datetime.

20. Posible refuerzo con Pydantic o Pandera

Para acercarse al techo 97-98, una evolución razonable sería combinar:
- DataSchema declarativo propio para gobernanza del contrato;
- Pandera o Pydantic para validación más rica y reusable.

Pero el núcleo no debe depender de vender “compile-time safety” inexistente en Python. Lo correcto es hablar de:
- validación declarativa runtime,
- estructuras tipadas y gobernadas,
- checks reproducibles.

21. Pruebas mínimas recomendadas

~~~python
def test_price_schema_has_expected_pk():
    schema = get_schema("prices_adjusted")
    assert schema.primary_key == ("date", "symbol")

def test_missing_required_column_fails():
    import pandas as pd
    df = pd.DataFrame({
        "date": ["2026-03-01"],
        "symbol": ["AAPL"],
        "open": [100.0],
        "high": [101.0],
        "low": [99.0],
        "volume": [1000000],
    })
    result = validate_schema(df, "prices_adjusted")
    assert result.passed is False
    assert any(issue.code == "MISSING_COLUMN" for issue in result.issues)

def test_extra_column_fails_when_not_allowed():
    import pandas as pd
    df = pd.DataFrame({
        "date": ["2026-03-01"],
        "symbol": ["AAPL"],
        "open": [100.0],
        "high": [101.0],
        "low": [99.0],
        "close": [100.5],
        "volume": [1000000],
        "foo": [1],
    })
    result = validate_schema(df, "prices_adjusted")
    assert result.passed is False
    assert any(issue.code == "UNEXPECTED_COLUMN" for issue in result.issues)

def test_primary_key_violation_detected():
    import pandas as pd
    df = pd.DataFrame({
        "date": ["2026-03-01", "2026-03-01"],
        "symbol": ["AAPL", "AAPL"],
        "open": [100.0, 100.0],
        "high": [101.0, 101.0],

def test_domain_violation_close_nonpositive():
    import pandas as pd
    df = pd.DataFrame({
        "date": ["2026-03-01"],
        "symbol": ["AAPL"],
        "open": [100.0],
        "high": [101.0],
~~~

Además, conviene mantener tests de snapshot del registry para detectar cambios accidentales en:
- nombres;
- versiones;
- PK;
- allow_extra_columns.

22. MVP obligatorio

El MVP del módulo debe incluir:
- ColumnSpec
- DataSchema
- ValidationIssue
- ValidationResult
- SCHEMA_REGISTRY
- schemas mínimos para prices_adjusted, features, labels y predictions
- validate_schema
- assert_schema
- checks de columnas, dtypes, nullabilidad, PK y dominios básicos
- tests deterministas básicos

23. Segunda fase razonable

Como extensiones útiles:
- MultiIndex support para PK nativa pandas
- validación temporal más rica
- null fraction / quality gates opcionales
- support de migración v1 -> v2
- integración opcional con Pandera/Pydantic
- quality_score ∈ [0,1] como complemento, no sustituto, de passed/fail

24. No-objetivos

No pertenece al alcance de este módulo:
- imputación automática;
- drift detection avanzada;
- leakage detection completa;
- entrenamiento de modelos;
- feature engineering;
- Great Expectations completo;
- inferencia estadística sobre calidad del dataset.

La misión aquí es contractual y estructural.

25. Criterios de aceptación

El módulo sólo debe considerarse correcto si cumple simultáneamente:

- existen schemas canónicos para los datasets centrales;
- cada schema declara columnas, tipos, nullabilidad, PK y restricciones;
- la validación produce incidencias estructuradas y accionables;
- la evolución de contratos está versionada;
- los consumidores del pipeline pueden validar sus artefactos contra una única fuente de verdad.

26. Conclusión

simons_core/schemas.py es una de las piezas fundacionales del core porque traduce DataFrames potencialmente ambiguos en artefactos con semántica estructural explícita. Su valor no está en sonar industrial, sino en imponer disciplina: columnas gobernadas, claves formales, dominios claros, errores precisos y versionado estable. En un framework cuantitativo serio, eso no es burocracia de datos; es la condición necesaria para que el pipeline no derive en incompatibilidades silenciosas, leakage encubierto y validaciones ad hoc. Ese es exactamente el estándar que este módulo debe imponer.
