\section{Propósito del módulo}

El módulo \texttt{simons\_core/calendar.py} proporciona una \emph{fuente de verdad temporal} para el sistema. Su función es traducir fechas de calendario civil a fechas y sesiones de mercado, de forma consistente, reproducible y explícita.

En un stack cuantitativo, este módulo existe para evitar una familia de errores muy concreta:

\begin{itemize}
    \item confundir días naturales con días de mercado;
    \item avanzar horizontes temporales con reglas inconsistentes entre submódulos;
    \item alinear timestamps a sesiones distintas según el componente que consuma los datos;
    \item introducir drift temporal entre research, backtest, validación y ejecución.
\end{itemize}

El objetivo del módulo no es ``optimizar rentabilidad'', sino garantizar integridad temporal. En sistemas de trading, la capa de calendario es infraestructura: si falla, todo lo demás queda contaminado.

\section{Responsabilidad exacta}

\texttt{calendar.py} debe resolver cinco problemas y solo esos cinco:

\begin{enumerate}[label=\arabic*.]
    \item Definir qué días son sesiones válidas para un mercado dado.
    \item Avanzar o retroceder en días de trading.
    \item Contar distancias en trading days entre dos fechas.
    \item Alinear fechas o timestamps a una sesión válida.
    \item Exponer estas reglas mediante una API pequeña, estable y verificable.
\end{enumerate}

No debe contener lógica de features, labels, entrenamiento, señales ni ejecución. Tampoco debe inferir reglas temporales implícitas fuera de su contrato.

\section{Modelo conceptual}

Sea \( \mathcal{C} \) un calendario de mercado y sea

\[
\mathcal{S} = \{ s_1, s_2, \dots, s_n \}
\]

el conjunto ordenado de sesiones válidas del mercado dentro de un horizonte temporal. Cada \( s_i \) representa una sesión de trading, no un día natural arbitrario.

La función central del módulo es operar siempre sobre \( \mathcal{S} \), no sobre el calendario gregoriano bruto.

En particular, si \( d \) es una fecha civil, las transformaciones relevantes son:

\[
d \mapsto s(d)
\]

donde \( s(d) \) puede significar:

\begin{itemize}
    \item la siguiente sesión válida;
    \item la sesión previa válida;
    \item la sesión que contiene un timestamp;
    \item la sesión más cercana bajo una regla explícita de alineación.
\end{itemize}

\section{Mercado y zona horaria}

Para el caso base del repositorio, el calendario debe tratar \textbf{US equities} bajo la zona horaria:

\[
\texttt{America/New\_York}
\]

Esta decisión no es cosmética. Toda operación temporal del módulo debe ser coherente con la convención horaria del mercado, no con la timezone local del servidor ni con UTC sin contextualización.

El módulo debe distinguir claramente entre:

\begin{itemize}
    \item \textbf{fecha de trading}: etiqueta de sesión;
    \item \textbf{timestamp timezone-aware}: instante temporal concreto;
    \item \textbf{fecha naive}: entrada civil que requiere normalización.
\end{itemize}

\section{API pública esperada}

Una API pública mínima y suficiente para \texttt{calendar.py} es la siguiente:

\begin{lstlisting}
load_market_calendar(market: str = "US_EQ") -> MarketCalendar

next_trading_day(date_like, n: int = 1, market: str = "US_EQ") -> date
prev_trading_day(date_like, n: int = 1, market: str = "US_EQ") -> date

trading_days_between(start, end, market: str = "US_EQ",
                     inclusive: str = "left") -> int

align_to_session(ts_like, mode: str = "next_open",
                 market: str = "US_EQ") -> pd.Timestamp
\end{lstlisting}

Si se desea un diseño más limpio, estas funciones pueden ser wrappers sobre una clase principal:

\begin{lstlisting}
class MarketCalendar:
    def next_trading_day(self, date_like, n: int = 1) -> date: ...
    def prev_trading_day(self, date_like, n: int = 1) -> date: ...
    def trading_days_between(self, start, end, inclusive="left") -> int: ...
    def align_to_session(self, ts_like, mode="next_open") -> pd.Timestamp: ...
\end{lstlisting}

\section{Semántica de cada operación}

\subsection{\texttt{load\_market\_calendar}}

Carga o construye la representación interna del calendario de mercado. Debe devolver un objeto con semántica estable y reutilizable, preferiblemente cacheado.

\textbf{Invariante:} dos llamadas equivalentes para el mismo mercado y mismo rango no deben producir reglas distintas.

\subsection{\texttt{next\_trading\_day(date, n=1)}}

Devuelve la \(n\)-ésima sesión estrictamente posterior a la fecha dada, salvo que la especificación indique explícitamente una convención distinta.

Formalmente, si \(d\) es una fecha y \(s_k\) es la primera sesión válida posterior, entonces:

\[
\texttt{next\_trading\_day}(d, n) = s_{k+n-1}
\]

Debe quedar documentado si una fecha que ya es sesión válida cuenta o no como punto de partida inclusivo. En general, para evitar ambigüedad, es preferible comportamiento \emph{estrictamente posterior}.

\subsection{\texttt{prev\_trading\_day(date, n=1)}}

Análogo al caso anterior, pero hacia atrás en el conjunto ordenado de sesiones.

\subsection{\texttt{trading\_days\_between(start, end)}}

Cuenta el número de sesiones válidas entre dos extremos bajo una convención explícita de inclusividad.

No basta con ``hacer una resta de fechas'' ni con contar business days genéricos. El resultado debe depender del calendario real del mercado y de la convención elegida.

Debe definirse, por ejemplo, si para un intervalo \([d_1, d_2)\) la cuenta incluye la sesión de \(d_1\), la de \(d_2\), ambas o ninguna.

\subsection{\texttt{align\_to\_session(ts, mode)}}

Mapea un timestamp o fecha a un instante representativo de una sesión válida. La semántica debe depender de \texttt{mode}, por ejemplo:

\begin{itemize}
    \item \texttt{next\_open}
    \item \texttt{prev\_close}
    \item \texttt{nearest\_open}
    \item \texttt{session\_label}
\end{itemize}

Este punto es especialmente importante porque la palabra ``align'' es ambigua si no se fija exactamente el resultado esperado.

\section{Normalización de entradas}

El módulo debe aceptar, como mínimo, tipos razonables de entrada:

\begin{itemize}
    \item \texttt{str}
    \item \texttt{datetime.date}
    \item \texttt{datetime.datetime}
    \item \texttt{pandas.Timestamp}
\end{itemize}

Toda entrada debe pasar por una fase explícita de normalización:

\begin{enumerate}[label=\arabic*.]
    \item parseo;
    \item asignación o validación de timezone;
    \item reducción a fecha o timestamp según la operación;
    \item validación de rango y tipo.
\end{enumerate}

Un módulo de calendario robusto no debe depender de coerciones implícitas opacas.

\section{Representación interna}

La representación interna puede basarse en una secuencia ordenada de sesiones, por ejemplo un \texttt{DatetimeIndex}, una estructura propia o un backend externo. La implementación concreta es secundaria; lo importante es preservar estas propiedades:

\begin{enumerate}[label=\textbf{R\arabic*.}]
    \item orden total;
    \item unicidad de sesiones;
    \item acceso eficiente por búsqueda;
    \item semántica determinista;
    \item independencia respecto al consumidor.
\end{enumerate}

Una estructura razonable para el caso base es un índice ordenado y cacheado de sesiones válidas por mercado.

\section{Casos límite que deben quedar resueltos}

El módulo debe documentar y testear explícitamente los siguientes edge cases:

\begin{itemize}
    \item entrada en fin de semana;
    \item entrada en festivo de mercado;
    \item entrada fuera de horario regular;
    \item timestamp timezone-naive;
    \item cambio de horario de verano (DST);
    \item rango invertido en \texttt{trading\_days\_between};
    \item \(n \le 0\) en desplazamientos;
    \item consulta fuera del rango del calendario precargado;
    \item medias sesiones o cierres tempranos, si el sistema decide soportarlos.
\end{itemize}

Estos casos no son periféricos: constituyen la mayor parte de los fallos reales en infraestructura temporal.

\section{Medias sesiones y cierres tempranos}

Una decisión arquitectónica importante es si el módulo modela solo sesiones \emph{válidas/inválidas} o si además distingue tipos de sesión:

\begin{itemize}
    \item sesión regular;
    \item cierre temprano;
    \item sesión especial;
    \item sesión no operativa.
\end{itemize}

Si el repositorio no necesita aún esa granularidad, el módulo puede empezar con un modelo binario ``hay sesión / no hay sesión''. Pero la interfaz debe diseñarse de forma que ampliar a tipos de sesión no rompa la API pública.

\section{Invariantes formales}

El módulo debe satisfacer, como mínimo, las siguientes invariantes:

\begin{enumerate}[label=\textbf{I\arabic*.}]
    \item \textbf{Monotonía:}
    \[
    \texttt{next\_trading\_day}(d,1) > d
    \]
    bajo orden temporal de sesiones.

    \item \textbf{Consistencia de ida y vuelta:} si \(d\) es una sesión válida y la convención es estricta,
    \[
    \texttt{prev\_trading\_day}(\texttt{next\_trading\_day}(d,1),1) = d
    \]

    \item \textbf{Conteo no negativo:}
    \[
    \texttt{trading\_days\_between}(d_1,d_2) \ge 0
    \]
    cuando \(d_2 \ge d_1\) bajo la convención documentada.

    \item \textbf{Determinismo:} mismas entradas y mismo calendario implican mismo resultado.

    \item \textbf{Cierre bajo normalización:} distintos formatos válidos de una misma fecha deben producir la misma sesión resultante.
\end{enumerate}

\section{Complejidad y rendimiento}

La prioridad del módulo es corrección semántica. Una vez garantizada, el rendimiento deseable es:

\begin{itemize}
    \item búsqueda de siguiente o anterior sesión en tiempo sublineal o amortizado constante sobre estructura cacheada;
    \item conteos sobre rangos sin necesidad de iterar día a día;
    \item coste de inicialización controlado mediante caché por mercado.
\end{itemize}

En otras palabras: primero exactitud temporal, después eficiencia. Nunca al revés.

\section{Diseño recomendado}

Una implementación sobria podría verse así:

\begin{lstlisting}
from dataclasses import dataclass
from datetime import date
import pandas as pd

@dataclass(frozen=True)
class MarketCalendar:
    market: str
    timezone: str
    sessions: pd.DatetimeIndex

    def next_trading_day(self, date_like, n: int = 1) -> date:
        if n < 1:
            raise ValueError("n must be >= 1")
        d = self._normalize_date(date_like)
        pos = self.sessions.searchsorted(pd.Timestamp(d), side="right")
        return self.sessions[pos + n - 1].date()

    def prev_trading_day(self, date_like, n: int = 1) -> date:
        if n < 1:
            raise ValueError("n must be >= 1")
        d = self._normalize_date(date_like)
        pos = self.sessions.searchsorted(pd.Timestamp(d), side="left") - 1
        return self.sessions[pos - (n - 1)].date()

    def trading_days_between(self, start, end, inclusive: str = "left") -> int:
        s = self._normalize_date(start)
        e = self._normalize_date(end)
        if e < s:
            raise ValueError("end must be >= start")
        left = self.sessions.searchsorted(pd.Timestamp(s), side="left")
        right = self.sessions.searchsorted(pd.Timestamp(e), side="left")
        count = right - left
        if inclusive == "both" and pd.Timestamp(e) in self.sessions:
            count += 1
        return count

    def align_to_session(self, ts_like, mode: str = "next_open") -> pd.Timestamp:
        ts = self._normalize_timestamp(ts_like)
        # explicit mode semantics go here
        ...

    def _normalize_date(self, x) -> date:
        ts = pd.Timestamp(x)
        if ts.tzinfo is not None:
            ts = ts.tz_convert(self.timezone)
        return ts.date()

    def _normalize_timestamp(self, x) -> pd.Timestamp:
        ts = pd.Timestamp(x)
        if ts.tzinfo is None:
            return ts.tz_localize(self.timezone)
        return ts.tz_convert(self.timezone)
\end{lstlisting}

\section{Errores y política de fallos}

El módulo debe fallar de forma explícita cuando:

\begin{itemize}
    \item el mercado solicitado no está soportado;
    \item el rango consultado cae fuera de cobertura;
    \item la entrada no puede parsearse;
    \item la convención de \texttt{inclusive} o \texttt{mode} es inválida;
    \item \(n < 1\) en desplazamientos.
\end{itemize}

No debe devolver resultados silenciosamente aproximados en estos casos. En infraestructura temporal, una excepción explícita es preferible a una corrección implícita no documentada.

\section{Pruebas mínimas recomendadas}

El módulo debe ir acompañado, al menos, de las siguientes pruebas:

\begin{lstlisting}
def test_next_trading_day_skips_weekend():
    cal = load_market_calendar("US_EQ")
    assert cal.next_trading_day("2026-01-16") > date(2026, 1, 16)

def test_prev_trading_day_moves_back_to_valid_session():
    cal = load_market_calendar("US_EQ")
    d = cal.prev_trading_day("2026-01-20")
    assert isinstance(d, date)

def test_trading_days_between_is_consistent():
    cal = load_market_calendar("US_EQ")
    n = cal.trading_days_between("2026-01-05", "2026-01-12")
    assert n >= 0

def test_align_to_session_returns_tz_aware_timestamp():
    cal = load_market_calendar("US_EQ")
    ts = cal.align_to_session("2026-01-17 12:00:00", mode="next_open")
    assert ts.tzinfo is not None

def test_invalid_n_raises():
    cal = load_market_calendar("US_EQ")
    try:
        cal.next_trading_day("2026-01-05", n=0)
        assert False
    except ValueError:
        assert True
\end{lstlisting}

Además, conviene mantener tests de regresión sobre fechas conocidas del calendario soportado, especialmente alrededor de festivos y cambios DST.

\section{Definition of Done}

\texttt{simons\_core/calendar.py} queda correctamente resuelto cuando:

\begin{itemize}
    \item existe una única fuente de verdad para sesiones válidas;
    \item la API pública tiene semántica explícita y estable;
    \item fines de semana, festivos y DST están resueltos de forma consistente;
    \item los desplazamientos temporales usan trading days reales, no aproximaciones civiles;
    \item el módulo es determinista, testeable y reusable por research, backtest y ejecución.
\end{itemize}

\section{Conclusión}

La calidad de \texttt{calendar.py} no depende de sonar sofisticado, sino de fijar con precisión la geometría temporal del sistema. En un framework cuantitativo serio, el calendario no es un detalle utilitario: es la base que evita errores silenciosos de horizonte, alineación y conteo de sesiones.

Por eso, el estándar correcto para este módulo es: API mínima, semántica explícita, edge cases bien definidos y cero ambigüedad temporal.
