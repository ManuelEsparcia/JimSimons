\section{Propósito del módulo}

El módulo \texttt{simons\_core/interfaces.py} define los \emph{contratos abstractos} que conectan los subsistemas principales del framework. Su función no es implementar lógica cuantitativa concreta, sino fijar una gramática común entre productores y consumidores de datos, features, modelos, decisiones de cartera y validaciones.

En términos arquitectónicos, este módulo existe para resolver cinco problemas:

\begin{enumerate}[label=\arabic*.]
    \item evitar acoplamiento directo entre implementaciones;
    \item imponer firmas y semánticas estables entre etapas del pipeline;
    \item hacer verificable la compatibilidad entre módulos;
    \item facilitar sustitución de componentes sin reescribir el sistema;
    \item permitir evolución interna conservando una API contractual estable.
\end{enumerate}

Si \texttt{calendar.py} fija la geometría temporal y \texttt{schemas.py} fija la geometría de datos, \texttt{interfaces.py} fija la geometría de interacción entre componentes.

\section{Principio rector}

La idea central puede expresarse así:

\[
\text{Implementation} \;\prec\; \text{Contract}
\]

donde el símbolo \( \prec \) no indica inferioridad funcional, sino dependencia arquitectónica: las implementaciones concretas deben adaptarse al contrato, no al revés.

Por tanto, el módulo debe responder a la pregunta:

\begin{quote}
``¿Qué debe garantizar un componente para ser enchufable en el sistema?''
\end{quote}

No debe responder a:

\begin{quote}
``¿Qué algoritmo específico debe usarse dentro de ese componente?''
\end{quote}

\section{Alcance del módulo}

Las interfaces nucleares del pipeline son:

\begin{itemize}
    \item \texttt{DataProvider}
    \item \texttt{FeatureBuilder}
    \item \texttt{ModelTrainer}
    \item \texttt{PortfolioEngine}
    \item \texttt{Validator}
\end{itemize}

Además, el módulo debe definir tipos transversales que viajan entre etapas o describen el contexto de una ejecución:

\begin{itemize}
    \item \texttt{RunContext}
    \item \texttt{GateResult}
    \item opcionalmente \texttt{ModelRef}, \texttt{ValidationReport}, \texttt{PortfolioDecision} u otros tipos pequeños y estables.
\end{itemize}

La calidad del módulo no depende de cuántas interfaces define, sino de si esas interfaces capturan correctamente las fronteras del sistema.

\section{Cadena semántica del pipeline}

El flujo de alto nivel puede expresarse de forma abstracta como:

\[
D_t \xrightarrow{\texttt{DataProvider}} X_t
\xrightarrow{\texttt{FeatureBuilder}} F_t
\xrightarrow{\texttt{ModelTrainer}} M_t
\xrightarrow{\texttt{PortfolioEngine}} W_t
\xrightarrow{\texttt{Validator}} G_t
\]

donde:

\begin{itemize}
    \item \(D_t\) representa datos fuente observados a fecha \(t\),
    \item \(F_t\) representa features construidas y validadas,
    \item \(M_t\) representa un artefacto de modelo o referencia a modelo,
    \item \(W_t\) representa una decisión de cartera o pesos,
    \item \(G_t\) representa un resultado de validación o gating.
\end{itemize}

La finalidad de \texttt{interfaces.py} es que cada flecha de esa composición tenga semántica estable, explícita y testeable.

\section{Diseño de contratos}

\subsection{Criterios}

Cada interfaz debe cumplir simultáneamente:

\begin{enumerate}[label=\textbf{C\arabic*.}]
    \item \textbf{Minimalidad:} solo exponer métodos esenciales.
    \item \textbf{Claridad semántica:} el nombre del método debe indicar exactamente su responsabilidad.
    \item \textbf{Separación de concerns:} no mezclar preparación de datos, entrenamiento y decisión en un único contrato.
    \item \textbf{Testabilidad:} debe ser posible verificar conformidad sin depender de infraestructura pesada.
    \item \textbf{Extensibilidad:} debe admitir nuevas implementaciones sin cambio de firma base.
\end{enumerate}

\subsection{ABC vs Protocol}

El módulo puede usar \texttt{abc.ABC} o \texttt{typing.Protocol}. La decisión depende de la filosofía del repositorio:

\begin{itemize}
    \item \texttt{ABC} si se desea enforcement nominal y jerarquía explícita;
    \item \texttt{Protocol} si se desea compatibilidad estructural y menor acoplamiento.
\end{itemize}

Para un core reusable, ambas opciones son válidas. Lo importante es mantener consistencia. Si el sistema ya depende de herencia explícita, \texttt{ABC} suele ser más fácil de gobernar. Si se busca flexibilidad máxima y typing moderno, \texttt{Protocol} puede ser preferible.

\section{Contrato \texttt{DataProvider}}

\subsection{Responsabilidad}

\texttt{DataProvider} define cómo un componente produce datos fuente consumibles por etapas posteriores. No construye señales, no entrena modelos y no decide pesos.

\subsection{Forma abstracta}

\begin{lstlisting}
class DataProvider(ABC):
    @abstractmethod
    def fetch(self, asof_date, context: "RunContext"):
        ...
\end{lstlisting}

\subsection{Precondiciones}
\begin{itemize}
    \item \texttt{asof\_date} debe representar una fecha o sesión válida bajo el calendario del sistema.
    \item \texttt{context} debe contener la información mínima para reproducibilidad y trazabilidad.
\end{itemize}

\subsection{Postcondiciones}
\begin{itemize}
    \item el resultado debe estar tipado y estructurado de forma compatible con \texttt{schemas.py};
    \item no debe mezclar datos futuros respecto a \texttt{asof\_date};
    \item debe ser determinista dadas mismas entradas y misma configuración.
\end{itemize}

\subsection{Invariantes}
\[
\texttt{fetch}(t, c) = Y_t
\quad \text{con} \quad
\max(\text{timestamp}(Y_t)) \le t
\]

Esta desigualdad resume una propiedad crítica: el proveedor no debe introducir look-ahead bias.

\section{Contrato \texttt{FeatureBuilder}}

\subsection{Responsabilidad}

\texttt{FeatureBuilder} transforma datos fuente validados en una matriz o tabla de features con semántica estable.

\subsection{Forma abstracta}

\begin{lstlisting}
class FeatureBuilder(ABC):
    @abstractmethod
    def build(self, raw_data, context: "RunContext"):
        ...
\end{lstlisting}

\subsection{Precondiciones}
\begin{itemize}
    \item \texttt{raw\_data} debe cumplir schema de entrada conocido;
    \item deben estar resueltas las claves de indexación temporal e instrumental.
\end{itemize}

\subsection{Postcondiciones}
\begin{itemize}
    \item el output debe cumplir un schema de features canónico;
    \item debe preservar alineación por fecha/símbolo;
    \item no debe introducir columnas implícitas o metadatos no documentados.
\end{itemize}

\subsection{Invariante de compatibilidad}
Si \(Y\) es un dataset válido y \(f\) una implementación conforme, entonces:

\[
f(Y, c) = X
\]

con \(X\) tal que sus ejes, claves y nombres de columnas satisfacen el contrato esperado por etapas posteriores.

\section{Contrato \texttt{ModelTrainer}}

\subsection{Responsabilidad}

\texttt{ModelTrainer} encapsula el entrenamiento o ajuste de un modelo a partir de features y objetivos. No ejecuta rebalanceos ni aplica reglas de cartera.

\subsection{Forma abstracta}

\begin{lstlisting}
class ModelTrainer(ABC):
    @abstractmethod
    def fit(self, features, labels, context: "RunContext"):
        ...
\end{lstlisting}

\subsection{Precondiciones}
\begin{itemize}
    \item \texttt{features} y \texttt{labels} deben estar alineados;
    \item no debe existir leakage temporal;
    \item el contexto debe fijar semilla, configuración y horizonte de run.
\end{itemize}

\subsection{Postcondiciones}
\begin{itemize}
    \item debe devolver una referencia de modelo, artefacto entrenado o estructura equivalente;
    \item el resultado debe contener suficiente metadata para trazabilidad;
    \item la operación debe ser reproducible dentro de tolerancias documentadas.
\end{itemize}

\subsection{Observación}

La interfaz no debe imponer un algoritmo concreto. Puede respaldar modelos lineales, árboles, boosting, deep learning o ensamblados, siempre que todos respeten la misma semántica externa.

\section{Contrato \texttt{PortfolioEngine}}

\subsection{Responsabilidad}

\texttt{PortfolioEngine} traduce predicciones, scores o señales en una decisión de cartera bajo restricciones explícitas.

\subsection{Forma abstracta}

\begin{lstlisting}
class PortfolioEngine(ABC):
    @abstractmethod
    def rebalance(self, signals, context: "RunContext"):
        ...
\end{lstlisting}

\subsection{Precondiciones}
\begin{itemize}
    \item \texttt{signals} debe tener índice e identificadores consistentes;
    \item las restricciones necesarias deben estar presentes en el contexto o en la propia configuración del engine;
    \item los inputs deben referirse a una misma fecha efectiva de decisión.
\end{itemize}

\subsection{Postcondiciones}
\begin{itemize}
    \item la salida debe representar una decisión de cartera bien formada;
    \item la semántica de la decisión debe ser inequívoca: pesos, órdenes objetivo o ranking elegible;
    \item deben cumplirse restricciones de factibilidad documentadas.
\end{itemize}

\subsection{Forma matemática abstracta}

Si \(s \in \mathbb{R}^n\) representa una señal sobre \(n\) activos, el engine aplica una transformación

\[
\Phi : \mathbb{R}^n \to \mathcal{W}
\]

donde \(\mathcal{W}\) es el conjunto de decisiones factibles bajo las restricciones del sistema. El módulo de interfaces no impone la forma exacta de \(\Phi\), pero sí que su salida pertenezca al espacio semántico correcto.

\section{Contrato \texttt{Validator}}

\subsection{Responsabilidad}

\texttt{Validator} evalúa si un artefacto, dataset, modelo o decisión cumple criterios de aceptación antes de avanzar de etapa.

\subsection{Forma abstracta}

\begin{lstlisting}
class Validator(ABC):
    @abstractmethod
    def validate(self, artifact, context: "RunContext") -> "GateResult":
        ...
\end{lstlisting}

\subsection{Postcondición clave}

La salida debe ser un \texttt{GateResult} con estructura suficiente para distinguir:

\begin{itemize}
    \item si la validación pasa o falla;
    \item qué score o métrica resume el resultado;
    \item qué razones o violaciones justifican la decisión;
    \item qué metadata contextual acompaña la evaluación.
\end{itemize}

\subsection{Importante}

El core no debe acoplar esta interfaz a una única filosofía de validación. Puede cubrir checks duros, validaciones estadísticas, reglas de calidad de datos o gates compuestos, siempre que la interfaz siga siendo pequeña y clara.

\section{Tipos transversales}

\subsection{\texttt{RunContext}}

\texttt{RunContext} debe ser el contenedor mínimo de información compartida entre etapas. Su propósito es transportar estado inmutable o cuasi-inmutable de ejecución.

Una versión razonable:

\begin{lstlisting}
from dataclasses import dataclass
from typing import Optional

@dataclass(frozen=True)
class RunContext:
    run_id: str
    asof_date: str
    seed: int
    config_hash: str
    pipeline_version: str
    market: str = "US_EQ"
    parent_run_id: Optional[str] = None
\end{lstlisting}

\subsubsection*{Invariantes}
\begin{itemize}
    \item \texttt{run\_id} identifica unívocamente la ejecución;
    \item \texttt{config\_hash} fija la configuración efectiva;
    \item \texttt{seed} permite reproducibilidad controlada;
    \item el objeto debe ser pequeño, serializable y estable.
\end{itemize}

\subsection{\texttt{GateResult}}

\texttt{GateResult} debe ser una estructura explícita, no un booleano desnudo. Una forma recomendable:

\begin{lstlisting}
from dataclasses import dataclass, field
from typing import Any

@dataclass(frozen=True)
class GateResult:
    passed: bool
    score: float
    reasons: tuple[str, ...] = ()
    metrics: dict[str, float] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)
\end{lstlisting}

\subsubsection*{Invariantes}
\begin{itemize}
    \item \texttt{score} debe tener semántica documentada;
    \item \texttt{reasons} debe poder explicar un fallo sin inspección externa;
    \item \texttt{metrics} debe contener señales numéricas útiles y no ambiguas.
\end{itemize}

\section{Contratos de entrada y salida}

El módulo debe dejar claro que la conformidad no se limita a la firma de métodos. Una implementación puede compilar y seguir siendo contractualmente incorrecta.

Por ello, cada interfaz debe entenderse como:

\[
\text{Contract} = \text{Signature} + \text{Semantics} + \text{Invariants}
\]

No basta con que exista un método \texttt{fit}. También debe cumplir:

\begin{itemize}
    \item significado correcto de inputs;
    \item forma válida de outputs;
    \item ausencia de side effects no documentados;
    \item compatibilidad temporal y estructural con el resto del pipeline.
\end{itemize}

\section{Errores y política de fallos}

Las implementaciones concretas pueden lanzar excepciones específicas, pero las interfaces deben favorecer fallos explícitos cuando se rompa el contrato.

Ejemplos típicos:

\begin{itemize}
    \item schema de entrada inválido;
    \item desalineación entre features y labels;
    \item señales duplicadas por activo/fecha;
    \item contexto incompleto;
    \item output no factible.
\end{itemize}

Una política sana es que las violaciones estructurales del contrato fallen pronto y de forma ruidosa, en vez de producir resultados silenciosamente incoherentes.

\section{Versionado del contrato}

Las interfaces son API pública y deben someterse a una política de versionado explícita.

\begin{itemize}
    \item \textbf{Patch:} cambios internos sin alterar firma ni semántica.
    \item \textbf{Minor:} adiciones retrocompatibles, por ejemplo nuevos campos opcionales.
    \item \textbf{Major:} cualquier cambio incompatible en firma, tipos o semántica.
\end{itemize}

Cambiar la forma del output de \texttt{rebalance}, renombrar campos de \texttt{RunContext} o modificar el significado de \texttt{score} en \texttt{GateResult} no son detalles menores: son cambios de contrato.

\section{Diseño recomendado}

Una versión sobria y clara del módulo podría adoptar esta forma:

\begin{lstlisting}
from __future__ import annotations

from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import Any, Optional

@dataclass(frozen=True)
class RunContext:
    run_id: str
    asof_date: str
    seed: int
    config_hash: str
    pipeline_version: str
    market: str = "US_EQ"
    parent_run_id: Optional[str] = None

@dataclass(frozen=True)
class GateResult:
    passed: bool
    score: float
    reasons: tuple[str, ...] = ()
    metrics: dict[str, float] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)

class DataProvider(ABC):
    @abstractmethod
    def fetch(self, asof_date, context: RunContext):
        raise NotImplementedError

class FeatureBuilder(ABC):
    @abstractmethod
    def build(self, raw_data, context: RunContext):
        raise NotImplementedError

class ModelTrainer(ABC):
    @abstractmethod
    def fit(self, features, labels, context: RunContext):
        raise NotImplementedError

class PortfolioEngine(ABC):
    @abstractmethod
    def rebalance(self, signals, context: RunContext):
        raise NotImplementedError

class Validator(ABC):
    @abstractmethod
    def validate(self, artifact, context: RunContext) -> GateResult:
        raise NotImplementedError
\end{lstlisting}

\section{Pruebas de conformidad recomendadas}

El módulo debe venir acompañado por tests de contrato, no solo tests unitarios de implementaciones.

\begin{lstlisting}
def test_gate_result_has_required_fields():
    result = GateResult(passed=True, score=0.99)
    assert result.passed is True
    assert isinstance(result.score, float)

def test_run_context_is_hashable_or_serializable_enough():
    ctx = RunContext(
        run_id="r1",
        asof_date="2026-03-01",
        seed=42,
        config_hash="abc123",
        pipeline_version="1.0.0",
    )
    assert ctx.run_id == "r1"

def test_portfolio_engine_contract():
    class DummyEngine(PortfolioEngine):
        def rebalance(self, signals, context):
            return {"weights": {}}

    engine = DummyEngine()
    out = engine.rebalance({}, RunContext(
        run_id="r1",
        asof_date="2026-03-01",
        seed=42,
        config_hash="abc123",
        pipeline_version="1.0.0",
    ))
    assert "weights" in out
\end{lstlisting}

Además, cada implementación concreta debería pasar un conjunto de tests parametrizados comunes que verifiquen la semántica del contrato.

\section{Casos límite y errores frecuentes}

Los fallos de diseño más comunes en un módulo de interfaces son:

\begin{itemize}
    \item interfaces demasiado anchas, llenas de métodos accesorios;
    \item firmas vagas que no fijan semántica;
    \item mezclar tipos internos experimentales con contratos públicos;
    \item usar el contexto como cajón de sastre para cualquier dato;
    \item devolver diccionarios ad hoc sin estructura estable;
    \item hacer depender el contrato de una implementación concreta.
\end{itemize}

Un buen módulo de interfaces evita justo eso: define poco, pero lo define con mucha disciplina.

\section{Definition of Done}

\texttt{simons\_core/interfaces.py} queda bien resuelto cuando:

\begin{itemize}
    \item cada etapa principal del pipeline tiene un contrato pequeño y estable;
    \item las implementaciones son sustituibles sin cambios aguas abajo;
    \item \texttt{RunContext} y \texttt{GateResult} capturan el estado transversal mínimo;
    \item la semántica de entrada y salida está fijada más allá de la firma;
    \item existen tests de conformidad reutilizables para todas las implementaciones.
\end{itemize}

\section{Conclusión}

La calidad de \texttt{interfaces.py} no reside en sonar sofisticado, sino en imponer orden. En un framework cuantitativo serio, este módulo es la capa que convierte una colección de componentes en un sistema composable. Si el contrato está bien diseñado, el pipeline puede crecer sin degenerar en acoplamiento accidental. Si el contrato está mal diseñado, ninguna cantidad de sofisticación algorítmica compensa la fragilidad estructural.

Por eso, el estándar correcto para este módulo es: minimalidad, semántica explícita, invariantes fuertes y versionado disciplinado.
