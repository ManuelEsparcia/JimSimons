Especificación Cuantitativa: simons_core/interfaces.py

0. Propósito y papel dentro del stack

simons_core/interfaces.py define la capa institucional de contratos abstractos del pipeline cuantitativo. Su función no es implementar lógica de datos, features, entrenamiento, optimización o validación, sino fijar una gramática estable de interacción entre componentes desacoplados. Es, en sentido fuerte, la frontera contractual del sistema: especifica qué debe garantizar cada etapa, qué tipo de objeto recibe, qué tipo de objeto devuelve y qué invariantes semánticos deben cumplirse para que la composición del pipeline sea segura, reproducible y libre de ambigüedad. :contentReference[oaicite:0]{index=0}

En un stack cuantitativo serio, este módulo no aporta alpha ni reduce drawdown directamente. Aporta algo más estructural: evita que la arquitectura derive en acoplamiento accidental, firmas vagas, outputs ad hoc y dependencias ocultas entre implementaciones. Su valor reside en imponer el principio:

    Implementation ≺ Contract

es decir, las implementaciones concretas deben adaptarse al contrato público, y no al revés.

La regla institucional debe ser esta:

    toda etapa del pipeline
    debe ser intercambiable, testeable y composable
    porque obedece un contrato explícito
    de firma, semántica, invariantes y errores.

1. Objetivo formal del módulo

Sea la cadena abstracta del pipeline:

    D_t -> X_t -> F_t -> M_t -> W_t -> G_t

donde:
- D_t representa datos fuente observados a fecha t;
- X_t representa datos limpios o estructurados;
- F_t representa features;
- M_t representa un artefacto de modelo;
- W_t representa una decisión de cartera;
- G_t representa un resultado de validación o gate.

El módulo implementa una familia de aplicaciones contractuales:

    DataProvider    : (asof_date, RunContext) -> PriceData
    FeatureBuilder  : (PriceData, RunContext) -> FeatureMatrix
    ModelTrainer    : (FeatureMatrix, Labels, RunContext) -> ModelArtifact
    PortfolioEngine : (Signals, RunContext) -> PortfolioDecision
    Validator       : (Artifact, RunContext) -> GateResult

La propiedad central del módulo es que la composición entre etapas sea semánticamente válida:

    output(Stage_i) ⊨ input(Stage_{i+1})

Esto significa que el contrato no se limita a la existencia de un método, sino a la compatibilidad estructural, temporal y semántica entre productores y consumidores.

2. Principios no negociables

P1. Contract-first.
Las interfaces describen el sistema; las implementaciones lo satisfacen.

P2. Minimalidad.
Cada contrato debe exponer sólo los métodos esenciales de su responsabilidad.

P3. Semántica explícita.
No basta con una firma válida; el significado de entradas y salidas debe quedar fijado.

P4. Composabilidad.
Las salidas de una etapa deben poder alimentar la siguiente sin adaptadores ambiguos.

P5. No look-ahead.
Ningún contrato puede permitir datos con timestamp futuro respecto a asof_date.

P6. Reproducibilidad.
El comportamiento de una implementación conforme debe ser reproducible dado el mismo RunContext, semilla y configuración.

P7. Error model explícito.
Las violaciones de contrato deben fallar de forma temprana y tipada.

P8. Evolución disciplinada.
Cambios incompatibles en firma o semántica deben versionarse, no introducirse silenciosamente.

3. Ámbito y no-objetivos

El módulo cubre:

- contratos abstractos de etapas del pipeline;
- tipos transversales de ejecución;
- tipado semántico mínimo de I/O;
- invariantes de composición;
- error hierarchy contractual;
- base para tests de conformidad.

Queda fuera de alcance:

- implementación concreta de loaders, modelos o motores;
- decisiones de negocio sobre universos, señales o restricciones;
- persistencia física de artefactos;
- lógica específica de calendarización;
- detalles internos de optimización o entrenamiento.

4. Álgebra del contrato

Cada contrato debe entenderse como:

    Contract = Signature × Semantics × Invariants × ErrorModel

No basta con definir:

    def fit(features, labels, context): ...

También debe quedar fijado:
- qué significa features;
- qué alineación temporal se exige;
- qué tipo de objeto debe devolverse;
- qué errores son admisibles;
- qué propiedades deben mantenerse para que el pipeline siga siendo correcto.

5. Tipos transversales obligatorios

5.1. RunContext

RunContext debe ser el contenedor mínimo de estado transversal de ejecución. Debe ser pequeño, inmutable, serializable y suficiente para:
- trazabilidad;
- reproducibilidad;
- contextualización temporal;
- evolución de versión del pipeline.

Implementación recomendada:

~~~python
from dataclasses import dataclass
import pandas as pd

@dataclass(frozen=True)
class RunContext:
    run_id: str
    asof_date: pd.Timestamp
    seed: int
    config_hash: str
    pipeline_version: str = "1.0.0"
    market: str = "US_EQ"
    parent_run_id: str | None = None
~~~

Invariantes:

I1. run_id identifica unívocamente la ejecución.
I2. asof_date define el horizonte temporal máximo permitido para datos de entrada.
I3. seed fija reproducibilidad pseudoaleatoria.
I4. config_hash identifica la configuración efectiva.
I5. pipeline_version fija la semántica contractual vigente.

5.2. GateResult

GateResult no debe ser un bool desnudo. Debe representar una decisión de validación con score normalizado y razones trazables.

Implementación recomendada:

~~~python
from dataclasses import dataclass, field
from typing import Any

@dataclass(frozen=True)
class GateResult:
    passed: bool
    score: float
    threshold: float = 0.8
    reasons: tuple[str, ...] = ()
    metrics: dict[str, float] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)

    def __post_init__(self):
        if not (0.0 <= self.score <= 1.0):
            raise ValueError("GateResult.score must belong to [0,1].")
        if not (0.0 <= self.threshold <= 1.0):
            raise ValueError("GateResult.threshold must belong to [0,1].")
        if self.passed != (self.score >= self.threshold):
            raise ValueError("GateResult.passed inconsistent with score >= threshold.")
~~~

Invariantes:

I1. score ∈ [0,1]
I2. threshold ∈ [0,1]
I3. passed <=> score >= threshold
I4. reasons explica fallos o borderline outcomes
I5. metrics contiene evidencia cuantitativa, no texto libre ambiguo

6. Tipos de I/O recomendados

Para subir el módulo a estándar realmente fuerte, las interfaces no deberían trabajar con Any o DataFrame genéricos sin semántica. Deben apoyarse en tipos/schemas explícitos.

Ejemplo razonable:

~~~python
from pydantic import BaseModel
from typing import Any
import pandas as pd

class PriceData(BaseModel):
    asof_date: pd.Timestamp
    symbols: list[str]
    prices: pd.DataFrame
    metadata: dict[str, Any] = {}

class FeatureMatrix(BaseModel):
    asof_date: pd.Timestamp
    features: pd.DataFrame
    feature_names: list[str]
    metadata: dict[str, Any] = {}

class ModelArtifact(BaseModel):
    model_ref: str
    artifact_path: str
    params: dict[str, Any]
    metrics: dict[str, float]

class PortfolioDecision(BaseModel):
    timestamp: pd.Timestamp
    weights: pd.Series
    turnover_l1: float
    notional_exposure: float
    constraints_active: list[str] = []
~~~

Regla crítica:
si el repositorio no quiere acoplarse a pydantic, al menos debe definir dataclasses o Protocols estables equivalentes. Lo importante no es la librería, sino que el contrato de tipos deje de ser “un DataFrame cualquiera”.

7. Contrato DataProvider

Responsabilidad:
producir datos point-in-time válidos para una fecha asof_date dada, sin leakage temporal.

Firma recomendada:

~~~python
from abc import ABC, abstractmethod

class DataProvider(ABC):
    @abstractmethod
    def fetch(self, asof_date: pd.Timestamp, context: RunContext) -> PriceData:
        """
        Fetch point-in-time data.
        Must satisfy max(timestamp(output)) <= asof_date.
        """
        raise NotImplementedError
~~~

Precondiciones:
- asof_date debe ser válida bajo el calendario del sistema.
- context debe contener run_id, seed y config_hash consistentes.

Postcondiciones:
- el output debe cumplir schema PriceData o equivalente.
- no debe contener datos futuros.

Invariante central:

    max(timestamp(fetch(asof_date, ctx))) <= asof_date

Ésta es la formulación contractual mínima anti-look-ahead.

8. Contrato FeatureBuilder

Responsabilidad:
transformar datos válidos y alineados en una matriz de features usable por etapas posteriores.

Firma recomendada:

~~~python
class FeatureBuilder(ABC):
    @abstractmethod
    def build(self, raw_data: PriceData, context: RunContext) -> FeatureMatrix:
        """
        Transform raw point-in-time data into aligned features.
        """
        raise NotImplementedError
~~~

Precondiciones:
- raw_data debe ser PIT-correct y estructuralmente válido.

Postcondiciones:
- el output debe estar alineado por índices temporales e instrumentales.
- no puede introducir columnas implícitas no documentadas.
- debe ser compatible con el contrato esperado por ModelTrainer.

Invariante de compatibilidad:

    FeatureBuilder.build(raw_data, ctx) ⊨ FeatureMatrix

9. Contrato ModelTrainer

Responsabilidad:
ajustar un modelo o construir un artefacto de entrenamiento reproducible a partir de features y labels alineadas.

Firma recomendada:

~~~python
class ModelTrainer(ABC):
    @abstractmethod
    def fit(
        self,
        features: FeatureMatrix,
        labels: pd.Series,
        context: RunContext,
    ) -> ModelArtifact:
        """
        Train a model artifact reproducibly given the same seed and data.
        """
        raise NotImplementedError

    @abstractmethod
    def predict(
        self,
        features: FeatureMatrix,
        model: ModelArtifact,
        context: RunContext,
    ) -> pd.DataFrame:
        """
        Produce prediction scores aligned with the feature matrix.
        """
        raise NotImplementedError
~~~

Precondiciones:
- features y labels deben estar alineados temporalmente.
- no debe existir leakage entre train y target.
- context.seed debe controlar fuentes pseudoaleatorias relevantes.

Postcondiciones:
- fit devuelve un artefacto trazable.
- predict devuelve scores alineados con el contrato de PortfolioEngine.

Invariante de reproducibilidad:

    fit(features, labels, context(seed=s)) = model_s

y, dentro de tolerancias documentadas,

    model_s = model_s'

para entradas idénticas y misma semilla.

10. Contrato PortfolioEngine

Responsabilidad:
transformar señales o predicciones en una decisión de cartera factible.

Firma recomendada:

~~~python
class PortfolioEngine(ABC):
    @abstractmethod
    def rebalance(
        self,
        signals: pd.DataFrame,
        context: RunContext,
    ) -> PortfolioDecision:
        """
        Convert signals into a feasible portfolio decision.
        """
        raise NotImplementedError
~~~

Precondiciones:
- signals debe referirse a una única fecha efectiva de decisión.
- los nombres de activos y el índice deben ser consistentes.

Postcondiciones:
- la salida debe pertenecer al espacio de decisiones factibles.
- debe quedar claro si la semántica es pesos, órdenes o target portfolio.

Invariante de factibilidad:

    rebalance(signals, ctx) ∈ 𝓦

donde 𝓦 es el conjunto de decisiones admitidas por el contrato del engine.

11. Contrato Validator

Responsabilidad:
evaluar si un artefacto cumple criterios de calidad y permitir o impedir su promoción.

Firma recomendada:

~~~python
class Validator(ABC):
    @abstractmethod
    def validate(self, artifact: object, context: RunContext) -> GateResult:
        """
        Validate an artifact and return a normalized gate result.
        """
        raise NotImplementedError
~~~

Postcondición:
- siempre debe devolver GateResult, no bool, int ni dict ad hoc.

Regla clave:
score debe tener una semántica consistente y orientada. Convención recomendada:

    1.0 = mejor / plenamente satisfactorio
    0.0 = peor / plenamente insatisfactorio

Esto elimina la ambigüedad típica de gates donde no se sabe si “menos es mejor” o “más es mejor”.

12. Error hierarchy contractual

Una arquitectura PhD-level seria no puede dejar el modelo de error en ValueError genérico para todo.

Se recomienda al menos:

~~~python
class ContractViolation(Exception):
    """Base class for contract-level violations."""

class DataLeakageError(ContractViolation):
    """Detected future information beyond asof_date."""

class SchemaViolation(ContractViolation):
    """Input or output structure does not satisfy expected schema."""

class FeasibilityError(ContractViolation):
    """Portfolio decision violates declared feasibility rules."""

class ReproducibilityError(ContractViolation):
    """Same seed/input produced inconsistent outputs."""

class GateSemanticsError(ContractViolation):
    """GateResult is internally inconsistent."""
~~~

Regla institucional:
violaciones estructurales del contrato deben fallar pronto y de forma tipada.

13. Versionado del contrato

Las interfaces del core son API pública. Por tanto, deben admitir evolución explícita.

Política recomendada:
- cambios retrocompatibles: minor
- cambios incompatibles: major

Si se requiere evolución sustancial, puede adoptarse un patrón tipo:

    DataProviderV1
    DataProviderV2

o mantener un único nombre con pipeline_version y migración documentada. Lo importante es no cambiar la semántica silenciosamente.

14. Protocol vs ABC

El módulo puede optar por:
- abc.ABC si quiere herencia nominal fuerte;
- typing.Protocol si quiere tipado estructural flexible.

Ambas opciones son válidas. Para un framework gobernado y estable, ABC suele ser más sencillo de auditar; para integraciones flexibles, Protocol puede ser más cómodo.

Una opción híbrida razonable es:
- ABC para contratos nucleares;
- Protocol para roles pequeños como Validator o Logger-like interfaces.

15. Tests de conformidad

El módulo debe acompañarse de tests reutilizables que verifiquen que una implementación concreta cumple el contrato.

16. Failure modes críticos

El módulo debe proteger al sistema contra:

F1. Interfaces demasiado anchas.
Más métodos ≠ mejor contrato.

F2. Firmas vagas.
DataFrame in / DataFrame out sin semántica es deuda contractual.

F3. Output ad hoc.
Dicts arbitrarios rompen composabilidad.

F4. GateResult ambiguo.
Si score no tiene orientación clara, el gate deja de ser auditable.

F5. Look-ahead no formalizado.
Si DataProvider no fija max(timestamp(output)) <= asof_date, la arquitectura ya nace contaminada.

F6. Context como cajón de sastre.
RunContext no debe convertirse en bolsa de cualquier estado mutable.

17. MVP obligatorio

El MVP del módulo debe incluir:
- RunContext inmutable y tipado;
- GateResult normalizado en [0,1];
- contratos DataProvider, FeatureBuilder, ModelTrainer, PortfolioEngine y Validator;
- error hierarchy base;
- tests mínimos de conformidad;
- convención explícita anti-look-ahead;
- política de versionado contractual.

18. Segunda fase razonable

Como extensiones útiles:
- Protocols adicionales para async/batch interfaces;
- soporte batch o fetch_batch;
- streaming interfaces controladas;
- richer schema types;
- integración con tracing u observabilidad;
- versioning explícito V1/V2 de interfaces si el framework crece.

19. No-objetivos

No pertenece al alcance de este módulo:
- definir algoritmos concretos de ML;
- optimizar pesos;
- leer de APIs externas;
- validar estadísticamente estrategias;
- imponer formatos internos de almacenamiento;
- implementar lógica de negocio de cada etapa.

La misión aquí es estrictamente contractual.

20. Criterios de aceptación

El módulo sólo debe considerarse correcto si cumple simultáneamente:

- cada etapa principal del pipeline tiene un contrato pequeño y estable;
- las implementaciones pueden sustituirse sin reescribir consumidores;
- RunContext y GateResult capturan el estado transversal mínimo necesario;
- la semántica de entrada y salida está fijada más allá de la firma;
- existen errores tipados para violaciones de contrato;
- la composición del pipeline es estructural y temporalmente verificable.

21. Conclusión

simons_core/interfaces.py es una de las piezas más importantes del core porque convierte una colección de módulos potencialmente heterogéneos en un sistema realmente composable. Su valor no está en sonar abstracto, sino en imponer disciplina: firmas pequeñas, semántica explícita, invariantes fuertes, errors tipados y versionado contractual. En un framework cuantitativo serio, eso no es decoración arquitectónica; es la condición necesaria para que el pipeline crezca sin degenerar en acoplamiento accidental y comportamiento implícito. Ese es exactamente el estándar que este módulo debe imponer.
