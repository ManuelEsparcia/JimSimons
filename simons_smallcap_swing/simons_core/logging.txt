\section{Propósito del módulo}

El módulo \texttt{simons\_core/logging.py} define la capa de observabilidad estructurada del sistema. Su función no es ``monitorizar rentabilidad'' ni emitir conclusiones cuantitativas sobre estrategia, sino proporcionar un registro consistente, parseable y trazable de lo que hace cada componente del pipeline.

En un framework cuantitativo serio, el logging existe para resolver cinco problemas muy concretos:

\begin{enumerate}[label=\arabic*.]
    \item reconstruir causalmente qué ocurrió en una ejecución;
    \item depurar fallos sin depender de prints ad hoc;
    \item correlacionar eventos entre módulos y etapas;
    \item exponer evidencia operativa en tests, backtests, research y producción;
    \item separar observabilidad de lógica de negocio.
\end{enumerate}

Si \texttt{interfaces.py} fija contratos entre componentes, \texttt{logging.py} fija el contrato observacional del sistema.

\section{Principio rector}

La regla de diseño es:

\[
\text{Log event} = \text{structured fact about execution}
\]

Un log no debe ser una cadena literaria ni un volcado arbitrario de objetos Python. Debe ser un hecho estructurado, contextualizado y consumible por humanos y máquinas.

Por tanto, la calidad del módulo no depende de cuántos logs produce, sino de si esos logs son:

\begin{itemize}
    \item consistentes;
    \item suficientemente informativos;
    \item baratos de emitir;
    \item seguros desde el punto de vista de datos;
    \item estables desde el punto de vista de esquema.
\end{itemize}

\section{Responsabilidad exacta}

\texttt{logging.py} debe encargarse de:

\begin{itemize}
    \item construir loggers con contexto base;
    \item definir el esquema canónico de evento;
    \item propagar identificadores de correlación;
    \item ofrecer utilidades ergonómicas para instrumentar pasos del pipeline;
    \item serializar excepciones y payloads de manera controlada;
    \item imponer reglas mínimas de redacción, niveles y parseabilidad.
\end{itemize}

No debe encargarse de:

\begin{itemize}
    \item configurar métricas de negocio;
    \item decidir alertas de trading;
    \item deducir conclusiones estadísticas sobre resultados;
    \item mezclar logging con tracing distribuido complejo salvo interfaz mínima;
    \item acoplarse a un backend único de observabilidad.
\end{itemize}

\section{API pública mínima}

Una API pública razonable para el módulo es:

\begin{lstlisting}
get_logger(name: str, run_context: RunContext | None = None) -> StructuredLogger

with_step(logger: StructuredLogger, step: str, **fields)

bind_context(logger: StructuredLogger, **fields) -> StructuredLogger

serialize_exception(exc: Exception) -> dict[str, object]
\end{lstlisting}

Si se desea una clase explícita:

\begin{lstlisting}
class StructuredLogger:
    def debug(self, event: str, **payload) -> None: ...
    def info(self, event: str, **payload) -> None: ...
    def warning(self, event: str, **payload) -> None: ...
    def error(self, event: str, **payload) -> None: ...
    def exception(self, event: str, exc: Exception | None = None, **payload) -> None: ...
    def bind(self, **fields) -> "StructuredLogger": ...
\end{lstlisting}

El objetivo no es inventar una jerarquía compleja, sino fijar una interfaz pequeña y estable para observabilidad estructurada.

\section{Esquema canónico de evento}

Todo evento emitido por el sistema debe seguir un sobre canónico. Como mínimo:

\begin{center}
\begin{tabular}{lll}
\toprule
Campo & Tipo & Obligatorio \\
\midrule
\texttt{ts\_utc} & string ISO-8601 UTC & Sí \\
\texttt{level} & string & Sí \\
\texttt{logger} & string & Sí \\
\texttt{event} & string & Sí \\
\texttt{run\_id} & string & Recomendado / Sí en pipeline \\
\texttt{step} & string & Opcional \\
\texttt{module} & string & Opcional \\
\texttt{payload} & object & Sí \\
\texttt{error\_code} & string & Opcional \\
\texttt{trace\_id} & string & Opcional \\
\texttt{span\_id} & string & Opcional \\
\bottomrule
\end{tabular}
\end{center}

Una forma JSON de referencia sería:

\begin{lstlisting}
{
  "ts_utc": "2026-03-06T10:15:24.183Z",
  "level": "INFO",
  "logger": "simons.features.momentum",
  "event": "features_built",
  "run_id": "run_20260306_001",
  "step": "feature_build",
  "module": "features",
  "payload": {
    "rows": 18420,
    "cols": 57,
    "symbols": 312
  }
}
\end{lstlisting}

\section{Semántica de niveles}

Los niveles deben usarse con disciplina:

\begin{itemize}
    \item \texttt{DEBUG}: detalle técnico útil para depuración local o entornos de test.
    \item \texttt{INFO}: hitos normales del pipeline y hechos operativos esperables.
    \item \texttt{WARNING}: desviaciones recuperables o situaciones anómalas no fatales.
    \item \texttt{ERROR}: fallo de una operación concreta con impacto local o de etapa.
    \item \texttt{CRITICAL}: fallo sistémico o imposibilidad de continuar la ejecución.
\end{itemize}

Un error común es usar \texttt{INFO} para todo o, al contrario, inflar \texttt{ERROR} con incidencias que no rompen contrato. El valor del módulo depende de que la taxonomía de niveles sea consistente entre componentes.

\section{Context propagation}

La observabilidad del pipeline requiere propagar contexto mínimo entre etapas. Sea \(L\) un logger y \(C\) un conjunto de campos contextuales:

\[
L' = \texttt{bind}(L, C)
\]

Entonces todo evento emitido por \(L'\) debe incorporar \(C\) salvo sobrescritura explícita. Los campos contextuales mínimos deseables son:

\begin{itemize}
    \item \texttt{run\_id}
    \item \texttt{asof\_date}
    \item \texttt{pipeline\_version}
    \item \texttt{module}
    \item \texttt{step}, cuando proceda
\end{itemize}

La regla es simple: el consumidor de logs no debe reconstruir contexto a partir del nombre del archivo ni de inferencias frágiles.

\section{Context manager de paso}

El patrón \texttt{with\_step(...)} debe permitir instrumentar bloques de ejecución con inicio, fin, duración y fallo, sin repetir boilerplate.

Semántica esperada:

\begin{itemize}
    \item al entrar, emitir un evento \texttt{step\_started};
    \item al salir con éxito, emitir \texttt{step\_completed};
    \item al salir por excepción, emitir \texttt{step\_failed} con serialización controlada del error;
    \item incluir duración en milisegundos o segundos.
\end{itemize}

Ejemplo:

\begin{lstlisting}
with with_step(logger, "feature_build", universe="smallcap") as step_logger:
    features = builder.build(raw_data, context)
\end{lstlisting}

Esto debería producir eventos estructurados como:

\begin{lstlisting}
{"event": "step_started", "step": "feature_build", ...}
{"event": "step_completed", "step": "feature_build", "payload": {"duration_ms": 281.4}, ...}
\end{lstlisting}

o, en caso de error:

\begin{lstlisting}
{"event": "step_failed", "step": "feature_build", "error_code": "FEATURE_BUILD_ERROR", ...}
\end{lstlisting}

\section{Serialización de excepciones}

La serialización de excepciones debe ser explícita y estable. No basta con hacer \texttt{str(exc)}. La forma recomendada es:

\begin{lstlisting}
{
  "type": "ValueError",
  "message": "n must be >= 1",
  "repr": "ValueError('n must be >= 1')",
  "traceback": "...",   # opcional o truncado según entorno
  "caused_by": null
}
\end{lstlisting}

Reglas:

\begin{itemize}
    \item incluir tipo de excepción;
    \item incluir mensaje;
    \item permitir traceback controlado;
    \item truncar o sanitizar contenidos excesivos;
    \item evitar serializar objetos no JSON-compatibles sin transformación.
\end{itemize}

\section{Payload policy}

El campo \texttt{payload} debe seguir reglas estrictas:

\begin{enumerate}[label=\textbf{P\arabic*.}]
    \item Debe ser JSON-serializable o convertible de forma controlada.
    \item Debe contener hechos operativos, no objetos arbitrarios gigantes.
    \item Debe evitar duplicar campos que pertenecen al sobre principal.
    \item Debe respetar política de redacción de datos.
\end{enumerate}

Ejemplos correctos:

\begin{itemize}
    \item número de filas procesadas;
    \item nombres de split o etapa;
    \item duración;
    \item conteos de símbolos;
    \item parámetros pequeños y estables.
\end{itemize}

Ejemplos incorrectos:

\begin{itemize}
    \item DataFrames completos;
    \item arrays masivos;
    \item objetos de modelo no serializados;
    \item secretos o credenciales;
    \item información personal o sensible.
\end{itemize}

\section{Redacción y seguridad de datos}

El módulo debe imponer una política base de redacción:

\begin{itemize}
    \item no PII;
    \item no secretos;
    \item no tokens de acceso;
    \item no rutas o URIs sensibles salvo necesidad operativa explícita;
    \item no payloads crudos con datos de entrada completos si eso rompe privacidad o volumen.
\end{itemize}

Idealmente, el logger debe poder aplicar una función de sanitización previa a la emisión. En forma abstracta, si \(x\) es un payload:

\[
x \mapsto \texttt{sanitize}(x) = x'
\]

donde \(x'\) conserva utilidad operativa pero elimina campos prohibidos o peligrosos.

\section{Sinks y desacoplamiento}

\texttt{logging.py} debe desacoplar la emisión del backend. El módulo puede escribir a:

\begin{itemize}
    \item stdout/stderr en JSON;
    \item fichero local rotado;
    \item collector externo;
    \item test sink o memory sink.
\end{itemize}

La interfaz no debe acoplarse a una tecnología concreta como Splunk, Datadog o ELK. La política correcta es: formato estable, transporte intercambiable.

\section{Sampling y control de volumen}

En sistemas de research y backtesting el volumen de eventos puede crecer rápido. El módulo debe prever, al menos a nivel conceptual:

\begin{itemize}
    \item no muestrear \texttt{ERROR}/\texttt{CRITICAL};
    \item muestrear selectivamente \texttt{DEBUG} de alta frecuencia;
    \item agregar eventos repetitivos cuando sea razonable;
    \item permitir niveles distintos por módulo o entorno.
\end{itemize}

La observabilidad útil no es ``loggear todo'', sino registrar lo suficiente sin degradar rendimiento ni generar ruido estructural.

\section{Compatibilidad de esquema}

El evento de log debe tratarse como un contrato de datos. Sea \(E_v\) el esquema de eventos en versión \(v\). El módulo debe preservar:

\[
E_{v+1} \supseteq E_v
\]

para cambios retrocompatibles, o bien versionar explícitamente el formato cuando se rompa compatibilidad.

Cambios incompatibles típicos:

\begin{itemize}
    \item renombrar \texttt{event};
    \item mover \texttt{error\_code} dentro de \texttt{payload};
    \item cambiar \texttt{ts\_utc} de ISO-8601 a entero sin coordinación;
    \item convertir \texttt{payload} en string arbitrario.
\end{itemize}

Un buen módulo de logging no solo emite eventos; gobierna la estabilidad de su esquema.

\section{Diseño recomendado}

Una implementación sobria podría verse así:

\begin{lstlisting}
from __future__ import annotations

from contextlib import contextmanager
from dataclasses import asdict, is_dataclass
from datetime import datetime, timezone
import json
import time
import traceback
from typing import Any

def _utc_now_iso() -> str:
    return datetime.now(timezone.utc).isoformat(timespec="milliseconds").replace("+00:00", "Z")

def _to_jsonable(value: Any) -> Any:
    if is_dataclass(value):
        return asdict(value)
    if isinstance(value, (str, int, float, bool)) or value is None:
        return value
    if isinstance(value, dict):
        return {str(k): _to_jsonable(v) for k, v in value.items()}
    if isinstance(value, (list, tuple)):
        return [_to_jsonable(v) for v in value]
    return repr(value)

def serialize_exception(exc: Exception) -> dict[str, Any]:
    return {
        "type": exc.__class__.__name__,
        "message": str(exc),
        "repr": repr(exc),
        "traceback": "".join(traceback.format_exception(type(exc), exc, exc.__traceback__)),
    }

class StructuredLogger:
    def __init__(self, name: str, base_fields: dict[str, Any] | None = None):
        self.name = name
        self.base_fields = base_fields or {}

    def bind(self, **fields) -> "StructuredLogger":
        merged = {**self.base_fields, **fields}
        return StructuredLogger(self.name, merged)

    def _emit(self, level: str, event: str, **payload) -> None:
        record = {
            "ts_utc": _utc_now_iso(),
            "level": level,
            "logger": self.name,
            "event": event,
            **self.base_fields,
            "payload": _to_jsonable(payload),
        }
        print(json.dumps(record, ensure_ascii=False))

    def debug(self, event: str, **payload) -> None:
        self._emit("DEBUG", event, **payload)

    def info(self, event: str, **payload) -> None:
        self._emit("INFO", event, **payload)

    def warning(self, event: str, **payload) -> None:
        self._emit("WARNING", event, **payload)

    def error(self, event: str, **payload) -> None:
        self._emit("ERROR", event, **payload)

    def exception(self, event: str, exc: Exception | None = None, **payload) -> None:
        if exc is not None:
            payload["exception"] = serialize_exception(exc)
        self._emit("ERROR", event, **payload)

def get_logger(name: str, run_context=None) -> StructuredLogger:
    base = {}
    if run_context is not None:
        for field in ("run_id", "asof_date", "pipeline_version", "market"):
            if hasattr(run_context, field):
                base[field] = getattr(run_context, field)
    return StructuredLogger(name, base)

@contextmanager
def with_step(logger: StructuredLogger, step: str, **fields):
    step_logger = logger.bind(step=step, **fields)
    t0 = time.perf_counter()
    step_logger.info("step_started")
    try:
        yield step_logger
    except Exception as exc:
        dt_ms = 1000.0 * (time.perf_counter() - t0)
        step_logger.exception("step_failed", exc=exc, duration_ms=dt_ms)
        raise
    else:
        dt_ms = 1000.0 * (time.perf_counter() - t0)
        step_logger.info("step_completed", duration_ms=dt_ms)
\end{lstlisting}

\section{Buenas prácticas de uso}

Uso correcto:

\begin{lstlisting}
logger = get_logger(__name__, run_context)

with with_step(logger, "feature_build", universe="smallcap") as log:
    features = builder.build(raw_data, context)
    log.info("features_built", rows=len(features), cols=len(features.columns))
\end{lstlisting}

Uso incorrecto:

\begin{lstlisting}
logger.info("Sharpe is amazing!!!")
logger.info("dump_everything", payload=df.to_dict())
logger.error("something bad happened maybe idk")
\end{lstlisting}

El primer bloque produce hechos observables y parseables. El segundo produce ruido, payload excesivo y semántica pobre.

\section{Métricas derivables del log}

Aunque \texttt{logging.py} no define métricas de negocio, sí debe permitir derivar métricas operativas de forma fiable, por ejemplo:

\begin{itemize}
    \item \texttt{log\_count\_by\_level}
    \item \texttt{error\_count\_by\_module}
    \item \texttt{step\_duration\_p50/p95/p99}
    \item \texttt{step\_failure\_rate}
    \item \texttt{events\_per\_run}
\end{itemize}

La regla es importante: primero hechos estructurados; las métricas se construyen encima.

\section{Casos límite y errores frecuentes}

El módulo debe manejar o prevenir explícitamente:

\begin{itemize}
    \item payloads no serializables;
    \item colisión entre campos base y payload;
    \item excepciones anidadas;
    \item timestamps naive;
    \item logs emitidos sin \texttt{run\_id} en etapas que lo exigen;
    \item emisión recursiva por errores dentro del propio logger.
\end{itemize}

Errores habituales de diseño:

\begin{itemize}
    \item mezclar mensaje libre y esquema estructurado;
    \item registrar objetos demasiado grandes;
    \item duplicar contexto en cada llamada manualmente;
    \item no distinguir fallo recuperable de fallo fatal;
    \item cambiar formato de evento sin gobernanza.
\end{itemize}

\section{Pruebas mínimas recomendadas}

\begin{lstlisting}
def test_get_logger_binds_run_context():
    class Ctx:
        run_id = "r1"
        asof_date = "2026-03-06"
        pipeline_version = "1.0.0"
        market = "US_EQ"

    logger = get_logger("test.logger", Ctx())
    assert logger.base_fields["run_id"] == "r1"

def test_with_step_emits_completion():
    logger = StructuredLogger("test")
    # In production tests, capture stdout or sink output
    with with_step(logger, "dummy_step"):
        pass

def test_serialize_exception_contains_type_and_message():
    try:
        raise ValueError("bad input")
    except ValueError as exc:
        data = serialize_exception(exc)
    assert data["type"] == "ValueError"
    assert "bad input" in data["message"]

def test_payload_is_jsonable():
    logger = StructuredLogger("test")
    logger.info("sample_event", x=1, y={"a": [1, 2, 3]})
\end{lstlisting}

Además, conviene mantener tests de snapshot o schema para garantizar que el formato JSON no cambia accidentalmente.

\section{Definition of Done}

\texttt{simons\_core/logging.py} queda correctamente resuelto cuando:

\begin{itemize}
    \item todo evento emitido sigue un esquema estable;
    \item los loggers propagan contexto mínimo sin boilerplate excesivo;
    \item los pasos del pipeline pueden instrumentarse con inicio, fin, duración y fallo;
    \item excepciones y payloads se serializan de manera controlada;
    \item no se filtran datos sensibles;
    \item el backend de transporte es intercambiable;
    \item existen tests que fijan parseabilidad y compatibilidad del formato.
\end{itemize}

\section{Conclusión}

La calidad de \texttt{logging.py} no reside en sonar industrial, sino en imponer disciplina observacional. En un sistema cuantitativo serio, el logging estructurado es la memoria verificable del pipeline: permite entender qué ocurrió, dónde falló y bajo qué contexto, sin contaminar la lógica de negocio ni depender de convenciones frágiles.

Por eso, el estándar correcto para este módulo es: esquema estable, contexto propagado, serialización segura, semántica clara y cero ambigüedad operativa.
