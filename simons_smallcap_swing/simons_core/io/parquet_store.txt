\section{Propósito}

Este módulo define una capa de persistencia columnar para datasets de investigación y producción en un sistema de \emph{small-cap swing trading}. Su misión es proporcionar lectura y escritura reproducibles, eficientes y auditables sobre artefactos tabulares de gran volumen, con semántica explícita de esquema, particionado y versionado.

El objetivo del módulo es infraestructural: reducir I/O innecesario, estabilizar tiempos de acceso, mejorar interoperabilidad analítica y formalizar contratos de almacenamiento. No se atribuyen a este componente mejoras directas de Sharpe, CAGR, hit rate o drawdown, pues dichas métricas pertenecen al dominio del modelo, los datos y la ejecución, no al del formato de persistencia.

\section{Ámbito}

El módulo cubre:

\begin{itemize}[leftmargin=1.5em]
    \item escritura de tablas en formato Parquet;
    \item lectura selectiva con proyección de columnas y filtros;
    \item validación de esquema antes y después de escritura;
    \item particionado físico por claves de acceso frecuentes;
    \item manifiestos y metadatos de snapshot;
    \item detección de corrupción y aislamiento de particiones inválidas;
    \item observabilidad operativa de volumen, latencia y cobertura de lectura.
\end{itemize}

Quedan fuera de alcance:

\begin{itemize}[leftmargin=1.5em]
    \item transacciones ACID multiobjeto al estilo \emph{table formats} avanzados;
    \item control distribuido de concurrencia fuerte;
    \item evolución automática de esquema sin validación explícita;
    \item gobierno integral de data lake.
\end{itemize}

\section{Motivación formal}

Sea una tabla lógica $T \in \mathbb{R}^{N \times d}$ con $N$ filas y $d$ columnas. Si una consulta requiere sólo un subconjunto $C \subseteq \{1,\dots,d\}$ y un predicado $P$ que activa poda de particiones o \emph{row-group pruning}, el coste de I/O efectivo puede aproximarse por

\[
\mathrm{IO}_{\mathrm{eff}}
\approx
S_{\mathrm{phys}} \cdot \rho_C \cdot \rho_P,
\]

donde:

\begin{itemize}[leftmargin=1.5em]
    \item $S_{\mathrm{phys}}$ es el tamaño físico almacenado;
    \item $\rho_C \in (0,1]$ es la fracción de columnas materializadas;
    \item $\rho_P \in (0,1]$ es la fracción de datos no descartada por poda.
\end{itemize}

Frente a formatos fila-a-fila o serializaciones monolíticas, Parquet permite explotar dos propiedades estructurales útiles para pipelines cuantitativos:

\begin{enumerate}[leftmargin=1.7em]
    \item \textbf{proyección columnar}, que evita cargar columnas no requeridas;
    \item \textbf{pruning físico}, que evita leer particiones o grupos irrelevantes.
\end{enumerate}

La mejora buscada, por tanto, es operacional: menor latencia y menor volumen leído/escrito bajo consultas parciales.

\section{Principios de diseño}

\begin{enumerate}[label=\textbf{P\arabic*.},leftmargin=1.7em]
    \item \textbf{Contrato explícito de esquema.} Ninguna tabla se persiste sin validar estructura y tipos.
    \item \textbf{Persistencia reproducible.} Una misma tabla lógica y una misma configuración de escritura deben generar un artefacto compatible y auditable.
    \item \textbf{Separación entre snapshot lógico y layout físico.} El consumidor razona sobre datasets y versiones, no sobre rutas ad hoc.
    \item \textbf{Atomicidad a nivel de publicación.} Un snapshot sólo es visible cuando su publicación queda completada.
    \item \textbf{Lectura sobria.} El lector debe poder proyectar columnas, aplicar filtros y validar metadatos sin acoplarse a detalles accidentales del layout.
    \item \textbf{Fallo recuperable.} Una partición corrupta no debe invalidar el dataset completo salvo decisión explícita.
\end{enumerate}

\section{Modelo de datos y snapshot}

El store debe tratar cada escritura como una publicación de snapshot. Un snapshot está compuesto por:

\begin{itemize}[leftmargin=1.5em]
    \item uno o varios ficheros Parquet;
    \item un manifiesto de snapshot;
    \item metadatos de esquema;
    \item estadísticas resumidas de publicación.
\end{itemize}

Definimos un snapshot como la tupla
\[
\mathcal{S} = (\Pi, \Sigma, M, \tau),
\]
donde:

\begin{itemize}[leftmargin=1.5em]
    \item $\Pi$ es el conjunto de paths físicos publicados;
    \item $\Sigma$ es el esquema lógico validado;
    \item $M$ es el manifiesto con hashes, conteos y particiones;
    \item $\tau$ es el instante de publicación.
\end{itemize}

La unidad visible para el consumidor es $\mathcal{S}$, no cada fichero individual.

\section{Invariantes}

El módulo debe respetar, como mínimo, los siguientes invariantes:

\begin{enumerate}[label=\textbf{I\arabic*.},leftmargin=1.7em]
    \item Todo snapshot visible tiene manifiesto íntegro.
    \item Todo snapshot visible referencia un esquema compatible con \texttt{schema\_ref}.
    \item Ningún fichero temporal se considera parte del dataset publicado.
    \item Una lectura versionada nunca mezcla particiones de snapshots distintos.
    \item Toda discrepancia entre esquema esperado y esquema observado produce error explícito o política controlada de rechazo.
    \item La corrupción de una partición se aísla y registra; no se sirve silenciosamente.
\end{enumerate}

\section{Esquema y contratos}

Cada dataset debe estar asociado a una referencia de esquema \texttt{schema\_ref}. El contrato mínimo debe definir:

\begin{itemize}[leftmargin=1.5em]
    \item nombres de columnas;
    \item tipos;
    \item nulabilidad;
    \item columnas de partición;
    \item claves lógicas o restricciones esperadas;
    \item versión del contrato.
\end{itemize}

La validación debe separar dos niveles:

\subsection{Validación estructural}
Comprueba que las columnas requeridas existen y que los tipos son compatibles con el contrato esperado.

\subsection{Validación semántica ligera}
Comprueba propiedades de bajo coste que evitan errores gruesos antes de persistir, por ejemplo:

\begin{itemize}[leftmargin=1.5em]
    \item ausencia de duplicados en clave lógica esperada;
    \item no negatividad en columnas de volumen si así procede;
    \item monotonicidad temporal por partición si el dataset lo exige.
\end{itemize}

El módulo no debe fingir validaciones más fuertes de las que realmente ejecuta.

\section{Estrategia de particionado}

El particionado debe obedecer patrones de acceso y no meras intuiciones estéticas. En datasets de mercado, una política razonable de partida es:

\begin{center}
\texttt{partition\_cols = ['date', 'symbol']}
\end{center}

pero sólo cuando:

\begin{itemize}[leftmargin=1.5em]
    \item las consultas filtran con frecuencia por fecha y/o símbolo;
    \item la cardinalidad no produce explosión patológica de ficheros pequeños;
    \item el tamaño medio de partición permanece operativo.
\end{itemize}

Por ello, la decisión de particionado debe evaluarse con tres tensiones:

\[
\text{granularidad} \;\leftrightarrow\; \text{pruning} \;\leftrightarrow\; \text{small-file overhead}.
\]

El documento de implementación debe dejar claro que particionar demasiado puede degradar el rendimiento aunque mejore la poda teórica.

\section{Modos de escritura}

Se proponen tres modos mínimos, con semántica estricta:

\subsection{\texttt{fail\_if\_exists}}
Falla si el destino lógico ya existe. Es el modo correcto cuando el dataset se considera inmutable para una versión dada.

\subsection{\texttt{overwrite\_partition}}
Reemplaza sólo las particiones afectadas por la nueva publicación. Debe dejar constancia explícita de qué particiones se sustituyeron y bajo qué snapshot quedan vigentes.

\subsection{\texttt{append\_versioned}}
Publica un nuevo snapshot versionado sin destruir snapshots previos. Es el modo más limpio para auditoría, reproducibilidad y backtesting trazable.

\bigskip
En los tres casos, la publicación debe apoyarse en un manifiesto visible sólo al final del proceso. La atomicidad real depende del backend subyacente; por tanto, el contrato debe distinguir claramente entre:

\begin{itemize}[leftmargin=1.5em]
    \item \textbf{atomicidad de fichero local vía rename,} que suele ser fuerte;
    \item \textbf{publicación lógica por manifiesto,} que es la técnica portable;
    \item \textbf{backend object store,} donde la semántica de renombrado puede no ser realmente atómica.
\end{itemize}

\section{Protocolo de escritura}

Una implementación seria debe seguir, de forma aproximada, el siguiente protocolo:

\begin{enumerate}[leftmargin=1.7em]
    \item validar esquema de \texttt{df} frente a \texttt{schema\_ref};
    \item validar reglas semánticas ligeras;
    \item materializar escritura en staging temporal;
    \item recopilar metadatos físicos y hashes de salida;
    \item construir manifiesto del snapshot;
    \item publicar el snapshot haciéndolo visible sólo al final;
    \item registrar métricas y trazas.
\end{enumerate}

El pseudocódigo conceptual sería:

\begin{lstlisting}
def write_parquet(...):
    schema = resolve_schema(schema_ref)
    validate_schema(df, schema)
    validate_business_constraints(df, schema)

    staging_path = allocate_staging_area(path)
    materialize_parquet(df, staging_path, partition_cols)

    manifest = build_manifest(
        staging_path=staging_path,
        schema=schema,
        dataset_version=dataset_version,
        metadata=metadata,
    )

    publish_snapshot(staging_path, path, manifest, write_mode)
    return manifest
\end{lstlisting}

\section{Lectura columnar}

La lectura debe poder restringir columnas y filtros sin cargar material innecesario. Un uso típico sería:

\begin{lstlisting}
df = read_parquet(
    path="s3://trading/ohlcv/",
    columns=["close", "volume", "rsi_14"],
    filters=[("date", ">=", "2026-01-01"), ("symbol", "in", ["AAPL", "NVDA"])],
    schema_ref="features_v2"
)
\end{lstlisting}

La eficiencia real de esta operación dependerá de:

\begin{itemize}[leftmargin=1.5em]
    \item alineación entre filtros y columnas de partición;
    \item metadatos de row groups;
    \item granularidad de ficheros;
    \item backend físico.
\end{itemize}

Por tanto, la documentación debe hablar de \emph{habilitar} poda y proyección, no de prometer reducciones de I/O fijas en todo escenario.

\section{Manifiesto}

Cada snapshot debe llevar un manifiesto con información suficiente para trazabilidad y validación. Campos mínimos recomendados:

\begin{longtable}{>{\ttfamily}p{4cm}p{9.5cm}}
\toprule
Campo & Significado \\
\midrule
dataset\_version & versión lógica del snapshot \\
schema\_ref & referencia del contrato de esquema \\
schema\_hash & hash del esquema resuelto \\
created\_at & instante de publicación \\
row\_count & número total de filas \\
file\_count & número de ficheros publicados \\
partition\_cols & columnas de partición \\
partitions & particiones incluidas o afectadas \\
checksums & hashes por fichero o bloque lógico \\
writer\_version & versión del escritor \\
metadata & metadatos opcionales de ejecución \\
\bottomrule
\end{longtable}

El manifiesto es la pieza que permite distinguir entre ``ficheros presentes'' y ``snapshot válido''.

\section{Errores y tratamiento de fallos}

El módulo debe fallar de forma explícita y tipada. Como mínimo:

\begin{itemize}[leftmargin=1.5em]
    \item \texttt{SchemaValidationError}: esquema incompatible;
    \item \texttt{ManifestError}: manifiesto ausente o inconsistente;
    \item \texttt{DatasetNotFoundError}: dataset o versión inexistente;
    \item \texttt{CorruptPartitionError}: partición ilegible o checksum inválido;
    \item \texttt{InvalidPathError}: path rechazado por validación;
    \item \texttt{WriteConflictError}: conflicto con el modo de escritura elegido.
\end{itemize}

Una partición corrupta debe seguir la política:

\begin{enumerate}[leftmargin=1.7em]
    \item detectar;
    \item aislar;
    \item registrar;
    \item excluir del conjunto válido;
    \item decidir si la lectura total debe abortar o continuar según modo estricto o tolerante.
\end{enumerate}

\section{Hardening de rutas}

El escritor y el lector deben validar rutas de entrada y salida para evitar:

\begin{itemize}[leftmargin=1.5em]
    \item traversal no permitido;
    \item mezcla accidental de entornos;
    \item escritura fuera de raíces autorizadas;
    \item colisiones de nombres entre datasets lógicos.
\end{itemize}

La validación de rutas no es un detalle cosmético: es parte del contrato de integridad operacional.

\section{Observabilidad}

El módulo debe exponer métricas que permitan entender coste y salud operativa, por ejemplo:

\begin{center}
\begin{tabular}{>{\ttfamily}p{5cm}p{8.5cm}}
\toprule
Métrica & Interpretación \\
\midrule
rows\_written & filas publicadas por snapshot \\
bytes\_written & volumen físico escrito \\
write\_latency\_sec & latencia total de publicación \\
rows\_read & filas materializadas por lectura \\
bytes\_read & volumen físico realmente leído \\
projected\_columns & número de columnas solicitadas \\
pruned\_partitions & particiones descartadas por filtros \\
schema\_validation\_failures & fallos de contrato \\
corrupt\_partitions & incidencias de integridad detectadas \\
\bottomrule
\end{tabular}
\end{center}

Estas métricas sí son pertinentes para un módulo Parquet. No lo es reportar, como mérito del store, mejoras directas de performance financiera.

\section{Criterios de aceptación}

El módulo no debe considerarse listo sin pasar, al menos, estas pruebas:

\begin{enumerate}[leftmargin=1.7em]
    \item escritura válida produce snapshot visible con manifiesto íntegro;
    \item esquema incompatible produce error antes de publicación;
    \item modo \texttt{fail\_if\_exists} rechaza colisión de destino;
    \item modo \texttt{overwrite\_partition} sustituye sólo particiones objetivo;
    \item lectura versionada no mezcla snapshots;
    \item lectura con proyección devuelve exactamente las columnas pedidas;
    \item lectura con filtros poda correctamente particiones cuando aplica;
    \item partición corrupta se detecta y aísla;
    \item validación de ruta rechaza destinos no autorizados;
    \item métricas operativas quedan registradas tras lectura y escritura.
\end{enumerate}

\section{Definition of Done}

La \emph{Definition of Done} realista para este módulo es:

\begin{itemize}[leftmargin=1.5em]
    \item snapshots publicados con manifiesto y esquema validado;
    \item semántica de escritura definida y testeada por modo;
    \item lectura con proyección y filtros operativa;
    \item trazabilidad de versión, filas, ficheros y checksums;
    \item detección y aislamiento de corrupción;
    \item documentación explícita de garantías y límites del backend.
\end{itemize}

No forman parte de la DoD afirmaciones absolutas como ``zero data loss'' o ``100\% idempotente'' sin especificación formal exhaustiva del entorno, concurrencia y backend.

\section{Notas de arquitectura}

Parquet debe verse aquí como una capa de persistencia analítica bien adaptada a workloads cuantitativos tabulares. Su valor está en:

\begin{itemize}[leftmargin=1.5em]
    \item eficiencia en lecturas parciales;
    \item compresión razonable;
    \item integración amplia con tooling analítico;
    \item facilidad para construir pipelines reproducibles.
\end{itemize}

Si en el futuro el sistema necesita garantías ACID más fuertes, evolución de esquema compleja o control transaccional sobre catálogos, entonces podrá evaluarse la adopción de formatos de tabla superiores. Pero ese es otro nivel del stack y no debe confundirse con el contrato actual de \texttt{parquet\_store.py}.

\section{Resumen}

\texttt{simons\_core/io/parquet\_store.py} debe implementar un store Parquet determinista, versionable y auditable, con publicación por snapshots, validación de esquema, particionado razonado, lectura selectiva y tratamiento sobrio de fallos. La calidad del módulo no proviene de promesas grandilocuentes, sino de la precisión con la que formaliza persistencia tabular para investigación y producción cuantitativa.
