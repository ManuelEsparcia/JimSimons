Especificación Cuantitativa: simons_core/math/optimization.py

0. Propósito y papel dentro del stack

simons_core/math/optimization.py define la capa institucional de optimización convexa del stack cuantitativo. Su función no es contener lógica de alpha, timing o selección de señales, sino proporcionar primitivas matemáticas robustas, auditables y reutilizables para problemas de asignación y construcción de cartera. En particular, el módulo debe resolver programas cuadráticos convexos con restricciones lineales, cotas por activo y penalizaciones de turnover, y además devolver diagnósticos suficientes para certificar, no sólo asumir, que la solución es aceptable desde el punto de vista numérico.

A diferencia de otros módulos puramente infraestructurales, éste sí es un núcleo matemático real del sistema. Su calidad no se mide sólo por “que el solver devuelva algo”, sino por la claridad de la formulación, la robustez numérica, la trazabilidad de la solución y la calidad de los diagnósticos KKT. En un stack profesional de portfolio construction, eso marca la diferencia entre “solver said optimal” y “solución certificada, condicionada y utilizable en producción”.

La regla institucional debe ser esta:

    toda solución devuelta por el módulo
    debe corresponder a una formulación convexa explícita,
    respetar restricciones dentro de tolerancia,
    y acompañarse de diagnósticos de optimalidad y factibilidad
    suficientemente fuertes para su uso downstream.

1. Objetivo formal del módulo

El módulo implementa primitivas para resolver problemas de la forma canónica

    min_w   1/2 w^T Q w + c^T w

sujeto a:

    C w <= d
    E w  = f
    l <= w <= u

y extensiones estándar como penalización L1 de turnover mediante formulación epigráfica.

Aquí:
- w ∈ R^n es el vector de pesos;
- Q ∈ R^(n×n) es la matriz cuadrática;
- c ∈ R^n es el término lineal;
- C, d representan desigualdades lineales;
- E, f representan igualdades lineales;
- l, u son cotas inferiores y superiores.

El módulo debe actuar como capa general para:
- mean-variance optimization;
- portfolio construction con caps y neutralidades;
- control de exposición;
- mezcla convexa de señales;
- penalizaciones de rotación;
- proyección y postprocesado determinista de pesos.

2. Principios no negociables

P1. Semántica matemática explícita.
Toda restricción debe tener representación inequívoca: igualdad, desigualdad o cota.

P2. Convexidad verificable.
La ruta principal del módulo asume Q semidefinida positiva o regularizada explícitamente hasta entrar en régimen convexo usable.

P3. Diagnóstico antes que opacidad.
No basta con devolver x_opt; hay que devolver factibilidad, cierre KKT, conditioning y metadatos del solver.

P4. Determinismo operacional.
Mismo problema, misma configuración y mismo solver determinista deben producir resultado reproducible salvo tolerancias numéricas del backend.

P5. Separación entre formulación y heurística.
Turnover L1, restricciones y bounds deben modelarse correctamente, no con hacks ad hoc.

P6. Fail-closed numérico.
Problemas mal condicionados, no convexos o mal formados deben fallar explícitamente o regularizarse de forma trazable.

P7. Postprocesado disciplinado.
Funciones como simplex projection o box clipping son utilidades auxiliares y no sustituyen a una solución QP cuando ésta es requerida.

3. Formulación canónica base

La formulación institucional del módulo debe ser:

    min_w   1/2 w^T Q w + c^T w

    s.a.
           C w <= d
           E w  = f
           l <= w <= u

Interpretación operativa:
- presupuesto o fully-invested se expresa en E w = f;
- caps por nombre o exposición máxima se expresan en C w <= d o bounds;
- long-only corresponde a l = 0;
- neutrality constraints se expresan como igualdades o desigualdades lineales.

Esta convención debe mantenerse constante en toda la API para evitar ambigüedades entre Aw<=b, Gw=h, etc.

4. Penalización L1 de turnover

Si w_prev es la cartera anterior y se quiere penalizar

    λ_to ||w - w_prev||_1

la formulación correcta no consiste en alterar c linealmente, sino en introducir variables auxiliares t >= 0 y resolver:

    min_{w,t}   1/2 w^T Q w + c^T w + λ_to Σ_i t_i

    s.a.
              C w <= d
              E w  = f
              l <= w <= u
             -t <= w - w_prev <= t
              t >= 0

Esto preserva convexidad y modela correctamente el turnover absoluto.

Regla crítica:
el módulo no debe aceptar implementaciones “rápidas” que sustituyan esta construcción por aproximaciones no equivalentes sin bandera explícita.

5. Contrato de entrada de solve_qp

La función principal debe aceptar, como mínimo:

- Q
- c
- C, d opcionales
- E, f opcionales
- lb, ub opcionales
- w_prev opcional
- turnover_penalty opcional
- solver_cfg opcional
- warm_start opcional en fases avanzadas

Condiciones mínimas:
- Q debe ser cuadrada;
- Q y c deben ser dimensionalmente compatibles;
- C y d deben tener shapes consistentes;
- E y f deben tener shapes consistentes;
- lb y ub deben tener shape (n,);
- si turnover_penalty > 0, entonces w_prev debe existir y tener shape (n,).

6. Prevalidación matemática y numérica

Antes de instanciar el problema convexamente, el módulo debe ejecutar:

6.1. Simetrización de Q

    Q <- 1/2 (Q + Q^T)

Esto evita incoherencias leves introducidas por acumulación numérica.

6.2. PSD check

Debe calcularse λ_min(Q). Si

    λ_min(Q) < -tol_psd

el problema no es PSD dentro de tolerancia y no puede seguir como QP convexa sin intervención.

6.3. Adaptive regularization

En lugar de un umbral hard-coded opaco, la política recomendada es:

    δ = clamp(reg_eps - λ_min(Q), 0, reg_max)
    Q_reg = Q + δ I

con valores típicos de:
- reg_eps ≈ 1e-8
- reg_max ≈ 1e-4

Esto hace explícita la regularización mínima necesaria para restaurar convexidad numérica usable.

6.4. Condition number bound

Debe calcularse una aproximación de

    κ(Q_reg) = λ_max / max(λ_min, ε)

y compararse contra un umbral operativo, por ejemplo:

    κ(Q_reg) <= 1e6

Si κ explota, el módulo debe:
- fallar explícitamente;
- o marcar la solución como no certificable;
- o requerir escalado/regularización adicional.

Regla crítica:
no debe aceptarse silenciosamente una Q mal condicionada sólo porque el solver convergió “a algo”.

7. Solución primal y dual

El módulo debe resolver el problema primal y, cuando el solver lo permita, recuperar también variables duales asociadas a:
- desigualdades;
- igualdades;
- cotas inferiores;
- cotas superiores;
- restricciones auxiliares de turnover.

La recuperación dual no es un lujo. Es la base para diagnósticos KKT completos.

8. Diagnósticos KKT completos

Éste es el salto principal que distingue una spec buena de una spec PhD-level seria.

Para el problema base, el Lagrangiano es:

    L(w, λ, ν, α, β)
      = 1/2 w^T Q w + c^T w
        + λ^T (Cw - d)
        + ν^T (Ew - f)
        + α^T (l - w)
        + β^T (w - u)

con:
- λ >= 0
- α >= 0
- β >= 0

Las condiciones KKT relevantes son:

8.1. Primal feasibility

    C w* <= d
    E w*  = f
    l <= w* <= u

8.2. Dual feasibility

    λ* >= 0
    α* >= 0
    β* >= 0

8.3. Stationarity

    Qw* + c + C^T λ* + E^T ν* - α* + β* = 0

8.4. Complementary slackness

    λ_i* (Cw* - d)_i = 0
    α_i* (l - w*)_i = 0
    β_i* (w* - u)_i = 0

El módulo debe reportar residuales cuantitativos, no sólo un bool global. Como mínimo:

- r_stationarity
- r_eq_feas
- r_ineq_feas
- r_lb_feas
- r_ub_feas
- r_dual_feas
- r_complementarity

y una bandera agregada:

    is_certified = all(residual <= kkt_tol)

con kkt_tol configurable, por ejemplo 1e-8.

9. API principal recomendada

La función solve_qp debería aspirar a una interfaz como:


10. Implementación de referencia PhD-level

Una implementación base razonable debe seguir este flujo:

10.1. Input sanity
- cast a float
- validación de shapes
- simetrización de Q

10.2. Conditioning
- eigvalsh(Q)
- regularización adaptativa
- chequeo de condition number

10.3. Construcción CVXPY
- variable w
- variable t si turnover_penalty > 0
- objetivo cuadrático + lineal + turnover
- constraints por bloques separados

10.4. Solver waterfall controlado
Política razonable:

    OSQP -> CLARABEL -> SCS

pero no como heurística opaca. Debe persistirse:
- orden intentado
- solver utilizado
- status por solver
- tolerancias usadas

10.5. Dual recovery
Recuperar dual values cuando el solver y la canonicalización lo permitan.

10.6. KKT diagnostics
Calcular residuales completos y certificar o no la solución.

11. Pseudocódigo recomendado

~~~python
def solve_qp(...):
    Q = np.asarray(Q, dtype=float)
    c = np.asarray(c, dtype=float).reshape(-1)
    n = Q.shape[0]

    if Q.shape != (n, n):
        raise ValueError("Q must be square")
    if c.shape != (n,):
        raise ValueError("c must have shape (n,)")

    Q = 0.5 * (Q + Q.T)

    eigvals = np.linalg.eigvalsh(Q)
    lam_min = eigvals[0]
    lam_max = eigvals[-1]

    delta = min(max(reg_eps - lam_min, 0.0), reg_max)
    Q_reg = Q + delta * np.eye(n)

    eigvals_reg = np.linalg.eigvalsh(Q_reg)
    kappa = eigvals_reg[-1] / max(eigvals_reg[0], 1e-12)
    if kappa > cond_max:
        raise ValueError(f"Q ill-conditioned: kappa={kappa:.3e} > {cond_max:.3e}")

    w = cp.Variable(n)
    objective = 0.5 * cp.quad_form(w, Q_reg) + c @ w
    constraints = []

    if C is not None and d is not None:
        constraints.append(C @ w <= d)

    if E is not None and f is not None:
        constraints.append(E @ w == f)

    if lb is not None:
        constraints.append(w >= lb)

    if ub is not None:
        constraints.append(w <= ub)

    if turnover_penalty > 0.0:
        if w_prev is None:
            raise ValueError("w_prev required when turnover_penalty > 0")
        t = cp.Variable(n, nonneg=True)
        constraints += [
            w - w_prev <= t,
            -(w - w_prev) <= t,
        ]
        objective += turnover_penalty * cp.sum(t)

    prob = cp.Problem(cp.Minimize(objective), constraints)

    solver_order = [cp.OSQP, cp.CLARABEL, cp.SCS]
    used_solver = None
    last_error = None

    for solver in solver_order:
        try:
            prob.solve(
                solver=solver,
                eps_abs=kkt_tol,
                eps_rel=kkt_tol,
                warm_start=warm_start is not None,
            )
            used_solver = str(solver)
            if prob.status in ("optimal", "optimal_inaccurate"):
                break
        except Exception as exc:
            last_error = exc

    if prob.status not in ("optimal", "optimal_inaccurate"):
        raise RuntimeError(f"Optimization failed: {prob.status}; last_error={last_error}")

    x_opt = np.asarray(w.value).reshape(-1)
    kkt = compute_kkt_residuals(...)

    return {
        "x_opt": x_opt,
        "objective_value": float(prob.value),
        "status": prob.status,
        "solver": used_solver,
        "kappa_Q": float(kappa),
        "reg_delta": float(delta),
        "kkt": kkt,
        "is_certified": all(v <= kkt_tol for v in kkt.values()),
    }
~~~

12. compute_kkt_residuals

El módulo debe tener una rutina dedicada, no lógica dispersa dentro de solve_qp.

~~~python
def compute_kkt_residuals(Q, c, C, d, E, f, lb, ub, w_opt,
                          lambda_ineq=None, lambda_eq=None,
                          alpha_lb=None, beta_ub=None):
    residuals = {}

    residuals["r_eq_feas"] = (
        0.0 if E is None else float(np.max(np.abs(E @ w_opt - f)))
    )
    residuals["r_ineq_feas"] = (
        0.0 if C is None else float(np.max(np.maximum(C @ w_opt - d, 0.0)))
    )
    residuals["r_lb_feas"] = (
        0.0 if lb is None else float(np.max(np.maximum(lb - w_opt, 0.0)))
    )
    residuals["r_ub_feas"] = (
        0.0 if ub is None else float(np.max(np.maximum(w_opt - ub, 0.0)))
    )

    if lambda_ineq is not None:
        residuals["r_dual_feas_ineq"] = float(np.max(np.maximum(-lambda_ineq, 0.0)))
    else:
        residuals["r_dual_feas_ineq"] = float("inf")

    if alpha_lb is not None:
        residuals["r_dual_feas_lb"] = float(np.max(np.maximum(-alpha_lb, 0.0)))
    else:
        residuals["r_dual_feas_lb"] = float("inf")

    if beta_ub is not None:
        residuals["r_dual_feas_ub"] = float(np.max(np.maximum(-beta_ub, 0.0)))
    else:
        residuals["r_dual_feas_ub"] = float("inf")

    if lambda_ineq is not None and lambda_eq is not None:
        stat = Q @ w_opt + c
        if C is not None:
            stat = stat + C.T @ lambda_ineq
        if E is not None:
            stat = stat + E.T @ lambda_eq
        if alpha_lb is not None:
            stat = stat - alpha_lb
        if beta_ub is not None:
            stat = stat + beta_ub
        residuals["r_stationarity"] = float(np.linalg.norm(stat, ord=np.inf))
    else:
        residuals["r_stationarity"] = float("inf")

    return residuals
~~~

13. Solver tolerances y warm-start

Éste debe ser un punto explícito del módulo, no un detalle oculto.

13.1. Tolerancias
Para producción quant seria, una recomendación razonable es:

- eps_abs = 1e-8
- eps_rel = 1e-8

o tolerancias algo más laxas si el problema es grande, pero siempre explicitadas y persistidas.

13.2. Warm-start
Debe permitirse warm_start en fase 2 para:
- runs recurrentes;
- rebalanceos secuenciales;
- problemas con pequeñas perturbaciones en c, d o w_prev.

14. Funciones auxiliares

14.1. project_to_simplex

Debe proyectar z sobre:

    Δ_n = { w : w_i >= 0, Σ_i w_i = 1 }

Una implementación robusta y estándar es:

~~~python
import numpy as np

def project_to_simplex(z: np.ndarray) -> np.ndarray:
    z = np.asarray(z, dtype=float).reshape(-1)
    if z.size == 0:
        return z.copy()

    u = np.sort(z)[::-1]
    cssv = np.cumsum(u) - 1.0
    rho = np.nonzero(u - cssv / (np.arange(z.size) + 1) > 0)[0][-1]
    theta = cssv[rho] / (rho + 1.0)
    return np.maximum(z - theta, 0.0)
~~~

Esta rutina debe entenderse como postprocesado geométrico, no como sustituto de solve_qp.

14.2. enforce_box_constraints

~~~python
def enforce_box_constraints(w: np.ndarray, lb: np.ndarray, ub: np.ndarray) -> np.ndarray:
    w = np.asarray(w, dtype=float)
    lb = np.asarray(lb, dtype=float)
    ub = np.asarray(ub, dtype=float)
    return np.minimum(np.maximum(w, lb), ub)
~~~

Debe existir como helper para no duplicar lógica algebraica.

15. Casos de uso institucionales

El módulo debe servir a:

- mean-variance optimization;
- long-only portfolio construction;
- sector/factor constraints;
- leverage and exposure control;
- blending convexa de señales;
- turnover-aware rebalancing;
- sanitización/proyección de pesos preliminares.

Regla importante:
este módulo no debe contener suposiciones sobre:
- horizonte temporal,
- alfa concreto,
- slippage específico,
- reglas de entrada/salida,
- o semántica del modelo generador de señales.

Eso pertenece a capas superiores.

16. Failure modes críticos

El módulo debe detectar y tratar explícitamente, al menos:

F1. Q no PSD.
No debe seguir como si nada.

F2. Q mal condicionada.
Puede producir soluciones numéricamente frágiles aunque el solver declare optimal.

F3. Restricciones incompatibles.
Debe distinguir infeasible de solver failure.

F4. Igualdades degeneradas o casi singulares.
Puede producir duales inestables y stationarity poco interpretable.

F5. Bounds flotantes o mal definidos.
Pueden inducir unboundedness o comportamiento no deseado.

F6. Colapso numérico del solver.
No debe traducirse en x_opt utilizable sin chequeos.

18. Validación y pruebas

El módulo no debe considerarse listo sin:

18.1. Casos analíticos pequeños
Problemas 2×2 o 3×3 con solución conocida.

18.2. Test de shapes y errores tipados
Fallo claro ante entradas mal formadas.

18.3. Test de factibilidad
Casos factibles e infactibles.

18.4. Test de bounds
Comprobar respeto de lb/ub.

18.5. Test de turnover
Confirmar que λ_to > 0 reduce rotación respecto al caso λ_to = 0, salvo casos degenerados.

18.6. Test KKT
Residuales pequeños en instancias bien condicionadas.

18.7. Self-test opcional
Un entry point como:

~~~bash
python -m simons_core.math.optimization --self-test
~~~

19. MVP y segunda fase

19.1. MVP obligatorio

El MVP debe incluir:
- solve_qp en forma canónica;
- turnover L1 epigraph correcto;
- simetrización de Q;
- PSD check + regularización adaptativa;
- condition number bound;
- solver waterfall explícito;
- full KKT residuals;
- project_to_simplex;
- enforce_box_constraints;
- tests deterministas básicos.

19.2. Segunda fase razonable

Como extensiones útiles:
- warm-start;
- soporte más rico de dual recovery según solver;
- structured Hessian exploitation;
- proyecciones paralelas o más optimizadas;
- regularización adaptativa más sofisticada;
- tolerancias por solver/backend.

20. No-objetivos

No pertenece al alcance de este módulo:
- implementar un interior-point solver desde cero;
- ADMM custom;
- SQP general no convexo;
- lógica específica de trading;
- backtesting;
- execution;
- cost models microestructurales avanzados.

La misión aquí es una capa de optimización convexa sólida y certificable.

21. Criterios de aceptación

El módulo sólo debe considerarse correcto si cumple simultáneamente:

- formula problemas QP convexos con semántica inequívoca;
- modela turnover L1 correctamente mediante epigraph;
- detecta y trata Q no PSD o mal condicionada;
- devuelve solución primal consistente;
- reporta residuales KKT completos;
- distingue claramente éxito, infeasibility y solver failure;
- mantiene utilidades auxiliares deterministas y bien testeadas;
- no mezcla álgebra de optimización con lógica de estrategia.

22. Conclusión

simons_core/math/optimization.py es uno de los pocos módulos del stack donde sí procede exigir nivel claramente doctoral. Su valor no está en envolver CVXPY, sino en fijar una semántica matemática estricta para problemas convexos de construcción de cartera, con modelado correcto de turnover, control de conditioning y diagnósticos KKT suficientemente completos para certificar la solución. En un laboratorio cuantitativo serio, ese estándar no es opcional: si no puedes verificar stationarity, primal feasibility y comportamiento numérico, entonces no tienes una primitive de optimización de producción, sólo tienes un solver call decorado. Ese es exactamente el estándar que este módulo debe imponer.
