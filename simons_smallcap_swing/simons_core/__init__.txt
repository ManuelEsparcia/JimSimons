Especificación Cuantitativa: simons_core/__init__.py

0. Propósito y papel dentro del stack

simons_core/__init__.py define la superficie pública estable del paquete simons_core. Su responsabilidad no es implementar lógica de negocio, ni contener heurísticas cuantitativas, ni introducir side effects operativos, sino resolver cuatro problemas de arquitectura muy concretos:

1. Exponer un conjunto pequeño, coherente y estable de símbolos públicos.
2. Ocultar detalles internos de organización del paquete.
3. Establecer una frontera clara entre API soportada y API privada.
4. Garantizar que el acto de importar simons_core sea determinista, barato y libre de efectos colaterales.

En un repositorio cuantitativo serio, el __init__.py del core cumple la misma función que una interfaz estable en una librería de infraestructura: reduce acoplamiento, simplifica consumo aguas abajo y permite evolucionar la implementación interna sin romper a los módulos clientes. :contentReference[oaicite:0]{index=0}

1. Qué debe hacer y qué no debe hacer

1.1. Debe hacer

- Re-exportar únicamente APIs estables y deliberadamente soportadas.
- Centralizar la definición de __all__.
- Permitir imports ergonómicos del tipo:

    from simons_core import RunContext, GateResult, MarketCalendar

- Preservar compatibilidad semántica entre versiones.
- Servir como punto de lectura rápida para entender el núcleo público del paquete.

1.2. No debe hacer

- No debe ejecutar I/O.
- No debe leer variables de entorno para inicialización operativa.
- No debe configurar logging global.
- No debe materializar caches, abrir conexiones, tocar disco ni red.
- No debe importar submódulos pesados solo por conveniencia si eso degrada el coste de import.
- No debe mezclar símbolos experimentales con símbolos estables.

2. Principio rector

La regla de diseño es:

    Public API << Internal implementation surface

Es decir, la superficie pública debe ser estrictamente menor, más estable y más conservadora que la superficie total del paquete. La calidad del módulo no se mide por cuántas cosas exporta, sino por cuán bien seleccionadas, mantenidas y documentadas están las que exporta.

3. Contrato formal de exportación

Sea

    E = {e_1, e_2, ..., e_n}

el conjunto de símbolos exportados por __all__. El módulo debe satisfacer las siguientes invariantes:

I1. Estabilidad:
todo símbolo e_i ∈ E pertenece a la API pública soportada.

I2. Trazabilidad:
cada e_i debe tener origen explícito en un submódulo identificado.

I3. No ambigüedad:
no deben exportarse alias redundantes salvo decisión consciente de compatibilidad.

I4. Importabilidad:
todo e_i debe resolverse mediante import simons_core.

I5. Documentabilidad:
todo e_i debe estar documentado o ser autoexplicativo dentro del contexto del paquete.

I6. Compatibilidad:
eliminar o modificar semánticamente un e_i requiere política explícita de versionado.

En términos prácticos, __all__ no es una lista decorativa: es el contrato de consumo del paquete.

4. Criterio de selección de símbolos públicos

Un símbolo debe ser re-exportado desde simons_core solo si cumple al menos una de las siguientes condiciones:

- Es un tipo central consumido transversalmente por varios módulos.
- Es un contrato abstracto que define integración entre subsistemas.
- Es una utilidad fundacional cuyo acceso directo mejora ergonomía sin aumentar ambigüedad.
- Es parte del lenguaje común del framework y su ubicación natural para el consumidor es el namespace raíz.

Por el contrario, no deben subirse al namespace raíz:

- Helpers locales.
- Constantes internas de implementación.
- Clases experimentales.
- Funciones altamente especializadas usadas por un único submódulo.
- Objetos cuyo nombre solo tiene sentido dentro de una subcarpeta concreta.

5. Estructura pública esperada

Para este bloque de simons_core, una política razonable es re-exportar solo los elementos que definen contratos, tipos y servicios nucleares. Por ejemplo:

- Desde interfaces: contratos abstractos y tipos transversales.
- Desde schemas: estructuras canónicas de validación.
- Desde calendar: servicio temporal de calendario de mercado.
- Desde logging: primitivas de observabilidad del core.

La idea no es exponer todo lo que existe en esos archivos, sino solo lo que constituye lenguaje compartido entre módulos.

6. Disciplina de imports

6.1. Regla general

El coste de

    import simons_core

debe ser bajo, determinista y libre de side effects. Formalmente, si T_import es el tiempo total de importación:

    T_import = Σ t_k

donde t_k es el coste inducido por cada import transitivo necesario. La política del módulo consiste en minimizar el conjunto de dependencias transitivas cargadas en import-time.

6.2. Consecuencias prácticas

- No importar dependencias pesadas que no sean necesarias para definir el contrato público.
- Mantener fuera del namespace raíz componentes de alto coste o acoplamiento opcional.
- Evitar ciclos de import entre interfaces, schemas, calendar, logging y submódulos descendientes.

6.3. Ciclos de import

Si el grafo dirigido de dependencias de import se denota por

    G = (V, A)

el objetivo es que la porción relevante del core sea acíclica o, al menos, que __init__.py no participe en ciclos. Un __init__ que dispara ciclos suele ser síntoma de frontera pública mal diseñada.

7. Side effects prohibidos

El módulo debe ser observacionalmente puro en import-time: importar el paquete no debe cambiar estado global del proceso salvo la carga de definiciones Python necesaria.

En particular, quedan prohibidos en simons_core/__init__.py:

- Configuración automática de handlers de logging.
- Registro global de plugins.
- Validaciones de entorno costosas.
- Lectura de configuración runtime.
- Escritura de ficheros temporales.
- Inicialización de caches.

Esta disciplina es especialmente importante en research, backtesting, notebooks, tests y entornos CI, donde imports repetidos y composición modular son la norma.

8. Compatibilidad y versionado

La semántica del módulo debe alinearse con una política de versionado explícita:

- Patch version:
  correcciones internas sin cambio de API pública.

- Minor version:
  adición retrocompatible de nuevos símbolos públicos.

- Major version:
  eliminación, renombrado o cambio semántico incompatible de símbolos exportados.

Si un símbolo exportado deja de ser recomendable, la ruta correcta no es borrarlo silenciosamente, sino marcarlo como deprecado, anunciar horizonte de retirada y mantener compatibilidad durante una ventana razonable.

9. Relación con documentación y tooling

__init__.py también cumple una función documental. Cuando un desarrollador inspecciona el paquete raíz, debe poder inferir rápidamente:

1. qué conceptos son centrales;
2. qué objetos son estables;
3. qué piezas forman el lenguaje común del core.

Por ello, la lista de exports debe ser corta, semántica y mantenible. Un namespace raíz sobrecargado incrementa fricción cognitiva y vuelve opaco el contrato del paquete.

10. Diseño recomendado

Una implementación sobria y correcta podría adoptar una forma como la siguiente:

~~~python
"""
Public API for simons_core.

This module re-exports the stable, low-level contracts and services
that define the package's supported surface.
"""

from .interfaces import (
    DataProvider,
    FeatureBuilder,
    ModelTrainer,
    PortfolioEngine,
    Validator,
    RunContext,
    GateResult,
)

from .schemas import (
    PriceSchema,
    FeatureSchema,
    LabelSchema,
    PredictionSchema,
)

from .calendar import MarketCalendar
from .logging import get_logger, LogEvent

__all__ = [
    "DataProvider",
    "FeatureBuilder",
    "ModelTrainer",
    "PortfolioEngine",
    "Validator",
    "RunContext",
    "GateResult",
    "PriceSchema",
    "FeatureSchema",
    "LabelSchema",
    "PredictionSchema",
    "MarketCalendar",
    "get_logger",
    "LogEvent",
]
~~~

11. Criterios de calidad

El módulo se considera correcto si satisface simultáneamente los siguientes criterios:

Q1. Claridad:
un consumidor entiende la API pública leyendo solo este archivo.

Q2. Minimalidad:
no se exportan símbolos superfluos.

Q3. Estabilidad:
la selección de exports refleja compromiso real de compatibilidad.

Q4. Pureza de import:
no hay side effects.

Q5. Bajo acoplamiento:
el paquete raíz no obliga a cargar subsistemas innecesarios.

Q6. Testabilidad:
la resolución de exports puede verificarse automáticamente.

12. Casos límite y fallos frecuentes

Los errores más habituales en este tipo de módulo son:

- Exportar símbolos internos “por comodidad” y luego no poder retirarlos.
- Re-exportar objetos cuyos nombres colisionan semánticamente.
- Introducir dependencias pesadas que ralentizan todo el stack.
- Convertir el namespace raíz en un agregador indiscriminado.
- Romper imports descendientes al mover código sin mantener alias de compatibilidad.

Un __init__.py bien diseñado evita precisamente ese tipo de deuda arquitectónica temprana.

La definición del módulo debe ir acompañada, al menos, de las siguientes verificaciones:

14. Definition of Done

El archivo simons_core/__init__.py queda bien resuelto cuando:

- la API pública está definida explícitamente;
- no hay side effects al importar;
- el namespace raíz es pequeño y semántico;
- la compatibilidad está gobernada por política clara;
- cualquier consumidor puede apoyarse en este punto de entrada sin depender de detalles internos.

15. Conclusión

La calidad de simons_core/__init__.py no reside en sonar ambicioso, sino en ser disciplinado. En un framework cuantitativo serio, este módulo actúa como frontera contractual del core: selecciona qué es estable, qué es público y qué puede cambiar internamente sin propagar fragilidad al resto del sistema.

Por tanto, su estándar correcto no es el de una pieza “vistosa”, sino el de una interfaz mínima, rigurosa y durable.
