\section{Propósito del módulo}

El archivo \texttt{simons\_core/\_\_init\_\_.py} define la \emph{superficie pública estable} del paquete \texttt{simons\_core}. Su responsabilidad no es implementar lógica de negocio, ni contener heurísticas cuantitativas, ni introducir side effects operativos, sino resolver cuatro problemas de arquitectura muy concretos:

\begin{enumerate}[label=\arabic*.]
    \item Exponer un conjunto pequeño, coherente y estable de símbolos públicos.
    \item Ocultar detalles internos de organización del paquete.
    \item Establecer una frontera clara entre API soportada y API privada.
    \item Garantizar que el acto de importar \texttt{simons\_core} sea determinista, barato y libre de efectos colaterales.
\end{enumerate}

En un repositorio cuantitativo serio, el \texttt{\_\_init\_\_.py} del core cumple la misma función que una interfaz estable en una librería de infraestructura: reduce acoplamiento, simplifica consumo aguas abajo y permite evolucionar la implementación interna sin romper a los módulos clientes.

\section{Qué debe hacer y qué no debe hacer}

\subsection{Debe hacer}
\begin{itemize}
    \item Re-exportar únicamente APIs estables y deliberadamente soportadas.
    \item Centralizar la definición de \texttt{\_\_all\_\_}.
    \item Permitir imports ergonómicos del tipo:
    \begin{lstlisting}
from simons_core import RunContext, GateResult, MarketCalendar
    \end{lstlisting}
    \item Preservar compatibilidad semántica entre versiones.
    \item Servir como punto de lectura rápida para entender el núcleo público del paquete.
\end{itemize}

\subsection{No debe hacer}
\begin{itemize}
    \item No debe ejecutar I/O.
    \item No debe leer variables de entorno para inicialización operativa.
    \item No debe configurar logging global.
    \item No debe materializar caches, abrir conexiones, tocar disco ni red.
    \item No debe importar submódulos pesados solo por conveniencia si eso degrada el coste de import.
    \item No debe mezclar símbolos experimentales con símbolos estables.
\end{itemize}

\section{Principio rector}

La regla de diseño es:

\[
\text{Public API} \ll \text{Internal implementation surface}
\]

Es decir, la superficie pública debe ser estrictamente menor, más estable y más conservadora que la superficie total del paquete. La calidad del módulo no se mide por cuántas cosas exporta, sino por cuán bien seleccionadas, mantenidas y documentadas están las que exporta.

\section{Contrato formal de exportación}

Sea
\[
E = \{e_1, e_2, \dots, e_n\}
\]
el conjunto de símbolos exportados por \texttt{\_\_all\_\_}. El módulo debe satisfacer las siguientes invariantes:

\begin{enumerate}[label=\textbf{I\arabic*.}]
    \item \textbf{Estabilidad:} todo símbolo \( e_i \in E \) pertenece a la API pública soportada.
    \item \textbf{Trazabilidad:} cada \( e_i \) debe tener origen explícito en un submódulo identificado.
    \item \textbf{No ambigüedad:} no deben exportarse alias redundantes salvo decisión consciente de compatibilidad.
    \item \textbf{Importabilidad:} todo \( e_i \) debe resolverse mediante \texttt{import simons\_core}.
    \item \textbf{Documentabilidad:} todo \( e_i \) debe estar documentado o ser autoexplicativo dentro del contexto del paquete.
    \item \textbf{Compatibilidad:} eliminar o modificar semánticamente un \( e_i \) requiere política explícita de versionado.
\end{enumerate}

En términos prácticos, \texttt{\_\_all\_\_} no es una lista decorativa: es el contrato de consumo del paquete.

\section{Criterio de selección de símbolos públicos}

Un símbolo debe ser re-exportado desde \texttt{simons\_core} solo si cumple al menos una de las siguientes condiciones:

\begin{itemize}
    \item Es un tipo central consumido transversalmente por varios módulos.
    \item Es un contrato abstracto que define integración entre subsistemas.
    \item Es una utilidad fundacional cuyo acceso directo mejora ergonomía sin aumentar ambigüedad.
    \item Es parte del lenguaje común del framework y su ubicación natural para el consumidor es el namespace raíz.
\end{itemize}

Por el contrario, \emph{no} deben subirse al namespace raíz:

\begin{itemize}
    \item Helpers locales.
    \item Constantes internas de implementación.
    \item Clases experimentales.
    \item Funciones altamente especializadas usadas por un único submódulo.
    \item Objetos cuyo nombre solo tiene sentido dentro de una subcarpeta concreta.
\end{itemize}

\section{Estructura pública esperada}

Para este bloque de \texttt{simons\_core}, una política razonable es re-exportar solo los elementos que definen contratos, tipos y servicios nucleares. Por ejemplo:

\begin{itemize}
    \item Desde \texttt{interfaces}: contratos abstractos y tipos transversales.
    \item Desde \texttt{schemas}: estructuras canónicas de validación.
    \item Desde \texttt{calendar}: servicio temporal de calendario de mercado.
    \item Desde \texttt{logging}: primitivas de observabilidad del core.
\end{itemize}

La idea no es exponer \emph{todo} lo que existe en esos archivos, sino solo lo que constituye lenguaje compartido entre módulos.

\section{Disciplina de imports}

\subsection{Regla general}
El coste de
\begin{lstlisting}
import simons_core
\end{lstlisting}
debe ser bajo, determinista y libre de side effects. Formalmente, si \(T_{\text{import}}\) es el tiempo total de importación:

\[
T_{\text{import}} = \sum_{k=1}^{m} t_k
\]

donde \(t_k\) es el coste inducido por cada import transitivo necesario. La política del módulo consiste en minimizar el conjunto de dependencias transitivas cargadas en import-time.

\subsection{Consecuencias prácticas}
\begin{itemize}
    \item No importar dependencias pesadas que no sean necesarias para definir el contrato público.
    \item Mantener fuera del namespace raíz componentes de alto coste o acoplamiento opcional.
    \item Evitar ciclos de import entre \texttt{interfaces}, \texttt{schemas}, \texttt{calendar}, \texttt{logging} y submódulos descendientes.
\end{itemize}

\subsection{Ciclos de import}
Si el grafo dirigido de dependencias de import se denota por
\[
G = (V, A),
\]
el objetivo es que la porción relevante del core sea acíclica o, al menos, que \texttt{\_\_init\_\_.py} no participe en ciclos. Un \texttt{\_\_init\_\_} que dispara ciclos suele ser síntoma de frontera pública mal diseñada.

\section{Side effects prohibidos}

El módulo debe ser \emph{observacionalmente puro} en import-time: importar el paquete no debe cambiar estado global del proceso salvo la carga de definiciones Python necesaria.

En particular, quedan prohibidos en \texttt{\_\_init\_\_.py}:

\begin{itemize}
    \item Configuración automática de handlers de logging.
    \item Registro global de plugins.
    \item Validaciones de entorno costosas.
    \item Lectura de configuración runtime.
    \item Escritura de ficheros temporales.
    \item Inicialización de caches.
\end{itemize}

Esta disciplina es especialmente importante en research, backtesting, notebooks, tests y entornos CI, donde imports repetidos y composición modular son la norma.

\section{Compatibilidad y versionado}

La semántica del módulo debe alinearse con una política de versionado explícita:

\begin{itemize}
    \item \textbf{Patch version:} correcciones internas sin cambio de API pública.
    \item \textbf{Minor version:} adición retrocompatible de nuevos símbolos públicos.
    \item \textbf{Major version:} eliminación, renombrado o cambio semántico incompatible de símbolos exportados.
\end{itemize}

Si un símbolo exportado deja de ser recomendable, la ruta correcta no es borrarlo silenciosamente, sino marcarlo como deprecado, anunciar horizonte de retirada y mantener compatibilidad durante una ventana razonable.

\section{Relación con documentación y tooling}

\texttt{\_\_init\_\_.py} también cumple una función documental. Cuando un desarrollador inspecciona el paquete raíz, debe poder inferir rápidamente:

\begin{enumerate}[label=\arabic*.]
    \item qué conceptos son centrales;
    \item qué objetos son estables;
    \item qué piezas forman el lenguaje común del core.
\end{enumerate}

Por ello, la lista de exports debe ser corta, semántica y mantenible. Un namespace raíz sobrecargado incrementa fricción cognitiva y vuelve opaco el contrato del paquete.

\section{Diseño recomendado}

Una implementación sobria y correcta podría adoptar una forma como la siguiente:

\section{Criterios de calidad}

El módulo se considera correcto si satisface simultáneamente los siguientes criterios:

\begin{enumerate}[label=\textbf{Q\arabic*.}]
    \item \textbf{Claridad:} un consumidor entiende la API pública leyendo solo este archivo.
    \item \textbf{Minimalidad:} no se exportan símbolos superfluos.
    \item \textbf{Estabilidad:} la selección de exports refleja compromiso real de compatibilidad.
    \item \textbf{Pureza de import:} no hay side effects.
    \item \textbf{Bajo acoplamiento:} el paquete raíz no obliga a cargar subsistemas innecesarios.
    \item \textbf{Testabilidad:} la resolución de exports puede verificarse automáticamente.
\end{enumerate}

\section{Casos límite y fallos frecuentes}

Los errores más habituales en este tipo de módulo son:

\begin{itemize}
    \item Exportar símbolos internos ``por comodidad'' y luego no poder retirarlos.
    \item Re-exportar objetos cuyos nombres colisionan semánticamente.
    \item Introducir dependencias pesadas que ralentizan todo el stack.
    \item Convertir el namespace raíz en un agregador indiscriminado.
    \item Romper imports descendientes al mover código sin mantener alias de compatibilidad.
\end{itemize}

Un \texttt{\_\_init\_\_.py} bien diseñado evita precisamente ese tipo de deuda arquitectónica temprana.

La definición del módulo debe ir acompañada, al menos, de las siguientes verificaciones:

    assert set(simons_core.__all__) == expected
\end{lstlisting}

\section{Definition of Done}

El archivo \texttt{simons\_core/\_\_init\_\_.py} queda bien resuelto cuando:

\begin{itemize}
    \item la API pública está definida explícitamente;
    \item no hay side effects al importar;
    \item el namespace raíz es pequeño y semántico;
    \item la compatibilidad está gobernada por política clara;
    \item cualquier consumidor puede apoyarse en este punto de entrada sin depender de detalles internos.
\end{itemize}

\section{Conclusión}

La calidad de \texttt{\_\_init\_\_.py} no reside en sonar ambicioso, sino en ser disciplinado. En un framework cuantitativo serio, este módulo actúa como frontera contractual del core: selecciona qué es estable, qué es público y qué puede cambiar internamente sin propagar fragilidad al resto del sistema.

Por tanto, su estándar correcto no es el de una pieza ``vistosa'', sino el de una interfaz mínima, rigurosa y durable.
