\section{Modulo: \texttt{data/price/fetch\_prices.py}}

\subsection{Objetivo y contexto de mercado}

El modulo \texttt{data/price/fetch\_prices.py} descarga y normaliza precios diarios OHLCV (open, high, low, close, volumen, y campos adicionales) para el \textbf{universo de small caps de EE. UU.} en un rango temporal definido, garantizando:

\begin{itemize}
  \item \textbf{Trazabilidad}: cada dato se asocia explícitamente a un proveedor, run\_id y fecha de ingesta.
  \item \textbf{Reproducibilidad}: el mismo \texttt{run\_id} + \texttt{config} debe generar exactamente el mismo dataset raw.
  \item \textbf{Consistencia temporal PIT} (point–in–time) para que el backtest posterior no use información futura. [web:56][web:62]
\end{itemize}

Este módulo es la \textbf{capa de ingestión primaria} sobre la que se construyen ajustes corporativos, features, labels y backtest, y por tanto debe ser robusto, observable y resiliente a fallos parciales.

\subsection{Marco de datos y convenciones de esquema}

El universo de small caps de EE. UU. se define aguas arriba en \texttt{data/universe/*}, y el módulo asume una lista de tickers efectiva por fecha. [web:56]

El esquema de salida canonico es:

\begin{align}
  &\texttt{symbol}        && \in \text{string} \\
  &\texttt{date}          && \in \text{date, timezone-naive (sesión de mercado)} \\
  &\texttt{open, high, low, close} && \in \mathbb{R}_{>0} \\
  &\texttt{volume}        && \in \mathbb{R}_{\geq 0} \text{ o } \mathbb{N} \\
  &\texttt{vwap}          && \in \mathbb{R}_{>0} \text{ (nullable)} \\
  &\texttt{trades\_count} && \in \mathbb{N} \text{ (nullable)} \\
  &\texttt{source\_provider} && \in \text{string} \\
  &\texttt{ingest\_ts\_utc} && \in \text{timestamp} \\
  &\texttt{run\_id}       && \in \text{string}
\end{align}

La clave primaria lógica es:

\[
\text{PK} = (\texttt{symbol}, \texttt{date}, \texttt{source\_provider}),
\]

y se exige ausencia de duplicados por esta clave. [web:56]

\subsection{Politicas de fetch e incrementalidad}

El módulo soporta:

\begin{itemize}
  \item \texttt{force\_full\_refresh = False}: fetch incremental detectando la última fecha disponible por ticker y descargando solo desde:
    \[
    t_{\min} = \max(\text{max\_local\_date} + 1, \texttt{start\_date}).
    \]
  \item \texttt{force\_full\_refresh = True}: reingestión completa del rango [\texttt{start\_date}, \texttt{end\_date}].
\end{itemize}

Esto se alinea con pipelines de ingestión bulk descritos en la literatura cuantitativa, donde:

\begin{itemize}
  \item Se backfill completan los históricos una vez.
  \item Se ejecutan runs diarios incrementales optimizando costos y latencia. [web:56][web:62]
\end{itemize}

\subsection{Estrategia de retry y rate limit}

Para cada ticker, la función \texttt{fetch\_symbol\_prices} encapsula:

\begin{itemize}
  \item Retries configurables para errores transitorios:
    \begin{itemize}
      \item time–out.
      \item 429 (rate limiting).
      \item 5xx (servidor proveedor).
    \end{itemize}
  \item Backoff exponencial con jitter:
    \[
    \text{delay}_k = \text{backoff\_base\_sec} \cdot \alpha^k \cdot \text{random\_jitter},
    \]
    con \(\alpha > 1\).
  \item Rate limit global por provider, independiente de la lógica de negocio. [web:53][web:54]
\end{itemize}

Esto es estándar en pipelines de datos financieros que usan APIs de proveedores como Polygon, Tiingo, Stooq, etc. [web:51][web:54][web:60]

\subsection{Validaciones de integridad en ingestión}

El módulo implementa validaciones mínimas para garantizar que el dataset crudo es coherente antes de pasar a \texttt{adjust\_prices.py}.

\subsubsection{Por fila}

\begin{itemize}
  \item \textbf{Ordenamiento de OHLC} (cuando todos no nulos):
    \begin{align}
      \texttt{high} &\geq \max(\texttt{open}, \texttt{close}, \texttt{low}), \\
      \texttt{low}  &\leq \min(\texttt{open}, \texttt{close}, \texttt{high}).
    \end{align}
  \item \texttt{volume} \(\geq 0\).
  \item \texttt{date} parseable y dentro de \([\texttt{start\_date}, \texttt{end\_date}]\).
\end{itemize}

\subsubsection{Por ticker}

\begin{itemize}
  \item Fechas \texttt{date} crecientes (sin reversión).
  \item No duplicados en \texttt{date} (por \texttt{symbol}).
  \item Porcentaje de días faltantes frente al calendario de sesiones de mercado, registrado en \texttt{coverage\_report}.
\end{itemize}

Estas reglas se alinean con buenas prácticas de chequeo de calidad de datos en backtest, donde se exige que el OHLC mantenga la relación lógica derivada de la definición de vela. [web:41][web:62]

\subsection{Normalización de esquema y proveedor}

El flujo por ticker es:

\begin{equation*}
\texttt{fetch\_symbol\_prices} \xrightarrow{\text{parseo}} \texttt{normalize\_provider\_frame}(\cdot, \texttt{provider}) \xrightarrow{\text{validación}} \texttt{validate\_ingestion\_frame}.
\end{equation*}

La función \texttt{normalize\_provider\_frame}:

\begin{itemize}
  \item Mapea campos de proveedor (\texttt{open, close, \dots}) al esquema canonico.
  \item Normaliza símbolos a \texttt{uppercase} y trim.
  \item Ensaya tipificación fuerte: \texttt{open, high, \dots} como \texttt{float64}, \texttt{volume} como \texttt{float64/int64}, etc.
\end{itemize}

Esta capa de normalización asegura que múltiples proveedores (Polygon, Tiingo, Stooq, etc.) alimenten el mismo esquema de datos usado por el resto del pipeline de investigación cuantitativa. [web:54][web:58]

\subsection{Reportes de cobertura y fallos}

Además del dataset principal en Parquet, se generan:

\begin{itemize}
  \item \texttt{coverage\_report}: indica por ticker:
    \[
    \texttt{pct\_missing\_sessions} = \frac{ \#\text{sesiones esperadas} - \#\text{sesiones recibidas} }{ \#\text{sesiones esperadas} }.
    \]
  \item \texttt{failures\_report}: por ticker / código de error / mensaje, para que \texttt{qc\_prices.py} pueda decidir politicas de exclusion de small caps ilíquidos o con datos escasos.
  \item \texttt{run\_manifest.json}: con parámetros de entrada, versión de código, hash de la config, y contadores de filas/tickers procesados.
\end{itemize}

Esto encaja con el paradigma de \textit{Data Quality} en sistemas de backtest, donde se documenta explícitamente la cobertura de datos por activo y fecha. [web:41][web:62]

\subsection{Interfaz de ejecución y configuración}

CLI sugerida:

\begin{verbatim}
python -m data.price.fetch_prices                \
  --provider polygon                               \
  --start-date 2014-01-01                          \
  --end-date 2026-03-01                            \
  --universe-path data/universe/current.parquet    \
  --frequency 1d                                   \
  --run-id fetch_prices_20260305_001
\end{verbatim}

Los parámetros clave son:

\begin{itemize}
  \item \texttt{provider}: “polygon”, “tiingo”, “stooq”, etc.
  \item \texttt{start\_date, end\_date}: ventana de tiempo.
  \item \texttt{run\_id}: identificador único de ejecución para trazabilidad y replay.
  \item \texttt{frequency}: en esta fase solo \texttt{"1d"}.
  \item \texttt{chunk\_size\_tickers}: lote de símbolos por ejecución (para limitar memory/RAM).
\end{itemize}

\subsection{Dependencias aguas arriba y aguas abajo}

El módulo se sitúa en el eje central:

\begin{itemize}
  \item \textbf{Upstream}:
    \begin{itemize}
      \item Lista de tickers PIT del universo de small caps.
      \item Calendario de sesiones de mercado (NYSE/NASDAQ).
      \item Configuración de proveedor de datos.
    \end{itemize}
  \item \textbf{Downstream}:
    \begin{itemize}
      \item \texttt{data/price/adjust\_prices.py}.
      \item \texttt{data/price/qc\_prices.py}.
      \item \texttt{features/build\_features.py}.
      \item \texttt{labels/build\_labels.py}.
    \end{itemize}
\end{itemize}

\texttt{fetch\_prices.py} asegura que el resto de módulos trabajen sobre un dataset raw homogeneizado, sin decisiones de normalización diseminadas en el pipeline. [web:56][web:62]

\subsection{Reglas de negocio de diseño de sistema}

Entre las reglas de negocio están:

\begin{itemize}
  \item \textbf{No usar información futura}: el proveedor devuelve únicamente barras historicas \textit{publicadas} para cada \texttt{date}.
  \item \textbf{No imputar datos faltantes}: se mantienen como \texttt{null} y se delega cualquier imputación al módulo de QC o features, si procede.
  \item \textbf{Reintentos solo para errores transitorios}: 429, time–out, 5xx; se excluyen errores permanentes (404, símbolo inválido).
  \item \textbf{Hash/versionado de config y datasets}: para garantizar que el mismo \texttt{run\_id} + \texttt{config} produzcan exáctamente el mismo dataset raw.
\end{itemize}

Estas políticas son estándar en pipelines de datos cuantitativos donde la \textbf{auditoría de backtest} es crítica y el error de data es mucho más difícil de detectar que el error de código. [web:56][web:62]

\subsection{Estructura de funciones recomendada}

El módulo está organizado en capas lógicas:

\begin{itemize}
  \item \texttt{load\_config(...)}, \texttt{load\_universe\_tickers(...)}.
  \item \texttt{get\_incremental\_window(...)}: decide el rango a descargar por ticker.
  \item \texttt{fetch\_symbol\_prices(..., provider\_client)}: capa de negocio sobre el cliente de proveedor.
  \item \texttt{normalize\_provider\_frame(...)}, \texttt{validate\_ingestion\_frame(...)}.
  \item \texttt{persist\_prices(...)}: escritura Parquet particionada por \texttt{provider}, \texttt{freq}, \texttt{date}.
  \item \texttt{build\_coverage\_report(...)}, \texttt{build\_failures\_report(...)}.
  \item \texttt{main()} que orquesta todo el flujo.
\end{itemize}

Esta arquitectura permite:

\begin{itemize}
  \item Desacoplar el cliente del proveedor hacia una interfaz (\texttt{provider\_client}).
  \item Reutilizar la misma lógica de fetch tanto para backfill histórico como para ejecuciones diarias incrementales.
\end{itemize}

\subsection{Criterios de aceptación (Definition of Done)}

El módulo cumple la definición de terminado si:

\begin{itemize}
  \item Ejecuta \textbf{end‑to‑end} sin caer por el fallo de uno o pocos tickers.
  \item Genera un dataset Parquet con esquema canonico y sin duplicados por PK lógica.
  \item Produce \texttt{coverage\_report} y \texttt{failures\_report} consumibles por QC.
  \item Soporta modo \texttt{incremental} y \texttt{full\_refresh}.
  \item Provee logs y métricas suficientes para auditar una corrida:
    \begin{itemize}
      \item \texttt{total\_tickers, tickers\_ok, tickers\_failed}.
      \item \texttt{rows\_ingested}.
      \item \texttt{pct\_missing\_sessions\_avg}.
      \item \texttt{provider\_http\_error\_rate}.
      \item \texttt{run\_duration\_sec}.
    \end{itemize}
  \item Es \textbf{reproducible} por \texttt{run\_id} + hash de la configuración.
\end{itemize}

\subsection{Casos de prueba mínimos recomendados}

Los tests unitarios e integrados deben cubrir:

\begin{itemize}
  \item \textbf{Caso feliz}: 3 tickers, rango corto, salida con todas las columnas correctas, sin duplicados, OHLC coherente.
  \item \textbf{Retry}: proveedor responde 429/5xx en la primera petición, y luego exito.
  \item \textbf{Error permanente}: ticker inválido queda en \texttt{failures\_report} pero no aborta el pipeline.
  \item \textbf{Incremental}: segunda corrida solo trae fechas nuevas.
  \item \textbf{Duplicados en respuesta del proveedor}: se deduplican correctamente por PK lógica.
  \item \textbf{Schema drift}: columna inesperada/ausente según la política de \texttt{normalize\_provider\_frame} y \texttt{validate\_ingestion\_frame}.
\end{itemize}

\subsection{Riesgos, decisiones abiertas y checklist}

Entre los riesgos:

\begin{itemize}
  \item Coste y licencia de proveedor primario y posible fallback.
  \item Política de días faltantes aceptables para small caps ilíquidos.
  \item Regla de versionado: partición por \texttt{run} vs. \texttt{overwrite} controlado.
  \item Convención final de \texttt{timezone/date} cuando el proveedor usa \texttt{UTC timestamp} en lugar de calendario nativo.
\end{itemize}

\textbf{Checklist rápido de implementación:}

\begin{itemize}
  \item [ ] Cliente de proveedor desacoplado por interfaz.
  \item [ ] Rate limit centralizado.
  \item [ ] Retries con backoff exponencial y jitter.
  \item [ ] Normalización de esquema por proveedor.
  \item [ ] Validaciones básicas de ingestión.
  \item [ ] Persistencia Parquet + particiones + manifest.
  \item [ ] Reportes de cobertura y fallos.
  \item [ ] CLI con argumentos y valores por defecto en config.
\end{itemize}

Este módulo configura así la base sólida de datos raw para todo el sistema de swing trading, asegurando que el backtest industrial que construiréis en \texttt{backtest/engine.py} se base en un dataset \textbf{limpio, reproducible y sin leakage}. [web:56][web:62]
