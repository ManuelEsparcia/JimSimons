\section{Módulo: \texttt{data/edgar/ticker\_cik.py}}

\subsection{Objetivo general del módulo}

El módulo \texttt{ticker\_cik.py} construye y mantiene un \emph{mapping versiónado} entre \texttt{ticker} y \texttt{CIK}, así como otros identificadores relacionados (por ejemplo, \texttt{source}, \texttt{exchange}, \texttt{sector}), de forma \emph{trazable} y \emph{historificada}, para enlazar el universo de acciones negociables con la base de datos EDGAR de la SEC.

Formalmente, el módulo define una función de \emph{mapping master}:
\[
  \mathcal{M}: \mathcal{U}_{\text{raw}} \times \Theta_{\text{map}} \longrightarrow \mathcal{H} \times \mathcal{C},
\]
donde:
\begin{itemize}
  \item \(\mathcal{U}_{\text{raw}}\) es el conjunto de mappings crudos de entrada (SEC y mappers internos),
  \item \(\Theta_{\text{map}}\) es el conjunto de parámetros de validación y dedupe,
  \item \(\mathcal{H}\) es el historial versionado de mappings \texttt{ticker}<->\texttt{CIK},
  \item \(\mathcal{C}\) es el snapshot \texttt{current} del mapping vigente.
\end{itemize}

Este módulo es la \emph{capa de resolución de identidad} entre el universo cuantitativo (ticker/asset) y la base de datos regulatoria (CIK/filer), crítica para evitar \emph{mismatches} y \emph{leakage} al cruzar series de cotización con series de datos fundamentales\cite{DePrado2018Advances, SECEDGARAPI}.

\subsection{Alcance del módulo}

El alcance incluye:

\begin{itemize}
  \item \textbf{Ingestión de fuentes de mapping}:
    \begin{itemize}
      \item fuentes oficiales de la SEC (por ejemplo, JSON de \texttt{ticker}<->\texttt{CIK}),
      \item y mappers internos (\texttt{internal\_symbol\_master}) con información adicional (exchange, sector, etc.).
    \end{itemize}
  \item \textbf{Normalización de formatos}:
    \begin{itemize}
      \item \texttt{ticker} en mayúsculas y sin espacios extras,
      \item \texttt{CIK} en formato \emph{zero‑padded} (10 dígitos),
      \item y campos de metadatos (como \texttt{source}) bien tipados.
    \end{itemize}
  \item \textbf{Dedupe y resolución de conflictos}:
    \begin{itemize}
      \item eliminación de duplicados activos \texttt{symbol}->\texttt{CIK} solapados en tiempo,
      \item y resolución de discrepancias de \texttt{CIK} para el mismo \texttt{symbol} en distintas fuentes.
    \end{itemize}
  \item \textbf{Histórico versionado de cambios}:
    \begin{itemize}
      \item cada cambio en la asociación \texttt{ticker}<->\texttt{CIK} queda registrado con \texttt{effective\_from}, \texttt{effective\_to}, y \texttt{is\_active}.
    \end{itemize}
  \item \textbf{Snapshot \texttt{current}}:
    \begin{itemize}
      \item una vista actualizada del mapping vigente para el universo completo.
    \end{itemize}
\end{itemize}

El módulo se alinea con prácticas de \emph{master reference data} en sistemas de \emph{data mesh} financieros, donde se mantiene un historial de cambios de identificación para evitar \emph{entity matching errors} en backtesting\cite{DePrado2018Advances, WorldbankDataQuality}.

\subsection{Entradas del módulo}

Las entradas del módulo son:

\begin{itemize}
  \item \texttt{sec\_ticker\_source\_path/api}: ruta o API a la fuente oficial de mapeo \texttt{ticker}<->\texttt{CIK} de la SEC (por ejemplo, un archivo JSON o un endpoint REST).
  \item \texttt{internal\_symbol\_master\_path} (opcional): ruta a un mapeo interno de símbolos con metadatos adicionales (exchange, sector, etc.).
  \item \texttt{config\_path}: ruta a un archivo YAML de configuración con reglas de normalización, pesaje de fuentes y políticas de dedupe.
  \item \texttt{run\_id}: identificador único de la ejecución para trazabilidad y replicabilidad.
\end{itemize}

En notación formal, el conjunto de entradas es:
\[
  \mathcal{I} = \{ \mathcal{U}_{\text{sec}}, \mathcal{U}_{\text{int}}, \Theta_{\text{map}}, r \},
\]
donde:
\begin{itemize}
  \item \(\mathcal{U}_{\text{sec}}\) es el universo de mappings de la SEC,
  \item \(\mathcal{U}_{\text{int}}\) es el universo interno de símbolos,
  \item \(\Theta_{\text{map}}\) es el conjunto de parámetros de resolución de conflictos,
  \item \(r\) es el \texttt{run\_id}.
\end{itemize}

\subsection{Salidas y artefactos}

El módulo genera tres artefactos clave:

\begin{itemize}
  \item \texttt{data/edgar/mappings/ticker\_cik\_history.parquet}: tabla de histórico de mappings \texttt{ticker}<->\texttt{CIK} con ventana temporal \texttt{effective\_from}--\texttt{effective\_to}.
  \item \texttt{data/edgar/mappings/ticker\_cik\_current.parquet}: snapshot del mapping vigente por \texttt{symbol} y \texttt{CIK}.
  \item \texttt{data/edgar/mappings/ticker\_cik\_manifest\_\{run\_id\}.json}: metadatos de la ejecución (fuentes utilizadas, fecha de ingestión, etc.).
\end{itemize}

Estos artefactos permiten:
\begin{itemize}
  \item reconstruir exactamente el mismo snapshot de mappings a partir de \texttt{run\_id},
  \item monitorear cambios históricos de \texttt{ticker} y \texttt{CIK} (por ejemplo, cambios de ticker, reestructuraciones de empresas),
  \item y usar \texttt{ticker\_cik\_history} para cruzar cotizaciones y datos fundamentales en \emph{point‑in‑time} de forma consistente\cite{SECEDGARAPI, DePrado2018Advances}.
\end{itemize}

\subsection{Modelo de datos y campos mínimos}

El módulo asegura que cada fila de mapping contenga al menos los siguientes campos:

\begin{itemize}
  \item \texttt{symbol}: símbolo de la acción en notación \texttt{UPPERCASE} y sin espacios superfluos.
  \item \texttt{cik}: CIK de 10 dígitos, con \emph{zero‑padding} consistente.
  \item \texttt{source}: identificador de la fuente (por ejemplo, \texttt{SEC\_TICKER\_CIK}, \texttt{INTERNAL\_MASTER}).
  \item \texttt{effective\_from}: fecha de inicio de vigencia de la asociación \texttt{symbol}<->\texttt{CIK}.
  \item \texttt{effective\_to}: fecha de fin de vigencia (o \texttt{NULL} si es vigente).
  \item \texttt{is\_active}: flag que indica si la asociación es vigente en la fecha de \texttt{run\_id}.
  \item \texttt{run\_id}: identificador de la ejecución.
  \item \texttt{ingest\_ts\_utc}: timestamp UTC de ingestión del registro.
\end{itemize}

En notación matemática, cada registro de mapping se representa como:
\[
  m = (s, c, \sigma, f, t, a, r, \tau),
\]
donde:
\begin{itemize}
  \item \(s\) es el \texttt{symbol},
  \item \(c\) es el \texttt{CIK},
  \item \(\sigma\) es el \texttt{source},
  \item \(f\) es \texttt{effective\_from},
  \item \(t\) es \texttt{effective\_to},
  \item \(a\) es \texttt{is\_active},
  \item \(r\) es el \texttt{run\_id},
  \item \(\tau\) es \texttt{ingest\_ts\_utc}.
\end{itemize}

\subsection{Checks y validaciones}

El módulo aplica varios checks obligatorios:

\begin{itemize}
  \item \textbf{Formato canonico de CIK}:
    \[
      c \in \{0,1,\dots,9\}^{10},
    \]
    es decir, el CIK debe ser exactamente 10 dígitos, comprensible con la definición de CIK de la SEC\cite{SECEDGARAPI, SECEDGARDataQuality}.

  \item \textbf{Normalización de \texttt{symbol}}:
    \[
      s \leftarrow \texttt{UPPER}( \texttt{TRIM}(s) ),
    \]
    garantizando consistencia de notación a lo largo de fuentes dispares (SEC, API externas, bases internas).

  \item \textbf{No duplicados activos de \texttt{symbol}->\texttt{CIK} simultáneos}:
    \[
      \forall (s, c_1, c_2) \text{ con } c_1 \ne c_2, \ \nexists t \text{ tal que } t \in [f_1, t_1] \cap [f_2, t_2] \text{ y } a = 1,
    \]
    donde \(f_i, t_i\) son los intervalos de \texttt{effective\_from}--\texttt{effective\_to} para cada asociación. Esto evita \emph{overlapping} activos de \texttt{ticker}<->\texttt{CIK}.

  \item \textbf{Versionado de cambios históricos}:
    Las actualizaciones de \texttt{CIK} se manejan mediante:
    \begin{itemize}
      \item cierre de \texttt{effective\_to} del registro anterior,
      \item y apertura de \texttt{effective\_from} del nuevo registro,
      manteniendo así un historial continuo de cambios.
    \end{itemize}
\end{itemize}

\subsection{Flujo de ejecución}

El flujo del módulo se puede descomponer en cinco pasos:

\begin{enumerate}
  \item \textbf{Cargar fuente SEC y mappings internos}:
    \begin{itemize}
      \item Se descarga o lee la fuente \texttt{sec\_ticker\_source\_path/api}.
      \item Se carga \texttt{internal\_symbol\_master\_path} si está disponible.
    \end{itemize}

  \item \textbf{Normalizar campos}:
    \begin{itemize}
      \item \texttt{symbol} se lleva a \texttt{UPPERCASE} y se limpian espacios.
      \item \texttt{CIK} se rellena a 10 dígitos.
      \item Se unifican esquemas de campos (por ejemplo, \texttt{ingest\_ts\_utc} se genera para todos los registros).
    \end{itemize}

  \item \textbf{Resolver conflictos y deduplicar}:
    \begin{itemize}
      \item Se detectan \texttt{ticker}s con múltiples \texttt{CIK} vigentes en el mismo intervalo temporal.
      \item Se aplican reglas de resolución de conflicto (por ejemplo, preferir \texttt{source} SEC sobre \texttt{INTERNAL\_MASTER}, o aplicar \emph{last writer wins} con trazabilidad).
    \end{itemize}

  \item \textbf{Construir \texttt{history} y \texttt{current}}:
    \begin{itemize}
      \item \texttt{ticker\_cik\_history} almacena todas las asociaciones con \texttt{effective\_from}--\texttt{effective\_to}.
      \item \texttt{ticker\_cik\_current} es el subconjunto donde \texttt{effective\_to = NULL} o donde \texttt{is\_active = 1}.
    \end{itemize}

  \item \textbf{Persistir outputs}:
    \begin{itemize}
      \item Se guarda \texttt{ticker\_cik\_history.parquet}.
      \item Se guarda \texttt{ticker\_cik\_current.parquet}.
      \item Se genera \texttt{ticker\_cik\_manifest\_\{run\_id\}.json} con metadatos de la ejecución.
    \end{itemize}
\end{enumerate}

Este flujo reproduce el patrón estándar de \emph{historical mapping} para identificadores de entidades, similar a \emph{CIK mapping services} y \emph{ticker‑CIK crosswalks} usados en pipelines de datos fundamentales\cite{sec_cik_mapper, cik-mapping-SEC}.

\subsection{CLI y uso operativo}

El módulo se invoca mediante la línea de comandos:

\begin{verbatim}
python -m data.edgar.ticker_cik ^
  --source-path data/edgar/source/ticker_cik_raw.parquet ^
  --config-path configs/data/edgar_ticker_cik.yaml ^
  --run-id ticker_cik_20260305_001
\end{verbatim}

Esto permite:
\begin{itemize}
  \item integrar el módulo en pipelines de construcción de datos fundamentales,
  \item actualizar periódicamente el mapping conforme cambian \texttt{ticker}s y \texttt{CIK}s en la base de datos de la SEC,
  \item y mantener versiones reproducibles de mapeos para cada \texttt{run\_id}.
\end{itemize}

\subsection{Definición de término (DoD)}

La definición de término del módulo establece que:

\begin{itemize}
  \item El mapping \texttt{ticker}<->\texttt{CIK} debe ser \emph{estable}:
    \begin{itemize}
      \item sin duplicados activos solapados temporalmente,
      \item y con formato consistente de \texttt{CIK} y \texttt{symbol}.
    \end{itemize}
  \item El mapping debe ser \emph{trazable}:
    \begin{itemize}
      \item se debe poder recuperar el historial de cambios de \texttt{ticker}<->\texttt{CIK} desde \texttt{ticker\_cik\_history.parquet},
      \item y se debe poder reproducir exactamente el \texttt{current} snapshot por \texttt{run\_id}.
    \end{itemize}
  \item El módulo debe generar outputs listos para ser consumidos por:
    \begin{itemize}
      \item \texttt{point\_in\_time} (para cruces \texttt{ticker}<->\texttt{CIK} PIT),
      \item y \texttt{edgar\_qc} (para validación de cobertura de CIK sobre el universo de símbolos).
    \end{itemize}
\end{itemize}

\subsection{Conexión con estado del arte y literatura}

El diseño de \texttt{ticker\_cik.py} se alinea con:

\begin{itemize}
  \item \textbf{Mapping de ticker<->CIK de la SEC}: la SEC y terceros mantienen tablas de \texttt{ticker} a \texttt{CIK} en formato \emph{crosswalk} para facilitar el acceso a filings\cite{SECEDGARAPI, sec_cik_mapper}.
  \item \textbf{Historización de cambios de identidad}: el uso de \texttt{effective\_from}--\texttt{effective\_to} y \texttt{is\_active} se inspira en prácticas de \emph{slowly changing dimensions} (SCD) en sistemas de datos financieros, donde se preserva la historia de cambios de identificación para evitar \emph{entity matching errors} en series de tiempo\cite{KimballSCD, DePrado2018Advances}.
  \item \textbf{Calidad de datos de referencia}: el módulo implementa reglas de validación de \texttt{CI
