\section{Módulo: \texttt{data/edgar/parse\_xbrl.py}}

\subsection{Objetivo general del módulo}

El módulo \texttt{parse\_xbrl.py} normaliza y transforma \emph{facts XBRL} provenientes de \texttt{submissions} y \texttt{companyfacts} de EDGAR en una \emph{tabla de facts fundamentales canónica}, lista para ser usada en la construcción de series \emph{point‑in‑time} (PIT) y modelos de trading cuantitativo.

Formalmente, el módulo implementa una función de normalización:
\[
  \mathcal{N}: \mathcal{B} \times \mathcal{M} \longrightarrow \mathcal{F}^{\text{c}},
\]
donde:
\begin{itemize}
  \item \(\mathcal{B}\) es el conjunto de datos brutos de XBRL (submissions y companyfacts),
  \item \(\mathcal{M}\) es el conjunto de mappings de tags a métricas internas,
  \item \(\mathcal{F}^{\text{c}}\) es el espacio de facts fundamentales en una métrica canónica, con unidades, periodos y contexto homogéneo.
\end{itemize}

Este módulo es \emph{la capa de traducción} entre la taxonomía XBRL (US‑GAAP, DEI, etc.) y el esquema de metricas interno usado por el sistema de trading, permitiendo comparabilidad trans‑temporal y trans‑sectorial\cite{XBRLGuide, XBRLUSGAAPGuide}.

\subsection{Alcance del módulo}

El alcance incluye:

\begin{itemize}
  \item \textbf{Mapping de tags a métricas internas}: asociar conceptos XBRL (por ejemplo, \texttt{RevenueFromContractWithCustomerExcludingAssessedTax}) a métricas internas normalizadas (por ejemplo, \texttt{Revenue\_Canonical}).
  \item \textbf{Normalización de unidades y escala}: convertir todas las unidades (USD, shares, PER‑share, miles, millones, etc.) a una unidad base definida.
  \item \textbf{Selección de periodos}: seleccionar el registro más relevante por \((cik, \texttt{metric\_name}, \texttt{period\_end}, \texttt{filed\_date})\).
  \item \textbf{Deduplicación de versiones}: eliminar duplicados de facts alineados por \emph{periodo de informe} y \emph{fecha de presentación}.
\end{itemize}

El módulo actúa como \emph{ETL de normalización} para datos XBRL, análogo a un \emph{data normalization engine} en sistemas de \emph{structured data for financial analysis}\cite{XBRLWhyNormalize, XBRLUSGAAPGuide}. Su salida es la base de datos inmediata de \texttt{facts\_canonical} consumida por \texttt{point\_in\_time} y modelos de factores fundamentales.

\subsection{Entradas del módulo}

Las entradas del módulo son:

\begin{itemize}
  \item \texttt{submissions\_raw\_path}: ruta al almacén raw de submissions de EDGAR (por ejemplo, \texttt{data/edgar/raw/submissions}).
  \item \texttt{companyfacts\_raw\_path}: ruta al almacén raw de companyfacts XBRL (por ejemplo, \texttt{data/edgar/raw/companyfacts}).
  \item \texttt{xbrl\_mapping\_config\_path}: ruta a un archivo de configuración donde se definen mappings de tags a \texttt{metric\_name} y reglas de normalización.
  \item \texttt{run\_id}: identificador único de la ejecución para trazabilidad y replicabilidad.
\end{itemize}

En notación formal, el conjunto de entradas es:
\[
  \mathcal{I} = \{ \mathcal{B}_S, \mathcal{B}_F, \Theta_{\text{map}}, r \},
\]
donde:
\begin{itemize}
  \item \(\mathcal{B}_S\) es el conjunto de submissions,
  \item \(\mathcal{B}_F\) es el conjunto de companyfacts,
  \item \(\Theta_{\text{map}}\) es el conjunto de mappings y reglas de normalización,
  \item \(r\) es el \texttt{run\_id}.
\end{itemize}

\subsection{Salidas y artefactos}

El módulo genera tres artefactos clave:

\begin{itemize}
  \item \texttt{data/edgar/parsed/facts\_canonical\_\{run\_id\}.parquet}: tabla de facts fundamentales canónicos, lista para ser usada en \texttt{point\_in\_time}.
  \item \texttt{data/edgar/parsed/facts\_rejected\_\{run\_id\}.parquet}: registros de facts que fallaron alguno de los criterios de normalización o validación.
  \item \texttt{data/edgar/parsed/manifest\_\{run\_id\}.json}: metadatos de la ejecución (número de facts procesados, dropped, etc.).
\end{itemize}

Estos artefactos permiten:
\begin{itemize}
  \item reconstruir la misma versión de datos canónicos a partir de \texttt{run\_id},
  \item monitorear la tasa de facts \emph{rejected} y detectar cambios en la taxonomía o en el pipeline de ingesta,
  \item y usar \texttt{facts\_rejected} para auditoría de casos marginales (por ejemplo, nuevos tags US‑GAAP, extensiones de filer, etc.).
\end{itemize}

\subsection{Modelo de datos y campos mínimos}

El módulo asegura que cada fila de \texttt{facts\_canonical} contenga al menos los siguientes campos:

\begin{itemize}
  \item \texttt{cik}: clave de identificación de la entidad.
  \item \texttt{symbol} (si está disponible en el mapping): ticker asociado al CIK.
  \item \texttt{metric\_name}: nombre de métrica canonico interno (por ejemplo, \texttt{Revenue\_Canonical}, \texttt{NetIncome\_Canonical}).
  \item \texttt{value}: valor numérico de la métrica, en la unidad de base definida.
  \item \texttt{period\_end}: fecha de término del período de informe (por ejemplo, 31‑12‑2023).
  \item \texttt{period\_type} (\texttt{Q} o \texttt{YTTM}): tipo de período (trimestral o anual/TTM).
  \item \texttt{filed\_date}: fecha de presentación del filing.
  \item \texttt{acceptance\_datetime}: timestamp de aceptación en EDGAR.
  \item \texttt{source\_tag}: tag XBRL original (por ejemplo, \texttt{RevenueFromContractWithCustomerExcludingAssessedTax}).
  \item \texttt{unit}: unidad de medida, homogeneizada a una base común.
  \item \texttt{run\_id}: identificador de la ejecución.
\end{itemize}

En notación matemática, cada registro canónico se representa como:
\[
  f^{\text{c}} = (c, s, m, v, e, \tau, f_d, \alpha, t, u, r),
\]
donde:
\begin{itemize}
  \item \(c \in \mathcal{C}\) es el CIK,
  \item \(s\) es el \texttt{symbol},
  \item \(m\) es el \texttt{metric\_name},
  \item \(v \in \mathbb{R}\) es el valor normalizado,
  \item \(e\) es \texttt{period\_end},
  \item \(\tau \in \{\texttt{Q}, \texttt{YTTM}\}\) es el \texttt{period\_type},
  \item \(f_d\) es \texttt{filed\_date},
  \item \(\alpha\) es \texttt{acceptance\_datetime},
  \item \(t\) es el \texttt{source\_tag},
  \item \(u\) es la unidad normalizada,
  \item \(r\) es el \texttt{run\_id}.
\end{itemize}

\subsection{Reglas de normalización y selección}

El módulo aplica varias reglas clave:

\begin{itemize}
  \item \textbf{Mapping tag \(\rightarrow\) metric\_name versionado}:
    \[
      m = \Phi_{\theta}(t),
    \]
    donde \(\Phi_{\theta}\) es una función de mapping parametrizada por \(\theta\), versionada por \texttt{run\_id}, tal que:
    \[
      \forall t \in \mathcal{T}, \ \Phi_{\theta}(t) \in \mathcal{M}_{\text{internal}},
    \]
    y \(\mathcal{T}\) es el conjunto de tags XBRL y \(\mathcal{M}_{\text{internal}}\) es el conjunto de métricas internas.

  \item \textbf{Conversión de unidades a base definida}:
    Para una unidad original \(u_0\) y un factor de escala \(s_u\), el valor se normaliza como:
    \[
      v = v_0 \cdot s_u,
    \]
    de forma que todas las unidades se expresen en una base común (por ejemplo, USD en millones, PER‑share sin escala, etc.)\cite{XBRLUnitDeclarations}. Esto permite comparabilidad homogénea entre empresas y periodos\cite{XBRLWhyNormalize}.

  \item \textbf{Selección del registro más reciente por clave}:
    Se define la clave:
    \[
      k = (c, m, e, f_d),
    \]
    y se selecciona:
    \[
      f^{\text{c}}_{\text{best}} = \arg\max_{f \in \mathcal{F}(k)} f_d,
    \]
    es decir, se retiene el registro con la fecha de presentación más reciente para cada combinación \((cik, metric\_name, period\_end, filed\_date)\), garantizando la versión más actualizada de cada hecho\cite{XBRLGuide}.

  \item \textbf{No mezclar taxonomías incompatibles sin flag}:
    Si un hecho proviene de una taxonomía distinta a la base (por ejemplo, \texttt{ifrs} en lugar de \texttt{us‑gaap}), se debe:
    \begin{itemize}
      \item o convertirlo a la métrica canónica mediante un mapeo bien definido,
      \item o mantenerlo en un conjunto separado con un flag que indique la fuente.
    \end{itemize}
    Esto evita bugs de \emph{mixing taxonomies} que pueden introducir sesgos de comparabilidad entre mercados\cite{XBRLTechnicalGuide}.
\end{itemize}

\subsection{Flujo de ejecución}

El flujo del módulo se puede descomponer en cuatro pasos:

\begin{enumerate}
  \item \textbf{Carga de raw facts y mapping config}:
    Se leen \texttt{submissions\_raw\_path} y \texttt{companyfacts\_raw\_path}, y se cargan los mappings de tags definidos en \texttt{xbrl\_mapping\_config\_path}.

  \item \textbf{Aplicación de mapping y conversión de unidades}:
    \begin{itemize}
      \item Se mapea cada tag XBRL a \texttt{metric\_name}.
      \item Se aplican conversiones de unidad y escala según una tabla de factores predefinidos.
    \end{itemize}

  \item \textbf{Resolución de conflictos y deduplicados}:
    Para cada clave \((cik, metric\_name, period\_end, filed\_date)\):
    \begin{itemize}
      \item se selecciona el registro más reciente,
      \item se marcan como \texttt{rejected} los duplicados y los registros con taxonomías incompatibles o sin metric\_name.
    \end{itemize}

  \item \textbf{Emisión de outputs}:
    \begin{itemize}
      \item Se guarda \texttt{facts\_canonical\_\{run\_id\}.parquet}.
      \item Se guarda \texttt{facts\_rejected\_\{run\_id\}.parquet}.
      \item Se genera \texttt{manifest\_\{run\_id\}.json} con metadatos de la ejecución.
    \end{itemize}
\end{enumerate}

Este flujo implementa un patrón de \emph{normalization ETL} para datos XBRL, donde se transforma una estructura anidada y heterogénea de facts en una tabla tabular consistente, lista para ser consumida por sistemas de time series PIT y modelos de factores fundamentales\cite{CampbellEtAl2001, DePrado2018Advances}.

\subsection{Validaciones aplicadas}

El módulo aplica varias validaciones para asegurar calidad:

\begin{itemize}
  \item \textbf{metric\_name no nulo para registros aceptados}:
    \[
      \forall f \in \mathcal{F}_{\text{accepted}}, \ m \neq \emptyset,
    \]
    es decir, se rechazan automática o explícitamente los facts que no pueden ser mapeados a una métrica interna.

  \item \textbf{value en rango razonable según métrica}:
    Para cada métrica \(m\), se verifica que:
    \[
      v \in [\underline{v}_m, \overline{v}_m],
    \]
    donde \(\underline{v}_m\) y \(\overline{v}_m\) son límites de rango económico (por ejemplo, \texttt{Revenue} y \texttt{TotalAssets} no pueden ser negativos, \texttt{EPS} no debe ser infinito, etc.). Valores fuera de rango se marcan como \texttt{rejected} o se señalan con flags de anomalía.

  \item \textbf{fechas coherentes}:
    Debe cumplirse:
    \[
      f_d \ge e,
    \]
    es decir, la fecha de presentación no puede ser anterior a la fecha de término del período de informe, evitando \emph{look‑ahead bias} en el manejo de facts temporales\cite{SECXBRLGuide}.
\end{itemize}

\subsection{CLI y uso operativo}

El módulo se invoca mediante la línea de comandos:

\begin{verbatim}
python -m data.edgar.parse_xbrl ^
  --submissions-path data/edgar/raw/submissions ^
  --companyfacts-path data/edgar/raw/companyfacts ^
  --config-path configs/data/edgar_parse_xbrl.yaml ^
  --run-id parse_xbrl_20260305_001
\end{verbatim}

Esto permite:
\begin{itemize}
  \item integrar el módulo en pipelines de features fundamental y PIT,
  \item actualizar periódicamente la tabla canónica conforme llegan nuevos filings,
  \item y mantener versiones reproducibles de mappings y normalizaciones mediante \texttt{run\_id}.
\end{itemize}

\subsection{Definición de término (DoD)}

La definición de término del módulo establece que:

\begin{itemize}
  \item La tabla \texttt{facts\_canonical\_\{run\_id\}.parquet} debe ser \emph{consistente} y lista para ser consumida por el módulo \texttt{point\_in\_time}, sin discrepancias de:
    \begin{itemize}
      \item métricas internas,
      \item unidades de medida,
      \item y estructura de periodos.
    \end{itemize}
  \item No debe haber facts aceptados sin \texttt{metric\_name} ni facts aceptados con violaciones graves de rango o fechas.
  \item Debe existir un registro auditado de rechazos y metadatos en \texttt{facts\_rejected\_\{run\_id\}.parquet} y \texttt{manifest\_\{run\_id\}.json}, que permita rastrear la evolución de la calidad de los mappings XBRL a lo largo del tiempo.
\end{itemize}
