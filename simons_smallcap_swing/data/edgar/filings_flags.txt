\section{Módulo: \texttt{data/edgar/filings\_flags.py}}

\subsection{Objetivo general del módulo}

El módulo \texttt{filings\_flags.py} genera \emph{flags de calidad y evento} sobre los filings EDGAR de un conjunto de empresas, con el fin de:

\begin{itemize}
  \item filtrar o ponderar observaciones fundamentales en \emph{research} y \emph{modelado},
  \item identificar periodos de baja confiabilidad de los datos financieros,
  \item y capturar señales de riesgo contable y de governance que se pueden usar como features de \emph{credit risk} o \emph{liquidity risk} en estrategias de swing trading.
\end{itemize}

Formalmente, el módulo define una \emph{función de flags}:
\[
  \mathcal{F}: \mathcal{S} \times \mathcal{F} \longrightarrow \{0,1\}^d,
\]
donde:
\begin{itemize}
  \item \(\mathcal{S}\) es el conjunto de submissions raw,
  \item \(\mathcal{F}\) es el conjunto de facts financieros parseados,
  \item \(\{0,1\}^d\) es un vector de \(d\) flags binarios (por ejemplo, \texttt{late\_filer}, \texttt{amended}, \texttt{restatement}, etc.).
\end{itemize}
Este vector de flags se usa luego como \emph{máscara de calidad} en la construcción de series fundamentales PIT y en la generación de \emph{labels} de riesgo\cite{DeFondZhang2014, DeepQuarryRestatements}.

\subsection{Alcance del módulo}

El alcance del módulo incluye:

\begin{itemize}
  \item \textbf{Detección de late filers}: identificar empresas que retrasan sistemáticamente sus presentaciones regulatorias (10‑K, 10‑Q, etc.).
  \item \textbf{Detección de amended forms}: identificar presentaciones corregidas (10‑K/A, 10‑Q/A), que pueden ser indicadores de errores o ajustes contables significativos.
  \item \textbf{Detección de restatements}: identificar periodos donde se han corregido estados financieros previos, asociados a errores de auditoría o ajustes de estimación\cite{DeFondZhang2014, DeepQuarryRestatements}.
  \item \textbf{Detección de fields faltantes}: detectar registros de facts donde campos clave (por ejemplo, \texttt{Revenue}, \texttt{NetIncome}, \texttt{TotalAssets}) están ausentes o no reportados.
  \item \textbf{Eventos contables anormales}: detectar cambios bruscos o no racionales en series de facts (por ejemplo, saltos de \texttt{TotalAssets} o \texttt{RetainedEarnings} que exceden umbrales económicos).
\end{itemize}

Estos flags se alinean con la literatura de \emph{accounting quality} y \emph{audit risk}, donde:
\begin{itemize}
  \item los \emph{late filers} se asocian a debilidades de controles internos y a mayor probabilidad de errores\cite{SECDeadlineImpact, SECWarningLateFilings},
  \item los \emph{amended filings} y \emph{restatements} se usan como proxies de baja calidad de auditoría\cite{DeFondZhang2014, DeepQuarryRestatements},
  \item y la ausencia de campos clave se asocia a opacidad de información y riesgo de ocultación de problemas de liquidez o rentabilidad.
\end{itemize}

\subsection{Entradas del módulo}

Las entradas del módulo son:

\begin{itemize}
  \item \texttt{submissions\_raw\_path}: ruta a los datos raw de submissions de EDGAR (por ejemplo, \texttt{data/edgar/raw/submissions}).
  \item \texttt{parsed\_facts\_path}: ruta a los facts financieros parseados (por ejemplo, \texttt{data/edgar/parsed/facts\_canonical\_latest.parquet}).
  \item \texttt{flags\_config\_path}: ruta a un archivo YAML de configuración con umbrales, reglas de flagging y reglas de \emph{quality scoring}.
  \item \texttt{run\_id}: identificador único de la ejecución para trazabilidad y replicabilidad.
\end{itemize}

En notación formal, el conjunto de entradas es:
\[
  \mathcal{I} = \{ \mathcal{S}, \mathcal{F}, \theta_{\text{flags}}, r \},
\]
donde:
\begin{itemize}
  \item \(\mathcal{S}\) es el conjunto de submissions,
  \item \(\mathcal{F}\) es el conjunto de facts financieros,
  \item \(\theta_{\text{flags}}\) agrupa umbrales y reglas de flagging,
  \item \(r\) es el \texttt{run\_id}.
\end{itemize}

\subsection{Salidas y artefactos}

El módulo genera tres artefactos clave:

\begin{itemize}
  \item \texttt{data/edgar/flags/filings\_flags\_\{run\_id\}.parquet}: tabla de flags por filing y empresa, con campos de calidad y eventos.
  \item \texttt{data/edgar/flags/filings\_flags\_summary\_\{run\_id\}.json}: resumen de distribución de flags (por ejemplo, porcentaje de \texttt{late\_filers}, \texttt{amended}, \texttt{restatement}).
  \item \texttt{data/edgar/flags/manifest\_\{run\_id\}.json}: metadatos de la ejecución (número de filings procesados, umbrales aplicados, etc.).
\end{itemize}

Estos artefactos permiten:
\begin{itemize}
  \item construir \emph{cohorts} de empresas con distinta calidad de reporting,
  \item ponderar o descartar observaciones fundamentales en la construcción de features PIT,
  \item y usar los flags como \emph{features de riesgo} en modelos de credito y estrategias de swing trading\cite{CampbellEtAl2001, Chan2009QuantitativeTrading}.
\end{itemize}

\subsection{Modelo de datos y campos mínimos}

El módulo asegura que cada fila de flags contenga al menos los siguientes campos:

\begin{itemize}
  \item \texttt{cik}: clave de identificación de la entidad.
  \item \texttt{accession\_number}: identificador único del filing.
  \item \texttt{filing\_date}: fecha de presentación del filing.
  \item \texttt{form\_type}: tipo de formulario (por ejemplo, \texttt{10‑K}, \texttt{10‑K/A}, \texttt{10‑Q}).
  \item \texttt{flag\_late\_filer}: flag binario que indica si el filing es \emph{late} respecto a la normativa de plazos de la SEC.
  \item \texttt{flag\_amended}: flag binario que indica si el filing es una versión corregida (por ejemplo, \texttt{10‑K/A}).
  \item \texttt{flag\_restatement}: flag binario que indica si el filing contiene o está asociado a una \emph{restatement}.
  \item \texttt{flag\_missing\_core\_fields}: flag binario que indica la ausencia de campos clave en el conjunto de facts.
  \item \texttt{quality\_score}: puntuación de calidad agregada, normalizada en \([0,1]\), que combina todos los flags.
  \item \texttt{run\_id}: identificador de la ejecución.
\end{itemize}

En notación matemática, cada registro de flags se representa como:
\[
  f = (c, a, f_d, \tau, \ell, m, r, \mu, q, r),
\]
donde:
\begin{itemize}
  \item \(c \in \mathcal{C}\) es el CIK,
  \item \(a\) es el \texttt{accession\_number},
  \item \(f_d\) es \texttt{filing\_date},
  \item \(\tau\) es el \texttt{form\_type},
  \item \(\ell \in \{0,1\}\) es \texttt{flag\_late\_filer},
  \item \(m \in \{0,1\}\) es \texttt{flag\_missing\_core\_fields},
  \item \(r \in \{0,1\}\) es \texttt{flag\_restatement},
  \item \(\mu \in \{0,1\}\) es \texttt{flag\_amended},
  \item \(q \in [0,1]\) es el \texttt{quality\_score},
  \item \(r\) es el \texttt{run\_id}.
\end{itemize}

\subsection{Reglas de flags y calidad}

El módulo aplica varias reglas de flagging:

\begin{itemize}
  \item \textbf{Late filer}:
    \[
      \ell = \mathbb{I}\big( f_d > d_{\text{deadline}}(c, \tau) \big),
    \]
    donde \(d_{\text{deadline}}(c, \tau)\) es el plazo regulatorio de presentación para el CIK \(c\) y el tipo de formulario \(\tau\) (por ejemplo, 60 días para 10‑Q, 90 días para 10‑K para non‑accelerated filers). La literatura muestra que los \emph{late filers} presentan mayor probabilidad de errores de reporting y de restatements\cite{SECDeadlineImpact, SECWarningLateFilings}.

  \item \textbf{Amended form}:
    \[
      m = \mathbb{I}\big( \tau \text{ termina con } \texttt{/A} \big),
    \]
    es decir, si el \texttt{form\_type} es una versión corregida. Los amended filings se interpretan como \emph{red flags} por parte de los inversores\cite{ToppanMerrillAmendedFilings, DeepQuarryRestatements}.

  \item \textbf{Restatement}:
    \[
      r = \mathbb{I}\big( \exists \text{ restatement asociado al filing } a \big),
    \]
    donde la detección de restatement puede basarse en:
    \begin{itemize}
      \item presencia de \texttt{error correction} checkbox en 10‑K/A,
      \item cambios significativos en \texttt{RetainedEarnings} o \texttt{NetIncome} de un periodo a otro,
      \item o señales de cambio de auditoría o comentarios de la SEC.
    \end{itemize}
    La literatura de \emph{accounting quality} usa restatements como indicador de baja calidad de auditoría\cite{DeFondZhang2014, DeepQuarryRestatements}.

  \item \textbf{Missing core fields}:
    \[
      \mu = \mathbb{I}\big( \text{faltar alguno de los campos clave en } \mathcal{F} \big),
    \]
    donde los campos clave pueden incluir \texttt{Revenue}, \texttt{NetIncome}, \texttt{TotalAssets}, \texttt{TotalLiabilities}, etc. La ausencia de estos campos se asocia a opacidad de información y riesgo de \emph{data snooping} sesgada\cite{DePrado2018Advances}.

  \item \textbf{Quality score}:
    \[
      q = \frac{1}{w} \sum_{i=1}^{d} w_i (1 - f_i),
    \]
    donde:
    \begin{itemize}
      \item \(f_i \in \{0,1\}\) es el \(i\)-ésimo flag de calidad,
      \item \(w_i\) es el peso asociado al flag,
      \item \(w = \sum_{i=1}^{d} w_i\) normaliza el score entre 0 y 1.
    \end{itemize}
    El \texttt{quality\_score} se puede usar como \emph{weight} en la construcción de features PIT, de modo que observaciones con baja calidad se ponderen menos en el backtest\cite{CampbellEtAl2001, Chan2009QuantitativeTrading}.
\end{itemize}

\subsection{Flujo de ejecución}

El flujo del módulo se puede descomponer en cuatro pasos:

\begin{enumerate}
  \item \textbf{Carga de filings y facts}:
    Se leen \texttt{submissions\_raw\_path} y \texttt{parsed\_facts\_path}, y se unen por \texttt{cik} y \texttt{accession\_number} para obtener el contexto de cada filing.

  \item \textbf{Evaluación de reglas de flags}:
    Se aplican las reglas de \texttt{late\_filer}, \texttt{amended}, \texttt{restatement}, \texttt{missing\_core\_fields} y otros eventos contables anormales, generando un vector binario por registro.

  \item \textbf{Asignación de quality score}:
    Se calcula el \texttt{quality\_score} agregado por empresa y por periodo de informe, usando pesos especificados en \texttt{flags\_config\_path}.

  \item \textbf{Persistencia de outputs}:
    \begin{itemize}
      \item Se guarda \texttt{filings\_flags\_\{run\_id\}.parquet} con todos los flags por filing.
      \item Se genera \texttt{filings\_flags\_summary\_\{run\_id\}.json} con estadísticas de cobertura y distribución de flags.
      \item Se guarda un \texttt{manifest\_\{run\_id\}.json} con metadatos de la ejecución.
    \end{itemize}
\end{enumerate}

Este flujo se puede ver como un \emph{feature engineering} de calidad de datos sobre la base de datos regulatoria, añadiendo \emph{metadata de riesgo contable} que se integra directamente en el pipeline de series PIT y modelos de trading\cite{DePrado2018Advances}.

\subsection{CLI y uso operativo}

El módulo se invoca mediante la línea de comandos:

\begin{verbatim}
python -m data.edgar.filings_flags ^
  --submissions-path data/edgar/raw/submissions ^
  --facts-path data/edgar/parsed/facts_canonical_latest.parquet ^
  --config-path configs/data/edgar_filings_flags.yaml ^
  --run-id filings_flags_20260305_001
\end{verbatim}

Esto permite:
\begin{itemize}
  \item integrar el módulo en pipelines de features y backtesting,
  \item actualizar periódicamente los flags conforme llegan nuevos filings,
  \item y mantener versiones reproducibles de calidad de datos mediante \texttt{run\_id}.
\end{itemize}

\subsection{Definición de término (DoD)}

La definición de término del módulo establece que:

\begin{itemize}
  \item Los flags generados deben ser \emph{accionables} para el módulo \texttt{point\_in\_time}, es decir:
    \begin{itemize}
      \item se pueden usar para filtrar observaciones con \texttt{flag\_late\_filer=1}, \texttt{flag\_restatement=1}, etc.,
      \item o se pueden usar para ponderar features fundacionales con \texttt{quality\_score}.
    \end{itemize}
  \item Los flags también deben ser \emph{accionables} para la generación de \emph{labels} y \emph{features de riesgo} en modelos de credit risk, default risk y estrategias de swing trading.
  \item Debe existir un registro auditado de distribución de flags en \texttt{filings\_flags\_summary\_\{run\_id\}.json}, que se pueda usar para monitorear el riesgo de calidad de datos en el universo de empresas.
\end{itemize}

  \item \textbf{Accounting quality y audit risk}: los restatements se usan ampliamente en la literatura de contabilidad como medida de baja calidad de auditoría\cite{DeFondZhang20
