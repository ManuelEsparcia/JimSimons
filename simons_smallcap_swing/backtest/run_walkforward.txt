Especificación Cuantitativa e Institucional: backtest/run_walkforward.py

0. Propósito y función real del módulo

backtest/run_walkforward.py no existe para “partir el histórico en trozos” ni para producir una colección cómoda de métricas OOS aparentes. Su misión institucional es mucho más exigente: simular, con disciplina temporal estricta, el ciclo real de entrenamiento, purga, embargo, inferencia OOS y monetización secuencial de una estrategia cuantitativa. En otras palabras, debe responder a esta pregunta:

    si el sistema hubiera sido reentrenado y desplegado de forma progresiva en el tiempo,
    usando sólo la información disponible en cada fecha de decisión,
    ¿qué trayectoria OOS neta, qué heterogeneidad entre ventanas
    y qué evidencia metodológicamente defendible habría producido realmente?

Ésa es la frontera entre un backtest cómodo y una validación temporal seria.

1. Objetivo formal del módulo

Sea D el conjunto de datos PIT disponibles:

    D = {
        Features,
        Labels_H,
        Prices,
        Universe,
        Calendar
    }

y sea Config_WF la configuración del experimento walk-forward.

El módulo debe implementar una transformación:

    WF : D × Config_WF -> R_WF

donde R_WF contiene:
- resultados por ventana;
- artefactos entrenados por ventana;
- predicciones OOS por ventana;
- backtests OOS por ventana;
- trayectoria OOS agregada no solapada;
- métricas globales de estabilidad, calidad y heterogeneidad;
- manifiesto metodológico completo.

En forma estructural:

    WF =
        Aggregate(
            {
                Backtest_OOS_k(
                    Predict_OOS_k(
                        Train_k(
                            PurgeEmbargo(Window_k(D))
                        )
                    )
                )
            } for k = 1..K
        )

sujeto a tres restricciones no negociables:

    1. temporalidad causal estricta;
    2. no solapamiento OOS en el agregado final;
    3. trazabilidad completa por ventana.

2. Pregunta institucional que debe responder

Toda ejecución de backtest/run_walkforward.py debe permitir responder, sin ambigüedad, a estas preguntas:

- ¿Cuál es el rendimiento neto verdaderamente OOS de la estrategia cuando se reentrena secuencialmente?
- ¿Es ese rendimiento estable entre ventanas o depende de pocos periodos favorables?
- ¿Qué sensibilidad muestra el edge a cambios de régimen o a la fecha de entrenamiento?
- ¿La señal sobrevive a purga y embargo correctamente aplicados?
- ¿La estrategia parece robusta a la heterogeneidad temporal o colapsa fuera de ciertos subperiodos?
- ¿El esquema rolling o expanding altera materialmente la lectura del edge?
- ¿Hay evidencia de leakage o de contaminación metodológica?
- ¿Existe base suficiente para promoción a un estadio superior de validación?

Si el módulo no puede responder con claridad a estas preguntas, no está cumpliendo su papel.

3. Principios no negociables

P1. PIT estricto.
Toda observación usada para entrenar o validar debe respetar la disponibilidad real de datos.

P2. Purga antes que comodidad.
Cuando el label tenga horizonte o estructura temporal extendida, la exclusión conservadora de observaciones es obligatoria.

P3. Embargo explícito.
Embargo y purge no son sinónimos; ambos deben estar definidos, versionados y trazados.

P4. OOS real, no “pseudo-OOS”.
Sólo puede llamarse OOS a un bloque que haya sido predicho por un modelo entrenado sin acceso directo ni indirecto a su información futura.

P5. Agregación sin solape.
No puede existir doble contabilización ni reutilización temporal ambigua de segmentos OOS.

P6. Window-level accountability.
Cada ventana debe ser tratada como una unidad experimental reconstruible.

P7. Fail loudly.
Cualquier sospecha material de leakage, solape o inconsistencia debe bloquear o degradar el run de forma visible.

4. Papel del módulo dentro del sistema

4.1. Upstream

El módulo consume:
- features PIT ya generadas;
- labels con horizonte explícito o metadata equivalente;
- precios compatibles con el engine;
- universo histórico por fecha;
- calendario de sesiones;
- entrenamiento / inferencia de modelos;
- configuración del engine.

4.2. Downstream

El módulo alimenta a:
- backtest/report.py;
- backtest/diagnostics.py;
- módulos de validación estadística;
- comparadores entre estrategias o experimentos;
- decisiones de promoción o descarte.

Por tanto, run_walkforward.py no es un “helper”; es el filtro metodológico que separa un single-run optimista de una señal temporalmente defendible.

5. Estructura conceptual del walk-forward

Para cada ventana k se definen:

- bloque de entrenamiento T_k;
- bloque de validación V_k, si existe;
- bloque de test OOS S_k.

La condición estructural mínima es:

    T_k ≺ V_k ≺ S_k

tras aplicar purge y embargo.

Cada iteración debe seguir el patrón:

    Train_k
    -> Purge(Labels_H)
    -> Embargo_E
    -> Predict_OOS_k
    -> Engine_OOS_k
    -> Persist_k

y el resultado global debe surgir sólo de la concatenación ordenada y no solapada de los segmentos OOS válidos.

6. Notación temporal formal

Sea D = {t_1, ..., t_T} la secuencia ordenada de sesiones de mercado.

Para una observación anclada en t con label de horizonte H, definimos un intervalo informacional efectivo:

    I(t) = [t, t + H]

o, más generalmente, el intervalo exacto asociado a la construcción del label.

Si una observación de train tiene I(t) que solapa con S_k, entonces esa observación no puede pertenecer al entrenamiento de la ventana k.

Formalmente:

    I(t) ∩ S_k != ∅  ->  t debe ser purgado del train

Esto es el corazón metodológico del módulo.

7. Purga y embargo

7.1. Purga

La purga elimina observaciones del bloque train cuyo label o intervalo informacional invade el bloque test.

No basta con que la fecha del feature sea anterior a test_start; si el label usa información que toca test, esa observación está contaminada.

Regla general:

    si label_end_ts(t) >= min(S_k)
    entonces t no puede usarse en train_k

7.2. Embargo

Además de la purga, puede imponerse un embargo de E sesiones o E días efectivos entre el final del bloque train/valid y el inicio del bloque OOS.

Formalmente:

    min(S_k) >= max(T_k ∪ V_k) + E

El embargo no sustituye a la purga. Protege la frontera temporal; la purga protege el contenido del label.

7.3. Principio institucional

El módulo debe registrar separadamente:
- purge_days o purge_rule;
- embargo_days;
- effective_train_after_purge;
- effective_gap_to_test.

Sin esta trazabilidad, el experimento no es institucionalmente defendible.

8. Esquemas de ventana soportados

8.1. Rolling

En modo rolling, el train tiene longitud aproximadamente constante.

Si L_train es el tamaño del train y L_test el del test:

    T_k = [a_k, a_k + L_train]
    S_k = [b_k, b_k + L_test]

con b_k determinado tras purge y embargo.

Este modo favorece adaptación a regímenes recientes y control del envejecimiento de señal.

8.2. Expanding

En modo expanding, el train crece con el tiempo:

    T_k = [t_0, a_k]
    S_k = [b_k, b_k + L_test]

Se usa cuando se desea aprovechar toda la historia disponible hasta cada frontera.

8.3. Validación interna

Si existe V_k, debe definirse de forma temporalmente consistente y nunca puede contaminar S_k. Puede ser:
- una cola temporal del train;
- un split temporal interno;
- un esquema nested temporal explícito.

9. Construcción de ventanas válidas

Una ventana sólo puede ejecutarse si cumple simultáneamente:

- número mínimo de sesiones en train antes de purga;
- número mínimo de observaciones después de purga;
- bloque OOS no vacío;
- cobertura mínima del universo en train y test;
- datos suficientes para labels y engine;
- consistencia del calendario;
- ausencia de solape con ventanas OOS ya aceptadas.

Si no se cumple, la ventana no desaparece silenciosamente: debe etiquetarse como:

    OK
    SKIPPED
    INVALID
    ERROR

12. Precondiciones obligatorias

Antes de ejecutar, el módulo debe verificar:

- features, labels y universe con PK única en (date, asset_id) o equivalente;
- compatibilidad entre el universo y los precios monetizados;
- labels con metadata suficiente para purge;
- calendario ordenado y sin duplicados;
- consistencia de fechas y zonas horarias si las hubiera;
- existencia y legibilidad de configs externas;
- coherencia entre frecuencia temporal de features, labels y engine;
- ausencia de NaNs críticos en campos núcleo.

Si falla una precondición crítica, el run debe abortar.

13. Flujo operativo obligatorio

El flujo correcto del módulo debe ser:

1. Cargar config, calendario y metadata del label.
2. Construir candidatas de ventana.
3. Aplicar purge y embargo por ventana.
4. Validar elegibilidad operativa de cada ventana.
5. Entrenar modelo sobre train efectivo.
6. Validar o seleccionar hiperparámetros sólo con información permitida.
7. Generar predicciones exclusivamente sobre OOS.
8. Monetizar cada OOS mediante backtest/engine.py.
9. Persistir artefactos de ventana.
10. Calcular métricas por ventana.
11. Agregar sólo ventanas OOS válidas y no solapadas.
12. Calcular métricas globales y heterogeneidad.
13. Persistir manifiesto final y outputs globales.

18. Agregación OOS: regla institucional

El agregado global sólo puede construirse con ventanas cuyo bloque OOS cumpla:

    S_i ∩ S_j = ∅   para todo i != j

No se admite solape silencioso. Si hay solape:
- o se rechaza la agregación;
- o se resuelve con política explícita, documentada y auditable.

Nunca debe concatenarse todo “porque funciona”.

19. Política de OOS_non_overlap

El módulo debe definir una política única y explícita para OOS_non_overlap. Una convención razonable es:

- ordenar todas las filas OOS por fecha;
- verificar unicidad temporal por fila monetizable;
- descartar bloques o filas conflictivas si la política lo permite;
- construir NAV agregado por reinversión secuencial:

    NAV_t = NAV_{t-1} * (1 + r_t^OOS)

Esta convención debe quedar congelada en run_manifest.json.

21. Heterogeneidad entre ventanas

Un walk-forward útil no es sólo una curva agregada. Debe cuantificar estabilidad entre ventanas.

Si K es el número de ventanas válidas y SR_k el Sharpe de cada una:

    mean_SR = (1 / K) * sum_k SR_k

    std_SR =
        sqrt(
            sum_k (SR_k - mean_SR)^2 / (K - 1)
        )

Análogamente deben poder calcularse:
- dispersión del CAGR o retorno anualizado por ventana;
- dispersión del drawdown;
- dispersión del coste;
- dispersión del turnover.

Una estrategia con OOS agregado atractivo pero varianza extrema entre ventanas debe quedar bajo sospecha metodológica.

22. Statistical process control

Para elevar el módulo a nivel verdaderamente institucional, no basta con mirar medias y desviaciones entre ventanas. Debe incorporarse control secuencial de estabilidad.

22.1. CUSUM por ventana

Una versión mínima puede construir, sobre retornos OOS o una métrica rolling, una estadística tipo:

    CUSUM_t =
        sum_{u <= t} max(0, |X_u - mu_ref| - kappa * sigma_ref)

o una versión estándar con acumulación positiva/negativa.

Objetivo:
- detectar deriva estructural entre ventanas;
- distinguir edge estable de edge errático.

Si la estabilidad temporal es un criterio real de promoción, el módulo debe registrarla.

23. Regime awareness

Una agregación walk-forward ciega puede ocultar que ciertas ventanas pertenecen a regímenes radicalmente distintos. Por eso el módulo debería, como mínimo, dejar hueco para:

- etiquetado de régimen por ventana;
- probabilidades HMM o equivalente;
- grouping posterior de métricas por régimen.

No hace falta sobrecargar el MVP, pero sí dejar la interfaz preparada.

24. FDR y multiple testing across windows

Cuando se evalúan muchas ventanas, también existe riesgo de interpretar “ventanas buenas” como evidencia limpia cuando son parcialmente producto del azar. Por tanto, si se reportan p-values por ventana, debe existir soporte para:

- corrección FDR Benjamini-Hochberg;
- o ajuste más conservador si el contexto lo requiere.

No para convertir el módulo en un paper estadístico, sino para evitar la lectura ingenua de heterogeneidad temporal.

25. Deflated OOS Sharpe

Si el walk-forward se usa como prueba de promoción, el sistema debería poder integrar una versión deflactada del Sharpe OOS agregado, por ejemplo:

    Deflated_SR_OOS =
        SR_OOS / sqrt(1 + multiple_tests_factor)

o una variante compatible con el marco del sistema.

Esto es especialmente importante cuando el run forma parte de una familia de experimentos.

26. Combinatorial Purged CV

El módulo no tiene por qué implementar en el MVP todo CPCV, pero sí debe estar diseñado de forma compatible con una futura extensión hacia:

- generación de múltiples combinaciones válidas train/test;
- purga formal basada en label_end;
- agregación más rica que el simple rolling walk-forward.

Si la arquitectura lo impide, el diseño está demasiado estrecho.

27. Criterios de aceptación del run

Un run walk-forward no debe aceptarse simplemente porque produjo resultados. Como mínimo debe exigirse:

- leakage = 0
- ventanas válidas >= min_windows
- solape OOS = 0
- cobertura OOS suficiente
- consistencia metodológica entre ventanas
- engine errors críticos = 0 en ventanas agregadas

Y, además, conviene exigir:
- heterogeneidad temporal no patológica;
- estabilidad mínima de capacidad o sizing si se calcula;
- resultados económicos no puramente episódicos.

28. Sistema de severidades

Cada ventana y el run global deben etiquetarse con severidad:

- OK
- WARNING
- ERROR
- CRITICAL

Reglas mínimas:
- evidencia de leakage -> CRITICAL
- solape OOS no resuelto -> CRITICAL
- ventanas válidas insuficientes -> ERROR o CRITICAL
- engine failure material en ventana -> ERROR
- heterogeneidad extrema -> WARNING o ERROR según config

29. Failure modes que el módulo debe detectar

Como mínimo:
- purge incompleta para labels con horizonte;
- embargo aplicado pero insuficiente;
- OOS solapado;
- ventanas demasiado pequeñas tras purga;
- train/test construidos con calendarios inconsistentes;
- OOS monetizado sobre universo o precios no alineados;
- run agregado con configuraciones no homogéneas entre ventanas;
- pérdida silenciosa de filas OOS;
- window_status ignorado al agregar;
- demasiadas ventanas inválidas pero run presentado como robusto.

- run_id

32. MVP obligatorio

El MVP institucionalmente aceptable debe incluir:
- rolling y expanding;
- purge y embargo diferenciados;
- entrenamiento / inferencia / engine por ventana;
- persistencia por ventana;
- agregado OOS sin solape;
- métricas por ventana y globales;
- heterogeneidad básica entre ventanas;
- manifiesto final reproducible.

33. Fase 1 razonable

Como evolución de alto valor:
- CUSUM de estabilidad;
- FDR sobre métricas por ventana;
- Deflated OOS Sharpe;
- capacity decay o sizing proxy por ventana;
- grouping por régimen;
- compatibilidad con CPCV.

34. No-objetivos

No pertenece al módulo:
- redefinir el modelo entrenado;
- optimizar señales;
- sustituir diagnostics;
- inventar explicaciones económicas;
- maquillar runs con pocas ventanas válidas;
- aceptar como OOS una validación temporalmente dudosa por conveniencia.

35. Criterio de calidad institucional

backtest/run_walkforward.py sólo puede considerarse institucional si:
- impone causalidad temporal estricta;
- trata purge y embargo con rigor y trazabilidad;
- produce ventanas reconstruibles;
- agrega únicamente OOS no solapado;
- expone heterogeneidad temporal en lugar de ocultarla;
- y bloquea cualquier run con contaminación metodológica material.

36. Conclusión

backtest/run_walkforward.py debe ser el suero de la verdad temporal del sistema. Su función no es multiplicar experimentos, sino someter una estrategia a la prueba más cercana posible a su explotación real: entrenar, esperar, predecir, monetizar, repetir y juzgar sólo sobre segmentos verdaderamente OOS. En un stack como simons_smallcap_swing, donde el riesgo de single-run optimism, leakage y fragilidad temporal es enorme, este módulo debe comportarse como un árbitro metodológico severo: hostil a la ambigüedad, conservador con la causalidad y obsesionado con la trazabilidad. Sólo así el OOS agregado puede interpretarse como evidencia y no como una ilusión retrospectiva.
