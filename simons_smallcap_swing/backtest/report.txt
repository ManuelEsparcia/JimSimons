\title{Especificación Cuantitativa e Institucional del Módulo \texttt{backtest/report.py}}
\author{Sistema \texttt{simons\_smallcap\_swing}}
\date{\today}

\begin{document}

\maketitle

\begin{abstract}
Este documento define de forma operativa, cuantitativa y de ingeniería el módulo
\texttt{backtest/report.py} del sistema \texttt{simons\_smallcap\_swing}. Su misión
no es simplemente presentar métricas aisladas, sino transformar los artefactos
generados por el motor de backtest, attribution, diagnostics y, cuando aplica,
walkforward, en un informe institucionalmente útil, internamente consistente y
decisionalmente accionable. El módulo debe producir un reporte que permita evaluar,
en una sola pieza, la calidad económica, la robustez estadística, la fricción de
implementación, la estabilidad temporal y la promocionabilidad de una estrategia.
La especificación fija objetivos, narrativa obligatoria, contratos de entrada y
salida, jerarquía de secciones, métricas mínimas, reglas de comparabilidad, manejo
de benchmarks, control de NaNs y faltantes, criterios de severidad, lógica de
consistencia con el engine, artefactos persistidos, \textit{failure modes} y tests
mínimos. El principio rector es inequívoco: un buen reporte cuantitativo no es una
colección bonita de gráficos, sino una interfaz de decisión entre investigación,
riesgo y capital.
\end{abstract}

\tableofcontents

\section{Propósito institucional del módulo}

El módulo \texttt{backtest/report.py} existe para convertir un conjunto disperso de
artefactos de backtest en una narrativa cuantitativa unificada, auditable y útil
para toma de decisiones. Un equity curve aislado no basta. Un Sharpe aislado tampoco.
El reporte debe responder, con claridad y sin contradicciones internas, a la pregunta:

\begin{quote}
\emph{¿Merece esta estrategia más tiempo de investigación, promoción a validación
superior o asignación de capital, una vez considerados rendimiento neto, drawdowns,
costes, estabilidad, concentración, calidad de ejecución y consistencia interna
del run?}
\end{quote}

Por tanto, el módulo cumple simultáneamente cinco funciones:

\begin{enumerate}[label=\arabic*.]
    \item \textbf{Síntesis ejecutiva}: condensar el estado económico del run en una
    forma entendible para PMs, risk y dirección de research.
    \item \textbf{Auditoría técnica}: mostrar suficiente detalle para que quant
    researchers y quant developers puedan verificar calidad, estabilidad y anomalías.
    \item \textbf{Interfaz entre capas}: integrar de forma coherente engine,
    attribution, diagnostics y benchmarking.
    \item \textbf{Comparabilidad}: permitir comparar runs distintos sin ambigüedad
    metodológica.
    \item \textbf{Soporte a decisión}: terminar en una lectura clara sobre si el run
    es promocionable, revisable o descartable.
\end{enumerate}

\section{Principio rector}

La filosofía del módulo debe ser:

\begin{quote}
\emph{No presentar métricas en bruto sin jerarquía interpretativa; ordenar la
información de modo que el usuario vea primero lo que decide el destino del run
y después lo que explica sus matices.}
\end{quote}

Esto implica que el reporte no debe organizarse como una colección plana de tablas,
sino como una narrativa cuantitativa estructurada en capas:

\begin{itemize}
    \item primero la conclusión económica;
    \item después la robustez y los riesgos;
    \item después la explicación del resultado;
    \item finalmente el detalle técnico y la trazabilidad.
\end{itemize}

\section{Preguntas que el reporte debe responder}

Toda ejecución del módulo debe producir un informe capaz de contestar, sin ambigüedad,
a las siguientes preguntas:

\begin{itemize}
    \item ¿Cuál ha sido el retorno neto realista del run y con qué nivel de riesgo?
    \item ¿La estrategia gana de forma suficientemente estable o depende de episodios aislados?
    \item ¿Qué parte del edge se erosiona por costes y fricciones de ejecución?
    \item ¿El drawdown observado es compatible con el mandato del sistema?
    \item ¿El resultado está concentrado en pocos nombres, pocos días o pocos sectores?
    \item ¿Cómo se compara frente a benchmark, si aplica?
    \item ¿Qué dicen attribution y diagnostics sobre la credibilidad del run?
    \item ¿Debe promocionarse, revisarse o descartarse?
\end{itemize}

Si el informe no responde estas preguntas con claridad, el reporte es insuficiente
aunque contenga muchas métricas.

\section{Alcance}

\subsection{Incluido}

El módulo debe cubrir, al menos:

\begin{itemize}
    \item consolidación de métricas netas y brutas provenientes del engine;
    \item integración opcional de attribution, diagnostics y walkforward;
    \item generación de resumen ejecutivo y anexo técnico;
    \item tablas y series de performance, riesgo, costes, exposures y estabilidad;
    \item comparativa contra benchmark, si existe;
    \item breakdown por subperiodo, lado, decil, sector o sleeve, si los artefactos existen;
    \item export en formatos consumibles y persistibles.
\end{itemize}

\subsection{Excluido}

El módulo \textbf{no} debe:

\begin{itemize}
    \item recalcular el backtest desde cero;
    \item redefinir métricas inconsistentes con el engine;
    \item inferir arbitrariamente datos ausentes del run;
    \item reemplazar attribution o diagnostics;
    \item introducir reglas de negocio no documentadas en configuración.
\end{itemize}

\section{Principios de diseño}

\begin{enumerate}[label=\arabic*.]
    \item \textbf{Consistencia con el engine}: toda métrica primaria debe derivarse
    de artefactos oficiales del run.
    \item \textbf{Net-first}: la capa principal del reporte debe priorizar rendimiento
    neto, no bruto.
    \item \textbf{Jerarquía narrativa}: lo decisivo primero, lo complementario después.
    \item \textbf{Comparabilidad explícita}: nunca comparar runs con definiciones
    incompatibles sin advertencia.
    \item \textbf{Separación entre dato y juicio}: distinguir métricas observadas de
    conclusiones interpretativas.
    \item \textbf{Persistencia total}: todas las tablas relevantes deben quedar exportadas.
    \item \textbf{Fallo ruidoso}: si faltan artefactos críticos o hay inconsistencias
    materiales, el reporte debe marcarlo visiblemente.
\end{enumerate}

\section{Jerarquía del informe}

El informe institucional debe seguir esta estructura obligatoria:

\subsection{Capa I: Executive summary}

Debe contener sólo aquello que determina la primera decisión sobre el run:

\begin{itemize}
    \item retorno total neto;
    \item CAGR;
    \item volatilidad anualizada;
    \item Sharpe neto;
    \item max drawdown;
    \item Calmar;
    \item turnover medio;
    \item coste total en bps;
    \item veredicto de diagnostics si existe;
    \item una conclusión resumida sobre promoción, revisión o descarte.
\end{itemize}

\subsection{Capa II: Economic quality}

Debe explicar si el resultado es monetizable y robusto:

\begin{itemize}
    \item equity curve;
    \item drawdown curve;
    \item erosión por costes;
    \item estabilidad rolling;
    \item comparación con benchmark, si aplica.
\end{itemize}

\subsection{Capa III: Structural explanation}

Debe responder \emph{de dónde viene el resultado}:

\begin{itemize}
    \item attribution por activo/sector/factor/sleeve si existe;
    \item breakdown por lado long/short;
    \item breakdown por deciles de señal;
    \item concentración de P\&L y exposición.
\end{itemize}

\subsection{Capa IV: Technical appendix}

Debe contener el detalle para investigadores y developers:

\begin{itemize}
    \item definiciones exactas de métricas;
    \item contratos de datos;
    \item anomalías detectadas;
    \item metadata del run;
    \item manifest de reproducibilidad.
\end{itemize}

\section{Narrativa obligatoria del informe}

Un reporte institucional no debe limitarse a listar cifras. Debe seguir esta lógica narrativa:

\subsection{Paso 1: Qué ha ocurrido}

Resumen cuantitativo del run:
\begin{itemize}
    \item rentabilidad neta;
    \item riesgo principal;
    \item coste total;
    \item situación frente a benchmark;
    \item decisión preliminar.
\end{itemize}

\subsection{Paso 2: Cómo de fiable parece}

\begin{itemize}
    \item estabilidad rolling;
    \item drawdowns;
    \item concentración;
    \item dependencia temporal;
    \item severidades de diagnostics.
\end{itemize}

\subsection{Paso 3: De dónde viene el resultado}

\begin{itemize}
    \item attribution;
    \item long vs short;
    \item sectores;
    \item deciles;
    \item factores si aplica.
\end{itemize}

\subsection{Paso 4: Qué debe hacerse}

El informe debe terminar con una conclusión accionable, por ejemplo:

\begin{itemize}
    \item promocionar a walkforward más estricto;
    \item revisar costes y ejecución;
    \item investigar concentración;
    \item descartar por inestabilidad o fragilidad.
\end{itemize}

\section{Métricas obligatorias}

\subsection{Performance neta}

Sean \(V_0\) y \(V_T\) el NAV inicial y final, y \(T_y\) la duración en años del run.

\[
\text{Total Return} = \frac{V_T}{V_0} - 1
\]

\[
\text{CAGR} = \left(\frac{V_T}{V_0}\right)^{1/T_y} - 1
\]

Si \(r_t\) son retornos diarios netos:

\[
\mu_d = \frac{1}{T}\sum_{t=1}^{T} r_t,
\qquad
\sigma_d = \sqrt{\frac{1}{T-1}\sum_{t=1}^{T}(r_t-\mu_d)^2}
\]

\[
\text{Annualized Return} \approx 252\,\mu_d
\quad \text{o la convención compuesta oficial del sistema}
\]

\[
\text{Annual Vol} = \sigma_d \sqrt{252}
\]

\[
\text{Sharpe} = \frac{\mu_d \sqrt{252}}{\sigma_d}
\]

Si existe tasa libre de riesgo, debe incluirse de forma consistente y explícita.

\subsection{Downside y drawdown}

\[
\text{Sortino} = \frac{\mu_d \sqrt{252}}{\sigma_{\text{down}}}
\]

donde \(\sigma_{\text{down}}\) se calcula sólo con retornos negativos o bajo el
objetivo fijado por configuración.

Sea \(E_t\) la equity curve y \(M_t=\max_{s\le t} E_s\). Entonces:

\[
DD_t = \frac{E_t}{M_t} - 1
\]

\[
\text{Max Drawdown} = \min_t DD_t
\]

\[
\text{Calmar} = \frac{\text{CAGR}}{|\text{Max Drawdown}|}
\]

\[
\text{TUW} = \frac{1}{T}\sum_t \mathbb{I}(DD_t < 0)
\]

\subsection{Costes y ejecución}

\[
C_t^{\text{total}} =
C_t^{\text{commission}} +
C_t^{\text{fees}} +
C_t^{\text{slippage}} +
C_t^{\text{impact}} +
C_t^{\text{borrow}} +
C_t^{\text{financing}}
\]

\[
\text{Total Cost Bps}
=
10^4 \cdot
\frac{\sum_t C_t^{\text{total}}}{\sum_t \text{capital\_base}_t}
\]

La base de capital usada en bps debe definirse unívocamente en configuración:
AUM, notional negociado o gross exposure, según diseño.

\[
\text{Win Rate} = \frac{\#\{t : r_t > 0\}}{T}
\]

\[
\text{Implementation Shortfall}_t
=
\text{modeled\_return}_t - \text{executed\_return}_t
\]

si ambas series existen de forma consistente.

\subsection{Métricas de estabilidad}

El módulo debe calcular, como mínimo:

\begin{itemize}
    \item rolling return;
    \item rolling vol;
    \item rolling Sharpe;
    \item retorno por subperiodo;
    \item métricas por año y por trimestre;
    \item estabilidad por régimen si existen etiquetas.
\end{itemize}

\section{Comparabilidad entre runs}

Éste es uno de los puntos críticos para llevar el módulo a nivel institucional.
El reporte debe indicar explícitamente si dos runs son comparables. Para ello debe
verificar, al menos:

\begin{itemize}
    \item misma convención temporal del engine;
    \item misma definición de retorno;
    \item misma base de costes;
    \item misma moneda o FX treatment;
    \item benchmark equivalente o ausencia equivalente;
    \item horizonte temporal comparable;
    \item misma familia estratégica o etiqueta de experimento.
\end{itemize}

Si alguna de estas condiciones no se cumple, el reporte debe marcar la comparación
como \texttt{NON-COMPARABLE} o \texttt{COMPARABLE WITH CAVEATS}.

\section{Benchmarking}

Si se proporciona benchmark, el módulo debe producir:

\begin{itemize}
    \item retorno acumulado relativo;
    \item active return;
    \item active drawdown si aplica;
    \item exceso de Sharpe o información relativa si la definición está soportada;
    \item visualización superpuesta de equity vs benchmark.
\end{itemize}

La alineación por calendario es obligatoria. No se permite comparar series con
fechas desalineadas sin advertencia formal.

\section{Breakdowns opcionales pero recomendados}

\subsection{Por subperiodo}

El reporte debe construir tablas por:

\begin{itemize}
    \item año;
    \item trimestre;
    \item mes, si el horizonte lo justifica;
    \item régimen, si existe etiquetado externo.
\end{itemize}

\subsection{Por lado}

Si la estrategia tiene libros long y short distinguibles, el reporte debe incluir:

\begin{itemize}
    \item contribución long;
    \item contribución short;
    \item costes del lado long;
    \item costes del lado short;
    \item exposición media por lado.
\end{itemize}

\subsection{Por deciles}

Si existen scores o rankings persistidos:

\[
\text{LS Spread} = \mu_{\text{top decile}} - \mu_{\text{bottom decile}}
\]

Este bloque es útil para evaluar si el alpha monetizado preserva ordenamiento
transversal razonable.

\subsection{Por sector o sleeve}

Si los artefactos lo permiten, el reporte debe incluir tablas de contribución
agregada y estabilidad por sector o sleeve.

\section{Reglas de interpretación}

El módulo debe distinguir entre:

\begin{itemize}
    \item \textbf{métrica observada}: valor calculado;
    \item \textbf{lectura interpretativa}: qué significa para la calidad del run;
    \item \textbf{acción sugerida}: qué conviene hacer a continuación.
\end{itemize}

Por ejemplo, un Sharpe alto no debe presentarse sin contexto si coincide con:

\begin{itemize}
    \item drawdown excesivo;
    \item coste demasiado alto;
    \item concentración extrema;
    \item inestabilidad rolling;
    \item flags severos en diagnostics.
\end{itemize}

\section{Reglas de severidad del reporte}

El propio reporte debe poder etiquetar su estado:

\begin{itemize}
    \item \textbf{OK}: todos los artefactos críticos presentes y métricas coherentes.
    \item \textbf{WARNING}: faltan algunos bloques opcionales o existen pequeñas inconsistencias.
    \item \textbf{ERROR}: faltan datos clave o existen incoherencias materiales.
\end{itemize}

Además, si diagnostics existe, el reporte debe heredar su veredicto y mostrarlo
visiblemente, no esconderlo en anexo.

\section{Manejo de NaNs, faltantes y datos parciales}

El módulo debe imponer reglas explícitas:

\begin{itemize}
    \item si falta NAV o retorno diario, el informe no debe continuar silenciosamente;
    \item si faltan fills o costes detallados, deben desactivarse sólo las secciones
    dependientes, con advertencia visible;
    \item si falta benchmark, se omiten las métricas relativas sin generar error estructural;
    \item toda imputación o fallback debe quedar registrada en \texttt{manifest.json}.
\end{itemize}

\section{Failure modes que el módulo debe detectar}

Como mínimo, debe detectar y señalar:

\begin{itemize}
    \item métricas recalculadas que no coinciden con las del engine;
    \item equity curve incoherente con retornos diarios;
    \item benchmark desalineado;
    \item costes ausentes pero reclamados en secciones ejecutivas;
    \item atribución o diagnostics correspondientes a otro \texttt{run\_id};
    \item comparaciones entre runs metodológicamente incompatibles;
    \item gráficos bonitos construidos sobre series parciales o no reconciliadas.
\end{itemize}

\section{Flujo operativo del módulo}

El pipeline recomendado es:

\begin{enumerate}[label=\arabic*.]
    \item Cargar configuración y localizar artefactos del run.
    \item Verificar integridad, fechas, run ids y precondiciones.
    \item Construir tabla maestra de performance/risk/costs.
    \item Reconciliar métricas núcleo con artefactos oficiales del engine.
    \item Integrar benchmark si existe.
    \item Integrar attribution y diagnostics si existen.
    \item Construir secciones ejecutivas y técnicas.
    \item Renderizar tablas, markdown y HTML si procede.
    \item Persistir outputs y manifiesto final.
\end{enumerate}

\section{Manifest y reproducibilidad}

El fichero \texttt{manifest.json} del reporte debe contener:

\begin{itemize}
    \item \texttt{run\_id}
    \item commit hash o versión del código;
    \item lista exacta de artefactos consumidos;
    \item hashes o timestamps de datasets;
    \item configuración de secciones habilitadas;
    \item convención de métricas usada;
    \item benchmark activo y su fuente si aplica;
    \item formatos de salida generados;
    \item timestamp de generación.
\end{itemize}

\section{Tests mínimos exigibles}

El módulo debe venir acompañado, como mínimo, de:

\begin{enumerate}[label=\arabic*.]
    \item \textbf{Test de consistencia}: métricas recalculadas coinciden con el engine.
    \item \textbf{Test de benchmark}: alineación correcta de fechas y retornos relativos.
    \item \textbf{Test de drawdown}: serie sintética con drawdown conocido.
    \item \textbf{Test de CAGR/Sharpe}: serie sintética con resultados esperables.
    \item \textbf{Test de datos faltantes}: desactivación correcta de secciones opcionales.
    \item \textbf{Test de run mismatch}: rechazo si attribution o diagnostics pertenecen a otro run.
    \item \textbf{Test de comparabilidad}: flag correcto ante runs incompatibles.
    \item \textbf{Test de manifest}: persistencia íntegra de metadatos.
\end{enumerate}

\section{Criterio de calidad institucional}

El módulo puede considerarse institucional si cumple simultáneamente:

\begin{enumerate}[label=\arabic*.]
    \item presenta primero lo que decide y después lo que explica;
    \item es consistente con el engine y no reinventa la verdad del run;
    \item integra de forma visible attribution y diagnostics;
    \item distingue métricas, interpretación y acción sugerida;
    \item soporta comparabilidad seria entre runs;
    \item falla de forma visible cuando los datos no permiten una lectura fiable.
\end{enumerate}

\section{Conclusión}

\texttt{backtest/report.py} debe ser la capa que convierte un backtest en una pieza
de comunicación cuantitativa institucional. No basta con mostrar rendimiento; hay que
mostrar si ese rendimiento es neto, estable, interpretable, comparable y promocionable.
En un sistema serio de swing trading cuantitativo sobre small caps, el reporte no es
un decorado al final del pipeline, sino el puente entre investigación técnica y
decisión de capital. Por eso debe ser claro para un PM, riguroso para un researcher,
inconsistente con nada del engine y hostil a toda ambigüedad metodológica.
