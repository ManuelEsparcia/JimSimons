Modulo: ops/data_health.py

1) Objetivo
Monitorear salud de datos diarios en todas las capas (data/features/labels/predictions) para detectar degradaciones tempranas que afecten modelo, backtest o ejecucion.

2) Alcance
Incluye:
- checks de freshness, completitud, schema drift y outliers.
- score de salud por dataset y score global diario.
- alertas por severidad y bloqueo de pipeline segun policy.

No incluye:
- correccion automatica compleja de datos.
- model drift (cubierto por drift_detection).

3) Dependencias
Upstream:
- outputs diarios de data pipeline y feature/label stores
- metadata de schemas esperados
- thresholds de calidad

Downstream:
- ops/daily_pipeline gate
- ops/monitoring alertas
- validation quick checks

4) Contrato de entrada
Entradas minimas:
- data_health_config_path
- run_id
- execution_date

Fuentes monitoreadas (ejemplo):
- data/price/*
- data/universe/*
- data/edgar/*
- data/borrow/*
- features/store
- labels/store
- predictions/*

5) Contrato de salida
- ops/data_health/{run_id}/data_health_summary.json
- ops/data_health/{run_id}/dataset_health.parquet
- ops/data_health/{run_id}/health_alerts.parquet
- ops/data_health/{run_id}/manifest.json

Campos minimos dataset_health:
- dataset_name
- freshness_status
- completeness_rate
- schema_status
- anomaly_score
- health_status (pass/warn/fail)
- run_id

6) Checks obligatorios
Freshness:
- ultimo timestamp >= expected_cutoff.

Completeness:
- volumen de filas vs baseline.
- cobertura por fecha/simbolo/feature.

Schema:
- columnas requeridas presentes.
- tipos esperados.
- columnas nuevas inesperadas flagged.

Anomalias:
- spikes en valores clave.
- pct_missing inusual.
- distribucion fuera de banda historica.

7) Score de salud sugerido
health_score_dataset = w1*freshness + w2*completeness + w3*schema + w4*anomaly
health_score_global = agregacion ponderada por criticidad de dataset.

8) Gates y severidad
- pass: score y checks criticos dentro de umbral.
- warn: degradacion moderada.
- fail: freshness/schema critico roto o score muy bajo.

9) Flujo del proceso
1. Cargar config y catalogo de datasets.
2. Ejecutar checks por dataset.
3. Calcular score por dataset y global.
4. Generar alertas por severidad.
5. Persistir resumen y hallazgos.

10) Manejo de casos especiales
- dataset opcional ausente:
  - warning no bloqueante.
- evento calendario (holiday parcial):
  - ajustar expectativa de volumen.
- backfill activo:
  - modo tolerante configurable.

11) Observabilidad
Logging:
- INFO: health checks por dataset.
- WARNING: degradacion moderada.
- ERROR: fail critico de salud de datos.

Metricas:
- n_datasets_checked
- n_pass/n_warn/n_fail
- health_score_global
- critical_fail_count
- runtime_sec

12) Interfaz de ejecucion (CLI sugerida)
python -m ops.data_health ^
  --config-path configs/ops/data_health.yaml ^
  --execution-date 2026-03-05 ^
  --run-id data_health_20260305_001

13) Estructura interna recomendada (funciones)
- load_data_health_config(...)
- discover_datasets(...)
- check_dataset_freshness(...)
- check_dataset_completeness(...)
- check_dataset_schema(...)
- check_dataset_anomalies(...)
- compute_health_scores(...)
- build_health_alerts(...)
- persist_data_health_outputs(...)
- main()

14) Criterios de aceptacion (Definition of Done)
- visibilidad diaria de salud de datos.
- gates accionables para bloquear etapas riesgosas.
- alertas claras por severidad.

15) Casos de prueba minimos (para tests/)
- freshness fail en dataset critico.
- schema drift detectado.
- caida de cobertura dispara warn/fail.
- score global agregado correctamente.

16) Riesgos y decisiones abiertas
- calibracion de umbrales por dataset.
- sensibilidad a eventos de mercado extremos.
- tradeoff ruido vs deteccion temprana.

17) Checklist rapido de implementacion
- [ ] checks freshness/completeness/schema/anomaly.
- [ ] score dataset/global.
- [ ] alertas y gates.
- [ ] salida diaria versionada.
- [ ] tests de fail criticos.
