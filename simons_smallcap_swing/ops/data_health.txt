Especificación Cuantitativa: ops/data_health.py

0. Propósito y papel dentro del sistema

ops/data_health.py es la capa de gobernanza operativa que decide si los datasets que alimentan el pipeline diario son aptos para ser consumidos por etapas posteriores. Su función no es corregir datos, imputarlos silenciosamente ni “arreglar” una fuente defectuosa, sino medir su estado contra contratos explícitos, clasificar incidencias, producir una decisión operacional PASS/WARN/FAIL y dejar una traza auditable de por qué esa decisión se tomó. El documento base que has subido ya fija muy bien ese espíritu: separar score continuo de gate discreto, distinguir checks por tipo, mantener hard-fail dominance y permitir integración limpia con el daily pipeline. :contentReference[oaicite:1]{index=1}

En un stack cuantitativo serio, este módulo no es un “extra de observabilidad”, sino una barrera de protección metodológica y operativa. Si falla, el sistema puede:
- entrenar sobre features corruptas;
- construir señales con universos incompletos;
- inferir sobre labels o precios atrasados;
- o ejecutar una pipeline entera apoyándose en inputs contaminados.

La regla institucional debe ser esta:

    ningún dataset
    debe ser consumido por etapas sensibles
    si antes no ha pasado
    un contrato de salud de datos
    explícito, reproducible y auditable.

1. Objetivo formal del módulo

Sea d un dataset y c_d su contrato asociado. El módulo debe implementar una función de calidad:

    Q(d, c_d) -> (score_d, gate_d, audit_d)

donde:
- score_d ∈ [0,1] es una medida continua agregada de conformidad;
- gate_d ∈ {PASS, WARN, FAIL} es la decisión operacional discreta;
- audit_d es el conjunto estructurado de checks, métricas, umbrales y razones.

El objetivo no es maximizar el score, sino garantizar que:
- los checks críticos bloquean el consumo si fallan;
- la decisión es determinista;
- la agregación global preserva criticidad;
- y la justificación de cada gate queda persistida.

Formalmente, cada dataset d se evalúa contra un conjunto de checks:

    h_d = {h_{d,1}, ..., h_{d,K_d}}

con resultados individuales en forma de métricas, estados y severidades.

2. Principios no negociables

P1. Contract-first.
Todo dataset debe evaluarse contra un contrato formal; sin contrato válido no puede haber PASS.

P2. Hard-fail dominance.
Un fallo crítico no puede quedar oculto por un score agregado alto.

P3. Determinismo.
Mismo dataset, misma config y misma fecha lógica deben producir exactamente el mismo resultado.

P4. Auditabilidad.
Cada PASS/WARN/FAIL debe poder trazarse al subconjunto exacto de checks que lo causan.

P5. No corrección silenciosa.
El módulo mide y clasifica; no arregla inputs sin dejar rastro.

P6. Separación score/gate.
El score sirve para observabilidad; el gate sirve para control operacional.

P7. Contrato por tipo de dataset.
No todos los datasets comparten los mismos checks ni la misma criticidad.

3. Ámbito y no-objetivos

El módulo cubre:
- validación de contratos de dataset;
- disponibilidad;
- freshness;
- completitud;
- schema validation;
- consistencia temporal;
- plausibilidad estadística;
- drift distribucional;
- agregación dataset-level;
- agregación global;
- overrides auditados;
- persistencia de resultados;
- eventos de monitoring.

Queda fuera de alcance:
- imputación de datos;
- feature engineering correctivo;
- label construction;
- model training;
- strategy evaluation;
- portfolio optimization;
- anomaly explanation profunda;
- causal discovery.

La misión aquí es quantitative ops governance, no research.

4. Unidad básica: Dataset Contract

Todo dataset supervisado por data_health debe estar descrito por un contrato mínimo. Una estructura razonable es:

~~~python
@dataclass
class DatasetContract:
    name: str
    dataset_type: str
    path_or_table: str
    criticality: float
    expected_schema: dict
    freshness_sla_hours: float
    required_checks: list[str]
    gating_policy: dict
~~~

Mínimos conceptuales:
- name: identificador único;
- dataset_type: price, universe, features, labels, predictions, borrow, etc.;
- path_or_table: localización lógica o física;
- criticality: peso relativo en la agregación global;
- expected_schema: columnas, tipos, claves y restricciones básicas;
- freshness_sla_hours o equivalente;
- required_checks: checks obligatorios;
- gating_policy: reglas de hard-fail, thresholds y overrides.

Un dataset sin contrato válido no debe evaluarse como PASS. Debe producir:
- FAIL contractual;
- o ERROR de configuración;
- pero nunca conformidad implícita.

5. Taxonomía de checks

El módulo debe separar checks por naturaleza. Mezclarlos indiscriminadamente degrada interpretabilidad y acción correctiva.

5.1. Availability

Pregunta:

    ¿el dataset existe y puede leerse?

Checks típicos:
- recurso presente;
- permisos de lectura;
- tamaño no nulo;
- formato parseable;
- latencia de acceso dentro de umbral.

5.2. Freshness

Pregunta:

    ¿el dataset está actualizado dentro de la ventana contractual?

Definimos:

    Δt_d = t_eval - t_last_update,d

y el check es conforme si:

    Δt_d <= τ_d

donde τ_d depende del contrato del dataset y no debe fijarse globalmente sin contexto.

5.3. Completeness

Pregunta:

    ¿la cobertura observada coincide con la cobertura esperada?

En un panel completo ideal:

    C_d = N_obs / N_exp

con:

    N_exp = |U_d| * |T_d|

pero el módulo debe permitir contratos no cartesianos y no asumir panel completo cuando no corresponde.

5.4. Structural consistency

Pregunta:

    ¿el dataset respeta el schema pactado?

Checks mínimos:
- columnas requeridas presentes;
- dtypes compatibles;
- claves únicas;
- ausencia de columnas prohibidas si aplica;
- dominios lógicos básicos.

5.5. Temporal consistency

Pregunta:

    ¿la estructura temporal del dataset es coherente con su contrato?

Checks típicos:
- monotonicidad de timestamps;
- ausencia de fechas futuras indebidas;
- no duplicados temporales inconsistentes;
- huecos explicables por calendario;
- coherencia con sesiones de mercado si aplica.

5.6. Plausibility

Pregunta:

    ¿los valores observados son plausibles para este tipo de dato?

Ejemplos:
- precios > 0;
- volúmenes >= 0;
- retornos dentro de rangos razonables;
- features no explosivas;
- probas en [0,1] si aplica.

5.7. Drift

Pregunta:

    ¿la distribución actual se desvía materialmente de la referencia?

Esto no implica corrupción por sí mismo, pero sí alerta operacional. El contrato debe definir:
- referencia;
- ventana;
- estadístico;
- threshold;
- y efecto sobre el gate.

6. Modelo de scoring

6.1. Score por check

Cada check puede producir una puntuación:

    h_{d,k} ∈ [0,1]

donde:
- 1 = conformidad total;
- 0 = fallo máximo;
- valores intermedios = degradación graduada si el check lo soporta.

6.2. Score por dataset

Un score agregado por dataset puede definirse como:

    H_d = Σ_k w_{d,k} h_{d,k}
    con Σ_k w_{d,k} = 1

Pero H_d no debe tener prioridad sobre los hard-fails.

6.3. Gate por dataset

La regla contractual recomendada es:

    gate_d = FAIL   si existe hard_fail_k
    gate_d = WARN   si no hay hard_fail y H_d < θ_warn,d
    gate_d = PASS   en otro caso

También puede descomponerse en:
- blocking_fail
- warning_region
- pass_region

pero siempre manteniendo la dominancia de fallos críticos.

7. Separación score vs gate

Esta separación es crucial y debe quedar explícita:

- score:
  indicador continuo útil para dashboards, tendencia y priorización.

- gate:
  decisión discreta usada por el pipeline para continuar, degradar o abortar.

No es aceptable que el pipeline decida solo por H_d si existen checks críticos que pueden invalidar el dataset aunque el promedio sea alto.

Ejemplo correcto:
- prices con freshness perfecta y completitud perfecta, pero schema roto en una columna crítica -> FAIL
- no WARN, no score alto “compensa”, no degradación automática.

8. Agregación global

Sea D = {d_1, ..., d_m} el conjunto de datasets evaluados. Una agregación global razonable del score es:

    H_global = (Σ_d c_d H_d) / (Σ_d c_d)

donde c_d representa criticidad relativa.

Pero la decisión global debe seguir la política:

    global_gate = FAIL
        si existe dataset crítico con gate_d = FAIL

    global_gate = WARN
        si no hay FAIL crítico y existe al menos un dataset relevante en WARN
        o un FAIL no bloqueante degradado

    global_gate = PASS
        en otro caso

Esto evita que datasets secundarios contaminen la operación principal o que datasets críticos queden diluidos por promedios.

9. Estados operativos

A nivel dataset, el resultado debe pertenecer exactamente a uno de:

- PASS
- WARN
- FAIL
- ERROR
- SKIPPED

Semántica:
- PASS: dataset consumible sin reservas operativas.
- WARN: consumible bajo alerta o degradación aceptada.
- FAIL: no consumible bajo la política actual.
- ERROR: la evaluación misma falló.
- SKIPPED: el contrato no aplicaba a esa ejecución.

A nivel global del run de salud:
- SUCCESS
- PARTIAL
- FAILED

SUCCESS:
todos los datasets relevantes fueron evaluados y el gate global no bloquea.

PARTIAL:
la evaluación terminó con resultados parciales válidos bajo política explícita.

FAILED:
la evaluación no pudo completarse o generó gate global bloqueante.

10. Robustez estadística recomendada

Aunque este módulo es ops y no research, debe evitar estimadores frágiles cuando se trate de datos financieros. La regla recomendable es:

- por defecto, preferir estimadores robustos en plausibility checks;
- usar mean/std solo cuando el contrato lo justifique.

Un z-score robusto puede definirse como:

    z_i = (x_i - median(x)) / (1.4826 * MAD(x))

Implementación razonable:

~~~python
import numpy as np

def robust_zscore(x: np.ndarray) -> np.ndarray:
    x = np.asarray(x, dtype=float)
    med = np.nanmedian(x)
    mad = np.nanmedian(np.abs(x - med))
    scale = 1.4826 * mad if mad > 0 else 1.0
    return np.abs((x - med) / scale)
~~~

Esto protege mucho mejor frente a colas pesadas y outliers que un mean/std ingenuo.

11. Checks mínimos recomendados

11.1. Availability

~~~python
def check_availability(df, contract):
    if df is None:
        return fail_check("availability", hard_fail=True, message="dataset missing")
    if len(df) == 0:
        return fail_check("availability", hard_fail=True, message="dataset empty")
    return pass_check("availability", metric=1.0)
~~~

11.2. Freshness

~~~python
def check_freshness(last_update_ts, eval_ts, sla_hours):
    delta_h = (eval_ts - last_update_ts).total_seconds() / 3600.0
    if delta_h > sla_hours:
        return fail_check("freshness", metric=delta_h, threshold=sla_hours, hard_fail=False)
    return pass_check("freshness", metric=delta_h, threshold=sla_hours)
~~~

11.3. Completeness

~~~python
def check_completeness(n_obs, n_exp, min_coverage):
    coverage = n_obs / n_exp if n_exp > 0 else 0.0
    if coverage < min_coverage:
        return fail_check("completeness", metric=coverage, threshold=min_coverage, hard_fail=False)
    return pass_check("completeness", metric=coverage, threshold=min_coverage)
~~~

11.4. Schema

~~~python
def check_schema(df, contract):
    required = set(contract.expected_schema["required_columns"])
    observed = set(df.columns)
    missing = required - observed
    if missing:
        return fail_check("schema", observed=sorted(missing), hard_fail=True, message="missing columns")
    return pass_check("schema", metric=1.0)
~~~

11.5. Plausibility

~~~python
def check_positive_prices(df, price_col="close"):
    if (df[price_col] <= 0).any():
        return fail_check("plausibility_price_positive", hard_fail=True, message="non-positive prices")
    return pass_check("plausibility_price_positive", metric=1.0)
~~~

11.6. Drift

~~~python
def check_mean_shift(current_mean, ref_mean, max_shift):
    shift = abs(current_mean - ref_mean)
    if shift > max_shift:
        return warn_check("drift_mean_shift", metric=shift, threshold=max_shift)
    return pass_check("drift_mean_shift", metric=shift, threshold=max_shift)
~~~

12. Power and insufficient sample

Un error clásico en data health es declarar PASS sobre checks estadísticos con muestra ridículamente pequeña. Por eso el módulo debe poder producir algo como:

- status = WARN o ERROR
- reason = insufficient_sample

cuando el tamaño de muestra no permite interpretar el check con sentido.

No debe hacer:
- false PASS por σ=0;
- false PASS por n muy pequeño;
- thresholds estadísticos interpretados sin soporte muestral.

13. Overrides

Un override no debe borrar el fallo original. Debe persistirse como excepción auditada. Información mínima:

- dataset_name
- check_name
- original_status
- overridden_status
- reason
- actor
- created_at
- expires_at

Ejemplo:

~~~python
override = {
    "dataset_name": "features_daily",
    "check_name": "drift_mean_shift",
    "original_status": "WARN",
    "overridden_status": "PASS",
    "reason": "known rebalance universe change",
    "actor": "manolo",
    "created_at": "2026-03-08T08:00:00Z",
    "expires_at": "2026-03-15T08:00:00Z",
}
~~~

Un override expirado no debe aplicarse. Un override sin actor o razón tampoco.

15. Integración con daily_pipeline

daily_pipeline.py no debe leer logs libres para decidir. Debe consumir un output estructurado de data_health.py y, como mínimo:
- inspeccionar global_gate;
- obtener la lista de datasets bloqueantes si existe FAIL;
- propagar esa razón a su propio audit trail.

Semántica operativa recomendada:
- global_gate = FAIL -> abortar o no lanzar etapas downstream críticas;
- global_gate = WARN -> continuar bajo policy de degradación;
- global_gate = PASS -> continuar normalmente.

16. Pseudocódigo de referencia

~~~python
def main(config_path: str, run_id: str, execution_date: str):
    cfg = load_and_validate_config(config_path)
    contracts = load_dataset_contracts(cfg.catalog)

    validate_contracts(contracts)

    started_at = now_utc()
    dataset_results = []

    for contract in contracts:
        if not applies_to_execution(contract, execution_date, cfg):
            dataset_results.append(
                DatasetHealthResult(
                    dataset_name=contract.name,
                    dataset_type=contract.dataset_type,
                    criticality=contract.criticality,
                    status="SKIPPED",
                    score=None,
                    hard_fail=False,
                    checks=[],
                    observed_at=now_utc(),
                    details={"reason": "not_applicable"},
                )
            )
            continue

        try:
            df = load_dataset(contract)
            checks = []

            checks.append(check_availability(df, contract))
            checks.append(check_schema(df, contract))
            checks.append(check_freshness(extract_last_update(df), parse_ts(execution_date), contract.freshness_sla_hours))
            checks.append(check_completeness(len(df), expected_rows(contract, execution_date), contract.gating_policy["min_coverage"]))

            if contract.dataset_type == "price":
                checks.append(check_positive_prices(df, "close"))

            checks.extend(run_statistical_checks(df, contract))

            score = compute_dataset_score(checks, contract)
            status = derive_dataset_gate(checks, contract)
            hard_fail = any(c.hard_fail for c in checks)

            dataset_results.append(
                DatasetHealthResult(
                    dataset_name=contract.name,
                    dataset_type=contract.dataset_type,
                    criticality=contract.criticality,
                    status=status,
                    score=score,
                    hard_fail=hard_fail,
                    checks=checks,
                    observed_at=now_utc(),
                    details={},
                )
            )

        except Exception as exc:
            dataset_results.append(
                DatasetHealthResult(
                    dataset_name=contract.name,
                    dataset_type=contract.dataset_type,
                    criticality=contract.criticality,
                    status="ERROR",
                    score=None,
                    hard_fail=True,
                    checks=[],
                    observed_at=now_utc(),
                    details={"error_type": type(exc).__name__, "message": str(exc)},
                )
            )

    overrides = load_active_overrides(run_id, execution_date)
    final_results = apply_overrides(dataset_results, overrides)

    global_score = compute_global_score(final_results)
    global_gate = derive_global_gate(final_results)

    result = DataHealthRunResult(
        run_id=run_id,
        execution_date=execution_date,
        global_gate=global_gate,
        global_score=global_score,
        datasets=final_results,
        overrides_applied=overrides,
        started_at=started_at,
        ended_at=now_utc(),
    )

    persist_data_health_result(result)
    emit_data_health_events(result)

    return result
~~~

17. Casos límite que el módulo debe resolver

17.1. Dataset vacío pero existente
Debe contarse como FAIL de disponibilidad o completitud, no como PASS.

17.2. Gap explicado por festivo
No debe producir FAIL si el contrato y el calendario lo explican.

17.3. Cambio de esquema compatible
Puede ser WARN si la política lo permite, no necesariamente FAIL.

17.4. Drift elevado en dataset no crítico
Debe poder quedarse en WARN y no bloquear globalmente.

17.5. Muestra insuficiente
Debe producir insufficient_sample o WARN, no PASS artificial.

17.6. Dataset presente pero corrupto
Debe producir ERROR o FAIL estructural, nunca degradarse a score parcial sin trazabilidad.

17.7. Override expirado
No debe aplicarse aunque exista en storage.

18. Failure modes críticos que debe detectar o evitar

F1. Hard-fail oculto por score agregado.
F2. Dataset sin contrato tratado como válido.
F3. Freshness evaluado sin contexto de SLA contractual.
F4. Completeness calculado contra una malla esperada incorrecta.
F5. Schema drift mezclado con drift estadístico.
F6. Plausibility check frágil por mean/std en presencia de outliers.
F7. Overrides silenciosos sin actor ni expiración.
F8. Global gate incoherente con criticidad de datasets.
F9. Muestras pequeñas aprobadas sin warning.
F10. Persistencia incompleta del audit trail.

19. MVP obligatorio

El MVP del módulo debe incluir:
- DatasetContract explícito;
- checks de availability, freshness, completeness y schema;
- al menos un plausibility check por tipo crítico de dataset;
- score por dataset;
- gate por dataset con hard-fail dominance;
- agregación global por criticidad;
- persistencia de CheckResult y DatasetHealthResult;
- overrides auditados;
- integración limpia con daily_pipeline.

20. Segunda fase razonable

Como extensiones útiles:
- robust z-score por defecto en plausibility;
- drift estadístico más rico;
- multivariate drift para features;
- contract-aware gap detection con calendario;
- power analysis por check;
- reglas de schema evolution;
- detección secuencial de drift en runs repetidos.

21. No-objetivos

No pertenece al alcance de este módulo:
- imputar o arreglar datos automáticamente;
- hacer feature engineering;
- optimizar la estrategia;
- explicar causalmente los fallos;
- hacer monitoring predictivo sofisticado;
- resolver research questions.

La misión aquí es convertir “parece que los datos están bien” en un sistema de contratos, checks, scores y gates operacionalmente fiables.

22. Criterios de aceptación

El módulo solo debe considerarse correcto si cumple simultáneamente:

- todo dataset relevante tiene contrato explícito;
- los checks críticos pueden bloquear independientemente del score;
- freshness depende del contrato y no de una heurística global;
- completitud se calcula contra el esperado real del contrato;
- schema y drift se evalúan por separado;
- los resultados son persistidos con detalle suficiente;
- la agregación global respeta criticidad;
- los overrides quedan auditados y expiran;
- la salida es consumible inequívocamente por el pipeline diario.

23. Conclusión

ops/data_health.py debe entenderse como la capa formal de gobernanza cuantitativa de inputs. Su valor no está en producir dashboards bonitos ni en “poner notas” a los datos, sino en imponer una disciplina contractual que proteja el pipeline frente a inputs incompletos, atrasados, rotos o estadísticamente sospechosos. En un framework cuantitativo serio, esa capa no es opcional: es la condición necesaria para que cualquier etapa downstream opere sobre una base verificable y no sobre una suposición optimista de que “todo parece bien”. Ese es exactamente el estándar que este módulo debe imponer.
