Modulo: ops/daily_pipeline.py

1) Objetivo
Orquestar la ejecucion diaria end-to-end del sistema cuantitativo (datos -> features -> labels -> inference -> portfolio -> backtest/controles), con tolerancia a fallos y trazabilidad completa.

2) Alcance
Incluye:
- planificacion y ejecucion de jobs por dependencias.
- control de prechecks/postchecks por etapa.
- gestion de run_id, manifests y estados de pipeline.
- retry policy, fail-fast configurable y alertas.
- registro de tiempos/SLA y artefactos diarios.

No incluye:
- logica de negocio interna de cada modulo.
- orquestacion externa en cloud (si existe, este modulo actua como runner).

3) Dependencias aguas arriba y aguas abajo
Upstream:
- configuraciones globales ops.
- calendario de mercado y ventanas operativas.
- credenciales y fuentes de datos disponibles.

Downstream:
- ops/monitoring.py
- ops/data_health.py
- ops/drift_detection.py
- reporting diario de estado del sistema

4) DAG logico diario recomendado
1. data_universe + data_price + data_edgar + data_borrow
2. qc_data (price/universe/edgar/borrow)
3. features_build + feature_qc
4. labels_build + label_qc + splits
5. train/inference (segun modo daily/retrain)
6. portfolio (sizing/constraints/rebalance)
7. execution sim + cost/impact
8. risk + validation quick gates
9. reporting + publish artifacts

5) Contrato de entrada
Entradas minimas:
- daily_pipeline_config_path
- execution_date
- run_mode (daily|retrain|backfill)
- run_id

Opcionales:
- selective_steps (subset de etapas)
- force_rerun_steps
- dry_run

Parametros clave:
- fail_fast
- max_retries_per_step
- retry_backoff_sec
- step_timeouts
- required_gates
- sla_thresholds

6) Contrato de salida
- ops/runs/{run_id}/pipeline_state.json
- ops/runs/{run_id}/step_status.parquet
- ops/runs/{run_id}/pipeline_summary.json
- ops/runs/{run_id}/pipeline_logs/*.log
- ops/runs/{run_id}/manifest.json

Campos minimos step_status:
- step_name
- status (pending/running/success/warn/fail/skipped)
- start_ts
- end_ts
- duration_sec
- retry_count
- error_code (nullable)
- output_artifact_ref

7) Reglas operativas y de seguridad
- no avanzar a etapas dependientes si gate critico previo falla.
- reproducibilidad: cada paso debe escribir run_id y config_hash.
- idempotencia: re-ejecucion controlada sin corromper snapshots.
- bloquear publicacion final si validation_suite diaria falla.

8) Flujo del proceso (alto nivel)
1. Resolver plan de etapas segun run_mode.
2. Ejecutar preflight checks (inputs, conectividad, espacio, config).
3. Correr DAG por topologia con retries.
4. Capturar estado y artefactos de cada etapa.
5. Ejecutar postchecks de coherencia global.
6. Emitir resumen final + alertas.

9) Validaciones minimas
- todas las etapas requeridas completadas o justificadamente skipped.
- gates criticos en pass (o override auditado).
- artefactos obligatorios presentes.
- tiempos dentro de SLA (o warning).

10) Manejo de fallos y recuperacion
- fallos transitorios: retry con backoff.
- fallos permanentes: marcar fail y detener ramas dependientes.
- recovery mode:
  - resume_from_last_successful_step.
  - rerun_step_with_cleanup policy configurable.

11) Observabilidad y trazabilidad
Logging:
- INFO: inicio/fin de cada step.
- WARNING: retries, SLA violados, gates borderline.
- ERROR: fallo de step critico.

Metricas operativas:
- pipeline_success_flag
- total_duration_sec
- n_steps_success/warn/fail
- n_retries_total
- sla_breach_count

12) Interfaz de ejecucion (CLI sugerida)
python -m ops.daily_pipeline ^
  --config-path configs/ops/daily_pipeline.yaml ^
  --execution-date 2026-03-05 ^
  --run-mode daily ^
  --run-id daily_pipeline_20260305_001

13) Estructura interna recomendada (funciones)
- load_pipeline_config(...)
- build_execution_plan(...)
- run_preflight_checks(...)
- execute_step(step, context)
- handle_step_retry(...)
- evaluate_step_gate(...)
- persist_step_state(...)
- run_postflight_checks(...)
- finalize_pipeline_summary(...)
- main()

14) Criterios de aceptacion (Definition of Done)
- pipeline diario reproducible y auditable.
- recuperacion de fallos funcional.
- integracion completa con monitoring/alerts.

15) Casos de prueba minimos (para tests/)
- ejecucion feliz end-to-end en modo dry_run.
- fallo transitorio con retry exitoso.
- fallo critico que bloquea dependencias.
- resume_from_last_successful_step funcional.

16) Riesgos y decisiones abiertas
- granularidad de steps vs complejidad operativa.
- politicas de override en gates criticos.
- estrategia de paralelizacion en entornos limitados.

17) Checklist rapido de implementacion
- [ ] DAG y dependencias de etapas.
- [ ] estados + retries + timeouts.
- [ ] gates criticos y bloqueos.
- [ ] resumen final y manifests.
- [ ] tests de fallo/recuperacion.
