Especificación Cuantitativa: ops/daily_pipeline.py

0. Propósito y papel dentro del sistema

ops/daily_pipeline.py es el orquestador operativo diario del sistema cuantitativo. Su función no es calcular señales, construir features, entrenar modelos ni optimizar carteras, sino ejecutar de forma determinista, segura, idempotente y auditable el flujo completo de producción para una fecha lógica dada. Es, en términos prácticos, la pieza que transforma un conjunto de módulos de negocio en una realidad operativa diaria reproducible.

Su misión es garantizar simultáneamente:
- orden correcto de ejecución;
- respeto de dependencias;
- persistencia consistente del estado;
- reanudación segura tras fallos parciales;
- retry disciplinado ante errores transitorios;
- observabilidad completa;
- y separación estricta entre lógica de negocio y control operacional.

La regla institucional del módulo debe ser esta:

    mismo input lógico,
    misma configuración efectiva,
    misma versión de código,
    mismo estado upstream válido
    => mismo plan de ejecución
    y mismos artefactos lógicos resultantes.

1. Objetivo formal del módulo

El pipeline se modela como un DAG:

    G = (V, E)

donde:
- V es el conjunto de etapas operativas;
- E es el conjunto de dependencias de precedencia.

Una ejecución diaria para fecha tau debe producir un run r tal que:
- cada etapa se ejecute solo cuando todas sus dependencias requeridas hayan concluido en estado compatible;
- las transiciones de estado sean persistidas de forma atómica;
- los artefactos se emitan una sola vez por step_key lógico;
- el run pueda reanudarse desde un prefijo consistente del estado.

La abstracción principal del módulo puede escribirse como:

    Execution : (tau, config, code_version, upstream_state) -> RunRecord

con propiedades adicionales de:
- determinismo;
- idempotencia;
- auditabilidad;
- reanudación.

2. Principios no negociables

P1. Determinismo operacional.
El plan y la semántica de ejecución no deben depender del azar ni del orden accidental de iteración.

P2. Idempotencia fuerte.
Reejecutar el mismo run o la misma etapa no debe duplicar artefactos ni corromper estado.

P3. Persistencia atómica.
El estado nunca debe reflejar “éxito” si los artefactos no han quedado correctamente persistidos y verificados.

P4. Reanudación segura.
Un run interrumpido debe poder reconciliarse y continuar sin recomputación ciega.

P5. Separación de responsabilidades.
El runner orquesta; las etapas computan; el almacenamiento persiste; la monitorización observa.

P6. Observabilidad completa.
Toda transición importante debe generar un evento estructurado.

P7. Fail-fast configurable.
Los fallos críticos deben abortar el pipeline si así lo exige la política.

3. Ámbito y no-objetivos

El módulo cubre:
- construcción y validación del DAG;
- topological scheduling;
- control del ciclo de vida del run;
- control del ciclo de vida de las etapas;
- idempotent step keys;
- retry policy;
- preflight y postflight;
- persistencia de estado;
- integración con monitoring;
- reanudación;
- ejecución parcial controlada;
- paralelización segura bajo restricciones.

Queda fuera de alcance:
- cálculo de features;
- entrenamiento de modelos;
- lógica financiera;
- portfolio optimization;
- alerting final concreto;
- deployment de modelos;
- strategy research.

La misión aquí es infraestructura operativa robusta.

4. Contrato funcional mínimo

4.1. Entradas obligatorias

El módulo debe recibir, como mínimo:
- config_path
- execution_date
- run_mode

Parámetros opcionales razonables:
- run_id
- fail_fast
- max_retries
- resume
- selected_steps

4.2. Salidas obligatorias

Toda ejecución debe producir:
- un RunRecord persistido;
- StepRecords persistidos;
- un resumen final de ejecución;
- eventos de monitorización;
- y, si procede, referencias a artefactos producidos.

Incluso en caso de fallo, debe quedar un cierre explícito del run con:
- estado final;
- etapa de fallo;
- causa;
- intentos consumidos;
- artefactos válidos preservados.

5. Modelo del DAG y validación del plan

El DAG debe validarse antes de ejecutar nada. Las comprobaciones mínimas son:

1. aciclicidad:
   no puede existir ningún ciclo en G.

2. unicidad de nombre de etapa:
   cada nodo debe tener un nombre estable y único.

3. validez de dependencias:
   toda dependencia debe referirse a una etapa existente.

4. cierre transitivo sobre selected_steps:
   si el usuario pide un subconjunto, el plan debe incluir todas sus dependencias necesarias.

5. existencia de implementación:
   cada etapa del plan debe tener una función o runner asociado.

Un orden topológico válido es una permutación (v_1, ..., v_n) tal que:

    (v_i, v_j) in E  =>  i < j

Esto no obliga a ejecución serial completa; solo fija precedencia.

6. Modelo de estado del run

El run debe pertenecer exactamente a uno de los estados:

- created
- running
- success
- failed
- aborted
- partial

Semántica mínima:
- created: run persistido pero aún no arrancado;
- running: ejecución activa;
- success: todas las etapas requeridas terminaron correctamente;
- failed: una o más etapas críticas fallaron sin recuperación válida;
- aborted: ejecución detenida deliberadamente o por preflight;
- partial: ejecución cerrada válidamente sobre un subconjunto permitido.

7. Modelo de estado de etapa

Cada etapa debe pertenecer exactamente a uno de:

- pending
- ready
- running
- success
- failed
- skipped
- blocked

Invariantes:
1. una etapa no entra en running si alguna dependencia requerida no está en estado compatible;
2. success es terminal salvo invalidación explícita por cambio upstream;
3. failed es terminal salvo retry o relanzamiento explícito;
4. skipped requiere razón persistida;
5. blocked significa imposibilidad de ejecución por fallo ascendente o política.

8. Idempotencia

8.1. Run key

La identidad lógica del run debe venir dada por una clave estable como:

    run_key = H(execution_date, run_mode, config_version, code_version)

donde H es un hash estable y reproducible.

8.2. Step key

Cada etapa debe tener además una clave idempotente propia:

    step_key(s) = H(run_key, step_name, upstream_fingerprint)

donde upstream_fingerprint resume:
- inputs efectivos;
- huellas de artefactos previos;
- versión de configuración relevante;
- y cualquier cambio que invalide reutilización.

Propiedades exigidas:
- si step_key no cambia y el estado previo es success consistente, la etapa puede reutilizarse;
- si step_key cambia, la etapa debe considerarse inválida y reejecutarse;
- si una etapa se reejecuta y cambia output_fingerprint, su subárbol descendente puede requerir invalidación.

9. Persistencia de estado

La persistencia debe ocurrir a dos niveles.

9.1. RunRecord

Estructura mínima recomendada:

~~~python
@dataclass
class RunRecord:
    run_id: str
    run_key: str
    execution_date: str
    run_mode: str
    status: str
    created_at: str
    started_at: str | None
    ended_at: str | None
    config_version: str
    code_version: str
    fail_fast: bool
~~~

9.2. StepRecord

~~~python
@dataclass
class StepRecord:
    run_id: str
    step_name: str
    step_key: str
    status: str
    attempt: int
    started_at: str | None
    ended_at: str | None
    input_fingerprint: str
    output_fingerprint: str | None
    error_type: str | None
    error_message: str | None
~~~

9.3. Atomicidad

La persistencia debe obedecer la semántica:

    delta_state AND delta_artifacts
    OR rollback

Es decir:
- nunca marcar success antes de validar artefactos;
- nunca dejar artefactos “huérfanos” sin estado reconciliable;
- nunca cerrar el run como success con etapas no confirmadas.

10. Preflight

Antes de pasar a running, el módulo debe ejecutar una batería de comprobaciones críticas. Mínimos razonables:

- acceso al almacén de estado;
- acceso a rutas de artefactos;
- credenciales necesarias;
- configuración válida;
- calendario/ventana operativa válida para execution_date;
- espacio disponible;
- DAG acíclico;
- compatibilidad de selected_steps.

Si falla una comprobación crítica:
- el run no debe pasar a running;
- debe cerrarse como aborted;
- debe emitirse evento preflight_failed;
- y debe quedar resumen persistido del motivo.

11. Política de errores

La política del pipeline debe distinguir, al menos, entre:

11.1. Errores transitorios
Ejemplos:
- TimeoutError
- ConnectionError
- rate limit temporal
- lock efímero de base de datos
- cola saturada
- I/O intermitente recuperable

11.2. Errores permanentes
Ejemplos:
- configuración inválida
- contrato roto
- input corrupto
- ausencia definitiva de credenciales
- bug reproducible
- DAG inválido

11.3. Errores de gate
Ejemplos:
- día no hábil
- datos obligatorios ausentes
- preflight contract failure
- condición de negocio que fuerza skip o abort

Solo los errores transitorios deben pasar por retry automático por defecto.

12. Retry policy

La espera entre intentos debe seguir exponential backoff acotado:

    B_k = min(B_max, B_0 * gamma^k)

para k = 0, ..., R-1

donde:
- B_0 es la espera base;
- gamma > 1 es el factor multiplicativo;
- B_max acota la espera máxima;
- R es el número máximo de reintentos.

Una implementación razonable:

~~~python
def backoff_delay(attempt: int, base_s: float = 5.0, gamma: float = 2.0, max_s: float = 300.0) -> float:
    return min(max_s, base_s * (gamma ** attempt))
~~~

Opcionalmente puede añadirse jitter, pero si se hace debe estar gobernado y no romper reproducibilidad cuando ésta sea contractual.

El módulo debe registrar en cada retry:
- attempt number;
- tipo de error;
- delay aplicado;
- resultado final del nuevo intento.

13. Timeouts

Cada etapa debe declarar timeout_s o heredar uno por defecto. Si una etapa excede su timeout:
- debe marcarse como failed por timeout;
- debe ejecutar cleanup seguro si aplica;
- y solo después, en su caso, entrar en política de retry si timeout es considerado transitorio.

14. Reanudación

La semántica de reanudación correcta no es “continuar desde la siguiente etapa”, sino:

1. reconstruir el plan;
2. cargar RunRecord y StepRecords;
3. reconciliar estados inconsistentes;
4. comparar step_keys persistidos con los step_keys actuales;
5. reutilizar solo etapas success compatibles;
6. reejecutar lo inválido o lo dudoso.

Reglas mínimas:
- success + mismo step_key => reusable;
- failed => candidato a retry o rerun;
- running heredado de un proceso muerto => debe reconciliarse como failed recoverable o pending según política;
- cambio en config/code/upstream => invalida compatibilidad.

15. Paralelización y recursos

La paralelización solo es admisible si:
- no existe camino u -> v ni v -> u entre dos etapas;
- no se supera un presupuesto global de recursos;
- el acceso a recursos compartidos está coordinado;
- la persistencia de estados sigue siendo consistente.

Si cada etapa s tiene un vector de recursos:

    r_s = (cpu_s, mem_s, io_s)

y existe un presupuesto total B, entonces en cualquier instante t debe cumplirse:

    sum_{s running at t} r_s <= B

El scheduler no debe lanzar etapas “solo porque están ready”; debe respetar también restricciones de capacidad.

16. Detección de bloqueos y planes inválidos

Aunque el DAG se valide al inicio, conviene mantener comprobaciones runtime de coherencia:
- no aparición de ciclos en runtime graph derivado;
- no estado donde existen etapas ready pero ninguna puede ejecutarse sin razón explícita;
- no mezcla incoherente entre blocked y success en dependencias.

Una utilidad razonable:

~~~python
def detect_deadlock(current_graph):
    try:
        list(nx.topological_sort(current_graph))
        return None
    except nx.NetworkXUnfeasible:
        return "cycle_detected"
~~~

17. Integración con monitorización

El pipeline debe emitir eventos estructurados, no strings informales. Eventos mínimos:

- run_started
- run_finished
- preflight_failed
- step_started
- step_succeeded
- step_failed
- retry_scheduled
- postflight_warning

Cada evento debe contener al menos:
- event_type
- run_id
- step_name si aplica
- severity
- timestamp
- payload

El pipeline no decide necesariamente el sink final, pero sí debe producir facts observables completos.

18. Interfaz recomendada

~~~python
@dataclass
class PipelineConfig:

@dataclass
class PipelineContext:

~~~

Y una firma operativa razonable:

~~~python
def main(config_path: str, execution_date: str, run_id: str | None = None) -> None:
    ...
~~~

19. Pseudocódigo de referencia

~~~python
def main(config_path: str, execution_date: str, run_id: str | None = None):
    cfg = load_and_validate_config(config_path)
    ctx = build_context(cfg, execution_date, run_id)

    plan = build_execution_plan(cfg, ctx)
    validate_plan(plan)

    create_run_record(ctx, status="created")

    if not run_preflight(ctx, plan):
        close_run(ctx, status="aborted")
        emit_event("preflight_failed", ctx)
        return

    mark_run_running(ctx)
    emit_event("run_started", ctx)

    while not all_steps_terminal(ctx, plan):
        ready_steps = scheduler(plan, ctx)

        for step in ready_steps:
            if not dependencies_satisfied(step, plan, ctx):
                mark_step_blocked(step, ctx)
                continue

            if is_step_reusable(step, ctx):
                mark_step_success_reused(step, ctx)
                continue

            mark_step_running(step, ctx)
            emit_event("step_started", ctx, step)

            for attempt in range(cfg.max_retries + 1):
                try:
                    output = execute_step(step, ctx)
                    validate_step_output(step, output, ctx)
                    persist_step_artifacts(step, output, ctx)
                    mark_step_success(step, ctx, output)
                    emit_event("step_succeeded", ctx, step)
                    break

                except TransientError as exc:
                    if attempt == cfg.max_retries:
                        mark_step_failed(step, ctx, exc)
                        emit_event("step_failed", ctx, step, exc)
                        if cfg.fail_fast and is_critical(step):
                            close_run(ctx, status="failed")
                            emit_event("run_finished", ctx)
                            return
                    else:
                        delay = backoff_delay(attempt, cfg.retry_base_seconds, cfg.retry_backoff_factor)
                        schedule_retry(step, ctx, attempt, exc, delay)
                        emit_event("retry_scheduled", ctx, step, exc)

                except Exception as exc:
                    mark_step_failed(step, ctx, exc)
                    emit_event("step_failed", ctx, step, exc)
                    if cfg.fail_fast and is_critical(step):
                        close_run(ctx, status="failed")
                        emit_event("run_finished", ctx)
                        return
                    break

    final_status = reconcile_run_status(ctx, plan)
    run_postflight(ctx, plan, final_status)
    close_run(ctx, status=final_status)
    emit_event("run_finished", ctx)
~~~

20. Casos límite que el módulo debe resolver

20.1. Caída tras persistir artefactos pero antes de marcar success
La reanudación debe reconciliar artefactos existentes con estado pendiente o running heredado.

20.2. Cambio de configuración entre fallo y resume
La compatibilidad debe romperse; no se puede reutilizar ciegamente un step success bajo otra config efectiva.

20.3. Dependencia opcional ausente
Debe marcarse skipped o blocked con razón explícita, nunca success implícito.

20.4. Rerun de una etapa con descendientes ya confirmados
Debe existir invalidación controlada del subárbol descendente afectado.

20.5. Dos procesos intentando el mismo run_key
Debe haber exclusión lógica o lock operacional; no pueden coexistir dos writers del mismo run.

20.6. Preflight pasa pero fuente devuelve datos incompletos
La etapa debe fallar por contrato, no esconder el problema como empty success.

21. Failure modes críticos que debe detectar o evitar

F1. DAG con ciclo.
F2. Topological order incorrecto.
F3. Reejecución que duplica artefactos.
F4. Run success con artefactos incompletos.
F5. Etapas running “zombies” tras crash.
F6. Retry sobre errores permanentes.
F7. Resume incompatible por cambio upstream.
F8. Paralelización que excede recursos y degrada estabilidad.
F9. Run final incoherente con estados de etapas.
F10. Ausencia de eventos operativos críticos.

22. MVP obligatorio

El MVP del módulo debe incluir:
- construcción y validación del DAG;
- topological scheduling;
- run_key y step_key idempotentes;
- persistencia de RunRecord y StepRecord;
- retry con exponential backoff;
- preflight;
- postflight;
- reanudación básica;
- emisión de eventos;
- fail_fast configurable.

23. Segunda fase razonable

Como extensiones útiles:
- resource-aware scheduling;
- deadlock detection runtime;
- clasificación transitorio/permanente más rica;
- state reconciliation multi-process;
- invalidación transitiva de descendientes;
- control de concurrencia por resource budgets;
- observabilidad más profunda.

24. No-objetivos

No pertenece al alcance de este módulo:
- optimizar la estrategia;
- mejorar métricas financieras;
- seleccionar modelos;
- calcular señales;
- ajustar thresholds de negocio;
- hacer scheduling estocástico sofisticado;
- formal verification completa del state machine.

La misión aquí es reliability engineering disciplinada.

25. Criterios de aceptación

El módulo solo debe considerarse correcto si cumple simultáneamente:

- el DAG es validado y acíclico;
- el orden topológico es correcto;
- run y etapas son idempotentes por clave lógica;
- la persistencia de estado es consistente y atómica;
- los reintentos solo actúan sobre errores recuperables;
- la reanudación evita recomputación ciega;
- los eventos operativos cubren el ciclo de vida completo;
- el estado final del run reconcilia correctamente las etapas;
- el pipeline puede abortar de forma segura ante fallos críticos.

26. Conclusión

ops/daily_pipeline.py es una pieza de infraestructura crítica, no un módulo de research. Su calidad no debe medirse por Sharpe ni por alpha, sino por la robustez con la que convierte módulos heterogéneos en una ejecución diaria determinista, auditable, reanudable y segura. En un framework cuantitativo serio, esa disciplina operativa es exactamente lo que separa un repositorio prometedor de un sistema de producción fiable. Ese es el estándar que este módulo debe imponer.
