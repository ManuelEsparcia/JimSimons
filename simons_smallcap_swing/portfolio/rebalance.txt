Especificación Cuantitativa: portfolio/rebalance.py

1. Propósito del módulo

El objetivo de portfolio/rebalance.py es transformar un portafolio objetivo ya validado aguas arriba en un conjunto de órdenes ejecutables, respetando simultáneamente:

- presupuesto de turnover,
- thresholds operativos,
- lot sizing,
- negociabilidad real,
- y restricciones hard heredadas del portfolio final.

La pregunta correcta del módulo es:

“Dado un portafolio actual y un portafolio objetivo, ¿cuál es la implementación discreta más cercana al target que sigue siendo contablemente coherente, operativamente ejecutable y compatible con el budget disponible?”

Este módulo no decide cuál debe ser la cartera óptima. Esa decisión ya ha sido tomada aguas arriba por:
- optimizer.py,
- constraints.py,
- neutralize.py,
- o una arquitectura equivalente.

Su función correcta es traducir un target ya aprobado en un plan de órdenes consistente.

2. Principios no negociables

2.1 Separación de responsabilidades
rebalance.py no optimiza alpha-riesgo. Solo traduce pesos objetivo a órdenes operativas.

2.2 Semántica cerrada
El módulo debe tener una semántica única y reproducible:
- target continuo ejecutable,
- traducción a acciones,
- rounding,
- reparación discreta,
- postcheck final.

2.3 Precedencia explícita
Las reglas deben tener prioridad fija:
1. validez de datos y negociabilidad
2. órdenes obligatorias
3. presupuesto de turnover
4. thresholds discrecionales
5. rounding por lotes
6. repair final
7. postcheck hard

2.4 Consistencia contable
Pesos, acciones, notional y efectivo deben cuadrar dentro de tolerancia.

2.5 Determinismo
Mismas entradas -> misma salida.
No se permiten heurísticas opacas no trazadas.

2.6 Auditabilidad
Toda orden omitida, truncada, redondeada o diferida debe dejar:
- razón explícita,
- prioridad,
- y efecto sobre drift residual.

3. Qué hace y qué no hace

3.1 Qué hace
- recibe pesos objetivo y pesos actuales;
- clasifica órdenes en obligatorias, discrecionales o bloqueadas;
- construye un portafolio ejecutable continuo bajo budget;
- lo convierte a acciones;
- redondea a lotes;
- repara residuos discretos;
- genera órdenes con `skip_reason`;
- y produce el portafolio estimado post-trade.

3.2 Qué no hace
- no genera alpha;
- no recalcula optimizer;
- no sustituye execution;
- no estima trayectorias intradía;
- no inventa liquidez ni precios faltantes.

1. orders.parquet
2. rebalance_report.json
3. posttrade_portfolio.parquet
4. status in {ok, partial, blocked, fail_postcheck}

6. Notación

Sea N el número de activos en la fecha t.

- w_cur in R^N: pesos actuales
- w_tgt in R^N: pesos objetivo
- p in R_+^N: precios
- s_cur in Z^N: acciones actuales
- NAV > 0

Diferencia ideal en pesos:
    Delta w_i^ideal = w_tgt_i - w_cur_i

Turnover ideal:
    TO_ideal = sum_i |Delta w_i^ideal|

Acciones objetivo continuas:
    s_i^(tgt,cont) = NAV * w_tgt_i / p_i

Orden continua ideal:
    Delta s_i^ideal = s_i^(tgt,cont) - s_cur_i

Notional ideal:
    n_i^ideal = p_i * |Delta s_i^ideal|
              = NAV * |Delta w_i^ideal|

7. Clasificación exacta de activos

Cada activo debe pertenecer exactamente a una de estas clases:

7.1 Mandatory
Órdenes obligatorias:
- salida completa de nombres no elegibles,
- cierre forzado,
- reducción hard impuesta upstream,
- cualquier otra orden marcada como no discrecional.

7.2 Discretionary
Órdenes destinadas a acercar la cartera al target, sujetas a threshold y budget.

7.3 Blocked
Activos no operables:
- tradable_flag = False
- halt_flag = True
- price faltante o no válido
- lote inválido
- o cualquier bloqueo real equivalente

8. Threshold canónico

La regla de threshold se aplica solo a órdenes discrecionales.

Un activo discrecional queda congelado en:
    w_i = w_cur_i

si simultáneamente:
    |Delta w_i^ideal| < theta_w
y
    n_i^ideal < n_min

La lógica oficial es AND, no OR.

Las órdenes obligatorias quedan exentas de este filtro salvo bloqueo operativo real.

9. Target ejecutable continuo

9.1 Idea central
La salida continua del módulo no debe surgir de un simple ranking codicioso por tamaño.
La semántica canónica es un problema de proyección convexa bajo budget.

9.2 Problema canónico
Sea w el portafolio ejecutable continuo buscado.

Resolver:

    minimize    1/2 (w - w_tgt)^T D (w - w_tgt)

    subject to
                sum_i |w_i - w_cur_i| <= B_TO
                w_i = w_tgt_i       para i en mandatory si factible
                w_i = w_cur_i       para i en blocked
                w_i = w_cur_i       para discrecionales bajo threshold
                restricciones hard heredadas del upstream

Default:
    D = I

Si el sistema quiere ponderar por volatilidad o liquidez, debe declararlo explícitamente en config.

9.3 Restricciones hard heredadas
Si el target upstream ya venía condicionado por gross/net/bounds y se desea preservarlos tras budgeting, el módulo puede imponer adicionalmente:

- box:
    l <= w <= u

- gross:
    sum_i |w_i| <= G_max

- net:
    N_min <= sum_i w_i <= N_max

Estas restricciones deben activarse solo si la configuración las declara como hard también en rebalance.

10. Reformulación convexa explícita

Para implementar la proyección continua sin ambigüedad:

10.1 Turnover budget
Introducir variable u_i >= 0:
    u_i >= w_i - w_cur_i
    u_i >= w_cur_i - w_i
    sum_i u_i <= B_TO

10.2 Gross heredado opcional
Si gross está activo, introducir z_i >= 0:
    z_i >= w_i
    z_i >= -w_i
    sum_i z_i <= G_max

10.3 Net heredado opcional
    N_min <= sum_i w_i <= N_max

10.4 Mandatory / blocked / threshold
- mandatory:
    w_i = w_tgt_i
- blocked:
    w_i = w_cur_i
- below-threshold discretionary:
    w_i = w_cur_i

11. Fallback determinista

Si la configuración desactiva la proyección continua, el único fallback permitido será un ranking determinista con precedencia explícita:

1. ejecutar primero mandatory,
2. sobre discrecionales restantes, ordenar por prioridad
   priority_i = |Delta w_i^ideal| / max(c_i, c_floor)
3. acumular hasta agotar B_TO,
4. congelar el resto.

Si se usa:
    used_fallback = True

Nunca debe confundirse con la semántica oficial del módulo.

12. Conversión a acciones

Una vez obtenido w_exe_cont:

    s_i^(exe,cont) = NAV * w_i^(exe,cont) / p_i

Orden continua:
    Delta s_i^(exe,cont) = s_i^(exe,cont) - s_cur_i

13. Política de rounding

13.1 Política oficial
Default:
    rounding_policy = toward_zero_to_lot

Es decir:

    Delta s_i^(lot) = lotfloor(Delta s_i^(exe,cont), lot_i)

donde el redondeo se hace hacia cero al múltiplo válido más cercano.

13.2 Razón
Esta política evita:
- sobrepasar el target continuo,
- consumir más budget del previsto,
- y fabricar notional discreto artificialmente agresivo.

13.3 Notional discreto
    n_i^(lot) = p_i * |Delta s_i^(lot)|

14. Repair loop canónico

Tras el rounding, el módulo debe ejecutar una reparación discreta final.

14.1 Acciones post-órdenes
    s_i^post = s_cur_i + Delta s_i^(lot)

14.2 Pesos post-órdenes
    w_i^post = s_i^post * p_i / NAV

14.3 Drift residual respecto al target continuo
    r_i = w_i^(exe,cont) - w_i^post

Métrica agregada:
    Drift_1 = sum_i |r_i|

14.4 Repair loop
El repair loop por defecto será:

1. ordenar activos por mayor |r_i| entre los que admiten un lote adicional factible;
2. proponer un lote extra en la dirección que reduzca |r_i|;
3. aceptar solo si no viola:
   - turnover budget,
   - efectivo disponible,
   - lot validity,
   - caps hard heredadas,
   - y, si está activo, no excede el target continuo más allá de tolerancia;
4. repetir hasta que:
   - no haya mejora factible,
   - o el drift residual quede por debajo de tolerancia.

14.5 Naturaleza del método
Este repair loop es heurístico pero determinista.
No garantiza optimalidad global discreta, pero sí una política clara, estable y suficiente para el MVP.

15. Liquidez opcional

Si se activa cap de participación por nombre, calcular:

    pi_i = p_i * |Delta s_i^(lot)| / (phi * ADV_i^USD)

Restricción:
    pi_i <= pi_max

Si una orden la viola:
- truncarla al máximo factible,
- o marcarla como parcialmente ejecutada / bloqueada por liquidez.

Esto debe aplicarse antes del cierre final del repair loop.

16. Postcheck hard

Después del repair loop, el módulo debe verificar:

16.1 Turnover
    TO_exec <= B_TO + eps

16.2 Gross y net heredados, si activos
- gross_post <= G_max + eps
- N_min - eps <= net_post <= N_max + eps

16.3 Posición por nombre si aplica
- bounds individuales respetados

16.4 Efectivo residual
- cash_post dentro de tolerancia

16.5 Lote válido
- todas las órdenes múltiples válidos de lot_size

Si una restricción hard queda violada materialmente:
    status = fail_postcheck

Si la violación proviene de imposibilidad operativa real documentada:
    status = partial

17. Taxonomía obligatoria de skip_reason

El módulo debe usar vocabulario controlado:

- below_threshold
- budget_exhausted
- blocked_nontradable
- lot_round_to_zero
- liquidity_cap
- cash_insufficient
- hard_constraint_repair
- fully_executed
- partially_executed

No se permiten razones libres no normalizadas en la salida principal.

18. Métricas before/after obligatorias

El reporte agregado debe incluir al menos:

- turnover_ideal
- turnover_executed
- budget_usage
- num_orders
- num_skipped
- num_mandatory_blocked
- cash_residual
- l1_drift_to_target
- l1_drift_to_cont
- gross_before / after
- net_before / after
- max_name_drift

19. Invariantes del módulo

Toda salida válida debe cumplir:

1. las órdenes respetan coherencia contable entre pesos, acciones y notional;
2. ninguna orden discrecional supera el presupuesto total;
3. ninguna orden obligatoria se elimina por threshold salvo bloqueo real;
4. el portafolio posttrade es exactamente el que resulta de aplicar las órdenes persistidas;
5. cualquier desviación material respecto a w_tgt o w_exe_cont queda cuantificada y explicada.

20. Estado final

La semántica recomendada es:

- ok:
  ejecución coherente y postcheck limpio

- partial:
  algunas órdenes no pudieron ejecutarse plenamente por restricciones reales, pero la salida sigue siendo usable y auditada

- blocked:
  la mayor parte de la implementación queda impedida por no negociabilidad o bloqueos dominantes

- fail_postcheck:
  la salida final rompe restricciones hard por encima de tolerancia

21. Tests mínimos obligatorios

21.1 Test identidad
Si w_cur = w_tgt y no hay mandatory externos:
    no debe generarse orden discrecional

21.2 Test threshold
Una orden discrecional por debajo de theta_w y n_min:
    skip_reason = below_threshold

21.3 Test mandatory
Una orden mandatory no debe desaparecer por threshold

21.4 Test budget
    TO_exec <= B_TO + eps

21.5 Test rounding
Todas las órdenes deben ser múltiplos válidos de lot_size

21.6 Test repair
El repair loop no puede aumentar drift si rompe budget o constraints

21.7 Test contable
Para cada activo:
    |p_i * Delta s_i - NAV * Delta w_i_est| <= tol_disc razonable

21.8 Test postcheck
La cartera final debe respetar constraints heredadas o marcar fail_postcheck/partial

22. Integración correcta en el pipeline

Ubicación semántica natural:

signal
-> optimizer / constraints / neutralize
-> rebalance
-> execution

rebalance.py define el plan inicial de implementación.
execution.py o el simulador posterior modelarán:
- fills parciales,
- slippage real,
- retrasos,
- y demás fricciones de mercado.

23. Fase 2 opcional: LP exacta de rounding+repair

Como mejora opcional, puede existir una versión avanzada que resuelva conjuntamente rounding y repair como LP/MILP pequeño bajo budget y cash.

Pero:
- no debe ser la semántica base del MVP,
- ni justificar complejidad excesiva en small caps swing si el gain marginal es pequeño.

24. Interpretación correcta

La lectura correcta de rebalance.py no es:
“filtramos un target y redondeamos como podamos”.

La lectura correcta es:
“construimos primero el portafolio continuo ejecutable más cercano al target bajo budget, luego lo discretizamos y finalmente reparamos de forma determinista el residuo sin romper constraints.”

Eso convierte el rebalanceo en una capa:
- programable,
- testeable,
- auditable,
- y compatible con una arquitectura cuantitativa seria.

25. Conclusión

portfolio/rebalance.py debe ser una especificación cerrada de traducción operativa.

Recibe un target ya validado.
Clasifica activos por prioridad y ejecutabilidad.
Construye un target continuo bajo budget.
Lo convierte a acciones.
Redondea de forma conservadora.
Repara el residuo discreto.
Postchequea constraints hard.
Y persiste órdenes junto con una explicación completa de toda desviación material.

Ese es exactamente el comportamiento que debe tener una capa seria de rebalanceo en un stack cuantitativo institucional.
