Especificación Cuantitativa: portfolio/neutralize.py

1. Propósito del módulo

El objetivo de portfolio/neutralize.py es transformar un vector de pesos preliminares

    w_in in R^N

en un vector de pesos neutralizados

    w_neu in R^N

tal que las exposiciones no deseadas a factores continuos y grupos categóricos queden llevadas a targets o bandas predefinidas, preservando simultáneamente la implementabilidad del portafolio.

La pregunta correcta del módulo es:

“¿Cuál es el portafolio factible más cercano al portafolio preliminar que satisface los targets de neutralización y respeta net, gross, posición, turnover y liquidez?”

El módulo no “purifica el alpha” en sentido metafísico. Su función operativa es más precisa:
eliminar o acotar exposiciones sistemáticas no intencionales sin romper la geometría económica del portafolio.

2. Principios no negociables

2.1 Point-in-time
Toda exposición, grupo, liquidez y tradabilidad deben ser PIT en la fecha de rebalanceo.

2.2 Separación de responsabilidades
neutralize.py ajusta pesos.
No:
- estima factores,
- recalibra el risk model,
- genera alpha,
- ni sustituye un optimizador global alpha+risk+constraints si este ya existe.

2.3 Método canónico único
La implementación oficial del módulo es una proyección convexa restringida.
Proyección ortogonal y residualización solo pueden existir como diagnóstico o warm start.

2.4 Preservación explícita
La preservación de net, gross, long gross o short gross debe imponerse mediante restricciones explícitas.
Nunca mediante renormalizaciones ambiguas a posteriori.

2.5 Infactibilidad visible
Si neutralidad y restricciones operativas son incompatibles, el módulo debe declararlo.
No debe fabricar una solución heurística silenciosa.

2.6 Auditabilidad
Toda neutralización debe dejar trazabilidad before/after, slacks, active constraints y distancia al input.

3. Qué hace y qué no hace

3.1 Qué hace
- neutraliza exposiciones continuas como beta, size, value, momentum, volatility u otras columnas de X;
- neutraliza exposiciones categóricas como sector, industria, país o buckets discretos;
- impone targets exactos o bandas de tolerancia;
- preserva net de forma exacta o en banda;
- respeta posición, gross, turnover y liquidez si se configuran;
- devuelve el vector factible más cercano al input.

3.2 Qué no hace
- no decide el universo;
- no crea señales;
- no hace PCA/SVD por defecto;
- no aprende factores latentes;
- no reinterpreta el alpha preliminar.

4. Entradas obligatorias

4.1 Pesos
- w_in
- w_current
- eligible_flag
- halt_flag si aplica
- side_flag si existe restricción long-only / short-only

4.2 Exposiciones continuas
Matriz

    X in R^(N x K)

cuyas columnas representan factores continuos PIT:
beta, size, value, momentum, residual vol, liquidity, etc.

4.3 Exposiciones categóricas
Matriz indicadora

    B in {0,1}^(N x G)

para sectores, industrias, países u otros grupos discretos.

4.4 Targets
- t_x in R^K
- t_b in R^G
- tau_x >= 0
- tau_b >= 0

donde:
- t_x, t_b son targets de exposición,
- tau_x, tau_b son bandas de tolerancia.

4.5 Restricciones operativas
- bounds por activo: l_i, u_i
- gross_max
- net_min, net_max o net_target exacto
- long_gross_max y short_gross_max si aplican
- turnover_max
- cap de liquidez por nombre
- AUM operativo
- phi de ADV utilizable

4.6 Métrica de distancia
Matriz diagonal positiva D.

Default recomendable:
- D = I
o
- D_ii = 1 / (sigma_i + eps)^2

según lo que se decida como convención del sistema.

5. Salidas obligatorias

El módulo debe devolver:

1. w_neu
2. neutralization_report.json
3. before_after.parquet
4. status in {optimal, optimal_inaccurate, infeasible, fail_postcheck}

6. Notación

Sea N el número de activos elegibles.

- w in R^N
- w_in in R^N
- w_cur in R^N

Definimos:

Gross:
    G(w) = sum_i |w_i|

Net:
    N(w) = sum_i w_i

Long gross:
    G_plus(w) = sum_i max(w_i, 0)

Short gross:
    G_minus(w) = sum_i max(-w_i, 0)

Turnover:
    TO(w, w_cur) = sum_i |w_i - w_cur_i|

Exposición continua:
    e_x(w) = X^T w

Exposición categórica:
    e_b(w) = B^T w

7. Definición correcta de neutralización

Neutralizar significa resolver un problema de proyección sobre el conjunto factible definido por:
- targets de exposición,
- límites por activo,
- net,
- gross,
- turnover,
- liquidez,
- y demás restricciones activas.

No significa:
- restar promedios,
- residualizar y renormalizar,
- ni “arreglar” pesos secuencialmente hasta que más o menos encajen.

8. Problema matemático canónico

El método oficial del módulo es:

    minimize    1/2 (w - w_in)^T D (w - w_in)

    subject to
        l <= w <= u
        N_min <= 1^T w <= N_max
        sum_i |w_i| <= G_max
        G_plus(w) <= G_plus_max                 si aplica
        G_minus(w) <= G_minus_max               si aplica
        t_x - tau_x <= X^T w <= t_x + tau_x
        t_b - tau_b <= B^T w <= t_b + tau_b
        TO(w, w_cur) <= TO_max                  si aplica
        restricciones de liquidez               si aplican

Si se requiere net exacto:
    1^T w = N_target

Interpretación:
w_neu es el punto factible más cercano a w_in bajo la métrica D.

9. Reformulación convexa explícita

Para evitar ambigüedad, el módulo debe explicitar la reformulación de absolutos.

9.1 Gross total
Introducir z_gross_i >= 0 tal que:
    z_gross_i >= w_i
    z_gross_i >= -w_i

y:
    sum_i z_gross_i <= G_max

9.2 Long gross
Introducir z_long_i >= 0 tal que:
    z_long_i >= w_i

y:
    sum_i z_long_i <= G_plus_max

9.3 Short gross
Introducir z_short_i >= 0 tal que:
    z_short_i >= -w_i

y:
    sum_i z_short_i <= G_minus_max

9.4 Turnover
Introducir v_i >= 0 tal que:
    v_i >= w_i - w_cur_i
    v_i >= w_cur_i - w_i

y:
    sum_i v_i <= TO_max

9.5 Liquidez
Sea A el AUM operativo y ADV_i el ADV monetario PIT.

Definir:
    cap_i = (pi_max * phi * ADV_i) / A

Entonces:
    w_i - w_cur_i <= cap_i
    w_cur_i - w_i <= cap_i

Esto implementa una cota lineal sobre participación por nombre.

10. Tratamiento separado de X y B

10.1 Factores continuos
X controla exposiciones continuas:

    X^T w ≈ t_x

Estas columnas deben venir ya definidas y normalizadas upstream.
neutralize.py no debe reinterpretar su escala ni hacer transformaciones latentes por defecto.

10.2 Grupos categóricos
B controla exposiciones discretas:

    B^T w ≈ t_b

La matriz B debe estar bien definida:
- filas alineadas con activos,
- columnas con semántica fija,
- y sin ambigüedad de pertenencia donde no corresponda.

10.3 Razón de la separación
No deben mezclarse factores continuos y dummies categóricas en una sola representación opaca si eso destruye la interpretabilidad de:
- targets,
- slacks,
- y binding constraints.

11. Preservación de invariantes económicos

11.1 Net
Default recomendado:
    preserve_net = exact

Implementación:
    1^T w = 1^T w_in
o
    1^T w = N_target

11.2 Gross
Default recomendado:
    preserve_gross = upper_bound

Es decir:
    G(w) <= G_max

No imponer gross exacto salvo necesidad explícita, porque aumenta infactibilidad sin aportar demasiado valor en la mayoría de casos.

11.3 Long/short balance
Si la estrategia es long-short y el balance por lado importa, pueden imponerse:
    G_plus(w) <= G_plus_max
    G_minus(w) <= G_minus_max

o bandas estrechas alrededor del balance original.

12. Métodos secundarios permitidos

12.1 Proyección ortogonal
Puede existir como diagnóstico:

    P_X = I - X (X^T X)^(-1) X^T
    w_tilde = P_X w_in

pero no como método oficial si hay constraints operativas simultáneas.

12.2 Residualización
También puede existir como diagnóstico:
    w_in = X beta + eps
    w_tilde = eps

pero tampoco como método canónico.

12.3 Regla de uso
Ambos métodos solo pueden existir como:
- diagnostic_projection
- diagnostic_residualization
- warm start opcional

Nunca como semántica principal del módulo.

13. Política de infactibilidad

13.1 Política por defecto
Default:
    infeasibility_policy = strict_fail

Si el problema es infactible:
- status = infeasible
- w_neu = None
- no se devuelve cartera parcheada
- se reportan las familias de constraints en conflicto

13.2 Relajación opcional
Solo si se activa explícitamente:
    infeasibility_policy = prioritized_relaxation

Orden permitido:
1. tolerancias de factores continuos secundarios
2. tolerancias de grupos secundarios
3. turnover
4. gross exacto, si existiera
5. net band estrecha, si no es exacta

Nunca deben relajarse silenciosamente:
- elegibilidad
- halt_flag
- bounds hard por activo
- liquidez por nombre marcada hard

14. Solver policy

14.1 Regla simple
- problema pequeño o relativamente denso: ECOS
- QP sparse con muchas restricciones lineales y turnover: OSQP
- problema muy grande / fallback first-order: SCS

14.2 Default recomendado
OSQP como default general razonable para el MVP.

14.3 Registro obligatorio
El solver usado debe quedar persistido en el reporte.

15. Post-checks numéricos

Aunque el solver devuelva optimal, el módulo debe verificar todas las restricciones con tolerancia específica por solver.

Tolerancias recomendadas:
- ECOS: 1e-8
- OSQP: 1e-6
- SCS: 1e-5

Checks mínimos:

15.1 Box
    max_i max(0, l_i - w_i, w_i - u_i) <= eps_solver

15.2 Gross
    max(0, G(w) - G_max) <= eps_solver

15.3 Long/short gross
    max(0, G_plus(w) - G_plus_max) <= eps_solver
    max(0, G_minus(w) - G_minus_max) <= eps_solver

15.4 Net
    max(0, N_min - N(w), N(w) - N_max) <= eps_solver
o igualdad exacta dentro de tolerancia si así se configuró.

15.5 Factores
    max_k violation_k_factor <= eps_solver

15.6 Grupos
    max_g violation_g_group <= eps_solver

15.7 Turnover
    max(0, TO(w, w_cur) - TO_max) <= eps_solver

15.8 Liquidez
    max_i liquidity_violation_i <= eps_solver

Si una violación supera tolerancia:
- status = optimal_inaccurate
o
- fail_postcheck si la violación es material

16. Before/after obligatorio

El reporte debe incluir al menos:

- net_before / net_after
- gross_before / gross_after
- long_gross_before / after
- short_gross_before / after
- turnover_before / after
- factor_exposure_before / after
- group_exposure_before / after
- max_abs_dw
- distance_to_input
- active_constraints
- binding_constraints
- slacks por familia

17. Invariantes del módulo

Toda salida válida debe satisfacer:

1. mismo universo e índice que la entrada elegible
2. ningún activo no elegible o suspendido con peso operativo
3. constraints activas satisfechas dentro de tolerancia
4. targets de neutralización respetados dentro de bandas
5. si w_in ya es factible, entonces w_neu = w_in salvo error numérico menor
6. strict_fail nunca devuelve cartera infactible disfrazada

18. Failure modes importantes

18.1 X mal condicionada
Si X^T X es casi singular, métodos diagnósticos ortogonales pueden ser inestables.
El método canónico restringido sigue siendo la referencia, pero la condición debe reportarse.

18.2 B mal especificada
Si las filas de B no están alineadas o la pertenencia categórica está mal definida, la neutralización categórica pierde sentido.

18.3 Targets incompatibles
Neutralidad demasiado estricta con bounds, net o turnover puede volver el problema infactible.
Eso debe reportarse, no maquillarse.

18.4 Liquidez demasiado dura
La combinación de turnover y límites de participación puede hacer imposible alcanzar el target neutralizado.
Eso es una señal útil, no un error del solver.

19. Salida recomendada

19.1 neutralization_report.json
Debe incluir:
- status
- solver_name
- solve_time
- objective_value
- active_constraints
- slacks
- max_violation_by_family
- used_fallback
- infeasibility_reason si aplica
- distance_to_input

19.2 before_after.parquet
Debe incluir por activo:
- w_in
- w_neu
- delta_abs
- delta_signed
- liquidity_cap
- group memberships
- factor exposures
- flags de constraint activa

20. Esqueleto de implementación

def run_neutralize(w_in, w_cur, X, B, targets_x, targets_b, config):
    validate_inputs(w_in, w_cur, X, B, targets_x, targets_b, config)

    problem = build_constrained_projection_problem(
        w_in=w_in,
        w_cur=w_cur,
        X=X,
        B=B,
        targets_x=targets_x,
        targets_b=targets_b,
        config=config
    )

    result = solve_problem(problem, config)

    postcheck = run_postchecks(result, config)

    report = build_neutralization_report(
        result=result,
        postcheck=postcheck,
        config=config
    )

    persist_outputs(result, report, config)

    return result["w_neu"], report, result["status"]

21. Tests mínimos obligatorios

21.1 Neutralidad exacta
Si w_in ya cumple todos los targets y constraints:
    w_neu ≈ w_in

21.2 Neutralización continua
Con un caso sintético de beta no nula:
    X^T w_neu debe caer en la banda objetivo

21.3 Neutralización categórica
Con una sobreexposición sectorial sintética:
    B^T w_neu debe corregirse sin romper net ni bounds

21.4 Preservación de net
Si preserve_net = exact:
    sum_i w_neu_i = sum_i w_in_i dentro de tolerancia

21.5 Gross y turnover
La solución no puede violar gross o turnover si están activos

21.6 Infactibilidad real
Si targets y bounds son incompatibles:
    status = infeasible bajo strict_fail

21.7 Auditoría
El reporte debe contener exposiciones before/after, slacks y estado final

22. Integración correcta en el pipeline

La ubicación semántica correcta suele ser:

signal
-> sizing
-> neutralize
-> constraints / optimizer
-> rebalance
-> execution

Si existe un optimizador global que ya incorpora neutralidad como constraint interna, neutralize.py puede usarse como:
- sanity check,
- diagnóstico,
- o capa auxiliar,
pero no debe duplicar silenciosamente la optimización global.

23. Defaults razonables

- method = constrained_projection
- distance_metric = weighted_l2
- preserve_net = exact
- preserve_gross = upper_bound
- infeasibility_policy = strict_fail
- solver_name = OSQP
- tolerances versionadas por solver

24. Interpretación correcta

La lectura correcta de neutralize.py no es:
“restamos exposición y luego arreglamos el portfolio”.

La lectura correcta es:
“definimos formalmente el conjunto de portafolios neutralizados e implementables y proyectamos sobre él el portafolio preliminar con una métrica única, reproducible y auditada”.

Eso convierte la neutralización en una propiedad verificable del output, no en una aspiración retórica.

25. Conclusión

portfolio/neutralize.py debe ser una especificación cerrada de neutralización implementable.

Recibe pesos y exposiciones PIT.
Define targets explícitos para factores y grupos.
Construye un conjunto factible con neutralidad y límites operativos.
Proyecta convexamente al punto más cercano.
Hace visible la infactibilidad.
Y deja una auditoría completa de cuánto alpha preliminar hubo que sacrificar para obtener un portafolio neutralizado y ejecutable.

Ese es exactamente el comportamiento que debe tener una capa seria de neutralización en un stack cuantitativo institucional.
