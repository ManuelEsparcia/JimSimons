Especificación Cuantitativa: portfolio/capacity.py

1. Propósito del módulo

El objetivo de portfolio/capacity.py es estimar la capacidad operable máxima de la estrategia, entendida como el mayor nivel de AUM para el cual la estrategia sigue siendo ejecutable sin violar restricciones duras de:

- participación de mercado,
- días para liquidar órdenes,
- días para liquidar posiciones,
- concentración por nombre,
- y degradación mínima aceptable del retorno neto.

La salida del módulo no es una opinión cualitativa sobre si la estrategia “escala bien”, sino una decisión reproducible:

capacity_limit = max { A_k in A : todas las reglas duras se satisfacen }

El módulo no reoptimiza la estrategia, no modifica el universo y no cambia la lógica de rebalanceo. Mide escalabilidad condicional a la estrategia ya fijada.

2. Principios no negociables

2.1 Point-in-time
Toda métrica de liquidez, spread y operabilidad debe ser PIT por fecha y activo.

2.2 Condicionalidad
La capacidad se estima condicional a:
- selección de activos,
- sizing,
- optimizer,
- rebalanceo,
- costes,
- y ejecución efectivos del pipeline actual.

2.3 Monotonicidad
Al aumentar AUM, las métricas de fricción no pueden mejorar estructuralmente.
Si el ruido empírico introduce pequeñas no monotonicidades, deben regularizarse antes de decidir.

2.4 Separación de responsabilidades
capacity.py mide escalabilidad.
No sustituye a:
- rebalance.py,
- execution/impact,
- optimizer.py,
- ni al motor de backtest.

3. Entradas

El módulo debe recibir:

3.1 Rebance tape efectivo
Una tabla PIT con una fila por activo y fecha de rebalanceo, al menos con:

- date
- asset_id
- price
- w_current
- w_target
- delta_w = w_target - w_current
- abs_delta_w = |delta_w|
- w_post
- flags de tradabilidad si aplican

El módulo debe usar órdenes efectivamente rebalanceadas, no órdenes teóricas previas a thresholds o filtros.

3.2 Datos PIT de liquidez
Por activo y fecha:

- adv_usd
- spread_bps
- volatilidad reciente si está disponible
- tradable_flag
- halt_flag o flags equivalentes
- cualquier campo necesario para excluir observaciones no operables

3.3 Configuración
Parámetros versionados como:

- aum_reference
- scenario_multipliers
- usable_adv_fraction = phi
- participation_cap_p95
- trade_dtl_cap_p95
- position_dtl_cap_p95
- max_single_name_participation
- impact_eta
- impact_gamma
- fee_bps
- min_return_retention
- min_net_return
- liquidity_floor_usd = eps_L
- participation_cap_hard = pi_cap_hard
- bootstrap_n
- bootstrap_confidence

4. Escenarios de AUM

Sea:
A_ref = AUM de referencia del baseline

y sea:
M = {m_1, ..., m_K}
una rejilla creciente de multiplicadores.

Entonces:
A_k = m_k * A_ref

Rejilla recomendada MVP:
M = {0.50, 0.75, 1.00, 1.25, 1.50, 2.00, 3.00, 4.00, 5.00}

La rejilla debe ser creciente, versionada y persistida con la salida.

5. Escalado de órdenes y posiciones

5.1 Orden monetaria
Para activo i y fecha t:

q_{i,t}(A_k) = A_k * |delta_w_{i,t}|

Este es el tamaño monetario de la orden bajo el escenario A_k.

5.2 Notional de posición
El tamaño monetario post-rebalance relevante para riesgo de liquidación es:

pos_{i,t}(A_k) = A_k * |w_post_{i,t}|

Este módulo asume que el turnover porcentual base se mantiene al escalar AUM.
Lo que cambia es el tamaño monetario, no la lógica de rebalanceo.

6. Liquidez utilizable

Definimos la liquidez utilizable por activo y fecha como:

L_{i,t} = max(phi * ADV_{i,t}^{USD}, eps_L)

donde:
- phi in (0,1] es la fracción de ADV considerada realmente accesible,
- eps_L > 0 es un suelo numérico y operativo mínimo.

Default MVP razonable:
- phi = 0.10
- eps_L = 10,000 USD

Interpretación:
- phi controla prudencia operativa,
- eps_L evita degeneraciones numéricas cuando ADV es casi cero o errático.

Si tradable_flag = False o halt_flag = True, la observación no debe tratarse como ejecutable normal.

7. Participación de mercado

7.1 Participación bruta
pi_raw_{i,t}(A_k) = q_{i,t}(A_k) / L_{i,t}

7.2 Participación robusta
Para blindar el módulo contra explosiones numéricas:

pi_{i,t}(A_k) = min(pi_raw_{i,t}(A_k), pi_cap_hard)

con un cap duro puramente defensivo, por ejemplo:
pi_cap_hard = 5.0

Esto no relaja reglas operativas; solo evita que nombres con ADV casi cero destruyan la agregación o la persistencia.

7.3 Métricas agregadas
El módulo debe reportar, al menos:

- pi_p50(A_k)
- pi_p95(A_k)
- pi_p99(A_k) opcional
- pi_max(A_k)

donde:
pi_p95(A_k) = Percentile_95({pi_{i,t}(A_k)})

8. Días para liquidar

8.1 DTL de trade
DTL_trade_{i,t}(A_k) = q_{i,t}(A_k) / L_{i,t}

8.2 DTL de posición
DTL_pos_{i,t}(A_k) = pos_{i,t}(A_k) / L_{i,t}

8.3 Métricas agregadas
El módulo debe reportar:

- DTL_trade_p95(A_k)
- DTL_trade_max(A_k)
- DTL_pos_p95(A_k)
- DTL_pos_max(A_k)

y opcionalmente p99 si el universo tiene colas muy problemáticas.

9. Modelo de coste

9.1 Coste unitario por orden
El coste total por orden, en bps, debe definirse como:

c_{i,t}(A_k) =
0.5 * spread_{i,t}
+ fee_bps
+ eta * [ min(pi_{i,t}(A_k), 1.0) ]^gamma

con:
- eta > 0
- gamma in [0.5, 1.0]

Esta forma es coherente con impacto cóncavo tipo square-root / power-law.

9.2 Coste medio ponderado por notional
c_bar(A_k) =
sum_{i,t} q_{i,t}(A_k) * c_{i,t}(A_k)
/
sum_{i,t} q_{i,t}(A_k)

9.3 Coste anualizado
Sea TO_ann el turnover anualizado del baseline, medido exactamente igual que en rebalance/backtest.

Entonces:
C_ann(A_k) = TO_ann * c_bar(A_k)

9.4 Retorno neto anualizado
r_net_ann(A_k) =
r_gross_ann - C_ann(A_k) / 10^4

9.5 Retención relativa
Sea el baseline en A_ref:

rho(A_k) =
r_net_ann(A_k) / r_net_ann(A_ref)

Si el denominador es demasiado pequeño o no interpretable, el módulo debe señalarlo explícitamente y no fingir estabilidad de rho.

10. Reglas duras

Un escenario A_k es factible si satisface simultáneamente:

1. pi_p95(A_k) <= pi_p95_max
2. DTL_trade_p95(A_k) <= DTL_trade_max
3. DTL_pos_p95(A_k) <= DTL_pos_max
4. pi_max(A_k) <= pi_single_name_max
5. rho(A_k) >= rho_min
6. r_net_ann(A_k) >= r_net_min

Defaults razonables MVP:
- pi_p95_max = 0.10
- DTL_trade_max = 1.0 día
- DTL_pos_max = 5.0 días
- pi_single_name_max = 0.20
- rho_min = 0.80
- r_net_min = 0.02

11. Precedencia de reglas

La precedencia correcta es:

1. validez de datos
2. liquidez / participación / DTL
3. concentración por nombre
4. retorno neto y retención

Esto implica:
- si fallan datos o liquidez, no tiene sentido discutir retorno,
- y si el baseline es ya infactible, no existe capacidad válida por encima.

12. Regularización monotónica

Por construcción económica, al crecer A_k deben crecer o no disminuir materialmente:
- pi_p95
- DTL_trade_p95
- DTL_pos_p95
- c_bar

Pero el ruido empírico puede romper esto localmente.

Por tanto, antes de aplicar reglas finales, el módulo debe regularizar:

for metric in ["pi_p95", "DTL_trade_p95", "DTL_pos_p95", "avg_cost_bps"]:
    metric = cumulative_max(metric)

o una regularización isotónica equivalente.

La decisión de capacidad se toma después de esta regularización, no antes.

13. Incertidumbre en percentiles

El módulo debe tratar explícitamente que un p95 empírico no es exacto.

MVP recomendable:
calcular intervalos bootstrap por fecha para:

- pi_p95(A_k)
- DTL_trade_p95(A_k)
- DTL_pos_p95(A_k)

Usar resampling por fecha o bloque temporal, no por fila aislada, para respetar dependencia cross-sectional básica del tape.

Salida mínima:
- p95_point
- p95_ci_low
- p95_ci_high

Regla conservadora opcional:
si el upper bound del IC viola una regla dura de forma material, degradar el escenario a FAIL o WARN según config.

14. Stress simple de liquidez

Como extensión inmediata de alto valor y bajo coste, el módulo puede soportar escenarios auxiliares de liquidez:

- ADV x 1.0
- ADV x 0.5
- ADV x 0.2

Esto no sustituye synthetic_shocks, pero ayuda a medir sensibilidad de capacidad a un deterioro simple de liquidez efectiva.

15. Algoritmo operativo

15.1 Validación inicial
Comprobar:
- duplicados por (date, asset_id)
- columnas obligatorias presentes
- spreads no negativos
- A_ref positivo
- scenario_multipliers crecientes
- % de notional con liquidez válida suficiente

Si la cobertura de liquidez válida cae por debajo del umbral configurado, abortar.

15.2 Construcción de escenarios
Para cada A_k:
- escalar órdenes y posiciones
- calcular L_{i,t}
- calcular participación robusta
- calcular DTL trade y DTL pos
- calcular costes
- calcular retorno neto y retención
- agregar métricas

15.3 Regularización monotónica
Aplicar cumulative_max a métricas de fricción.

15.4 Evaluación de reglas
Generar para cada escenario:
- hard_pass
- breach_reason
- first_breach_metric
- first_breach_value
- first_breach_threshold

15.5 Selección final
Regla conservadora:

- si baseline falla:
  baseline_feasible = False
  capacity_limit = null

- si baseline pasa:
  capacity_limit = mayor A_k factible contiguo hacia arriba

Una vez aparece el primer FAIL tras regularización, los superiores quedan descartados por construcción.

16. Invariantes de implementación

El módulo debe garantizar:

- A_{k+1} > A_k
- L_{i,t} >= eps_L > 0
- 0 <= pi_{i,t}(A_k) <= pi_cap_hard
- DTL_trade, DTL_pos >= 0
- avg_cost_bps >= 0
- métricas de fricción no decrecientes tras regularización
- ningún escenario FAIL puede ser seleccionado como capacity_limit
- el baseline siempre se evalúa explícitamente

17. Failure modes importantes

17.1 ADV casi cero
Si no se satura robustamente, explotan participación y DTL y el módulo puede romperse numéricamente.

17.2 Baseline ya infactible
La respuesta correcta no es “capacidad pequeña”, sino:
capacity_limit = null
y explicación explícita de que el baseline no es operable bajo las reglas.

17.3 Denominador de rho mal condicionado
Si r_net_ann(A_ref) es demasiado bajo, la retención puede perder significado.
Debe marcarse.

17.4 No monotonicidad por ruido
La regularización isotónica o cumulative_max no es opcional: es parte central del contrato.

19. Esqueleto de implementación

def usable_liquidity(adv_usd, phi, eps_L):
    return np.maximum(phi * adv_usd, eps_L)


def robust_participation(order_notional, usable_adv, pi_cap_hard):
    pi_raw = order_notional / usable_adv
    return np.minimum(pi_raw, pi_cap_hard)


def impact_cost_bps(spread_bps, fee_bps, eta, gamma, participation):
    return 0.5 * spread_bps + fee_bps + eta * np.power(np.minimum(participation, 1.0), gamma)


def bootstrap_percentile_ci(x, q=0.95, n_boot=1000, alpha=0.05, rng=None):
    # Resampling por fecha o bloque, no por fila suelta
    boot = []
    for _ in range(n_boot):
        sample = resample_by_date(x, rng=rng)
        boot.append(np.quantile(sample, q))
    lo = np.quantile(boot, alpha / 2)
    hi = np.quantile(boot, 1 - alpha / 2)
    return lo, hi


def run_capacity(rebalance_tape, liquidity_pit, config):
    validate_inputs(rebalance_tape, liquidity_pit, config)

    scenarios = []
    for m in config["scenario_multipliers"]:
        A_k = m * config["aum_reference"]

        merged = build_scenario_frame(rebalance_tape, liquidity_pit, A_k, config)

        metrics = compute_capacity_metrics(merged, config)
        scenarios.append(metrics)

    scenarios = enforce_monotonicity(scenarios)
    scenarios = evaluate_hard_rules(scenarios, config)

    summary = select_capacity_limit(scenarios, config)
    persist_outputs(scenarios, summary, config)

    return scenarios, summary

20. MVP vs versión avanzada

20.1 MVP
- rejilla fija de AUM
- phi constante
- eps_L fijo
- impacto cóncavo simple
- p95 + IC bootstrap
- regularización monotónica
- reglas duras con precedencia estricta

20.2 Versión avanzada
- phi por bucket de liquidez
- escenarios ADV stress 0.5x / 0.2x
- p99 adicional
- stress conjunto con synthetic_shocks
- sensibilidad explícita de capacity_limit a phi, eta y gamma

21. Interpretación correcta

La lectura correcta de capacity.py no es:
“¿cuánto capital podríamos gestionar si el mercado nos dejara?”

La lectura correcta es:
“¿hasta qué AUM podemos mantener esta estrategia, con esta lógica y este tape de rebalanceos, sin romper nuestras restricciones de liquidez y sin destruir demasiado el retorno neto?”

Eso convierte la capacidad en una magnitud operativa, auditable y útil para despliegue real.

22. Referencias orientativas

- Almgren y Chriss, por estructura de costes e impacto
- Gatheral y literatura de impact models, por ley cóncava
- López de Prado, por lógica PIT y medición condicional de capacidad

23. Conclusión

portfolio/capacity.py debe ser un módulo severo y explícito.

No debe intentar modelar toda la microestructura del mercado.
Debe hacer algo más útil:

tomar el tape real de la estrategia, escalarlo bajo distintos AUM, medir fricciones y límites operativos, imponer monotonicidad, y devolver el mayor tamaño de capital compatible con disciplina de ejecución y preservación razonable del edge neto.
