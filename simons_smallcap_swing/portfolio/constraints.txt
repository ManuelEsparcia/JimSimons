Especificación Cuantitativa: portfolio/constraints.py

1. Propósito del módulo

El objetivo de portfolio/constraints.py es transformar un vector de pesos preliminares

    w_pre in R^N

en un vector de pesos implementable y auditado

    w_constr in R^N

que respete simultáneamente las restricciones operativas y de riesgo definidas para la fecha t.

La función del módulo no es mejorar la señal ni reinterpretar el alpha.
Su cometido es estrictamente geométrico y operativo:

“dado un portafolio preliminar y un conjunto de límites duros, encontrar el portafolio factible más cercano bajo la métrica canónica del sistema.”

2. Principios no negociables

2.1 Contrato cerrado
Entradas, salidas, invariantes, política de infactibilidad y tolerancias deben estar definidos ex ante.

2.2 Método canónico único
La implementación por defecto no será una cascada heurística abierta, sino una única proyección convexa con restricciones explícitas.

2.3 Separación de responsabilidades
constraints.py no genera alpha, no hace sizing desde cero y no sustituye al optimizador global del portfolio si este ya existe.

2.4 Point-in-time
Todos los datos de universo, exposición, liquidez y tradabilidad deben ser PIT.

2.5 No compensabilidad
Ninguna restricción dura violada puede quedar “rescatada” por una distancia pequeña a w_pre.

2.6 Auditabilidad
Toda modificación relevante del portafolio debe quedar trazada before/after y con identificación de restricciones activas.

3. Qué hace y qué no hace

3.1 Qué hace
- impone límites de posición
- impone límites gross y net
- impone límites por grupo
- impone límites por factor
- impone restricciones de turnover
- impone restricciones de liquidez individual
- y devuelve el vector factible más cercano a w_pre

3.2 Qué no hace
- no genera la señal
- no recalibra ADV ni costes
- no neutraliza exposiciones salvo que estén expresadas como restricciones explícitas
- no optimiza alpha+risk de forma global salvo que se use como proyector final dentro de otra arquitectura

4. Entradas obligatorias

El snapshot temporal t debe aportar como mínimo:

4.1 Pesos
- w_pre
- w_current
- eligible_flag

4.2 Exposiciones
- sector_id
- industry_id
- factor_matrix X in R^(N x K)
- exposiciones continuas opcionales: beta, size, value, momentum, volatility, etc.

4.3 Liquidez
- price
- adv_usd
- spread_bps
- halt_flag
- lot_size si aplica

4.4 Configuración
Debe venir desde config, no desde constantes internas:
- gross_max
- net_min, net_max
- pos_lower, pos_upper
- turnover_max
- adv_participation_cap
- dtl_max si aplica
- group_bounds
- factor_bounds
- distance_metric
- infeasibility_policy
- solver_name
- solver_tolerances
- liquidity_fraction phi
- aum_reference o AUM operativo

5. Salidas obligatorias

El módulo devuelve:

1. w_constr
2. constraint_report.json
3. before_after.parquet
4. status in {optimal, optimal_inaccurate, infeasible, fail_postcheck}

6. Notación

Sea N el número de activos elegibles en t.

- w in R^N: vector candidato
- w_pre in R^N: vector preliminar
- w_cur in R^N: vector actual

Definimos:

Gross:
    G(w) = sum_i |w_i|

Net:
    N(w) = sum_i w_i

Turnover:
    TO(w, w_cur) = sum_i |w_i - w_cur_i|

Exposición de grupo g:
    E_g(w) = sum_{i in G_g} w_i

Exposición factorial:
    f(w) = X^T w

7. Restricciones duras canónicas

7.1 Límites por activo
Para cada i:
    l_i <= w_i <= u_i

7.2 Gross
    sum_i |w_i| <= G_max

7.3 Net
    N_min <= sum_i w_i <= N_max

7.4 Grupos
Para cada grupo g:
    L_g <= E_g(w) <= U_g

7.5 Factores
Para cada factor k:
    L_k^(f) <= (X^T w)_k <= U_k^(f)

7.6 Turnover
    sum_i |w_i - w_cur_i| <= TO_max

7.7 Liquidez individual
Sea A el AUM operativo y phi la fracción de ADV utilizable.
El notional negociado por nombre es:

    q_i(w) = A * |w_i - w_cur_i|

La participación implícita es:

    pi_i(w) = q_i(w) / (phi * ADV_i_usd)

Restricción:
    pi_i(w) <= pi_max     para todo i

Equivalente:
    |w_i - w_cur_i| <= (pi_max * phi * ADV_i_usd) / A

7.8 DTL opcional
Si se activa control por días para liquidar:

    DTL_i(w) = A * |w_i| / (phi * ADV_i_usd) <= d_max

8. Problema matemático canónico

El método por defecto del módulo es la proyección convexa:

    minimize    1/2 (w - w_pre)^T D (w - w_pre)
    subject to  todas las restricciones duras simultáneamente

donde D es diagonal positiva definida.

Default recomendado:
    D_ii = 1 / (sigma_i + eps)^2

Si no hay volatilidades fiables:
    D = I

Interpretación:
- el portafolio final es el punto factible más cercano a w_pre
- la solución es única si D es definida positiva y el conjunto factible es no vacío
- las restricciones definen factibilidad; el objetivo solo elige el mejor punto dentro de ella

9. Reformulación convexa explícita

Para evitar ambigüedad, el módulo debe explicitar la reformulación de términos absolutos.

9.1 Gross
Introducir z_i >= 0 con:
    z_i >= w_i
    z_i >= -w_i
    sum_i z_i <= G_max

9.2 Turnover
Introducir v_i >= 0 con:
    v_i >= w_i - w_cur_i
    v_i >= w_cur_i - w_i
    sum_i v_i <= TO_max

9.3 Liquidez
Definir:
    cap_i = (pi_max * phi * ADV_i_usd) / A

Entonces:
    w_i - w_cur_i <= cap_i
    w_cur_i - w_i <= cap_i

9.4 DTL opcional
Definir:
    pos_cap_i = (d_max * phi * ADV_i_usd) / A

Entonces:
    -pos_cap_i <= w_i <= pos_cap_i

10. Precedencia conceptual

La precedencia correcta no es secuencial, sino semántica:

1. validez de datos y elegibilidad
2. límites por activo
3. liquidez y turnover
4. gross y net
5. grupos y factores
6. criterio de proximidad a w_pre

Las primeras cinco capas definen el conjunto factible.
La sexta elige un punto dentro de él.

11. Política de infactibilidad

11.1 Política por defecto
Default:
    infeasibility_policy = strict_fail

Si el problema es infactible:
- status = infeasible
- w_constr = None
- el módulo no fabrica una cartera heurística silenciosa
- se persiste diagnóstico explícito

11.2 Relajación opcional
Solo si config lo activa:
    infeasibility_policy = prioritized_relaxation

Orden canónico de relajación:
1. factores secundarios
2. grupos secundarios
3. turnover
4. net estrecho
5. gross

Nunca deben relajarse silenciosamente:
- elegibilidad
- halt_flag
- límites hard por activo
- restricciones de liquidez individuales

Toda relajación debe quedar cuantificada y marcada.

12. Selección de solver

12.1 Regla simple
- N pequeño o problema denso: ECOS
- N medio/grande y estructura sparse QP: OSQP
- caso muy grande / first-order tolerante: SCS

12.2 Recomendación práctica
- ECOS: bueno para problemas pequeños y relativamente densos
- OSQP: default recomendado para QP sparse con turnover y muchas restricciones lineales
- SCS: fallback escalable si el resto falla o el problema es muy grande

12.3 No-go rule
El solver no puede elegirse de forma opaca.
Debe registrarse en la salida.

13. Validación inicial de datos

Antes de resolver, el módulo debe comprobar:

- índices alineados entre w_pre, w_cur, X, grupos y liquidez
- ausencia de duplicados por activo y fecha
- eligible_flag consistente
- halt_flag consistente
- ADV > 0 si liquidez activa
- spreads no negativos
- bounds por activo presentes
- matriz X compatible dimensionalmente
- grupo por activo bien definido si grupos activos

Cualquier fallo aquí debe abortar explícitamente.

14. Resolución

El módulo construye el problema convexo, lo resuelve con el solver configurado y registra:

- solver usado
- tiempo de resolución
- status del solver
- valor objetivo
- active constraints
- slacks
- violación máxima por familia de restricciones

15. Post-checks numéricos

Aunque el solver devuelva optimal, el módulo debe verificar numéricamente todas las restricciones.

Ejemplos:

Posición:
    max_i max(0, l_i - w_i, w_i - u_i) <= eps_solver

Gross:
    max(0, G(w) - G_max) <= eps_solver

Net:
    max(0, N_min - N(w), N(w) - N_max) <= eps_solver

y análogos para grupos, factores, turnover, liquidez y DTL.

Tolerancias recomendadas por solver:
- ECOS: 1e-8
- OSQP: 1e-6
- SCS: 1e-5

Si una violación supera tolerancia:
- degradar a optimal_inaccurate
o
- fail_postcheck si es material

16. Before/after obligatorio

El reporte debe incluir, como mínimo:

- gross_before / gross_after
- net_before / net_after
- turnover_before / turnover_after
- max_pos_before / max_pos_after
- liquidity_breach_count_before / after
- group_breach_count_before / after
- factor_breach_count_before / after
- distance_to_pre
- active_constraints
- binding_constraints
- slacks por familia

17. Invariantes del módulo

Toda salida válida debe cumplir:

1. mismo universo e índice que la entrada elegible
2. ningún activo no elegible o suspendido recibe peso operativo
3. todas las restricciones duras satisfechas dentro de tolerancia
4. si w_pre ya es factible, entonces w_constr = w_pre salvo error numérico mínimo
5. strict_fail nunca devuelve una cartera infactible disfrazada

18. Fallback heurístico

Puede existir un fallback solo para contingencias de solver, nunca como método principal.

Orden permitido:
1. clip por activo
2. clip por liquidez
3. rescale gross/net
4. ajuste proyectivo de grupos/factores
5. postcheck completo

Si se usa:
    used_fallback = True

Debe quedar explícitamente marcado en el reporte.

19. Tests mínimos obligatorios

19.1 Factibilidad trivial
Si w_pre es factible:
    w_constr ≈ w_pre

19.2 Bound por activo
Un solo nombre fuera de rango debe pegarse al bound correcto.

19.3 Liquidez
Si un activo excede pi_max, el resultado debe reducirlo o declarar infactibilidad.

19.4 Turnover
Si turnover excede el máximo, la solución debe ser la proyección factible más cercana.

19.5 Grupos
Si un grupo excede rango, la cartera final debe reintroducir factibilidad.

19.6 Infactibilidad real
Si gross, net y box son incompatibles:
    status = infeasible bajo strict_fail

19.7 Auditoría
constraint_report.json debe contener todas las familias de constraints y sus slacks.

20. Integración con el pipeline

La semántica correcta es:

signal
-> sizing
-> neutralize (si aplica)
-> constraints
-> rebalance / execution

Si existe optimizer global que ya resuelve alpha, riesgo y restricciones de una vez, entonces constraints.py debe actuar como:
- capa de validación final,
- o projector de seguridad antes de ejecución,
pero no duplicar innecesariamente la lógica de optimización.

21. Salida recomendada

21.1 constraint_report.json
Debe incluir:
- status
- solver_name
- solve_time
- objective_value
- active_constraints
- slacks
- max_violation_by_family
- used_fallback
- infeasibility_reason si aplica

21.2 before_after.parquet
Debe incluir, por activo y agregado:
- w_pre
- w_constr
- delta_abs
- liquidity_caps
- group exposures
- factor exposures
- flags de restricción activa

22. Defaults razonables

- distance_metric = weighted_l2
- solver_name = OSQP
- infeasibility_policy = strict_fail
- adv_participation_cap = 10%
- dtl_max = 5 días
- solver_tolerances versionadas por solver

Los límites de gross, net, posición, turnover, grupo y factor nunca deben ir hardcodeados en el módulo.

23. Interpretación correcta

La lectura correcta de constraints.py no es:
“vamos arreglando el portfolio paso a paso hasta que más o menos encaje”.

La lectura correcta es:
“definimos formalmente el conjunto factible y proyectamos sobre él el portafolio preliminar con una métrica única, reproducible y auditada”.

Eso convierte el módulo en una pieza:
- testeable,
- matemáticamente limpia,
- y operativamente robusta.

24. Referencias orientativas

- Boyd y Vandenberghe, por optimización convexa
- documentación de CVXPY, por reformulación estándar de valores absolutos
- OSQP paper, por QP sparse operativo
- literatura estándar de portfolio constraints y projection methods

25. Conclusión

portfolio/constraints.py debe ser una especificación cerrada de factibilidad, no un conjunto abierto de arreglos heurísticos.

Recibe un portafolio preliminar.
Define un conjunto factible con restricciones duras explícitas.
Proyecta de forma convexa al punto más cercano.
Maneja la infactibilidad con política clara.
Y deja un rastro auditable de cada concesión impuesta al alpha preliminar.

Ese es exactamente el comportamiento que debe tener una capa seria de constraints en un stack cuantitativo institucional.
