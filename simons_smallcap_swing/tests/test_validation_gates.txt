\section{Propósito del Módulo}

El objetivo de \texttt{tests/test\_validation\_gates.py} es verificar que la capa de validación final del pipeline transforma de forma correcta un conjunto de reportes de control en una decisión operativa coherente: \texttt{PASS}, \texttt{WARN} o \texttt{FAIL}.

Este módulo no evalúa si un modelo es rentable, ni si una métrica concreta es buena en términos absolutos. Su función es más estructural: asegurar que, dados unos resultados de validación ya calculados, el motor de gates aplica correctamente la semántica de severidad, prioridad, agregación y bloqueo.

La pregunta que debe responder esta suite es concreta:

\begin{quote}
\emph{Dado un conjunto de gates con severidad, estado, evidencia y artefactos requeridos, ¿la decisión final del sistema coincide exactamente con la política declarada?}
\end{quote}

Si esta capa falla, el sistema puede:
\begin{itemize}
    \item desplegar modelos que debían quedar bloqueados,
    \item bloquear ejecuciones válidas por una mala agregación,
    \item generar decisiones no reproducibles,
    \item o producir auditorías incompletas del proceso de validación.
\end{itemize}

\section{Principios de Diseño}

Este módulo debe escribirse como una especificación de gobernanza de decisión, no como una narrativa de performance. Sus principios son:

\begin{enumerate}
    \item \textbf{Severidad explícita}: cada gate debe tener una clase de severidad bien definida.
    \item \textbf{Prioridad determinista}: la combinación de resultados debe resolverse con reglas únicas y estables.
    \item \textbf{Separación entre métrica y política}: el cálculo de la métrica pertenece a otros módulos; aquí se testa la lógica de decisión.
    \item \textbf{Auditabilidad}: la decisión final debe poder explicarse a partir del conjunto de gates observados.
    \item \textbf{Fail-fast en ausencia crítica}: si faltan artefactos o reportes esenciales, el sistema debe fallar según contrato.
\end{enumerate}

\section{Modelo Conceptual de un Gate}

Un gate representa una restricción operacional evaluada sobre una ejecución del pipeline. Un gate bien formado debe contener, como mínimo:

\begin{itemize}
    \item \texttt{name}: nombre único del gate,
    \item \texttt{status}: resultado observado,
    \item \texttt{severity}: nivel de severidad,
    \item \texttt{critical}: indicador equivalente si el diseño lo usa,
    \item \texttt{observed\_value}: valor observado, si aplica,
    \item \texttt{threshold}: umbral o regla aplicada, si aplica,
    \item \texttt{message}: explicación legible del resultado,
    \item \texttt{artifact\_ref}: referencia al artefacto o evidencia subyacente, si aplica.
\end{itemize}

Una representación mínima y robusta es:

\begin{lstlisting}
gate = {
    "name": "leakage_check",
    "status": "FAIL",
    "severity": "CRITICAL",
    "observed_value": 0.003,
    "threshold": 0.0,
    "message": "Leakage temporal detectado",
    "artifact_ref": "validation/leakage_report.json",
}
\end{lstlisting}

\section{Espacio de Estados del Gate}

Para evitar ambigüedad, el sistema debe fijar un conjunto pequeño de estados permitidos. Una convención razonable es:

\[
\mathcal{S} = \{\texttt{PASS}, \texttt{WARN}, \texttt{FAIL}, \texttt{MISSING}\}
\]

y un conjunto de severidades:

\[
\mathcal{V} = \{\texttt{INFO}, \texttt{WARNING}, \texttt{CRITICAL}\}
\]

La decisión global del sistema debe derivarse de la combinación de ambos. Por ejemplo, un gate con \texttt{status=FAIL} y \texttt{severity=CRITICAL} tiene efecto bloqueante; un gate con \texttt{status=WARN} y \texttt{severity=WARNING} no necesariamente bloquea, pero debe elevar la salida global a \texttt{WARN} si no existe ningún fallo crítico.

\section{Regla de Decisión Global}

Sea \(G = \{g_1,\dots,g_n\}\) el conjunto de gates evaluados. Una política determinista y fácilmente testeable es:

\[
\mathrm{Decision}(G)=
\begin{cases}
\texttt{FAIL}, & \text{si existe } g_i \text{ con } \mathrm{status}(g_i)=\texttt{FAIL}
\text{ y } \mathrm{severity}(g_i)=\texttt{CRITICAL},\\[6pt]
\texttt{WARN}, & \text{si no existe fallo crítico y existe } g_i \text{ con }
\mathrm{status}(g_i)\in\{\texttt{WARN},\texttt{FAIL}\}
\text{ no crítico},\\[6pt]
\texttt{PASS}, & \text{en otro caso.}
\end{cases}
\]

Esta formulación puede adaptarse al repositorio, pero el punto clave es que la prioridad debe ser única y comprobable.

\subsection{Orden de precedencia}

La precedencia natural es:

\[
\texttt{FAIL} \succ \texttt{WARN} \succ \texttt{PASS}
\]

donde \(\succ\) significa ``domina a''. Si existe una sola razón bloqueante, la salida global debe ser \texttt{FAIL} independientemente del resto de gates en verde.

\section{Artefactos Críticos y Fallo por Ausencia}

No todos los gates se basan en una métrica numérica. Algunos verifican la existencia de artefactos indispensables para confiar en la ejecución:

\begin{itemize}
    \item reporte de leakage,
    \item reporte PIT,
    \item snapshot de configuración,
    \item manifest de features,
    \item labels validadas,
    \item reportes de backtest o validation summary.
\end{itemize}

Si un artefacto declarado como obligatorio no existe, el sistema no debe degradarse a \texttt{WARN} por defecto. Debe aplicar la política explícita del pipeline, que en un diseño serio suele ser:

\[
\texttt{artifact missing} \land \texttt{artifact required}
\Rightarrow \texttt{FAIL}
\]

\section{Casos Mínimos que Debe Cubrir el Módulo}

\subsection{\texttt{test\_validation\_pass\_when\_all\_critical\_gates\_pass}}

Verifica que la decisión global es \texttt{PASS} cuando todos los gates críticos y no críticos relevantes pasan.

\paragraph{Qué protege}
\begin{itemize}
    \item agregación errónea que promueve un falso warning,
    \item defaults conservadores mal aplicados,
    \item confusión entre gate informativo y gate bloqueante.
\end{itemize}

\subsection{\texttt{test\_validation\_fail\_on\_critical\_gate}}

Verifica que un único fallo crítico fuerza la decisión global a \texttt{FAIL}.

\paragraph{Qué protege}
\begin{itemize}
    \item pérdida de prioridad de severidad,
    \item agregación promedio que diluye un bloqueo,
    \item resolución incorrecta cuando coexisten gates en verde y un crítico rojo.
\end{itemize}

\subsection{\texttt{test\_validation\_warn\_on\_non\_critical\_borderline}}

Verifica que un gate no crítico en estado de advertencia produce salida \texttt{WARN} cuando no existen fallos críticos.

\paragraph{Qué protege}
\begin{itemize}
    \item promoción indebida a \texttt{FAIL},
    \item omisión de advertencias relevantes,
    \item colapso erróneo de \texttt{WARN} a \texttt{PASS}.
\end{itemize}

\subsection{\texttt{test\_missing\_artifact\_critical\_gate\_fails}}

Verifica que la ausencia de un artefacto obligatorio provoca \texttt{FAIL} si el contrato lo define como crítico.

\paragraph{Qué protege}
\begin{itemize}
    \item despliegues sin evidencia mínima,
    \item falsos positivos en pipelines incompletas,
    \item decisiones tomadas con reportes parciales.
\end{itemize}

\subsection{\texttt{test\_gate\_priority\_and\_decision\_consistency}}

Verifica que la prioridad global se resuelve de forma determinista cuando coexisten múltiples gates de distinto tipo.

\paragraph{Qué protege}
\begin{itemize}
    \item dependencia del orden de iteración del diccionario,
    \item resultados diferentes según orden de los reportes,
    \item ambigüedad al combinar \texttt{WARN} y \texttt{FAIL}.
\end{itemize}

\section{Reglas Formales de Consistencia}

El motor de decisión debe satisfacer al menos las siguientes propiedades.

\subsection{Idempotencia respecto al orden}

Si \(\pi(G)\) es cualquier permutación del conjunto de gates, entonces:

\[
\mathrm{Decision}(G) = \mathrm{Decision}(\pi(G))
\]

La decisión no puede depender del orden de serialización o de lectura de los reportes.

\subsection{Monotonicidad por empeoramiento}

Si un gate cambia de \texttt{PASS} a \texttt{WARN}, o de \texttt{WARN} a \texttt{FAIL} manteniendo la misma severidad o una severidad mayor, la decisión global no puede mejorar. Formalmente, si \(G'\) es un conjunto ``peor'' que \(G\), entonces:

\[
\mathrm{Decision}(G') \preceq \mathrm{Decision}(G)
\]

donde \(\preceq\) sigue el orden:

\[
\texttt{FAIL} \preceq \texttt{WARN} \preceq \texttt{PASS}
\]

\subsection{Dominancia de fallos críticos}

Si existe al menos un gate crítico fallido, ningún número de gates positivos puede compensarlo:

\[
\exists g_i \in G:\ 
\mathrm{severity}(g_i)=\texttt{CRITICAL}
\land
\mathrm{status}(g_i)=\texttt{FAIL}
\Rightarrow
\mathrm{Decision}(G)=\texttt{FAIL}
\]

\section{Implementación de Referencia}

Una implementación sobria del motor de decisión puede escribirse así:

\begin{lstlisting}
def aggregate_validation_decision(gate_reports):
    has_critical_fail = any(
        g["severity"] == "CRITICAL" and g["status"] in {"FAIL", "MISSING"}
        for g in gate_reports
    )
    if has_critical_fail:
        return "FAIL"

    has_warn = any(
        (
            g["status"] == "WARN"
            or (g["status"] == "FAIL" and g["severity"] != "CRITICAL")
        )
        for g in gate_reports
    )
    if has_warn:
        return "WARN"

    return "PASS"
\end{lstlisting}

La clave aquí no es que esta sea la única política posible, sino que:
\begin{itemize}
    \item la política sea única,
    \item el test la fije,
    \item y los estados especiales como \texttt{MISSING} estén resueltos explícitamente.
\end{itemize}

\section{Ejemplos de Tests Sobrios}

\subsection{Todo pasa}

\begin{lstlisting}
def test_validation_pass_when_all_critical_gates_pass(
    synthetic_gate_reports_pass,
):
    decision = aggregate_validation_decision(synthetic_gate_reports_pass)
    assert decision == "PASS"
\end{lstlisting}

\subsection{Fallo crítico}

\begin{lstlisting}
def test_validation_fail_on_critical_gate(synthetic_gate_reports_fail):
    decision = aggregate_validation_decision(synthetic_gate_reports_fail)
    assert decision == "FAIL"
\end{lstlisting}

\subsection{Advertencia no crítica}

\begin{lstlisting}
def test_validation_warn_on_non_critical_borderline(
    synthetic_gate_reports_warn,
):
    decision = aggregate_validation_decision(synthetic_gate_reports_warn)
    assert decision == "WARN"
\end{lstlisting}

\subsection{Artefacto obligatorio ausente}

\begin{lstlisting}
def test_missing_artifact_critical_gate_fails(
    synthetic_gate_reports_missing_artifact,
):
    decision = aggregate_validation_decision(
        synthetic_gate_reports_missing_artifact
    )
    assert decision == "FAIL"
\end{lstlisting}

\subsection{Independencia del orden}

\begin{lstlisting}
def test_gate_priority_and_decision_consistency():
    reports_a = [
        {"name": "coverage", "status": "WARN", "severity": "WARNING"},
        {"name": "leakage", "status": "FAIL", "severity": "CRITICAL"},
    ]
    reports_b = list(reversed(reports_a))

    decision_a = aggregate_validation_decision(reports_a)
    decision_b = aggregate_validation_decision(reports_b)

    assert decision_a == "FAIL"
    assert decision_b == "FAIL"
    assert decision_a == decision_b
\end{lstlisting}

\section{Schema del Reporte de Validación}

Además de la decisión final, el sistema suele persistir un resumen consolidado. Este módulo puede verificar que el reporte final tiene un schema estable. Una estructura razonable es:

\begin{lstlisting}
validation_manifest = {
    "run_id": "run_20260306_001",
    "decision": "FAIL",
    "generated_at": "2026-03-06T10:15:00Z",
    "gates": [
        {
            "name": "leakage_check",
            "status": "FAIL",
            "severity": "CRITICAL",
            "message": "Leakage detectado",
            "artifact_ref": "validation/leakage_report.json",
        }
    ],
}
\end{lstlisting}

El contrato mínimo del manifest debería fijar:
\begin{itemize}
    \item unicidad de \texttt{run\_id},
    \item presencia de \texttt{decision},
    \item listado completo de gates evaluados,
    \item mensajes legibles,
    \item referencias reproducibles a la evidencia.
\end{itemize}

\section{Casos Borde que Deben Añadirse}

Para que el módulo suba realmente de nivel, conviene cubrir además los siguientes escenarios.

\subsection{Duplicado de nombre de gate}

Si dos reportes comparten el mismo \texttt{name}, el motor debe:
\begin{itemize}
    \item prohibirlo,
    \item o resolverlo con una política explícita.
\end{itemize}

La suite debe verificar esa decisión, porque los duplicados rompen la auditabilidad.

\subsection{Estado desconocido}

Si aparece un \texttt{status} fuera del espacio permitido, como \texttt{"BORDERLINE"} o \texttt{"UNKNOWN"}, el sistema no debe degradarse silenciosamente. Debe fallar o normalizarlo según contrato.

\subsection{Severidad ausente}

Un gate sin severidad no es operacionalmente interpretable. El test debe verificar si eso:
\begin{itemize}
    \item provoca error,
    \item se trata como crítico por defecto,
    \item o se rechaza en validación de schema.
\end{itemize}

\subsection{Lista vacía de gates}

La política para \(G=\varnothing\) debe ser explícita. En un diseño serio, una ausencia total de validación rara vez debería traducirse automáticamente en \texttt{PASS}. Este caso merece un test específico.

\subsection{Fail no crítico}

Algunos diseños permiten gates con \texttt{status=FAIL} pero severidad no crítica, cuyo efecto global es \texttt{WARN}. Si esa semántica existe, debe estar testeada y no quedar implícita.

\section{Separación entre Umbrales y Lógica de Agregación}

Una debilidad habitual de estos documentos es mezclar dos capas distintas:

\begin{enumerate}
    \item \textbf{qué threshold usa cada métrica};
    \item \textbf{cómo se agrega el resultado de cada gate a una decisión global}.
\end{enumerate}

Este archivo debe centrarse en la segunda. Los umbrales concretos de Sharpe, coverage, leakage o drift pueden variar por estrategia, mercado, horizonte, universo o fase de investigación. Por eso no deben codificarse aquí como si fueran universales. Lo que sí debe ser universal dentro del repositorio es la semántica de la decisión una vez que cada gate ya viene evaluado.

\section{Qué No Debe Hacer Este Módulo}

Para mantener nivel alto, \texttt{test\_validation\_gates.py} no debe:

\begin{itemize}
    \item justificar la calidad del sistema por umbrales decorativos de Sharpe o IC,
    \item presentar thresholds concretos como verdades generales,
    \item mezclar la lógica de gates con entrenamiento, tuning o despliegue real,
    \item convertir el test en una discusión de MLOps genérico,
    \item depender de herramientas externas cuando bastan estructuras sintéticas mínimas.
\end{itemize}

El lenguaje correcto aquí es el de política de decisión, precedencia, evidencia y consistencia.

\section{Definition of Done}

El módulo puede considerarse sólido si cumple simultáneamente lo siguiente:

\begin{enumerate}
    \item Verifica que todos los gates en verde producen \texttt{PASS}.
    \item Verifica que un único fallo crítico produce \texttt{FAIL}.
    \item Verifica que advertencias no críticas producen \texttt{WARN} cuando no existen fallos críticos.
    \item Verifica que la ausencia de artefactos obligatorios se trata según contrato y, si es crítica, bloquea la ejecución.
    \item Verifica que la decisión no depende del orden de los reportes.
    \item Verifica el schema mínimo del manifest de validación.
    \item Verifica manejo explícito de estados desconocidos, severidades ausentes o duplicados.
    \item Falla con mensajes que identifiquen exactamente qué regla de decisión fue violada.
\end{enumerate}

\section{Conclusión}

\texttt{tests/test\_validation\_gates.py} debe entenderse como una capa de gobernanza determinista del pipeline. No prueba si un modelo es bueno; prueba si el sistema sabe decidir correctamente cuándo una ejecución queda autorizada, advertida o bloqueada.

En un stack cuantitativo serio, esa distinción es esencial. Las métricas pueden cambiar con el tiempo y los thresholds pueden recalibrarse, pero la semántica de prioridad, evidencia mínima y bloqueo no debe quedar ambigua. Por eso este módulo debe redactarse como una especificación de decisión auditable: estados bien definidos, prioridades únicas, artefactos requeridos, reglas de agregación explícitas y fallos imposibles de malinterpretar.
