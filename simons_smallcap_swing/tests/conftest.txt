\section{Propósito del Módulo}

El archivo \texttt{tests/conftest.py} define la infraestructura común de la suite de tests. Su función no es validar señales, rentabilidades ni hipótesis de mercado, sino proporcionar un entorno de prueba \emph{determinista, aislado, rápido y auditable} sobre el que puedan construirse tests fiables para backtesting, datasets point-in-time, features, labels y validación de pipeline.

En una arquitectura cuantitativa seria, \texttt{conftest.py} cumple cuatro objetivos:

\begin{enumerate}
    \item \textbf{Reproducibilidad}: toda fixture debe producir exactamente el mismo resultado bajo la misma semilla y los mismos parámetros.
    \item \textbf{Aislamiento}: ningún test debe depender de estado global, red, reloj del sistema, ficheros persistentes ni artefactos creados por otros tests.
    \item \textbf{Componibilidad}: las fixtures deben poder combinarse de forma limpia para construir escenarios más complejos sin duplicar lógica.
    \item \textbf{Legibilidad operativa}: la suite debe expresar con claridad qué datos mínimos necesita cada test y qué supuestos estructurales están siendo impuestos.
\end{enumerate}

\section{Principios de Diseño}

Las fixtures de este módulo se diseñan bajo principios de ingeniería de tests, no de modelado predictivo:

\begin{itemize}
    \item \textbf{Datos mínimos suficientes}: se generan paneles pequeños pero estructuralmente válidos.
    \item \textbf{Determinismo fuerte}: toda aleatoriedad usa un generador explícito con semilla fija.
    \item \textbf{Semántica estable}: las columnas, índices, claves primarias y tipos deben permanecer controlados.
    \item \textbf{No I/O externo}: ningún test debe depender de APIs, bases de datos, S3, ficheros de mercado ni servicios de terceros.
    \item \textbf{Escopos correctos}: \texttt{function}, \texttt{module} o \texttt{session} se eligen por coste, mutabilidad y riesgo de contaminación.
    \item \textbf{Fallos informativos}: cuando una fixture no puede construirse, el error debe exponer qué contrato se incumplió.
\end{itemize}

\section{Alcance de las Fixtures Centrales}

\subsection{\texttt{tmp\_project\_dir}}

Fixture de sistema de ficheros temporal para tests que requieran escribir artefactos efímeros.

\paragraph{Contrato}
\begin{itemize}
    \item Debe crear un directorio vacío y exclusivo por test.
    \item Debe limpiarse automáticamente al finalizar.
    \item Ningún path generado por esta fixture debe sobrevivir entre ejecuciones.
\end{itemize}

\paragraph{Uso esperado}
Tests de serialización, escritura de reportes, snapshots, logs temporales, cachés locales o artefactos intermedios de pipeline.

\subsection{\texttt{sample\_run\_context}}

Fixture que representa un contexto mínimo de ejecución.

\paragraph{Contenido típico}
\begin{itemize}
    \item \texttt{run\_id}
    \item \texttt{seed}
    \item \texttt{asof\_date}
    \item \texttt{train\_start}, \texttt{train\_end}
    \item parámetros de configuración mínimos
\end{itemize}

\paragraph{Contrato}
\begin{itemize}
    \item Debe ser hashable o serializable si el sistema lo requiere.
    \item Debe fijar una semilla explícita.
    \item Debe evitar timestamps dinámicos no controlados salvo que el test los congele.
\end{itemize}

\subsection{\texttt{synthetic\_price\_panel}}

Fixture con un panel sintético de precios OHLCV para varios símbolos y varias fechas.

\paragraph{Objetivo}
No pretende replicar fielmente la microestructura real del mercado, sino ofrecer una base consistente para tests de joins, retornos, accounting, ventanas temporales y validación de esquemas.

\paragraph{Contrato mínimo}
\begin{itemize}
    \item Índice temporal monótono creciente.
    \item Universo de símbolos fijo y explícito.
    \item Columnas OHLCV bien formadas.
    \item Restricciones lógicas:
    \[
    \mathrm{low}_t \le \min(\mathrm{open}_t,\mathrm{close}_t) \le
    \max(\mathrm{open}_t,\mathrm{close}_t) \le \mathrm{high}_t
    \]
    \item Volumen no negativo.
    \item Ausencia de valores infinitos.
\end{itemize}

\subsection{\texttt{synthetic\_feature\_panel}}

Fixture con features derivadas o prefabricadas para tests de schema y alineación temporal.

\paragraph{Contrato}
\begin{itemize}
    \item Debe compartir clave primaria con el panel base esperado por el sistema, típicamente \texttt{(date, symbol)}.
    \item Debe incluir un subconjunto pequeño pero representativo de features numéricas.
    \item Debe permitir inyectar \texttt{NaN}, \texttt{Inf}, columnas faltantes o dtypes erróneos cuando el test lo necesite.
\end{itemize}

\subsection{\texttt{synthetic\_label\_panel}}

Fixture con labels forward construidas de forma controlada.

\paragraph{Contrato}
Para un horizonte \(h\), una construcción canónica es:
\[
y_t = f(P_{t+h}, P_t),
\]
donde \(f\) puede ser retorno simple, retorno logarítmico o etiqueta binaria derivada. Lo esencial es que:
\begin{itemize}
    \item la etiqueta dependa solo de información futura respecto a \(t\),
    \item las últimas \(h\) observaciones inválidas queden explícitamente marcadas,
    \item exista alineación exacta entre decisión, features y label.
\end{itemize}

\subsection{\texttt{synthetic\_splits}}

Fixture con particiones temporales de train/validation/test.

\paragraph{Contrato}
\begin{itemize}
    \item Las particiones deben respetar orden temporal.
    \item No puede existir solape indebido entre bloques si el módulo exige purgado o embargo.
    \item Deben poder generarse versiones mínimas para tests rápidos.
\end{itemize}

\section{Modelo de Datos Canónico para la Suite}

Para maximizar interoperabilidad entre tests, conviene que las fixtures adopten una representación canónica. Una opción robusta es:

\begin{itemize}
    \item \textbf{Panel largo} con índice \texttt{(date, symbol)}
    \item \textbf{Fechas} en \texttt{DatetimeIndex} normalizado
    \item \textbf{Símbolos} como cadenas estables
    \item \textbf{Columnas numéricas} en \texttt{float64}/\texttt{int64} salvo motivo expreso
\end{itemize}

Esto reduce ambigüedad en merges, validaciones de schema y comparaciones exactas.

\section{Generación Sintética: Requisitos, no Ornamentación}

La generación sintética debe ser \emph{suficientemente realista para ejercer el código}, pero no más compleja de lo necesario. En testing, la prioridad no es sofisticación estadística, sino control de invariantes.

\subsection{Precios}

Puede emplearse una dinámica simple basada en retornos gaussianos acumulados:
\[
P_t = P_0 \exp\left(\sum_{u=1}^{t} r_u\right), \qquad r_u \sim \mathcal{N}(\mu,\sigma^2),
\]
siempre que:
\begin{itemize}
    \item la semilla esté fijada,
    \item el número de fechas sea pequeño,
    \item la salida sea exactamente reproducible.
\end{itemize}

En muchos tests, incluso es preferible una serie \emph{completamente determinista} con retornos prefijados, porque facilita asserts exactos y depuración.

\subsection{Features}

Las features sintéticas deben diseñarse para cubrir casos estructurales:
\begin{itemize}
    \item feature válida numérica,
    \item feature con \texttt{NaN},
    \item feature constante,
    \item feature desalineada,
    \item feature duplicada o con nombre prohibido.
\end{itemize}

\subsection{Labels}

Las labels deben poder producir:
\begin{itemize}
    \item filas válidas,
    \item cola inválida por horizonte,
    \item ejemplos con huecos de calendario,
    \item universos fragmentados por símbolo.
\end{itemize}

\section{Implementación de Referencia}

El siguiente ejemplo ilustra una implementación sobria y adecuada para \texttt{conftest.py}. No intenta cubrir toda la complejidad del sistema; define una base fiable y extensible.

\begin{lstlisting}
import pytest
import numpy as np
import pandas as pd

from pathlib import Path

SEED = 42


@pytest.fixture(scope="function")
def rng():
    return np.random.default_rng(SEED)


@pytest.fixture(scope="function")
def tmp_project_dir(tmp_path):
    project_dir = tmp_path / "project"
    project_dir.mkdir(parents=True, exist_ok=True)
    return project_dir


@pytest.fixture(scope="function")
def sample_run_context():
    return {
        "run_id": "test_run_001",
        "seed": SEED,
        "asof_date": pd.Timestamp("2024-12-31"),
        "train_start": pd.Timestamp("2024-01-01"),
        "train_end": pd.Timestamp("2024-09-30"),
    }


@pytest.fixture(scope="function")
def synthetic_price_panel(rng):
    dates = pd.bdate_range("2024-01-01", periods=40)
    symbols = ["AAA", "BBB", "CCC"]

    frames = []
    for symbol in symbols:
        rets = rng.normal(loc=0.0005, scale=0.01, size=len(dates))
        close = 100.0 * np.exp(np.cumsum(rets))
        open_ = close * (1.0 + rng.normal(0.0, 0.002, len(dates)))
        high = np.maximum(open_, close) * (1.0 + 0.003)
        low = np.minimum(open_, close) * (1.0 - 0.003)
        volume = rng.integers(100_000, 500_000, size=len(dates))

        df = pd.DataFrame({
            "date": dates,
            "symbol": symbol,
            "open": open_,
            "high": high,
            "low": low,
            "close": close,
            "volume": volume,
        })
        frames.append(df)

    out = pd.concat(frames, ignore_index=True)
    out = out.sort_values(["date", "symbol"]).set_index(["date", "symbol"])

    return out


@pytest.fixture(scope="function")
def synthetic_feature_panel(synthetic_price_panel):
    df = synthetic_price_panel.reset_index().copy()

    df["ret_1d"] = df.groupby("symbol")["close"].pct_change()
    df["ma_5"] = (
        df.groupby("symbol")["close"]
        .rolling(5)
        .mean()
        .reset_index(level=0, drop=True)
    )
    df["vol_5"] = (
        df.groupby("symbol")["ret_1d"]
        .rolling(5)
        .std()
        .reset_index(level=0, drop=True)
    )

    out = df[["date", "symbol", "ret_1d", "ma_5", "vol_5"]]
    out = out.set_index(["date", "symbol"]).sort_index()
    return out


@pytest.fixture(scope="function")
def synthetic_label_panel(synthetic_price_panel):
    horizon = 5
    df = synthetic_price_panel.reset_index().copy()

    fwd_return = (
        df.groupby("symbol")["close"]
        .shift(-horizon) / df.groupby("symbol")["close"] - 1.0
    )

    df["label_fwd_5d"] = (fwd_return > 0.02).astype("float")
    df.loc[fwd_return.isna(), "label_fwd_5d"] = np.nan

    out = df[["date", "symbol", "label_fwd_5d"]]
    out = out.set_index(["date", "symbol"]).sort_index()
    return out


@pytest.fixture(scope="function")
def synthetic_splits():
    return {
        "train": ("2024-01-01", "2024-06-30"),
        "validation": ("2024-07-01", "2024-09-30"),
        "test": ("2024-10-01", "2024-12-31"),
    }
\end{lstlisting}

\section{Invariantes que \texttt{conftest.py} Debe Preservar}

El valor real del módulo no está en ``tener fixtures'', sino en preservar invariantes compartidas por toda la suite.

\subsection{Invariantes estructurales}

\begin{itemize}
    \item El índice debe ser único cuando la tabla represente una PK lógica.
    \item Las fechas deben estar ordenadas de forma estricta dentro de cada símbolo.
    \item Las columnas obligatorias deben existir con nombres exactos.
    \item Los tipos deben ser compatibles con las funciones consumidoras.
\end{itemize}

\subsection{Invariantes numéricas}

\begin{itemize}
    \item No deben aparecer \texttt{Inf} ni \texttt{-Inf}.
    \item Los volúmenes no pueden ser negativos.
    \item Las relaciones OHLC deben ser coherentes.
    \item Los asserts de coma flotante deben usar tolerancias explícitas:
    \[
    |x - y| \le \varepsilon_{\text{abs}} + \varepsilon_{\text{rel}} |y|.
    \]
\end{itemize}

\subsection{Invariantes temporales}

\begin{itemize}
    \item Features y labels deben alinearse sobre la misma clave temporal.
    \item Las labels forward deben invalidar correctamente la cola.
    \item Los splits no deben introducir look-ahead.
\end{itemize}

\section{Casos Borde que las Fixtures Deben Poder Soportar}

Un \texttt{conftest.py} maduro no solo genera el caso feliz; también debe facilitar escenarios controlados para fallo.

\begin{itemize}
    \item Símbolo con historia corta
    \item Fecha ausente en un símbolo pero no en otro
    \item Columna obligatoria faltante
    \item Dtype incorrecto
    \item Duplicados en \texttt{(date, symbol)}
    \item \texttt{NaN} en features
    \item Cola inválida de labels
    \item Universo cambiante en el tiempo
\end{itemize}

Esto no implica que todas esas variantes deban vivir en una única fixture base; a menudo es preferible construir fixtures derivadas o helper factories específicas.

\section{Factories Auxiliares frente a Fixtures Monolíticas}

Una mala práctica común es convertir \texttt{conftest.py} en un contenedor monolítico de fixtures rígidas. La alternativa correcta es combinar:

\begin{itemize}
    \item \textbf{fixtures base} pequeñas y estables,
    \item \textbf{helper factories} parametrizables,
    \item \textbf{fixtures derivadas} para escenarios concretos.
\end{itemize}

Por ejemplo, una factoría de paneles permite alterar número de símbolos, longitud temporal, presencia de huecos o intensidad de ruido sin duplicar lógica.

\begin{lstlisting}
def make_price_panel(
    rng,
    n_days=40,
    symbols=("AAA", "BBB"),
    start="2024-01-01",
    with_gaps=False
):
    dates = pd.bdate_range(start, periods=n_days)
    if with_gaps:
        dates = dates.delete([5, 12])

    frames = []
    for symbol in symbols:
        rets = rng.normal(0.0005, 0.01, len(dates))
        close = 100.0 * np.exp(np.cumsum(rets))
        df = pd.DataFrame({
            "date": dates,
            "symbol": symbol,
            "close": close,
        })
        frames.append(df)

    out = pd.concat(frames, ignore_index=True)
    return out.sort_values(["date", "symbol"]).set_index(["date", "symbol"])
\end{lstlisting}

\section{Escopos de Fixture y Riesgo de Contaminación}

La elección del \texttt{scope} es parte de la especificación.

\begin{center}
\begin{tabular}{lll}
\toprule
Fixture & Scope recomendado & Razón \\
\midrule
\texttt{tmp\_project\_dir} & \texttt{function} & aislamiento total \\
\texttt{rng} & \texttt{function} & independencia de tests \\
\texttt{sample\_run\_context} & \texttt{function} & mutabilidad baja pero segura \\
\texttt{synthetic\_price\_panel} & \texttt{function} o \texttt{module} & según coste \\
\texttt{synthetic\_splits} & \texttt{function} & claridad temporal \\
\bottomrule
\end{tabular}
\end{center}

Si una fixture mutable usa \texttt{session} sin necesidad, la suite queda expuesta a dependencia implícita entre tests. En testing cuantitativo, ese tipo de contaminación es especialmente peligroso porque genera falsos positivos intermitentes.

\section{Hooks y Marcadores de \texttt{pytest}}

Los hooks deben existir solo si mejoran la gobernanza de la suite.

\subsection{\texttt{pytest\_addoption}}

Útil para activar subconjuntos de tests costosos o integraciones opcionales.

\begin{lstlisting}
def pytest_addoption(parser):
    parser.addoption(
        "--runslow",
        action="store_true",
        default=False,
        help="Ejecuta tests marcados como lentos."
    )
\end{lstlisting}

\subsection{\texttt{pytest\_collection\_modifyitems}}

Permite excluir tests lentos por defecto y mantener rápida la ruta principal de CI.

\begin{lstlisting}
def pytest_collection_modifyitems(config, items):
    if config.getoption("--runslow"):
        return

    skip_slow = pytest.mark.skip(reason="usa --runslow para ejecutar este test")
    for item in items:
        if "slow" in item.keywords:
            item.add_marker(skip_slow)
\end{lstlisting}

\paragraph{Criterio}
Los hooks son infraestructura de ejecución. No deben mezclar lógica de negocio, métricas de estrategia ni decisiones de investigación.

\section{Qué No Debe Hacer Este Módulo}

Para que \texttt{conftest.py} mantenga nivel alto, conviene explicitar sus límites:

\begin{itemize}
    \item No debe fijar umbrales de Sharpe, win rate o profit factor.
    \item No debe prometer validez productiva del modelo.
    \item No debe incrustar lógica de entrenamiento, tuning o inferencia.
    \item No debe depender de librerías pesadas si no son necesarias para la suite base.
    \item No debe simular demasiada ``realidad'' a costa de perder control y velocidad.
\end{itemize}

Cuando esos elementos aparecen en una especificación de tests base, suelen introducir ruido conceptual y dificultar el mantenimiento.

\section{Definition of Done del Módulo}

\texttt{tests/conftest.py} estará bien diseñado si cumple simultáneamente lo siguiente:

\begin{enumerate}
    \item Todas las fixtures son deterministas.
    \item Ninguna fixture depende de I/O externo.
    \item Los contratos de índices, columnas y tipos están explicitados.
    \item Los escenarios mínimos permiten construir tests unitarios e integrados.
    \item Las fixtures pueden extenderse sin romper compatibilidad semántica.
    \item La suite falla de forma informativa ante violaciones estructurales.
    \item El tiempo de ejecución de la ruta principal de tests sigue siendo bajo.
\end{enumerate}

\section{Conclusión}

\texttt{conftest.py} no es un módulo auxiliar menor; es la capa que determina si la suite de tests será una colección frágil de ejemplos o una infraestructura rigurosa de validación. En un stack cuantitativo, su misión correcta es imponer disciplina experimental: reproducibilidad, aislamiento, control temporal, contratos explícitos y capacidad de construir casos borde de manera barata.

Si esta base está bien resuelta, los tests de accounting, point-in-time correctness, leakage, schemas y validation gates pueden escribirse con mucha mayor precisión. Si esta base está mal resuelta, incluso tests aparentemente correctos pueden volverse no deterministas, opacos o metodológicamente ambiguos.
