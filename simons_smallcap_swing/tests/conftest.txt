Especificación Cuantitativa: tests/conftest.py

0. Propósito y papel dentro de la suite

tests/conftest.py define la infraestructura común de fixtures de la suite de tests. Su función no es validar alpha, optimización de cartera ni performance financiera, sino proporcionar un entorno de prueba determinista, aislado, rápido y auditable sobre el que puedan ejecutarse tests unitarios, de integración y de regresión del pipeline cuantitativo. La base que me has pasado ya fijaba correctamente el espíritu del módulo: datos sintéticos reproducibles, contratos DataFrame explícitos, cero I/O externo y soporte para escenarios típicos de prices/features/labels/splits. :contentReference[oaicite:0]{index=0}

En un stack cuantitativo serio, conftest.py no es un detalle auxiliar. Es el laboratorio controlado sobre el que descansa la credibilidad de toda la suite. Si esta capa falla, incluso tests aparentemente correctos pueden:
- depender de estado global oculto;
- dejar de ser reproducibles;
- mezclar train y test sin querer;
- pasar localmente y fallar en CI;
- esconder bugs de schema o alignment detrás de fixtures mal diseñadas.

La regla institucional debe ser esta:

    toda fixture del core de tests
    debe ser determinista, aislada, composable
    y estructuralmente válida
    antes de que cualquier test la consuma.

1. Objetivo formal del módulo

Sea un conjunto de escenarios de prueba Θ y una familia de fixtures Φ. El módulo implementa una aplicación:

    Λ : TestScenario -> DataFrameContract

donde cada escenario parametriza:
- universo sintético;
- rango temporal;
- estructura de OHLCV;
- features derivadas;
- labels forward;
- splits temporales;
- contexto de ejecución;
- edge cases controlados.

La propiedad central del módulo es:

    misma semilla + mismos parámetros + mismo código
    => mismo dataset sintético y mismo entorno de prueba

Formalmente:

    D(θ; seed = s) = D(θ; seed = s)

y además:

    D(θ) ⊨ Schema(θ)

antes de que el test comience su lógica propia.

2. Principios no negociables

P1. Determinismo fuerte.
Toda aleatoriedad debe proceder de un generador explícito con semilla fija.

P2. Aislamiento total.
Ninguna fixture debe depender de red, reloj del sistema, base de datos, caché persistente ni artefactos de otros tests.

P3. Contrato explícito.
Las fixtures no devuelven “DataFrames cualquiera”, sino artefactos con índice, columnas y semántica controlados.

P4. Minimalidad suficiente.
Los datos deben ser lo bastante ricos para ejercer el código, pero no tan complejos que oculten bugs o ralenticen la suite.

P5. Auto-validación.
Las fixtures base deben poder validarse contra los schemas del sistema.

P6. Temporal correctness.
Las fechas, splits y labels forward deben respetar orden temporal y no inducir look-ahead accidental.

P7. Componibilidad.
Las fixtures deben combinarse limpiamente sin duplicar lógica ni generar dependencias implícitas.

3. Alcance y no-objetivos

El módulo cubre:

- fixtures de RNG determinista;
- directorios temporales aislados;
- RunContext o equivalente;
- paneles sintéticos de precios OHLCV;
- paneles sintéticos de features;
- paneles sintéticos de labels;
- splits temporales;
- factories para edge cases;
- hooks ligeros de pytest para gobernanza de la suite.

Queda fuera de alcance:

- simulación realista de microestructura;
- procesos estocásticos sofisticados tipo Heston;
- market impact;
- tuning de modelos;
- entrenamiento real;
- integración con servicios externos;
- evaluación de estrategia productiva.

La misión aquí es infraestructura de tests, no modelización financiera “bonita”.

4. Arquitectura deseable de fixtures

La estructura recomendada combina tres capas:

4.1. Fixtures base
Pequeñas, estables y reutilizables:
- rng
- tmp_project_dir
- sample_run_context

4.2. Fixtures de datos sintéticos
- synthetic_price_panel
- synthetic_feature_panel
- synthetic_label_panel
- temporal_splits

4.3. Factories o fixtures derivadas
Para edge cases:
- missing_dates_panel
- duplicate_pk_panel
- zero_volume_panel
- invalid_schema_panel
- short_history_symbol_panel

Esto es preferible a una única fixture monolítica con cientos de parámetros ocultos.

5. RNG determinista

La fixture de RNG debe fijar una semilla única y explícita.

~~~python
import pytest
import numpy as np

SEED = 42

@pytest.fixture(scope="function")
def rng() -> np.random.Generator:
    return np.random.default_rng(SEED)
~~~

Reglas:
- scope=function por defecto;
- no usar np.random global;
- no compartir estado mutable entre tests;
- no reseedear dentro de la lógica del test salvo caso muy específico y explícito.

6. Directorio temporal aislado

Los tests que escriben artefactos necesitan filesystem efímero y aislado.

~~~python
from pathlib import Path
import pytest

@pytest.fixture(scope="function")
def tmp_project_dir(tmp_path: Path) -> Path:
    project_dir = tmp_path / "project"
    project_dir.mkdir(parents=True, exist_ok=True)
    return project_dir
~~~

Contrato:
- un directorio exclusivo por test;
- sin contaminación entre ejecuciones;
- sin persistencia fuera del ciclo pytest.

7. RunContext sintético

La suite necesita un contexto canónico mínimo y estable.

~~~python
import pandas as pd
import pytest

@pytest.fixture(scope="function")
def sample_run_context():
    return {
        "run_id": "test_run_001",
        "seed": SEED,
        "asof_date": pd.Timestamp("2024-12-20"),
        "train_start": pd.Timestamp("2024-01-01"),
        "train_end": pd.Timestamp("2024-09-30"),
        "pipeline_version": "1.0.0",
        "market": "US_EQ",
    }
~~~

Requisitos:
- asof_date debe ser consistente con el panel generado;
- el contexto no debe depender del reloj actual;
- debe ser serializable o fácilmente comparable en tests.

8. Panel sintético de precios

8.1. Objetivo

synthetic_price_panel debe generar una base OHLCV pequeña pero estructuralmente válida, suficiente para:
- pct_change;
- rolling windows;
- joins por PK;
- labels forward;
- validación de schemas;
- pruebas de accounting o backtesting básico.

8.2. Contrato estructural

El panel debe satisfacer:
- MultiIndex(date, symbol) o estructura equivalente clara;
- fechas monótonas;
- símbolo explícito;
- columnas open, high, low, close, volume;
- OHLC coherente;
- volumen no negativo;
- ausencia de Inf.

Formalmente, para cada fila t:

    low_t <= min(open_t, close_t) <= max(open_t, close_t) <= high_t

8.3. Generación sobria recomendada

No hace falta realismo extremo. Una dinámica simple basada en retornos gaussianos acumulados es suficiente, siempre que sea reproducible.

~~~python
import pandas as pd
import numpy as np
import pytest

@pytest.fixture(scope="function")
def synthetic_price_panel(rng: np.random.Generator) -> pd.DataFrame:
    dates = pd.bdate_range("2024-01-01", periods=40)
    symbols = ["AAA", "BBB", "CCC"]

    frames = []
    for symbol in symbols:
        rets = rng.normal(loc=0.0005, scale=0.01, size=len(dates))
        close = 100.0 * np.exp(np.cumsum(rets))
        open_ = close * (1.0 + rng.normal(0.0, 0.002, len(dates)))
        high = np.maximum(open_, close) * (1.0 + rng.uniform(0.001, 0.005, len(dates)))
        low = np.minimum(open_, close) * (1.0 - rng.uniform(0.001, 0.005, len(dates)))

        vol_signal = np.abs(rets) * 1e6
        volume = rng.integers(100_000, 500_000, size=len(dates)) + vol_signal.astype(int)

        df = pd.DataFrame({
            "date": dates,
            "symbol": symbol,
            "open": open_,
            "high": high,
            "low": low,
            "close": close,
            "volume": volume,
        })
        frames.append(df)

    out = pd.concat(frames, ignore_index=True)
    out = out.sort_values(["date", "symbol"]).set_index(["date", "symbol"])

    assert out.index.is_monotonic_increasing
    assert not out.index.duplicated().any()
    assert (out["high"] >= out[["open", "close"]].max(axis=1)).all()
    assert (out["low"] <= out[["open", "close"]].min(axis=1)).all()
    assert (out["volume"] >= 0).all()

    return out
~~~

8.4. Nota sobre realismo

La relación volumen ~ |retorno| añade realismo útil sin convertir la fixture en una simulación pesada. Eso es correcto para testing: control > sofisticación.

9. Auto-validación contra schema

Un salto de calidad importante es que las fixtures centrales se validen contra los schemas del sistema antes de ser devueltas.

~~~python
from simons_core.schemas import validate_schema

@pytest.fixture(scope="function")
def synthetic_price_panel(rng: np.random.Generator) -> pd.DataFrame:
    ...
    panel = out.reset_index()

    result = validate_schema(panel, "prices_adjusted")
    if not result.passed:
        pytest.fail(f"Fixture failed schema validation: {result.issues}")

    return out
~~~

Regla institucional:
una fixture base inválida no debe circular por la suite y contaminar tests aguas abajo.

10. Panel sintético de features

10.1. Objetivo

Debe generar features simples y estables para:
- probar schemas;
- alignment temporal;
- rolling ops;
- tolerancia a NaN iniciales;
- pipelines de modelado.

10.2. Features mínimas razonables

- ret_1d
- ma_5
- vol_5

10.3. Implementación recomendada

~~~python
@pytest.fixture(scope="function")
def synthetic_feature_panel(synthetic_price_panel: pd.DataFrame) -> pd.DataFrame:
    df = synthetic_price_panel.reset_index().copy()

    df["ret_1d"] = df.groupby("symbol")["close"].pct_change()
    df["ma_5"] = (
        df.groupby("symbol")["close"]
        .rolling(5)
        .mean()
        .reset_index(level=0, drop=True)
    )
    df["vol_5"] = (
        df.groupby("symbol")["ret_1d"]
        .rolling(5)
        .std()
        .reset_index(level=0, drop=True)
    )

    out = df[["date", "symbol", "ret_1d", "ma_5", "vol_5"]]
    out = out.sort_values(["date", "symbol"]).set_index(["date", "symbol"])

    assert out.index.is_monotonic_increasing
    assert not out.index.duplicated().any()

    return out
~~~

10.4. NaNs controlados

Es correcto que existan NaNs estructurales en los primeros puntos de rolling windows. Esos NaNs no son bugs; son parte del contrato esperado del dataset.

11. Panel sintético de labels

11.1. Objetivo

Debe proporcionar labels forward controladas, alineadas temporalmente y con cola inválida explícita.

11.2. Definición base

Para horizonte h:

    y_t = P_{t+h} / P_t - 1

o una binarización derivada.

11.3. Requisito crítico

Las últimas h observaciones por símbolo deben invalidarse explícitamente, típicamente con NaN.

11.4. Fixture parametrizada recomendada

~~~python
@pytest.fixture(params=[1, 5, 10], scope="function")
def synthetic_label_panel(request, synthetic_price_panel: pd.DataFrame):
    horizon = request.param
    df = synthetic_price_panel.reset_index().copy()

    fwd_ret = (
        df.groupby("symbol")["close"].shift(-horizon)
        / df.groupby("symbol")["close"] - 1.0
    )

    binary_label = (fwd_ret > 0.02).astype(float)
    binary_label[fwd_ret.isna()] = np.nan

    out = pd.DataFrame({
        "date": df["date"],
        "symbol": df["symbol"],
        "label_fwd": fwd_ret,
        "binary_fwd": binary_label,
    }).set_index(["date", "symbol"]).sort_index()

    return out, horizon
~~~

11.5. Beneficio

Parametrizar horizontes multiplica cobertura sin duplicar código.

12. Splits temporales

La suite debe disponer de splits explícitos, ordenados y no solapados.

~~~python
@pytest.fixture(scope="function")
def temporal_splits():
    return {
        "train": ("2024-01-01", "2024-06-30"),
        "validation": ("2024-07-01", "2024-09-30"),
        "test": ("2024-10-01", "2024-12-31"),
    }
~~~

Invariantes:
- train < validation < test
- sin overlap
- semántica temporal clara
- fácilmente consumibles por tests de walk-forward, purged splits o validación de leakage

13. Edge case factories

Una suite madura necesita generar fallos controlados, no solo el caso feliz.

13.1. Ejemplos deseables

- duplicate_pk_panel
- missing_column_panel
- negative_volume_panel
- zero_volume_symbol_panel
- missing_dates_panel
- short_history_symbol_panel
- feature_panel_with_nan_spike
- invalid_label_tail_panel

13.2. Ejemplo

~~~python
@pytest.fixture(scope="function")
def edge_case_price_panel(synthetic_price_panel: pd.DataFrame) -> pd.DataFrame:
    bad = synthetic_price_panel.reset_index().copy()
    bad.loc[0, "open"] = np.nan
    bad.loc[1, "low"] = -1.0
    bad = pd.concat([bad, bad.iloc[[0]]], ignore_index=True)
    return bad.set_index(["date", "symbol"]).sort_index()
~~~

14. Invariantes obligatorios del módulo

El valor real de conftest.py está en preservar invariantes compartidas por toda la suite.

14.1. Invariantes estructurales

- PK única cuando aplica;
- fechas ordenadas;
- columnas obligatorias presentes;
- dtypes compatibles;
- ausencia de Inf.

14.2. Invariantes temporales

- features y labels alineadas por (date, symbol);
- cola inválida explícita en labels forward;
- splits temporales sin solape indebido;
- no look-ahead accidental en los datos usados por tests.

14.3. Invariantes numéricas

- low <= min(open, close)
- high >= max(open, close)
- close > 0
- volume >= 0
- comparaciones de floats con tolerancias explícitas cuando proceda

15. Scopes correctos

La elección de scope es parte del contrato de testing.

Recomendación base:

- rng -> function
- tmp_project_dir -> function
- sample_run_context -> function
- synthetic_price_panel -> function
- synthetic_feature_panel -> function
- synthetic_label_panel -> function
- temporal_splits -> function

Puede relajarse a module para fixtures costosas e inmutables, pero la prioridad por defecto debe ser aislamiento, no micro-optimización prematura.

16. Hooks de pytest

Los hooks deben existir sólo cuando mejoran la gobernanza de la suite.

Ejemplo razonable:

~~~python
def pytest_addoption(parser):
    parser.addoption("--runslow", action="store_true", default=False, help="run slow tests")
    parser.addoption("--runschema", action="store_true", default=False, help="extra schema validation")

def pytest_collection_modifyitems(config, items):
    if not config.getoption("--runslow"):
        skip_slow = pytest.mark.skip(reason="need --runslow")
        for item in items:
            if "slow" in item.keywords:
                item.add_marker(skip_slow)
~~~

Eso sí pertenece al módulo. Lo que no pertenece es incrustar lógica de estrategia o thresholds de negocio en conftest.py.

17. Failure modes críticos que el módulo debe evitar

F1. Fixture no determinista.
Dos runs dan datos distintos sin cambio de semilla.

F2. Contaminación entre tests.
Uso incorrecto de scope o estado global mutable.

F3. Price panel estructuralmente inválido.
OHLC inconsistentes, PK duplicada, tipos erróneos.

F4. Labels forward mal alineadas.
La cola inválida no se marca, o se genera leakage.

F5. Features opacas.
No queda claro qué columnas son identificadores y cuáles señales.

F6. Dependencia de I/O externo.
Cualquier fixture que dependa de red, disco persistente o APIs rompe la disciplina del módulo.

18. Pruebas mínimas del propio conftest

Aunque conftest.py sea infraestructura, sus contratos deben verificarse al menos indirectamente. Ejemplos:

~~~python
def test_price_panel_is_deterministic(rng):
    pass  # compare two constructions under same seed if exposed via factory

def test_price_panel_has_valid_ohlc(synthetic_price_panel):
    df = synthetic_price_panel
    assert (df["high"] >= df[["open", "close"]].max(axis=1)).all()
    assert (df["low"] <= df[["open", "close"]].min(axis=1)).all()

def test_feature_panel_has_expected_columns(synthetic_feature_panel):
    assert {"ret_1d", "ma_5", "vol_5"} <= set(synthetic_feature_panel.columns)

def test_label_tail_contains_nan(synthetic_label_panel):
    labels, horizon = synthetic_label_panel
    assert labels["label_fwd"].isna().sum() >= horizon
~~~

19. MVP obligatorio

El MVP del módulo debe incluir:
- RNG determinista con SEED fijo;
- tmp_project_dir aislado;
- sample_run_context estable;
- synthetic_price_panel válido;
- synthetic_feature_panel básico;
- synthetic_label_panel con cola NaN;
- temporal_splits no solapados;
- hooks opcionales ligeros;
- auto-validación básica de fixtures centrales.

20. Segunda fase razonable

Como extensiones útiles:
- edge-case factories más ricas;
- validación automática con simons_core.schemas;
- fixtures parametrizadas por horizonte y universo;
- escenarios con missing dates o símbolos cortos;
- markers para tests lentos, schema-heavy o integration-like;
- memory profiling hooks si la suite crece mucho.

21. No-objetivos

No pertenece al alcance de este módulo:
- simulación de mercado avanzada;
- realismo microestructural completo;
- Heston/GARCH/impact models;
- tuning de modelos;
- benchmarks de estrategia;
- integración live con proveedores de datos.

La misión aquí es infraestructura de tests robusta y gobernada.

22. Criterios de aceptación

El módulo sólo debe considerarse correcto si cumple simultáneamente:

- todas las fixtures base son deterministas;
- no existe dependencia de I/O externo;
- los contratos estructurales de los paneles están explicitados;
- las fixtures pueden componer tests unitarios e integrados sin boilerplate excesivo;
- los edge cases pueden generarse de forma controlada;
- los errores de fixture fallan de forma informativa y temprana.

23. Conclusión

tests/conftest.py no es un módulo auxiliar menor; es la base experimental de toda la suite. Su valor no está en sonar sofisticado, sino en imponer disciplina de laboratorio: reproducibilidad, aislamiento, control temporal, schemas explícitos y datos sintéticos suficientemente válidos para ejercer el pipeline. En un stack cuantitativo serio, eso es exactamente lo que convierte una batería de tests en una infraestructura de validación fiable. Ese es el estándar que este módulo debe imponer.
