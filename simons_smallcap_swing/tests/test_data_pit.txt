Especificación Cuantitativa: tests/test_data_pit.py

0. Propósito y papel dentro de la suite

tests/test_data_pit.py define la barrera metodológica contra look-ahead bias y contaminación temporal en la capa de datos del sistema. Su función no es evaluar si una estrategia gana dinero ni si un modelo predice bien, sino verificar algo anterior y más básico: que toda observación usada por el pipeline era realmente observable en el instante de decisión. Si este módulo falla, cualquier métrica posterior —IC, Sharpe, turnover, drawdown o hit ratio— queda epistemológicamente contaminada aunque el resto del stack funcione “correctamente”. La base que me has pasado ya fija bien el núcleo matemático del módulo: semántica PIT mediante backward asof join, elegibilidad temporal src_ts ≤ decision_ts, universo histórico con ventanas list/delist y disponibilidad de fundamentales por acceptance_ts. :contentReference[oaicite:0]{index=0}

La regla institucional debe ser esta:

    ninguna fila usada por el sistema
    puede depender de información
    cuyo timestamp efectivo de disponibilidad
    sea posterior al instante de decisión.

1. Objetivo formal del módulo

Sea una tabla izquierda de decisiones o puntos de evaluación:

    L = {(t, s)}

donde t es el instante de decisión y s el símbolo, y sea una tabla fuente R con observaciones históricas. La operación PIT canónica debe satisfacer:

    R*(t, s) = argmax { u <= t : (u, s) in R }

siempre que exista observación elegible previa. Esto define la semántica de join point-in-time correcta: para cada par (decision_ts, symbol), el sistema solo puede consumir la observación más reciente no posterior al instante de decisión.

La suite debe verificar que el pipeline satisface simultáneamente cuatro familias de invariantes:

1. Temporal eligibility:
   src_ts <= decision_ts

2. Historical universe eligibility:
   list_date <= decision_ts < delist_date

3. Filing/fundamental availability:
   acceptance_ts <= decision_ts
   o, si existe una noción más fuerte,
   available_ts <= decision_ts

4. Determinism of PIT selection:
   |R*(t,s)| <= 1 bajo política de desempate explícita

2. Principios no negociables

P1. PIT explícito.
Toda elegibilidad temporal debe expresarse con timestamps, no con intuiciones del tipo “ese dato era más o menos conocido”.

P2. Backward-only semantics.
La semántica PIT por defecto debe ser backward merge, nunca forward fill desde el futuro.

P3. Economic date ≠ availability date.
No debe confundirse fecha económica del dato con fecha efectiva de observación.

P4. Historical universe real.
El universo en t no puede reconstruirse con el universo actual salvo que se quiera conscientemente introducir survivorship bias.

P5. Fail-fast.
Una sola violación src_ts > decision_ts debe hacer fallar el test.

P6. Determinismo.
Si varias filas fuente son candidatas, debe existir política explícita de resolución.

P7. Granularidad temporal honesta.
Si el sistema decide intradía, los tests PIT deben operar intradía; truncar a date no es aceptable.

3. Ámbito y no-objetivos

El módulo cubre:

- PIT correctness en joins de precios;
- PIT correctness en fundamentales;
- elegibilidad histórica del universo;
- no reutilización del universo actual en historia;
- helpers de aserción PIT;
- casos con datos ausentes o no elegibles;
- semántica backward merge o equivalente.

Queda fuera de alcance:

- evaluación de alpha;
- calidad predictiva del modelo;
- performance del join;
- drift estadístico de features;
- portfolio construction;
- métricas de riesgo;
- lógica de ejecución.

La misión aquí es validez temporal del dataset.

4. Modelo temporal canónico

Para que la suite sea rigurosa, conviene fijar un modelo de timestamps estable. Una tabla o fila de datos puede tener varias fechas distintas:

- event_ts:
  momento económico al que pertenece el dato.

- period_end:
  cierre del periodo económico, típico en fundamentales.

- publish_ts:
  momento en que la fuente publicó el dato.

- acceptance_ts:
  momento en que el filing o dato fue formalmente aceptado.

- available_ts:
  momento a partir del cual el sistema podía usar el dato.

- decision_ts:
  momento en que la estrategia decide.

- asof_ts:
  corte temporal usado por una consulta PIT.

La restricción relevante para PIT no es:

    period_end <= decision_ts

sino:

    available_ts <= decision_ts

y, en esquemas simplificados:

    src_ts <= decision_ts

donde src_ts es el timestamp que gobierna efectivamente la elegibilidad temporal del registro fuente.

5. Contrato PIT general

La suite debe fijar el siguiente contrato general:

    Dataset_PIT_Correct
    <=>
    forall rows:
        src_ts <= decision_ts
        and universe_eligible(symbol, decision_ts)
        and deterministic_join(symbol, decision_ts)

Esto implica prohibir expresamente:

- forward fills desde observaciones futuras;
- uso de close del mismo día para decisiones anteriores al close;
- uso de fundamentales por period_end en lugar de acceptance_ts;
- reconstrucción histórica con universo actual;
- joins ambiguos sobre timestamps duplicados sin política explícita.

6. Invariantes obligatorios

6.1. No future data

Para toda fila del output PIT:

    row.src_ts <= row.decision_ts

Ésta es la aserción cardinal del módulo.

6.2. Universe eligibility

Si un símbolo s tiene:
- list_date(s)
- delist_date(s)

entonces su pertenencia histórica debe obedecer:

    s in U_t  <=>  list_date(s) <= t < delist_date(s)

o la variante exacta que el sistema haya definido. Lo importante es que esté fijada y testeada.

6.3. No current universe leakage

En general, para t < T:

    U_t != U_T

La historia no debe reconstruirse usando el universo actual.

6.4. Determinism of PIT join

Para cada par (t,s), el join PIT debe devolver a lo sumo una observación lógica:

    |R*(t,s)| <= 1

Si hay múltiples candidatas con el mismo src_ts, el sistema debe declarar una política:
- first
- last
- max priority
- aggregate
- reject

Lo que no es aceptable es dependencia implícita del orden físico de la tabla.

6.5. Intraday consistency

Si la estrategia decide a las 09:30 y el precio corresponde al close de las 16:00 del mismo día, entonces:

    16:00 > 09:30

y esa observación no es elegible.

7. Helpers de aserción recomendados

La suite debe factorizar helpers para evitar lógica repetida y hacer el fallo más informativo.

~~~python
import pandas as pd

def assert_pit_correct(df: pd.DataFrame, decision_col: str, src_col: str) -> None:
    mask = df[src_col] > df[decision_col]
    if mask.any():
        violators = df.loc[mask, [decision_col, src_col]].head()
        raise AssertionError(
            f"PIT violation: {mask.sum()} rows with {src_col} > {decision_col}\n"
            f"First violators:\n{violators}"
        )

def assert_universe_eligibility(
    df: pd.DataFrame,
    universe: pd.DataFrame,
    symbol_col: str = "symbol",
    ts_col: str = "decision_ts",
) -> None:
    for _, row in df.iterrows():
        sym = row[symbol_col]
        t = row[ts_col]
        window = universe.loc[sym]
        if not (window["list_date"] <= t):
            raise AssertionError(f"{sym} pre-listing at {t}")
        if pd.notna(window["delist_date"]) and not (t < window["delist_date"]):
            raise AssertionError(f"{sym} post-delisting at {t}")
~~~

Estos helpers deben vivir o bien en el propio módulo de tests o en utilidades de testing, pero nunca incrustados de forma inconsistente test a test.

La fixture de precios debe contener timestamps explícitos de disponibilidad, no solo fechas económicas.

import pandas as pd

def price_panel_pit() -> pd.DataFrame:

9. Join PIT canónico para precios

La implementación de referencia debe usar backward merge y tablas ordenadas.

~~~python
def pit_join_prices(decision_table: pd.DataFrame, price_panel: pd.DataFrame) -> pd.DataFrame:
    left = decision_table.sort_values(["symbol", "decision_ts"])
    right = price_panel.sort_values(["symbol", "price_ts"])

    return pd.merge_asof(
        left,
        right,
        left_on="decision_ts",
        right_on="price_ts",
        by="symbol",
        direction="backward",
        allow_exact_matches=True,
    )
~~~

Contrato:
- right debe estar ordenada;
- direction debe ser backward;
- price_ts representa disponibilidad real;
- el resultado debe pasar assert_pit_correct.

10. Casos mínimos del módulo

10.1. Precios: no future dates

~~~python
def test_no_future_dates_in_prices(decision_table, price_panel_pit):
    joined = pit_join_prices(decision_table, price_panel_pit)
    assert_pit_correct(joined, "decision_ts", "price_ts")
~~~

Qué protege:
- look-ahead directo en precios
- close no disponible aún
- fill ilegal de huecos desde el futuro

10.2. Intradía: 09:30 no puede ver 16:00

~~~python
def test_intraday_close_unavailable_before_close(decision_table, price_panel_pit):
    joined = pit_join_prices(decision_table, price_panel_pit)

    row = joined[
        (joined["symbol"] == "AAA")
        & (joined["decision_ts"] == pd.Timestamp("2024-01-02 09:30"))
    ].iloc[0]

    assert row["price_ts"] == pd.Timestamp("2024-01-01 16:00")
~~~

Esto obliga a distinguir fecha de disponibilidad real, no solo day bucket.

10.3. Fundamentales: acceptance before decision

~~~python
def test_fundamentals_acceptance_before_decision(decision_table, fundamentals_pit):
    left = decision_table.sort_values(["symbol", "decision_ts"])
    right = fundamentals_pit.sort_values(["symbol", "acceptance_ts"])

    joined = pd.merge_asof(
        left,
        right,
        left_on="decision_ts",
        right_on="acceptance_ts",
        by="symbol",
        direction="backward",
        allow_exact_matches=True,
    )

    valid = joined["acceptance_ts"].isna() | (joined["acceptance_ts"] <= joined["decision_ts"])
    assert valid.all()
~~~

Qué protege:
- usar filing por period_end
- confundir dato económico con dato observable
- consumir revisiones futuras en historia

10.4. Universe membership respects list/delist

~~~python
def test_universe_membership_respects_list_delist_window(decision_table, universe_history):
    eligible = decision_table.copy()
    assert_universe_eligibility(eligible, universe_history)
~~~

Qué protege:
- pre-listing leakage
- post-delisting survivorship artificial

10.5. No current universe leakage

~~~python
def test_no_current_universe_leakage_into_history(decision_table, universe_history):
    jan2 = decision_table[decision_table["decision_ts"] < pd.Timestamp("2024-01-03")]
    assert "BBB" not in jan2["symbol"].values
~~~

Esto debe adaptarse a cómo se construye el universo histórico real, pero la idea es obligar a respetar la dinámica de altas y bajas.

11. Casos borde imprescindibles

Para acercarse al techo realista 96-97, la suite debe cubrir además:

11.1. Huecos de calendario por símbolo

Si BBB no tiene precio el 2024-01-02, el sistema puede:
- usar el último precio anterior si existe;
- devolver NaN si no existe;
- marcar missing_pit

pero nunca usar el precio del 2024-01-03 para decidir en 2024-01-02.

11.2. Duplicados en src_ts

Si existen dos filas con mismo (symbol, src_ts), el sistema debe:
- rechazar;
- o aplicar política determinista explícita.

La suite debe testear esa política.

11.3. Revisions / vintages

Un filing revisado en febrero no puede contaminar decisiones de enero. Si el sistema modela vintages, el test debe verificar que:

    revision.acceptance_ts > decision_ts

implica no elegibilidad en ese decision_ts.

11.4. No eligible source row

Si no hay observación previa elegible, el sistema no debe forward-fill desde el futuro. Debe:
- devolver NaN,
- excluir la fila,
- o marcarla explícitamente.

La suite debe fijar y testear esta política.

12. Determinismo del join

La suite no debe limitarse a src_ts <= decision_ts. También debe exigir que el PIT join sea determinista.

Ejemplo razonable:

~~~python
def test_duplicate_src_ts_requires_explicit_policy():
    dup = pd.DataFrame([
        {"symbol": "AAA", "price_ts": pd.Timestamp("2024-01-02 16:00"), "price": 101.0},
        {"symbol": "AAA", "price_ts": pd.Timestamp("2024-01-02 16:00"), "price": 101.5},
    ])

    # El sistema debe decidir si:
    # - rechaza el dataset,
    # - usa first,
    # - usa last,
    # - agrega.
    # Lo importante es que exista política fija.
    assert len(dup) == 2
~~~

Idealmente esto se conecta con un helper de validación que rechace ambigüedad si la política es strict.

13. Métrica diagnóstica opcional

Como ayuda de depuración, puede ser útil calcular:

    LeakRate = (1/N) * Σ 1[src_ts > decision_ts]

pero esto es solo diagnóstico. En tests PIT correctos, la condición exigida es:

    LeakRate = 0

La suite no debe aceptar “leakage pequeño”.

14. Failure modes críticos que debe detectar

F1. Uso de cierre del mismo día antes del cierre.
F2. Forward fill desde observación futura.
F3. Uso de fundamentals por period_end en lugar de acceptance_ts.
F4. Revisión futura usada retroactivamente.
F5. Universo actual reutilizado en historia.
F6. Símbolos elegibles antes de list_date.
F7. Símbolos mantenidos tras delist_date.
F8. Ambigüedad en selección cuando hay múltiples filas elegibles.

15. MVP obligatorio

El MVP del módulo debe incluir:
- assert_pit_correct helper
- test_no_future_dates_in_prices
- test_intraday_close_unavailable_before_close
- test_fundamentals_acceptance_before_decision
- test_universe_membership_respects_list_delist_window
- test_no_current_universe_leakage_into_history
- al menos un caso con fila sin observación previa elegible

16. Segunda fase razonable

Como extensiones útiles:
- data vintages con revisiones
- duplicate src_ts policy tests
- coverage matrix por fecha y símbolo
- distribución de lags src_ts -> decision_ts
- granularidad intradía más rica
- PIT protocol versioning si el stack crece

17. No-objetivos

No pertenece al alcance de este módulo:
- performance del join
- métricas predictivas
- backtest accounting
- optimization
- factor evaluation
- drift estadístico
- strategy scoring

La misión aquí es exclusivamente validez temporal.

18. Criterios de aceptación

El módulo solo debe considerarse correcto si cumple simultáneamente:

- toda observación fuente usada por el join es elegible temporalmente;
- el universo histórico respeta ventanas reales de vida del símbolo;
- los fundamentales respetan su timestamp de disponibilidad;
- la suite cubre al menos un caso intradía y uno de universo dinámico;
- el join PIT es backward y determinista bajo política explícita;
- una sola fila futura hace fallar el test.

19. Conclusión

tests/test_data_pit.py es una de las suites metodológicamente más importantes del sistema porque protege la validez experimental de todo lo demás. Su valor no está en parecer sofisticada, sino en imponer una disciplina exacta sobre qué se sabía, cuándo se sabía y cuándo podía usarse. En un stack cuantitativo serio, esa barrera PIT no es opcional: es la condición necesaria para que cualquier resultado posterior tenga derecho a ser interpretado como evidencia y no como contaminación del futuro. Ese es exactamente el estándar que este módulo debe imponer.
