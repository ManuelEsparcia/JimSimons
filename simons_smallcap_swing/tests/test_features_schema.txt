Especificación Cuantitativa: tests/test_features_schema.py

0. Propósito y papel dentro de la suite

tests/test_features_schema.py define la barrera estructural del feature store dentro de la suite de tests. Su función no es evaluar si las features tienen poder predictivo, mejoran un modelo o elevan el Sharpe de una estrategia, sino verificar algo anterior y más básico: que la salida de la pipeline de features respeta un contrato de datos estable, consumible y libre de roturas silenciosas. La base que has pasado ya fija bien el núcleo correcto del módulo: columnas requeridas, unicidad de PK=(date,symbol), dtypes exactos, ausencia de Inf y umbrales de cobertura por feature. :contentReference[oaicite:0]{index=0}

En un stack cuantitativo serio, este módulo no es “cosmético”. Es el cortafuegos entre generación de features y todo lo demás:
- training,
- inference,
- validation,
- attribution,
- backtest.

Si falla esta capa, el resto del sistema puede seguir corriendo y aun así producir resultados sin significado operativo.

La regla institucional debe ser esta:

    ninguna tabla de features
    debe entrar en entrenamiento o inferencia
    si no satisface primero
    un contrato estructural explícito, versionado y auditable.

1. Objetivo formal del módulo

Sea un DataFrame de features D y un contrato estructural S. El módulo debe verificar si:

    D ⊨ S

donde el schema S incluye, como mínimo:

- columnas obligatorias;
- tipos esperados;
- política de nullabilidad;
- clave primaria lógica;
- restricciones de validez numérica;
- umbrales mínimos de cobertura;
- opcionalmente, dominios plausibles por familia de feature;
- versión contractual.

Una formalización útil es:

    S = (C_req, T, N, K, R, V, Θ)

donde:
- C_req es el conjunto de columnas requeridas;
- T asigna tipos esperados a columnas;
- N define nullabilidad;
- K define la PK lógica;
- R define dominios válidos;
- V define la versión del contrato;
- Θ define umbrales de cobertura.

Entonces:

    D ⊨ S

si y solo si se cumplen simultáneamente:

1. C_req ⊆ C_obs
2. PK_unique(D, K)
3. dtype_D(c) = T(c) o dtype compatible declarado
4. no Inf en columnas numéricas
5. domain(c) respetado para columnas con restricciones
6. Coverage(c) ≥ Θ(c) cuando el contrato lo exige
7. la versión observada coincide con la versión aceptada o su política de compatibilidad

2. Principios no negociables

P1. Schema-first.
El feature store se valida como contrato de datos, no como artefacto “informal” de pandas.

P2. Fail-fast.
Una violación estructural debe detener el flujo antes de training o inference.

P3. Precisión diagnóstica.
El test debe decir exactamente qué falló: columna, tipo, cobertura, PK o dominio.

P4. Determinismo.
El mismo input y la misma configuración deben producir el mismo schema efectivo.

P5. No ambigüedad.
No se permiten coerciones silenciosas, columnas implícitas ni tipos laxos no documentados.

P6. Evolución disciplinada.
Los cambios de schema deben ser deliberados y versionados.

P7. Separación de capas.
Este módulo valida consumibilidad estructural, no utilidad predictiva.

3. Ámbito y no-objetivos

El módulo cubre:

- presencia de columnas obligatorias;
- unicidad de PK=(date,symbol) o la PK definida;
- dtypes exactos o compatibles explícitos;
- ausencia de ±Inf;
- cobertura mínima por feature;
- dominios plausibles si el contrato lo fija;
- compatibilidad con el schema central del sistema.

Queda fuera de alcance:

- feature importance;
- stationarity completa;
- selección de variables;
- calidad predictiva;
- causalidad;
- SHAP;
- model readiness profunda;
- multicolinealidad compleja como criterio fuerte del schema base.

Puede haber extensiones razonables, pero no deben desplazar el foco principal del módulo.

4. Qué debe proteger este módulo

4.1. Rotura de columnas requeridas

Si una columna obligatoria desaparece, se renombra o no se produce, el módulo debe fallar inmediatamente. Es uno de los errores más caros aguas abajo porque:
- rompe joins,
- rompe consumers de inferencia,
- invalida training sets,
- y a menudo produce fallos tardíos difíciles de diagnosticar.

La condición mínima es:

    C_req ⊆ C_obs

4.2. Duplicados de clave lógica

Si la PK lógica es:

    K = (date, symbol)

entonces debe cumplirse:

    | { rows in D : (date, symbol) = k } | <= 1   para todo k

Un duplicado en PK convierte el dataset en ambiguo:
- una señal deja de representar una observación única;
- los joins pueden multiplicarse;
- labels y features pueden desalinearse;
- la inferencia puede depender del orden físico del DataFrame.

4.3. Deriva de tipos

Una feature numérica convertida a object, una fecha como string o una columna categórica casteada erróneamente pueden no romper la ejecución de inmediato, pero sí contaminar comparaciones, serialización, joins y modelos.

Este módulo debe tratar cualquier dtype drift no declarado como una violación contractual real.

4.4. Valores infinitos

Los infinitos suelen emerger de:
- divisiones por cero,
- logs inválidos,
- escalados degenerados,
- normalizaciones defectuosas.

No deben tratarse como “detalle menor”. Una feature con Inf no pertenece a un dataset consumible.

4.5. Cobertura insuficiente

Una feature puede existir nominalmente y aun así ser inútil o peligrosa si la mayor parte de sus valores son NaN. Por eso la cobertura debe validarse con thresholds explícitos, distintos por familia de features.

5. Contrato estructural recomendado

Una representación sobria y operativa del schema de features puede ser:

~~~python
schema_v1 = {
    "version": "1.0.0",
    "required_columns": {
        "date",
        "symbol",
        "run_id",
        "config_hash",
        "ret_1d",
        "vol_20d",
    },
    "expected_dtypes": {
        "date": "datetime64[ns]",
        "symbol": "object",
        "run_id": "object",
        "config_hash": "object",
        "ret_1d": "float64",
        "vol_20d": "float64",
    },
    "primary_key": ["date", "symbol"],
    "coverage_thresholds": {
        "ret_1d": 0.95,
        "vol_20d": 0.80,
    },
}
~~~

Esto ya fija:
- identidad;
- estructura mínima;
- tipos;
- cobertura;
- versión.

Si además se añaden dominios plausibles, el contrato se vuelve mucho más útil sin sobrediseñarse.

6. Contrato de columnas obligatorias

El test base debe verificar:

    C_req ⊆ C_obs

Implementación recomendada:

~~~python
def test_features_required_columns_present(feature_df, feature_schema_ref):
    required = feature_schema_ref["required_columns"]
    missing = required.difference(feature_df.columns)
    assert not missing, f"Faltan columnas requeridas: {sorted(missing)}"
~~~

Qué protege:
- renombrados accidentales;
- salidas parciales de pipeline;
- columnas eliminadas en refactors;
- drift entre módulos productores y consumidores.

7. Contrato de clave primaria

La clave lógica mínima recomendada es:

    PK = (date, symbol)

La condición formal es:

    PK_unique(D)

Implementación recomendada:

~~~python
def test_features_primary_key_unique_date_symbol(feature_df, feature_schema_ref):
    pk = feature_schema_ref["primary_key"]
    dup_mask = feature_df.duplicated(subset=pk, keep=False)
    assert not dup_mask.any(), f"Duplicados detectados en PK {pk}"
~~~

Esto debe fallar con mensajes precisos, idealmente mostrando ejemplos de duplicados.

8. Contrato de tipos

Los dtypes deben validarse de forma estricta o mediante una clase declarada de compatibilidad. Si el sistema exige exactitud total, la comparación puede ser directa:

~~~python
def test_features_dtypes_match_schema(feature_df, feature_schema_ref):
    expected = feature_schema_ref["expected_dtypes"]

    for col, exp_dtype in expected.items():
        assert col in feature_df.columns, f"Columna ausente: {col}"
        obs_dtype = str(feature_df[col].dtype)
        assert obs_dtype == exp_dtype, (
            f"{col}: dtype observado={obs_dtype}, esperado={exp_dtype}"
        )
~~~

Esto protege contra:
- coerciones silenciosas;
- cambios en parsers;
- serializaciones que transforman numéricos en texto;
- cambios de backend de lectura/escritura.

Si el sistema decide permitir “familias compatibles” de tipos, esa compatibilidad debe estar centralizada y no repartida test a test.

9. Contrato de validez numérica

Para toda feature numérica f_j:

    f_j ∈ ℝ ∪ {NaN}

pero:

    f_j ∉ {+∞, -∞}

Implementación recomendada:

~~~python
import numpy as np

def test_no_inf_values_in_numeric_features(feature_df):
    num_cols = feature_df.select_dtypes(include=["number"]).columns
    has_inf = np.isinf(feature_df[num_cols].to_numpy()).any()
    assert not has_inf, "Se detectaron valores +/-Inf en features numéricas"
~~~

En un módulo más completo, conviene además informar qué columnas contienen Inf.

10. Cobertura por feature

Definimos cobertura de una feature f_j como:

    Coverage(f_j) = (1/N) * Σ 1{f_ij no es NaN}

El contrato exige:

    Coverage(f_j) >= θ_j

para cada feature cuyo threshold se haya fijado.

Implementación recomendada:

~~~python
def test_feature_coverage_above_threshold(feature_df, feature_schema_ref):
    thresholds = feature_schema_ref["coverage_thresholds"]

    for col, thr in thresholds.items():
        coverage = feature_df[col].notna().mean()
        assert coverage >= thr, (
            f"{col}: coverage={coverage:.3f} < {thr:.3f}"
        )
~~~

Nota importante:
los thresholds no deben ser universales ni arbitrarios. Deben reflejar la naturaleza de cada feature:
- una contemporánea tipo ret_1d puede exigir cobertura casi total;
- una rolling de 20 días necesita tolerar warm-up NaNs;
- una feature fundamental puede tener cobertura estructuralmente menor si el contrato así lo admite.

11. Domain constraints recomendadas

Para subir este módulo a un nivel realmente fuerte, conviene introducir restricciones de dominio plausibles por feature. Ejemplos razonables:

- ret_1d ∈ [-0.20, 0.20]
- ret_5d ∈ [-0.50, 0.50]
- vol_20d ∈ [0.0, 1.5]
- ma_ratio ∈ [0.5, 2.0]
- rsi_14 ∈ [0, 100]

Esto no pretende modelar toda la estadística del mercado; pretende detectar valores físicamente absurdos o errores de transformación.

Implementación recomendada:

~~~python
DOMAIN_BOUNDS = {
    "ret_1d": (-0.20, 0.20),
    "ret_5d": (-0.50, 0.50),
    "vol_20d": (0.00, 1.50),
    "ma_ratio": (0.5, 2.0),
    "rsi_14": (0.0, 100.0),
}

def test_feature_domain_constraints(feature_df):
    for col, (lo, hi) in DOMAIN_BOUNDS.items():
        if col not in feature_df.columns:
            continue
        valid = feature_df[col].between(lo, hi) | feature_df[col].isna()
        assert valid.all(), f"{col}: valores fuera de [{lo}, {hi}]"
~~~

Esto protege especialmente contra:
- errores de normalización;
- features mal escaladas;
- retornos imposibles;
- ratios degenerados.

12. Nullability policy

No todas las columnas deben tratarse igual:

- date, symbol, run_id, config_hash:
  no deben ser nulas.

- rolling features:
  pueden admitir nulos en warm-up.

- features fundamentales:
  la nullabilidad depende del contrato explícito.

Por tanto, el test no debe preguntar solo “hay nulos o no”, sino “hay nulos donde están permitidos y en la magnitud permitida”.

13. Fixture base recomendada

La fixture debe ser pequeña, determinista y fácil de manipular para inducir fallos controlados.

~~~python
import numpy as np
import pandas as pd
import pytest

@pytest.fixture
def synthetic_feature_panel():
    dates = pd.bdate_range("2024-01-01", periods=6)
    symbols = ["AAA", "BBB"]

    idx = pd.MultiIndex.from_product(
        [dates, symbols],
        names=["date", "symbol"]
    )

    df = pd.DataFrame(index=idx).reset_index()
    df["run_id"] = "run_001"
    df["config_hash"] = "cfg_abc123"
    df["ret_1d"] = np.array([0.01, 0.02, -0.01, 0.00, 0.03, 0.01, -0.02, 0.01, 0.00, 0.02, 0.01, -0.01])
    df["vol_20d"] = np.array([np.nan, np.nan, 0.12, 0.10, 0.11, 0.13, np.nan, np.nan, 0.09, 0.08, 0.10, 0.11])

    return df
~~~

Esta fixture es suficientemente rica para:
- probar coverage;
- probar PK;
- probar dtypes;
- inducir fallos por transformación mínima;
- seguir siendo auditable a mano.

14. Fixtures derivadas para fallo controlado

Una buena suite debe disponer de variaciones mínimas sobre la fixture válida:

14.1. Missing column

~~~python
@pytest.fixture
def feature_panel_missing_run_id(synthetic_feature_panel):
    return synthetic_feature_panel.drop(columns=["run_id"])
~~~

14.2. PK duplicate

~~~python
@pytest.fixture
def feature_panel_with_duplicate_pk(synthetic_feature_panel):
    dup = synthetic_feature_panel.iloc[[0]].copy()
    return pd.concat([synthetic_feature_panel, dup], ignore_index=True)
~~~

14.3. Dtype drift

~~~python
@pytest.fixture
def feature_panel_dtype_drift(synthetic_feature_panel):
    df = synthetic_feature_panel.copy()
    df["ret_1d"] = df["ret_1d"].astype("object")
    return df
~~~

14.4. Inf value

~~~python
@pytest.fixture
def feature_panel_with_inf(synthetic_feature_panel):
    df = synthetic_feature_panel.copy()
    df.loc[0, "ret_1d"] = np.inf
    return df
~~~

14.5. Low coverage

~~~python
@pytest.fixture
def feature_panel_low_coverage(synthetic_feature_panel):
    df = synthetic_feature_panel.copy()
    df["ret_1d"] = np.nan
    return df
~~~

La ventaja es clara: cada test falla por una razón única y comprensible.

15. Integración con la capa formal de schemas

Si el repositorio dispone de una capa central como simons_core.schemas, este módulo no debería reinventar contratos incompatibles. Lo correcto es actuar como validación operacional de esa capa central.

Una interfaz razonable sería:

~~~python
from simons_core.schemas import validate_schema

def test_feature_schema_matches_central_contract(feature_df):
    result = validate_schema(feature_df, "features")
    assert result.passed, result.issues
~~~

Así se evita:
- lógica duplicada;
- drift entre schema formal y test local;
- criterios distintos según el fichero.

16. Versionado del schema

Este módulo debe tratar el schema como artefacto versionado. Una estrategia mínima razonable:

~~~python
schema_v1 = {..., "version": "1.0.0"}
schema_v2 = {..., "version": "2.0.0"}
~~~

Y tests del tipo:

- v1 sigue siendo aceptado por consumidores v1;
- v2 introduce cambios deliberados;
- si v2 rompe compatibilidad, el test debe hacerlo explícito.

Ejemplo:

~~~python
def test_schema_version_changes_are_explicit(feature_df, schema_v1, schema_v2):
    assert schema_v1["version"] != schema_v2["version"]
~~~

La clave no es el formalismo por sí mismo, sino impedir cambios silenciosos de contrato.

17. Extensiones razonables sin sobrediseño

Para acercarse al techo realista 90-92, pueden añadirse:

17.1. Temporal alignment check
Verificar que las features están alineadas con la fecha de decisión o label_date esperada.

17.2. Column order test
Solo si el downstream consumer depende contractualmente del orden.

17.3. No duplicated feature names
Evitar alias o colisiones.

17.4. Optional feature groups
Permitir familias opcionales bien definidas sin debilitar el contrato.

17.5. Run-to-run schema stability
Confirmar que distintas ejecuciones con misma config mantienen mismo schema estructural.

18. Failure modes críticos que debe detectar

F1. Columna obligatoria ausente.
F2. Duplicados en PK=(date, symbol).
F3. Dtype drift silencioso.
F4. Valores ±Inf.
F5. Cobertura por debajo del umbral contractual.
F6. Valores físicamente absurdos fuera de dominio.
F7. Drift de schema entre runs no versionado.
F8. Incompatibilidad con el schema central del sistema.

19. MVP obligatorio

El MVP del módulo debe incluir, como mínimo:

- test_features_required_columns_present
- test_features_primary_key_unique_date_symbol
- test_features_dtypes_match_schema
- test_no_inf_values_in_numeric_features
- test_feature_coverage_above_threshold

Y, a ser posible ya en primera iteración:
- test_feature_domain_constraints
- test_feature_schema_matches_central_contract

20. Segunda fase razonable

Como extensiones útiles:
- schema versioning tests;
- temporal alignment tests;
- no duplicated feature names;
- nullability policy by feature family;
- domain constraints más ricas;
- run-to-run schema stability;
- checks ligeros de correlación extrema solo como warning contractual si el sistema lo necesita.

21. No-objetivos

No pertenece al alcance de este módulo:
- demostrar predictividad;
- medir IC o Sharpe;
- probar feature importance;
- hacer selección de variables;
- validar stationarity completa;
- discutir arquitectura de modelos.

La misión aquí es estructural y contractual.

22. Criterios de aceptación

El módulo solo debe considerarse correcto si cumple simultáneamente:

- detecta ausencia de columnas obligatorias;
- detecta duplicados de clave lógica;
- detecta dtypes incorrectos;
- detecta ±Inf en columnas numéricas;
- detecta cobertura insuficiente según thresholds explícitos;
- falla con mensajes claros y accionables;
- permanece alineado con la capa central de schema del sistema.

23. Conclusión

tests/test_features_schema.py debe entenderse como la muralla estructural del feature store. Su valor no está en sonar sofisticado, sino en imponer una disciplina mínima sin la cual el resto del stack deja de ser fiable: columnas explícitas, PK única, tipos correctos, valores finitos, cobertura suficiente y evolución versionada. En un framework cuantitativo serio, eso no es burocracia; es la condición necesaria para que training, inference y backtesting operen sobre artefactos realmente consumibles. Ese es exactamente el estándar que este módulo debe imponer.
