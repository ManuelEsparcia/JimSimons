Especificación Cuantitativa: models/regimes/regime_features.py

1. Propósito del módulo

El objetivo de models/regimes/regime_features.py es construir una representación causal, robusta, parsimoniosa y operativamente estable del estado observable del mercado para alimentar modelos de inferencia de regímenes y mecanismos downstream de adaptación.

El módulo no está diseñado para generar alpha directa ni para acumular indicadores técnicos ornamentales. Su misión es producir un vector temporal de estado:

    z_t ∈ R^p

que resuma dimensiones económicamente relevantes del entorno de mercado:
- escala de volatilidad,
- inestabilidad de volatilidad,
- dispersión cross-sectional,
- concentración de correlaciones,
- liquidez y fricción,
- breadth y participación,
- tendencia sistémica,
- y proxies de estrés.

La pregunta exacta que debe responder el módulo es:

“Dado el conjunto de datos disponible hasta t, qué vector de features z_t describe de forma más útil, causal y robusta el estado del mercado para inferencia posterior de regímenes?”

2. Principios no negociables

2.1 Point-in-time estricto
Toda feature en fecha t debe depender solo de información disponible hasta t, o hasta t-1 si la convención operativa de ejecución así lo exige.

2.2 Representación de estado, no señal de alpha
Las regime features describen el entorno; no deben optimizarse para predecir retorno inmediato.

2.3 Robustez > sofisticación
Es preferible una feature simple, robusta y estable a una métrica más sofisticada pero frágil a outliers, cambios de universo o missing data.

2.4 Parsimonia con cobertura conceptual
El conjunto final debe cubrir las grandes dimensiones del entorno de mercado sin degenerar en una colección redundante de indicadores casi equivalentes.

2.5 Freeze del feature set canónico
La lista canónica de features debe ser versionada, auditada y estable en el tiempo. Las features experimentales pueden existir, pero fuera del contrato base.

2.6 Defaults internos primero
El sistema debe funcionar correctamente usando solo datos internos del propio universo y benchmark. Los proxies externos son opcionales y nunca deben ser condición de existencia del módulo.

2.7 Downstream awareness
Una feature solo es valiosa si mejora la calidad de inferencia en hmm.py o la utilidad de regime_gating.py sin introducir fragilidad estructural.

3. Qué hace y qué no hace

3.1 Qué hace
- carga inputs internos y opcionalmente proxies externos;
- define benchmark y universo efectivo por fecha;
- calcula familias canónicas de regime features;
- aplica transformaciones robustas y smoothing causal;
- calcula flags de validez y confianza;
- filtra redundancia obvia;
- congela el conjunto canónico final;
- persiste outputs, manifiestos y métricas de calidad.

3.2 Qué no hace
- no infiere estados latentes;
- no etiqueta regímenes económicos;
- no decide pesos de cartera;
- no optimiza online el subconjunto de features;
- no utiliza datos futuros para estabilizar distribuciones;
- no es una capa de representation learning profunda.

4. Rol dentro del stack

Secuencia arquitectónica correcta:

    market data
    -> regime_features.py
    -> hmm.py
    -> regime_gating.py
    -> portfolio / execution / risk modules

Interpretación correcta:
- regime_features.py construye la capa observable del estado;
- hmm.py estima estados latentes sobre esa representación;
- regime_gating.py traduce esos estados a decisiones operativas.

Una mala capa de regime_features no solo empeora la inferencia; puede inducir:
- falsos cambios de régimen,
- chattering en gating,
- sobresegmentación del HMM,
- y destrucción innecesaria de alpha vía prudencia excesiva.

7. Semántica temporal

Sea D_{≤t} toda la información disponible hasta la fecha t.

El módulo implementa una transformación causal:

    T : D_{≤t} -> z_t

Toda rolling window debe cerrarse en t.
Ninguna normalización puede usar observaciones posteriores.
Ningún proxy externo puede entrar sin respetar su lag real.
Toda imputación debe quedar explícitamente auditada.

8. Universo elegible por fecha

Sea U_t el universo elegible en fecha t.

Un símbolo i pertenece a U_t si cumple simultáneamente:
- datos de precio válidos en t;
- volumen o notional válidos;
- no suspendido, no halt, no status incompatible si así lo exige la política;
- elegibilidad de universo heredada de upstream;
- cobertura mínima suficiente para la familia de features que lo use.

Regla crítica:
las features cross-sectional solo se consideran plenamente válidas si:

    |U_t| >= n_min

y, para features de correlación rolling, además:
- el símbolo debe cumplir cobertura mínima dentro de la ventana;
- si no la cumple, queda fuera del bloque de correlación de esa fecha.

9. Benchmark y representación del mercado

9.1 Benchmark por defecto
Default recomendado:
- benchmark interno equal-weighted o robustamente equal-weighted del universo elegible.

9.2 Alternativas
- benchmark cap-weighted interno;
- benchmark externo, solo si representa razonablemente el libro real o el mercado objetivo.

9.3 Regla operativa
Si se usa benchmark externo:
- debe persistirse también benchmark interno;
- debe compararse su coherencia con la versión interna;
- y debe declararse explícitamente la dependencia de esa serie externa.

10. Taxonomía canónica de regime features

10.1 Volatilidad agregada del mercado

Si se trabaja con barras diarias, la medida canónica debe tratarse como rolling quadratic-return volatility proxy:

    vol_t^(k)
    =
    sqrt( A / k * Σ_{s=t-k+1}^t r_s^2 )

donde:
- r_s = retorno diario del benchmark;
- A = factor de anualización, típicamente 252.

Features core:
- vol_10d
- vol_20d
- vol_63d

10.2 Volatility of volatility

    vov_t^(k,m)
    =
    sqrt(
      1/m * Σ_{s=t-m+1}^t
      ( vol_s^(k) - mean_t^(m)(vol^(k)) )^2
    )

Feature core:
- vov_20d_on_vol_10d
o
- vov_20d_on_vol_20d

10.3 Tendencia sistémica

    trend_ret_t^(k)
    =
    1/k * Σ_{s=t-k+1}^t r_s

Features core:
- trend_ret_20d
- trend_ret_63d

Opcional:
- ma_spread_50_200
- trend_persistence_proxy

10.4 Breadth

    pct_above_ma_{k,t}
    =
    #{ i ∈ U_t : p_{i,t} > MA_{k,i,t} } / |U_t|

Features core:
- pct_above_ma_50
- pct_above_ma_200

Opcionales:
- advance_decline_ratio
- breadth_slope

10.5 Dispersión cross-sectional

La dispersión cross-sectional no debe usar por defecto varianza cruda si el universo tiene heavy tails o microcaps extremas.

Default canónico:
escala robusta tipo MAD:

    disp_t
    =
    1.4826 *
    median_{i∈U_t}
    | r_{i,t} - median_{j∈U_t}(r_{j,t}) |

Alternativa aceptable:
- winsorized variance cross-sectional
si está formalmente definida y auditada.

10.6 Concentración de correlaciones

Construir una matriz de retornos rolling:

    R_t ∈ R^{l × n_t}

sobre símbolos con cobertura suficiente en [t-l+1, t].

Matriz de correlación muestral:

    C_t^sample

Si use_corr_shrinkage = true:

    C_t^shr = (1 - δ_t) C_t^sample + δ_t I

donde:
- δ_t puede ser fijo o estimado por un shrinkage canónico;
- el target por defecto es identidad.

Feature core:

    corr_conc_t
    =
    λ_1(C_t^shr) / trace(C_t^shr)

Como C_t es matriz de correlación:
trace(C_t^shr) = n_t

por tanto:

    corr_conc_t = λ_1(C_t^shr) / n_t

Interpretación:
- valor alto = fuerte dominancia del factor común;
- valor bajo = estructura más distribuida.

Regla de implementación:
si n_t es grande, no es necesario autodescomposición completa; basta el mayor autovalor.

10.7 Ilíquidez / fricción

Feature por símbolo tipo Amihud:

    illiq_{i,t}
    =
    |r_{i,t}| / max(notional_{i,t}, ε)

Agregación core:
usar mediana o trimmed mean robusta:

    illiq_t = Agg_{i∈U_t}(illiq_{i,t})

No usar media simple como default si existen colas extremas.

10.8 Stress proxies

Ejemplo 1:
tail loss frequency

    tail_loss_freq_t^(m,α)
    =
    Σ_{s=t-m+1}^t 1{ r_s < q_{α,t}^{rolling} }

Ejemplo 2:
drawdown speed

    dd_speed_t
    =
    dd_t - dd_{t-1}

o una versión suavizada/robusta.

11. Features internas vs proxies externos

11.1 Features internas
Se derivan exclusivamente del universo y benchmark del sistema.
Deben constituir la base canónica del módulo.

11.2 Proxies externos
Pueden incluir:
- VIX
- spreads de crédito
- slope de tipos
- cross-asset returns
- ETFs sectoriales externos

11.3 Regla de diseño
El sistema debe ser funcional y útil sin proxies externos.
Los externos solo pueden añadir señal marginal, nunca ser soporte estructural.

11.4 Requisitos por proxy externo
Toda proxy externa debe declarar:
- frecuencia real de actualización;
- lag de disponibilidad;
- forward-fill policy;
- max_staleness;
- fallback policy.

12. Transformaciones robustas y pipeline de feature engineering

Pipeline recomendado por feature:

1. cálculo bruto:
   z_t^(raw)

2. clipping/winsor causal:
   z_t^(clip)

3. normalización robusta rolling:
   z_t^(norm)

4. rank-gauss opcional:
   z_t^(rg)

5. smoothing causal opcional:
   z_t^(smooth)

Regla:
no deben convivir múltiples versiones equivalentes de una misma feature en el set canónico salvo justificación fuerte.

13. Normalización robusta

Default:

    z_t^(norm)
    =
    ( z_t - median(z_{t-L:t}) )
    /
    ( MAD(z_{t-L:t}) + ε )

donde:
- L = lookback_norm
- MAD = median(|z - median(z)|)

Si use_rank_gauss = true:

    u_t = Fhat_t(z_t)
    z_t^(rg) = Φ^{-1}( clip(u_t, ε_u, 1-ε_u) )

Objetivo:
- homogeneizar escalas,
- reducir sensibilidad a outliers,
- facilitar inferencia downstream.

14. Smoothing causal

El smoothing no debe definirse con un α arbitrario difícil de interpretar.
Se parametriza por half-life h:

    zbar_t = α z_t + (1-α) zbar_{t-1}
    α = 1 - 2^(-1/h)

Reglas:
- h corto para features muy ruidosas;
- h largo para proxies lentos;
- no suavizar si la feature debe reaccionar rápidamente a transiciones reales.

15. Confidence score y validez por fecha

Cada fecha t debe incluir un confidence_score_t ∈ [0,1].

Este score resume:
- tamaño de universo relativo;
- missingness;
- staleness de proxies;
- estabilidad de estimadores cross-sectional;
- fracción de familias válidas.

Ejemplo conceptual:

    confidence_t
    =
    w_u c_u(t)
    + w_m c_m(t)
    + w_s c_s(t)
    + w_p c_p(t)

Además:
- feature_valid_flag por feature crítica;
- family_valid_flag por familia;
- low_confidence_flag_t si confidence_t cae por debajo del umbral.

16. Gobernanza de redundancia y selección canónica

16.1 Principio
La selección del conjunto final no debe ser una optimización abierta ejecutada online en cada run.
Debe ser una política offline, versionada y auditable.

16.2 Secuencia recomendada

Paso 1: quality filter
Eliminar features con:
- cobertura insuficiente,
- missing elevado,
- staleness recurrente,
- varianza efectiva casi nula.

Paso 2: redundancy filter
Si dos features tienen correlación absoluta persistentemente alta:
- conservar la más interpretable,
- la más estable,
- o la más útil downstream;
- mover la otra a experimental.

Paso 3: downstream ablation
Evaluar offline si la feature mejora:
- separabilidad del HMM,
- estabilidad inter-run,
- calidad del regime_gating.

Paso 4: freeze
Congelar el conjunto final por policy_version.

16.3 Sobre selección L1
L1 o Lasso pueden usarse como herramienta exploratoria offline.
No deben ser el motor principal de selección porque:
- el objetivo aquí es mayormente no supervisado;
- un target tipo HMM likelihood puede introducir circularidad;
- y la estabilidad de selección puede ser pobre.

17. Invariantes del módulo

17.1 Causalidad
Toda feature usa solo información disponible hasta t.

17.2 Cobertura
Las features cross-sectional no son plenamente válidas si |U_t| < n_min.

17.3 Robustez
Las features core usan agregadores robustos o clipping si la familia lo requiere.

17.4 Freeze del core set
El set canónico no cambia silenciosamente entre runs.

17.5 Funcionalidad sin externos
El módulo sigue siendo operativo sin proxies externos.

18. Complejidad computacional

18.1 Bajo coste
- vol
- vov
- trend
- breadth
- illiquidity aggregation

18.2 Alto coste
- corr_conc_t

Para corr_conc:
- evitar espectro completo si n_t es grande;
- usar mayor autovalor solamente;
- usar shrinkage si la muestra es ruidosa;
- exigir cobertura mínima por símbolo.

19. Failure modes principales

- colapso del universo en ciertas fechas;
- correlación rolling inestable por mala cobertura;
- iliquidez agregada dominada por pocos nombres;
- proxies externos stale;
- smoothing excesivo que borra cambios reales;
- drift no controlado del set canónico.

20. Criterios de aceptación del run

Un run se considera accepted solo si:
- no hay leakage;
- coverage suficiente del core set;
- confidence razonable en la mayoría de fechas;
- redundancia controlada;
- proxies externos dentro de política de staleness;
- set canónico coincide con una versión permitida;
- no deteriora materialmente el stack downstream en validación offline.

21. Tests mínimos obligatorios

- rolling windows estrictamente causales;
- outputs sin NaN inesperados en features core;
- confidence_score en [0,1];
- corr_conc reproducible bajo misma config;
- fallback de proxies externos correctamente auditado;
- freeze del core set reproducible;
- feature dictionary consistente con outputs.

23. MVP vs versión avanzada

23.1 MVP
- familias canónicas internas
- robust aggregation
- robust z-score
- rank-gauss opcional
- EMA por half-life
- shrinkage opcional en corr_conc
- confidence score
- freeze del core set

23.2 Avanzado
- shrinkage adaptativo formal
- variantes liquidity-weighted o size-weighted como companion features
- sparse selection offline secundaria
- descomposición sectorial
- cálculo incremental de corr_conc
- calibración más formal de confidence score

24. Qué NO conviene meter ahora

No conviene introducir:
- autoencoders
- diffusion models
- causal representation learning
- selección automática online del feature set
- exceso de proxies externos frágiles
- target-supervised feature mining agresivo

25. Interpretación correcta

La lectura correcta de regime_features.py no es:
“generamos muchas features de mercado”.

La lectura correcta es:
“construimos una representación observable, robusta y causal del estado de mercado, de baja dimensión efectiva, estable en el tiempo y útil para la inferencia posterior de regímenes”.

26. Conclusión

models/regimes/regime_features.py debe ser una infraestructura de representación, no una colección ornamental de indicadores.

Su misión es:
- resumir el entorno,
- preservar causalidad,
- resistir colas y universos cambiantes,
- controlar redundancia,
- y entregar a hmm.py una base estadística estable y económicamente interpretable.

Ese es exactamente el comportamiento que debe tener una capa seria de regime representation en un stack cuantitativo institucional.
