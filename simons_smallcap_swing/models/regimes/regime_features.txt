Modulo: models/regimes/regime_features.py

1) Objetivo
Construir features de regimen de mercado (estado macro/micro de riesgo, volatilidad, liquidez y tendencia) para clasificar contextos de mercado y habilitar decision adaptativa en modelos y portfolio.

2) Alcance
Incluye:
- Ingenieria de variables de estado de mercado multi-horizonte.
- Features de volatilidad, correlacion, breadth, stress y liquidez.
- Normalizacion temporal robusta y smoothing.
- Persistencia PIT de regime_features para HMM/regime_gating.

No incluye:
- Entrenamiento del modelo de regimen (hmm.py).
- Aplicacion de politica de gating final (regime_gating.py).

3) Dependencias aguas arriba y aguas abajo
Upstream:
- data/price/adjusted (indices, proxy mercado, universo)
- data/price/market_proxies (si existe)
- data/universe/history
- opcional: macro proxies/cross-asset proxies

Downstream:
- models/regimes/hmm.py
- models/regimes/regime_gating.py
- validation/walkforward_bias.py

4) Taxonomia de features de regimen
Volatilidad:
- realized_vol_10d/20d/63d
- vol_of_vol
- jump proxy (tail moves)

Tendencia/momentum de mercado:
- trend_ret_20d/63d en benchmark
- moving-average spread

Correlacion y dispersion:
- avg_pair_corr_cross_section
- dispersion_returns_cross_section
- correlation_regime_index

Liquidity/risk-off:
- market_turnover_proxy
- spread_proxy_index
- illiquidity_proxy (Amihud-like aggregate)

Breadth/sentiment tecnico:
- pct_symbols_above_ma_50/200
- advance_decline_ratio

Stress proxies:
- tail_loss_frequency
- drawdown_speed

5) Contrato de entrada
Entradas minimas:
- market_prices_path
- universe_history_path
- regime_features_config_path
- start_date/end_date
- run_id

Opcionales:
- macro_proxy_path
- sector_indices_path

Campos minimos esperados:
- date
- close/volume para proxies de mercado
- universe constituents por fecha

Parametros clave de config:
- lookbacks
- smoothing_method (ema/rolling median)
- winsor_limits
- standardization_mode
- min_history_days

6) Contrato de salida
- models/regimes/features/date={yyyy-mm-dd}/part-*.parquet
- models/regimes/features/regime_feature_dictionary_{run_id}.json
- models/regimes/features/regime_feature_summary_{run_id}.json
- models/regimes/features/manifest_{run_id}.json

Campos minimos:
- date
- regime_feature_{name}
- feature_valid_flag
- run_id
- config_hash

7) Reglas PIT y anti-leakage
- cada feature de fecha t usa solo datos <= t (o <= t-1 segun decision timing).
- no backfill con datos de futuro.
- si hay forward-fill de datos externos, limitar con max_staleness.
- registrar corte temporal de cada fuente en manifest.

8) Flujo del proceso (alto nivel)
1. Cargar fuentes y config.
2. Construir series base de mercado.
3. Calcular proxies por familia de features.
4. Aplicar smoothing/winsor/normalizacion.
5. Ejecutar validaciones de cobertura/sanidad.
6. Persistir outputs y metadata.

9) Transformaciones recomendadas
- zscore temporal robusto (rolling mediana/MAD)
- rank_gauss para colas pesadas
- smoothing EMA para evitar chattering de regimen
- clipping a percentiles extremos

10) Validaciones minimas
- cobertura temporal continua en rango objetivo
- sin NaN criticos en features core
- valores dentro de rangos esperables por feature
- correlaciones extremadamente altas entre features flagged

11) Manejo de casos especiales
- fechas con pocos activos en universo:
  - flag de baja confianza para features cross-sectional.
- gaps de mercado/holiday mismatch:
  - alinear a calendario canonico.
- proxy faltante externo:
  - fallback interno + warning.

12) Observabilidad y trazabilidad
Logging:
- INFO: progreso por familia de features.
- WARNING: cobertura baja o proxies inestables.
- ERROR: falta de inputs criticos.

Metricas operativas:
- n_features_regime
- n_dates
- pct_missing_core_features
- n_low_confidence_dates
- runtime_sec

15) Criterios de aceptacion (Definition of Done)
- dataset de regime features PIT reproducible.
- cobertura y sanidad suficientes para entrenar HMM.
- metadata y diccionario listos para auditoria.

16) Casos de prueba minimos (para tests/)
- calculo correcto de vol/trend en serie sintetica.
- no leakage temporal en ventanas rolling.
- handling de datos faltantes segun policy.
- smoothing reduce ruido sin desplazar fechas indebidamente.

17) Riesgos y decisiones abiertas
- set final de features (parsimonia vs expresividad).
- sensibilidad a smoothing y lookbacks.
- dependencia de proxies externos.
