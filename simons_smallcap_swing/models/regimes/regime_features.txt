\section{Ingeniería de \texttt{regime\_features} para detección de regímenes en swing trading}

\subsection{Objetivo del módulo}
El módulo \texttt{models/regimes/regime\_features.py} construye \textbf{features de régimen de mercado} que capturan el \textbf{estado macro/micro de riesgo}, \textbf{volatilidad}, \textbf{liquidez}, \textbf{tendencia} y \textbf{amplitud de mercado} sobre diferentes horizontes temporales, con el fin de:
\begin{itemize}
  \item Clasificar contextos de mercado (\emph{risk‑on} vs \emph{risk‑off}), regímenes de volatilidad y liquidity shocks.
  \item Habilitar decisiones adaptativas en modelos cuantitativos (HMM, regime‑gating, portfolio risk allocation) \cite{Hamilton2010, Ang2012, Admati1988}.
\end{itemize}
Formalmente, el módulo transforma series de precios y datos de mercado en un vector de \texttt{regime\_features} \(\mathbf{z}_t \in \mathbb{R}^p\) cuya estructura funcional y escala son robustas a cola pesada, ruido y cambios estructurales.

\subsection{Taxonomía de \texttt{regime\_features}}
\begin{itemize}
  \item \textbf{Volatilidad}:
    \begin{itemize}
      \item \texttt{realized\_vol\_10d/20d/63d}: \(\sigma^{(k)}_t = \sqrt{\frac{1}{k}\sum_{i=t-k+1}^{t} r^2_i}\), donde \(r_i\) es el retorno diario del índice de mercado.
      \item \texttt{vol\_of\_vol}: \(\sqrt{\text{Var}(\sigma^{(k)}_s\ \mid\ s=t-m+1,\dots,t)}\), captura “volatility of volatility” como proxy de incertidumbre de segundo orden \cite{Graves2012}.
      \item \texttt{jump\_proxy}: \(\mathbb{I}(|r_t| > \text{threshold})\) o estadístico de colas \(\text{VaR}_\alpha(r_t)\), para identificar spikes de volatilidad.
    \end{itemize}
  \item \textbf{Tendencia / momentum de mercado}:
    \begin{itemize}
      \item \texttt{trend\_ret\_20d/63d}: \(\mu^{(k)}_t = \frac{1}{k} \sum_{i=t-k+1}^{t} r_i\), media de retornos en el horizonte.
      \item \texttt{ma\_spread\_50/200}: \(\text{MA}_{50,t} - \text{MA}_{200,t}\), donde \(\text{MA}_{k,t} = \frac{1}{k} \sum_{i=t-k+1}^{t} p_i\) para el precio de mercado \cite{Moskowitz2012}.
    \end{itemize}
  \item \textbf{Correlación y dispersión}:
    \begin{itemize}
      \item \texttt{avg\_pair\_corr\_cross\_section}: \(\frac{2}{n(n-1)} \sum_{i<j} \text{Corr}(r_{i,t:l}, r_{j,t:l})\), correlación media entre pares de activos del universo en ventana \([t-l, t]\) \cite{Chen2004}.
      \item \texttt{dispersion\_returns\_cross\_section}: \(\text{Var}(\{r_{i,t}\}_{i\in\mathcal{U}_t})\), varianza de retornos de universo, proxy de “picking risk” y oportunidad de alojamiento multi‑factor \cite{Pástor2017}.
      \item \texttt{correlation\_regime\_index}: \(\text{eigen\_ratio}(\mathbf{C}_t)\), donde \(\mathbf{C}_t\) es la matriz de correlaciones de universo.
    \end{itemize}
  \item \textbf{Liquidez / risk‑off}:
    \begin{itemize}
      \item \texttt{market\_turnover\_proxy}: \(\sum_{i\in\mathcal{U}_t} v_{i,t} / \text{mcap}_{i,t}\) como flujo relativo de volumen.
      \item \texttt{spread\_proxy\_index}: \(\sum_{i\in\mathcal{U}_t} \text{spread}_i v_{i,t} / \sum v_{i,t}\), media ponderada por volumen.
      \item \texttt{illiquidity\_proxy} (tipo Amihud agregado): \(\sum_{i\in\mathcal{U}_t} \frac{|r_{i,t}|}{v_{i,t}} \big/ \sum v_{i,t}\), captura el impacto de volumen sobre el precio de mercado \cite{Amihud2002, Hasbrouck2013}.
    \end{itemize}
  \item \textbf{Breadth / sentimiento técnico}:
    \begin{itemize}
      \item \texttt{pct\_symbols\_above\_ma\_50/200}: \(\#\{i \in \mathcal{U}_t : p_{i,t} > \text{MA}_{k,i,t}\} / |\mathcal{U}_t|\), grado de participación alcista/bajista.
      \item \texttt{advance\_decline\_ratio}: \(\text{advances}_t / \text{declines}_t\), medida clásica de breadth de mercado \cite{Kraus1989}.
    \end{itemize}
  \item \textbf{Stress proxies}:
    \begin{itemize}
      \item \texttt{tail\_loss\_frequency}: \(\sum_{t=m}^{t} \mathbb{I}(r_t < -\text{VaR}_\alpha)\), frecuencia de pérdidas extremas.
      \item \texttt{drawdown\_speed}: \(\Delta (\text{max\_drawdown}_t)\) por ventana, para identificar aceleración de correcciones \cite{Benoit2017}.
    \end{itemize}
\end{itemize}

\subsection{Transformaciones y normalización robustas}
Cada \(\texttt{regime\_feature}_{t}\) se construye bajo principio de \emph{no–look‑ahead} y se transforma para:
\begin{itemize}
  \item \textbf{Robust z‑score}: \(\tilde{z}_t = \frac{z_t - \text{med}_{t-l}^{t}(z_s)}{\text{MAD}_{t-l}^{t}(z_s)}\), usando \texttt{rolling mediana} y \texttt{MAD} para colas pesadas \cite{Huber2011}.
  \item \textbf{rank‑gauss}: \(\text{rank}(z_t)\) transformada a cuantiles gaussianos \(\Phi^{-1}(F(z_t))\) para estabilizar colas \cite{Rubinstein2017}.
  \item \textbf{Smoothing EMA}: \(\tilde{z}_t^{\text{smooth}} = \alpha z_t + (1-\alpha)\tilde{z}_{t-1}^{\text{smooth}}\), con \(\alpha \propto 1/\text{lookback}\), evitando \emph{chattering} en señales de régimen \cite{Hamilton2010}.
  \item \textbf{clipping}: limitado a percentiles \texttt{winsor\_limits} para evitar outlier‑inflation de momentos.
\end{itemize}

\subsection{Reglas PIT y anti‑leakage}
\begin{itemize}
  \item Todas las features de fecha \(t\) se calculan únicamente con información \(\leq t\) (o \(\leq t-1\) si el timing de entrada exige lag de ejecución).
  \item No se backfill con datos futuros: \(\texttt{lookback}\) y \(\texttt{min\_history\_days}\) garantizan que las series sean \emph{causales} y reproducibles en walk‑forward.
  \item En caso de \texttt{forward‑fill} de proxies externos, se limita \texttt{max\_staleness} y se registra en \texttt{manifest} el origen y fecha de última actualización \cite{Bailey2014}.
\end{itemize}

\subsection{Flujo de procesamiento (alto nivel)}
\begin{enumerate}
  \item \texttt{load\_regime\_feature\_config} carga \texttt{lookbacks}, \texttt{smoothing\_method}, \texttt{winsor\_limits}, \texttt{standardization\_mode}.
  \item \texttt{load\_market\_inputs} integra \texttt{market\_prices\_path}, \texttt{universe\_history\_path}, opcionalmente \texttt{macro\_proxy\_path}, \texttt{sector\_indices\_path}.
  \item \texttt{build\_volatility\_regime\_features}, \texttt{build\_trend\_regime\_features}, \texttt{build\_corr\_dispersion\_features}, \texttt{build\_liquidity\_stress\_features} construyen familias de features \(\mathbf{z}_t\).
  \item \texttt{normalize\_and\_smooth\_features} aplica z‑score, rank‑gauss, EMA, clipping.
  \item \texttt{validate\_regime\_features} comprueba cobertura, rangos y correlaciones extremas.
  \item \texttt{persist\_regime\_feature\_outputs} serializa:
    \begin{itemize}
      \item \texttt{models/regimes/features/date=\{yyyy-mm-dd\}/part-*.parquet} por partición temporal.
      \item \texttt{regime\_feature\_dictionary\_\{run\_id\}.json} (descripción de cada \texttt{regime\_feature\_\{name\}}).
      \item \texttt{regime\_feature\_summary\_\{run\_id\}.json} (estadísticas por feature y fecha).
      \item \texttt{manifest\_\{run\_id\}.json} (metadatos de \texttt{run\_id}, ventanas, features, \texttt{config\_hash}).
    \end{itemize}
\end{enumerate}

\subsection{Validaciones mínimas}
\begin{itemize}
  \item Cobertura temporal continua en \([start\_date, end\_date]\), alineada a calendario canónico.
  \item Sin \texttt{NaN} en features core (\texttt{realized\_vol\_20d, trend\_ret\_20d}, \texttt{illiquidity\_proxy}, etc.).
  \item Rangos por feature compatibles con literatura (e.g. volatilidad anualizada en \([0.05, 0.6]\), \texttt{advance\_decline\_ratio} en \([0.1, 10]\)).
  \item Correlaciones \(\rho_{ij} > 0.95\) entre features redundantes etiquetadas para posible reducción de dimensionalidad (PCA / factorization).
\end{itemize}

\subsection{Manejo de casos especiales}
\begin{itemize}
  \item \texttt{pocos activos} en universo: \texttt{feature\_valid\_flag} marcado como bajo nivel de confianza para features cross‑sectionales (\texttt{avg\_pair\_corr}, \texttt{dispersion\_returns}).
  \item \texttt{gaps de mercado} o \texttt{holiday mismatch}: se alinea todo a calendario canónico usando \texttt{next\_business\_date} y \texttt{fillna(limit=max\_staleness)}.
  \item \texttt{proxy externo faltante} (macro, cross‑asset): se utiliza proxy interno con \texttt{fallback} y \texttt{WARNING} en log, con clara documentación en \texttt{manifest}.
\end{itemize}

\subsection{Observabilidad y trazabilidad}
Logging:
\begin{itemize}
  \item \texttt{INFO}: \texttt{build\_volatility\_regime\_features} completado, \texttt{build\_corr\_dispersion\_features} completado, etc.
  \item \texttt{WARNING}: \texttt{pct\_missing\_core\_features} alto, \texttt{low\_confidence\_dates} identificados, correlaciones extremas.
  \item \texttt{ERROR}: falta de \texttt{market\_prices\_path} o \texttt{universe\_history\_path}.
\end{itemize}
Métricas operativas:
\begin{itemize}
  \item \texttt{n\_features\_regime}.
  \item \texttt{n\_dates}.
  \item \texttt{pct\_missing\_core\_features}.
  \item \texttt{n\_low\_confidence\_dates}.
  \item \texttt{runtime\_sec}.
\end{itemize}

\subsection{Reglas de diseño de \texttt{regime\_features} para modelo HMM y regime‑gating}
\begin{itemize}
  \item \textbf{Parsimonia}: un subconjunto \(\mathcal{I} \subset \{1,\dots,p\}\) con \texttt{AIC} o \texttt{BIC} óptimo para HMM gaussiano \cite{Bozdogan1987}.
  \item \textbf{Multi‑horizonte}: combinación de \texttt{realized\_vol\_10d} (micro‑risk) y \texttt{vol\_63d} (macro‑regime) para capturar escala de tiempo de cambios de régimen \cite{Andersen2016}.
  \item \textbf{Liquidity/stress}: \texttt{illiquidity\_proxy} y \texttt{tail\_loss\_frequency} permiten separar estados \emph{risk‑on} (liquidez alta, bumps de vol moderados) de \emph{risk‑off/crisis} (liquidez baja, colas extremas).
  \item \textbf{Breadth/tendencia}: \texttt{pct\_symbols\_above\_ma} y \texttt{trend\_ret\_63d} definen regímenes de momentum fuerte vs debilidad de tendencia, críticos para swing‑trading multi‑horizonte \cite{Moskowitz2012}.
\end{itemize}

\subsection{Casos de prueba mínimos}
\begin{itemize}
  \item \texttt{build\_volatility\_regime\_features} calcula \(\sigma^{(20)}_t\) correctamente en serie sintética con conocimiento analítico.
  \item No hay \emph{leakage temporal} en ventanas rolling: \(\texttt{regime\_feature}_t\) no depende de \(\texttt{close}_{t+1}\).
  \item \texttt{normalize\_and\_smooth\_features} maneja \texttt{NaN} según política (forward / backward / interpolación), con \texttt{feature\_valid\_flag} bien seteado.
  \item \texttt{EMA} y \texttt{clipping} reducen ruido sin transladarse fuera de \texttt{[t-1, t]}\ en el tiempo.
\end{itemize}

\subsection{Decisiones de riesgo abiertas}
\begin{itemize}
  \item \textbf{Set final de features}: trade‑off entre expresividad (modelo HMM más rico) y riesgo de overfitting y \emph{transaction cost} inducido por ruido \cite{Bailey2014}.
  \item \textbf{Sensibilidad a \texttt{smoothing} y \texttt{lookbacks}}: mayor \texttt{lookback} estabiliza features pero introduce lag, crítico en swing trading con horizonte \(\sim\) 20–100 días \cite{Hamilton2010}.
  \item \textbf{Dependencia de proxies externos}: si se añaden \texttt{macro\_proxy\_path}, se debe evaluar \texttt{refresh\_frequency} y \texttt{data\_lag} para evitar \emph{regime mistagging} \cite{Adrian2019}.
\end{itemize}

Este módulo es la base de toda la \emph{regime‑aware pipeline} (HMM, regime‑gating, risk‑allocation) y debe ser \textbf{PIT‑reproducible, robusto a ruido} y \textbf{versionado} para su consumo en backtesting industrial y producción en vivo.
