Especificación Cuantitativa: models/inference/explain.py

0. Propósito y papel dentro del stack

models/inference/explain.py define la capa institucional de explicabilidad descriptiva del sistema de inferencia. Su función no es reinterpretar el modelo en abstracto ni ofrecer narrativas causales, sino generar artefactos de explicación reproducibles, auditables y útiles para debugging, monitoring, control de drift explicativo y validación cuantitativa de hipótesis de señal. La base que me has pasado ya cerraba muy bien este propósito, el alcance funcional, la separación entre explicación y scoring, la trazabilidad por run_id/model_version/data_hash y la política explícita de resolución de método según tipo de modelo y presupuesto computacional. :contentReference[oaicite:0]{index=0}

En un sistema de small-cap swing trading, explain.py debe entenderse como instrumentación científica del modelo en producción. Su papel no es “demostrar por qué el modelo tiene razón”, sino responder de forma controlada a preguntas operativas como:
- qué familias de features dominan el score;
- si una promoción o retrain ha alterado la jerarquía explicativa;
- si las explicaciones locales extremas son coherentes o sospechosas;
- si la estructura explicativa se mantiene estable entre ventanas y regímenes.

La regla de diseño debe ser prudente:

    toda explicación producida por el módulo
    es una descomposición descriptiva del comportamiento del modelo
    sobre una ejecución concreta,
    no una inferencia causal ni una validación económica por sí sola.

1. Objetivo formal del módulo

El módulo implementa una transformación del tipo:

    E : (model_ref, X_inference, ŷ_persisted, config, run_id)
        -> (global_explanations, local_explanations, explain_summary, manifest)

donde:
- model_ref resuelve a una versión exacta del modelo;
- X_inference es el conjunto de features efectivamente usado en scoring;
- ŷ_persisted son las predicciones ya generadas y persistidas;
- config fija método, baseline, presupuesto y política de muestreo;
- run_id identifica unívocamente la ejecución de explicación.

El output debe incluir, como mínimo:
- explicabilidad global por feature y por familia;
- explicabilidad local por observación cuando el método lo permita;
- métricas de consistencia y cierre numérico;
- metadatos de trazabilidad completa.

El módulo no entrena, no recalibra y no altera scores ya persistidos. Sólo explica lo que ya fue puntuado.

2. Principios no negociables

P1. Separación estricta entre scoring y explicación.
explain.py no modifica predicciones ya persistidas.

P2. Fail-closed ante inconsistencia.
Si el contrato entre modelo, features y predicciones no se cumple, la ejecución falla.

P3. Trazabilidad completa.
Toda explicación debe enlazarse con model_version, inference_run_id, data_hash, explain_config_hash y código/librerías.

P4. Método explícito.
El método de explicación no puede quedar implícito en el implementador; debe resolverse por política documentada.

P5. Baseline explícito.
Toda explicación aditiva debe fijar y persistir su baseline y conjunto de referencia.

P6. Prudencia semántica.
Las explicaciones son descriptivas, no causales.

P7. Presupuesto gobernado.
Si el método exacto no cabe dentro del presupuesto computacional, el módulo debe degradar de forma explícita o fallar, nunca recortar silenciosamente.

3.3. Validaciones previas obligatorias

Antes de calcular cualquier explicación, el módulo debe verificar:
- compatibilidad exacta entre schema del modelo y features persistidas;
- reconciliación entre predictions_path y features_path sobre claves lógicas;
- consistencia de model_version;
- ausencia de NaN/inf en inputs críticos;
- disponibilidad de todas las columnas requeridas por el método resuelto.

Si falla cualquiera de estas validaciones, la ejecución debe abortar y no generar artefactos parciales silenciosos.

4. Resolución formal del método

La base ya fija una política correcta de resolución por tipo de modelo. Conviene convertirla en una matriz institucional explícita. :contentReference[oaicite:2]{index=2}

4.1. Method Resolution Matrix recomendada

    ┌────────────┬────────────┬──────────────────┬─────────────┐
    │ Model Type │ Budget Low │ Budget Medium    │ Budget High │
    ├────────────┼────────────┼──────────────────┼─────────────┤
    │ Linear     │ coef       │ coef             │ coef        │
    │ GBDT       │ PFI_global │ TreeSHAP_global  │ TreeSHAP_*  │
    │ Ensemble   │ PFI_global │ component+PFI    │ component_* │
    └────────────┴────────────┴──────────────────┴─────────────┘

    * = global + local si cobertura y coste lo permiten

4.2. Reglas duras

- Si el modelo es lineal y la transformación entre entrada persistida y score es lineal y conocida, el método por defecto debe ser coef.
- Si el modelo es GBDT compatible y el presupuesto lo permite, el método por defecto debe ser TreeSHAP.
- Si TreeSHAP no cabe en presupuesto o no es compatible, debe usarse permutation importance para global y desactivar local salvo override explícito.
- Si el modelo es ensemble, debe intentarse explicación por componente + agregación cuando matemáticamente sea válida; en otro caso, explicación global conservadora del ensemble final.
- Forzar un método incompatible debe producir error explícito.

Esto elimina ambigüedad y hace el módulo realmente implementable.

5. Baseline y conjunto de referencia

Toda explicación aditiva debe explicitar el baseline. La base ya lo deja bien planteado; aquí conviene cerrar las validaciones cuantitativas. :contentReference[oaicite:3]{index=3}

5.1. Definición base

Para métodos aditivos:

    ŷ(x) ≈ ϕ_0 + Σ_j ϕ_j(x)

donde:
- ϕ_0 es el valor esperado del modelo sobre un conjunto de referencia reproducible;
- ϕ_j(x) es la contribución atribuida a la feature j.

5.2. Baseline obligatorio

El módulo debe persistir:
- tipo de baseline;
- tamaño del background set;
- política de muestreo;
- hash o IDs de filas del baseline set.

5.3. Baseline convergence check

Debe añadirse una validación explícita:

    | E[ ŷ(X_ref) ] - ϕ_0 | <= δ_baseline

con δ_baseline configurable y suficientemente estricto para el método usado. Esto mejora mucho la confiabilidad de ϕ_0 y evita que el baseline quede como una noción difusa.

6. Metodologías soportadas

6.1. Modelos lineales

Para un modelo lineal:

    ŷ(x) = β_0 + Σ_j β_j x_j

la explicación local debe definirse como:

    ϕ_j(x) = β_j x_j

siempre que x corresponda exactamente al espacio consumido por el modelo.

Deben reportarse al menos dos medidas globales:

    I_j^(coef) = |β_j|

    I_j^(contrib) = (1/n) Σ_i |β_j x_{ij}|

La primera refleja sensibilidad estructural. La segunda, importancia efectiva sobre la muestra explicada.

6.2. GBDT: TreeSHAP como método principal

Para árboles, el método principal debe ser TreeSHAP, con:
- explicaciones locales por observación;
- importancia global como media de contribuciones absolutas:

    ϕ̄_j = (1/n) Σ_i |ϕ_j(x_i)|

Debe verificarse siempre el cierre aditivo:

    | ŷ(x_i) - (ϕ_0 + Σ_j ϕ_j(x_i)) | <= ε_closure

y persistir el error de cierre por observación.

6.3. Permutation Feature Importance como método global complementario

La definición operativa debe ser:

    PFI_j = E[ L(y, ŷ_perm^j) ] - E[ L(y, ŷ) ]

donde L es la loss fijada en configuración.

PFI debe interpretarse como sensibilidad predictiva global, no como atribución local exacta. Deben persistirse:
- métrica L;
- número de repeticiones;
- seeds;
- universo de evaluación;
- dispersión entre repeticiones.

6.4. Ensembles

Caso aditivo lineal entre componentes:

    ŷ(x) = Σ_b w_b ŷ^(b)(x)

si los pesos w_b son conocidos y constantes, puede agregarse:

    ϕ_j(x) = Σ_b w_b ϕ_j^(b)(x)

Caso no lineal o no aditivo:
- explicar componentes cuando sea posible;
- producir explicación global conservadora del ensemble final;
- marcar explícitamente que no existe lectura aditiva exacta del ensemble completo.

7. Axiomas y consistencia de explicaciones aditivas

Este es uno de los puntos que faltaba reforzar para subir el nivel del módulo. Para métodos tipo SHAP, explain.py debe verificar y documentar explícitamente, cuando aplique, los axiomas relevantes del marco de additive feature attribution:

7.1. Efficiency
La suma de atribuciones más baseline debe reconstruir la predicción:

    Σ_j ϕ_j(x) + ϕ_0 = ŷ(x)

hasta tolerancia ε.

7.2. Dummy
Si una feature no afecta al modelo en el contexto del método y del background set, su atribución debe ser 0 o numéricamente despreciable.

7.3. Symmetry
Features intercambiables bajo el modelo y la distribución de referencia no deberían recibir atribuciones sistemáticamente distintas.

7.4. Linearity
Cuando el modelo explicado pertenece a una familia aditiva compatible, la combinación lineal de modelos debe reflejarse en la combinación lineal de atribuciones.

No todas estas propiedades podrán testearse exhaustivamente en producción para todos los métodos, pero el manifest debe indicar al menos:
- qué axiomas aplican;
- cuáles se verificaron numéricamente;
- cuáles quedan como “not applicable”;
- y cuáles no se comprobaron.

8. Faithfulness cuantitativa

La base ya incluía cierre aditivo y consistencia, pero eso no basta para medir fidelidad real del ranking de features. Aquí conviene introducir una validación cuantitativa mínima.

8.1. Deletion AUC

Debe permitirse, al menos opcionalmente y recomendado por defecto para explain_mode global de modelos no lineales, una curva de deletion:
- ordenar features por |ϕ_j(x)| o importancia local/media;
- eliminar progresivamente las más importantes;
- medir la degradación del score o de la salida del modelo.

La métrica resumen puede definirse como AUC de deletion curve. La interpretación institucional debe ser:
- AUC alta o degradación rápida cuando se eliminan las top features -> explicaciones más fieles;
- degradación plana -> explicaciones posiblemente poco informativas.

8.2. Insertion curve
Opcionalmente, puede añadirse la curva inversa:
- partir de baseline o input neutral;
- insertar features por orden de importancia;
- medir cuánto se reconstruye el score.

Estas métricas no deben bloquear el MVP, pero sí deben formar parte de la primera fase seria del módulo.

9. Estabilidad temporal de importancias

La base ya contempla stability_report y comparación entre ventanas. Conviene convertir eso en una validación cuantitativa explícita. :contentReference[oaicite:4]{index=4}

9.1. Rank-stability entre ventanas

Para ventanas consecutivas u, u+1, el módulo debe calcular:
- Spearman rank correlation entre rankings globales;
- opcionalmente Kendall-τ;
- cambios de top-k families/features.

Ejemplo de criterio útil:

    Spearman( rank_t , rank_{t+1} ) >= ρ_min

con ρ_min configurable.

9.2. Interpretación

- estabilidad alta -> estructura explicativa robusta;
- estabilidad muy baja -> posible drift, colinealidad, retrain no estable o cambio de régimen.

9.3. Política

La estabilidad temporal no debe usarse para rechazar automáticamente toda ejecución, pero sí para:
- generar alertas;
- alimentar monitoring;
- y contextualizar cualquier lectura de global importance.

10. Cobertura y presupuesto computacional

10.1. Cobertura mínima

Debe cumplirse:

    n_explained / n_eligible >= γ_min

con γ_min configurable.

10.2. Política de muestreo

Si el universo es demasiado grande para explicaciones locales completas, debe usarse muestreo persistido y reproducible:
- estratificado por score,
- por fecha,
- por símbolo,
- por cuantiles,
- o mixto.

Nunca se permite truncado silencioso.

10.3. Degradación por presupuesto

Si el método preferido excede max_compute_budget_sec:
- degradar explícitamente a un método más barato permitido por la matrix de resolución;
- o fallar si la configuración prohíbe degradación.

Toda degradación debe registrarse en manifest.

11. Validaciones obligatorias

11.1. Finitud numérica

    ϕ_j(x_i) ∈ R

para todas las atribuciones persistidas.

11.2. Cierre aditivo
Para métodos aditivos:

    | ŷ_i - (ϕ_0 + Σ_j ϕ_{ij}) | <= ε_closure

11.3. Cobertura mínima

    n_explained / n_eligible >= γ_min

11.4. Baseline convergence

    | E[ ŷ(X_ref) ] - ϕ_0 | <= δ_baseline

11.5. Faithfulness
Si está activado:

    deletion_auc >= τ_del

o, al menos, generar warning si cae por debajo del umbral.

11.6. Estabilidad temporal
Si está activada:

    Spearman(rank_t, rank_{t+1}) >= ρ_min

o warning/alerta si se incumple.

12. Edge cases críticos

El módulo debe contemplar explícitamente:

- features faltantes o reordenadas -> fail inmediato;
- predictions presentes pero features ausentes -> no explicación retroactiva incompleta;
- universo demasiado grande -> sampleo persistido, nunca truncado silencioso;
- todas las contribuciones casi nulas -> posible baseline trivial, score saturado o fallo metodológico;
- features altamente colineales -> warning fuerte, especialmente para lineales y PFI;
- ensemble no aditivo -> impedir lectura de suma exacta;
- degradación de método por budget -> no silenciosa.

13. Métricas adicionales recomendadas

Como extensiones razonables de primera o segunda fase:
- attribution variance vía bootstrap por bloques;
- outlier score de atribuciones, por ejemplo max|ϕ_j| / std(ϕ_j);
- VIF o diagnóstico de colinealidad entre features explicadas;
- top-k family churn entre ventanas;
- frecuencia con que una feature entra/sale del top-k global.

Estas métricas mejoran mucho el valor del módulo para monitoring real sin convertirlo en un framework de investigación XAI ornamental.

16. MVP y versión avanzada

16.1. MVP obligatorio

El MVP debe incluir:
- coef para lineales;
- TreeSHAP para GBDT cuando sea viable;
- PFI como fallback global;
- matrix explícita de resolución de método;
- baseline persistido;
- cierre aditivo;
- cobertura mínima;
- manifest completo;
- artefactos globales/locales estructurados.

16.2. Primera fase recomendada

Añadir ya:
- baseline convergence test;
- verificación explícita de axiomas aplicables;
- deletion AUC;
- rank-stability Spearman entre ventanas.

16.3. Segunda fase

Como extensiones razonables:
- insertion curves;
- attribution variance;
- VIF/colinealidad;
- análisis más rico por régimen;
- métricas de outlier attribution.

17. No-objetivos

No pertenece al alcance de este módulo:
- inferencia causal;
- contrafactuales avanzados;
- mechanistic interpretability;
- attribution patching;
- frameworks de explicación no trazables al run real;
- investigación metodológica profunda sobre XAI.

La misión aquí es explicabilidad operativa, reproducible y útil para control cuantitativo, no investigación ornamental.

18. Criterios de aceptación

El módulo sólo debe considerarse correcto si cumple simultáneamente:
- contrato exacto entre modelo, features y predicciones;
- método resuelto de forma explícita y compatible;
- baseline persistido y coherente;
- cierre aditivo dentro de tolerancia cuando aplique;
- cobertura mínima suficiente;
- degradación por presupuesto no silenciosa;
- artefactos persistidos con trazabilidad completa;
- semántica prudente y no causal en outputs y documentación.

19. Conclusión

models/inference/explain.py debe entenderse como la capa de instrumentación explicativa seria del sistema de inferencia. Su valor no está en “decorar” predicciones con gráficos o narrativas, sino en convertir scores opacos en artefactos auditables, comparables y útiles para debugging, monitoring y control de drift explicativo. En un laboratorio cuantitativo profesional, esa diferencia es decisiva: una explicación buena no es la que suena convincente, sino la que es reproducible, metodológicamente honesta y suficientemente fiel como para ayudar a detectar cuándo el modelo está haciendo algo razonable, algo frágil o algo directamente sospechoso.
