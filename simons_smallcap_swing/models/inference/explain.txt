\section{Módulo \texttt{models/inference/explain.py}}

\subsection{Objetivo}

El módulo \texttt{models.inference.explain.py} tiene como objetivo proporcionar una capa de
\textbf{explicabilidad operativa, reproducible y auditable} sobre las predicciones generadas por los
modelos registrados en el sistema. Su función no es “interpretar” el modelo en abstracto, sino
\textbf{producir artefactos de explicación consistentes con una ejecución concreta de inferencia},
de forma que una predicción individual o un conjunto de scores puedan ser analizados ex post con
suficiente precisión para:

\begin{itemize}
    \item auditar qué grupos de features dominaron el comportamiento del modelo;
    \item depurar fallos de señal, drift o sensibilidad excesiva a determinadas variables;
    \item validar que la estructura explicativa observada es coherente con la hipótesis de investigación;
    \item documentar el comportamiento del modelo para validación interna, monitoring y revisión cuantitativa.
\end{itemize}

El módulo soporta explicabilidad:

\begin{itemize}
    \item \textbf{global}: importancia agregada de features, estabilidad por grupos y jerarquías de señal;
    \item \textbf{local}: atribuciones por observación, fecha y símbolo;
    \item \textbf{por tipo de modelo}: coeficientes para modelos lineales, \texttt{TreeSHAP} para modelos de árboles y
    estrategias de agregación para ensamblados;
    \item \textbf{versionada por run}: toda explicación queda ligada a un \texttt{run\_id}, a un
    \texttt{model\_version}, a una versión de datos y a una configuración explícita.
\end{itemize}

En un sistema de \textit{small-cap swing trading}, esto es especialmente importante porque la explicación
de una señal no es solo una comodidad analítica: es una herramienta de control de modelo, de análisis
de robustez y de detección temprana de degradación estructural.

\subsection{Principio de diseño}

El módulo adopta un principio deliberadamente conservador:

\begin{quote}
\textit{Una explicación es un artefacto descriptivo del comportamiento del modelo sobre una distribución
de entrada y una ejecución concretas; no constituye, por sí sola, una inferencia causal ni una garantía
de validez económica.}
\end{quote}

Por tanto:

\begin{itemize}
    \item las explicaciones se interpretan como \textbf{descomposiciones del score del modelo};
    \item no se usa lenguaje causal fuerte salvo que exista un procedimiento externo que lo justifique;
    \item las explicaciones locales se consideran potencialmente ruidosas y se contextualizan siempre con
    explicaciones globales, estabilidad temporal y agrupaciones por familia de features.
\end{itemize}

\subsection{Contexto de XAI en Trading Cuantitativo}

En investigación cuantitativa industrial, la explicabilidad cumple una función distinta a la de entornos
puramente regulatorios o de producto. Aquí no se trata solo de justificar una predicción, sino de responder
preguntas operativas del tipo:

\begin{itemize}
    \item ¿está el modelo concentrando demasiada sensibilidad en una familia concreta de señales?
    \item ¿ha cambiado la jerarquía de importancias tras un retrain o una promoción a \texttt{staging}/\texttt{production}?
    \item ¿están las explicaciones locales extremas asociadas a outliers, errores de datos o comportamiento legítimo?
    \item ¿son las señales explicativamente estables entre regímenes de mercado o dependen de un subperiodo estrecho?
\end{itemize}

Por ello, el módulo se diseña para producir no solo importancias puntuales, sino también artefactos que
permitan estudiar \textbf{consistencia, estabilidad, cobertura y coste computacional}.

\subsection{Alcance Funcional}

\subsubsection{Incluye}

\begin{itemize}
    \item \textbf{Explicabilidad global}:
    \begin{itemize}
        \item importancia global por feature;
        \item ranking de features;
        \item agregación por grupos de features usando el \texttt{feature dictionary};
        \item comparación temporal opcional entre ventanas o runs.
    \end{itemize}

    \item \textbf{Explicabilidad local}:
    \begin{itemize}
        \item atribuciones por observación para un subconjunto o para la totalidad del universo explicado;
        \item ranking por contribución absoluta;
        \item top contributors positivos y negativos por predicción.
    \end{itemize}

    \item \textbf{Soporte por clase de modelo}:
    \begin{itemize}
        \item modelos lineales: descomposición determinista basada en coeficientes;
        \item GBDT: \texttt{TreeSHAP} como método preferente para explicaciones locales y globales;
        \item importancia por permutación como método complementario o \textit{fallback} global;
        \item ensembles: explicación por componente y agregación conforme a la regla de combinación del ensemble.
    \end{itemize}

    \item \textbf{Persistencia estructurada}:
    \begin{itemize}
        \item artefactos tabulares en Parquet;
        \item metadatos de ejecución en JSON;
        \item manifiesto con hashes, configuración, seeds, librerías y validaciones.
    \end{itemize}
\end{itemize}

\subsubsection{No incluye}

\begin{itemize}
    \item entrenamiento o recalibración del modelo;
    \item selección de cartera o sizing;
    \item inferencia causal, \textit{counterfactual reasoning} o análisis estructural fuera de la clase descriptiva;
    \item explicación de modelos no registrados o ejecuciones no trazables.
\end{itemize}

\subsection{Dependencias}

\subsubsection{Upstream}

\begin{itemize}
    \item \textbf{Model registry}: resolución determinista de \texttt{model\_ref} a una versión concreta;
    \item \textbf{Artefactos de inferencia}: features efectivamente usadas en scoring y predicciones persistidas;
    \item \textbf{Training contract}: lista de features de entrenamiento, transformaciones y metadatos de compatibilidad;
    \item \textbf{Feature dictionary}: taxonomía de features, grupos, horizontes, unidades y familia económica/estadística.
\end{itemize}

\subsubsection{Downstream}

\begin{itemize}
    \item validación cuantitativa de señales;
    \item monitoring de drift explicativo;
    \item debugging de predicciones extremas;
    \item evidencia de research y documentación de comportamiento de modelo.
\end{itemize}

\subsection{Invariantes Operativos}

El módulo debe satisfacer las siguientes propiedades:

\begin{itemize}
    \item \textbf{Determinismo condicionado}: dado el mismo \texttt{run\_id}, la misma versión de modelo,
    la misma muestra y la misma configuración, los artefactos explicativos deben ser reproducibles salvo
    diferencias numéricas acotadas por tolerancia.
    \item \textbf{Trazabilidad completa}: toda explicación debe poder enlazarse con
    \texttt{model\_version}, \texttt{inference\_run\_id}, \texttt{data\_hash}, \texttt{code\_commit} y
    \texttt{explain\_config\_hash}.
    \item \textbf{Separación entre explicación y scoring}: el módulo no altera scores ya persistidos;
    solo los explica.
    \item \textbf{Fail-closed en inconsistencia}: si el contrato entre features, predicciones y modelo no
    se cumple, la ejecución falla; no se generan explicaciones parciales silenciosas.
\end{itemize}

\subsection{Contrato de Entrada}

\subsubsection{Entradas mínimas}

\begin{itemize}
    \item \texttt{model\_ref}: referencia lógica o versión exacta del modelo;
    \item \texttt{features\_path}: conjunto de features efectivamente usado por la inferencia;
    \item \texttt{predictions\_path}: salida persistida del scoring;
    \item \texttt{explain\_config\_path}: configuración de explicabilidad;
    \item \texttt{run\_id}: identificador único de esta ejecución de explicación;
    \item \texttt{inference\_run\_id}: identificador del batch de inferencia que se desea explicar.
\end{itemize}

\subsubsection{Parámetros clave}

\begin{itemize}
    \item \texttt{explain\_mode}: \texttt{global}, \texttt{local}, \texttt{both};
    \item \texttt{method\_policy}: \texttt{auto}, \texttt{force\_shap}, \texttt{force\_permutation}, \texttt{force\_coef};
    \item \texttt{sample\_size\_local}: tamaño máximo para explicaciones locales;
    \item \texttt{top\_k\_features}: número de features destacadas en resúmenes;
    \item \texttt{background\_sample\_size}: tamaño del conjunto de referencia para métodos que lo requieran;
    \item \texttt{max\_compute\_budget\_sec}: presupuesto temporal máximo permitido;
    \item \texttt{group\_by\_feature\_family}: si se agregan importancias por grupos taxonómicos;
    \item \texttt{stability\_window\_spec}: ventanas opcionales para medir estabilidad temporal.
\end{itemize}

\subsection{Resolución Formal del Método}

Uno de los puntos críticos del módulo es que la selección del método de explicación no quede a criterio
implícito del implementador. La política es:

\begin{enumerate}
    \item Si el modelo es \textbf{lineal} y la transformación entre entrada persistida y score final es lineal
    y conocida, el método por defecto es \texttt{coef}.
    \item Si el modelo es \textbf{GBDT} compatible con \texttt{TreeSHAP} y el coste estimado cabe dentro del
    presupuesto computacional, el método por defecto es \texttt{shap}.
    \item Si el modelo es GBDT pero el coste de \texttt{TreeSHAP} excede el presupuesto o existe incompatibilidad
    de librerías/serialización, se usa \texttt{permutation} para importancia global y se desactiva la explicación
    local salvo override explícito.
    \item Si el modelo es \textbf{ensemble}, se intenta explicar cada componente con su método nativo y luego
    agregar; si eso no es posible, se produce únicamente explicación global sobre el ensemble final con el método
    compatible más conservador.
    \item Si el usuario fuerza un método incompatible con la clase de modelo, la ejecución falla.
\end{enumerate}

Esta política elimina ambigüedad y evita dos errores comunes: explicar con una técnica conceptualmente inadecuada
o activar un método demasiado costoso sin control.

\subsection{Baseline y Conjunto de Referencia}

Para métodos basados en contribuciones aditivas, la noción de baseline debe quedar fijada. El módulo define:

\begin{itemize}
    \item \textbf{Baseline local}:
    \[
    \hat{y}(x) \approx \phi_0 + \sum_{j=1}^{p} \phi_j(x),
    \]
    donde \(\phi_0\) representa el valor esperado del modelo bajo un conjunto de referencia reproducible.
    \item \textbf{Conjunto de referencia por defecto}: muestra estratificada y persistida del universo de
    entrenamiento válido o, en su defecto, del universo de inferencia del run, siempre que esa elección quede
    registrada en \texttt{manifest.json}.
\end{itemize}

El baseline nunca se deja implícito. Debe almacenarse:

\begin{itemize}
    \item tipo de baseline;
    \item tamaño de muestra;
    \item criterio de muestreo;
    \item hash o identificadores de filas del conjunto de referencia.
\end{itemize}

\subsection{Contrato de Salida}

Los artefactos se persisten en:

\begin{center}
    \texttt{models/explain/\{run\_id\}/}
\end{center}

\subsubsection{Ficheros generados}

\begin{itemize}
    \item \texttt{global\_importance.parquet}
    \begin{itemize}
        \item importancia global por feature;
        \item ranking;
        \item familia/grupo de feature;
        \item signo neto opcional y métricas de estabilidad.
    \end{itemize}

    \item \texttt{local\_explanations.parquet}
    \begin{itemize}
        \item atribuciones por observación y feature;
        \item baseline asociado;
        \item ranking por contribución absoluta;
        \item flags de calidad numérica.
    \end{itemize}

    \item \texttt{explain\_summary.json}
    \begin{itemize}
        \item cobertura de observaciones explicadas;
        \item método resuelto;
        \item coste computacional;
        \item tests de consistencia;
        \item top features globales;
        \item observaciones con mayor score absoluto y mayor inestabilidad explicativa.
    \end{itemize}

    \item \texttt{stability\_report.parquet} (opcional)
    \begin{itemize}
        \item estabilidad temporal de importancias por ventana;
        \item correlaciones rank-rank entre periodos;
        \item dispersión de contribuciones.
    \end{itemize}

    \item \texttt{manifest.json}
    \begin{itemize}
        \item metadatos completos de run, hashes, versiones y validaciones.
    \end{itemize}
\end{itemize}

\subsubsection{Campos mínimos de \texttt{local\_explanations}}

\begin{itemize}
    \item \texttt{run\_id}
    \item \texttt{inference\_run\_id}
    \item \texttt{date}
    \item \texttt{symbol}
    \item \texttt{model\_version}
    \item \texttt{method}
    \item \texttt{baseline\_value}
    \item \texttt{y\_pred}
    \item \texttt{feature\_name}
    \item \texttt{feature\_group}
    \item \texttt{feature\_value}
    \item \texttt{contribution}
    \item \texttt{abs\_contribution}
    \item \texttt{rank\_abs\_contribution}
    \item \texttt{is\_finite}
\end{itemize}

\subsection{Metodologías Implementadas}

\subsubsection{Modelos lineales}

Para modelos lineales de la forma

\[
\hat{y}(x) = \beta_0 + \sum_{j=1}^{p}\beta_j x_j,
\]

la explicación local se obtiene mediante:

\[
\phi_j(x)=\beta_j x_j,
\]

siempre que los \(x_j\) persistidos correspondan exactamente a la representación usada por el modelo.
Si el pipeline incluye escalado, normalización o transformaciones no triviales, el módulo debe explicar
en el espacio realmente consumido por el predictor, no en una reconstrucción aproximada posterior.

Para importancia global, se reportan al menos dos medidas:

\[
I^{(\text{coef})}_j = |\beta_j|,
\qquad
I^{(\text{contrib})}_j = \frac{1}{n}\sum_{i=1}^{n}|\beta_j x_{ij}|.
\]

La primera mide sensibilidad estructural del modelo; la segunda refleja importancia efectiva sobre la muestra explicada.

\subsubsection{GBDT: \texttt{TreeSHAP} como método principal}

Para modelos de árboles, el método principal es \texttt{TreeSHAP}. El módulo utiliza:

\begin{itemize}
    \item explicaciones locales aditivas por observación;
    \item importancia global como media de contribuciones absolutas:
    \[
    \bar{\phi}_j = \frac{1}{n}\sum_{i=1}^{n}|\phi_j(x_i)|.
    \]
\end{itemize}

El uso de \texttt{TreeSHAP} requiere:

\begin{itemize}
    \item compatibilidad confirmada entre la clase del modelo y la librería de explicación;
    \item baseline explícito;
    \item tolerancia máxima permitida en el cierre:
    \[
    \left| \hat{y}(x_i) - \left(\phi_0 + \sum_{j=1}^{p}\phi_j(x_i)\right) \right| \le \epsilon.
    \]
\end{itemize}

Si el cierre falla sistemáticamente fuera de tolerancia, la ejecución se marca como inválida.

\subsubsection{Permutation Importance como método global complementario}

La importancia por permutación se usa para contrastar o sustituir importancias globales cuando:

\begin{itemize}
    \item \texttt{TreeSHAP} no es viable;
    \item se desea una medida directamente ligada a degradación de pérdida out-of-sample;
    \item el interés principal es ranking global, no explicación local.
\end{itemize}

Se define:

\[
\mathrm{PFI}_j = \mathbb{E}[L(\mathbf{y},\hat{\mathbf{y}}_{j^{\mathrm{perm}}})]
-
\mathbb{E}[L(\mathbf{y},\hat{\mathbf{y}})].
\]

Aquí el signo queda fijado de modo que valores altos impliquen mayor pérdida al permutar y, por tanto,
mayor importancia. El módulo debe registrar:

\begin{itemize}
    \item métrica \(L\) usada;
    \item número de repeticiones;
    \item universo de evaluación;
    \item semillas por repetición.
\end{itemize}

Dado que la permutación rompe estructuras de dependencia entre variables, sus resultados deben leerse como
\textbf{medida global de sensibilidad predictiva}, no como atribución local exacta.

\subsubsection{Ensembles}

Para ensembles, el módulo distingue dos casos:

\paragraph{(a) Agregación lineal explícita entre modelos base}
Si el ensemble produce

\[
\hat{y}(x)=\sum_{b=1}^{B} w_b \hat{y}^{(b)}(x),
\]

con pesos \(w_b\) conocidos y constantes, entonces se agregan explicaciones base como

\[
\phi_j(x)=\sum_{b=1}^{B} w_b \phi_j^{(b)}(x).
\]

\paragraph{(b) Meta-modelos o combinaciones no lineales}
Si la capa final no es lineal o no existe correspondencia aditiva fiable, el módulo:

\begin{itemize}
    \item explica cada componente por separado, cuando sea posible;
    \item produce una explicación global del ensemble final con el método compatible;
    \item marca explícitamente que la agregación entre componentes y salida final no admite una lectura
    aditiva exacta.
\end{itemize}

\subsection{Control de Coste Computacional}

La explicabilidad puede ser prohibitivamente costosa en universos amplios. Por eso el módulo implementa
reglas explícitas de presupuesto:

\begin{itemize}
    \item \texttt{sample\_size\_local} limita el número de observaciones locales explicadas;
    \item se permite muestreo estratificado por score, fecha, símbolo o cuantiles;
    \item si el coste estimado excede \texttt{max\_compute\_budget\_sec}, el módulo degrada de
    \texttt{shap} a \texttt{permutation} para global o falla según configuración;
    \item el tiempo total, tiempo por bloque y memoria aproximada se registran en el \texttt{manifest}.
\end{itemize}

La política por defecto es priorizar:

\begin{enumerate}
    \item cobertura suficiente para debugging;
    \item consistencia metodológica;
    \item respeto al presupuesto;
    \item ausencia de degradaciones silenciosas.
\end{enumerate}

\subsection{Reglas de Reproducibilidad}

La reproducibilidad no se limita a fijar una semilla. El módulo debe almacenar:

\begin{itemize}
    \item \texttt{seed\_global} y, en su caso, seeds por bloque;
    \item versión exacta de librerías (\texttt{shap}, \texttt{numpy}, \texttt{sklearn}, etc.);
    \item \texttt{model\_version}, \texttt{data\_hash}, \texttt{code\_commit};
    \item índices o claves primarias de la muestra explicada;
    \item baseline y conjunto de referencia;
    \item configuración resuelta tras defaults.
\end{itemize}

No se recomienda usar una semilla derivada de \texttt{timestamp} si eso dificulta la repetibilidad exacta.
Por defecto, la seed debe venir de configuración o derivarse determinísticamente de
\texttt{run\_id} mediante una función estable.

\subsection{Flujo de Ejecución}

\begin{enumerate}
    \item \textbf{Resolver modelo y verificar contrato}
    \begin{itemize}
        \item cargar la versión exacta del modelo desde el registry;
        \item verificar compatibilidad entre features persistidas, schema y predicciones.
    \end{itemize}

    \item \textbf{Resolver método}
    \begin{itemize}
        \item clasificar el tipo de modelo;
        \item estimar coste;
        \item aplicar \texttt{method\_policy}.
    \end{itemize}

    \item \textbf{Construir muestra y baseline}
    \begin{itemize}
        \item fijar subconjunto local si aplica;
        \item construir o recuperar el conjunto de referencia;
        \item persistir selección.
    \end{itemize}

    \item \textbf{Calcular explicaciones globales}
    \begin{itemize}
        \item SHAP, coeficientes o permutación;
        \item agregar por feature y, opcionalmente, por grupo.
    \end{itemize}

    \item \textbf{Calcular explicaciones locales}
    \begin{itemize}
        \item ejecutar solo si el método y el presupuesto lo permiten;
        \item persistir atribuciones por observación.
    \end{itemize}

    \item \textbf{Ejecutar validaciones}
    \begin{itemize}
        \item finitud numérica;
        \item cierre aditivo cuando corresponda;
        \item cobertura mínima;
        \item integridad de schemas y cardinalidades.
    \end{itemize}

    \item \textbf{Persistir artefactos}
    \begin{itemize}
        \item escritura atómica de tablas y manifiesto;
        \item marcar run como válido solo si todas las validaciones críticas pasan.
    \end{itemize}
\end{enumerate}

\subsection{Validaciones Implementadas}

\begin{itemize}
    \item \textbf{Finitud numérica}:
    \[
    \phi_j(x_i)\in\mathbb{R}, \quad \forall i,j.
    \]

    \item \textbf{Cierre aditivo}:
    para métodos aditivos válidos,
    \[
    \left| y_{\mathrm{pred},i} - \left(\phi_0+\sum_{j=1}^{p}\phi_j(x_i)\right)\right|\le\epsilon.
    \]

    \item \textbf{Cobertura mínima}:
    \[
    \frac{n_{\mathrm{explained}}}{n_{\mathrm{eligible}}}\ge \gamma.
    \]

    \item \textbf{Consistencia de ranking}:
    ausencia de empates patológicos o artefactos de ordenación por overflow/redondeo.

    \item \textbf{Integridad entre artefactos}:
    las claves \texttt{date-symbol-model\_version-run\_id} deben reconciliar entre
    \texttt{predictions}, \texttt{local\_explanations} y \texttt{manifest}.

    \item \textbf{Estabilidad temporal} (opcional pero recomendada):
    correlación de Spearman/Kendall entre rankings globales en ventanas consecutivas,
    con alertas si la estructura explicativa rota bruscamente.
\end{itemize}

\subsection{Tratamiento de Edge Cases}

Para llevar el módulo a estándar de producción, deben contemplarse explícitamente los siguientes casos:

\begin{itemize}
    \item \textbf{Features faltantes o reordenadas}: fallo inmediato si el conjunto explicado no coincide con
    el contrato del modelo.
    \item \textbf{Predicciones presentes pero features ausentes}: no se permite explicación retroactiva incompleta.
    \item \textbf{Universo demasiado grande}: explicación local parcial con muestreo persistido; nunca truncado silencioso.
    \item \textbf{Contribuciones todas casi nulas}: marcar posible degeneración del método o saturación del score.
    \item \textbf{Features altamente colineales}: advertencia en el resumen, especialmente para lineales y PFI.
    \item \textbf{Ensemble no aditivo}: impedir una lectura de suma exacta si no está matemáticamente justificada.
\end{itemize}

\subsection{Interpretación Recomendada}

Las salidas del módulo deben leerse con la siguiente jerarquía:

\begin{enumerate}
    \item primero, importancias globales y su estabilidad;
    \item después, agrupación por familias de features;
    \item por último, explicaciones locales individuales, especialmente en casos extremos o anómalos.
\end{enumerate}

La política recomendada es no tomar decisiones de investigación o de producción basadas en una única explicación local
aislada. Una explicación local solo adquiere significado fuerte cuando es consistente con patrones agregados, estabilidad
temporal y evidencia de validación externa.

\subsection{Resumen}

En conjunto, \texttt{models/inference/explain.py} no debe entenderse como un módulo cosmético de visualización,
sino como una capa de \textbf{instrumentación científica del modelo en producción}. Su valor real reside en que:

\begin{itemize}
    \item convierte predicciones opacas en artefactos auditables;
    \item obliga a fijar baseline, método, muestra y presupuesto computacional;
    \item reduce ambigüedad de implementación mediante políticas explícitas;
    \item facilita debugging, monitoring y validación cuantitativa;
    \item mantiene una postura metodológicamente prudente: explicación descriptiva fuerte, causalidad no inferida.
\end{itemize}

Este enfoque es el adecuado para una plataforma cuantitativa seria: no promete más interpretabilidad de la que el método
puede ofrecer, pero sí exige que toda explicación generada sea reproducible, trazable y operativamente útil.
