\section{Módulo \texttt{models/inference/model\_registry.py}}

\subsection{Objetivo}
El módulo \texttt{models/inference/model\_registry.py} tiene como objetivo gestionar un \textbf{registro central de modelos} que almacene:
\begin{itemize}
    \item \textbf{Metadatos}: \texttt{model\_id}, \texttt{task\_type}, \texttt{algo\_family}, \texttt{stage}, etc.
    \item \textbf{Artefactos versionados}: rutas \texttt{artifact\_uri} inmutables.
    \item \textbf{Versiones y etapas}: \texttt{candidate}, \texttt{staging}, \texttt{production}, \texttt{archived}.
    \item \textbf{Historial de promociones y rollbacks}: trazabilidad de decisiones de despliegue.
\end{itemize}

Este sistema garantiza:
\begin{itemize}
    \item \textbf{Reproducibilidad plena}: trazabilidad de \texttt{model\_version} \(\leftrightarrow\) \texttt{training\_run\_id} \(\leftrightarrow\) \texttt{data\_hash} \(\leftrightarrow\) \texttt{code\_commit}.
    \item \textbf{Gobernanza de riesgo}: sólo modelos con validación aprobada pueden ascender a producción, con políticas de \textit{override} auditado.
    \item \textbf{Despliegue seguro}: resolución determinista de la versión activa para inferencia (\texttt{resolve\_model}).
\end{itemize}

Este diseño se alinea con estándares industriales de \textbf{model registry} como \texttt{MLflow Model Registry} \cite{mlflow2020modelregistry}, \texttt{Vertex AI Model Registry} \cite{google2025modelregistry} y guías de MLOps para entornos de trading cuantitativo \cite{mlinproduction2020registries}.

\subsection{Contexto de Model Registry en Trading Cuantitativo}
En estrategias de \textit{swing trading} de alta rotación, la gestión de modelos debe ser tan rigurosa como la gestión de carteras. El registro de modelos permite:
\begin{itemize}
    \item \textbf{Separación clara} entre entrenamiento (\texttt{training}) e inferencia (\texttt{predict}).
    \item \textbf{Iteración controlada} de versiones, evitando despliegues caóticos.
    \item \textbf{Auditoría regulatoria}: qué modelo, qué dataset, qué código, en qué fecha.
\end{itemize}

El uso de \textit{append-only} de eventos de promoción/rollback, junto con alias inmutables (\texttt{production}, \texttt{staging}), es un patrón recomendado en guías de MLOps \cite{modelregistry2025bestpractices}.

\subsection{Alcance Funcional}

\subsubsection{Incluye}
\begin{itemize}
    \item \textbf{Alta de nuevas versiones}:
    \begin{itemize}
        \item Registro de \texttt{model\_version} con metadatos, métricas y hashes de datos/features.
    \end{itemize}
    \item \textbf{Consulta por alias/etapa}:
    \begin{itemize}
        \item \texttt{latest\_candidate}, \texttt{staging}, \texttt{production}, \texttt{archived}.
    \end{itemize}
    \item \textbf{Resolución de modelo activo}:
    \begin{itemize}
        \item Operación \texttt{resolve\_model(ref)} que devuelve la versión actualmente activa para inferencia.
    \end{itemize}
    \item \textbf{Versionado de artefactos y dependencias}:
    \begin{itemize}
        \item \texttt{feature\_set\_hash}, \texttt{config\_hash}, \texttt{code\_commit}.
    \end{itemize}
    \item \textbf{Trazabilidad de decisiones}:
    \begin{itemize}
        \item Historial de promociones y rollbacks, con motivo y aprobador.
    \end{itemize}
\end{itemize}

\subsubsection{No incluye}
\begin{itemize}
    \item \textbf{Entrenamiento de modelos}: el módulo asume completitud de \texttt{training\_run} y \texttt{validation\_suite}.
    \item \textbf{Almacenamiento binario pesado}: solo gestiona \texttt{paths} hacia artefactos externos (S3, GCS, etc.).
\end{itemize}

\subsection{Dependencias}

\subsubsection{Upstream}
\begin{itemize}
    \item \textbf{Outputs de \texttt{training}}:
    \begin{itemize}
        \item Métricas, artefactos, manifests generados por el pipeline de entrenamiento.
    \end{itemize}
    \item \textbf{Decisiones de \texttt{validation\_suite}}:
    \begin{itemize}
        \item \texttt{validation\_decision} (pass/fail/override) que determina si una versión puede ascender a \texttt{production}.
    \end{itemize}
\end{itemize}

\subsubsection{Downstream}
\begin{itemize}
    \item \texttt{models/inference/predict}: consulta el modelo resuelto para realizar inferencia en tiempo real.
    \item \texttt{backtest/run\_walkforward}: selecciona versiones de modelo para backtesting y walk‑forward testing.
    \item \texttt{ops/monitoring}: obtiene el modelo activo para monitorear su comportamiento en producción.
\end{itemize}

\subsection{Entidades Clave}

\subsubsection{Model}
\begin{itemize}
    \item \texttt{model\_id} (estable, único a nivel de problema).
    \item \texttt{task\_type} (e.g.\ \texttt{regression\_returns}, \texttt{binary\_signal}).
    \item \texttt{algo\_family} (e.g.\ \texttt{gbdt}, \texttt{nn}, \texttt{ensemble}).
\end{itemize}

\subsubsection{Version}
Para cada versión de modelo \((m, v)\) se almacena:
\begin{itemize}
    \item \texttt{model\_version} (semver \texttt{v1.2.3} o timestamp \texttt{2026.03.05.1}).
    \item \texttt{artifact\_uri}: ruta inmutable al artefacto serializado (Weights, \texttt{joblib}, \texttt{pmml}, etc.).
    \item \texttt{training\_run\_id}: referencia al \texttt{run} de entrenamiento en el sistema de experimentación.
    \item \texttt{training\_data\_hash}: huella del dataset de entrenamiento (SHA‑256 o hash de partición).
    \item \texttt{feature\_set\_hash}: hash del conjunto de features utilizadas.
    \item \texttt{config\_hash}: hash de la configuración de entrenamiento (\texttt{hyperparams}).
    \item \texttt{code\_commit}: SHA del commit de código fuente.
    \item \texttt{metrics\_snapshot}: métricas principales (e.g.\ \texttt{sharpe}, \texttt{accuracy}, \texttt{AUC}).
    \item \texttt{validation\_decision}: \texttt{pass}, \texttt{fail}, \texttt{override}.
    \item \texttt{stage}: \texttt{candidate}, \texttt{staging}, \texttt{production}, \texttt{archived}.
\end{itemize}

Estos campos siguen el patrón de metadatos recomendado por MLflow \cite{mlflow2020modelregistry} y guías de MLOps \cite{modelregistry2025bestpractices}.

\subsubsection{Alias}
\begin{itemize}
    \item \texttt{latest\_candidate}: versión más reciente aprobada por \texttt{validation\_suite}.
    \item \texttt{staging}: versión en prueba para entornos de \texttt{staging}.
    \item \texttt{production}: versión activa en producción.
\end{itemize}

\subsection{Contrato de Entrada}

\subsubsection{Operaciones mínimas}
\begin{itemize}
    \item \texttt{register\_model\_version(payload)}: registra una nueva versión candidata.
    \item \texttt{promote\_model(model\_id, version, stage)}: promueve una versión a nueva etapa.
    \item \texttt{rollback\_model(model\_id, to\_version)}: revierte el alias \texttt{production} a una versión previa.
    \item \texttt{resolve\_model(ref)}: resuelve el modelo activo dado un \texttt{ref} (alias, stage, versión).
    \item \texttt{list\_models(filters)}: consulta el catálogo de modelos.
\end{itemize}

\subsubsection{Payload mínimo de registro}
\begin{itemize}
    \item \texttt{model\_id}
    \item \texttt{model\_version}
    \item \texttt{artifact\_uri}
    \item \texttt{training\_run\_id}
    \item \texttt{metrics\_summary} (representación serializada de métricas clave).
    \item \texttt{validation\_decision}
    \item \texttt{created\_ts}
\end{itemize}

Estos campos forman el núcleo de trazabilidad necesaria para un \textit{model card} minimalista \cite{modelcards2019}.

\subsection{Contrato de Salida}

\subsubsection{Persistencia}
\begin{itemize}
    \item \texttt{models/registry/registry.parquet} (o base de datos ligera \texttt{sqlite}/\texttt{jsonl}):
    \begin{itemize}
        \item Tabla principal de \texttt{model\_version} con todos sus metadatos.
    \end{itemize}
    \item \texttt{models/registry/history/\{date\}/events.parquet}:
    \begin{itemize}
        \item Log de eventos \textit{append-only} (registrar, promover, rollback).
    \end{itemize}
    \item \texttt{models/registry/model\_cards/\{model\_id\}\_\{version\}.json}:
    \begin{itemize}
        \item \textit{Model Card} por versión con métricas y metadatos de uso.
    \end{itemize}
\end{itemize}

\subsubsection{Respuestas API internas}
\begin{itemize}
    \item \texttt{resolved\_model\_metadata}: conjunto completo de metadatos de la versión resuelta.
    \item \texttt{promotion\_history}: historial de transiciones de etapa.
    \item \texttt{current\_status\_by\_alias}: estado actual de cada alias.
\end{itemize}

\subsection{Reglas de Gobernanza}

\begin{itemize}
    \item \textbf{Sólo versiones con \texttt{validation\_suite=pass} pueden ir a producción} (salvo override auditado explicitamente registrado).
    \item \textbf{Promotion y rollback son eventos auditables}: cada cambio de \texttt{stage} genera un evento persistido.
    \item \textbf{No se reescribe historia}: el registro de eventos es \textit{append-only}; se mantiene la trazabilidad completa.
    \item \textbf{Cada versión apunta a artefacto inmutable}: el \texttt{artifact\_uri} no debe ser modificado después del registro.
\end{itemize}

Estas reglas responden a recomendaciones de gobernanza de ML en entornos de trading de alta frecuencia \cite{mlops2023trading}.

\subsection{Flujo de Registro y Promoción}

\begin{enumerate}
    \item \textbf{Registrar versión candidata} tras el entrenamiento:
    \begin{itemize}
        \item \texttt{register\_model\_version} con métricas y hashes.
    \end{itemize}
    \item \textbf{Adjuntar métricas y hashes de reproducibilidad}:
    \begin{itemize}
        \item \texttt{training\_data\_hash}, \texttt{feature\_set\_hash}, \texttt{config\_hash}, \texttt{code\_commit}.
    \end{itemize}
    \item \textbf{Evaluar \texttt{validation\_decision}}:
    \begin{itemize}
        \item \texttt{pass} \(\rightarrow\) candidatura a \texttt{staging}/\texttt{production}.
        \item \texttt{fail} \(\rightarrow\) versión bloqueada a salvo de overrides.
    \end{itemize}
    \item \textbf{Promover a etapa según política}:
    \begin{itemize}
        \item \texttt{promote\_model(model\_id, version, stage)}.
    \end{itemize}
    \item \textbf{Registrar evento y actualizar alias}:
    \begin{itemize}
        \item Actualización atómica de \texttt{production}/\texttt{staging} y escritura de evento.
    \end{itemize}
\end{enumerate}

Este flujo sigue el patrón de \textit{lifecycle stages} descrito en \cite{mlflow2020modelregistry} y \cite{modelregistry2025bestpractices}.

\subsection{Validaciones Mínimas}

\begin{itemize}
    \item \textbf{Unicidad} \texttt{(model\_id, model\_version)}:
    \begin{itemize}
        \item Evitar versiones duplicadas para el mismo modelo.
    \end{itemize}
    \item \textbf{Accesibilidad de \texttt{artifact\_uri}}:
    \begin{itemize}
        \item Verificar que el artefacto existe y es legible (p.ej.\ \texttt{HEAD} HTTP o similar).
    \end{itemize}
    \item \textbf{Campos obligatorios no nulos}.
    \item \textbf{Stage válido} (perteneciente al conjunto predefinido).
    \item \textbf{Coherencia entre \texttt{validation\_decision} y \texttt{stage} target}:
    \begin{itemize}
        \item No se permite promover a \texttt{production} una versión con \texttt{validation\_decision=fail} salvo \texttt{override} auditado.
    \end{itemize}
\end{itemize}

\subsection{Manejo de Rollback}

\begin{itemize}
    \item \textbf{Rollback rápido por alias}:
    \begin{itemize}
        \item Operación \texttt{rollback\_model(model\_id, to\_version)} que actualiza el alias \texttt{production} a una versión anterior estable.
    \end{itemize}
    \item \textbf{Snapshot de estado de alias}:
    \begin{itemize}
        \item Guardar el estado de los alias antes y después del rollback para análisis forense.
    \end{itemize}
    \item \textbf{Registro de motivo y aprobador}:
    \begin{itemize}
        \item Persistir metadatos de operación (¿por qué se retrocedió?, ¿quién lo aprobó?).
    \end{itemize}
\end{itemize}

Esta estrategia de rollback es recomendada en guías de MLOps para entornos de alta frecuencia \cite{modelregistry2025rollback}.

\subsection{Observabilidad}

\subsubsection{Logging}
\begin{itemize}
    \item \texttt{INFO}: registros de alta de versiones, promociones exitosas.
    \item \texttt{WARNING}: promociones forzadas con \texttt{override} o rollback no planificado.
    \item \texttt{ERROR}: conflictos de versión, artefacto inexistente, fallo de \texttt{resolve\_model}.
\end{itemize}

\subsubsection{Métricas}
\begin{itemize}
    \item \texttt{n\_models\_total}: número total de modelos registrados.
    \item \texttt{n\_versions\_total}: número total de versiones registradas.
    \item \texttt{n\_promotions\_prod}: cuántas promociones a \texttt{production} se han realizado.
    \item \texttt{n\_rollbacks}: número de rollback a versiones anteriores.
    \item \texttt{avg\_time\_candidate\_to\_prod}: tiempo medio desde \texttt{latest\_candidate} hasta \texttt{production}.
\end{itemize}

\subsection{CLI Sugerida}

\begin{verbatim}
python -m models.inference.model_registry register \
  --model-id lgbm_main \
  --version 2026.03.05.1 \
  --artifact-uri models/artifacts/lgb
