Especificación Cuantitativa: models/inference/model_registry.py

1. Propósito del módulo

El objetivo de model_registry.py no es guardar una lista de modelos entrenados ni actuar como catálogo descriptivo de artefactos. Su función es mucho más crítica: definir la fuente única de verdad operacional sobre el ciclo de vida de los modelos del sistema.

La pregunta correcta del módulo es:

“¿Qué modelo exacto está autorizado para servir inferencia en este instante, bajo qué decisión formal, con qué artefacto íntegro, con qué trazabilidad completa y con qué historial auditable de promociones, overrides y rollbacks?”

En un stack cuantitativo serio, esta pregunta no es administrativa. Es control de riesgo de modelo. Si el sistema no puede responder de forma determinista y auditable qué versión exacta está activa, toda predicción producida downstream queda epistemológicamente debilitada.

Por eso este módulo no debe entenderse como un inventario de versiones, sino como infraestructura transaccional de gobernanza. Su misión es eliminar ambigüedad operacional, garantizar atomicidad de promociones, imponer invariantes de integridad y preservar la historia completa del lifecycle de cada artefacto predictivo.

2. Alcance y no-objetivos

2.1 Alcance

El módulo:
- registra nuevas versiones de modelo con metadatos completos;
- persiste referencias a artefactos, hashes, métricas y lineage operativo;
- gobierna estados como candidate, staging, production y archived;
- resuelve alias lógicos como production o staging a una versión única;
- ejecuta promociones y rollbacks auditables;
- mantiene historial append-only de eventos;
- valida integridad mínima de artefactos antes de resolver o promocionar;
- y expone una interfaz determinista para inferencia, auditoría y monitoring.

2.2 Fuera de alcance

El módulo no:
- entrena modelos;
- no almacena necesariamente el binario pesado del artefacto;
- no decide por sí solo qué modelo “es mejor” desde el punto de vista cuantitativo;
- no ejecuta inferencia ni scoring;
- no sustituye a validation_suite ni a la capa de promotion decision;
- no debe convertirse en un data catalog genérico;
- ni debe promocionar una versión solo porque exista o porque sea la última registrada.

3. Objeto formal del módulo

El módulo gobierna cuatro entidades lógicas principales:

- Model
- Version
- Alias
- Event

3.1 Model

Representa una familia lógica estable de predictor, identificada por model_id y metadatos estructurales como task_type, algo_family y status lógico.

3.2 Version

Cada versión (model_id, version) representa un artefacto entrenado e inmutable, asociado a:
- artifact_uri
- artifact_hash
- training_run_id
- training_data_hash
- feature_hash o feature_set_hash
- config_hash
- code_commit
- metrics_snapshot
- validation_decision
- stage
- created_ts
- registered_by

3.3 Alias

Un alias no es un modelo, sino una referencia lógica mutable y auditable hacia una versión concreta. Ejemplos:
- latest_candidate
- staging
- production

3.4 Event

Toda mutación relevante del lifecycle se representa como evento append-only, por ejemplo:
- REGISTER_VERSION
- PROMOTE_TO_STAGING
- PROMOTE_TO_PRODUCTION
- ROLLBACK_PRODUCTION
- ARCHIVE_VERSION
- OVERRIDE_PROMOTION

4. Principio central: gobernanza transaccional, no catálogo

La regla estructural del módulo es:

el registry no describe modelos; gobierna el derecho operacional de una versión a existir, resolverse y servir.

Eso implica que:
- no puede haber resolución ambigua;
- no puede haber promoción parcial;
- no puede reescribirse la historia;
- no puede asumirse que “latest” equivale a “eligible”;
- y no puede servirse un artefacto cuyo estado o integridad sea incierto.

La ausencia de ambigüedad es aquí más importante que la comodidad. El módulo debe fallar de forma conservadora antes que resolver “algo razonable”.

5. Invariantes fundamentales

El módulo debe imponer, al menos, los siguientes invariantes:

5.1 Unicidad de versión

No pueden existir dos registros con la misma clave:

(model_id, version)

5.2 Inmutabilidad de artefacto

Una vez registrada una versión, no pueden mutarse:
- artifact_uri
- artifact_hash
- training_run_id
- training_data_hash
- feature_hash
- config_hash
- code_commit

5.3 Unicidad de alias activo

Para cada model_id y cada alias crítico, especialmente staging y production, debe existir como máximo una versión activa.

5.4 Consistencia entre validación y promoción

Ninguna versión con validation_decision=fail puede ocupar production salvo override explícito, auditado y trazable.

5.5 Historial no reescribible

Los eventos son append-only. El estado actual puede derivarse de ellos, pero los eventos pasados no deben editarse ni borrarse.

5.6 Resolución determinista

Toda llamada a resolve(ref) debe devolver exactamente una versión o fallar de forma explícita. Nunca debe devolver múltiples candidatas ni elegir heurísticamente sin política registrada.

6. Modelo de estados

La máquina de estados mínima recomendada es:

candidate → staging → production → archived

Interpretación:

- candidate: versión registrada pero todavía no necesariamente desplegable
- staging: versión habilitada para preproducción o smoke checks finales
- production: versión actualmente autorizada para inferencia real
- archived: versión preservada solo para reproducibilidad o histórico

La transición entre estados no es libre. Debe respetar una política explícita. Las transiciones inválidas deben fallar.

7. Semántica de alias

Los alias deben tener semántica cerrada y estable.

7.1 latest_candidate

No significa “la última versión por timestamp”.
Debe significar:
la última versión candidata elegible bajo la política vigente.

7.2 staging

Debe apuntar a una única versión activa de preproducción.

7.3 production

Debe apuntar a una única versión servible y autorizada.

Si varias versiones son potencialmente elegibles para un alias, el registry no debe desempatar implícitamente. Debe aplicar una política explícita o fallar.

8. Esquema relacional mínimo recomendado

El módulo se beneficia claramente de una formulación relacional formal.

8.1 Tabla models

Campos mínimos:
- model_id (PK)
- task_type
- algo_family
- status
- created_ts

8.2 Tabla versions

Campos mínimos:
- model_id (FK)
- version
- artifact_uri
- artifact_hash
- training_run_id
- data_hash
- feature_hash
- config_hash
- code_commit
- metrics_json
- validation_decision
- stage
- created_ts
- registered_by

Restricción principal:
UNIQUE(model_id, version)

8.3 Tabla aliases

Campos mínimos:
- model_id (FK)
- alias
- version (FK)
- set_ts

Debe imponerse unicidad por (model_id, alias).

8.4 Tabla events

Campos mínimos:
- event_id (PK)
- event_ts
- event_type
- model_id (FK)
- from_version
- to_version
- actor
- reason
- approval_ref
- payload_hash

8.5 Constraints mínimos

- CHECK(stage IN ('candidate','staging','production','archived'))
- CHECK(validation_decision IN ('pass','fail','override'))
- FK versions.model_id → models.model_id
- FK aliases.version → versions.version con clave compuesta correcta
- ON DELETE RESTRICT en entidades críticas

9. Registro de versión

La operación register_version(payload) debe:

1. validar campos obligatorios;
2. verificar unicidad de (model_id, version);
3. validar que el artefacto exista y sea legible;
4. verificar artifact_hash;
5. persistir la versión como candidate;
6. registrar evento REGISTER_VERSION;
7. dejar el sistema en estado consistente incluso si falla cualquier subpaso.

El registro no debe aceptar versiones “a medio documentar”. Un artefacto sin hashes ni trazabilidad suficiente no merece entrar en la capa oficial.

10. Resolución determinista

La función resolve(ref) es el corazón operativo del módulo.

Debe aceptar referencias del tipo:
- alias
- versión exacta
- model_id + alias
- model_id + version
- opcionalmente referencia histórica por fecha

La secuencia correcta es:

1. validar sintaxis del ref;
2. expandir a consulta explícita;
3. recuperar candidatos elegibles;
4. aplicar política de unicidad;
5. verificar servibilidad e integridad del artefacto;
6. devolver una única versión;
7. si no existe una única respuesta válida, fallar.

La política prohibida es:
“devolver la primera fila encontrada”.

11. SQL explícito de resolución

Conviene cerrar semánticamente resolve() con consultas explícitas.

11.1 Resolve production

SELECT v.*
FROM versions v
JOIN aliases a
  ON v.model_id = a.model_id
 AND v.version = a.version
WHERE a.model_id = ?
  AND a.alias = 'production'
LIMIT 1;

11.2 Resolve version exacta

SELECT *
FROM versions
WHERE model_id = ?
  AND version = ?;

11.3 Ambigüedad

Si la consulta devuelve:
- cero filas → FAIL_NOT_FOUND
- más de una fila cuando debería haber una → FAIL_AMBIGUOUS

12. Atomicidad de promociones

La promoción a staging o production debe ser una transacción lógica indivisible.

Los tres subpasos críticos son:
1. precheck de elegibilidad;
2. actualización del alias;
3. escritura del evento de promoción.

Estos tres pasos deben ejecutarse como una única operación consistente.
No puede ocurrir:
- que cambie el alias y no exista el evento;
- que exista el evento y no haya cambiado el alias;
- o que el alias apunte a una versión no íntegra.

13. CAS y concurrencia

El módulo debería especificar explícitamente una estrategia compare-and-swap o equivalente bajo transacción/lock.

Pseudocódigo conceptual:

def promote(model_id, version, target_alias):
    with txn:
        v = load_version(model_id, version)
        assert v exists
        assert v.validation_decision in {'pass', 'override'}
        assert artifact_hash_matches(v)

        old_version = load_alias(model_id, target_alias)

        success = compare_and_set_alias(
            model_id=model_id,
            alias=target_alias,
            expected_old=old_version,
            new_version=version
        )

        if not success:
            raise ConcurrencyError()

        append_event(
            event_type="PROMOTE",
            model_id=model_id,
            from_version=old_version,
            to_version=version
        )

        confirm_post_write_consistency()

Aunque el módulo no necesite teoría distribuida avanzada, sí debe cerrar:
- retry policy;
- backoff;
- conflicto esperado;
- y política fail-closed bajo concurrencia.

14. Integridad de artefactos

Una versión registrada pero no servible destruye el valor operacional del registry.
Por ello, antes de permitir registro, promoción o resolución a production, el módulo debe verificar:

- existencia física de artifact_uri;
- legibilidad del artefacto;
- coincidencia entre artifact_hash y hash calculado;
- presencia del manifiesto asociado;
- compatibilidad mínima con la clase de modelo declarada.

Si falla cualquiera de estos checks, la versión puede existir en histórico, pero no debe resolverse como production.

15. Política de promoción

La promoción a production solo debe permitirse si:
- la versión existe;
- el artefacto es íntegro;
- validation_decision es pass o existe override explícito;
- la transición de stage es válida;
- la versión cumple el contrato de inferencia vigente;
- y la promoción queda registrada como evento auditable.

La decisión de “esta versión merece promotion” debe venir de validation_suite o de una decision layer explícita. El registry ejecuta y gobierna la decisión; no la inventa.

16. Rollback semánticamente correcto

El rollback no es borrar historia.
Es crear un nuevo evento que reasigna un alias, normalmente production, a una versión anterior conocida y servible.

Un rollback correcto debe:
- especificar to_version de forma explícita;
- verificar que esa versión sigue existiendo;
- verificar integridad de su artefacto;
- registrar from_version y to_version;
- conservar intacta la historia de promociones previas;
- y dejar evidencia de actor, motivo y aprobación si aplica.

La política incorrecta es:
“volver al estado anterior” sin identificar formalmente la versión destino.

17. Event sourcing y reconstrucción de estado

El módulo encaja bien con un esquema híbrido de estado materializado + eventos append-only.

Conceptualmente:

state(model_id) = foldl(initial_state, ordered_events(model_id), apply_event)

Esto aporta:
- auditabilidad fuerte;
- reconstrucción histórica;
- explicación de cómo se llegó al estado actual;
- y mayor resiliencia frente a dudas sobre mutaciones pasadas.

No hace falta convertir todo en event sourcing puro, pero sí garantizar que el estado actual sea compatible con el historial.

18. Failure modes frecuentes

El módulo debe capturar explícitamente:

18.1 Duplicate version key
Dos versiones con la misma clave lógica.

18.2 Alias ambiguity
Un alias apuntando a múltiples versiones o sin política de unicidad.

18.3 Artifact corruption
artifact_uri existe pero hash no coincide o el artefacto no es legible.

18.4 Concurrent promotion race
Dos promociones simultáneas compiten y dejan el alias en estado incierto si no existe CAS/txn correcto.

18.5 Event loss
Se actualiza el alias pero no se escribe el evento, o viceversa.

18.6 FK violation
Referencias colgantes entre aliases, versions y models.

18.7 Invalid override
Una versión fail llega a production sin aprobación formal trazable.

18.8 Ambiguous resolve
resolve(ref) devuelve varias candidatas o depende del orden físico de lectura.

19. Flags y salidas mínimas recomendadas

El output del módulo debería incluir algo del tipo:

{
  "model_id": "...",
  "version": "...",
  "gate": "PASS|WARN|FAIL",
  "stage": "candidate|staging|production|archived",
  "artifact_integrity": {
      "artifact_exists": true,
      "hash_match": true,
      "manifest_exists": true
  },
  "resolution": {
      "resolved_from_ref": "...",
      "resolved_uniquely": true
  },
  "governance": {
      "validation_decision": "pass|fail|override",
      "override_ref": "...",
      "promotion_eligible": true
  },
  "hard_failures": [...],
  "soft_flags": [...],
  "binding_failure": "...",
  "recommended_action": "...",
  "notes": [...]
}

Cada fallo relevante debe indicar:
- operación afectada;
- entidad afectada;
- severidad;
- causa concreta;
- y acción recomendada.

20. Observabilidad

20.1 Logging

- INFO: registro correcto, promoción correcta, rollback correcto, resolución correcta
- WARNING: override, fallback explícito, retry por concurrencia
- ERROR: hash inconsistente, artefacto ausente, resolución ambigua, fallo transaccional, violación de invariantes

20.2 Métricas

El módulo debería emitir al menos:
- n_models_total
- n_versions_total
- n_promotions_production
- n_rollbacks
- n_override_promotions
- resolve_failures
- artifact_integrity_failures
- avg_candidate_to_production_latency
- cas_conflicts
- promotion_retries

21. Qué ya está muy bien en esta especificación

La especificación base ya acierta en piezas importantes:

- visión del módulo como control de riesgo de modelo;
- estados candidate → staging → production → archived;
- metadatos fuertes de versión como artifact_hash, training_data_hash y code_commit;
- aliases lógicos que resuelven a una versión única;
- historial append-only de eventos;
- promociones y rollbacks auditables;
- inmutabilidad conceptual de artefactos;
- y atomicidad entendida como requisito estructural.

Eso da una base institucional sólida, aunque no sea un módulo de densidad matemática extrema.

22. Qué falta para empujarlo al techo realista

Dado el tipo de módulo, el techo realista no es 95+ matemático. Es más bien 82–85 con excelencia infra.

Para acercarse a ese techo, faltan sobre todo:

22.1 ERD formal completo
Con PK, FK, CHECK, índices y claves compuestas bien especificadas.

22.2 SQL explícito de operaciones críticas
Especialmente resolve(), list_history() y transitions.

22.3 Política de concurrencia cerrada
CAS, retries, backoff y failover bajo conflicto.

22.4 Reconstrucción formal desde eventos
Para reforzar auditabilidad.

22.5 Bounds operativos
No teoría pesada, pero sí métricas de conflict rate, resolve failure rate y promotion safety.

23. MVP vs versión avanzada

23.1 MVP

- esquema relacional formal;
- unicidad de (model_id, version);
- aliases únicos;
- resolve determinista;
- validación de integridad de artefacto;
- promoción atómica;
- rollback auditable;
- eventos append-only;
- output conservador fail-closed.

23.2 Versión avanzada

- event sourcing más formal;
- version vectors o metadata extra de concurrencia si hace falta;
- dependency graph modelo→features→data;
- réplicas de lectura con garantías explícitas;
- monitorización de conflictos de promoción;
- validación de integridad también en resolve de forma periódica.

24. Interpretación correcta

La lectura correcta de model_registry.py es:

“este módulo gobierna transaccionalmente qué versión exacta de un modelo puede vivir operativamente en staging o production, con integridad, trazabilidad y resolución determinista.”

La lectura incorrecta es:

“este módulo lista modelos y guarda metadatos para consultarlos después.”

No está para consultar cómodamente.
Está para impedir ambigüedad operacional sobre el modelo activo.

25. Referencias orientativas

- Codd, por modelo relacional y normalización
- Gray, por concepto transaccional y atomicidad
- literatura de artifact registries y ML lineage
- patrones append-only y event sourcing
- prácticas institucionales de model governance y model risk management

26. Conclusión

model_registry.py es una pieza de infraestructura crítica y su valor está en la disciplina operacional, no en la complejidad matemática.

La combinación de:
- metadatos inmutables por versión,
- aliases lógicos,
- promociones atómicas,
- rollbacks auditables,
- historial append-only,
- y resolución determinista

define una base muy seria de gobernanza de modelos.

No necesita sobreingeniería distribuida para ser excelente. Lo que necesita es cerrar bien invariantes, esquema relacional, semántica de concurrencia y trazabilidad de artefactos.

Como transactional model governance layer, la especificación es buena y con esos cierres puede quedar institucionalmente muy sólida.
