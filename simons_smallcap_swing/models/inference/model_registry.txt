Modulo: models/inference/model_registry.py

1) Objetivo
Gestionar el registro central de modelos (metadata, artefactos, versiones y estado de promocion) para garantizar reproducibilidad, gobernanza y despliegue seguro de inferencia.

2) Alcance
Incluye:
- Alta de nuevas versiones de modelo.
- Consulta por alias/etapa (candidate, staging, production, archived).
- Resolucion de modelo activo para inferencia.
- Versionado de artefactos y dependencias (features/config/code hash).
- Trazabilidad de decisiones de promocion/rollback.

No incluye:
- Entrenamiento de modelos.
- Almacenamiento binario pesado fuera de paths referenciados.

3) Dependencias
Upstream:
- outputs de training (metrics, artifacts, manifests)
- validation_suite decision

Downstream:
- models/inference/predict
- backtest/run_walkforward (seleccion por version)
- ops/monitoring (modelo activo)

4) Entidades clave
Model:
- model_id (estable), task_type, algo_family.

Version:
- model_version (semver o timestamped id)
- artifact_uri
- training_run_id
- training_data_hash
- feature_set_hash
- config_hash
- code_commit
- metrics_snapshot
- validation_decision
- stage

Alias:
- latest_candidate
- staging
- production

5) Contrato de entrada
Operaciones minimas:
- register_model_version(payload)
- promote_model(model_id, version, stage)
- rollback_model(model_id, to_version)
- resolve_model(ref)
- list_models(filters)

Payload minimo de registro:
- model_id
- model_version
- artifact_uri
- training_run_id
- metrics_summary
- validation_decision
- created_ts

6) Contrato de salida
Persistencia (ejemplo):
- models/registry/registry.parquet (o sqlite/jsonl)
- models/registry/history/{date}/events.parquet
- models/registry/model_cards/{model_id}_{version}.json

Respuestas API internas:
- resolved model metadata completo
- historial de promociones
- estado actual por alias

7) Reglas de gobernanza
- Solo versiones con validation_suite=pass pueden ir a production (salvo override auditado).
- Promotion y rollback deben generar evento auditable.
- No se reescribe historia; solo append de eventos.
- Cada version debe apuntar a artefacto inmutable.

8) Flujo de registro/promotion
1. Registrar version candidata post-training.
2. Adjuntar metricas y hashes de reproducibilidad.
3. Evaluar decision de validacion.
4. Promover a staging/production segun policy.
5. Registrar evento y actualizar alias atomicamente.

9) Validaciones minimas
- unicidad de (model_id, model_version).
- artifact_uri accesible.
- campos obligatorios no nulos.
- stage valido.
- coherencia entre validation_decision y stage target.

10) Manejo de rollback
- rollback rapido por alias a version previa estable.
- mantener snapshot de alias antes/despues.
- registrar motivo y aprobador en metadata.

11) Observabilidad
Logging:
- INFO: registros/promociones exitosas.
- WARNING: promociones forzadas con override.
- ERROR: conflictos de version/artefacto inexistente.

Metricas:
- n_models_total
- n_versions_total
- n_promotions_prod
- n_rollbacks
- avg_time_candidate_to_prod

12) CLI sugerida
python -m models.inference.model_registry register ^
  --model-id lgbm_main ^
  --version 2026.03.05.1 ^
  --artifact-uri models/artifacts/lgbm/train_lgbm_20260305_001 ^
  --training-run-id train_lgbm_20260305_001 ^
  --metrics-path models/reports/lgbm/train_lgbm_20260305_001/metrics_summary.json

python -m models.inference.model_registry promote ^
  --model-id lgbm_main ^
  --version 2026.03.05.1 ^
  --stage staging

13) Estructura interna recomendada
- load_registry_store(...)
- validate_registration_payload(...)
- register_model_version(...)
- promote_model_version(...)
- rollback_model_version(...)
- resolve_model_ref(...)
- list_model_versions(...)
- append_registry_event(...)
- persist_registry(...)
- main()

14) Criterios de aceptacion
- Registro soporta trazabilidad completa version->artefacto->metricas.
- Resolucion de modelo para inferencia es determinista.
- Promotion/rollback auditables y seguros.

15) Casos de prueba minimos
- registrar version valida.
- rechazar duplicado de version.
- promover a production con pass.
- bloquear promotion a production con fail (sin override).
- rollback restaura alias anterior.

16) Checklist rapido
- [ ] Esquema de registro versionado.
- [ ] Operaciones register/promote/rollback/resolve.
- [ ] Eventos auditables append-only.
- [ ] Validaciones de gobernanza.
- [ ] Tests de integridad de alias.
