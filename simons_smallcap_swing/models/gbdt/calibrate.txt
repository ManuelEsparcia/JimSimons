Especificación Cuantitativa: models/gbdt/calibrate.py

1. Propósito del módulo

El objetivo de models/gbdt/calibrate.py es transformar los scores crudos producidos por un modelo GBDT en probabilidades utilizables, comparables y auditables.

La función correcta del módulo no es “mejorar el ranking” del modelo base, sino corregir la relación entre score y frecuencia empírica del evento para que una probabilidad predicha tenga interpretación estadística razonable fuera de muestra.

La pregunta exacta que debe responder el módulo es:

“Dado un score crudo s = f(x), qué transformación g(s) produce probabilidades p̂ coherentes con la frecuencia empírica observada en datos temporalmente limpios?”

2. Principios no negociables

2.1 El calibrador no reentrena el modelo base
El modelo base produce scores.
Este módulo aprende una transformación posterior:
    p̂ = g(s)

2.2 No leakage
Nunca se calibra sobre predicciones in-sample.

2.3 Cronología primero
En presencia de estructura temporal, la calibración debe respetar el orden causal de la información.

2.4 Parsimonia por defecto
La flexibilidad adicional debe ganarse empíricamente.
No se concede por defecto.

2.5 La calibración no puede comprar una mejora cosmética a costa de romper otras propiedades
Reducir ECE empeorando fuertemente NLL, Brier o estabilidad del score no se considera éxito.

2.6 Auditabilidad total
La salida debe incluir:
- método activo,
- muestra usada,
- métricas antes/después,
- decisión final,
- y artefacto serializado reproducible.

3. Qué hace y qué no hace

3.1 Qué hace
- recibe scores OOF o scores de un bloque cronológicamente válido;
- ajusta uno o varios calibradores candidatos;
- evalúa calibración probabilística;
- selecciona método activo o decide no calibrar;
- persiste calibrador y predicciones calibradas.

3.2 Qué no hace
- no reentrena el GBDT;
- no usa datos futuros para calibrar;
- no sustituye a inferencia o ensemble;
- no convierte automáticamente todo score en “mejor modelo”.

4. Formulación matemática

Sea:
- x_i: features
- y_i in {0,1}: etiqueta
- s_i = f(x_i): score crudo del modelo base

El calibrador aprende una función:
    g: R -> [0,1]

y produce:
    p̂_i = g(s_i)

La condición ideal de calibración perfecta es:

    P(y=1 | g(s)=p) = p

En práctica, se usa una versión agregada por grupos o bins:
    E[y | i in B] ≈ E[p̂ | i in B]

5. Rol dentro del stack

Este módulo vive después de:
- entrenamiento del GBDT
- generación de scores OOF o bloque de validación limpio

Y antes de:
- signal gating
- thresholding
- conviction sizing
- ensemble probabilístico
- monitorización de drift probabilístico

6. Entradas contractuales

6.1 Entradas mínimas
- predictions_oof.parquet o equivalente
- labels.parquet
- config de calibración

6.2 Campos recomendados
- timestamp
- asset_id
- fold_id
- model_id
- run_id
- score_type

6.3 Requisito crítico
La unión entre scores y labels debe ser uno a uno y auditada.

7. Salidas contractuales

El módulo debe producir al menos:

- calibrator.pkl
- calibrated_predictions.parquet
- calibration_summary.json
- manifest.json

Opcionalmente:
- reliability diagram
- histogramas de score/probabilidad
- tablas por bin
- estabilidad temporal por subperiodos

8. Métodos soportados

La versión canónica del módulo debe soportar:

8.1 Platt scaling
    p̂_i = sigma(A s_i + B)

con (A,B) estimados por minimización de NLL.

8.2 Isotonic regression
Función monótona no decreciente g que minimiza error cuadrático o equivalente sobre el conjunto de calibración.

8.3 Temperature scaling
    p̂_i = sigma(s_i / T)
o extensión multicategoría si aplica.

La implementación puede soportar métodos adicionales, pero no forman parte del contrato base.

9. Política canónica de selección

La regla recomendada es:

1. Platt scaling como método por defecto
2. Temperature scaling si el score tiene semántica de logit o escala compatible
3. Isotonic solo si hay muestra suficiente y mejora robusta

Interpretación correcta:
- Platt es el baseline institucional
- isotonic es una excepción justificada, no un derecho automático
- temperature es una alternativa ligera cuando la distorsión es principalmente de escala

10. Construcción correcta del conjunto de calibración

La calibración solo puede entrenarse sobre una de estas fuentes:

10.1 OOF estricto
Cada observación debe venir de un modelo que no la vio en entrenamiento.

10.2 Bloque cronológico separado
Si existe un bloque de validación posterior al entrenamiento y anterior al test o despliegue.

Están prohibidas:
- predicciones in-sample
- mezcla entrenamiento + calibración sin separación limpia
- uso del test final para elegir calibrador

11. Métricas obligatorias

El módulo debe reportar al menos:

- ECE
- MCE
- Brier score
- NLL

Y, como diagnóstico adicional:
- AUC o equivalente si la tarea es binaria
- Spearman / IC si el score se usa ordinalmente
- correlación score vs calibrated score
- drift por subperiodos si está habilitado

12. Definiciones métricas base

12.1 ECE
Con bins B_m:

    ECE = sum_m (|B_m| / n) * |Acc(B_m) - Conf(B_m)|

12.2 MCE

    MCE = max_m |Acc(B_m) - Conf(B_m)|

12.3 Brier

    Brier = (1/n) sum_i (y_i - p̂_i)^2

12.4 NLL

    NLL = (1/n) sum_i [ -y_i log(p̂_i) - (1-y_i) log(1-p̂_i) ]

13. Política de binning

La versión base debe fijar política explícita.
Recomendación:

- n_bins = 10 por defecto
- equal-frequency binning por defecto
- fallback a fewer bins si tamaño muestral por bin es insuficiente

Si un bin cae por debajo de `min_bin_size`, la ECE debe marcarse como inestable.

14. Regla de aceptación del calibrador

El calibrador no se acepta porque “reduce algo el ECE”.
Debe superar una política explícita.

Regla recomendada:

Aceptar calibrador C solo si:

1. mejora ECE de forma no trivial
2. no empeora materialmente NLL
3. no empeora materialmente Brier
4. no degrada de forma relevante la capacidad ordinal si el score se usa para ranking
5. la mejora no depende enteramente de un único subperiodo

Si no se cumple lo anterior:
- se conserva el score base
- o se usa identity calibrator

15. Política de “no calibrar” es válida

Este punto debe quedar claro en el contrato:
el módulo puede concluir que el mejor calibrador es “ninguno”.

Eso no es fallo.
Es una decisión válida si:
- la muestra es pequeña,
- la mejora es inestable,
- o el coste en robustez supera el beneficio.

16. Estabilidad temporal

La especificación base debe incluir, al menos, una evaluación descriptiva por bloques temporales.

Ejemplos:
- train calibrator on OOF global
- evaluar por año / semestre / terciles temporales
- comparar ECE, Brier, NLL por subperiodo

La estabilidad temporal es parte de la validación.
No hace falta imponer CUSUM como contrato base, pero sí hace falta que el módulo detecte si la mejora está concentrada en un solo tramo del histórico.

17. Isotonic: política de uso responsable

Isotonic no debe activarse si:
- la muestra de calibración es pequeña,
- el score tiene demasiados empates,
- la curva resultante es excesivamente escalonada,
- o la mejora desaparece fuera del bloque donde se ajustó.

La versión base debe exigir:
- tamaño mínimo de muestra,
- tamaño mínimo por bin o por tramo efectivo,
- y mejora robusta frente a Platt.

18. Temperature scaling: política correcta

Temperature scaling tiene sentido cuando:
- el score es cercano a logit,
- o se usa una familia de modelos donde la miscalibración parece principalmente de escala

No debe activarse automáticamente sobre cualquier score arbitrario sin justificación de escala.

19. Transformación y ranking

Para métodos monótonos:
- el orden relativo del score no debe invertirse salvo tolerancias numéricas o ties

Para isotonic:
- pueden aparecer ties
- eso debe aceptarse como parte de la semántica del método, no como bug

20. Clips y estabilidad numérica

Antes de computar NLL:
- clip de probabilidades a [eps, 1-eps]

Regla recomendada:
- eps configurable, por ejemplo 1e-6 o 1e-8

Además:
- no se permiten NaN
- no se permiten infinidades
- toda salida debe pertenecer a [0,1]

21. Flujo operativo canónico

Paso 1:
cargar scores, labels y config

Paso 2:
validar integridad del join y cobertura

Paso 3:
identificar score_type y elegibilidad temporal

Paso 4:
construir conjunto de calibración válido

Paso 5:
ajustar candidatos permitidos por config

Paso 6:
evaluar métricas antes/después

Paso 7:
aplicar regla formal de aceptación

Paso 8:
persistir calibrador activo, predicciones calibradas y resumen

22. Invariantes del módulo

Toda salida válida debe cumplir:

- 0 <= p̂_i <= 1
- no leakage
- artefacto serializable
- mismo número de filas alineadas tras calibración
- toda decisión final queda registrada
- si se activa identity/no calibration, debe quedar explícito

23. Tests mínimos obligatorios

23.1 Integridad
Predicciones y labels deben alinear uno a uno sin duplicados.

23.2 No leakage
El módulo debe rechazar un conjunto de calibración que incluya filas in-sample si esa información está disponible.

23.3 Rango probabilístico
Toda probabilidad calibrada debe estar en [0,1].

23.4 Monotonicidad
Platt y temperature deben preservar orden.
Isotonic debe ser no decreciente.

23.5 Reproducibilidad
Mismos inputs + misma config + misma seed => mismo calibrador

23.6 Métricas
ECE, Brier y NLL deben poder recomputarse desde calibrated_predictions.

27. Qué no debe hacer este módulo

La versión canónica no debe:
- imponer BBQ o beta calibration como default
- activar isotonic por moda
- confundir mejor ECE con mejor modelo
- usar datos futuros para recalibrar
- vender una mejora cosmética como mejora económica real

28. Extensiones futuras razonables

Como fase 2 o research:
- beta calibration / BBQ
- smoothed isotonic
- formal drift tests
- calibration by regime
- multiclass extensions más avanzadas

Pero ninguna de estas extensiones debe contaminar el contrato mínimo.

29. Interpretación correcta

La lectura correcta de calibrate.py no es:
“suavizamos los scores del GBDT”.

La lectura correcta es:
“aprendemos una transformación probabilística post-modelo, temporalmente limpia y auditable, que convierte scores en probabilidades defendibles para las capas downstream del sistema.”

Ese es su rol real.

30. Conclusión

models/gbdt/calibrate.py debe ser una pieza sobria y fuerte de infraestructura ML.

Recibe scores fuera de muestra y etiquetas válidas.
Ajusta calibradores candidatos.
Evalúa calibración con métricas apropiadas.
Acepta uno o ninguno bajo una regla explícita.
Serializa el artefacto.
Y deja una traza suficiente para justificar por qué una probabilidad final fue la que fue.

Ese es exactamente el comportamiento que debe tener una capa seria de calibración probabilística en un stack cuantitativo institucional.
