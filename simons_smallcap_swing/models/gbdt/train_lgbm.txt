\section{Objetivo estratégico del módulo}

El módulo \texttt{models/gbdt/train\_lgbm.py} implementa el entrenamiento de modelos \textbf{LightGBM} dentro de una arquitectura cuantitativa cross-sectional para \textbf{small caps}. Su función no es simplemente ajustar un modelo de boosting sobre una matriz tabular, sino producir una función predictiva temporalmente válida, metodológicamente auditable y operacionalmente reutilizable dentro de un pipeline de selección, ranking y decisión de cartera.

La unidad económica relevante del sistema no es la observación aislada, sino la sección cruzada por fecha. En cada instante \(t\), el modelo recibe un universo elegible \(\mathcal{U}_t\) de activos y asigna a cada nombre \(i \in \mathcal{U}_t\) un score que pretende capturar señal futura útil para trading. Formalmente, si \(\bm{x}_{i,t} \in \mathbb{R}^{d}\) denota el vector de características del activo \(i\) en fecha \(t\), el objetivo del módulo es estimar una función
\[
\hat{f}: \mathcal{X} \rightarrow \mathbb{R}
\]
tal que
\begin{equation}
\hat{y}_{i,t} = \hat{f}(\bm{x}_{i,t})
\approx
\E\!\left[y_{i,t}\mid \bm{x}_{i,t}\right],
\end{equation}
donde \(y_{i,t}\) puede representar:
\begin{itemize}[leftmargin=2em]
    \item un retorno forward continuo,
    \item una etiqueta binaria de dirección o superación de umbral,
    \item o una relevancia ordinal para ranking cross-sectional.
\end{itemize}

La salida del modelo no se interpreta de forma abstracta. Debe ser apta para alimentar downstream:
\begin{itemize}[leftmargin=2em]
    \item selección de candidatos,
    \item ordenación relativa dentro del universo,
    \item combinación con otros modelos en un ensemble,
    \item calibración posterior de probabilidades,
    \item y evaluación walk-forward genuinamente fuera de muestra.
\end{itemize}

Por ello, este módulo debe entenderse como una pieza de \textbf{research-to-production}, no como un experimento aislado.

\section{Principios de diseño}

El módulo se rige por una doctrina explícita:

\begin{enumerate}[leftmargin=2em]
    \item \textbf{La cronología es inviolable.} Ninguna observación puede contaminar el entrenamiento de un bloque temporal anterior a su realización económica.
    
    \item \textbf{El modelo se evalúa por utilidad cross-sectional fuera de muestra, no por ajuste in-sample.} Una pérdida baja en entrenamiento no tiene valor si no se traduce en ranking o discriminación robusta OOS.
    
    \item \textbf{La reproducibilidad es obligatoria.} Toda ejecución debe dejar artefactos, métricas, configuración y metadatos suficientes para reconstruir exactamente el experimento.
    
    \item \textbf{La complejidad del modelo debe merecerse.} El aumento de capacidad solo se acepta si produce mejora estable y no una simple captura de ruido histórico.
    
    \item \textbf{La promoción a ensemble o producción requiere gates explícitos.} Un modelo no avanza por intuición; avanza si cumple reglas formales.
\end{enumerate}

\section{Unidad de modelado y contexto cross-sectional}

En estrategias de small caps, el objetivo raramente es acertar el valor absoluto de retorno de un único activo de forma aislada. Lo relevante es ordenar el universo contemporáneo de forma que los activos con score alto presenten, en agregado, mejor comportamiento futuro que los de score bajo.

Sea \(\mathcal{U}_t = \{1,\dots,N_t\}\) el universo elegible en fecha \(t\). El modelo genera el conjunto de scores
\[
\{\hat{y}_{i,t}\}_{i \in \mathcal{U}_t}.
\]
La calidad de esos scores se juzga principalmente por su capacidad para inducir un ranking económicamente útil dentro de cada fecha, más que por minimizar un error global agregado mezclando periodos y activos heterogéneos.

Esta perspectiva tiene consecuencias importantes:
\begin{itemize}[leftmargin=2em]
    \item las métricas deben computarse por fecha y luego agregarse,
    \item la validación debe respetar dependencia temporal,
    \item y el tratamiento del universo, coverage y missingness no puede separarse del problema económico.
\end{itemize}

\section{Definición formal de los datos de entrada}

El módulo asume como inputs contractuales:

\begin{itemize}[leftmargin=2em]
    \item \texttt{features\_path}: matriz de features con clave lógica mínima \texttt{(date, symbol)};
    \item \texttt{labels\_path}: etiquetas forward compatibles con la misma clave lógica;
    \item \texttt{splits\_path}: asignación de folds y roles temporales \texttt{train/valid/test} generada aguas arriba;
    \item \texttt{config\_path}: configuración cerrada del experimento;
    \item \texttt{run\_id}: identificador único de ejecución.
\end{itemize}

El join entre tablas se realiza exclusivamente sobre claves canónicas. Cualquier ambigüedad, duplicado o desalineación se trata como error de integridad, no como advertencia cosmética.

\section{Splits temporales purged y embargo}

La validez del módulo depende críticamente del esquema de partición temporal. En presencia de labels forward, ventanas superpuestas y dependencia serial, la validación aleatoria clásica es metodológicamente inaceptable. Por ello, el entrenamiento se apoya en splits \textbf{purged} con \textbf{embargo}.

Intuitivamente:
\begin{itemize}[leftmargin=2em]
    \item el \textbf{purging} elimina del entrenamiento observaciones cuyo horizonte de evento intersecta el bloque de validación o test;
    \item el \textbf{embargo} retira además una banda temporal alrededor del bloque evaluado para reducir contaminación por proximidad.
\end{itemize}

Sea \(\mathcal{D}\) el conjunto de observaciones elegibles tras el join de features, labels y splits. Para un fold \(k\), el conjunto de entrenamiento efectivo se define como
\begin{equation}
\mathcal{D}^{(k)}_{\mathrm{train}}
=
\left\{
i \in \mathcal{D}
\;\middle|\;
\texttt{fold\_id}_i = k,\;
\texttt{split\_role}_i = \texttt{train},\;
\texttt{inclusion\_flag}_i = 1
\right\},
\end{equation}
mientras que validación y test se restringen a observaciones temporalmente posteriores y libres de solapamiento causal con el bloque de entrenamiento:
\begin{equation}
\mathcal{D}^{(k)}_{\mathrm{eval}}
=
\left\{
i \in \mathcal{D}
\;\middle|\;
\texttt{split\_role}_i \in \{\texttt{valid},\texttt{test}\},
\;
t_i > t_{\max}^{\mathrm{train},k},
\;
i \notin \texttt{purged/embargo}
\right\}.
\end{equation}

La propiedad que se busca no es solo separación cronológica superficial, sino \textbf{independencia operacional suficiente} entre bloques para que la performance observada se aproxime a un entorno de despliegue real.

\section{Formulación matemática de LightGBM}

Sea \(f^{(m)}\) el predictor tras \(m\) iteraciones de boosting. En LightGBM, el modelo final se construye de forma aditiva:
\begin{equation}
f^{(M)}(\bm{x}) = \sum_{m=1}^{M} \nu \, T_m(\bm{x}),
\end{equation}
donde \(T_m\) es el árbol añadido en la iteración \(m\) y \(\nu\) es la tasa de aprendizaje.

El problema de entrenamiento puede expresarse como la minimización de una pérdida regularizada:
\begin{equation}
\min_{f}
\;
\sum_{i=1}^{n}\ell\!\left(y_i, f(\bm{x}_i)\right)
+
\sum_{m=1}^{M}\Omega(T_m),
\end{equation}
con penalización de complejidad típicamente de la forma
\begin{equation}
\Omega(T_m)=\gamma L_m + \frac{\lambda}{2}\sum_{l=1}^{L_m} w_{m,l}^{2},
\end{equation}
donde \(L_m\) es el número de hojas del árbol \(m\), \(w_{m,l}\) son los pesos de hoja y \((\gamma,\lambda)\) controlan complejidad estructural y regularización.

Desde una perspectiva cuantitativa, los hiperparámetros principales no son meros knobs de librería: definen la capacidad efectiva del modelo para aproximar interacciones no lineales, efectos umbral, saturaciones y jerarquías entre señales. En particular:
\begin{itemize}[leftmargin=2em]
    \item \texttt{num\_leaves} regula granularidad funcional;
    \item \texttt{max\_depth} limita complejidad jerárquica;
    \item \texttt{min\_data\_in\_leaf} impone masa mínima por partición;
    \item \texttt{feature\_fraction} y \texttt{bagging\_fraction} actúan como regularización estocástica;
    \item \texttt{lambda\_l1}, \texttt{lambda\_l2} penalizan sobreajuste paramétrico;
    \item \texttt{learning\_rate} gobierna el compromiso entre plasticidad y estabilidad.
\end{itemize}

\section{Objetivos soportados}

\subsection{Regresión}

Cuando el target es un retorno continuo, el módulo entrena bajo objetivo de regresión. En ese caso, la pérdida puntual puede ser, según configuración, cuadrática o absoluta:
\begin{align}
\mathcal{L}_{\mathrm{MSE}}
&=
\frac{1}{n}\sum_{i=1}^{n}(y_i-\hat{y}_i)^2,\\
\mathcal{L}_{\mathrm{MAE}}
&=
\frac{1}{n}\sum_{i=1}^{n}|y_i-\hat{y}_i|.
\end{align}

Sin embargo, en un problema de stock selection cross-sectional, estas pérdidas son auxiliares. La métrica central es la calidad ordinal por fecha, medida típicamente mediante Information Coefficient de Spearman:
\begin{equation}
\IC_t
=
\Corr\!\left(
\rank(\hat{y}_{i,t}),
\rank(y_{i,t})
\right)_{i\in\mathcal{U}_t}.
\end{equation}
A nivel agregado, se reportan estadísticas como:
\[
\mathrm{median}(\IC_t), \qquad
\mathrm{mean}(\IC_t), \qquad
\frac{\mathrm{mean}(\IC_t)}{\mathrm{std}(\IC_t)}.
\]

También puede calcularse el \textit{hit ratio} direccional:
\begin{equation}
\mathrm{HitRatio}
=
\frac{1}{n}\sum_{i=1}^{n}\mathbf{1}\!\left(y_i\hat{y}_i>0\right),
\end{equation}
aunque esta métrica se considera secundaria frente a medidas ordinales y de ranking.

\subsection{Clasificación binaria}

Cuando el label es binario, el modelo optimiza una pérdida logística:
\begin{equation}
\mathcal{L}_{\mathrm{CE}}
=
-\frac{1}{n}
\sum_{i=1}^{n}
\left[
y_i\log \hat{p}_i + (1-y_i)\log(1-\hat{p}_i)
\right].
\end{equation}

Las métricas principales incluyen:
\begin{itemize}[leftmargin=2em]
    \item \(\AUC\)-ROC;
    \item \(\AUC\)-PR en clases desbalanceadas;
    \item logloss;
    \item y, si las probabilidades van a usarse downstream, métricas de calibración.
\end{itemize}

Si el dataset presenta desbalance severo o calidad heterogénea de observaciones, se admiten pesos muestrales:
\begin{equation}
\mathcal{L}_{\mathrm{weighted}}
=
-\frac{1}{n}
\sum_{i=1}^{n}
w_i
\left[
y_i\log \hat{p}_i + (1-y_i)\log(1-\hat{p}_i)
\right].
\end{equation}

\subsection{Ranking}

Para problemas donde el objetivo económico es explícitamente ordenar activos dentro de cada fecha, el módulo soporta \texttt{lambdarank}. Si \(\mathcal{Q}_t\) denota la lista de activos del día \(t\), la calidad del ranking puede medirse mediante:
\begin{equation}
\NDCG@k
=
\frac{\DCG@k}{\IDCG@k}.
\end{equation}

Lo relevante aquí es que el modelo deja de perseguir error puntual y optimiza directamente una funcional relacionada con la utilidad del ordenamiento. En contextos donde la cartera solo consume top-\(k\) nombres o percentiles extremos, este objetivo puede ser más fiel al problema económico que una regresión estándar.

\section{Selección de métrica principal}

Una de las carencias típicas en documentos técnicamente buenos pero no cerrados es dejar varias métricas coexistiendo sin una jerarquía oficial. Este módulo corrige eso estableciendo una doctrina explícita:

\begin{itemize}[leftmargin=2em]
    \item Si el uso downstream es \textbf{ranking cross-sectional}, la métrica principal es una métrica ordinal por fecha agregada OOS, normalmente \(\mathrm{median}(\IC_t)\) o \(\NDCG@k\).
    \item Si el uso downstream es \textbf{clasificación probabilística}, la métrica principal es logloss o AUC según el objetivo operativo exacto.
    \item Métricas auxiliares como RMSE, MAE, coverage o hit ratio nunca sustituyen a la métrica principal; solo contextualizan.
\end{itemize}

En consecuencia, el early stopping, la comparación entre folds y la promoción final deben referirse siempre a una métrica primaria definida ex ante en configuración.

\section{Preprocesado y contrato de features}

El módulo asume que la mayor parte del feature engineering ocurre aguas arriba, pero impone un contrato mínimo de consistencia:

\begin{enumerate}[leftmargin=2em]
    \item La lista \texttt{feature\_list\_final} debe ser idéntica entre folds.
    \item Se eliminan columnas no numéricas si no existe tratamiento explícito para ellas.
    \item Se excluyen columnas con varianza nula o cobertura insuficiente según umbrales configurados.
    \item Los missing values deben seguir una política definida y estable; no se admite tratamiento ad hoc distinto por fold sin quedar documentado.
\end{enumerate}

LightGBM puede manejar missingness de forma nativa, pero eso no exime de distinguir entre:
\begin{itemize}[leftmargin=2em]
    \item missing estructural con contenido informativo,
    \item missing accidental por problemas de datos,
    \item y columnas cuya tasa de ausencia las vuelve inestables o irrelevantes.
\end{itemize}

La carpeta necesita una política explícita porque el tratamiento de missing puede alterar de forma sustancial la estabilidad del modelo en small caps.

\section{Pesos muestrales y estructura por fecha}

El módulo permite pesos de observación cuando existe una hipótesis económica clara, por ejemplo:
\begin{itemize}[leftmargin=2em]
    \item penalizar observaciones de menor calidad,
    \item ponderar por liquidez,
    \item o modular el coste de errores en subuniversos con distinta relevancia económica.
\end{itemize}

En tareas de ranking, además, la estructura por fecha debe preservarse mediante información de grupos. Esto es crucial: la sección cruzada diaria no puede tratarse como una bolsa indiferenciada de filas si el objetivo económico vive en el orden relativo dentro de cada fecha.

\section{Controles anti-leakage}

El módulo define prohibiciones explícitas:

\begin{itemize}[leftmargin=2em]
    \item no se entrena sobre observaciones con \texttt{inclusion\_flag = 0};
    \item no se utiliza \texttt{test} para early stopping;
    \item no se usa performance de test para elegir hiperparámetros dentro de la misma ejecución;
    \item no se recalibra sobre el test final;
    \item no se permiten joins que introduzcan información futura en features o labels.
\end{itemize}

Estas restricciones no son un detalle de implementación; son parte de la definición del experimento. Un modelo que viola estas reglas queda invalidado aunque su score aparente sea excelente.

\section{Early stopping y regularización}

Cada fold se entrena con early stopping sobre un bloque de validación estrictamente separado:
\begin{itemize}[leftmargin=2em]
    \item si la métrica principal deja de mejorar durante \texttt{early\_stopping\_rounds}, el entrenamiento se detiene;
    \item el mejor número de iteraciones se congela para ese fold;
    \item el test se evalúa únicamente una vez finalizado el ajuste.
\end{itemize}

La política correcta de early stopping exige dos cosas:
\begin{enumerate}[leftmargin=2em]
    \item que la métrica monitorizada sea la métrica económicamente relevante, no una proxy cómoda;
    \item que la decisión no toque nunca el bloque de test.
\end{enumerate}

\section{Gates de aceptación del modelo}

Para pasar de research a candidato serio de ensemble o producción, el modelo debe superar un conjunto de gates. El módulo fija al menos los siguientes:

\begin{enumerate}[leftmargin=2em]
    \item \textbf{Gate de performance primaria}: la métrica principal OOS debe superar un umbral mínimo predefinido.
    
    \item \textbf{Gate de overfitting}: la divergencia entre entrenamiento y validación no puede exceder una tolerancia:
    \begin{equation}
    \Delta_{\mathrm{overfit}}
    =
    \mathcal{M}_{\mathrm{train}}-\mathcal{M}_{\mathrm{valid}}
    \le \varepsilon
    \quad
    \text{o equivalente según la orientación de la métrica.}
    \end{equation}
    
    \item \textbf{Gate de estabilidad}: la dispersión entre folds debe permanecer bajo control; no se aceptan modelos cuyo resultado dependa enteramente de uno o dos subperiodos.
    
    \item \textbf{Gate de coverage}: el porcentaje de filas válidas con score generado debe superar un umbral contractual.
    
    \item \textbf{Gate de integridad}: flags críticos de fuga, duplicados, fechas inválidas o labels corruptos deben ser cero.
\end{enumerate}

Un score medio atractivo no compensa una mala estabilidad o una cobertura insuficiente. En entornos cuantitativos reales, la robustez domina a la brillantez puntual.

\section{Flujo operativo del módulo}

El flujo oficial es:

\begin{enumerate}[leftmargin=2em]
    \item Cargar configuración, datos y \texttt{run\_id}.
    \item Unir features, labels y splits por clave canónica.
    \item Filtrar observaciones no elegibles.
    \item Construir datasets por fold y por rol temporal.
    \item Ejecutar prechecks de integridad.
    \item Entrenar LightGBM por fold con early stopping en validación.
    \item Generar predicciones OOF, valid y test según corresponda.
    \item Calcular métricas por fold y agregadas.
    \item Persistir artefactos del modelo y manifest de entrenamiento.
    \item Emitir veredicto \texttt{pass}, \texttt{warn} o \texttt{fail}.
\end{enumerate}

\section{Artefactos de salida}

Toda ejecución debe persistir, como mínimo:

\begin{itemize}[leftmargin=2em]
    \item modelo por fold;
    \item predicciones \texttt{oof};
    \item predicciones de test por fold;
    \item resumen de métricas;
    \item importancias de variables;
    \item manifest con configuración, timestamps, rutas, hashes y cobertura.
\end{itemize}

Una estructura típica sería:

\begin{verbatim}
models/artifacts/lgbm/{run_id}/fold_k/model.txt
models/predictions/lgbm/{run_id}/oof.parquet
models/predictions/lgbm/{run_id}/fold_k_test_pred.parquet
models/predictions/lgbm/{run_id}/latest_scores.parquet
models/reports/lgbm/{run_id}/metrics_summary.json
models/reports/lgbm/{run_id}/metrics_by_fold.parquet
models/reports/lgbm/{run_id}/feature_importance.parquet
models/reports/lgbm/{run_id}/training_manifest.json
\end{verbatim}

\section{Criterio de promoción a ensemble o producción}

La carpeta necesitaba cerrar esta parte con más fuerza. El hecho de que un fold concreto salga bien no convierte automáticamente al modelo en candidato real. La promoción exige:

\begin{enumerate}[leftmargin=2em]
    \item mejora neta frente a baselines relevantes;
    \item estabilidad inter-fold e inter-periodo;
    \item cobertura suficiente del universo objetivo;
    \item ausencia de señales evidentes de fragilidad por régimen;
    \item compatibilidad con calibración, ensemble y capa de ejecución.
\end{enumerate}

Si un modelo ofrece gran score medio pero comportamiento errático, alta sensibilidad a hiperparámetros o dependencia excesiva de un subperiodo, debe permanecer en research.

\section{Validaciones obligatorias}

Antes de dar el módulo por correcto, deben superarse como mínimo las siguientes pruebas:

\subsection{Integridad}
\begin{itemize}[leftmargin=2em]
    \item unicidad de \texttt{(date, symbol)};
    \item ausencia de desalineaciones entre features y labels;
    \item fechas ordenadas y compatibles con los splits.
\end{itemize}

\subsection{Sanidad de features}
\begin{itemize}[leftmargin=2em]
    \item no hay columnas completamente nulas;
    \item no hay columnas constantes no justificadas;
    \item la lista de features es idéntica en todos los folds.
\end{itemize}

\subsection{Sanidad de outputs}
\begin{itemize}[leftmargin=2em]
    \item número de predicciones consistente con coverage esperado;
    \item ausencia de NaNs o infinitos en scores;
    \item metadatos completos en archivos persistidos.
\end{itemize}

\subsection{Validez experimental}
\begin{itemize}[leftmargin=2em]
    \item ninguna métrica de test intervino en tuning o early stopping;
    \item embargo y purging fueron efectivamente respetados;
    \item el run puede reconstruirse a partir del manifest.
\end{itemize}

\section{Definición de Done}

El módulo \texttt{train\_lgbm.py} solo se considera terminado si:

\begin{enumerate}[leftmargin=2em]
    \item entrena LightGBM de forma temporalmente válida;
    \item produce predicciones OOF y OOS trazables;
    \item calcula métricas consistentes con el problema económico;
    \item persiste artefactos y metadatos suficientes para auditoría;
    \item aplica gates explícitos de aceptación;
    \item y deja una salida apta para calibración, ensemble o inferencia posterior.
\end{enumerate}

En consecuencia, \texttt{train\_lgbm.py} no debe interpretarse como un simple wrapper de LightGBM. Es el módulo que convierte una hipótesis de señal tabular en un predictor cross-sectional evaluado con disciplina temporal, gobernanza experimental y criterios explícitos de promoción dentro de un stack cuantitativo serio.
