Modulo: models/ensemble/stacking.py

1) Objetivo
Entrenar un meta-modelo que combine predicciones de modelos base para mejorar performance OOS y estabilidad temporal frente a cualquier modelo individual.

2) Alcance
Incluye:
- Construccion de matriz de meta-features a partir de OOF de modelos base.
- Entrenamiento de meta-modelo con CV temporal purged.
- Generacion de predicciones OOF/OOS del ensemble.
- Comparativa contra best-single-model y blender.

No incluye:
- Entrenamiento de modelos base.
- Orquestacion de deployment final en produccion.

3) Dependencias aguas arriba y aguas abajo
Upstream:
- outputs de diversity_metrics (candidate_sets)
- predicciones OOF/OOS de modelos base
- labels y splits purged
- config de stacking

Downstream:
- models/inference/predict.py (ensemble model_ref)
- backtest/run_walkforward.py
- validation/validation_suite.py

4) Contrato de entrada
Entradas minimas:
- base_oof_predictions_path
- base_oos_predictions_path (opcional en entrenamiento)
- labels_path
- splits_path
- stacking_config_path
- run_id

Opcionales:
- regime_features_path
- sample_weights_path
- candidate_set_ref

Campos minimos esperados:
- date, symbol
- model_id
- y_pred_base
- y_true
- fold_id/split_role

5) Contrato de salida
- models/artifacts/ensemble/stacking/{run_id}/meta_model.pkl (o json)
- models/predictions/ensemble/stacking/{run_id}/oof.parquet
- models/predictions/ensemble/stacking/{run_id}/oos.parquet
- models/reports/ensemble/stacking/{run_id}/metrics_summary.json
- models/reports/ensemble/stacking/{run_id}/meta_feature_importance.parquet
- models/reports/ensemble/stacking/{run_id}/manifest.json

Campos minimos en predicciones:
- date
- symbol
- y_true
- y_pred_stack
- split_role
- run_id

6) Regla anti-leakage fundamental
- El meta-modelo solo se entrena con predicciones OOF de nivel 0.
- Nunca usar predicciones in-sample de base models para fit del meta.
- Test OOS queda fuera del ajuste del meta.

7) Disenos de meta-modelo soportados
- lineal regularizado (ridge/lasso)
- logistic/regression simple
- GBDT ligero (con regularizacion fuerte)

Recomendacion default:
- ridge lineal por robustez e interpretabilidad.

8) Features del meta-modelo
Base:
- columna por cada modelo base (score/pred).

Opcionales:
- interacciones simples entre modelos.
- indicador de regimen.
- incertidumbre de base models (si disponible).

9) Flujo del proceso
1. Cargar candidate set y predicciones base OOF.
2. Pivotear panel a matriz meta-features por (date,symbol).
3. Alinear con y_true y splits temporales.
4. Entrenar meta por folds (purged).
5. Generar OOF del stack.
6. Entrenar version final para inferencia segun policy.
7. Evaluar OOS y persistir artefactos.

10) Validaciones minimas
- cobertura suficiente de todas las meta-features.
- sin leakage entre train/valid/test en meta-level.
- sin NaN/inf en matriz meta.
- mejora o no degradacion fuerte vs baseline.

11) Metricas recomendadas
- metrica principal OOS (IC/RMSE/AUC segun task).
- overfit_gap stack (train-valid).
- estabilidad por fold.
- improvement_vs_best_base.
- drawdown/turnover proxy downstream.

12) Gates sugeridos
- Gate 1: improvement_vs_best_base >= umbral.
- Gate 2: estabilidad folds <= umbral dispersion.
- Gate 3: sin leakage flags.
- Gate 4: complejidad meta controlada (evitar overfit).

13) Manejo de casos especiales
- base model faltante en ciertas fechas:
  - policy: imputar/descartar fila/modelo.
- alta colinealidad de meta-features:
  - regularizacion y/o pruning por diversity.
- pocos datos OOF:
  - fallback a blender.

14) Observabilidad y trazabilidad
Logging:
- INFO: progreso por fold y mejor config meta.
- WARNING: cobertura baja o colinealidad extrema.
- ERROR: inconsistencia OOF/splits.

Metricas operativas:
- n_base_models_used
- n_rows_meta_train
- n_rows_meta_valid
- pct_rows_dropped
- runtime_sec

15) Interfaz de ejecucion (CLI sugerida)

  --splits-path labels/splits/purged_splits_latest.parquet ^
  --config-path configs/models/stacking.yaml ^
  --run-id stacking_20260305_001

17) Criterios de aceptacion (Definition of Done)
- stacking OOF/OOS reproducible y sin leakage.
- mejora robusta vs mejores modelos individuales o justificacion de fallback.
- artefactos listos para model_registry e inference.

18) Casos de prueba minimos (para tests/)
- pipeline meta-features con 2 modelos base sinteticos.
- verificacion de no uso de in-sample base preds.
- fallback a blender cuando meta data insuficiente.
- comparativa improvement_vs_best_base correcta.

19) Riesgos y decisiones abiertas
- tipo de meta-modelo final (lineal vs no lineal).
- frecuencia de refit del meta.
- uso de features de regimen en meta-level.
