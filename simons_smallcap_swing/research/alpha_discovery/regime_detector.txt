Módulo: models/regimes/regime_detector.py

1. Propósito del módulo

regime_detector.py segmenta el mercado en una secuencia temporal de regímenes
r_t ∈ {1,…,K} usando proxies de mercado observables y point-in-time (PIT).

El rol del módulo NO es asignar pesos, hacer gating final, ni optimizar cartera.
Su rol es más preciso:

- producir etiquetas de régimen reproducibles por fecha;
- cuantificar su persistencia y estabilidad;
- permitir evaluación condicional de señales/alphas por régimen;
- dejar una base auditable para análisis downstream.

Debe responder a dos preguntas:

(1) ¿qué estado de mercado describe mejor la fecha t?
(2) ¿cambia la calidad de una señal cuando condicionamos por ese estado?

2. Qué hace y qué no hace

2.1 Qué hace
- consume una serie temporal de proxies de mercado z_t;
- aplica una metodología explícita de segmentación:
  threshold rules, GMM o HMM;
- genera una etiqueta r_t por fecha;
- persiste probabilidad/confianza si la metodología lo soporta;
- calcula estadísticas de duración y transición;
- opcionalmente calcula métricas condicionales de alphas por régimen.

2.2 Qué no hace
- no decide directamente si un alpha debe activarse o desactivarse;
- no reasigna capital ni modifica gross/net;
- no sustituye un módulo de regime_gating;
- no debe usar información futura para etiquetar fechas live;
- no debe convertirse en un laboratorio de modelos de series temporales complejos.

3. Definición formal del problema

Sea z_t ∈ ℝ^p el vector de proxies de mercado observables en la fecha t.
El módulo produce una secuencia:

    r_t ∈ {1,...,K}

y, cuando aplica, una distribución de probabilidad:

    p_t(k) = P(r_t = k | 𝓘_t)

donde 𝓘_t es la información disponible hasta t.

Una vez obtenidos los regímenes, para cualquier alpha α se define el IC diario:

    IC_t(α) = ρ_S( α_{t,·}, y_{t,·} )

y su media condicional por régimen:

    IC̄^(r)(α) = (1 / T_r) Σ_{t : r_t = r} IC_t(α)

donde T_r es el número de fechas asignadas al régimen r.

4. Jerarquía metodológica

El módulo soporta varias metodologías, pero no deben tratarse como intercambiables.
Debe existir una jerarquía explícita.

4.1 Nivel 1 — baseline interpretable
Método: threshold-based deterministic labeling.

Uso correcto:
- baseline productivo inicial;
- máxima interpretabilidad;
- validación interna;
- semántica de regímenes totalmente controlada.

Ejemplo:

    r_t =
      high_vol        if vix_t > c_1
      down_trend      if ret_mkt_20d < c_2
      calm_uptrend    if vix_t ≤ c_1 and ret_mkt_20d ≥ c_3
      neutral         otherwise

Ventajas:
- 100% interpretable
- 100% PIT si los proxies lo son
- reproducibilidad exacta
- robustez operacional

4.2 Nivel 2 — clustering exploratorio
Método: Gaussian Mixture Model (GMM).

Definición:

    p(z_t) = Σ_{k=1}^K π_k N(z_t ; μ_k, Σ_k)

Asignación por MAP:

    r_t = argmax_k P(r_t = k | z_t)

Uso correcto:
- research exploratorio offline;
- detectar agrupaciones empíricas de proxies;
- comparar si la estructura latente coincide con reglas threshold.

Limitación:
- no modela persistencia secuencial explícita;
- semántica de clusters puede rotar entre runs.

4.3 Nivel 3 — segmentación secuencial latente
Método: Hidden Markov Model (HMM).

Definición:

    s_t ∈ {1,...,K}
    P(s_t = j | s_{t-1} = i) = A_ij
    z_t | s_t = k ~ N(μ_k, Σ_k)

El output causal correcto para uso live es:

    p_t(k) = P(s_t = k | z_1,...,z_t)

y NO:

    P(s_t = k | z_1,...,z_T)

porque esta última usa smoothing retrospectivo y mete look-ahead.

Uso correcto:
- si se desea persistencia temporal explícita;
- si los regímenes son latentes y no sólo reglas visibles;
- si se acepta más complejidad a cambio de una estructura dinámica mejor.

5. Principio central PIT

La regla más importante del módulo es:

    si la etiqueta r_t va a usarse downstream en una fecha t,
    entonces debe depender sólo de información disponible hasta t.

En particular:

- threshold rules: triviales si z_t es PIT;
- GMM: si se reentrena rolling y luego se asigna sólo con z_t, es aceptable;
- HMM: usar filtering P(s_t | z_1:t), no smoothing P(s_t | z_1:T).

6. Construcción de proxies

Sea:

    z_t = (z_{1,t}, ..., z_{p,t})

Los proxies pueden incluir, por ejemplo:
- VIX o proxy equivalente de estrés;
- retorno benchmark 20d / 60d;
- volatilidad realizada 20d;
- breadth / pct above MA;
- realized correlation proxy;
- amplitud de spreads o liquidez agregada;
- corr concentration / turbulence proxies.

Cada proxy debe tener:
- definición exacta;
- unidad;
- ventana lookback;
- política de missing;
- garantía PIT.

7. Normalización temporal

Si se requiere normalización, debe ser rolling o expanding causal.

Ejemplo de z-score rolling:

    z̃_{j,t} = ( z_{j,t} - μ_{j,t}^{(W)} ) / ( σ_{j,t}^{(W)} + ε )

donde μ_{j,t}^{(W)} y σ_{j,t}^{(W)} usan sólo datos ≤ t.

Nunca usar:
- z-score global ex post;
- PCA entrenado con toda la muestra si luego se usa para etiquetar live;
- winsorización con cuantiles calculados usando futuro.

8. Métodos concretos

8.1 Threshold rules

Contrato:
- input: proxies PIT
- output: regime_id exacto por fecha
- optional: confidence heuristic if desired

Ventajas:
- baseline canónico
- mejor opción si el objetivo principal es interpretabilidad

MVP:
- rules versionadas en config
- precedence entre reglas
- bucket “unknown/neutral” explícito

8.2 GMM

Modelo:

    p(z_t) = Σ_{k=1}^K π_k N(z_t; μ_k, Σ_k)

Selección de K:
- BIC por defecto;
- AIC como diagnóstico secundario;
- no elegir K sólo por intuición visual.

Conteo de parámetros para covariance_type='full':

    d(K) = (K - 1)                  # mixture weights
           + K p                    # means
           + K p(p+1)/2             # covariances

BIC:

    BIC(K) = -2 logL_K + d(K) log(T)

AIC:

    AIC(K) = -2 logL_K + 2 d(K)

Regla recomendada:
- usar BIC como selector principal;
- si BIC y AIC divergen, persistir ambos y emitir warning.

8.3 HMM

Modelo gaussiano:

    s_t ~ Markov(A)
    z_t | s_t = k ~ N(μ_k, Σ_k)

Parámetros:

    θ = {π, A, μ_1,...,μ_K, Σ_1,...,Σ_K}

Likelihood:

    L(θ) = P(z_1,...,z_T | θ)

Estimación: EM / Baum-Welch.

Inferencia:
- filtering: α_t(k) ∝ P(z_t | s_t=k) Σ_j α_{t-1}(j) A_{jk}
- posterior filtered:
    p_t(k) = P(s_t=k | z_1:t)

Etiquetado causal:

    r_t = argmax_k p_t(k)

No usar por defecto para live:
- Viterbi path global si usa toda la muestra;
- smoothing posterior con datos futuros.

9. Selección de K

9.1 Threshold method
K viene impuesto por diseño de reglas.

9.2 GMM/HMM
K debe seleccionarse con:
- BIC primario;
- AIC secundario;
- filtros de estabilidad y viabilidad.

No basta con minimizar BIC.
También deben imponerse:

- min_dates_per_regime
- min_mean_duration
- max_ephemeral_fraction
- no estados degenerados

Score recomendado:

    score(K) = -BIC_norm(K)
               + λ_dur * duration_score(K)
               + λ_occ * occupancy_score(K)
               - λ_deg * degeneracy_score(K)

MVP:
- elegir K por BIC entre candidatos viables;
- si el mejor K por BIC produce un estado degenerado, bajar al siguiente viable.

10. Persistencia y duración de regímenes

Sea una racha consecutiva del régimen r de longitud L_{r,m}.
Persistir por régimen:

- n_blocks_r
- mean_duration_r
- median_duration_r
- p10/p50/p90 duration
- occupancy_r = T_r / T
- transitions_out_r

Si mean_duration_r o median_duration_r es demasiado baja,
la segmentación puede ser demasiado ruidosa para uso downstream.

11. Matriz de transición

Para HMM:
- persistir A estimada;
- persistir también una matriz empírica basada en r_t.

Para métodos no secuenciales:
- persistir transición empírica simple entre etiquetas consecutivas.

Campos mínimos:
- A_ij model
- A_ij empirical
- self_transition_i
- expected_duration_i ≈ 1 / (1 - A_ii)  [si HMM y A_ii<1]

12. Estabilidad semántica de estados

Problema:
en GMM/HMM el label “estado 1” puede no significar lo mismo entre runs.

Solución:
mapear estados latentes a nombres semánticos usando una convención estable.

Ejemplo de ordenación:
- ordenar estados por media de volatilidad proxy;
- desempatar por retorno benchmark medio;
- o construir vector resumen por estado:

    m_k = [E[VIX|k], E[ret_20d|k], E[rv_20d|k]]

y alinear estados entre runs minimizando distancia.

Coste sugerido:

    C_ij = w_vol |m_i(vol)-m_j(vol)|
         + w_ret |m_i(ret)-m_j(ret)|
         + w_rv  |m_i(rv)-m_j(rv)|

Luego resolver matching con Hungarian algorithm.

Esto no es obligatorio en threshold rules, pero sí muy recomendable en GMM/HMM.

13. Evaluación condicional de alphas

Para cada alpha α:

    IC_t(α) = ρ_S( α_{t,·}, y_{t,·} )

y por régimen:

    IC̄^(r)(α) = (1 / T_r) Σ_{t:r_t=r} IC_t(α)

Persistir además:
- std_ic_cond^(r)
- hit_rate_cond^(r) = mean(1{IC_t>0} | r_t=r)
- n_dates_cond^(r)
- coverage_cond^(r)
- delta_ic_vs_uncond^(r) = IC̄^(r) - IC̄

No interpretar regímenes con T_r demasiado pequeño.

14. Gates mínimos de validez del módulo

FAIL si cualquiera:
- proxies no PIT
- normalización usa futuro
- K seleccionado produce estados con n_dates < min_dates_per_regime
- HMM etiquetado con smoothing en modo live
- semántica de estados no resoluble / no persistida

WARN si:
- AIC y BIC discrepan de forma material
- un régimen está cerca del umbral mínimo de fechas
- mean_duration demasiado baja
- confidence posterior demasiado difusa
- mapping semántico inestable entre runs

PASS:
- segmentación PIT válida
- soporte por régimen suficiente
- estabilidad mínima cumplida
- outputs completos persistidos

1. cargar proxies
2. validar PIT y cobertura
3. normalizar causalmente
4. elegir metodología
5. si aplica, seleccionar K entre candidatos
6. ajustar modelo
7. etiquetar fechas de forma causal
8. asignar nombres semánticos estables
9. calcular duración, ocupación y transiciones
10. calcular estadísticas condicionales de alphas
11. persistir artefactos y diagnósticos

20. Invariantes matemáticos y operativos

20.1 Etiquetado válido
    r_t ∈ {1,...,K}  para todo t válido

20.2 PIT
    r_t depende sólo de información ≤ t

20.3 Probabilidades válidas
    Σ_k p_t(k) = 1  para todo t  si el método es probabilístico

20.4 Viabilidad de estados
    T_r ≥ min_dates_per_regime

20.5 Persistencia
    duration_stats disponibles para todo régimen

20.6 Reproducibilidad
    mismo snapshot + misma config + misma seed → mismos labels

21. Failure modes principales

- proxies con leakage por estandarización global;
- estados latentes degenerados con muy pocas fechas;
- switching excesivo entre regímenes;
- HMM mal usado con smoothing retrospectivo;
- semántica de estados inestable entre runs;
- K demasiado grande para T disponible;
- confundir clustering exploratorio con segmentación productiva.

22. Qué NO conviene introducir

No conviene meter aquí:
- MS-VAR;
- regime-switching GARCH;
- HDP-HMM / Bayesian nonparametrics;
- deep Markov models;
- copulas dinámicas;
- gating policies aprendidas;
- RL para asignar regímenes.

Todo eso cambia el rol del módulo.
Este módulo debe seguir siendo una capa de segmentación temporal interpretable y utilizable.

23. MVP vs versión avanzada

23.1 MVP
- threshold rules
- GMM con selección BIC
- HMM gaussiano con etiquetado online por filtering
- duración / ocupación / transición
- evaluación condicional de IC
- semántica estable básica de estados

23.2 Avanzado
- alineación de estados entre runs con Hungarian
- confidence / entropy diagnostics
- score compuesto BIC + viabilidad
- importance de proxies por separación entre estados
- monitoring de drift de proxies y de labels de régimen

24. Definition of Done

El módulo se considera terminado cuando:
- genera una etiqueta de régimen por fecha con garantía PIT;
- persiste resumen de ocupación, duración y transiciones;
- soporta threshold, GMM y HMM bajo un contrato común;
- en HMM usa filtering online para live labeling;
- en GMM/HMM selecciona K con criterio explícito;
- soporta evaluación condicional de alphas por régimen;
- deja artefactos reutilizables por módulos downstream.

25. Conclusión

regime_detector.py debe ser leído como un módulo de segmentación temporal,
no como un generador de narrativa ex post.

Su valor real no está en decir “el mercado estaba risk-off”,
sino en producir una partición temporal reproducible, causal y suficientemente estable
como para que el resto del stack pueda contrastar hipótesis condicionales con disciplina.

La implementación correcta aquí no es la más sofisticada,
sino la que preserve simultáneamente:
- causalidad temporal,
- semántica estable,
- soporte estadístico suficiente,
- y utilidad downstream real.

26. Base teórica mínima recomendada

- Hamilton (1989): Markov switching / regime models en macro-finance
- Rabiner (1989): forward-backward, filtering, Viterbi
- Schwarz (1978): BIC para selección de modelo
- Bishop (2006): mixture models y EM
- López de Prado (2018): disciplina PIT y evaluación temporal en research financiero

27. Técnica clave realmente adecuada aquí

Online HMM filtering es esencial aquí porque:
- elimina el look-ahead de usar smoothing P(s_t|z_1:T);
- mantiene el módulo usable en tiempo real;
- y conserva casi toda la utilidad de una segmentación secuencial
  sin convertir el sistema en un research toy de modelos latentes más complejos.
