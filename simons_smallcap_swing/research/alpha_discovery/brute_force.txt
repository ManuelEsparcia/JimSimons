\section{Propósito del módulo}

El objetivo de este módulo es generar, evaluar y filtrar de manera sistemática un conjunto amplio de candidatos de alpha construidos a partir de una gramática controlada de operadores, features y parámetros. Su función no es ``descubrir automáticamente el alpha definitivo'', sino actuar como \textbf{motor de exploración exhaustiva bajo restricciones estadísticas, computacionales y de gobernanza}.

En un pipeline cuantitativo serio, la exploración combinatoria tiene valor únicamente si se ejecuta con tres principios simultáneos:

\begin{enumerate}[leftmargin=1.5em]
    \item \textbf{disciplina point-in-time}: ninguna fórmula puede depender de información no disponible en el instante de decisión;
    \item \textbf{validación fuera de muestra}: toda métrica debe estimarse en un esquema temporal reproducible;
    \item \textbf{control de proliferación espuria}: el espacio de búsqueda, al crecer, aumenta mecánicamente la probabilidad de falsos positivos.
\end{enumerate}

Por tanto, este módulo debe entenderse como una \textbf{capa de proposal generation con evaluación proxy}, no como una certificación final de explotabilidad económica.

\section{Definición formal del espacio de búsqueda}

Sea \(\mathcal{X}_t(x)\) el conjunto de variables \emph{point-in-time} observables para el activo \(x\) en el instante \(t\). Definimos el espacio de candidatos como

\[
\mathcal{A} = \left\{ \alpha_j(x,t) = f_j\!\big(\mathcal{X}_t(x)\big) \; : \; f_j \in \mathcal{G},\; j=1,\dots,M \right\},
\]

donde \(\mathcal{G}\) es una gramática finita de composición de operadores válidos y \(M = |\mathcal{A}|\) es el número total de fórmulas generadas tras aplicar restricciones sintácticas y semánticas.

\subsection{Gramática}

La gramática debe componerse de:

\begin{itemize}[leftmargin=1.5em]
    \item \textbf{terminales}: precios, retornos, volumen, volatilidad, fundamentales PIT, métricas de liquidez, etc.;
    \item \textbf{operadores unarios}: \texttt{zscore}, \texttt{rank}, \texttt{abs}, \texttt{sign}, \texttt{log1p}, \texttt{ts\_mean}, \texttt{ts\_std}, \texttt{ts\_rank};
    \item \textbf{operadores binarios}: \(+\), \(-\), \(*\), \(/\), \(\min\), \(\max\), spreads, ratios;
    \item \textbf{parámetros discretos}: ventanas temporales, retrasos, percentiles, intensidades de winsorización.
\end{itemize}

El conjunto admisible no debe ser abierto. Toda expansión debe ocurrir sobre un \textbf{vocabulario declarado en configuración}, de modo que el espacio de fórmulas sea reproducible y auditable.

\subsection{Restricciones estructurales}

Para evitar explosión combinatoria no informativa, cada candidato debe respetar límites explícitos:

\begin{itemize}[leftmargin=1.5em]
    \item profundidad máxima del árbol sintáctico;
    \item número máximo de nodos;
    \item número máximo de features base por expresión;
    \item prohibición de composiciones redundantes o degeneradas;
    \item restricciones de dominio para operadores no definidos en todo \(\mathbb{R}\).
\end{itemize}

Ejemplos de fórmulas que deben ser descartadas antes de evaluación:

\begin{itemize}[leftmargin=1.5em]
    \item \(\texttt{rank(rank(x))}\) si la segunda aplicación no altera la semántica;
    \item \(\texttt{zscore(constant)}\);
    \item \(\texttt{x / x}\) fuera de un tratamiento explícito de ceros;
    \item fórmulas equivalentes por reordenación algebraica que ya hayan sido evaluadas.
\end{itemize}

\section{Rol del módulo dentro del pipeline}

\texttt{brute\_force} se sitúa entre el feature store y los módulos de validación avanzada. Su responsabilidad es:

\begin{enumerate}[leftmargin=1.5em]
    \item generar candidatos sintácticamente válidos;
    \item canonicalizar cada fórmula;
    \item deduplicar expresiones equivalentes;
    \item calcular una evaluación proxy fuera de muestra;
    \item registrar cobertura, estabilidad y flags de integridad;
    \item devolver un \emph{shortlist} priorizado para revisión posterior.
\end{enumerate}

No debe:

\begin{enumerate}[leftmargin=1.5em]
    \item ejecutar un backtest completo de cartera;
    \item optimizar pesos finales;
    \item inferir significancia económica definitiva;
    \item promover automáticamente un alpha a producción.
\end{enumerate}

\section{Representación canónica de candidatos}

Toda fórmula generada debe transformarse a una representación canónica \(\mathcal{C}(\alpha)\) antes de su evaluación. Esa representación sirve para:

\begin{itemize}[leftmargin=1.5em]
    \item detectar duplicados exactos;
    \item colapsar equivalencias sintácticas triviales;
    \item construir un \texttt{formula\_hash} estable;
    \item reutilizar cómputos intermedios en caché.
\end{itemize}

Formalmente, si \(\alpha\) es una expresión simbólica, definimos

\[
h(\alpha) = H\!\big(\mathrm{serialize}(\mathcal{C}(\alpha))\big),
\]

donde \(H\) es una función hash determinista y \(\mathcal{C}\) impone reglas como:

\begin{itemize}[leftmargin=1.5em]
    \item orden lexicográfico en operadores conmutativos;
    \item reducción de identidades algebraicas triviales;
    \item normalización de nombres de parámetros;
    \item serialización estable del árbol.
\end{itemize}

Sin esta capa, una parte relevante del presupuesto computacional se desperdicia reevaluando fórmulas esencialmente idénticas.

\section{Métrica primaria de evaluación}

La métrica primaria debe ser una medida de capacidad ordinal fuera de muestra, preferiblemente el \textbf{Information Coefficient cross-sectional} estimado por fecha y agregado temporalmente:

\[
IC_d^{(j)} = \rho_S\!\big(\alpha_{d,\cdot}^{(j)}, y_{d,\cdot}\big),
\]

donde:
\begin{itemize}[leftmargin=1.5em]
    \item \(d\) indexa fechas,
    \item \(\alpha_{d,\cdot}^{(j)}\) es el vector cross-sectional del candidato \(j\) en la fecha \(d\),
    \item \(y_{d,\cdot}\) es el vector cross-sectional del label futuro,
    \item \(\rho_S\) es la correlación de Spearman.
\end{itemize}

La agregación temporal recomendada es:

\[
\overline{IC}^{(j)} = \frac{1}{D_j}\sum_{d=1}^{D_j} IC_d^{(j)},
\qquad
\sigma_{IC}^{(j)} = \sqrt{\frac{1}{D_j-1}\sum_{d=1}^{D_j}\left(IC_d^{(j)}-\overline{IC}^{(j)}\right)^2},
\]

y adicionalmente:

\[
h^{(j)} = \frac{1}{D_j}\sum_{d=1}^{D_j}\mathbf{1}_{\{IC_d^{(j)} > 0\}},
\]

donde \(h^{(j)}\) es el \emph{hit rate} del signo del IC.

\subsection{Por qué una sola métrica primaria}

El módulo debe evitar combinar prematuramente IC, Sharpe, retorno anualizado y otras magnitudes heterogéneas en una sola puntuación opaca. En esta fase, el objetivo es priorizar candidatos con \textbf{capacidad de ordenación cross-sectional robusta}. Métricas de portafolio pueden añadirse más adelante, pero no deben contaminar el ranking base del generador.

\section{Esquema de validación temporal}

Toda evaluación debe realizarse en \textbf{walk-forward purgado}, con embargo cuando el label lo requiera. Si los folds se indexan por \(k = 1,\dots,K\), entonces para cada fold:

\begin{itemize}[leftmargin=1.5em]
    \item entrenamiento en \([t_0, t_k^{\mathrm{train}}]\),
    \item embargo de longitud \(e\),
    \item test en \([t_k^{\mathrm{test,start}}, t_k^{\mathrm{test,end}}]\).
\end{itemize}

Cada candidato produce una secuencia de métricas fuera de muestra por fold:
\[
\overline{IC}^{(j)}_1,\dots,\overline{IC}^{(j)}_K.
\]

La agregación final debe conservar tanto media como dispersión inter-fold:

\[
\mu^{(j)}_{\mathrm{OOS}} = \frac{1}{K}\sum_{k=1}^K \overline{IC}^{(j)}_k,
\qquad
\sigma^{(j)}_{\mathrm{fold}} = \sqrt{\frac{1}{K-1}\sum_{k=1}^K \left(\overline{IC}^{(j)}_k - \mu^{(j)}_{\mathrm{OOS}}\right)^2}.
\]

Un candidato con media elevada pero dispersión extrema entre folds debe tratarse con cautela y no ascender automáticamente en el ranking.

\section{Cobertura y viabilidad de evaluación}

No toda fórmula sintácticamente válida es estadísticamente evaluable. Deben imponerse, como mínimo, las siguientes condiciones:

\begin{enumerate}[leftmargin=1.5em]
    \item \textbf{coverage mínima}: proporción suficiente de observaciones no nulas;
    \item \textbf{mínimo cross-sectional por fecha}: evitar ICs calculados sobre universos demasiado pequeños;
    \item \textbf{mínimo número de fechas válidas}: el candidato debe generar historia suficiente;
    \item \textbf{dominio numérico estable}: no explosiones por divisiones cercanas a cero, logs inválidos, etc.
\end{enumerate}

La cobertura global puede definirse como

\[
c^{(j)} = \frac{N^{(j)}_{\mathrm{valid}}}{N_{\mathrm{eligible}}},
\]

y debe persistirse junto con el motivo de descarte si \(c^{(j)} < c_{\min}\).

\section{Penalización por complejidad}

El espacio de fórmulas contiene un riesgo natural de \emph{overfitting simbólico}: expresiones más complejas pueden ajustar artefactos del dataset sin mejorar la estructura predictiva subyacente. Por ello, además de rankear por métrica primaria, el módulo debe registrar una medida explícita de complejidad.

Una definición mínima razonable es:

\[
\kappa^{(j)} = w_n N^{(j)}_{\mathrm{nodes}} + w_d D^{(j)}_{\mathrm{depth}} + w_p P^{(j)}_{\mathrm{params}},
\]

donde:
\begin{itemize}[leftmargin=1.5em]
    \item \(N^{(j)}_{\mathrm{nodes}}\) es el número de nodos del árbol;
    \item \(D^{(j)}_{\mathrm{depth}}\) es la profundidad máxima;
    \item \(P^{(j)}_{\mathrm{params}}\) es el número de parámetros libres discretizados.
\end{itemize}

La complejidad no debe anular una buena señal, pero sí servir para desempatar candidatos de rendimiento parecido y para filtrar expresiones innecesariamente barrocas.

\section{Dependencia entre candidatos y multiple testing}

La exploración masiva genera un problema central: muchos candidatos comparten inputs, operadores y estructura, por lo que sus tests no son independientes. Por ello, un tratamiento ingenuo de \(M\) contrastes como si fueran i.i.d. es metodológicamente pobre.

Este módulo debe distinguir dos niveles:

\subsection{Nivel 1: ranking operacional}

Para la priorización interna del \emph{shortlist}, basta con ordenar candidatos por métrica proxy, estabilidad y cobertura, después de deduplicación y filtros de complejidad.

\subsection{Nivel 2: señal estadística ajustada}

Para reportes y auditoría, debe almacenarse una corrección por multiplicidad. Se recomienda:

\begin{itemize}[leftmargin=1.5em]
    \item \textbf{Benjamini--Hochberg} sobre familias homogéneas de candidatos;
    \item opcionalmente, \textbf{Benjamini--Yekutieli} cuando se desee una corrección robusta bajo dependencia arbitraria;
    \item reporte del número efectivo de candidatos únicos tras deduplicación y colapso de equivalencias.
\end{itemize}

El objetivo no es convertir el módulo en una máquina de inferencia perfecta, sino impedir que el ranking final ignore por completo el coste estadístico de haber explorado miles de fórmulas.

\section{Redundancia y clustering de shortlist}

Incluso tras deduplicación simbólica, muchos candidatos supervivientes pueden representar la misma intuición económica con ligeras variaciones paramétricas. Por ello, antes de devolver el \emph{shortlist}, debe aplicarse una reducción de redundancia.

Sea \(v^{(j)}\) el vector temporal de valores del candidato \(j\). Puede definirse una distancia funcional entre candidatos como:

\[
d(i,j) = 1 - |\mathrm{corr}(v^{(i)}, v^{(j)})|.
\]

Sobre esta base, el módulo puede:

\begin{itemize}[leftmargin=1.5em]
    \item agrupar candidatos altamente correlacionados;
    \item retener el representante más simple o mejor rankeado de cada cluster;
    \item evitar que el top-\(N\) quede monopolizado por variaciones triviales de la misma fórmula.
\end{itemize}

Este paso es esencial. Un buen motor de discovery no debe devolver cien veces la misma idea con distinta ventana.

\section{Caché y reutilización computacional}

Dado que numerosas expresiones comparten subárboles, el módulo debe implementar \textbf{memoization} de subexpresiones. Si \(g(x,t)\) aparece en múltiples candidatos, su evaluación PIT no debe recomputarse de cero en cada caso.

Conceptualmente, si
\[
\alpha_1 = \mathrm{rank}(g), \qquad \alpha_2 = \mathrm{zscore}(g),
\]
entonces \(g\) debe materializarse una sola vez y consumirse por ambos candidatos.

Sin esta política, el coste de búsqueda escala mucho más cerca del número bruto de nodos evaluados que del número de candidatos verdaderamente distintos.

\section{Validaciones obligatorias}

\subsection{Validaciones sintácticas}

\begin{enumerate}[leftmargin=1.5em]
    \item la fórmula debe parsear sin ambigüedad;
    \item todos los operadores deben pertenecer a la gramática autorizada;
    \item todos los parámetros deben estar dentro de rango;
    \item la representación canónica debe ser serializable.
\end{enumerate}

\subsection{Validaciones semánticas}

\begin{enumerate}[leftmargin=1.5em]
    \item todas las dependencias deben existir en el snapshot de datos;
    \item la fórmula no debe producir un valor constante;
    \item la cobertura debe superar el umbral mínimo;
    \item debe existir suficiente variación cross-sectional para calcular IC con sentido.
\end{enumerate}

\subsection{Validaciones PIT}

\begin{enumerate}[leftmargin=1.5em]
    \item ninguna feature puede mirar más allá de \(t\);
    \item toda variable fundamental debe respetar su timestamp de publicación;
    \item el universo de evaluación debe ser observable en el instante de decisión;
    \item el label no debe filtrarse directa ni indirectamente a la fórmula.
\end{enumerate}

\section{Flujo operacional}

El flujo recomendado del módulo es:

\[
\texttt{config}
\rightarrow
\texttt{generate}
\rightarrow
\texttt{canonicalize}
\rightarrow
\texttt{deduplicate}
\rightarrow
\texttt{evaluate\_OOS}
\rightarrow
\texttt{filter}
\rightarrow
\texttt{adjust}
\rightarrow
\texttt{cluster}
\rightarrow
\texttt{shortlist}
\rightarrow
\texttt{persist}.
\]

Desarrollado paso a paso:

\begin{enumerate}[leftmargin=1.5em]
    \item generar fórmulas dentro del presupuesto declarado;
    \item transformar cada una a forma canónica;
    \item eliminar duplicados exactos o equivalentes triviales;
    \item evaluar sólo los candidatos que superan validaciones básicas;
    \item calcular métricas OOS por fold;
    \item filtrar por cobertura, estabilidad y calidad mínima;
    \item aplicar corrección por multiple testing a nivel de familia;
    \item agrupar candidatos redundantes;
    \item devolver shortlist final y logs completos.
\end{enumerate}

\section{Criterios de calidad del módulo}

\texttt{brute\_force} estará bien implementado no cuando produzca muchas fórmulas, sino cuando produzca \textbf{candidatos útiles y auditables}. Como mínimo debe cumplir:

\begin{enumerate}[leftmargin=1.5em]
    \item toda fórmula generada tiene representación canónica y \texttt{formula\_hash};
    \item toda evaluación usa un protocolo temporal reproducible;
    \item todo descarte deja un motivo estructurado;
    \item la deduplicación reduce de forma sustantiva el número bruto de candidatos;
    \item el shortlist final contiene diversidad económica real, no sólo variantes paramétricas triviales;
    \item la ejecución es reproducible dado el mismo snapshot y configuración;
    \item el coste computacional está acotado por presupuesto y caché efectiva.
\end{enumerate}

\section{Definition of Done}

El módulo se considera completado cuando:

\begin{itemize}[leftmargin=1.5em]
    \item genera candidatos desde una gramática declarativa configurable;
    \item aplica canonicalización y deduplicación antes de evaluación intensiva;
    \item evalúa candidatos con walk-forward purgado y embargo;
    \item persiste IC OOS, cobertura, estabilidad y flags por candidato;
    \item ajusta por multiplicidad con un método explícito;
    \item agrupa candidatos redundantes antes del shortlist final;
    \item produce artefactos reproducibles para consumo de \texttt{alpha\_library} y validación posterior.
\end{itemize}

\section{Conclusión}

La exploración combinatoria de señales sólo aporta valor cuando se trata como un problema de búsqueda restringida bajo incertidumbre, y no como una competición por extraer el mayor número posible de métricas atractivas de un universo de fórmulas. El papel de \texttt{brute\_force} es convertir una gramática de hipótesis en un proceso ordenado de generación, evaluación y cribado, respetando simultáneamente restricciones PIT, validación temporal y costes de múltiples comparaciones.

En ese sentido, este módulo no debe vender promesas de rentabilidad. Debe entregar algo más valioso para el research serio: un subconjunto manejable de candidatos estadísticamente defendibles, computacionalmente trazables y conceptualmente diferenciados, listos para someterse a pruebas más exigentes en fases posteriores del pipeline.
