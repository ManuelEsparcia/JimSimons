Módulo: interaction_scan.py

1. Propósito del módulo

El objetivo de interaction_scan.py es detectar si combinaciones controladas de señales base
aportan información incremental real respecto a los mejores alphas individuales ya existentes.

La pregunta correcta no es:
“¿puedo construir una interacción no lineal con mejor métrica histórica?”

La pregunta correcta es:
“¿esta interacción aporta mejora fuera de muestra, estable, no trivialmente redundante
y suficientemente robusta como para merecer entrar en validación posterior?”

Este módulo es una capa de research screening.
No es un motor final de ensemble, no es un optimizer de cartera y no debe convertirse
en un buscador universal de no linealidades de alta capacidad.

2. Qué hace y qué no hace

2.1 Qué hace
- toma un conjunto ya filtrado de señales base PIT;
- genera interacciones dentro de una gramática restringida y auditable;
- evalúa cada interacción bajo el mismo protocolo temporal OOS que sus bases;
- mide mejora incremental frente al mejor componente individual;
- mide estabilidad por folds y subperiodos;
- controla redundancia funcional con las bases y con otras interacciones;
- corrige multiplicidad dentro de familias homogéneas;
- devuelve una shortlist trazable de interacciones candidatas.

2.2 Qué no hace
- no sustituye al training de modelos complejos;
- no debe usar árboles, redes o boosting como operadores de interacción por defecto;
- no debe optimizar directamente pesos de ensemble en producción;
- no certifica explotabilidad económica final;
- no resuelve causalidad entre señales;
- no debe crecer hasta convertirse en un módulo de AutoML.

3. Definición formal de interacción

Sea un conjunto de señales base elegibles:

    B = { alpha_1, ..., alpha_m }

donde cada señal base alpha_j(x,t) está disponible point-in-time para activo x en fecha t.

Una interacción candidata se define como:

    alpha_int(x,t) = g(alpha_i1(x,t), ..., alpha_ik(x,t); theta)

donde:
- g pertenece a una familia restringida de operadores admisibles;
- k es la cardinalidad de la interacción, típicamente k in {2,3};
- theta contiene parámetros discretos o de baja dimensión;
- todas las entradas deben existir PIT en la fecha t.

El módulo debe preservar una semántica clara:
la interacción es una transformación estructurada de señales base,
no un modelo de alta capacidad entrenado sobre labels.

4. Espacio de búsqueda

Sea G el conjunto de operadores permitidos y Theta_g el conjunto discreto de parámetros
admisibles de cada operador g. Entonces el espacio de candidatas es:

    I =
    {
      g(alpha_i1, ..., alpha_ik; theta) :
      g in G,
      (i1,...,ik) in C_k(B),
      theta in Theta_g,
      k <= k_max
    }

Este espacio debe ser explícitamente acotado por configuración.

5. Principio de evaluación

La validez de una interacción no debe medirse por su métrica absoluta aislada,
sino por su mejora incremental respecto al mejor alpha base que la compone,
bajo el mismo protocolo temporal OOS.

Sea M(.) la métrica primaria OOS. Entonces:

    DeltaM(alpha_int)
    =
    M(alpha_int)
    -
    max_{j in components(alpha_int)} M(alpha_j)

En este módulo, la métrica primaria por defecto debe ser el IC cross-sectional OOS.

6. Métrica primaria

Para una fecha d:

    IC_d(alpha) = rho_S(alpha_d, y_d)

donde:
- alpha_d es el vector cross-sectional de scores de la señal en d;
- y_d es el vector cross-sectional del label en d;
- rho_S es la correlación de Spearman.

La agregación primaria fuera de muestra es:

    IC_bar(alpha) = (1 / D) * sum_d IC_d(alpha)

Para una interacción candidata:

    DeltaIC
    =
    IC_bar(alpha_int)
    -
    max_j IC_bar(alpha_j)

donde el máximo se toma sobre sus componentes base.

La unidad estadística correcta para la comparación temporal no es la observación pooled
(date,symbol), sino la serie temporal de IC diario o los resúmenes por fold del mismo protocolo OOS.

7. Métrica incremental por fold

Si k indexa folds temporales:

    DeltaIC_k
    =
    IC_bar_k(alpha_int)
    -
    max_j IC_bar_k(alpha_j)

A partir de esta serie de mejoras por fold deben persistirse, como mínimo:

    mu_Delta = (1/K) * sum_k DeltaIC_k
    sigma_Delta = std_k(DeltaIC_k)
    h_Delta = (1/K) * sum_k 1{DeltaIC_k > 0}

Interpretación:
- mu_Delta mide mejora incremental media;
- sigma_Delta mide estabilidad entre folds;
- h_Delta mide frecuencia de mejora positiva.

8. Hipótesis nula correcta

La hipótesis nula natural es:

    H0: DeltaIC <= 0

La alternativa es:

    H1: DeltaIC > 0

El módulo no debe usar un t-test ingenuo sobre observaciones pooled.
La dependencia temporal y la dependencia entre señales base obligan a usar un procedimiento
de contraste más robusto y conservador.

9. Bootstrap temporal para DeltaIC

9.1 MVP recomendado
El procedimiento recomendado por defecto es bootstrap por bloques sobre la serie temporal
de contribuciones diarias o sobre la secuencia de IC_d, preservando estructura temporal.

Para una interacción candidata:

- calcular la serie diaria IC_d(alpha_int);
- calcular la serie diaria IC_d(alpha_j) de cada componente base;
- en cada réplica bootstrap, remuestrear bloques temporales de fechas;
- recomputar DeltaIC_boot;
- estimar:
    - p_value_boot
    - ci_lower_boot
    - ci_upper_boot

9.2 Razón metodológica
Este bootstrap es adecuado aquí porque:
- preserva la dependencia serial de primer orden de la señal;
- evita la falsa precisión de remuestreo i.i.d. por observación;
- mantiene comparable interacción y bases en la misma réplica temporal.

9.3 Salidas mínimas
- delta_ic_boot_mean
- delta_ic_boot_std
- delta_ic_boot_pvalue
- delta_ic_ci_95_low
- delta_ic_ci_95_high

10. Mejora incremental verdaderamente nueva: partial IC

Una interacción puede mejorar IC simplemente por ser una reexpresión no lineal
de la misma información contenida en una base dominante.
Por ello, además de DeltaIC, el módulo debe medir información incremental condicional.

10.1 Definición práctica
Para cada fecha d, sea:
- z_d = alpha_int,d
- B_d = [alpha_i1,d, ..., alpha_ik,d]
- y_d = label_d

Se residualizan:
- z_d sobre B_d
- y_d sobre B_d

y se calcula una correlación ordinal o lineal residual.

Versión simple MVP:
- residualización lineal cross-sectional por fecha;
- agregación temporal posterior.

Formalmente:

    z_res,d = z_d - Proj(z_d | B_d)
    y_res,d = y_d - Proj(y_d | B_d)

y:

    PIC_d = Corr(z_res,d, y_res,d)

donde Corr puede ser Pearson o Spearman sobre rankings residualizados,
según política declarada.

La métrica agregada es:

    PIC_bar = mean_d(PIC_d)

Interpretación:
- PIC_bar > 0 implica información incremental más allá de las bases;
- PIC_bar cerca de 0 sugiere que la interacción es una reparametrización redundante.

11. Complejidad de interacción

La complejidad adicional debe cuantificarse de forma explícita.
Una forma razonable para MVP:

    kappa_int
    =
    w_ops * N_ops
    + w_params * N_params
    + w_arity * (k - 1)

donde:
- N_ops = número de operadores adicionales de la fórmula;
- N_params = número de thresholds, slopes u otros parámetros libres;
- k = número de señales base involucradas.

La complejidad no invalida por sí sola, pero:
- debe penalizarse en ranking;
- debe usarse en desempates;
- debe impedir que mejoras muy marginales sobrevivan con estructuras demasiado complejas.

12. Penalización tipo información

Como versión avanzada útil, el módulo puede construir una puntuación penalizada:

    score_int
    =
    mu_Delta
    - lambda_sigma * sigma_Delta
    - lambda_kappa * kappa_int
    - lambda_red * redundancy_max

Esto no sustituye a los gates, pero sí ayuda a ordenar candidatas supervivientes.

No conviene llamar a esto “AIC” salvo que se derive de una log-likelihood explícita.
En este módulo, es preferible llamarlo penalized screening score.

13. Redundancia funcional

Una interacción puede tener DeltaIC > 0 pero ser casi idéntica funcionalmente
a una señal base o a otra interacción ya seleccionada.

Sea v(alpha) el vector apilado de scores OOS de una señal.
Se define una distancia simple:

    d(alpha_a, alpha_b) = 1 - | corr(v(alpha_a), v(alpha_b)) |

La redundancia máxima respecto a sus componentes base es:

    redundancy_base_max
    =
    max_j | corr(v(alpha_int), v(alpha_j)) |

Y la redundancia respecto a la shortlist ya seleccionada puede definirse como:

    redundancy_shortlist_max
    =
    max_s | corr(v(alpha_int), v(alpha_s)) |

Regla práctica:
si redundancy_base_max o redundancy_shortlist_max supera un threshold configurado,
la candidata debe degradarse o descartarse.

14. Clustering de redundancia

14.1 MVP
Clustering jerárquico o greedy sobre matriz de similitud:

    S_ij = | corr(v_i, v_j) |

con restricción de máximo una interacción por cluster en la shortlist final.

14.2 Versión avanzada útil
Spectral clustering sobre el Laplaciano construido desde S_ij.
Es razonable cuando:
- el número de supervivientes tras filtros previos es moderado;
- existe estructura de redundancia no capturada bien por greedy.

No hace falta ir a métodos más sofisticados.

15. Cobertura y viabilidad estadística

Una interacción debe superar validaciones mínimas de viabilidad:

15.1 Coverage diaria
Para cada fecha d:

    coverage_d = n_valid_d / n_eligible_d

15.2 Coverage media
    coverage_mean = mean_d(coverage_d)

15.3 Fechas válidas mínimas
Debe existir un mínimo de fechas válidas OOS.

15.4 Variación cross-sectional
Si una interacción colapsa a casi constante en muchas fechas,
no debe evaluarse como candidata válida.

Campos mínimos a persistir:
- n_valid_dates
- coverage_mean
- coverage_std
- n_dates_low_variation

16. Gramática permitida

La gramática debe ser restringida, interpretable y versionada.

16.1 Operadores algebraicos básicos
- product(a,b) = a * b
- min(a,b)
- max(a,b)
- sum_weighted(a,b; w)
- safe_ratio(a,b; eps)

16.2 Operadores condicionales
- gated_product(a,b; c) = a * 1{b > c}
- gated_abs(a,b; c) = a * 1{|b| > c}
- sigmoid_gate(a,b; slope, center) = a * sigma(slope * (b - center))

16.3 Restricciones
- k_max <= 3
- max_params_per_interaction <= 2 o 3
- prohibición de nesting profundo
- prohibición de operadores fuera de vocabulario

17. Interacciones que NO conviene permitir

No conviene incluir aquí:
- árboles entrenados sobre señales base;
- redes neuronales pequeñas “solo para probar”;
- kernels no paramétricos generales;
- stacking aprendido sobre labels;
- búsqueda simbólica abierta sin control fuerte.

Eso cambia la naturaleza del módulo y lo acerca a otro tipo de search.

18. Filtro previo de señales base

No debe combinarse cualquier cosa con cualquier cosa.
El conjunto de bases elegibles debe pasar gates mínimos:

- mean_ic >= threshold
- coverage_mean >= threshold
- n_valid_dates >= threshold
- no PIT flags
- no duplicidad funcional evidente
- opcional: max_bases por familia

Esto reduce explosión combinatoria y baja la tasa de falsos positivos.

19. Validación temporal obligatoria

Toda interacción y toda base comparada deben evaluarse bajo exactamente:
- mismo split protocol;
- mismo embargo/purge;
- mismo universo elegible;
- mismo label;
- mismo masking de fechas/activos.

No se debe permitir comparar:
- interacción OOS contra base IS;
- interacción con otra cobertura temporal;
- interacción calculada sobre una muestra distinta.

20. Multiple testing

La búsqueda de interacciones induce un problema severo de multiplicidad.
El módulo debe corregir al menos dentro de familias homogéneas de operador.

20.1 Recomendación MVP
Benjamini-Hochberg (BH) por familia de interacción:
- product family
- min/max family
- gated family
- sigmoid family

20.2 Cuándo usar corrección más conservadora
Benjamini-Yekutieli o bootstrap-based adjustment sólo si:
- el número de candidatas supervivientes es moderado;
- se desea mayor conservadurismo.

Por defecto, BH es suficiente como best practice práctica en esta capa.

21. Gates de selección

El módulo no debe usar un único score opaco.
Debe usar gates explícitos y luego ranking secundario.

Ejemplo de lógica base:

PASS si simultáneamente:
- mu_Delta > min_delta_ic_mean
- h_Delta >= min_delta_hit_rate
- delta_ic_boot_pvalue <= max_boot_pvalue
- PIC_bar >= min_partial_ic
- coverage_mean >= min_coverage
- kappa_int <= max_complexity
- redundancy_base_max <= max_abs_corr_with_base
- q_value_BH <= max_q_value

WARN si:
- no hay FAIL
- pero alguna métrica cae en banda intermedia

FAIL si:
- mu_Delta <= fail_delta_ic
- coverage insuficiente
- bootstrap no soporta mejora
- partial IC demasiado débil
- redundancia extrema
- complejidad excesiva
- interacción degenerada

26. Flujo canónico del módulo

1. load inputs
2. validate PIT and keys
3. filter base signals
4. enumerate interactions
5. discard semantic degeneracies
6. compute interaction panel PIT
7. evaluate bases and interactions under same OOS protocol
8. compute DeltaIC by fold and aggregate
9. compute partial IC
10. run block bootstrap for DeltaIC
11. compute complexity and redundancy
12. adjust p-values within families
13. apply gates
14. cluster/dedup surviving candidates
15. rank and persist shortlist

27. Invariantes del módulo

27.1 Comparabilidad
Cada interacción y sus bases se evalúan bajo exactamente el mismo protocolo OOS.

27.2 PIT
La interacción usa sólo inputs observables en t.

27.3 Dominio
IC_d in [-1, 1]

27.4 Viabilidad
No se computa interacción si no hay variación cross-sectional suficiente.

27.5 Reproducibilidad
Mismo config + mismo snapshot + misma seed -> mismos resultados

27.6 No duplicidad
Misma interacción canónica -> mismo interaction_id / same canonical hash

28. Canonical hash

Cada interacción debe tener una representación canónica única:

    h(alpha_int) = H( serialize( canonical_form(alpha_int) ) )

La forma canónica debe normalizar:
- orden de operandos si el operador es conmutativo;
- thresholds y params serializados de forma estable;
- nombres de alpha base normalizados;
- versión de gramática.

Esto evita duplicados algebraicamente equivalentes.

29. Failure modes principales

- interacciones degeneradas o casi constantes;
- mejora aparente impulsada por pocas fechas;
- bootstrap mal calibrado por bloques demasiado pequeños;
- confounding total con una base dominante;
- redundancia funcional extrema;
- cobertura insuficiente;
- multiplicidad severa ignorada;
- explosión combinatoria;
- errores PIT al mezclar señales base de snapshots distintos.

30. Complejidad computacional

Sea:
- M = número de señales base elegibles
- G = número de operadores
- P = número de parametrizaciones por operador
- Kmax = cardinalidad máxima
- D = número de fechas
- N_d = tamaño cross-sectional por fecha

El número bruto de candidatas puede crecer como:

    O( G * P * sum_{k=2}^{Kmax} C(M,k) )

El coste de evaluación domina y es aproximadamente:

    O( n_candidates * sum_d N_d log N_d )

si la métrica primaria es Spearman diario.

Por ello son esenciales:
- filtro previo de bases;
- gramática restringida;
- rechazo temprano de degeneradas;
- memoization de paneles intermedios cuando sea viable.

31. Qué NO conviene introducir

No conviene meter aquí:
- causal discovery;
- graphical models;
- deep interaction learners;
- genetic programming completo;
- symbolic regression abierta;
- policy learning;
- meta-learning de la gramática.

Todo eso cambia el tipo de módulo y reduce auditabilidad.

32. Pruebas mínimas del módulo

- interacción product correcta en caso sintético;
- interacción gated correcta con threshold conocido;
- comparación DeltaIC correcta frente al mejor componente base;
- bootstrap reproducible con seed fija;
- partial IC correcta en ejemplo donde interacción es pura redundancia;
- drop de interacciones constantes;
- BH correcto dentro de familia;
- canonical hash idéntico para expresiones equivalentes;
- dedup por redundancia correcto;
- shortlist reproducible.

33. MVP vs versión avanzada

33.1 MVP
- filtro previo de bases
- gramática restringida
- evaluación OOS purged
- DeltaIC por fold
- mu_Delta / sigma_Delta / h_Delta
- bootstrap por bloques
- complexity score
- redundancia por correlación absoluta
- BH por familia
- shortlist auditable

33.2 Avanzado
- partial IC más robusto
- clustering espectral
- score penalizado más refinado
- coverage-weighted DeltaIC secundario
- benchmark frente a shortlist histórica
- comparación por subperiodos o regímenes

34. Definition of Done

El módulo se considera terminado cuando:
- genera interacciones sólo desde señales base válidas;
- evalúa interacción y bases bajo el mismo protocolo temporal OOS;
- mide mejora incremental, estabilidad, redundancia y complejidad por separado;
- usa bootstrap temporalmente coherente para DeltaIC;
- corrige multiplicidad al menos dentro de familias;
- produce una shortlist pequeña, diversa y auditable;
- y deja candidatas listas para pasar a alpha_library o validation layers.

35. Interpretación correcta del módulo

La lectura correcta de interaction_scan.py no es:
“buscar cualquier no linealidad que suba el IC”.

La lectura correcta es:
“auditar si una combinación estructurada de señales base aporta información incremental,
estable y no redundante, suficiente como para justificar más complejidad en el pipeline”.

36. Conclusión

interaction_scan.py debe funcionar como un filtro disciplinado de sinergias plausibles,
no como un generador descontrolado de combinaciones.
Su valor real está en separar:
- mejoras triviales,
- duplicados ornamentales,
- y falsas sinergias por muestra,
de un conjunto pequeño de interacciones que sí parecen añadir valor incremental genuino.

Ese es el comportamiento correcto de un módulo institucional de interaction discovery.
