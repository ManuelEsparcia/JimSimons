Modulo: research/ablation/model_ablation.py

1) Objetivo
Cuantificar contribucion de componentes de arquitectura/modelado (calibration, ensemble, regime gating, optimizer settings, etc.) para identificar complejidad util vs complejidad innecesaria.

2) Alcance
Incluye:
- ablation de componentes de pipeline de modelo.
- comparacion sistematica contra baseline completo.
- analisis de tradeoff performance vs complejidad operativa.
- recomendacion de arquitectura minima robusta.

No incluye:
- ablation de features individuales (feature_ablation).
- rediseno manual de estrategia fuera de plan.

3) Dependencias
Upstream:
- baseline pipeline run (full stack)
- config modular de componentes
- outputs de training/backtest/validation

Downstream:
- research/research_pipeline decision bundle
- roadmap de simplificacion tecnica

4) Contrato de entrada
Entradas minimas:
- baseline_pipeline_ref
- model_ablation_config_path
- run_id

Componentes tipicos ablatables:
- calibrate on/off
- ensemble on/off
- stacking vs blender
- regime_gating on/off
- turnover_control variants
- optimizer_penalties variants

5) Contrato de salida
- research/ablation/model/{run_id}/ablation_results.parquet
- research/ablation/model/{run_id}/complexity_vs_perf.parquet
- research/ablation/model/{run_id}/ablation_summary.json
- research/ablation/model/{run_id}/manifest.json

Campos minimos:
- ablation_id
- component_name
- baseline_metric
- ablated_metric
- delta_metric
- complexity_delta
- operational_risk_delta
- recommendation

6) Metodologia recomendada
- cambiar un componente por vez (one-factor-at-a-time) para claridad causal.
- opcionalmente bloques de componentes relacionados.
- evaluar en mismo protocolo temporal y mismos gates.

7) Reglas anti-leakage y comparabilidad
- misma data window y splits para todas las variantes.
- mismo objective metric.
- mismas policies de coste/riesgo salvo componente ablatado.

8) Flujo del proceso
1. Cargar baseline pipeline context.
2. Construir plan de variantes por componente.
3. Ejecutar variantes y recolectar metricas.
4. Calcular deltas de performance/riesgo/coste.
5. Construir frontera complejidad vs beneficio.
6. Emitir recomendaciones.

9) Validaciones minimas
- baseline reproducido.
- todas las variantes etiquetadas y trazadas.
- metricas comparables y completas.
- variaciones no deseadas fuera del componente objetivo detectadas.

10) Manejo de casos especiales
- componente acoplado a otro:
  - documentar dependencia y usar ablation conjunta.
- variante inestable/fallida:
  - registrar como fail con razon tecnica.
- mejoras marginales:
  - priorizar opcion mas simple por governance.

11) Observabilidad
Logging:
- INFO: progreso por variante.
- WARNING: diferencias no controladas detectadas.
- ERROR: variante no ejecutable.

Metricas:
- n_variants
- n_variants_success
- best_delta_metric
- complexity_reduction_candidates
- runtime_sec

12) CLI sugerida
python -m research.ablation.model_ablation ^
  --baseline-pipeline-ref research_pipeline_20260305_001 ^
  --config-path configs/research/model_ablation.yaml ^
  --run-id model_ablation_20260305_001

13) Estructura interna recomendada
- load_model_ablation_config(...)
- load_baseline_pipeline_context(...)
- build_component_ablation_plan(...)
- run_component_variant(...)
- compute_variant_deltas(...)
- score_complexity_vs_performance(...)
- generate_recommendations(...)
- persist_model_ablation_outputs(...)
- main()

14) Criterios de aceptacion
- evidencia clara de valor marginal por componente.
- recomendaciones accionables para simplificar arquitectura.
- insumo directo para decision promote/iterate/reject.

15) Casos de prueba minimos
- quitar calibracion degrada calibration metrics.
- quitar regime gating aumenta drawdown en stress windows.
- blender vs stacking comparado en mismo protocolo.
- variante fallida queda en reporte con status correcto.

16) Checklist rapido
- [ ] plan de ablation por componente.
- [ ] ejecucion consistente con baseline.
- [ ] analisis complejidad vs beneficio.
- [ ] recomendaciones + reportes + manifest.
