\section{Objective}

The purpose of this module is to measure the \textbf{incremental out-of-sample value} of architectural components in the production trading stack. The question is not whether a component sounds sophisticated, but whether it improves the deployed system enough to justify its additional complexity and operational burden.

Formally, the module evaluates whether removing a component from the baseline pipeline leads to a material degradation in out-of-sample trading performance under a fixed and leakage-safe protocol.

This supports four practical decisions:
\begin{itemize}[leftmargin=*]
    \item retain a component that adds robust value;
    \item simplify a component that adds value but is operationally expensive;
    \item remove a component whose contribution is negligible or unstable;
    \item identify component interactions that require joint evaluation.
\end{itemize}

\section{Design Principles}

The model ablation study follows six principles.

\begin{enumerate}[leftmargin=*]
    \item \textbf{Out-of-sample first.} All conclusions are based on purged walk-forward evaluation.
    \item \textbf{Frozen baseline.} The baseline pipeline, labels, splits, execution assumptions, and objective are fixed before any ablation begins.
    \item \textbf{One primary performance metric.} Ranking is performed with a single predefined trading metric.
    \item \textbf{Complexity is reported separately.} Complexity is not mixed into the performance metric through arbitrary formulas.
    \item \textbf{Minimal intervention.} In canonical mode, the only intended change is removal of the component under test.
    \item \textbf{Operational interpretability.} The result must support an implementation decision, not a rhetorical conclusion.
\end{enumerate}

\section{Primary Metric and Secondary Diagnostics}

\subsection{Primary Metric}

Let
\[
M = \operatorname{median}_{w=1,\dots,W} \mathrm{IR}^{\text{net}}_w
\]
where $\mathrm{IR}^{\text{net}}_w$ is the net Information Ratio of the out-of-sample portfolio return stream in walk-forward window $w$, evaluated using the same transaction-cost and slippage assumptions as the baseline system.

The median across windows is used because it is more robust than a mean to isolated favorable periods and better reflects consistency across market states.

\subsection{Secondary Diagnostics}

The following quantities are reported separately and are not merged into the primary ranking score:

\begin{itemize}[leftmargin=*]
    \item net Sharpe ratio;
    \item turnover;
    \item maximum drawdown;
    \item tail-loss statistics, if available;
    \item latency impact;
    \item implementation footprint;
    \item fraction of OOS windows with positive contribution.
\end{itemize}

\section{Scope}

\subsection{Included Components}

Typical ablation candidates include, but are not limited to:

\begin{itemize}[leftmargin=*]
    \item probability calibration;
    \item ensembling or blending layers;
    \item stacking meta-models;
    \item regime-gating logic;
    \item turnover-control overlays;
    \item post-processing layers that alter sizing or trade selection.
\end{itemize}

\subsection{Excluded}

The following are out of scope for this module:

\begin{itemize}[leftmargin=*]
    \item feature ablation;
    \item label redesign;
    \item universe redesign;
    \item execution-engine redesign;
    \item unrestricted retuning of the remaining pipeline after each ablation.
\end{itemize}

If the study requires a re-optimization of the full architecture after removing a component, that experiment is classified as a model redesign study, not canonical model ablation.

\section{Component Definition}

A component is defined as a pipeline block that satisfies all three conditions:

\begin{enumerate}[leftmargin=*]
    \item it has a well-defined boundary in code;
    \item it can be disabled or bypassed without changing the rest of the data contract;
    \item its removal can be evaluated while keeping the remainder of the protocol fixed.
\end{enumerate}

Examples:
\begin{itemize}[leftmargin=*]
    \item \texttt{calibration}: transforms raw scores into calibrated probabilities;
    \item \texttt{ensemble}: combines predictions from multiple base models;
    \item \texttt{regime\_gating}: conditions participation or sizing on regime state;
    \item \texttt{turnover\_control}: suppresses low-conviction or excessively costly trades.
\end{itemize}

A component is not defined by vague ideas such as ``intelligence'' or ``stability layer.'' It must correspond to an explicit software module or switchable pipeline stage.

\section{Inputs and Outputs}

\subsection{Input Contract}

\begin{verbatim}
baseline_pipeline_ref
model_ablation_config_path
run_id
component_scope in {single, coupled}
evaluation_mode in {drop_retrain}
\end{verbatim}

\subsection{Upstream Dependencies}

The following artifacts must already exist:

\begin{itemize}[leftmargin=*]
    \item frozen baseline pipeline reference;
    \item labels and target horizon;
    \item purged walk-forward split specification;
    \item execution and cost model;
    \item model ablation configuration;
    \item reproducibility metadata including seeds and software version.
\end{itemize}

\subsection{Output Contract}

The canonical output is \texttt{model\_ablation\_results.parquet}. Each row represents one ablation candidate.

\section{Canonical Ablation Effect}

For component $k$, define the canonical ablation effect as

\[
\Delta_k = M_{\text{base}} - M_{-k}
\]

where:
\begin{itemize}[leftmargin=*]
    \item $M_{\text{base}}$ is the baseline primary metric under the frozen protocol;
    \item $M_{-k}$ is the primary metric after removing component $k$ and rerunning the pipeline under the same protocol.
\end{itemize}

Interpretation:
\begin{itemize}[leftmargin=*]
    \item $\Delta_k > 0$: the removed component was beneficial;
    \item $\Delta_k \approx 0$: the component adds little incremental value;
    \item $\Delta_k < 0$: the component is likely harmful or unnecessary in its current form.
\end{itemize}

This module does not collapse performance and complexity into a single formula such as
\[
\frac{\Delta \mathrm{IR}}{\Delta \mathrm{Comp} + \varepsilon}
\]
because that ratio is unstable, scale-dependent, and difficult to govern consistently across heterogeneous components.

\section{Complexity and Operational Burden}

Complexity is treated as a separate diagnostic axis rather than a pseudo-theoretical scalar.

\subsection{Complexity Dimensions}

Each component is assessed along three operational dimensions:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Implementation complexity:} code surface area, number of dependencies, maintenance burden.
    \item \textbf{Runtime complexity:} added latency, memory overhead, scheduling burden.
    \item \textbf{Governance complexity:} number of extra failure modes, monitoring burden, calibration drift risk.
\end{enumerate}

\subsection{Complexity Classification}

Instead of inventing an arbitrary continuous score, each component receives a coarse operational classification:
\[
\texttt{LOW},\quad \texttt{MEDIUM},\quad \texttt{HIGH}.
\]

This is intentional. For production decisions, a stable categorical assessment is often more useful than a fragile numeric proxy.

\section{Operational Risk}

Operational risk is also reported separately from performance.

Typical risk indicators include:
\begin{itemize}[leftmargin=*]
    \item increase in realized drawdown;
    \item increase in turnover or trading churn;
    \item increased sensitivity to missing inputs or delayed data;
    \item increase in instability across walk-forward windows;
    \item higher frequency of degenerate or empty decision states.
\end{itemize}

A component is not considered robust merely because it improves the central metric slightly. If it introduces fragile behavior, that must be visible in the ablation report.

\section{Evaluation Protocol}

\subsection{Baseline Freeze}

Before any run begins, the following are frozen:

\begin{itemize}[leftmargin=*]
    \item asset universe;
    \item label definition;
    \item purged split and embargo rules;
    \item execution and transaction-cost model;
    \item training configuration;
    \item seed configuration;
    \item baseline hyperparameters.
\end{itemize}

\subsection{Canonical Mode}

Canonical model ablation uses \texttt{drop\_retrain}.

For each component $k$:
\begin{enumerate}[leftmargin=*]
    \item disable or remove component $k$;
    \item keep the rest of the pipeline unchanged;
    \item rerun all training and evaluation stages needed by the remaining stack;
    \item evaluate on the identical purged walk-forward schedule;
    \item aggregate the out-of-sample results.
\end{enumerate}

\subsection{Retuning Policy}

Component-specific retuning is not permitted in canonical mode.

This rule is important. Otherwise the ablation result becomes a mixture of:
\begin{itemize}[leftmargin=*]
    \item the value of the removed component, and
    \item the quality of re-optimizing the rest of the system after removal.
\end{itemize}

Those are different research questions and must not be conflated.

\section{Window-Level Contribution and Stability}

For each component $k$ and window $w$, define
\[
\Delta_{k,w} = \mathrm{IR}^{\text{net}}_{\text{base},w} - \mathrm{IR}^{\text{net}}_{-k,w}.
\]

Then define the stability statistic
\[
S_k = \frac{1}{W}\sum_{w=1}^{W}\mathbf{1}(\Delta_{k,w} > 0).
\]

Interpretation:
\begin{itemize}[leftmargin=*]
    \item $S_k \approx 1$: contribution is persistent across windows;
    \item $S_k \approx 0.5$: contribution is regime-dependent or weak;
    \item $S_k \ll 0.5$: the component is likely unnecessary or harmful.
\end{itemize}

This statistic is simple, interpretable, and easier to govern than ad hoc Pareto scores.

\section{Coupled Ablations}

Some components are structurally dependent. For example:
\begin{itemize}[leftmargin=*]
    \item stacking may depend on calibration;
    \item regime gating may change the effective value of turnover control;
    \item a blender may be irrelevant unless multiple experts are active.
\end{itemize}

Therefore the module supports two scopes:
\begin{itemize}[leftmargin=*]
    \item \textbf{single:} remove one component at a time;
    \item \textbf{coupled:} remove a predefined dependency set jointly.
\end{itemize}

Coupled ablations are not searched combinatorially. They must be declared explicitly in the configuration, based on known architectural dependencies. This keeps the experiment tractable and interpretable.

\section{Execution Flow}

The expected execution flow is:

\begin{enumerate}[leftmargin=*]
    \item \texttt{load\_baseline\_pipeline\_context()}
    \item \texttt{build\_component\_plan()}
    \item \texttt{run\_ablated\_variant()}
    \item \texttt{compute\_window\_level\_deltas()}
    \item \texttt{aggregate\_component\_results()}
    \item \texttt{attach\_complexity\_and\_risk\_diagnostics()}
    \item \texttt{generate\_recommendations()}
    \item \texttt{persist\_results()}
\end{enumerate}

All runs must be deterministic conditional on fixed inputs and seeds.

\section{Invariants}

The following invariants are mandatory:

\begin{itemize}[leftmargin=*]
    \item no future information may enter training, calibration, or gating logic;
    \item baseline and ablated runs must use the same split structure;
    \item baseline and ablated runs must use the same execution simulator;
    \item in canonical mode, the only intentional architectural change is removal of the declared component;
    \item all results must be traceable to a unique baseline pipeline reference and configuration hash.
\end{itemize}

Any violation invalidates the ablation result.

\section{Uncertainty Quantification}

For each component, a paired bootstrap confidence interval is computed over the out-of-sample evaluation units consistent with the portfolio aggregation layer.

A positive point estimate is not sufficient. If the confidence interval is wide and materially overlaps zero, the component is classified as uncertain rather than valuable.

\section{Recommendation Logic}

Recommendations are conservative and rule-based.

\begin{itemize}[leftmargin=*]
    \item \textbf{KEEP:} $\Delta_k$ is clearly positive, the uncertainty is acceptable, and stability is high.
    \item \textbf{REVIEW:} contribution exists but is unstable, weak, or operationally awkward.
    \item \textbf{SIMPLIFY:} contribution is positive but comparable gains may be achievable with a lighter implementation.
    \item \textbf{REMOVE:} $\Delta_k \le 0$ or the component fails persistently across windows.
\end{itemize}

The recommendation is not a theorem. It is a governance output for the research and production roadmap.

\section{Validation and Acceptance Criteria}

The module is accepted only if the following conditions hold:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Reproducibility:} fixed seeds reproduce the same ranking up to numerical tolerance.
    \item \textbf{Leakage safety:} no future information enters any ablated or baseline path.
    \item \textbf{Isolation:} disabling a component does not silently alter unrelated pipeline behavior.
    \item \textbf{Dependency sanity check:} known coupled components show sensible behavior in coupled ablation mode.
    \item \textbf{Output completeness:} every candidate generates a full result row with delta, uncertainty, complexity class, operational diagnostics, and recommendation.
\end{enumerate}

\section{Operational Interpretation}

This module should be used when:
\begin{itemize}[leftmargin=*]
    \item the production stack has become too layered or fragile;
    \item a component is suspected of adding operational burden without clear trading value;
    \item a simpler architecture is being considered for deployment hardening;
    \item research performance is decaying and the system needs structural pruning.
\end{itemize}

It should not be used to advertise target performance levels or to justify arbitrary complexity metrics.

\section{Conclusion}

A strong model ablation module does not attempt to prove that a sophisticated architecture is intrinsically superior. It answers a narrower and more useful question:

\begin{quote}
\emph{Which architectural components produce robust out-of-sample value, and which ones should be simplified or removed?}
\end{quote}

That is the correct standard for model governance in a professional small-cap swing trading pipeline.
