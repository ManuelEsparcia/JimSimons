\section{Objective}

The purpose of this module is to quantify the \textbf{marginal out-of-sample utility} of individual features and feature groups within the production swing-trading pipeline. The goal is not to make causal claims. The goal is to answer a narrower and operationally useful question:

\begin{quote}
\emph{If a feature or feature family is removed from the modelling stack, how much predictive and trading performance is lost out-of-sample under a fixed and leakage-safe evaluation protocol?}
\end{quote}

This module is therefore a \textbf{model dependence and robustness study}. It is used to:
\begin{itemize}[leftmargin=*]
    \item rank features by incremental contribution to the deployed signal;
    \item detect redundant or weak features;
    \item support feature pruning without contaminating the evaluation protocol;
    \item identify feature groups whose contribution is unstable across time or regimes.
\end{itemize}

\section{Design Principles}

The ablation study must satisfy five principles.

\begin{enumerate}[leftmargin=*]
    \item \textbf{Out-of-sample only.} All rankings are based on purged walk-forward evaluation.
    \item \textbf{Retraining consistency.} Canonical ablation results are obtained by retraining the full pipeline after removing the feature set under test.
    \item \textbf{Single primary metric.} The primary metric is fixed in advance and remains unchanged across all ablations.
    \item \textbf{No hidden protocol drift.} Splits, labels, universe, costs, and training configuration remain fixed relative to the baseline.
    \item \textbf{Operational usefulness over rhetoric.} The output must support implementation decisions: keep, group, downgrade, or remove.
\end{enumerate}

\section{Primary Metric and Secondary Diagnostics}

\subsection{Primary Metric}

Let $M$ denote the primary model score. In this module, the primary score is:

\[
M = \operatorname{median}_{w=1,\dots,W} \mathrm{IR}^{\text{net}}_w
\]

where $\mathrm{IR}^{\text{net}}_w$ is the net Information Ratio measured on the out-of-sample portfolio return stream for walk-forward window $w$, after transaction costs and slippage assumptions already used by the baseline pipeline.

The use of the median across windows is intentional:
\begin{itemize}[leftmargin=*]
    \item it reduces sensitivity to isolated favorable periods;
    \item it is more stable than a mean when regime dispersion is material;
    \item it aligns better with model governance than a best-period score.
\end{itemize}

\subsection{Secondary Diagnostics}

The following are reported as diagnostics, not as ranking criteria:
\begin{itemize}[leftmargin=*]
    \item net Sharpe ratio;
    \item hit rate;
    \item turnover;
    \item average holding period;
    \item maximum drawdown;
    \item fraction of OOS windows with positive contribution.
\end{itemize}

No ablation decision is based on a moving mixture of metrics. The ranking is driven by the primary metric only.

\section{Scope}

\subsection{Included}

\begin{itemize}[leftmargin=*]
    \item \textbf{Single-feature ablation:} remove one feature at a time.
    \item \textbf{Group ablation:} remove economically or statistically related feature families.
    \item \textbf{Permutation diagnostic:} optional secondary test to detect local dependence and leakage sensitivity.
    \item \textbf{Stability analysis:} measure whether the feature contribution is persistent across windows.
\end{itemize}

\subsection{Excluded}

\begin{itemize}[leftmargin=*]
    \item hyperparameter retuning specific to each ablation candidate;
    \item architecture changes;
    \item label redesign;
    \item execution-model changes;
    \item causal inference claims.
\end{itemize}

If hyperparameter search is rerun, the result is not classified as canonical feature ablation. It is classified as a separate model redesign experiment.

\section{Inputs and Outputs}

\subsection{Upstream Dependencies}

The module assumes the following artifacts already exist:

\begin{itemize}[leftmargin=*]
    \item a frozen baseline run reference;
    \item feature matrix $X \in \mathbb{R}^{N \times p}$;
    \item label vector or panel $Y$;
    \item purged walk-forward split specification;
    \item fixed execution and cost assumptions;
    \item ablation configuration file.
\end{itemize}

\section{Definitions}

\subsection{Canonical Ablation Effect}

For candidate $j$, define the canonical ablation effect as:

\[
\Delta_j = M_{\text{base}} - M_{-j}
\]

where:
\begin{itemize}[leftmargin=*]
    \item $M_{\text{base}}$ is the baseline primary score under the frozen protocol;
    \item $M_{-j}$ is the primary score obtained after removing feature or group $j$ and retraining the pipeline from scratch.
\end{itemize}

Interpretation:
\begin{itemize}[leftmargin=*]
    \item $\Delta_j > 0$: removing the feature hurts performance; the feature is useful.
    \item $\Delta_j \approx 0$: the feature is likely redundant, weak, or already spanned by others.
    \item $\Delta_j < 0$: removing the feature improves performance; the feature is a candidate for pruning or redesign.
\end{itemize}

\subsection{Stability Statistic}

To measure temporal consistency, define:

\[
S_j = \frac{1}{W}\sum_{w=1}^{W} \mathbf{1}\!\left(\Delta_{j,w} > 0\right)
\]

where $\Delta_{j,w}$ is the OOS performance loss in walk-forward window $w$.

This statistic is preferred to ad hoc ratios such as $1-\mathrm{CoV}(\Delta)$ because it is easier to interpret operationally:
\begin{itemize}[leftmargin=*]
    \item $S_j \approx 1$: the feature is consistently useful;
    \item $S_j \approx 0.5$: the contribution is regime-dependent or weak;
    \item $S_j \ll 0.5$: the feature is likely harmful or unstable.
\end{itemize}

\section{Ablation Modes}

\subsection{Mode A: \texttt{drop\_retrain} (Canonical)}

This is the default and authoritative mode.

For each candidate:
\begin{enumerate}[leftmargin=*]
    \item remove the feature or group from the feature set;
    \item refit the preprocessing stack on training data only;
    \item retrain the model with the same baseline configuration;
    \item evaluate on the corresponding OOS segment;
    \item aggregate over all walk-forward windows.
\end{enumerate}

This mode measures the dependence of the trained system on the feature set under a clean retraining protocol.

\subsection{Mode B: \texttt{permute\_oos} (Diagnostic)}

This is a secondary diagnostic and must never replace canonical ranking.

For each candidate:
\begin{enumerate}[leftmargin=*]
    \item keep the trained baseline model fixed;
    \item permute the feature values within the OOS evaluation block under a leakage-safe rule;
    \item re-evaluate predictions and trading metrics;
    \item compare against the unpermuted baseline.
\end{enumerate}

This mode is useful for fast diagnostics, leakage sensitivity checks, and local dependence analysis, but it is not equivalent to retraining-based ablation.

\section{Evaluation Protocol}

\subsection{Baseline Freeze}

Before any ablation run, the following baseline elements are frozen:
\begin{itemize}[leftmargin=*]
    \item asset universe;
    \item labels and horizon;
    \item split logic, purge, and embargo;
    \item execution assumptions;
    \item model class and training configuration;
    \item random seeds.
\end{itemize}

\subsection{Retraining Rule}

For canonical results:
\begin{itemize}[leftmargin=*]
    \item retraining is performed from scratch;
    \item warm-start is not allowed;
    \item preprocessing objects are refit using training folds only;
    \item no candidate-specific hyperparameter retuning is permitted.
\end{itemize}

This prevents the ablation delta from being contaminated by optimization drift or by path dependence in the training procedure.

\subsection{Aggregation Across Windows}

For each candidate $j$ and walk-forward window $w$, compute:
\[
\Delta_{j,w} = \mathrm{IR}^{\text{net}}_{\text{base},w} - \mathrm{IR}^{\text{net}}_{-j,w}
\]

Then aggregate:
\[
\Delta_j = \operatorname{median}_{w=1,\dots,W} \Delta_{j,w}
\]

The median window-level delta is reported alongside the aggregate score delta. Both are useful:
\begin{itemize}[leftmargin=*]
    \item aggregate delta for executive ranking;
    \item window-level delta distribution for robustness and regime analysis.
\end{itemize}

\section{Uncertainty Quantification}

Each candidate receives a paired bootstrap confidence interval computed over OOS windows or OOS units consistent with the return aggregation layer.

If the bootstrap interval for $\Delta_j$ straddles zero widely, the feature is not considered reliably important even if the point estimate is positive.

The module does not translate statistical uncertainty into exaggerated scientific claims. It only reports whether the observed loss appears stable enough to matter for model governance.

\section{Feature Grouping and Redundancy}

\subsection{Economic Grouping}

Feature families may be defined by construction:
\begin{itemize}[leftmargin=*]
    \item momentum;
    \item liquidity and microstructure;
    \item volatility;
    \item price structure;
    \item fundamentals;
    \item event and calendar features.
\end{itemize}

\subsection{Statistical Grouping}

Highly correlated features may also be grouped using a training-set-only dependency analysis. A default option is absolute rank-correlation clustering above a configurable threshold such as $|\rho| \ge 0.80$.

This grouping is used to avoid misleading conclusions of the form:
\begin{quote}
\emph{feature A looks unimportant only because feature B is a near-duplicate.}
\end{quote}

Single-feature and grouped ablations should therefore be interpreted jointly.

\section{Missingness, Coverage, and Sparse Features}

Sparse features are not judged by performance impact alone. Each result should be interpreted jointly with its effective coverage.

A feature with moderate positive delta but very low coverage may still be useful for niche regimes, but it should not be treated as a general production driver.

Coverage statistics must therefore be logged explicitly:
\begin{itemize}[leftmargin=*]
    \item non-missing fraction;
    \item cross-sectional availability;
    \item stability of availability across time.
\end{itemize}

\section{Execution Flow}

The expected execution flow is:

\begin{enumerate}[leftmargin=*]
    \item \texttt{load\_baseline\_context()}
    \item \texttt{build\_ablation\_plan()}
    \item \texttt{run\_single\_ablation()}
    \item \texttt{compute\_window\_level\_deltas()}
    \item \texttt{aggregate\_candidate\_results()}
    \item \texttt{rank\_and\_recommend()}
    \item \texttt{persist\_results()}
\end{enumerate}

The module must be deterministic conditional on fixed seeds and fixed inputs.

\section{Invariants}

The following invariants are mandatory:

\begin{itemize}[leftmargin=*]
    \item no future information may enter any training or preprocessing stage;
    \item the split structure must be identical across baseline and ablated runs;
    \item the execution simulator must be identical across baseline and ablated runs;
    \item the only intentional change in canonical mode is the removal of the candidate feature set;
    \item all results must be traceable to the baseline run reference and ablation configuration.
\end{itemize}

Any run violating these invariants is invalid and must not be used for ranking.

\section{Recommendation Logic}

The recommendation field is rule-based and conservative.

\begin{itemize}[leftmargin=*]
    \item \textbf{KEEP:} $\Delta_j$ is positive, the confidence interval is acceptably tight, and $S_j$ is high.
    \item \textbf{REVIEW:} $\Delta_j$ is modest or unstable, or the uncertainty is too wide.
    \item \textbf{GROUP:} the feature appears weak alone but the family ablation is materially positive.
    \item \textbf{REMOVE:} $\Delta_j \le 0$ and the negative or null contribution is persistent.
\end{itemize}

These labels are implementation aids, not scientific truth claims.

\section{Validation and Acceptance Criteria}

The module is accepted only if the following conditions are met:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Reproducibility:} repeated runs with fixed seeds reproduce the same ranking and stored deltas up to numerical tolerance.
    \item \textbf{Leakage safety:} all preprocessing and feature grouping logic are trained on training data only.
    \item \textbf{Synthetic sanity check:} an injected predictive feature is ranked positively and a pure noise feature is not.
    \item \textbf{Redundancy sanity check:} near-duplicate features show weak single-feature deltas but stronger grouped deltas.
    \item \textbf{Output completeness:} every candidate produces a full result row with baseline score, ablated score, delta, uncertainty, and recommendation.
\end{enumerate}

\section{Operational Notes}

This module is a research and governance tool. It should be used:
\begin{itemize}[leftmargin=*]
    \item before feature pruning;
    \item after major feature-store revisions;
    \item when the live stack becomes too wide or difficult to maintain;
    \item when performance degradation suggests that legacy features are no longer paying their complexity cost.
\end{itemize}

It should not be used to justify unsupported claims about true economic causality.

\section{Conclusion}

A good ablation module does not try to sound profound. It isolates one question cleanly: \emph{what is the incremental out-of-sample value of each feature under the production protocol?}

That is enough to support feature selection, redundancy control, and research prioritization in a professional small-cap swing trading pipeline.
