Módulo: research/ablation/feature_ablation.py

1. Objetivo del módulo

El objetivo de feature_ablation.py es medir la utilidad marginal out-of-sample de cada
feature individual o grupo de features dentro del stack de modelado, cuantificando la
pérdida de rendimiento al eliminarlos bajo exactamente el mismo protocolo temporal,
mismo pipeline de entrenamiento y mismas hipótesis de costes que el baseline.

Este módulo no pretende explicar causalmente por qué una feature funciona ni estimar
“verdades estructurales” del mercado. Su función correcta es más precisa:
actuar como herramienta de gobernanza cuantitativa para responder, de forma reproducible
y auditable, a una pregunta operativa:

- cuánto valor incremental OOS aporta realmente esta feature al sistema actual.

En un stack serio de small-cap swing trading, feature_ablation.py debe impedir:
- conservar features redundantes por inercia;
- podar features útiles por ruido muestral;
- confundir ausencia de valor individual con redundancia dentro de grupo;
- interpretar como robusto un delta pequeño con incertidumbre alta;
- y comparar baseline y ablated bajo protocolos no equivalentes.

Su valor no está en “hacer inferencia sofisticada por sí misma”, sino en imponer
disciplina experimental al proceso de pruning, simplificación y priorización de research. :contentReference[oaicite:6]{index=6}

2. Qué hace y qué no hace

2.1 Qué hace
- ejecuta ablaciones de features individuales o grupos;
- compara baseline vs versión ablated bajo protocolo congelado;
- calcula deltas OOS sobre la métrica primaria;
- persiste contribuciones por ventana walk-forward;
- mide estabilidad temporal de la contribución;
- estima intervalos de confianza mediante bootstrap emparejado;
- opcionalmente ajusta multiplicidad con BH-FDR;
- registra coste de complejidad de cada feature o grupo;
- entrega ranking y recomendación operativa: KEEP / REVIEW / GROUP / REMOVE.

2.2 Qué no hace
- no rediseña labels;
- no cambia arquitectura del modelo;
- no rehace el pipeline de selección de universo;
- no reoptimiza hiperparámetros de forma específica para cada feature ablacionada;
- no reemplaza validation_suite.py ni multiple_testing.py;
- no demuestra causalidad económica;
- no decide por sí sola asignación de capital ni promoción a live.

3. Rol dentro del stack

Secuencia conceptual correcta:

    feature_store / labels / purged_splits / baseline_run
    -> feature_ablation.py
    -> pruning / simplification / validation downstream
    -> decision de mantenimiento, agrupación o eliminación

Interpretación correcta:
- el baseline define el sistema congelado de referencia;
- feature_ablation.py mide dependencia marginal respecto a ese baseline;
- otros módulos decidirán después si esa dependencia es suficiente para cambios
  de producción, rediseño del stack o investigación adicional.

4. Principios no negociables

4.1 OOS por construcción
Toda comparación debe hacerse fuera de muestra bajo walk-forward purged.

4.2 Invarianza de protocolo
Entre baseline y ablated no puede cambiar:
- split temporal;
- universo;
- labels;
- execution model;
- costes;
- seeds;
- ni configuración de entrenamiento salvo la ausencia intencional del candidato.

4.3 Retraining canónico
La ablación autoritativa debe hacerse reentrenando desde cero tras eliminar la feature
o grupo candidato.

4.4 Métrica primaria fija
La jerarquía de candidatos no puede depender de una mezcla móvil de métricas.
Debe existir una métrica primaria declarada ex ante.

4.5 Incertidumbre explícita
Un delta puntual no basta. Toda interpretación relevante debe acompañarse de
estabilidad temporal y banda de incertidumbre.

4.6 Redundancia reconocida
La contribución débil de una feature individual no implica inutilidad si existe otra
feature casi clonada dentro del stack.

4.7 Utilidad operativa antes que ornamento estadístico
El módulo debe ayudar a decidir qué mantener, revisar, agrupar o eliminar, no convertirse
en un laboratorio de inferencia excesivamente complejo.

5. Definición formal del problema

Sea M la métrica primaria OOS del sistema bajo protocolo congelado.
Para cada candidato j, individual o grupal, definimos su efecto de ablación como:

    Delta_j = M_base - M_-j

donde:
- M_base es la métrica primaria del baseline;
- M_-j es la métrica primaria tras eliminar j y reentrenar el pipeline completo.

La interpretación correcta es:
- Delta_j > 0  -> quitar j empeora el sistema; j aporta valor;
- Delta_j ~ 0  -> j parece redundante, débil o ya spanneada por otras;
- Delta_j < 0  -> quitar j mejora el sistema; j es candidata a poda o rediseño.

La ablación estima valor marginal discreto respecto al sistema desplegado de research,
no una derivada estructural del proceso generador del mercado. :contentReference[oaicite:7]{index=7}

6. Métrica primaria

La métrica primaria recomendada es la mediana del IR neto por ventana walk-forward:

    M = median_w IR_net,w

donde:
- w indexa ventanas OOS del protocolo purged walk-forward;
- IR_net,w se calcula sobre el stream OOS neto de costes de esa ventana.

La mediana se usa porque:
- reduce sensibilidad a periodos excepcionalmente favorables;
- penaliza menos la dependencia de una sola ventana extrema;
- y es más adecuada para gobernanza que una media frágil ante heterogeneidad temporal.

Las métricas secundarias pueden persistirse como diagnóstico, por ejemplo:
- Sharpe neto;
- hit rate;
- turnover;
- max drawdown;
- holding period;
- cobertura;
- share de ventanas con contribución positiva.

Pero la jerarquía principal debe venir de M y no de una combinación oportunista
de estadísticas heterogéneas.

7. Unidad temporal correcta de evaluación

La unidad estadística natural del módulo no es el trade individual ni la fila pooled
(date, symbol). La unidad correcta es la ventana walk-forward o, como mínimo,
una unidad OOS agregada coherente con el protocolo de evaluación.

Para cada candidato j y ventana w:

    Delta_j,w = IR_net,base,w - IR_net,-j,w

y la agregación principal recomendada es:

    Delta_j = median_w Delta_j,w

Persistir Delta_j,w es obligatorio, porque el valor del módulo depende tanto de la
magnitud agregada como de la estabilidad temporal de dicha contribución.

8. Estadístico de estabilidad

Para medir persistencia temporal, se define:

    S_j = (1 / W) * sum_w 1{ Delta_j,w > 0 }

donde W es el número de ventanas válidas.

Interpretación:
- S_j cercano a 1   -> la feature ayuda de forma sistemática;
- S_j cercano a 0.5 -> su valor es inestable o dependiente del régimen;
- S_j bajo          -> la feature es persistentemente irrelevante o dañina.

Este estadístico es preferible a medidas opacas tipo 1 - CoV(Delta),
porque tiene lectura operativa directa:
“¿en cuántas ventanas quitar esta feature empeoró realmente el sistema?”

9. Modos de ablación

9.1 drop_retrain
Es el modo canónico y autoritativo.

Flujo correcto:
- eliminar la feature o grupo del feature set;
- refitear preprocessing usando solo training;
- reentrenar el modelo con la misma configuración base;
- evaluar en el OOS correspondiente;
- agregar resultados sobre todas las ventanas.

Este modo mide dependencia real del sistema entrenado respecto al candidato ablacionado.

9.2 permute_oos
Es un modo diagnóstico secundario y nunca debe sustituir al ranking canónico.

Flujo correcto:
- mantener fijo el modelo baseline ya entrenado;
- permutar los valores OOS de la feature bajo una regla leakage-safe;
- reevaluar predicciones y métricas;
- comparar con baseline.

Su utilidad es:
- screening rápido;
- chequeos de dependencia local;
- detección de sensibilidad a permutación;
- inspección preliminar de features secundarias.

Pero no equivale a una ablación con retraining y no debe tratarse como prueba final
de valor marginal.

10. Features individuales y grupos

El módulo debe soportar dos escalas de análisis.

10.1 Single-feature ablation
Eliminar una única columna o feature lógica.

Uso correcto:
- detectar features claramente inútiles o dañinas;
- inspeccionar contribución individual;
- localizar candidatos evidentes a pruning.

10.2 Group ablation
Eliminar familias económicas o clusters estadísticos completos.

Uso correcto:
- distinguir redundancia interna de inutilidad real;
- medir valor conjunto de familias;
- evitar concluir erróneamente que una feature “no vale nada” sólo porque otra casi idéntica
  absorbe su señal.

Los grupos pueden definirse por:
- semántica económica: momentum, liquidity, volatility, price structure, fundamentals, etc.;
- o clustering estadístico training-only, por ejemplo mediante correlación absoluta alta.

Una regla razonable es agrupar cuando:

    |rho| >= rho_group

con rho_group configurable, por ejemplo 0.80, calculado únicamente en training
para evitar leakage.

11. Cuantificación de incertidumbre

Todo candidato debe recibir banda de incertidumbre sobre Delta_j.

La opción recomendada es bootstrap emparejado por ventanas completas:
- la unidad bootstrap es la ventana w;
- en cada réplica se remuestran con reemplazo ventanas completas;
- se recalcula el Delta agregado de la réplica;
- se obtienen percentiles para construir CI.

Esto preserva:
- comparabilidad baseline vs ablated dentro de cada réplica;
- estructura intraventana;
- y coherencia con la unidad real de decisión del módulo.

Columnas mínimas recomendadas:
- ci_lower
- ci_upper
- p_value_boot

Interpretación correcta:
- un Delta_j positivo con CI ancho y cruzando cero no debe tratarse como evidencia fuerte;
- un Delta_j negativo con intervalo consistentemente bajo cero refuerza la poda;
- un Delta_j modesto pero estable puede ser más útil operacionalmente que uno grande y errático.

Si las ventanas presentan solapamiento fuerte, puede habilitarse block bootstrap como
extensión metodológica, pero el MVP institucional puede empezar con bootstrap por ventana
bien documentado. :contentReference[oaicite:8]{index=8}

12. Multiple testing

Cuando el número de candidatos es grande, el módulo debe reconocer explícitamente
el riesgo de cosechar ruido.

Por ello, si el universo de ablaciones es suficientemente amplio, debe poder aplicarse
un ajuste BH-FDR sobre los p-values bootstrap:

    { p_j } -> { q_j }

Interpretación correcta:
- Delta_j sigue siendo la magnitud operativa principal;
- q_j no reemplaza el ranking;
- q_j sirve como señal de cautela cuando muchas features parecen “importantes” por azar.

Esto añade una capa pragmática de gobernanza sin convertir el módulo en una maquinaria
de testing excesivamente compleja.

13. Coste de complejidad

No toda feature con delta pequeño merece el mismo trato.
Debe registrarse también un coste de complejidad kappa_j por feature o grupo.

Forma recomendada:

    kappa_j = w_dim * p_j + w_cpu * c_j + w_ops * o_j

donde:
- p_j es el número de columnas o dimensión añadida;
- c_j es una estimación normalizada de coste computacional;
- o_j es un score discreto de complejidad operativa o mantenimiento.

Esto permite persistir:

    delta_per_complexity = Delta_j / kappa_j

Interpretación correcta:
- no sustituye al ranking principal;
- sí ayuda a priorizar pruning o simplificación cuando dos candidatas tienen efectos parecidos
  pero una cuesta mucho más en cómputo, dependencias o mantenimiento.

14. Cobertura y missingness

La utilidad de una feature no debe interpretarse al margen de su cobertura.

Cada resultado debería registrar, al menos:
- non_missing_fraction;
- cross_sectional_coverage;
- temporal_coverage_stability.

Una feature con valor positivo pero cobertura muy baja puede ser útil en nichos o contextos
particulares, pero no debe promocionarse como driver general del sistema sin esa salvedad.

17. Flujo de ejecución esperado

Secuencia correcta:

    load_baseline_context()
    -> build_ablation_plan()
    -> run_single_ablation()
    -> compute_window_level_deltas()
    -> bootstrap_candidate_delta()
    -> apply_multiple_testing_if_needed()
    -> compute_recommendation()
    -> persist_results()

El módulo debe ser determinista condicional en seeds fijas, baseline fijo e inputs fijos.

18. Invariantes

Los invariantes obligatorios son:

- no puede entrar información futura en training ni preprocessing;
- la estructura de splits debe ser idéntica entre baseline y ablated;
- el simulador de ejecución y los costes deben ser idénticos;
- la única diferencia intencional en modo canónico es la ausencia del candidato j en X;
- el bootstrap debe remuestrear unidades temporales completas, no trades aislados;
- todo resultado debe poder trazarse a baseline_run_ref y a la configuración exacta de ablación;
- un candidato sólo es rankeable si dispone de un número mínimo de ventanas válidas W_min.

Si cualquiera de estos invariantes falla, la corrida no es válida para ranking.

19. Lógica de recomendación

La recomendación debe ser conservadora y rule-based.

19.1 KEEP
- Delta_j positivo;
- estabilidad alta;
- incertidumbre razonablemente acotada;
- y cobertura suficiente.

19.2 REVIEW
- Delta_j modesto;
- o estabilidad intermedia;
- o CI demasiado ancho;
- o cobertura dudosa.

19.3 GROUP
- contribución individual floja;
- pero aportación conjunta material en family/group ablation;
- o fuerte sospecha de redundancia con features hermanas.

19.4 REMOVE
- Delta_j <= 0 de forma persistente;
- o evidencia de daño estable;
- o beneficio tan pequeño que no compensa su complejidad operativa.

Estas etiquetas son ayudas de implementación, no afirmaciones ontológicas sobre la “verdad”
de la feature.

20. Failure modes

Los errores más peligrosos del módulo son:
- interpretar un Delta_j pequeño y ruidoso como evidencia fuerte;
- ejecutar muchas ablaciones y no corregir multiplicidad;
- usar permute_oos como si fuera equivalente a drop_retrain;
- cambiar aunque sea levemente el protocolo entre baseline y ablated;
- no evaluar grupos en presencia de colinealidad fuerte;
- sobrepremiar features con cobertura ínfima;
- y confundir utilidad marginal dentro del stack actual con valor económico profundo.

21. Criterios de aceptación

El módulo sólo debe considerarse listo si cumple:

- reproducibilidad:
  mismas seeds e inputs -> mismo ranking salvo tolerancia numérica;

- leakage safety:
  preprocessing, clustering y grouping calculados sólo con training cuando corresponda;

- sanity check positivo:
  una feature sintética predictiva debe salir con Delta positivo;

- sanity check negativo:
  una feature ruido no debe mostrar contribución robusta;

- sanity check de redundancia:
  duplicados cercanos deben mostrar deltas individuales débiles pero deltas grupales más claros;

- completitud:
  cada candidato debe producir fila completa con baseline, ablated, delta, incertidumbre
  y recomendación.

22. Uso correcto dentro del research process

feature_ablation.py debe usarse:
- antes de podar features del stack;
- después de revisiones grandes del feature store;
- cuando el modelo acumula demasiada anchura o deuda de mantenimiento;
- cuando se sospecha que parte del edge histórico ya no paga su coste de complejidad;
- y como puente entre discovery exploratorio y simplificación productiva.

No debe usarse para:
- justificar claims causales;
- reemplazar validación fuerte downstream;
- ni vender como “importantes” features cuya señal desaparece al mirar incertidumbre,
  cobertura o redundancia.

23. Conclusión

feature_ablation.py no existe para producir una lista bonita de rankings.
Existe para aislar una pregunta crítica de gobernanza del modelo:

- qué valor incremental OOS aporta realmente cada feature o grupo dentro del sistema actual.

Eso basta para soportar pruning disciplinado, control de redundancia, reducción de
complejidad y priorización de research en una pipeline profesional de small-cap swing trading.
