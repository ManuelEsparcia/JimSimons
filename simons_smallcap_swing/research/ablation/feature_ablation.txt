Modulo: research/ablation/feature_ablation.py

1) Objetivo
Medir contribucion causal de features/familias removiendolas o perturbandolas de forma controlada para identificar dependencia real y robustez del modelo.

2) Alcance
Incluye:
- ablation por feature individual.
- ablation por familia (momentum/liquidity/fundamental/etc).
- permutation/dropout ablation temporal.
- ranking de importancia robusta OOS.

No incluye:
- tuning de hiperparametros durante ablation.
- cambios de arquitectura de modelo (model_ablation lo cubre).

3) Dependencias
Upstream:
- feature store + feature dictionary
- labels + splits temporales
- pipeline de training configurable

Downstream:
- research/research_pipeline decision bundle
- explainability reports

4) Contrato de entrada
Entradas minimas:
- baseline_run_ref
- features_path
- labels_path
- splits_path
- ablation_config_path
- run_id

Parametros clave:
- ablation_mode (drop|permute|mask)
- scope (single_feature|feature_group)
- max_features_per_run
- retrain_policy (full|warm_start)

5) Contrato de salida
- research/ablation/feature/{run_id}/ablation_results.parquet
- research/ablation/feature/{run_id}/importance_ranking.parquet
- research/ablation/feature/{run_id}/ablation_summary.json
- research/ablation/feature/{run_id}/manifest.json

Campos minimos:
- ablation_id
- feature_or_group
- baseline_metric
- ablated_metric
- delta_metric
- stability_score
- significance_flag

6) Metodologia recomendada
- comparar baseline vs ablated en mismo protocolo temporal.
- repetir en multiples folds/regimenes.
- medir impacto medio y dispersion.

Importancia robusta:
- mayor deterioro consistente al remover => mayor importancia.

7) Reglas anti-leakage
- mismo splits purged para baseline y ablations.
- no retune ad-hoc para favorecer ablated run.
- usar exactamente mismo target/metrica primaria.

8) Flujo del proceso
1. Cargar baseline y config de ablation.
2. Generar variantes ablated.
3. Re-entrenar/evaluar cada variante.
4. Calcular delta de metricas y estabilidad.
5. Construir ranking final.
6. Persistir reportes.

9) Validaciones minimas
- baseline reproducido correctamente.
- ablations ejecutadas con cobertura suficiente.
- metricas comparables en mismo horizonte.
- sin trials truncados sin etiquetar.

10) Manejo de casos especiales
- features altamente correlacionadas:
  - reportar clusters y evitar sobreinterpretacion.
- feature con alta missingness:
  - interpretar impacto junto a coverage.
- costo computacional alto:
  - limitar max_features_per_run y priorizar grupos.

11) Observabilidad
Logging:
- INFO: avance por ablation.
- WARNING: inestabilidad alta entre folds.
- ERROR: ablation run fallido.

Metricas:
- n_ablations_executed
- avg_delta_metric
- n_high_impact_features
- runtime_sec

12) CLI sugerida
python -m research.ablation.feature_ablation ^
  --baseline-run-ref train_lgbm_20260305_001 ^
  --features-path features/store ^
  --labels-path labels/store ^
  --splits-path labels/splits/purged_splits_latest.parquet ^
  --config-path configs/research/feature_ablation.yaml ^
  --run-id feature_ablation_20260305_001

13) Estructura interna recomendada
- load_feature_ablation_config(...)
- load_baseline_context(...)
- build_ablation_plan(...)
- run_single_feature_ablation(...)
- run_group_ablation(...)
- compute_ablation_deltas(...)
- rank_feature_importance(...)
- persist_feature_ablation_outputs(...)
- main()

14) Criterios de aceptacion
- ranking de importancia robusta reproducible.
- evidencia clara de dependencia/robustez de features.
- resultados integrables en decision card.

15) Casos de prueba minimos
- feature sintetica importante produce delta fuerte.
- feature ruido produce delta bajo.
- grupo completo removido refleja impacto esperado.
- fallo parcial de ablation queda registrado sin perder trazabilidad.

16) Checklist rapido
- [ ] plan de ablation por feature/grupo.
- [ ] ejecucion comparativa baseline vs ablated.
- [ ] ranking robusto y estabilidad.
- [ ] reportes + manifest.
