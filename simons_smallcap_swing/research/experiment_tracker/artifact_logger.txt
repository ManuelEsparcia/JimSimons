Modulo: research/experiment_tracker/artifact_logger.py

1) Objetivo
Registrar artefactos de experimentos (modelos, reportes, datos intermedios, manifests) con metadata rica para reproducibilidad, auditoria y comparacion.

2) Alcance
Incluye:
- logging de artifacts y metrics al tracker.
- validacion de existencia/checksum/schema_version.
- convenciones de rutas y nombres de artifacts.
- linking run->artifact lineage.

No incluye:
- inicializacion del tracker (init_mlflow).
- ejecucion de experimento/sweep.

3) Dependencias
Upstream:
- runs activos en experiment tracker
- artifacts generados por pipelines de research
- schema catalog y hash utilities

Downstream:
- research/research_pipeline.py
- validation y reporting comparativo
- model_registry governance

4) Contrato de entrada
Entradas minimas:
- tracking_run_ref
- artifact_list (path, type, schema_version, checksum)
- metrics_dict (opcional)
- params_dict (opcional)
- run_id

5) Contrato de salida
- research/experiment_tracker/artifacts/{run_id}/artifact_log_summary.json
- research/experiment_tracker/artifacts/{run_id}/artifact_lineage.parquet
- research/experiment_tracker/artifacts/{run_id}/manifest.json

Campos minimos lineage:
- artifact_id
- artifact_type
- local_path
- tracker_uri
- checksum
- schema_version
- producer_step
- run_id

6) Tipos de artifacts recomendados
- model_artifact
- prediction_artifact
- metrics_report
- validation_report
- config_snapshot
- data_sample_snapshot
- diagnostics_report

7) Reglas de logging
- no loggear artifact sin checksum.
- incluir schema_version cuando aplica.
- usar rutas deterministas con run_id.
- evitar sobrescritura silenciosa de artifacts.

8) Flujo del proceso
1. Validar lista de artifacts y metadata.
2. Calcular/verificar checksum.
3. Loggear artifacts al tracker.
4. Loggear metrics/params asociados.
5. Persistir lineage local.

9) Validaciones minimas
- paths existentes y legibles.
- checksum consistente pre/post upload (si aplica).
- artifact_type permitido.
- metric names dentro de convencion.

10) Manejo de errores
- artifact missing -> fail por item (policy configurable).
- upload parcial -> retry por artifact.
- mismatch checksum -> mark corrupt y fail.

11) Observabilidad
Logging:
- INFO: artifacts loggeados por tipo.
- WARNING: metadata incompleta no critica.
- ERROR: fallo upload/checksum.

Metricas:
- n_artifacts_logged
- bytes_logged
- n_artifact_failures
- log_latency_sec

12) CLI sugerida
python -m research.experiment_tracker.artifact_logger ^
  --run-ref mlflow_run_abc123 ^
  --artifact-manifest-path tmp/artifacts_to_log.json ^
  --run-id artifact_logger_20260305_001

13) Estructura interna recomendada
- load_artifact_manifest(...)
- validate_artifact_metadata(...)
- compute_or_verify_checksums(...)
- log_artifacts_to_tracker(...)
- log_metrics_and_params(...)
- build_artifact_lineage(...)
- persist_artifact_logging_outputs(...)
- main()

14) Criterios de aceptacion
- todos los artifacts criticos quedan trazables en tracker.
- lineage completo run->artifact reproducible.
- errores de logging detectados y reportados.

15) Casos de prueba minimos
- logging exitoso de artifacts mixtos.
- checksum mismatch detectado.
- artifact faltante dispara fallo controlado.
- retry de upload parcial funciona.

16) Checklist rapido
- [ ] validar metadata/checksum.
- [ ] loggear artifacts + metrics + params.
- [ ] persistir lineage.
- [ ] manejar fallos parciales y retries.
