Módulo: ops/tracking/init_mlflow.py

1. Objetivo del módulo

El objetivo de init_mlflow.py es dejar preparado, de forma determinista y auditable,
el plano mínimo de tracking experimental sobre el que podrán ejecutarse después los
runs de research.

Este módulo no entrena modelos, no lanza sweeps, no registra artefactos complejos y
no reemplaza a artifact_logger.py. Su responsabilidad es más estrecha y más crítica:

- resolver exactamente qué backend de MLflow se usará;
- verificar que ese backend responde;
- resolver o crear el experimento de destino de forma idempotente;
- fijar la identidad mínima del contexto experimental;
- y persistir evidencia local suficiente para reconstruir qué quedó inicializado.

La pregunta correcta que responde no es “si MLflow está instalado”, sino algo mucho
más útil para reproducibilidad:

- si este pipeline puede empezar a registrar runs bajo una identidad de backend,
  experimento y metadatos que después podamos reconstruir sin ambigüedad.

2. Qué hace y qué no hace

2.1 Qué hace
- carga y valida configuración de inicialización;
- resuelve tracking_uri con precedencia explícita;
- resuelve artifact_location cuando aplica;
- sanitiza URIs antes de persistirlas;
- construye cliente MLflow y valida conectividad básica;
- resuelve o crea el experimento objetivo con semántica get-or-create;
- valida tags obligatorios;
- construye una clave de idempotencia del contexto de init;
- escribe init_summary.json;
- escribe manifest.json;
- y devuelve un estado ready, partial o failed claramente interpretable.

2.2 Qué no hace
- no abre runs de entrenamiento;
- no registra métricas ni artefactos del experimento;
- no hace sweeps ni búsqueda de hiperparámetros;
- no versiona datasets completos;
- no sustituye un artifact registry;
- no debe caer silenciosamente a un file:// local salvo política explícita;
- no debe inventar tags obligatorios faltantes.

3. Rol dentro del stack

Secuencia conceptual correcta:

    config + project context + env
    -> init_mlflow.py
    -> run creation / training / logging / artifact publishing
    -> auditabilidad y reproducibilidad downstream

Interpretación correcta:
- init_mlflow.py no es el tracker completo;
- es el bootstrap reproducible del entorno de tracking;
- deja cerrado dónde registrar, bajo qué experimento y con qué identidad mínima;
- y crea la base sobre la que luego operarán training, evaluation y artifact logging.

4. Qué significa “estado listo”

El módulo debe producir un estado listo verificable, no una intuición vaga de que
“parece que todo está configurado”.

Conceptualmente, el estado ready debe incluir al menos:

- tracking_uri resuelto y sanitizado;
- artifact_uri o artifact_location resuelto, si aplica;
- experiment_name;
- experiment_id;
- required_tags;
- manifest persistido;
- summary persistido;
- status final consistente.

La semántica correcta es:

- ready:
  backend validado, experimento resuelto, evidencia local escrita;

- partial:
  backend o experimento resueltos, pero evidencia local incompleta o inconsistente;

- failed:
  no existe garantía suficiente para iniciar tracking de forma reproducible.

5. Inputs requeridos

El módulo debe consumir tres bloques de entrada:

5.1 Configuración declarativa
Como mínimo:
- tracking_uri
- artifact_location
- experiment_name
- schema_version
- timeout_seconds
- max_retries
- required_tags opcionales o derivables

5.2 Contexto de proyecto
Como mínimo:
- project_name
- module_name
- git_commit
- config_hash
- data_snapshot_hash si ya existe
- timestamp UTC de inicialización

5.3 Entorno de ejecución
Como mínimo:
- variables de entorno relevantes;
- credenciales necesarias para el backend;
- hostname o identidad de ejecución;
- directorio local de escritura de evidencia.

6. Resolución determinista del backend

Uno de los puntos más importantes del módulo es evitar que tracking_uri dependa
de una mezcla opaca de defaults implícitos.

La resolución debe seguir una precedencia fija y documentada:

- override explícito por CLI o llamada;
- config declarativa;
- variable de entorno MLFLOW_TRACKING_URI;
- error explícito si no existe ninguna fuente válida.

La regla clave es esta:
- la resolución debe ser determinista;
- y el orden no puede depender de ramas ocultas del código ni de fallback silencioso.

Lo mismo aplica, cuando corresponda, a artifact_location.

7. Política de secretos y sanitización

Este módulo puede leer secretos del entorno o de un proveedor seguro,
pero no debe persistirlos nunca en ficheros de evidencia ni en logs.

Toda URI persistida debe pasar por una función de sanitización que elimine, al menos:
- tokens;
- passwords;
- query params sensibles;
- credenciales embebidas en la URI.

Regla correcta:
- el runtime puede usar la URI real;
- manifest.json e init_summary.json solo deben guardar la URI sanitizada.

Si faltan credenciales necesarias:
- el fallo debe ser temprano;
- explícito;
- y clasificarse como permanente, no como retryable.

8. Tags obligatorios

Los tags obligatorios no son decoración.
Son el mínimo contexto necesario para poder interpretar un run después.

El conjunto mínimo recomendado es:
- git_commit
- config_hash
- schema_version

Y, cuando ya existe en la pipeline:
- data_snapshot_hash
- project_name
- research_stage
- universe_id

La política correcta es:
- si falta un tag obligatorio, la inicialización no debe declararse ready;
- queda prohibido rellenar con “unknown” o valores ficticios por conveniencia.

9. Contrato de salida

La inicialización debe producir al menos dos artefactos locales:

- init_summary.json
- manifest.json

9.1 init_summary.json
Debe resumir el resultado operacional de la inicialización e incluir, como mínimo:
- status
- tracking_uri_sanitized
- artifact_location_sanitized
- experiment_name
- experiment_id
- schema_version
- resolved_at_utc
- latency_ms
- retry_count

9.2 manifest.json
Debe ser la pieza de trazabilidad y contener, como mínimo:
- project_name
- module_name
- git_commit
- config_hash
- data_snapshot_hash si existe
- required_tags
- experiment_name
- experiment_id
- init_key

Estos ficheros no son accesorios.
Son la evidencia mínima local de que el plano de tracking quedó realmente inicializado.

10. Idempotencia

La inicialización debe poder reintentarse sin generar estados inconsistentes.

Una clave razonable de idempotencia es:

    K_init = (
        tracking_uri_sanitized,
        experiment_name,
        config_hash,
        git_commit,
        schema_version
    )

Interpretación:
- si dos ejecuciones comparten K_init, deben converger al mismo estado lógico;
- no deben crear experimentos duplicados;
- no deben producir manifiestos contradictorios;
- y la segunda ejecución debe resolverse como replay seguro.

La idempotencia aquí no es un lujo.
Es una propiedad crítica para reintentos tras fallos transitorios o relanzamientos de pipeline.

11. Invariantes

Los invariantes obligatorios son:

- si status = ready, el backend responde dentro del timeout configurado;
- si status = ready, existe exactamente un experiment_id resuelto para el experiment_name usado;
- el manifest persistido debe reflejar exactamente config_hash y experiment_id del runtime;
- la URI persistida debe estar sanitizada;
- no debe existir fallback silencioso a backend alternativo salvo política explícita;
- el mismo contexto idempotente no debe inflar el estado ni crear duplicados.

Si alguno de estos invariantes falla, el módulo no debe declarar ready.

12. Resolución del experimento

La semántica correcta para el experimento es get-or-create robusto, no create ciego.

Secuencia correcta:
- consultar si el experimento ya existe;
- si existe, devolver experiment_id;
- si no existe, intentar create_experiment;
- si hay conflicto o carrera, volver a consultar;
- solo declarar éxito cuando existe un experiment_id final y estable.

Esto es especialmente importante bajo concurrencia moderada.
El módulo no debe asumir que “si create falla, entonces no existe”.
Puede tratarse de una carrera con otra inicialización concurrente.

13. Race conditions y creación concurrente

La creación concurrente del mismo experimento es uno de los fallos reales más importantes.

La política correcta es:
- query;
- create si no existe;
- si create devuelve conflicto, esperar brevemente;
- requery;
- repetir un número acotado de veces;
- abortar con error tipado si no se alcanza convergencia.

Esto elimina la tentación de:
- crear nombres alternativos silenciosos;
- duplicar experimentos;
- o suponer éxito sin confirmar el experiment_id final.

No hace falta vestir esto de teoría cuantitativa.
Basta con implementarlo como un patrón robusto de concurrencia moderada.

14. Reintentos y backoff

Los reintentos solo deben aplicarse a fallos transitorios.

Clasificación mínima recomendable:
- transitorios:
  timeout, connection error, backend 5xx;
- permanentes:
  credenciales inválidas, config inválida, nombre de experimento inválido, conflicto no resoluble.

La política correcta es:
- retry solo si el error es transitorio;
- número finito de reintentos;
- exponential backoff explícito y acotado;
- registro de retry_count y latency total.

Una forma razonable es:
- delay_i = base_delay * 2^i hasta max_retries;
- con un bound total visible en config o documentación.

Esto evita retry storms y hace el comportamiento operacional predecible.

15. Escritura local atómica

Aunque no exista atomicidad global entre backend MLflow y filesystem local,
la escritura de init_summary.json y manifest.json sí debe ser atómica localmente.

Pauta correcta:
- escribir en fichero temporal;
- flush y fsync si procede;
- rename atómico al destino final.

Esto evita:
- manifiestos truncados;
- resúmenes corruptos;
- y estados locales ambiguos tras interrupciones.

16. Estados parciales y recovery

El módulo debe reconocer que puede fallar a mitad de camino.

Estados intermedios útiles:
- started
- backend_validated
- experiment_resolved
- summary_written
- manifest_written
- ready
- failed

Situaciones reales:
- backend validado pero manifest no escrito;
- experimento resuelto remotamente pero evidencia local incompleta;
- summary escrito y manifest fallido;
- contradicción entre estado remoto y local.

La rutina de recovery debe poder distinguir entre:
- no inicializado;
- inicializado remotamente pero no confirmado localmente;
- inicializado y consistente;
- inconsistencia que requiere abortar.

La política correcta no es ocultar estos estados,
sino hacerlos detectables y reconciliables.

17. Política de fallbacks

La degradación debe ser explícita, no silenciosa.

Fallbacks permitidos:
- solo los declarados por configuración;
- por ejemplo, primary_tracking_uri -> secondary_tracking_uri.

Fallbacks prohibidos por defecto:
- cambiar silenciosamente a file:// local;
- renombrar el experimento para “hacerlo pasar”;
- eliminar tags obligatorios para no fallar.

En research reproducible, un fallback silencioso suele ser peor que un fallo temprano,
porque rompe trazabilidad sin que el usuario lo perciba.

18. Observabilidad

La observabilidad correcta aquí no consiste en “loggear mucho”,
sino en loggear con contexto suficiente.

Cada evento importante debería incluir:
- event_time_utc
- module = init_mlflow
- status
- tracking_uri_sanitized
- experiment_name
- experiment_id si existe
- retry_count
- latency_ms

Métricas mínimas útiles:
- init_ready_flag
- manifest_written_flag
- experiment_created_flag
- retry_count
- latency_ms
- error_class

19. Interfaz recomendada

La interfaz debe ser estrecha, tipada y explícita.

Objetos razonables:
- InitConfig
- InitContext
- InitResult

Campos mínimos de InitConfig:
- tracking_uri
- artifact_location
- experiment_name
- schema_version
- timeout_seconds
- max_retries

Campos mínimos de InitContext:
- project_name
- module_name
- git_commit
- config_hash
- data_snapshot_hash opcional

Campos mínimos de InitResult:
- status
- tracking_uri_sanitized
- artifact_location_sanitized
- experiment_name
- experiment_id
- manifest_path
- summary_path
- retry_count
- latency_ms

20. Tipos de error

Las excepciones deben estar tipadas al menos así:

- ConfigError
- CredentialError
- BackendConnectionError
- ExperimentResolutionError
- ManifestWriteError
- RecoveryError

Semántica correcta:
- ConfigError y CredentialError no se reintentan;
- BackendConnectionError puede ser retryable;
- inconsistencias remoto/local deben elevarse a RecoveryError o equivalente;
- ningún error debe dejar status = ready sin experiment_id verificable.

21. Matriz mínima de pruebas

Este módulo no debe validarse solo con casos felices.

Casos obligatorios:
- configuración válida y backend accesible;
- tracking_uri ausente;
- credenciales ausentes;
- backend inaccesible;
- experimento ya existente;
- carrera de creación con conflicto;
- fallo transitorio seguido de éxito;
- nombre de experimento inválido;
- ausencia de git_commit;
- fallo de escritura de manifest.json;
- replay idempotente completo;
- estado remoto válido con evidencia local faltante.

La calidad aquí no depende solo de coverage global,
sino de cubrir bien idempotencia, sanitización, recovery y concurrencia.

22. Complejidad y rendimiento

El coste relevante del módulo no está en estructuras algorítmicas complejas,
sino en latencia de red, round-trips al backend y escritura local.

La formalización útil aquí es simple:
- coste O(1) en metadata local por inicialización;
- más coste de llamadas remotas;
- más coste de reintentos si el backend falla transitoriamente.

Eso basta para documentación útil.
No hay que disfrazar este módulo de “math-heavy”.

23. Qué no debe afirmarse

init_mlflow.py no debe venderse como:
- módulo cuantitativo;
- sistema completo de experiment management;
- garantía mágica de consistencia perfecta global;
- ni capa que “resuelve toda la reproducibilidad”.

Su responsabilidad correcta es más concreta y más valiosa:
dejar inicializado, de forma determinista, segura y auditable,
el backend experimental sobre el que luego se apoyará el resto de la pipeline.

24. Conclusión

init_mlflow.py debe existir para cerrar, antes de cualquier training o logging posterior,
unas pocas preguntas críticas de infraestructura:

- qué backend exacto se va a usar;
- con qué experimento se va a registrar;
- con qué identidad mínima de configuración y commit;
- qué evidencia local queda escrita;
- y cómo se distingue un ready real de un estado parcial o fallido.

Ese es el estándar correcto para una inicialización seria de MLflow en una pipeline
profesional de research cuantitativo.
