Inicialización Determinista del Tracking Experimental en \textit{Small-Cap Swing Trading}}
\author{Research Specification}
\date{Marzo 2026}

\begin{document}

\maketitle

\section{Propósito del módulo}

El módulo \texttt{init\_mlflow.py} establece el contexto mínimo y verificable a partir del cual un experimento puede empezar a registrar métricas, parámetros, tags y artefactos en MLflow de forma reproducible.

Su objetivo no es entrenar modelos, lanzar \textit{sweeps} ni registrar artefactos complejos. Su responsabilidad es más estrecha y más importante: dejar creado, validado y documentado el \emph{plano de control experimental} sobre el que operarán los módulos posteriores.

Formalmente, el módulo debe producir un estado válido

\[
\mathcal{S}_{\mathrm{ready}}
=
(\texttt{tracking\_uri}, \texttt{artifact\_uri}, \texttt{experiment\_id}, \texttt{required\_tags}, \texttt{manifest})
\]

tal que cualquier \texttt{run} posterior herede una configuración consistente, auditable e idempotente.

\section{Objetivo operacional}

El problema que resuelve \texttt{init\_mlflow.py} puede expresarse así:

\begin{quote}
Dado un archivo de configuración, un contexto de proyecto y un conjunto de metadatos obligatorios, inicializar MLflow de manera que el sistema sepa exactamente \emph{dónde} registrar, \emph{bajo qué experimento}, \emph{con qué identidad de configuración} y \emph{con qué garantías mínimas de consistencia}.
\end{quote}

En términos prácticos, esto implica:

\begin{itemize}
    \item resolver el \texttt{tracking\_uri};
    \item resolver o crear el experimento de destino;
    \item validar accesibilidad del backend;
    \item fijar tags obligatorios;
    \item persistir un resumen de inicialización;
    \item dejar trazado si la inicialización fue completa, parcial o fallida.
\end{itemize}

\section{Principios de diseño}

Este módulo debe obedecer cinco principios:

\begin{enumerate}[label=\textbf{P\arabic*.}]
    \item \textbf{Enfoque único}: inicializa MLflow, no una mezcla difusa de trackers.
    \item \textbf{Idempotencia}: reejecutarlo con el mismo contexto no debe crear estados inconsistentes.
    \item \textbf{Explícito antes que implícito}: ningún URI, experimento o tag crítico debe derivarse silenciosamente.
    \item \textbf{Fallos observables}: toda degradación debe producir un estado y un error identificables.
    \item \textbf{Persistencia mínima suficiente}: debe dejar evidencia local de qué quedó inicializado y cómo.
\end{enumerate}

\section{Alcance y no-alcance}

\subsection{Alcance}

\texttt{init\_mlflow.py} cubre:

\begin{itemize}
    \item carga y validación de configuración;
    \item resolución de \texttt{tracking\_uri} y \texttt{artifact\_location};
    \item autenticación indirecta mediante variables de entorno o configuración segura;
    \item verificación básica de conectividad con el backend MLflow;
    \item resolución o creación del experimento;
    \item validación de tags obligatorios;
    \item escritura de \texttt{init\_summary.json} y \texttt{manifest.json}.
\end{itemize}

\subsection{No-alcance}

No corresponde a este módulo:

\begin{itemize}
    \item registrar artefactos de entrenamiento;
    \item iniciar \texttt{runs} hijos;
    \item optimizar hiperparámetros;
    \item resolver linaje de datasets;
    \item gestionar borrado de experimentos;
    \item ocultar complejidad del backend bajo abstracciones vacías.
\end{itemize}

\section{Modelo formal de entrada}

Definimos la entrada como

\[
\mathcal{I} = (\mathcal{C}, \mathcal{P}, \mathcal{E})
\]

donde:

\begin{itemize}
    \item \(\mathcal{C}\): configuración declarativa;
    \item \(\mathcal{P}\): contexto del proyecto;
    \item \(\mathcal{E}\): entorno de ejecución.
\end{itemize}

\subsection{Configuración declarativa \(\mathcal{C}\)}

La configuración debe incluir, como mínimo:

\begin{center}
\begin{tabular}{>{\ttfamily}p{4.5cm} p{8.5cm}}
\toprule
tracking\_uri & URI del servidor MLflow o backend local \\
artifact\_location & Ruta base de artefactos, si aplica \\
experiment\_name & Nombre lógico del experimento \\
required\_tags & Diccionario de tags mínimos obligatorios \\
schema\_version & Versión del esquema de configuración \\
timeout\_seconds & Timeout máximo por operación de inicialización \\
max\_retries & Número máximo de reintentos transitorios \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Contexto del proyecto \(\mathcal{P}\)}

Debe contener al menos:

\begin{itemize}
    \item nombre del proyecto;
    \item nombre del módulo o pipeline que inicializa;
    \item commit de Git o identificador equivalente;
    \item hash de configuración;
    \item hash de snapshot de datos, si ya existe en el pipeline.
\end{itemize}

\subsection{Entorno \(\mathcal{E}\)}

Incluye:

\begin{itemize}
    \item variables de entorno relevantes;
    \item credenciales necesarias para autenticar contra el backend;
    \item hostname o identidad de ejecución;
    \item marca temporal UTC.
\end{itemize}

\section{Precedencia de resolución}

Para evitar ambigüedad, la resolución del backend debe seguir un orden determinista. Para \texttt{tracking\_uri}:

\[
\texttt{tracking\_uri}^{*}
=
\begin{cases}
\texttt{CLI override} & \text{si existe} \\
\texttt{config.tracking\_uri} & \text{si existe} \\
\texttt{MLFLOW\_TRACKING\_URI} & \text{si existe} \\
\texttt{error} & \text{en otro caso}
\end{cases}
\]

El mismo principio aplica a \texttt{artifact\_location}, con la salvedad de que puede ser opcional en determinados backends.

Este orden debe quedar documentado y no puede depender de ramas implícitas del código.

\section{Credenciales y política de secretos}

Una crítica habitual en módulos de inicialización es hablar de reproducibilidad y olvidar la parte más frágil: la autenticación. La política correcta es:

\begin{enumerate}
    \item el módulo \textbf{puede leer} secretos del entorno o de un proveedor seguro;
    \item el módulo \textbf{no debe persistir} secretos en \texttt{manifest.json}, \texttt{init\_summary.json} ni logs;
    \item cualquier URI persistida debe almacenarse en versión \emph{sanitizada};
    \item si faltan credenciales necesarias, el fallo debe ser explícito y temprano.
\end{enumerate}

Sea \(\mathrm{sanitize}(\cdot)\) una función que elimina tokens, passwords o query parameters sensibles. Entonces toda persistencia de backend debe satisfacer:

\[
\texttt{persisted\_uri} = \mathrm{sanitize}(\texttt{tracking\_uri})
\]

No basta con “no imprimir secretos”; hay que impedir que entren en los ficheros de salida.

\section{Contrato de salida}

El módulo debe producir dos salidas persistentes mínimas:

\begin{enumerate}
    \item \texttt{init\_summary.json}
    \item \texttt{manifest.json}
\end{enumerate}

\subsection{\texttt{init\_summary.json}}

Resume el resultado operacional de la inicialización. Debe incluir:

\begin{itemize}
    \item \texttt{status}
    \item \texttt{tracking\_uri\_sanitized}
    \item \texttt{artifact\_location\_sanitized}
    \item \texttt{experiment\_name}
    \item \texttt{experiment\_id}
    \item \texttt{schema\_version}
    \item \texttt{resolved\_at\_utc}
    \item \texttt{latency\_ms}
    \item \texttt{retry\_count}
\end{itemize}

\subsection{\texttt{manifest.json}}

Es la pieza auditora. Debe incluir:

\begin{itemize}
    \item \texttt{project\_name}
    \item \texttt{module\_name}
    \item \texttt{git\_commit}
    \item \texttt{config\_hash}
    \item \texttt{data\_snapshot\_hash} (si existe)
    \item \texttt{required\_tags}
    \item \texttt{experiment\_name}
    \item \texttt{experiment\_id}
    \item \texttt{init\_key}
\end{itemize}

\section{Idempotencia}

La inicialización no debe depender de que “la primera vez salga bien”. Debe poder reintentarse sin inflar el estado.

Definimos la clave de idempotencia como:

\[
K_{\mathrm{init}}
=
(\texttt{tracking\_uri\_sanitized},
 \texttt{experiment\_name},
 \texttt{config\_hash},
 \texttt{git\_commit},
 \texttt{schema\_version})
\]

Si dos ejecuciones comparten \(K_{\mathrm{init}}\), entonces deben converger al mismo estado lógico:

\[
K_{\mathrm{init}}^{(1)} = K_{\mathrm{init}}^{(2)}
\Rightarrow
\mathcal{S}_{\mathrm{ready}}^{(1)} \equiv \mathcal{S}_{\mathrm{ready}}^{(2)}
\]

Esto implica dos reglas:

\begin{itemize}
    \item si el experimento ya existe, debe resolverse, no recrearse;
    \item si ya se produjo un \texttt{manifest} compatible, la reejecución debe tratarse como \emph{replay seguro}.
\end{itemize}

\section{Tags obligatorios}

Los tags obligatorios no son decoración documental; son el mínimo necesario para poder interpretar un \texttt{run} semanas después.

Definimos:

\[
T_{\mathrm{req}} =
\{
\texttt{git\_commit},
\texttt{config\_hash},
\texttt{schema\_version}
\}
\]

y opcionalmente, si la fase del pipeline ya lo permite,

\[
T_{\mathrm{opt}} =
\{
\texttt{data\_snapshot\_hash},
\texttt{project\_name},
\texttt{research\_stage},
\texttt{universe\_id}
\}
\]

Por tanto:

\[
T = T_{\mathrm{req}} \cup T_{\mathrm{opt}}
\]

La inicialización debe fallar si falta un tag obligatorio. El comportamiento correcto no es rellenarlo silenciosamente con \texttt{unknown}, sino abortar o exigir su provisión explícita.

\section{Estado interno e invariantes}

Un \texttt{init\_mlflow} serio debe mantener invariantes. Los principales son los siguientes.

\subsection{Invariante de backend válido}

Si \texttt{status = ready}, entonces el backend responde a una operación mínima de cliente dentro del timeout configurado.

\[
\texttt{ready} \Rightarrow \texttt{backend\_reachable} = 1
\]

\subsection{Invariante de experimento resuelto}

Si \texttt{status = ready}, entonces existe un \texttt{experiment\_id} único asociado al \texttt{experiment\_name} resuelto.

\[
\texttt{ready} \Rightarrow \exists!\ \texttt{experiment\_id}
\]

\subsection{Invariante de manifiesto consistente}

Si \texttt{manifest.json} se escribe, entonces su \texttt{config\_hash} y \texttt{experiment\_id} deben corresponder exactamente a la inicialización completada.

\[
\texttt{manifest\_written} \Rightarrow
(\texttt{config\_hash}, \texttt{experiment\_id})_{\mathrm{manifest}}
=
(\texttt{config\_hash}, \texttt{experiment\_id})_{\mathrm{runtime}}
\]

\section{Creación y resolución del experimento}

El punto delicado no es “crear el experimento”, sino manejar correctamente carreras, duplicidades y respuestas parciales del backend.

La secuencia correcta es:

\begin{enumerate}
    \item consultar si \texttt{experiment\_name} ya existe;
    \item si existe, recuperar su \texttt{experiment\_id};
    \item si no existe, intentar crear;
    \item si la creación responde con conflicto o estado ya existente, volver a consultar;
    \item solo declarar éxito cuando se obtiene un \texttt{experiment\_id} final y estable.
\end{enumerate}

Esta pauta implementa una semántica \texttt{get-or-create} robusta frente a concurrencia moderada.

\subsection{Semántica de atomicidad}

No hay transacción global perfecta entre servidor MLflow, sistema local y escritura de ficheros de manifiesto. Por tanto, el módulo no debe prometer atomicidad total; debe ofrecer \textbf{publicación ordenada y recuperación clara}.

La regla operativa recomendada es:

\begin{quote}
\textbf{primero} resolver backend y experimento, \textbf{después} escribir los ficheros locales de evidencia.
\end{quote}

Esto minimiza el peor caso: un manifiesto local que afirma un experimento que realmente no quedó disponible.

\section{Fallbacks y política de degradación}

La degradación debe ser explícita, nunca silenciosa.

\subsection{Fallback permitido}

Solo deben existir fallbacks declarados por configuración. Por ejemplo:

\[
\texttt{primary\_tracking\_uri} \rightarrow \texttt{secondary\_tracking\_uri}
\]

si y solo si la política lo permite.

\subsection{Fallback prohibido}

Quedan prohibidos por defecto:

\begin{itemize}
    \item cambiar de backend remoto a \texttt{file://} local sin aviso;
    \item crear un experimento alternativo con nombre modificado silenciosamente;
    \item eliminar tags obligatorios para “dejar pasar” la inicialización.
\end{itemize}

En investigación cuantitativa, un fallback silencioso suele ser peor que un fallo temprano, porque destruye la trazabilidad sin que el operador lo perciba.

\section{Semántica de reintentos}

Los reintentos deben aplicarse solo a errores transitorios. Denotemos por \(\mathcal{E}_T\) el conjunto de errores transitorios y por \(\mathcal{E}_P\) el de errores permanentes.

\[
\mathcal{E} = \mathcal{E}_T \cup \mathcal{E}_P,
\qquad
\mathcal{E}_T \cap \mathcal{E}_P = \varnothing
\]

Ejemplos razonables:

\begin{itemize}
    \item \(\mathcal{E}_T\): timeout, error temporal de red, respuesta 5xx;
    \item \(\mathcal{E}_P\): credenciales inválidas, configuración incompleta, nombre de experimento inválido.
\end{itemize}

La política correcta es:

\begin{itemize}
    \item reintentar solo si \(e \in \mathcal{E}_T\);
    \item número de reintentos finito y visible;
    \item espera entre reintentos explícita y determinista o con \textit{backoff} acotado;
    \item registrar cada intento en el resumen o logs estructurados.
\end{itemize}

\section{Estados parciales y recuperación}

El módulo debe modelar estados intermedios. Un conjunto suficiente es:

\[
\{\texttt{started}, \texttt{backend\_validated}, \texttt{experiment\_resolved}, \texttt{manifest\_written}, \texttt{ready}, \texttt{failed}\}
\]

Un fallo puede ocurrir, por ejemplo, después de resolver el experimento pero antes de escribir el manifiesto. Ese estado no es equivalente a un fallo completo; requiere reconciliación.

La rutina de recuperación debe seguir la lógica:

\[
\texttt{recover()} :
\begin{cases}
\text{si backend y experimento existen, escribir manifiesto faltante} \\
\text{si backend no responde, marcar fallo recuperable} \\
\text{si el estado local contradice el remoto, abortar con inconsistencia}
\end{cases}
\]

El principio clave es que el sistema pueda distinguir entre:

\begin{itemize}
    \item \textbf{no inicializado};
    \item \textbf{inicializado remotamente pero no confirmado localmente};
    \item \textbf{inicialización lista y consistente}.
\end{itemize}

\section{Validaciones obligatorias}

Antes de declarar \texttt{ready}, el módulo debe validar al menos:

\begin{enumerate}
    \item que la configuración carga y cumple el esquema esperado;
    \item que \texttt{tracking\_uri} está resuelto;
    \item que el backend responde;
    \item que el nombre del experimento es válido;
    \item que el experimento se puede consultar o crear;
    \item que los tags obligatorios existen;
    \item que \texttt{init\_summary.json} y \texttt{manifest.json} se pueden escribir localmente.
\end{enumerate}

\subsection{Validez del nombre de experimento}

Una política razonable consiste en imponer un patrón conservador, por ejemplo:

\[
\texttt{experiment\_name} \in \texttt{Regex}(\texttt{[A-Za-z0-9\_\-/]\{1,128\}})
\]

El objetivo no es ser ornamental, sino impedir nombres ambiguos, no portables o conflictivos.

\section{Observabilidad}

La observabilidad correcta aquí no es “muchos logs”, sino logs con contexto suficiente para diagnosticar.

Cada evento importante debe emitir, como mínimo:

\begin{itemize}
    \item \texttt{event\_time\_utc}
    \item \texttt{module = init\_mlflow}
    \item \texttt{status}
    \item \texttt{tracking\_uri\_sanitized}
    \item \texttt{experiment\_name}
    \item \texttt{experiment\_id} (si existe)
    \item \texttt{retry\_count}
    \item \texttt{latency\_ms}
\end{itemize}

Métricas mínimas útiles:

\[
\mathbb{I}(\texttt{ready}),\quad
\texttt{latency\_ms},\quad
\texttt{retry\_count},\quad
\mathbb{I}(\texttt{manifest\_written}),\quad
\mathbb{I}(\texttt{experiment\_created})
\]

\section{Interfaz propuesta}

La interfaz debe ser estrecha. Un diseño razonable es:

\begin{lstlisting}[language=Python]
from dataclasses import dataclass
from pathlib import Path
from typing import Optional

@dataclass(frozen=True)
class InitConfig:
    tracking_uri: str
    artifact_location: Optional[str]
    experiment_name: str
    schema_version: str
    timeout_seconds: float
    max_retries: int

@dataclass(frozen=True)
class InitContext:
    project_name: str
    module_name: str
    git_commit: str
    config_hash: str

@dataclass(frozen=True)
class InitResult:
    status: str
    tracking_uri_sanitized: str
    artifact_location_sanitized: Optional[str]

    experiment_id: str
    manifest_path: Path
    summary_path: Path
    retry_count: int
    latency_ms: int

def init_mlflow(
    config: InitConfig,
    context: InitContext,
    output_dir: Path,
) -> InitResult:
    ...
\end{lstlisting}

\section{Pseudocódigo operacional}

\begin{lstlisting}[language=Python]
def init_mlflow(config, context, output_dir):
    validate_config(config)
    validate_context(context)

    tracking_uri = resolve_tracking_uri(config)
    tracking_uri_sanitized = sanitize_uri(tracking_uri)

    client = build_mlflow_client(tracking_uri)
    validate_backend(client, timeout=config.timeout_seconds)

    experiment_id = get_or_create_experiment(
        client=client,
        experiment_name=config.experiment_name,
        artifact_location=config.artifact_location,
        max_retries=config.max_retries,
    )

    required_tags = {
        "git_commit": context.git_commit,
        "config_hash": context.config_hash,
        "schema_version": config.schema_version,
    }

    if context.data_snapshot_hash is not None:
        required_tags["data_snapshot_hash"] = context.data_snapshot_hash

    validate_required_tags(required_tags)

    summary = build_init_summary(...)
    manifest = build_manifest(...)

    write_json_atomic(output_dir / "init_summary.json", summary)
    write_json_atomic(output_dir / "manifest.json", manifest)

    return InitResult(
        status="ready",
        tracking_uri_sanitized=tracking_uri_sanitized,
        artifact_location_sanitized=sanitize_uri(config.artifact_location),
        experiment_name=config.experiment_name,
        experiment_id=experiment_id,
        manifest_path=output_dir / "manifest.json",
        summary_path=output_dir / "init_summary.json",
        retry_count=...,
        latency_ms=...,
    )
\end{lstlisting}

\section{Escritura atómica local}

Aunque el backend MLflow no garantice atomicidad total con el sistema local, la escritura de ficheros de resumen sí debe ser atómica a nivel local. La pauta recomendada es:

\begin{enumerate}
    \item escribir en un fichero temporal;
    \item hacer \texttt{flush} y \texttt{fsync} si procede;
    \item renombrar de forma atómica al destino final.
\end{enumerate}

Esto evita manifiestos truncados o resúmenes corruptos en caso de interrupción.

\section{Matriz mínima de pruebas}

Un módulo de inicialización serio debe probarse contra rutas felices y rutas de degradación.

\subsection{Casos obligatorios}

\begin{enumerate}
    \item configuración válida y backend accesible;
    \item \texttt{tracking\_uri} ausente;
    \item credenciales ausentes;
    \item backend inaccesible;
    \item experimento ya existente;
    \item carrera de creación con conflicto;
    \item fallo transitorio seguido de éxito en reintento;
    \item nombre de experimento inválido;
    \item ausencia de \texttt{git\_commit};
    \item escritura fallida de \texttt{manifest.json};
    \item replay idempotente completo;
    \item estado remoto válido con manifiesto local faltante.
\end{enumerate}

\subsection{Criterio de calidad}

No basta con cobertura superficial. El criterio útil es:

\begin{itemize}
    \item cobertura alta de funciones críticas;
    \item tests explícitos de idempotencia;
    \item tests de recuperación de estados parciales;
    \item tests de sanitización de secretos;
    \item tests de escritura atómica local.
\end{itemize}

\section{Errores y semántica de fallo}

Las excepciones deben ser tipadas. Un conjunto suficiente sería:

\begin{itemize}
    \item \texttt{ConfigError}
    \item \texttt{CredentialError}
    \item \texttt{BackendConnectionError}
    \item \texttt{ExperimentResolutionError}
    \item \texttt{ManifestWriteError}
    \item \texttt{RecoveryError}
\end{itemize}

Semántica correcta:

\begin{itemize}
    \item los errores de configuración o credenciales son permanentes;
    \item los errores de red pueden ser transitorios;
    \item las inconsistencias entre estado remoto y local requieren reconciliación, no ocultación;
    \item ningún error debe dejar un \texttt{status=ready} sin \texttt{experiment\_id} verificable.
\end{itemize}

\section{Conclusión}

\texttt{init\_mlflow.py} no debe vender “infraestructura industrial” ni acumular referencias laterales. Debe resolver con precisión una pregunta mucho más concreta y mucho más exigente:

\begin{quote}
¿Puede este pipeline empezar a registrar experimentos bajo una identidad de configuración, backend y experimento que podamos reconstruir exactamente más adelante?
\end{quote}

La respuesta correcta depende de pocas cosas, pero debe dejarlas completamente cerradas:

\begin{itemize}
    \item cómo se resuelve el backend;
    \item cómo se autentica sin filtrar secretos;
    \item cómo se crea o recupera el experimento sin duplicidades;
    \item qué tags son obligatorios;
    \item qué significa reintentar;
    \item qué evidencia local queda escrita;
    \item cómo se recupera un estado parcial.
\end{itemize}

Si este módulo satisface esos invariantes, entonces el resto del sistema ---\textit{sweeps}, training, artifact logging y evaluación OOS--- podrá apoyarse sobre una base experimental realmente reproducible. Esa es la propiedad valiosa.
