Especificación Cuantitativa: validation/walkforward_bias.py

1. Propósito del módulo

El objetivo de walkforward_bias.py no es resumir un protocolo walk-forward con una métrica estética ni “mejorar” el resultado OOS del backtest.
Su función es auditar si la propia geometría temporal del esquema de validación produce una estimación OOS creíble o si introduce sesgos por construcción.

En particular, el módulo debe detectar si la estimación OOS queda contaminada por:
- solapamiento económico entre train/valid y test,
- gap insuficiente respecto al horizonte real del experimento,
- efectos frontera inducidos por la geometría,
- fragilidad excesiva ante pequeñas perturbaciones del protocolo,
- o reutilización indebida de las mismas ventanas OOS.

La pregunta correcta del módulo es:

“¿El protocolo walk-forward define una estimación OOS temporalmente válida y geométricamente estable, o la aparente robustez depende de artefactos de particionado?”

2. Alcance y no-objetivos

2.1 Alcance
El módulo:
- recibe la geometría completa del protocolo walk-forward;
- usa metadatos de horizonte de label, lookbacks y holding period;
- audita train/valid/test en cada fold;
- detecta contaminación por solapamiento de información;
- verifica gap y purga efectivos;
- mide boundary effects;
- mide sensibilidad a perturbaciones geométricas;
- y emite PASS / WARN / FAIL con detalle auditable.

2.2 Fuera de alcance
El módulo no:
- mide calidad económica de la estrategia;
- no corrige leakage clásico de features directamente;
- no sustituye leakage_audit;
- no reemplaza multiple_testing o PBO;
- no optimiza la mejor geometría;
- ni debe convertirse en un sistema general de inferencia temporal.

3. Inputs del módulo

El módulo debe recibir al menos:

protocol:
- lista de folds F_k = (I_k^train, I_k^valid, I_k^test)

strategy_meta:
- label_horizon = h
- holding_period = L
- feature_lookbacks = {w_1, ..., w_p}
- rebalance_freq
- purge_rules si existen
- cualquier persistencia adicional conocida

dataset_meta:
- calendario temporal
- granularidad (D, W, M, etc.)
- timestamps válidos

config:
- thresholds y tolerancias
- reglas de embargo
- número de perturbaciones geométricas
- definición de métricas locales
- flags de logging y auditoría

4. Modelo formal

Sea una sucesión de folds:

F_k = (I_k^train, I_k^valid, I_k^test),  k = 1, ..., K

cada uno cronológicamente ordenado.

El módulo asume que el protocolo pretende estimar generalización fuera de muestra mediante repetición temporal de entrenar-validar-testear sobre ventanas sucesivas.

La validez del protocolo no depende solo de que:
max(I_k^train) < min(I_k^test)

sino de que no exista solapamiento de información económica relevante entre observaciones de train/valid y observaciones de test.

5. Intervalo efectivo de información J(t)

Este es el objeto central del módulo.

Para una observación con timestamp de decisión t, definimos el intervalo efectivo de información:

J(t) = [ t - w_eff_pre , t + w_eff_post ]

donde:

- w_eff_pre recoge toda la historia necesaria para construir la observación,
- w_eff_post recoge toda la dependencia futura relevante asociada a label y ejecución.

MVP recomendado:
w_eff_pre  = max(feature_lookbacks)
w_eff_post = max(h, L)

por tanto:

J(t) = [ t - max(feature_lookbacks), t + max(h, L) ]

Versión más rica:
si existen features con memoria especial, smoothing, inventory carry o dependencia adicional, puede ampliarse a:

J(t) = [ t - max(feature_lookbacks) - l_pre , t + max(h, L) + l_post ]

El módulo debe calcular J(t) de forma derivada exclusivamente de strategy_meta, nunca de heurísticas opacas.

6. Qué entendemos por walk-forward bias

Existe walk-forward bias cuando la estimación OOS está sesgada por geometría temporal defectuosa, aunque el pipeline no use de forma explícita información futura en las features.

Incluye al menos:

6.1 Contaminación por solapamiento
Train o valid comparten parte del conjunto de información J(t) con observaciones de test.

6.2 Gap insuficiente
La separación visible entre ventanas es inferior a la requerida por h, L y lookbacks.

6.3 Boundary effects
Las observaciones cercanas a los bordes del test se comportan de forma sistemáticamente distinta del interior, sugiriendo artefactos del fold.

6.4 Inestabilidad geométrica
Pequeñas perturbaciones del protocolo cambian materialmente la conclusión OOS.

6.5 Reuse bias
El mismo OOS se ha usado repetidamente para elegir diseño, hiperparámetros o thresholds.

7. Checks estructurales

Estos checks son no compensables. Su violación implica FAIL.

7.1 Check 1: geometría cronológica
Para cada fold k debe cumplirse:

max(I_k^train) < min(I_k^valid) <= max(I_k^valid) < min(I_k^test)

o la variante explícitamente declarada si no existe valid intermedia.

Cualquier permutación o inversión temporal es FAIL.

7.2 Check 2: gap sufficiency
Definir el gap observado:

g_k = min(I_k^test) - max(I_k^train)

y el gap requerido:

g_k^req = max( h, L, max(feature_lookbacks) ) + e_min

donde e_min es el embargo mínimo configurable.

Condición estructural:
g_k >= g_k^req

Regla MVP:
- FAIL si g_k < g_k^req
- WARN si g_k está excesivamente pegado al límite:
  g_k in [g_k^req, g_k^req + margin_warn)

7.3 Check 3: purga efectiva
Definir la región prohibida asociada al test del fold k:

P_k =
[
min(I_k^test) - e - max(feature_lookbacks),
max(I_k^test) + max(h,L)
]

Debe cumplirse:

I_k^train ∩ P_k = ∅
I_k^valid ∩ P_k = ∅

Cualquier intersección no justificada implica FAIL.

7.4 Check 4: overlap contamination
Para dos observaciones o_a y o_b, definir:

Overlap(o_a, o_b) =
| J(o_a) ∩ J(o_b) | / min( |J(o_a)| , |J(o_b)| )

A nivel de fold:

OverlapRate_k =
(1 / (|T_k| |E_k|))
sum_{o_a in T_k} sum_{o_b in E_k} 1{ Overlap(o_a,o_b) > 0 }

donde:
- T_k = observaciones de train/valid relevantes
- E_k = observaciones de test

Reglas recomendadas:
- FAIL si OverlapRate_k > 0.005
- FAIL si existe algún par con Overlap > 0.5
- WARN si OverlapRate_k > 0 pero queda por debajo del fail threshold y no hay pares severos

Estas tolerancias deben vivir en config.

8. Checks diagnósticos

Estos checks no invalidan por sí solos la geometría, pero sí pueden producir WARN o reforzar FAIL.

8.1 Boundary effects
Definir la zona frontera del test con anchura delta, por ejemplo:
delta = 10% de |I_k^test|

Sea b_{k,t} = 1 si t está en el borde y 0 si está en el interior.

Sea m_{k,t} una métrica local del fold, por ejemplo:
- contribución al PnL,
- error de predicción,
- hit-rate,
- o score local apropiado.

Definir:

Delta_k^boundary =
E[m_{k,t} | b_{k,t}=1] - E[m_{k,t} | b_{k,t}=0]

Regla MVP:
- WARN si la media absoluta de Delta_k^boundary sobre folds supera
  0.2 * std(m_{k,t})
- FAIL no automático, salvo coexistencia con otros defectos estructurales

Versión avanzada:
- añadir IC bootstrap o t-based si el tamaño por fold lo permite

8.2 Geometric stability
El módulo debe generar perturbaciones moderadas del protocolo.

MVP recomendado:
J = 8–12 variantes con cambios como:
- longitud de train ±10%
- origen del primer fold ±5 días
- embargo ±2 días
- longitud de test ±10%

Sea U(F^(j)) una utilidad o métrica OOS resumen bajo la geometría perturbada.

Definir:

Instab =
( max_j U(F^(j)) - min_j U(F^(j)) ) / median_j U(F^(j))

Regla recomendada:
- WARN si Instab > 0.25
- FAIL si Instab es extrema y coincide con otra señal dura o reuse severo

La métrica U debe fijarse ex ante y ser coherente con el objetivo del protocolo.

8.3 Reuse of OOS windows
El módulo debe auditar metadatos del proceso de research para identificar si las mismas ventanas OOS se han reutilizado repetidamente para:
- elegir hiperparámetros,
- redefinir features,
- ajustar thresholds,
- o descartar familias hasta encontrar una que “funcione”.

Regla práctica:
- WARN si el mismo OOS ha sido reutilizado más allá de un umbral razonable y no existe trazabilidad de ese uso
- FAIL si la ventana OOS promocionada como final fue claramente usada de forma iterativa e intensiva como mecanismo de selección

9. Mínimos metodológicos

El módulo debe degradar confianza si:
- K < 4 folds
- Sucede que |I_k^test| es demasiado pequeño para medir boundary effects con algo de estabilidad
- Las perturbaciones geométricas no pueden construirse de forma razonable por longitud de muestra insuficiente

En estos casos:
- puede seguir produciendo reporte,
- pero no debería conceder PASS fuerte.

10. Gobernanza

Definir:
N_hard = número de checks estructurales en FAIL
N_soft = número de checks diagnósticos en WARN

Regla recomendada:

FAIL si:
N_hard >= 1

WARN si:
N_hard = 0 y N_soft >= tau_warn

PASS si:
N_hard = 0 y N_soft < tau_warn

Regla MVP:
tau_warn = 2

Esta lógica es preferible a cualquier score lineal compuesto, porque no permite que una contaminación dura quede “pro-mediada” con señales suaves.

11. Umbrales y tolerancias

El módulo no debe vender umbrales universales.
Pero sí debe tener defaults explícitos y versionados.

Defaults razonables de arranque:
- OverlapRate_fail = 0.005
- PairOverlap_fail = 0.5
- Boundary_warn = 0.2 * std local
- Instab_warn = 0.25
- K_min = 4
- delta_boundary = 10% de test
- J_perturb = 8–12

Estos valores son iniciales y deben revisarse empíricamente, pero no pueden quedar implícitos.

12. Invariantes de implementación

El módulo debe imponer, al menos:

- J(t) derivado exclusivamente de strategy_meta
- geometría cronológica estricta
- ningún FAIL estructural compensable
- reporte granular por fold
- thresholds versionados en config
- identificación explícita de la binding failure
- reproducibilidad completa del audit report

13. Failure modes frecuentes

El módulo debe señalar explícitamente:

13.1 Lookbacks largos
Una geometría aparentemente limpia puede seguir contaminada si max(feature_lookbacks) es grande.

13.2 Labels solapados
Aunque no haya fechas repetidas, puede haber reutilización económica del mismo bloque futuro.

13.3 Holding persistence
Una posición abierta desde train puede contaminar la interpretación OOS si no se modela correctamente L.

13.4 Pocos folds
El protocolo puede parecer estable solo porque carece de potencia diagnóstica.

13.5 Reuse no documentado
La reutilización del mismo OOS destruye su valor como estimador limpio.

15. Esqueleto de implementación

def effective_information_interval(t, strategy_meta):
    w_max = max(strategy_meta["feature_lookbacks"])
    h = strategy_meta["label_horizon"]
    L = strategy_meta["holding_period"]
    return (t - w_max, t + max(h, L))


def compute_overlap(interval_a, interval_b):
    left = max(interval_a[0], interval_b[0])
    right = min(interval_a[1], interval_b[1])
    inter = max(0, right - left)
    len_a = interval_a[1] - interval_a[0]
    len_b = interval_b[1] - interval_b[0]
    denom = max(min(len_a, len_b), 1)
    return inter / denom


def audit_walkforward_bias(protocol, dataset_meta, strategy_meta, config):
    report = init_report(protocol, config)

    hard_failures = []
    soft_flags = []
    fold_summaries = []

    for fold in protocol["folds"]:
        geom = check_fold_geometry(fold, config)
        purge = check_purge_and_embargo(fold, strategy_meta, config)
        overlap = compute_information_overlap(fold, dataset_meta, strategy_meta, config)
        boundary = estimate_boundary_effect(fold, dataset_meta, strategy_meta, config)

        fold_report = merge_fold_checks(geom, purge, overlap, boundary)
        fold_summaries.append(fold_report)

        hard_failures.extend(extract_hard_failures(fold_report))
        soft_flags.extend(extract_soft_flags(fold_report))

    stability = run_geometry_perturbation_audit(protocol, dataset_meta, strategy_meta, config)
    reuse = audit_oos_reuse(protocol, strategy_meta, config)

    hard_failures.extend(stability.get("hard_failures", []))
    hard_failures.extend(reuse.get("hard_failures", []))
    soft_flags.extend(stability.get("soft_flags", []))
    soft_flags.extend(reuse.get("soft_flags", []))

    gate, binding_failure = apply_wf_governance(hard_failures, soft_flags, config)

    report["fold_summaries"] = fold_summaries
    report["geometry_stability"] = stability
    report["reuse_audit"] = reuse
    report["hard_failures"] = hard_failures
    report["soft_flags"] = soft_flags
    report["gate"] = gate
    report["binding_failure"] = binding_failure
    report["recommended_action"] = map_action(gate, binding_failure)

    return report

16. Interpretación correcta

La lectura correcta de walkforward_bias.py no es:
“hemos asignado un score geométrico al backtest”.

La lectura correcta es:
“hemos auditado si el propio protocolo walk-forward merece credibilidad como estimador OOS.”

Eso significa verificar:
- separación causal efectiva,
- ausencia de overlap económico relevante,
- gap suficiente,
- estabilidad geométrica,
- y no reutilización destructiva del OOS.

Si esa base falla, la aparente robustez del walk-forward es solo un artefacto de procedimiento.

18. Conclusión

walkforward_bias.py debe ser un auditor severo de la geometría temporal del protocolo.
No está para embellecer el walk-forward.
Está para verificar que el OOS que produces se parece, de verdad, a evidencia fuera de muestra y no a una ilusión generada por folds mal separados o reutilizados.

Si el protocolo falla aquí, el resto del stack valida una estimación contaminada.
