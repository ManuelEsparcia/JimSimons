Especificación Cuantitativa: validation/leakage_audit.py

1. Propósito del módulo

El objetivo de leakage_audit.py no es mejorar el rendimiento del sistema ni seleccionar estrategias atractivas, sino verificar que el experimento cuantitativo está temporalmente bien definido.

En particular, el módulo debe auditar que:
- toda la información usada por el pipeline habría estado realmente disponible en el instante de decisión;
- la construcción de labels no contamina las features;
- los joins y forward fills respetan disponibilidad económica real;
- la geometría de train/test evita dependencia indebida;
- y toda transformación ajustable se entrena exclusivamente dentro del fold correcto.

En un sistema cuantitativo serio, el leakage no es un problema de calidad cosmética. Es una violación de validez experimental. Si existe leakage estructural, cualquier Sharpe, IC o PnL deja de ser interpretable como evidencia económica.

2. Alcance y no-objetivos

2.1 Alcance
El módulo:
- audita disponibilidad temporal real de features;
- verifica integridad de labels;
- inspecciona joins y forward fill;
- verifica purga y embargo en splits temporales;
- audita el alcance de ajuste de transformaciones del pipeline;
- y produce una salida estructurada PASS / WARN / FAIL con causa vinculante.

2.2 Fuera de alcance
El módulo no:
- optimiza estrategias,
- no calibra modelos,
- no reemplaza revisión manual de código,
- no prueba por sí solo ausencia absoluta de leakage semántico oculto,
- ni debe convertirse en un sistema general de inferencia causal.

3. Modelo temporal y notación

Sea j una observación del dataset, normalmente asociada a una entidad i y un tiempo base t_j.

Para cada observación j definimos:

- X^(j): vector de features
- Y^(j): label
- tau_decision(j): timestamp efectivo de decisión
- tau_avail(j,k): timestamp real de disponibilidad de la componente k de X^(j)

Además, definimos tres ventanas temporales por observación:

3.1 Ventana de features
Sea:
[t_feat_start(j), t_feat_end(j)]

el intervalo que cubre toda la información temporal usada para construir X^(j).

Ejemplo:
si una feature usa retornos de los últimos L días hasta t_j, entonces:
t_feat_start(j) = t_j - L
t_feat_end(j)   = t_j

3.2 Ventana de label
Para un horizonte h:
[t_label_start(j), t_label_end(j)] = [t_j + 1, t_j + h]

3.3 Ventana total de información
Definimos:
[t_info_start(j), t_info_end(j)] =
[t_feat_start(j), t_label_end(j)]

Esta ventana resume toda la información temporal relevante asociada a la observación j.

4. Definición formal de leakage

Existe leakage si se cumple al menos una de las siguientes condiciones:

4.1 Leakage de disponibilidad
Existe una componente k tal que:
tau_avail(j,k) > tau_decision(j) - h_safety

donde h_safety >= 0 representa el margen operativo exigido por el sistema.

4.2 Leakage de label
La construcción de X^(j), de Y^(j) o de cualquier objeto derivado utiliza de forma indebida información perteneciente a:
[t_label_start(j), t_label_end(j)]

4.3 Leakage entre train y test
Existe una observación i en train y una observación j en test tales que sus ventanas relevantes se solapan de forma incompatible con la política de purga y embargo.

4.4 Leakage por transformaciones globales
Existe una transformación T con parámetros theta ajustados usando datos fuera del subconjunto train del fold correspondiente:
theta = F(D_full)
en lugar de:
theta_m = F(D_train,m)

5. Taxonomía operativa del leakage

El auditor debe cubrir, como mínimo, cinco clases:

5.1 Leakage temporal de disponibilidad
Uso de datos publicados, consolidados, corregidos o ingeridos después del instante real de decisión.

5.2 Leakage por etiquetas mal definidas
Uso encubierto del bloque futuro del label dentro de features, targets secundarios o transformaciones derivadas.

5.3 Leakage por joins y forward fill
Merges con timestamps nominales incorrectos, joins por fecha no económica o propagación excesiva de valores a través del tiempo.

5.4 Leakage entre entrenamiento y evaluación
Contaminación entre train y test por falta de purga, embargo o separación temporal efectiva.

5.5 Leakage por transformaciones ajustadas globalmente
Escalado, imputación, winsorización, PCA, rankings, neutralizaciones o thresholds aprendidos usando información del dataset completo.

6. Principios de diseño del auditor

6.1 Defensa en profundidad
El auditor debe combinar:
- invariantes deterministas,
- metadatos temporales,
- geometría de ventanas,
- checks estructurales del pipeline,
- pruebas diferenciales,
- y diagnósticos auxiliares.

6.2 Jerarquía de evidencia
No todos los checks pesan igual.

Se distinguen:
- violaciones duras: prueban inconsistencia estructural;
- violaciones blandas: indican sospecha fuerte o fragilidad;
- diagnósticos auxiliares: ayudan a priorizar investigación pero no son prueba autosuficiente.

6.3 Principio de gobernanza
Una violación dura no puede compensarse con varios checks favorables.
El auditor debe bloquear PASS cuando exista una inconsistencia estructural.

7. Checks nucleares

7.1 Check 1: disponibilidad punto-en-el-tiempo
Para toda feature X_k^(j), verificar:
tau_avail(j,k) <= tau_decision(j) - h_safety

Si una fuente carece de metadatos temporales fiables:
- marcar como UNVERIFIED,
- impedir PASS pleno,
- y elevar severidad al menos a WARN.

Salida mínima:
- número de observaciones verificadas,
- porcentaje de violaciones,
- fuentes afectadas,
- mayor violación observada,
- severidad.

7.2 Check 2: integridad de labels
Sea:
Y^(j) = g(P_{t_j+1}, ..., P_{t_j+h})

El auditor debe verificar que:
- la función de etiquetado usa exclusivamente el bloque futuro autorizado;
- ninguna feature o transformación consume total o parcialmente ese mismo bloque de forma ilegítima;
- los shifts, rollings y alineamientos temporales están correctamente orientados.

El módulo debe inspeccionar:
- horizonte efectivo del label,
- offsets temporales usados,
- dependencias de columnas derivadas,
- y coherencia entre timestamps de X e Y.

7.3 Check 3: joins y forward fill
Para cada join entre tabla left y tabla right, debe cumplirse la relación económica correcta de disponibilidad:

tau_left >= tau_avail,right

según el modelo temporal de la fuente derecha.

Además, el forward fill por feature k debe satisfacer:
Delta_ffill(k) <= Delta_max(k)

donde Delta_max(k) depende del tipo de dato.
Ejemplos:
- dato cuasi-estático: tolerancia moderada
- fundamental/evento sensible: tolerancia baja
- dato de mercado: tolerancia prácticamente nula salvo diseño explícito

Violaciones graves:
- joins usando fecha contable en vez de fecha de disponibilidad real
- forward fill económicamente injustificable

7.4 Check 4: purga y embargo
Sea un split temporal con conjuntos train y test.

El auditor debe verificar que para todo par (i en train, j en test) se respeta la política de purga y embargo.

Versión mínima:
si la parte futura relevante del sample train contamina el bloque test, hay violación.

Versión operativa:
t_label_end(i) + e <= t_label_start(j)

donde e >= 0 es el embargo.

En versión avanzada, el chequeo debe considerar no solo la ventana de label, sino la ventana total de información cuando corresponda.

7.5 Check 5: ajuste de transformaciones dentro del fold
El pipeline debe representarse al menos lógicamente como un DAG de transformaciones.

Para cada nodo m con parámetros entrenables theta_m, debe verificarse:
theta_m = F(D_train,m)

y nunca:
theta_m = F(D_train,m union D_valid,m)
ni:
theta_m = F(D_full)

Esto aplica a:
- escalado
- winsorización
- imputación
- PCA u otras reducciones
- neutralización cross-sectional
- rankings globales
- calibración de thresholds
- cualquier estadístico aprendido de datos

Toda transformación ajustada fuera de su train-scope es FAIL estructural.

8. Geometría de independencia train/test

En series financieras, “train/test independientes” no significa solo orden cronológico.

El auditor debe entender que la dependencia puede venir de:
- labels solapados,
- ventanas rolling compartidas,
- persistencia de holdings,
- autocorrelación de features,
- y contaminación por transformaciones cross-sectional.

Por tanto, la validación correcta exige:
- splits temporales ordenados,
- purga compatible con horizonte de label,
- embargo cuando la dependencia residual lo requiera,
- y verificación algorítmica de estas condiciones.

9. Diagnósticos auxiliares

9.1 Pruebas diferenciales de pipeline
El auditor puede ejecutar el pipeline bajo configuraciones más estrictas:
- mayor h_safety,
- menor tolerancia de forward fill,
- purga mayor,
- embargo mayor,
- y transformaciones más estrictamente acotadas al fold.

Si la performance colapsa de forma anómala bajo endurecimientos temporalmente plausibles, se genera un soft flag de fragilidad o sospecha de leakage.

9.2 Información mutua y dependencia adelantada
Puede medirse, como diagnóstico auxiliar:
I(X_k^(j); Y^(j))

o su equivalente estimado sobre muestras.

Pero:
- no debe usarse como prueba única,
- no debe tener umbral universal,
- y, si se usa, conviene compararlo con referencias por permutación o shuffled labels.

9.3 Decay de performance
Definimos:
R_decay = S_OOS / max(S_IS, eps)

Un valor muy bajo puede reflejar:
- overfitting,
- cambio de régimen,
- o leakage.

Es un indicador blando, nunca una demostración.

9.4 PBO / CSCV
Si la infraestructura existe, puede incorporarse Probability of Backtest Overfitting (PBO) como evidencia complementaria.
Pero debe interpretarse correctamente:
- PBO alto no prueba leakage estructural;
- simplemente refuerza sospecha de invalidez experimental cuando coincide con otros flags.

10. Sistema de severidad

Sea H el conjunto de checks duros y S el conjunto de checks blandos.

Definimos:
N_hard = sum_{c in H} 1{c viola invariante}
N_soft = sum_{c in S} w_c 1{c sospechoso}

con pesos w_c en (0,1] usados solo para priorización.

Reglas de gobernanza:

FAIL si:
N_hard >= 1

WARN si:
N_hard = 0
y N_soft >= tau_warn

PASS si:
N_hard = 0
y N_soft < tau_warn

Importante:
- los diagnósticos auxiliares pueden elevar severidad,
- pero nunca reducirla,
- y un hard fail no puede compensarse.

11. Invariantes de implementación

El auditor debe imponer, al menos, estos invariantes:

- ninguna fuente sin metadatos temporales suficientes puede recibir PASS pleno;
- cualquier transformación global pre-split provoca FAIL;
- cualquier overlap no purgado incompatible con el split provoca FAIL;
- cualquier join con disponibilidad posterior a la decisión provoca FAIL;
- cualquier label contaminado con información no permitida provoca FAIL;
- la salida final debe identificar la causa exacta, la localización y el tipo de fallo.

12. Failure modes y límites del auditor

El módulo debe reconocer explícitamente sus límites.

12.1 Lo que sí detecta bien
- violaciones temporales explícitas,
- joins mal alineados,
- transformaciones ajustadas fuera del fold,
- purga/embargo incorrectos,
- parte importante del leakage estructural reproducible.

12.2 Lo que puede no detectar directamente
- leakage puramente semántico escondido en código arbitrario,
- metadata incompleta o incorrecta de proveedores,
- contaminación lógica no representada en pipeline_spec,
- supuestos económicos falsos aunque temporalmente consistentes.

En estos casos, el auditor debe:
- degradar confianza,
- marcar UNVERIFIED,
- y recomendar revisión manual.

13. Salida del módulo

La salida debe ser auditable y granular.
Formato recomendado:

{
  "gate": "PASS|WARN|FAIL",
  "hard_failures": [...],
  "soft_flags": [...],
  "sources_unverified": [...],
  "timestamp_summary": {...},
  "label_alignment_summary": {...},
  "join_audit_summary": {...},
  "split_purge_summary": {...},
  "transform_fit_summary": {...},
  "diagnostics": {
      "performance_decay": ...,
      "mutual_information_flags": ...,
      "stress_of_leakage_flags": ...,
      "pbo_flags": ...
  },
  "binding_failure": "...",
  "recommended_action": "..."
}

Cada fallo debe incluir:
- check afectado,
- tipo de severidad,
- fuente o transformación implicada,
- localización temporal,
- descripción corta,
- acción recomendada.

14. Esqueleto de implementación

def audit_leakage(dataset, feature_meta, split_spec, pipeline_spec, config):
    report = init_report()

    report["timestamp_summary"] = check_point_in_time_availability(
        dataset, feature_meta, config
    )

    report["label_alignment_summary"] = check_label_feature_alignment(
        dataset, pipeline_spec, config
    )

    report["join_audit_summary"] = audit_joins_and_forward_fill(
        dataset, feature_meta, config
    )

    report["split_purge_summary"] = check_purged_embargo_splits(
        dataset, split_spec, pipeline_spec, config
    )

    report["transform_fit_summary"] = audit_transform_fit_scope(
        pipeline_spec, split_spec
    )

    report["diagnostics"] = run_auxiliary_diagnostics(
        dataset, split_spec, pipeline_spec, config
    )

    gate, binding_failure = apply_leakage_governance(report, config)

    report["gate"] = gate
    report["binding_failure"] = binding_failure
    report["recommended_action"] = map_action(gate, binding_failure)

    return report

15. MVP vs versión avanzada

15.1 MVP
- checks 1–5 implementados sobre metadatos y pipeline_spec,
- severidad basada en N_hard / N_soft,
- soporte básico de purga y embargo,
- MI opcional en pocas features,
- sin PBO obligatorio,
- sin stress múltiple pesado.

15.2 Versión avanzada
- ventanas de información completas por observación,
- verificación exhaustiva de purged CV,
- DAG más rico de transformaciones,
- stress de leakage bajo varias configuraciones,
- integración opcional con PBO/CSCV,
- trazabilidad más detallada de dependencia temporal entre samples.

16. Interpretación correcta

La interpretación correcta de leakage_audit.py es:

“Este módulo no busca estrategias rentables. Busca demostrar que el experimento cuantitativo no está contaminado por información ilegítima.”

Si leakage_audit falla, el backtest deja de tener valor epistemológico.
Primero se demuestra validez temporal.
Después, y solo después, se discute rendimiento.

17. Referencias orientativas

- López de Prado, M. (2018), Advances in Financial Machine Learning
- Bailey et al., trabajos sobre backtest overfitting y PBO
- documentación y literatura sobre purged k-fold CV y embargo
- referencias operativas sobre PIT data y disponibilidad real de fuentes

18. Conclusión

leakage_audit.py debe ser un módulo severo, interpretable y automatizable.
No debe convertirse en un paper teórico ni en un dashboard decorativo.
Su función es más básica y más importante:

verificar que el pipeline respeta causalidad temporal, separación legítima entre entrenamiento y evaluación, y alcance correcto de cada transformación.

Si esa base falla, cualquier resultado posterior es irrelevante.
