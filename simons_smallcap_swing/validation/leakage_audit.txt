\section{Prop\'osito del m\'odulo}

El objetivo de \texttt{leakage\_audit.py} no es ``mejorar el Sharpe'' ni seleccionar
estrategias atractivas, sino verificar que la informaci\'on utilizada por el pipeline
habr\'ia estado realmente disponible en el instante de decisi\'on y que la geometr\'ia
de entrenamiento, validaci\'on y etiquetado no introduce dependencia espuria entre pasado
y futuro.

En un sistema cuantitativo serio, el leakage no es un problema cosm\'etico: invalida la
interpretaci\'on causal de cualquier backtest. Una estrategia con leakage puede parecer
estable, robusta y estad\'isticamente significativa, cuando en realidad solo explota
informaci\'on inaccesible ex ante o una contaminaci\'on de muestras mal definida.
Por ello, este m\'odulo debe tratar el leakage como una \textbf{violaci\'on de validez del
experimento}, no como un simple warning de calidad.

\section{Definici\'on formal de leakage}

Sea $t$ el instante en que se forma una decisi\'on de trading y sea $Y_{t+h}$ la variable
objetivo a horizonte $h$. Sea $X_t$ el vector de features supuestamente disponible en $t$.
Diremos que existe leakage si alguna componente de $X_t$ depende, directa o indirectamente,
de informaci\'on cuya disponibilidad real satisface
\[
\tau_{\mathrm{avail}}(X_t^{(k)}) > t,
\]
o si la construcci\'on de conjuntos de entrenamiento y test permite que la p\'erdida o la
selecci\'on del modelo utilice observaciones temporalmente ligadas al bloque evaluado fuera
de muestra.

Equivalentemente, el principio central del m\'odulo es:

\begin{quote}
\emph{Ninguna transformaci\'on del pipeline puede usar, de forma expl\'icita o impl\'icita,
informaci\'on cuya disponibilidad real sea posterior al instante de decisi\'on ni permitir
dependencia indebida entre entrenamiento y evaluaci\'on.}
\end{quote}

\section{Taxonom\'ia operativa del leakage}

El m\'odulo debe auditar, como m\'inimo, cinco clases de leakage.

\subsection{Leakage temporal de disponibilidad}

Ocurre cuando una feature utiliza datos publicados, consolidados o corregidos despu\'es de la
hora efectiva de decisi\'on. La condici\'on correcta no es solo ``fecha del dato $\leq$ fecha
de la barra'', sino:
\[
\tau_{\mathrm{avail}}(X_{i,t}^{(k)}) \leq \tau_{\mathrm{decision}}(i,t) - h_{\mathrm{safety}},
\]
donde $h_{\mathrm{safety}} \geq 0$ representa el margen operativo exigido por el sistema
(latencia de proveedores, procesamiento, clock alignment, colas de ingesta, etc.).

\subsection{Leakage por etiquetas mal definidas}

Aparece cuando la variable objetivo y las features comparten informaci\'on solapada de forma
incorrecta. Si el label se define, por ejemplo, como
\[
Y_t = f(P_{t+1},\dots,P_{t+h}),
\]
entonces ninguna transformaci\'on de features en el instante $t$ puede incorporar de forma
encubierta parte de ese mismo intervalo futuro.

\subsection{Leakage por joins y forward fill}

Es frecuente en merges de fundamentales, datos corporativos, anal\'iticos o datasets alternativos.
Si una tabla de eventos se une a precios por ``fecha de referencia'' en lugar de por fecha real
de disponibilidad, el pipeline puede filtrar futuro sin que sea evidente en el c\'odigo.
Igualmente, un \texttt{forward fill} no controlado puede propagar informaci\'on m\'as all\'a
de su ventana econ\'omicamente justificable.

\subsection{Leakage entre train y test}

Aparece cuando la partici\'on temporal no purga observaciones ligadas por horizonte de label,
solapamiento de barras, holdings persistentes o efectos de autocorrelaci\'on de features.
En series financieras no basta con un split cronol\'ogico ingenuo; deben imponerse purga y,
cuando proceda, embargo temporal.

\subsection{Leakage por normalizaci\'on o transformaciones globales}

Cualquier transformaci\'on ajustada con estad\'isticas de todo el dataset antes del split
(escalados, winsorizaciones, PCA, imputaciones, neutralizaciones, rankings globales, etc.)
puede filtrar informaci\'on futura. Toda transformaci\'on dependiente de datos debe ajustarse
\emph{solo} en el subconjunto de entrenamiento pertinente.

\section{Principios de dise\~no del auditor}

\subsection{Defensa en profundidad}

El leakage no debe modelarse como un check binario \'unico. El auditor debe combinar:
\begin{itemize}[leftmargin=2em]
    \item invariantes deterministas,
    \item verificaciones estructurales de metadatos,
    \item tests sobre la geometr\'ia temporal del dataset,
    \item pruebas diferenciales de pipeline,
    \item diagn\'osticos estad\'isticos auxiliares.
\end{itemize}

La raz\'on es sencilla: algunos leaks son demostrables por l\'ogica temporal, mientras que otros
solo aparecen por contradicciones entre versiones del pipeline o por patrones de rendimiento
demasiado fr\'agiles para ser plausibles.

\subsection{Jerarqu\'ia de evidencia}

No todos los tests tienen el mismo peso. El m\'odulo debe distinguir entre:
\begin{enumerate}[leftmargin=2em]
    \item \textbf{Violaciones duras}: prueban inconsistencia temporal o contaminaci\'on estructural.
    \item \textbf{Violaciones blandas}: no prueban leakage por s\'i solas, pero son suficientemente
    sospechosas como para requerir revisi\'on.
    \item \textbf{Diagn\'osticos auxiliares}: ayudan a priorizar investigaci\'on, pero no deben
    usarse como prueba \'unica.
\end{enumerate}

\section{Checks nucleares}

\subsection{Check 1: disponibilidad temporal punto-en-el-tiempo}

Para cada feature $X_{i,t}^{(k)}$, el auditor debe verificar la desigualdad:
\[
\tau_{\mathrm{avail}}(X_{i,t}^{(k)}) \leq \tau_{\mathrm{decision}}(i,t) - h_{\mathrm{safety}}.
\]

Este check requiere que cada fuente relevante transporte metadatos de disponibilidad real,
no solo timestamp nominal. Si dichos metadatos no existen, el m\'odulo debe degradar la
confianza del check y marcar la fuente como \texttt{UNVERIFIED}.

\paragraph{Salida esperada.}
\begin{itemize}[leftmargin=2em]
    \item porcentaje de filas v\'alidas,
    \item recuento de violaciones,
    \item peores fuentes,
    \item m\'aximo retraso negativo observado,
    \item clasificaci\'on \texttt{PASS/WARN/FAIL}.
\end{itemize}

\subsection{Check 2: integridad de labels}

Sea $Y_{i,t}^{(h)}$ el label a horizonte $h$. Debe cumplirse que la ventana usada para construir
el target no se filtra hacia las features contempor\'aneas o hacia transformaciones posteriores.
Un invariante b\'asico es que la funci\'on de etiquetado sea reproducible a partir de precios
futuros \emph{solo} en el bloque permitido:
\[
Y_{i,t}^{(h)} = g\big(P_{i,t+1},\dots,P_{i,t+h}\big),
\]
sin que ninguna etapa de feature engineering consuma total o parcialmente ese mismo intervalo.

\paragraph{Implementaci\'on.}
El auditor debe inspeccionar:
\begin{itemize}[leftmargin=2em]
    \item horizon efectivo del label,
    \item offsets usados en features rezagadas,
    \item alineaci\'on entre timestamps de features y target,
    \item presencia de operaciones de \texttt{shift}, rolling o joins susceptibles de
    invertirse de signo por error.
\end{itemize}

\subsection{Check 3: joins y propagaci\'on temporal}

En datasets combinados, cada merge debe auditarse con una regla del tipo:
\[
\tau_{\mathrm{left}} \geq \tau_{\mathrm{avail,right}}
\]
bajo la l\'ogica econ\'omica correcta de la fuente derecha. Esto es especialmente importante
en fundamentales, estimaciones de analistas, corporate actions, short interest y datos
alternativos.

Adem\'as, el auditor debe parametrizar la propagaci\'on permitida:
\[
\Delta_{\mathrm{ffill}}^{(k)} \leq \Delta_{\max}^{(k)},
\]
donde $\Delta_{\max}^{(k)}$ depende del tipo de dato. Un forward fill de una sesi\'on puede
ser razonable para un campo cuasi-est\'atico; uno de varias semanas puede ser leakage o, como
m\'inimo, mala especificaci\'on econ\'omica.

\subsection{Check 4: purga y embargo en validaci\'on temporal}

Sea un label a horizonte $h$ y sean bloques de entrenamiento y test definidos sobre intervalos
temporales. Para evitar contaminaci\'on, el conjunto de entrenamiento debe excluir observaciones
cuyas ventanas de informaci\'on o de label interfieran con el test. Formalmente, si una muestra
de entrenamiento termina en $t_i^{\mathrm{end}}$ y el test comienza en $t_j^{\mathrm{start}}$,
debe respetarse una separaci\'on efectiva:
\[
t_i^{\mathrm{end}} + h + e \leq t_j^{\mathrm{start}},
\]
donde $e \geq 0$ es el embargo adicional.

Aqu\'i $h$ y $e$ no son constantes universales; deben derivarse de:
\begin{itemize}[leftmargin=2em]
    \item horizonte de label,
    \item holding period esperado,
    \item persistencia de se\~nales,
    \item frecuencia de rebalanceo,
    \item dependencia serial observada.
\end{itemize}

\subsection{Check 5: ajuste de transformaciones dentro del fold}

Toda transformaci\'on dependiente de datos debe cumplir:
\[
\theta_{\mathrm{transform},m} = \mathcal{F}(D^{\mathrm{train}}_m),
\]
y solo despu\'es aplicarse a
\[
D^{\mathrm{valid}}_m,\; D^{\mathrm{test}}_m.
\]

Esto incluye, entre otros:
\begin{itemize}[leftmargin=2em]
    \item estandarizaci\'on,
    \item winsorizaci\'on,
    \item imputaci\'on,
    \item PCA y otras reducciones de dimensi\'on,
    \item neutralizaciones cross-sectional,
    \item rankings o cuantiles estimados globalmente,
    \item calibraci\'on de thresholds.
\end{itemize}

Si el auditor detecta que una transformaci\'on se ajusta antes del split, el resultado debe ser
\texttt{FAIL} estructural.

\section{Diagn\'osticos auxiliares}

\subsection{Pruebas diferenciales de pipeline}

Un recurso muy potente consiste en recomputar el pipeline bajo versiones m\'as estrictas:
\begin{itemize}[leftmargin=2em]
    \item con mayor latencia de seguridad,
    \item con joins truncados,
    \item con ventanas de ffill reducidas,
    \item con purga y embargo m\'as severos,
    \item con todas las transformaciones re-ajustadas estrictamente por fold.
\end{itemize}

Si el rendimiento colapsa bruscamente bajo versiones m\'as plausibles temporalmente, ello no
prueba leakage de forma l\'ogica, pero s\'i constituye evidencia fuerte de fragilidad o de
especificaci\'on inconsistente.

\subsection{Medidas de dependencia estad\'istica}

Medidas como correlaci\'on adelantada, informaci\'on mutua o tests de permutaci\'on pueden ser
\'utiles como se\~nal de alerta:
\[
I(X_t^{(k)};Y_{t+h}) =
\sum_{x,y} p(x,y)\log\frac{p(x,y)}{p(x)p(y)}.
\]

Sin embargo, el auditor no debe usar un umbral fijo del tipo ``$I>0.1$ implica leakage'' como
regla universal. Esa cantidad depende de discretizaci\'on, tama\~no muestral, no linealidad,
estructura de dependencia y definici\'on del label. Su funci\'on correcta es auxiliar:
priorizar investigaci\'on, no certificar culpabilidad.

\subsection{Decay de performance}

La comparaci\'on entre rendimiento in-sample y out-of-sample tambi\'en puede ser informativa.
Sea
\[
R_{\mathrm{decay}} = \frac{S_{\mathrm{OOS}}}{\max(S_{\mathrm{IS}},\varepsilon)}.
\]

Un valor bajo de $R_{\mathrm{decay}}$ puede indicar sobreajuste, cambio de r\'egimen, mala
especificaci\'on o leakage. Por s\'i solo, no permite distinguir estas causas. Por tanto,
debe tratarse como \textbf{diagn\'ostico blando} y nunca como prueba autosuficiente.

\section{Sistema de severidad}

En lugar de una puntuaci\'on arbitraria, se recomienda construir una matriz de incidencias.
Sea $\mathcal{H}$ el conjunto de checks duros y $\mathcal{S}$ el conjunto de checks blandos.

\[
N_{\mathrm{hard}} = \sum_{c \in \mathcal{H}} \mathbf{1}\{c \text{ viola invariante}\},
\qquad
N_{\mathrm{soft}} = \sum_{c \in \mathcal{S}} \omega_c \mathbf{1}\{c \text{ sospechoso}\},
\]
con pesos $\omega_c \in (0,1]$ solo para priorizaci\'on.

La gobernanza recomendada es:
\[
\texttt{FAIL} \iff N_{\mathrm{hard}} \geq 1,
\]
\[
\texttt{WARN} \iff N_{\mathrm{hard}} = 0 \;\; \text{y} \;\; N_{\mathrm{soft}} \geq \tau_{\mathrm{warn}},
\]
\[
\texttt{PASS} \iff N_{\mathrm{hard}} = 0 \;\; \text{y} \;\; N_{\mathrm{soft}} < \tau_{\mathrm{warn}}.
\]

Este esquema es mejor que un score agregado opaco porque evita compensar una violaci\'on temporal
grave con varios checks menores favorables.

\section{Casos l\'imite que el m\'odulo debe cubrir}

\begin{enumerate}[leftmargin=2em]
    \item \textbf{Corporate actions retroajustadas.} Una serie ajustada por dividendos o splits
    puede introducir informaci\'on posterior si no se trata correctamente.
    \item \textbf{Restatements fundamentales.} Debe distinguirse entre fecha fiscal, fecha de
    publicaci\'on y fecha de ingesta efectiva.
    \item \textbf{Delistings y survivorship.} La ausencia de nombres desaparecidos puede inducir
    sesgo similar a leakage econ\'omico.
    \item \textbf{Asincron\'ia entre mercados y horarios.} Una feature macro o sectorial puede
    parecer contempor\'anea cuando en realidad se publica tras el cierre relevante.
    \item \textbf{Universe selection ex post.} Un filtro de liquidez calculado con informaci\'on
    futura tambi\'en constituye leakage, aunque no aparezca en las features finales.
    \item \textbf{Ranking cross-sectional con datos incompletos.} El orden relativo puede usar
    observaciones que a\'un no estaban disponibles para todos los nombres.
\end{enumerate}

\section{Invariantes de implementaci\'on}

El auditor debe garantizar al menos los siguientes invariantes:

\begin{itemize}[leftmargin=2em]
    \item ninguna fuente sin metadatos temporales suficientes puede recibir \texttt{PASS} pleno;
    \item cualquier transformaci\'on global pre-split provoca \texttt{FAIL};
    \item cualquier overlap no purgado entre ventana de label y test provoca \texttt{FAIL};
    \item cualquier merge con disponibilidad posterior a la decisi\'on provoca \texttt{FAIL};
    \item los diagn\'osticos estad\'isticos solo elevan severidad, nunca la reducen;
    \item el resultado final debe identificar la causa exacta del fallo y su localizaci\'on
    en el pipeline.
\end{itemize}

\section{Esqueleto de implementaci\'on}

\begin{verbatim}
def audit_leakage(dataset, feature_meta, split_spec, pipeline_spec, config):
    report = init_report()

    report["timestamp_summary"] = check_point_in_time_availability(
        dataset, feature_meta, config
    )

    report["label_alignment_summary"] = check_label_feature_alignment(
        dataset, pipeline_spec, config
    )

    report["join_audit_summary"] = audit_joins_and_forward_fill(
        dataset, feature_meta, config
    )

    report["split_purge_summary"] = check_purged_embargo_splits(
        dataset, split_spec, pipeline_spec, config
    )

    report["transform_fit_summary"] = audit_transform_fit_scope(
        pipeline_spec, split_spec
    )

    report["diagnostics"] = run_auxiliary_diagnostics(
        dataset, split_spec, pipeline_spec, config
    )

    gate, binding_failure = apply_leakage_governance(report, config)

    report["gate"] = gate
    report["binding_failure"] = binding_failure
    report["recommended_action"] = map_action(gate, binding_failure)

    return report
\end{verbatim}

\section{Interpretaci\'on correcta}

La lectura correcta de \texttt{leakage\_audit.py} no es:
\begin{quote}
``este m\'odulo selecciona estrategias m\'as rentables''.
\end{quote}

La lectura correcta es:
\begin{quote}
``este m\'odulo comprueba si el experimento es temporalmente v\'alido; si falla, cualquier
m\'etrica de backtest deja de ser interpretable como evidencia econ\'omica''.
\end{quote}

Esa diferencia es esencial. En validaci\'on cuantitativa, primero se demuestra que el
experimento est\'a bien definido; solo despu\'es tiene sentido discutir rendimiento.

\section{Conclusi\'on}

Un auditor serio de leakage no debe vender ``SOTA industrial'' ni prometer que, tras filtrar
errores, aparecer\'an Sharpe elevados. Su tarea es bastante m\'as importante: verificar que el
pipeline respeta causalidad temporal, separaci\'on correcta entre entrenamiento y evaluaci\'on,
y alcance leg\'itimo de cada transformaci\'on. Si esa base falla, todo lo dem\'as es irrelevante.
