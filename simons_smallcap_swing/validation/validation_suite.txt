Especificación Cuantitativa: validation/validation_suite.py

1. Propósito del módulo

validation_suite.py es la capa final de gobernanza del stack de validación.

Su función no es construir un meta-score decorativo ni seleccionar “estrategias élite”.
Su función es consolidar la evidencia producida por los módulos de validación especializados y traducirla a una decisión operacional reproducible para una estrategia candidata ya congelada.

La pregunta correcta no es:

“¿qué estrategia tiene mejor score agregado?”

La pregunta correcta es:

“¿existe evidencia suficiente para considerar que esta estrategia es:
- temporalmente válida,
- estadísticamente no espuria,
- robusta bajo perturbaciones plausibles,
- y operativamente ejecutable?”

La suite debe responder esa pregunta sin permitir compensaciones indebidas entre fallos de distinta naturaleza.

2. Principio rector

La suite debe imponer una regla central:

Un fallo estructural no puede compensarse con una buena métrica en otra dimensión.

En particular:
- leakage no puede compensarse con buen OOS;
- selection bias no puede compensarse con buena capacidad;
- inviabilidad operativa no puede compensarse con buen multiple testing;
- y ningún score agregado puede convertir un FAIL estructural en PASS.

Esto es lo que separa una suite seria de un ranking cosmético.

3. Objeto de validación

La suite recibe una estrategia candidata congelada, entendida como un artefacto ya definido por:
- universo y reglas de elegibilidad,
- features y labels,
- modelo o regla de señal,
- construcción de cartera,
- sizing y rebalanceo,
- modelo de costes y ejecución,
- protocolo de evaluación temporal,
- y configuración versionada de validación.

La suite no:
- reentrena el modelo,
- no recalibra hiperparámetros,
- no reoptimiza cartera,
- no redefine familias de hipótesis,
- ni cambia la métrica objetivo.

4. Módulos que la suite integra

Una configuración mínima razonable debe consolidar, al menos, estos submódulos:

1. leakage_audit
2. walkforward_bias
3. multiple_testing
4. pbo_cscv
5. synthetic_shocks
6. capacity_sanity
7. oos_quality

La suite no reemplaza estos módulos.
Solo orquesta su salida y aplica una lógica global de decisión.

5. Contrato mínimo común de salida por submódulo

Para que validation_suite.py sea implementable sin ambigüedad, todo submódulo integrado debe exponer un schema mínimo común obligatorio.

Formato conceptual mínimo por submódulo:

{
  "gate": "PASS|WARN|FAIL",
  "metrics": {...},
  "hard_failures": [...],
  "soft_flags": [...],
  "binding_failure": "... | null",
  "recommended_action": "... | null",
  "notes": [...],
  "severity": ...   # opcional para gates blandos; obligatorio cuando aplique
}

Interpretación:
- gate: estado categórico final del submódulo;
- metrics: métricas crudas necesarias para auditoría;
- hard_failures: lista de violaciones estructurales si las hay;
- soft_flags: fragilidades o advertencias no estructurales;
- binding_failure: razón principal que explica FAIL o WARN;
- recommended_action: acción sugerida a nivel de módulo;
- notes: contexto adicional auditable;
- severity: severidad normalizada s_g in [0,1] cuando el módulo participe en agregación blanda.

Cada módulo puede añadir campos propios, pero no puede omitir este contrato base.

6. Taxonomía de gates

La suite debe distinguir tres clases de gates:

6.1 Gates estructurales
Son condiciones cuya violación invalida el experimento como tal.

Ejemplos:
- leakage confirmado,
- transformaciones ajustadas fuera del fold correcto,
- purga o embargo insuficientes,
- familia de pruebas no registrada,
- fallo crítico de integridad temporal,
- protocolo walk-forward geométricamente inválido.

Estos gates son no compensables.

6.2 Gates de fragilidad estadística
No invalidan necesariamente el experimento, pero indican que la evidencia a favor de la estrategia es frágil o probablemente espuria.

Ejemplos:
- multiple testing no superado,
- PBO alta,
- estabilidad OOS insuficiente,
- dependencia excesiva de un subperiodo o subuniverso.

6.3 Gates económicos y operativos
Evalúan si la estrategia sigue siendo utilizable bajo restricciones reales.

Ejemplos:
- capacidad insuficiente,
- costes excesivos,
- drawdowns inaceptables bajo shocks,
- sensibilidad operativa desproporcionada.

7. Conjuntos formales de gates

Definimos:

G_struct = {g_leakage, g_walkforward, g_family_integrity, ...}
G_stat   = {g_multiple_testing, g_pbo, g_oos_quality, ...}
G_econ   = {g_shocks, g_capacity, ...}

y:
G_soft = G_stat union G_econ

Para cada gate g, la suite debe registrar:
- estado categórico e_g in {PASS, WARN, FAIL}
- métricas subyacentes
- razón vinculante
- y, si aplica, una severidad normalizada s_g in [0,1]

8. Filtro duro estructural

Para gates estructurales, definimos:

h_g = 1{e_g = PASS} in {0,1}

y el filtro global duro:

H = product_{g in G_struct} h_g

Regla:

Si H = 0, entonces:
- global_gate = FAIL
- recommended_action = reject
- soft_priority_score = None

No se permite ninguna compensación posterior.

9. Severidades para gates blandos

Una vez superado el filtro duro, los módulos blandos pueden mapearse a severidades normalizadas:

s_g = f_g(m_g ; theta_g)

donde:
- m_g son las métricas crudas del submódulo,
- theta_g son thresholds y parámetros versionados,
- f_g debe ser monotónica, explícita y auditable.

Requisitos mínimos para cada f_g:
- f_g(metric ideal) = 0
- f_g(metric intolerable) = 1
- monotonicidad correcta según el sentido de la métrica
- definición documentada en config, no hardcodeada opacamente
- misma convención de escala para toda la suite: 0 = sano, 1 = intolerable

Ejemplos:

9.1 Métrica “más es mejor”
Si x = Sharpe_OOS y theta = (x_min, x_max):

s_g(x) =
    1                                  si x <= x_min
    (x_max - x) / (x_max - x_min)      si x_min < x < x_max
    0                                  si x >= x_max

9.2 Métrica “menos es mejor”
Si x = PBO y theta = (x_good, x_bad):

s_g(x) =
    0                                  si x <= x_good
    (x - x_good) / (x_bad - x_good)    si x_good < x < x_bad
    1                                  si x >= x_bad

9.3 Métrica por tramos o máximo de brechas
Cuando un submódulo tiene varias métricas relevantes, la severidad puede definirse como:
- máximo de severidades parciales,
- combinación convexa explícita,
- o regla piecewise documentada.

La suite no exige sofisticación matemática.
Exige trazabilidad, monotonicidad y coherencia entre módulos.

10. Mapeo mínimo de severidad por familia de módulos blandos

Para dejar cerrada la integración, la suite debe exigir que los siguientes módulos blandos expongan severity de forma explícita:

10.1 multiple_testing
severity basada, por ejemplo, en:
- adjusted p-values,
- DSR deficit,
- o breach de threshold principal de fragilidad inferencial.

10.2 pbo_cscv
severity basada en PBO_hat y, si aplica, su banda o IC operativo.

10.3 oos_quality
severity basada en:
- Sharpe neto OOS,
- estabilidad por subventanas,
- concentración de PnL,
- degradación vs training,
- drawdown o tail concentration si forman parte de su contrato.

10.4 synthetic_shocks
severity basada en:
- peor breach relativo bajo shocks,
- número de escenarios críticos fallidos,
- o binding critical scenario.

10.5 capacity_sanity
severity basada en:
- breach de participación,
- caída de Sharpe neto,
- capacidad efectiva,
- o binding capacity constraint.

La suite no obliga a una única fórmula universal.
Sí obliga a que cada severidad exista, quede documentada y comparta la misma semántica de escala.

11. Bundle de fragilidad estadística

La suite debe reconocer explícitamente que:
- multiple_testing,
- pbo_cscv,
- y oos_quality

no son tres fuentes ortogonales de evidencia.

Los tres módulos pertenecen al mismo bloque de fragilidad estadística, y por tanto no deben sumarse ingenuamente como si fueran independientes.

Denotamos:

G_stat_bundle = {g_multiple_testing, g_pbo, g_oos_quality}

Principio de agregación:
- este bundle debe tratarse como una familia correlacionada;
- la suite debe evitar triple penalización automática por el mismo patrón subyacente de fragilidad;
- y el resultado estadístico agregado debe interpretarse como un bloque, no como suma ciega de tres votos independientes.

12. Regla práctica para agregar el statistical fragility bundle

Dos reglas prácticas razonables son válidas; la suite debe fijar una de ellas de forma explícita en config.

12.1 Regla max-dominante
Definir la severidad del bundle como:

s_stat_bundle = max(s_multiple_testing, s_pbo, s_oos_quality)

Ventaja:
- conservadora;
- evita doble conteo;
- fácil de auditar.

12.2 Regla weighted bundle
Definir:

s_stat_bundle = alpha_1 s_multiple_testing + alpha_2 s_pbo + alpha_3 s_oos_quality
con:
alpha_i >= 0
sum alpha_i = 1

y con pesos modestos y documentados.

Regla por defecto recomendada:
usar max-dominante en MVP, y weighted bundle solo si los pesos han sido justificados y versionados.

13. Agregación blanda

Solo si H = 1, se puede definir un score auxiliar de prioridad.

La agregación debe operar sobre bloques blandos, no sobre módulos correlacionados sumados ingenuamente.

Forma recomendada:

R = omega_stat * s_stat_bundle
  + omega_shocks * s_shocks
  + omega_capacity * s_capacity

con:
omega_g >= 0
sum omega_g = 1

Interpretación:
- R pequeño = candidato relativamente priorizable entre los ya válidos
- R grande = candidato más frágil entre los supervivientes

Importante:
R no decide validez estructural.
Solo ordena entre candidatos que ya han pasado la capa dura.

14. Regla formal PASS / WARN / FAIL

Además de R, la suite debe usar contadores explícitos.

Definir:
N_soft_fail = numero de gates blandos con e_g = FAIL
N_warn      = numero de gates con e_g = WARN
s_max       = max(s_stat_bundle, s_shocks, s_capacity)

Regla base configurable:

14.1 FAIL global
Si:
- H = 0
o
- N_soft_fail >= 1
o
- N_warn >= max_warn
o
- s_max >= s_fail

entonces:
global_gate = FAIL

14.2 WARN global
Si no hay FAIL, pero:
- R > R_warn
o
- s_max in [s_warn, s_fail)
o
- existe combinación coherente de warnings relevantes

entonces:
global_gate = WARN

14.3 PASS global
Si:
- H = 1
- no hay FAIL blando
- y la evidencia blanda queda dentro de tolerancia

entonces:
global_gate = PASS

15. Dependencia entre módulos

La suite debe reconocer que algunos módulos no son evidencia independiente.

En particular:
- multiple_testing,
- pbo_cscv,
- y oos_quality

pertenecen a un mismo cluster de fragilidad estadística.

Asimismo, leakage_audit y walkforward_bias pueden estar relacionados en la capa temporal,
pero no deben doble contar el mismo defecto raíz.
La frontera correcta es:

- leakage_audit:
  causalidad PIT del pipeline, joins, labels, availability, transformaciones ajustadas fuera del fold;

- walkforward_bias:
  credibilidad geométrica del protocolo OOS, gap real, overlap contamination, boundary effects y reuse audit.

La suite debe usar esta separación conceptual para no castigar dos veces exactamente la misma infracción.

16. Score auxiliar y prioridad entre supervivientes

El score auxiliar debe existir solo para:
- ordenar candidatos PASS o WARN aceptables,
- priorizar revisión manual,
- decidir paper trading entre alternativas válidas,
- o seleccionar qué candidatos merecen recursos limitados.

No debe usarse para:
- salvar un FAIL,
- justificar promoción automática,
- ni comparar estrategias que fallaron por razones estructurales distintas.

17. OOS quality como resumen, no como argumento soberano

El bloque oos_quality debe resumir, como mínimo:
- Sharpe neto OOS,
- retorno neto anualizado,
- volatilidad,
- drawdown,
- estabilidad por subventanas,
- concentración de PnL,
- y degradación respecto al entorno de entrenamiento.

Pero debe leerse después de:
- leakage,
- multiple testing,
- PBO,
- y validación temporal correcta.

Un buen OOS por sí solo no puede dominar la suite.

18. Trampas frecuentes

La suite debe proteger explícitamente contra:

18.1 Compensación indebida
No permitir que una fortaleza operativa compense una invalidez estructural.

18.2 Doble conteo de evidencia
No tratar PBO, multiple testing y OOS como si fueran pruebas totalmente independientes.

18.3 Thresholds heredados sin contexto
Los umbrales deben venir de configuración versionada y con justificación contextual.

18.4 Registro incompleto del research
Si la familia real de pruebas está mal registrada, la corrección estadística pierde significado.

18.5 Reducir todo a un score
La validación no es un leaderboard.

18.6 Promoción sin trazabilidad
Toda decisión debe reconstruirse por strategy_id, run_id y config_hash.

19. Invariantes de implementación

La implementación debe imponer, como mínimo:

- ningún FAIL estructural puede convertirse en PASS por agregación;
- soft_priority_score solo puede calcularse si H = 1;
- todos los submódulos deben cumplir el contrato mínimo común de salida;
- toda decisión debe ser reproducible desde run_id y config_hash;
- todos los thresholds deben venir de configuración versionada;
- la suite debe conservar el detalle por módulo, no solo el estado final;
- y el reporte final debe identificar la binding constraint principal.

21. Esqueleto de implementación

def run_validation_suite(candidate, config):
    report = init_validation_report(candidate, config)

    report["module_gates"]["leakage"] = run_leakage_audit(candidate, config)
    report["module_gates"]["walkforward_bias"] = run_walkforward_audit(candidate, config)
    report["module_gates"]["multiple_testing"] = run_multiple_testing_audit(candidate, config)
    report["module_gates"]["pbo_cscv"] = run_pbo_audit(candidate, config)
    report["module_gates"]["synthetic_shocks"] = run_shock_audit(candidate, config)
    report["module_gates"]["capacity_sanity"] = run_capacity_audit(candidate, config)
    report["module_gates"]["oos_quality"] = summarize_oos_quality(candidate, config)

    enforce_common_module_schema(report["module_gates"], config)

    hard_failures = collect_structural_failures(report["module_gates"], config)

    report["structural_validity"] = {
        "passed": len(hard_failures) == 0,
        "hard_failures": hard_failures
    }

    if hard_failures:
        report["global_gate"] = "FAIL"
        report["binding_failures"] = hard_failures
        report["recommended_action"] = "reject"
        report["soft_priority_score"] = None
        return report

    soft_summary = aggregate_soft_evidence_with_bundle_logic(report["module_gates"], config)

    report["soft_priority_score"] = soft_summary["priority_score"]
    report["warnings"] = soft_summary["warnings"]
    report["binding_failures"] = soft_summary["binding_failures"]

    report["global_gate"] = decide_global_gate(soft_summary, config)
    report["recommended_action"] = map_action(report["global_gate"])

    return report

22. MVP vs versión avanzada

22.1 MVP
- taxonomía fija de gates,
- contrato mínimo común obligatorio para todos los submódulos,
- filtro duro estructural,
- funciones f_g simples y explícitas,
- statistical fragility bundle agregado por regla max-dominante,
- score auxiliar lineal solo entre supervivientes,
- regla formal de PASS/WARN/FAIL basada en contadores y máximos de severidad,
- thresholds versionados en config.

22.2 Versión avanzada
- weighted bundle para el bloque estadístico,
- confianza por módulo basada en calidad de evidencia,
- clusters de dependencia más finos entre módulos blandos,
- logging histórico de decisiones,
- revisión periódica manual de thresholds y pesos,
- pero sin optimización automática de la propia gobernanza.

23. Interpretación correcta

La lectura correcta de validation_suite.py es:

“este módulo consolida evidencia heterogénea y decide si una estrategia merece avanzar de fase, sin permitir que una debilidad estructural quede enmascarada por otras fortalezas parciales.”

La lectura incorrecta es:

“este módulo encuentra el pequeño subconjunto de estrategias heroicas con perfil top-tier.”

No está para vender narrativas.
Está para bloquear errores costosos.

25. Conclusión

validation_suite.py debe ser un módulo severo, transparente y auditable.

Primero debe verificar validez estructural.
Después debe medir fragilidad estadística y factibilidad operativa.
Y solo al final puede producir un score auxiliar entre estrategias ya válidas.

Ese orden no es decorativo.
Es la base de una gobernanza cuantitativa seria.
