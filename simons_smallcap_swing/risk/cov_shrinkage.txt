Especificación Cuantitativa: risk/cov_shrinkage.py

1. Propósito del módulo

El objetivo de risk/cov_shrinkage.py es estimar una matriz de covarianza ex-ante robusta y utilizable para:

- optimización de cartera,
- control de riesgo,
- reporting de exposición,
- y stress testing aguas abajo.

La pregunta correcta del módulo es:

“Dado un bloque temporal de retornos PIT con cobertura imperfecta y posible ruido transversal elevado, ¿qué matriz de covarianza estable y numéricamente utilizable debemos entregar al resto del stack?”

El módulo no busca maximizar Sharpe ni seleccionar estrategias.
Su función es reducir error de estimación y mejorar estabilidad numérica en la capa de riesgo.

2. Principios no negociables

2.1 Validez primero
La matriz final debe ser:
- simétrica,
- PSD o al menos corregida a PSD dentro de política explícita,
- y compatible con optimización cuadrática.

2.2 Robustez sobre sofisticación
El método canónico debe priorizar:
- trazabilidad,
- estabilidad,
- comportamiento predecible bajo cobertura imperfecta,
sobre técnicas más sofisticadas pero más frágiles.

2.3 Point-in-time estricto
La estimación debe usar exclusivamente retornos y universo disponibles en la fecha efectiva de cálculo.

2.4 Separación de responsabilidades
cov_shrinkage.py construye infraestructura de riesgo.
No:
- estima alpha,
- no selecciona targets macro,
- no hace modelado dinámico de regímenes,
- no sustituye a factor_model.py,
- ni a stress_tests.py.

2.5 Fallbacks conservadores
Si el estimador “bonito” falla, el módulo debe caer a un target más conservador, no inventar estructura frágil.

3. Qué hace y qué no hace

3.1 Qué hace
- recibe una matriz de retornos;
- filtra activos con cobertura insuficiente;
- estima covarianza muestral;
- construye un target estructural;
- aplica shrinkage lineal;
- corrige PSD si hace falta;
- computa diagnósticos;
- y persiste la matriz final con trazabilidad suficiente.

3.2 Qué no hace
- no ejecuta nonlinear shrinkage por defecto;
- no estima dinámicas GARCH multivariantes;
- no hace clustering jerárquico complejo;
- no decide allocation;
- no “predice” la covarianza futura con modelos de ML.

4. Entradas obligatorias

4.1 Retornos
Matriz:

    X in R^(T x N)

con:
- filas = fechas
- columnas = activos
- retornos PIT ya alineados temporalmente

4.2 Configuración
La config debe incluir al menos:
- lookback
- min_obs
- target_type
- shrinkage_method
- nan_policy
- psd_policy
- diag_floor
- jitter
- universe_policy
- max_condition_number
- fallback_policy

4.3 Universo efectivo
El módulo debe recibir o inferir:
- símbolos activos en la fecha de cálculo,
- máscara de elegibilidad,
- metadata suficiente para registrar cobertura.

5. Salidas obligatorias

El módulo debe persistir:

1. cov_shrunk.parquet
2. cov_sample.parquet
3. cov_target.parquet
4. diagnostics.json
5. universe_stats.json

La salida principal es:

    Sigma_final in R^(N x N)

indexada por símbolo y lista para consumo downstream.

6. Problema estadístico que resuelve

La covarianza muestral clásica:

    Sigma_sample =
    (1/(T-1)) (X - X_bar)^T (X - X_bar)

es válida en teoría, pero en práctica se vuelve muy inestable cuando:
- N no es pequeño frente a T,
- el universo rota,
- hay missingness,
- hay colinealidad,
- o el régimen de correlación es ruidoso.

Por ello, el módulo reemplaza Sigma_sample por un estimador shrinked:

    Sigma_shrink =
    (1 - delta) Sigma_sample
    + delta Sigma_target

con:
    delta in [0,1]

7. Semántica canónica del módulo

La semántica oficial del módulo es:

    mode = linear_shrinkage_canonical

Es decir:
- estimación muestral,
- target explícito,
- shrinkage lineal tipo Ledoit-Wolf / OAS / fijo,
- simetrización,
- reparación PSD,
- diagnósticos.

Razón:
esto ofrece el mejor equilibrio entre:
- rigor,
- estabilidad,
- facilidad de test,
- y utilidad operativa.

8. Targets soportados

8.1 Diagonal escalada
    Sigma_target = diag(Sigma_sample)

Preserva varianzas individuales y elimina correlaciones cruzadas.
Es el fallback conservador por defecto.

8.2 Identidad escalada
    Sigma_target = sigma_bar^2 I

donde sigma_bar^2 es la media diagonal.

Debe reservarse para fallbacks severos, no como target principal salvo configuración explícita.

8.3 Correlación constante
Sea rho_bar la media de las correlaciones válidas.
Entonces:

- diagonal = varianzas muestrales
- off-diagonal = rho_bar * sigma_i * sigma_j

Este target es útil cuando se desea capturar una correlación media sin heredar toda la inestabilidad del estimador completo.

8.4 Target factorial
Si existe un factor model ya validado:

    Sigma_target = B Omega_f B^T + D

Solo debe activarse si esa infraestructura existe y está validada aguas arriba.

9. Métodos de shrinkage soportados

9.1 Ledoit-Wolf
Default recomendado del módulo.

Ventajas:
- robusto,
- ampliamente aceptado,
- buena estabilidad cuando T y N son del mismo orden.

9.2 OAS
Adecuado cuando T es relativamente pequeño y se desea una aproximación MMSE simple.

9.3 Delta fijo
Útil para debugging, estudios controlados y comparación de sensibilidad.

9.4 Nonlinear shrinkage
Puede existir como:

    mode = nonlinear_shrinkage_experimental

pero no debe ser la semántica base del módulo canónico.
Si se implementa, debe quedar claramente marcado como experimental o fase 2.

10. Selección de método por defecto

Regla práctica razonable:

- si T >= 3N y coverage alta:
  Ledoit-Wolf con target diagonal o const_corr

- si N < T < 3N:
  Ledoit-Wolf u OAS

- si T <= N o coverage débil:
  target diagonal + shrinkage conservador + fallback preparado

- si hay factor model validado:
  target factorial opcional

No debe usarse una regla ingenua del tipo “si T < 50 entonces identidad”.
La lógica correcta depende de:
- T,
- N,
- cobertura,
- y conditioning.

11. Política de min_obs y universo

11.1 Mínimo de observaciones
Un activo i entra en el cálculo solo si:

    obs_i >= min_obs

Default razonable:
    min_obs = max(30, floor(0.6 * lookback))

11.2 Política de NaN
El comportamiento canónico debe ser:

1. filtrar activos que no alcanzan min_obs
2. trabajar sobre el subconjunto resultante
3. estimar con observaciones válidas del bloque
4. evitar imputaciones arbitrarias de correlación cero salvo fallback explícito

11.3 Universo cambiante
El módulo debe registrar:
- número de activos de entrada,
- número final usados,
- cobertura media,
- cobertura mínima,
- símbolos excluidos por baja observación.

12. Algoritmo canónico

Paso 1:
validar matriz X y metadatos

Paso 2:
filtrar activos con obs < min_obs

Paso 3:
construir Sigma_sample

Paso 4:
construir Sigma_target según target_type

Paso 5:
estimar delta según shrinkage_method

Paso 6:
construir Sigma_shrink

Paso 7:
simetrizar

Paso 8:
aplicar PSD repair si hace falta

Paso 9:
añadir jitter si la condición sigue siendo mala

Paso 10:
calcular diagnósticos y persistir artefactos

13. Simetrización y PSD repair

13.1 Simetrización
Antes de cualquier corrección:

    Sigma <- 0.5 * (Sigma + Sigma^T)

13.2 Reparación espectral
Si Sigma no es PSD dentro de tolerancia:
- diagonalizar,
- truncar autovalores por debajo del floor,
- reconstruir.

Forma recomendada:

    lambda_i <- max(lambda_i, eps_psd)

o una versión relativa:
    lambda_i <- max(lambda_i, floor * lambda_max)

13.3 Jitter diagonal
Si tras PSD repair el número de condición sigue siendo excesivo:
    Sigma <- Sigma + jitter * I

Todo esto debe quedar registrado.

14. Diagnósticos obligatorios

El módulo debe computar al menos:

14.1 delta final
- valor de shrinkage efectivo

14.2 coverage
    coverage_mean
    coverage_min

14.3 número de condición
    cond_num = lambda_max / lambda_min

14.4 effective_rank
Diagnóstico recomendado:

    p_i = lambda_i / sum_j lambda_j
    effective_rank = exp( - sum_i p_i log p_i )

o alternativamente otro rango efectivo documentado.

14.5 distancia al target
Norma tipo Frobenius:

    d_target = ||Sigma_shrink - Sigma_target||_F

14.6 magnitud de PSD repair
- min_eig_before
- min_eig_after
- num_fixed_eigs

14.7 dimensión final
- N_used
- T_effective

15. Fallback policy

Los fallbacks deben ser explícitos y conservadores.

Activadores típicos:
- N_used < 2
- Sigma_sample con NaN/inf no resolubles
- método automático falla
- repair PSD excesivo
- condición numérica descontrolada tras repair

Jerarquía recomendada:
1. target diagonal escalado
2. identidad escalada
3. abortar con error controlado si ni la diagonal es fiable

Nunca debe activarse un fallback silencioso sin dejar constancia en diagnostics.

16. Políticas auto y experimentales

16.1 target_type = auto
Permitido, pero con lógica acotada y trazable.
Ejemplo razonable:
- empezar por const_corr si coverage alta
- si no, diagonal
- si factor model validado y config lo pide, factorial

No debe convertirse en un selector “inteligente” opaco sin trazabilidad.

16.2 nonlinear_shrinkage_experimental
Puede existir en fase 2, pero:
- no es contrato base,
- debe ir detrás de flag explícito,
- y sus diagnósticos deben distinguirse claramente.

17. Invariantes del módulo

Toda salida válida debe cumplir:

1. Sigma_final = Sigma_final^T dentro de tolerancia
2. lambda_min(Sigma_final) >= 0 dentro de política PSD
3. cond_num finito y reportado
4. solo activos con coverage suficiente participan
5. target, delta y fallback quedan persistidos
6. la salida se alinea exactamente con el universo final usado

18. Failure modes importantes

18.1 T muy pequeño frente a N
El shrinkage ayuda, pero no hace magia.
La salida puede seguir siendo pobre; debe quedar reflejado en diagnostics.

18.2 Cobertura heterogénea extrema
Un activo con cobertura marginal puede contaminar estabilidad global si no se filtra bien.

18.3 Colinealidad extrema
Puede destruir conditioning y exigir PSD repair fuerte.

18.4 Factor target débil
Si el target factorial entra mal validado, puede empeorar el estimador en vez de ayudar.

18.5 Uso aguas abajo ciego
optimizer.py no debe asumir que todas las matrices vienen de la misma política.

19. Tests mínimos obligatorios

19.1 Simetría
    ||Sigma - Sigma^T|| pequeño

19.2 PSD
autovalores no negativos dentro de tolerancia

19.3 Robustez ante missingness
Dado un input con NaN estructurados:
- o matriz válida
- o error controlado y trazable

19.4 Mejora o no empeoramiento fuerte de conditioning
En casos mal condicionados, el shrinkage no debe empeorar sistemáticamente cond_num de forma absurda

19.5 Persistencia
Todos los artefactos deben escribirse con índices y metadata consistentes

19.6 Reproducibilidad
Mismas entradas y config deben producir la misma salida

20. Salida recomendada de diagnostics.json

Debe incluir como mínimo:

- calc_date
- lookback
- N_input
- N_used
- T_effective
- target_type
- shrinkage_method
- delta
- coverage_mean
- coverage_min
- cond_num
- effective_rank
- min_eig_before
- min_eig_after
- num_fixed_eigs
- psd_policy
- jitter_used
- fallback_used
- fallback_reason
- config_hash

21. Integración con el stack

La ubicación semántica natural es:

returns / universe PIT
-> cov_shrinkage
-> factor_model / optimizer / risk reports / stress tests

Este módulo entrega la matriz base robusta.
No debe cargar con lógica que pertenece a:
- modelado factorial profundo,
- simulación de régimen,
- ni selección táctica de cartera.

22. Defaults razonables

- mode = linear_shrinkage_canonical
- shrinkage_method = ledoit_wolf
- target_type = diag o auto_conservative
- min_obs = max(30, floor(0.6 * lookback))
- psd_policy = eigenvalue_floor
- diag_floor = 1e-8 relativo
- jitter pequeño y explícito
- fallback_policy conservadora

23. Interpretación correcta

La lectura correcta de cov_shrinkage.py no es:
“hemos metido una técnica elegante de shrinkage”.

La lectura correcta es:
“hemos construido una matriz de covarianza utilizable y estable para que el resto del stack no optimice sobre ruido transversal puro”.

Eso es mucho más importante que tener la versión más sofisticada del estimador.

24. Conclusión

risk/cov_shrinkage.py debe ser una pieza de infraestructura cuantitativa sobria y confiable.

Recibe retornos PIT con cobertura imperfecta.
Construye la covarianza muestral.
La regulariza hacia un target estructural.
Asegura simetría y PSD.
Controla conditioning.
Aplica fallback conservador si hace falta.
Y persiste suficiente diagnóstico para que downstream sepa exactamente qué matriz está usando y por qué.

Ese es exactamente el comportamiento que debe tener una capa seria de covarianza shrinked en un stack cuantitativo institucional.
