Especificación Cuantitativa: risk/factor_model.py

1. Propósito del módulo

El objetivo de risk/factor_model.py es construir el modelo multifactor canónico de riesgo del sistema, produciendo para cada fecha t:

- matriz de exposiciones por activo B_t,
- retornos históricos de factores f_t,
- covarianza factorial Omega_f,t,
- riesgo específico D_t,
- y covarianza total:
    Sigma_t = B_t Omega_f,t B_t^T + D_t

La pregunta correcta del módulo es:

“¿Cómo descomponemos el riesgo transversal del universo en una parte común explicada por factores y una parte idiosincrática, de forma PIT, estable y utilizable downstream?”

El módulo no existe para contar una narrativa bonita sobre “factores”.
Existe para entregar objetos matemáticos:
- coherentes,
- utilizables por optimizer y reporting,
- y auditables en producción.

2. Principios no negociables

2.1 Point-in-time estricto
Toda exposición, clasificación, retorno y covarianza debe construirse exclusivamente con información disponible hasta la fecha t.

2.2 Semántica canónica estable
El módulo debe tener una versión base cerrada y comprensible antes de admitir extensiones más sofisticadas.

2.3 Separación de responsabilidades
factor_model.py construye B_t, f_t, D_t, Omega_f,t y Sigma_t.
No:
- estima alpha,
- no hace stress testing completo,
- no optimiza cartera,
- no sustituye a cov_shrinkage.py,
- ni a exposure_report.py.

2.4 Robustez sobre sofisticación
En la versión base, se prioriza:
coherencia econométrica -> estabilidad -> trazabilidad -> sofisticación.

2.5 Cobertura explícita
Símbolos nuevos, factores faltantes o exposiciones dudosas no pueden resolverse con defaults silenciosos.

3. Qué hace y qué no hace

3.1 Qué hace
- construye factores estructurales PIT;
- estima retornos diarios de factores por regresión transversal;
- calcula residuos idiosincráticos;
- estima riesgo específico;
- estima covarianza factorial;
- compone la covarianza total;
- y persiste diagnósticos suficientes para auditoría.

3.2 Qué no hace
- no usa Kalman como contrato base;
- no usa sparse PCA por defecto;
- no hace selección automática agresiva de factores en cada fecha;
- no cambia la taxonomía de factores según “lo que funcione mejor” ex post.

4. Modelo matemático

Para cada activo i y fecha t:

    r_{i,t} = alpha_{i,t} + beta_{i,t}^T f_t + epsilon_{i,t}

donde:
- r_{i,t} es el retorno del activo,
- beta_{i,t} es su exposición a K factores,
- f_t es el retorno del vector de factores,
- epsilon_{i,t} es el residuo idiosincrático.

La versión canónica puede fijar alpha_{i,t} = 0 si la convención de centrado lo justifica y se desea evitar grados de libertad innecesarios.

La covarianza total se define como:

    Sigma_t = B_t Omega_f,t B_t^T + D_t

donde:
- B_t in R^(N x K)
- Omega_f,t in R^(K x K)
- D_t = diag(sigma^2_{eps,1,t}, ..., sigma^2_{eps,N,t})

5. Taxonomía de factores

La especificación debe distinguir explícitamente entre:

5.1 Factores estructurales
Ejemplos:
- mercado,
- sector,
- industria,
- tamaño,
- momentum,
- volatilidad,
- liquidez.

5.2 Factores estadísticos
Factores latentes o componentes principales.
No forman parte del contrato base.

5.3 Factores experimentales
Derivados de investigación interna, ML, macro, sentimiento, etc.
Deben vivir detrás de flags explícitos.

Regla canónica:
la versión base del módulo usa un conjunto estructural fijo y versionado.

6. Especificación base recomendada

Para small caps swing, la versión canónica debe trabajar con un conjunto base como:

- beta de mercado rolling,
- sectores/industrias mediante dummies,
- tamaño,
- momentum,
- volatilidad,
- liquidez.

Este conjunto debe ser:
- explícito,
- estable,
- versionado,
- y no cambiar silenciosamente entre ejecuciones.

7. Construcción de exposiciones B_t

7.1 Factores categóricos
Para sectores o industrias, usar variables dummy PIT.

Si hay G grupos:
    B_{i,g,t} = 1 si el activo i pertenece al grupo g en t
              = 0 en otro caso

Debe evitarse colinealidad perfecta:
- eliminando una categoría de referencia,
- o imponiendo restricciones de suma cero de forma documentada.

7.2 Factores continuos
Para tamaño, momentum, volatilidad o liquidez, las exposiciones deben:
- winsorizarse o truncarse robustamente,
- y normalizarse transversalmente dentro de cada fecha.

Convención recomendada:
z-score robusto vía mediana y MAD:

    x_tilde_{i,t} =
        (x_{i,t} - median(x_{.,t})) / MAD(x_{.,t})

o equivalente robusto, fijado por config.

7.3 Regla importante
B_t debe ser, en la versión canónica, principalmente una matriz de exposiciones estructurales contemporáneas.
No debe convertirse en una mezcla opaca de:
- betas rolling,
- componentes latentes,
- factores estructurales,
- y señales internas,
sin taxonomía clara.

8. Beta de mercado y factores rolling

La beta de mercado sí puede estimarse por regresión temporal rolling:

    r_{i,tau} = a_i + beta_i^(mkt,t) r_{m,tau} + eta_{i,tau}

sobre una ventana:
    tau in [t-L+1, t]

donde:
- L = lookback_beta

La política base puede ser:
- OLS rolling,
- o EWLS rolling si se desea media vida.

Pero esta lógica debe reservarse a factores donde la interpretación rolling tiene sentido económico claro, no a todos los factores indiscriminadamente.

9. Estimación de retornos factoriales

9.1 Regresión transversal diaria
Dado el vector de retornos de activos r_t y la matriz de exposiciones B_t:

    r_t = B_t f_t + u_t

La estimación base de f_t es:

    fhat_t = (B_t^T W_t B_t)^(-1) B_t^T W_t r_t

9.2 Política canónica para W_t
Aquí sí conviene cerrar mejor la especificación.

Versión base recomendada:
- si no existe aún estimación fiable de varianza específica rezagada:
    W_t = I
- si sí existe:
    W_t = diag(1 / sigma^2_{eps,i,t-1})

Esta es la versión WLS canónica del módulo.

Razón:
permite corrección de heterocedasticidad transversal sin convertir el módulo en una arquitectura demasiado compleja.

9.3 Colinealidad
Si B_t^T W_t B_t es singular o muy mal condicionada, el módulo debe:
1. detectar el problema,
2. aplicar pruning de factores redundantes,
3. o regularizar la inversión de forma explícita.

Nunca debe resolverse silenciosamente.

10. Política de colinealidad y pruning

La versión canónica debe definir reglas explícitas, por ejemplo:

- eliminar factores constantes o casi constantes en la fecha,
- eliminar dummies con breadth demasiado bajo,
- eliminar columnas casi colineales si el condition number excede umbral,
- registrar qué columnas fueron descartadas.

Campos obligatorios en diagnósticos:
- num_factors_input
- num_factors_used
- dropped_factors
- collinearity_flag
- cond_BWB

11. Residuos y riesgo específico

Una vez estimado f_t:

    epsilon_{i,t} = r_{i,t} - beta_{i,t}^T f_t

La versión base debe estimar la varianza específica por EWMA:

    sigma^2_{eps,i,t} =
        lambda_eps * sigma^2_{eps,i,t-1}
        + (1 - lambda_eps) * epsilon_{i,t}^2

con:
    0 < lambda_eps < 1

Debe existir un suelo mínimo:

    sigma^2_{eps,i,t} <- max(sigma^2_{eps,i,t}, eps_D)

Así se evita:
- riesgo específico cero,
- y matrices degeneradas.

12. Covarianza factorial Omega_f,t

Sea F_t la ventana histórica de retornos factoriales.
La covarianza muestral es:

    Omegahat_f,t =
        (1/(L_f-1)) (F_t - Fbar_t)^T (F_t - Fbar_t)

La política correcta del módulo no es reinventar shrinkage dentro de cada subfunción.
Debe integrarse limpiamente con la infraestructura de covarianza ya existente.

Regla recomendada:
- estimar Omegahat_f,t,
- pasarlo por una política explícita de shrinkage / PSD,
- y registrar el método usado.

13. Integración con cov_shrinkage.py

La política base debe ser:

- `factor_model.py` construye la historia de factor returns
- `cov_shrinkage.py` puede ser el proveedor estándar de shrinkage, PSD repair y conditioning para Omega_f,t

Esto evita duplicación de lógica y mejora consistencia del stack.

14. Símbolos nuevos y universo cambiante

14.1 Problema
En small caps el universo rota constantemente.
Habrá activos con:
- poca historia,
- características parciales,
- beta de mercado no fiable,
- cobertura incompleta.

14.2 Política base correcta
Para símbolos nuevos:

1. construir exposiciones estructurales con características PIT si existen;
2. si falta beta rolling, usar proxy conservadora documentada por sector/bucket;
3. asignar riesgo específico prudente, no artificialmente bajo;
4. marcar:
   fallback_used = True

14.3 Regla crítica
Un símbolo nuevo nunca debe parecer más “seguro” que uno maduro solo porque no tiene datos.

15. Política de cobertura y NaN

15.1 Cobertura mínima
Antes de construir el modelo, verificar:
- cobertura de retornos por símbolo,
- cobertura de features por factor,
- cobertura sectorial.

15.2 Reglas prohibidas
No se debe usar como fallback silencioso:
- beta = 0 por defecto,
- riesgo específico = 0,
- borrar factores enteros sin registrar el motivo.

15.3 Cobertura degradada
Si la cobertura cae por debajo del umbral, el módulo debe:
- emitir warning,
- degradar confianza,
- o abortar si la materialidad es alta.

16. Algoritmo operativo

Paso 1:
cargar retornos, universo y features PIT

Paso 2:
aplicar filtros de cobertura y elegibilidad

Paso 3:
construir B_t estructural por fecha

Paso 4:
estimar beta rolling solo en factores donde se haya decidido explícitamente

Paso 5:
estimar f_t vía regresión transversal diaria

Paso 6:
calcular residuos epsilon_{i,t}

Paso 7:
estimar riesgo específico por EWMA

Paso 8:
estimar Omega_f,t sobre historia de f_t y regularizarla

Paso 9:
construir:
    Sigma_t = B_t Omega_f,t B_t^T + D_t

Paso 10:
simetrizar, reparar PSD si hace falta y persistir

18. Invariantes del módulo

Toda salida válida debe cumplir:

1. B_t con dimensiones correctas
2. Omega_f,t simétrica
3. D_t diagonal con diagonal estrictamente positiva
4. Sigma_t simétrica
5. Sigma_t PSD dentro de política configurada
6. toda la construcción es PIT
7. cobertura y fallbacks quedan persistidos

19. Tests mínimos obligatorios

19.1 Dimensiones
    B_t in R^(N x K)
    Omega_f,t in R^(K x K)
    D_t in R^(N x N)
    Sigma_t in R^(N x N)

19.2 Simetría y PSD
Sigma_t debe ser simétrica dentro de tolerancia y PSD tras repair.

19.3 Recuperación sintética
En un dataset sintético con factores conocidos, la regresión transversal debe recuperar aproximadamente f_t.

19.4 Riesgo específico positivo
Para todo activo elegible:
    sigma^2_{eps,i,t} >= eps_D

19.5 Símbolo nuevo
Debe activarse fallback explícito, nunca comportamiento silencioso.

19.6 PIT
Cambios futuros del dataset no deben alterar retroactivamente artefactos pasados.

21. Qué no debe hacer este módulo

La versión canónica no debe:
- mezclar factores estructurales y latentes sin semántica clara,
- prometer alpha o mejora de Sharpe como propiedad directa,
- introducir Kalman, sparse PCA o Bayesian hierarchies como contrato base,
- usar selección dinámica agresiva de factores sin trazabilidad.

22. Extensiones futuras razonables

Como flags experimentales o fase 2:
- nonlinear shrinkage sobre Omega_f
- selección de factores por criterios de información
- betas dinámicas por state-space
- riesgo específico por régimen
- factores estadísticos adicionales

Pero ninguna de estas extensiones debe contaminar el contrato mínimo.

23. Interpretación correcta

La lectura correcta de factor_model.py no es:
“hemos construido un modelo sofisticado de factores”.

La lectura correcta es:
“hemos definido de forma programable y PIT:
- qué exposiciones existen,
- cómo se estiman los retornos factoriales,
- cómo se estima el riesgo específico,
- y cómo se compone una covarianza estructurada válida.”

Ese es su rol real.

24. Conclusión

risk/factor_model.py debe ser la fuente canónica de verdad para:
- exposiciones B_t,
- retornos factoriales f_t,
- riesgo específico D_t,
- y covarianza estructurada Sigma_t.

Debe ser sobrio, estable y auditable.
Debe preferir una arquitectura clara a una sofisticación prematura.
Y debe dejar suficientemente cerradas cuatro preguntas:
1. qué factores usamos,
2. cómo construimos B_t,
3. cómo estimamos f_t y D_t,
4. cómo ensamblamos Sigma_t.

Ese es exactamente el comportamiento que debe tener una capa seria de factor model dentro de un stack cuantitativo institucional.
