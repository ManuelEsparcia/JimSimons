4. Marco formal y notaciĂłn

`Q^(f) = (Q_struct^(f), Q_cov^(f), Q_num^(f), Q_leak^(f), Q_drift^(f), Q_col^(f))`

`Q_agg^(f) = Omega(Q^(f))`

7. Integridad estructural

7.1. Schema y contrato mĂ­nimo

Debe verificarse:

- claves coherentes
- ausencia de particiones con schema incompatible

8. Cobertura y missingness

8.1. Cobertura global

La cobertura global de una feature `f` se define como:

`rho_cov^(f) =
 sum_{t in T} sum_{s in S_t} I(X_t^(f)(s) vĂĄlido)
 /
 sum_{t in T} |S_t|`

8.2. Cobertura por fecha

`rho_cov^(f,t) =
 sum_{s in S_t} I(X_t^(f)(s) vĂĄlido)
 /
 |S_t|`

8.3. Tasa de nulos

`rho_null^(f) = 1 - rho_cov^(f)`

AdemĂĄs de la media global, deben calcularse cuantiles robustos de cobertura temporal:

`Q_0.05(rho_cov^(f,t)), Q_0.50(...), Q_0.95(...)`

porque una feature puede tener cobertura media aceptable y, aun asĂ­, colapsar repetidamente en fechas crĂ­ticas. :contentReference[oaicite:10]{index=10}

8.4. ClasificaciĂłn de missingness

Los missing deben distinguirse entre:

- `structural_missing`
- `economic_missing`
- `operational_missing`
- `policy_missing`

El QC no debe tratarlos como equivalentes. Una feature sparse esperada no debe recibir el mismo castigo que una feature densa rota por fallo operativo.

8.5. Gate de cobertura

Una feature falla cobertura si se cumple cualquiera de:

`rho_cov^(f) < tau_cov_global`

`Q_0.05(rho_cov^(f,t)) < tau_cov_p05`

`breach_rate^(f) > tau_cov_breach`

donde:

`breach_rate^(f) =
 (1/|T|) sum_t I(rho_cov^(f,t) < tau_cov_date)`

Esta combinaciĂłn de cobertura media, cuantiles bajos y frecuencia de breach es parte del contrato institucional del mĂłdulo. :contentReference[oaicite:11]{index=11}

9. Sanidad numĂ©rica

9.1. Valores no finitos

Debe verificarse ausencia de:

- `inf`
- `-inf`
- `NaN` fuera de polĂ­tica explĂ­cita

9.2. Constancia o casi constancia

Una feature debe marcarse como degenerada si:

`var_panel^(f) < epsilon_var`

y tambiĂ©n conviene medir constancia por fecha para distinguir entre degeneraciĂłn persistente y colapso ocasional.

9.3. Outliers

El mĂ©todo robusto principal debe basarse en mediana y MAD:

`z_rob,t^(f)(s) =
 (X_t^(f)(s) - med(X_t^(f)))
 /
 (1.4826 * MAD(X_t^(f)) + epsilon)`

y la tasa de outliers puede definirse como:

`rho_out^(f,t) =
 (1/|S_t|) sum_{s in S_t} I(|z_rob,t^(f)(s)| > gamma_rob)`

Un outlier rate alto no implica automĂĄticamente fallo. El gate debe distinguir entre colas econĂłmicas plausibles y patologĂ­a derivada de mala construcciĂłn o corrupciĂłn. :contentReference[oaicite:12]{index=12}

10. Leakage temporal

10.1. Principio general

La prueba principal de leakage debe basarse en disponibilidad temporal real, no en correlaciones aparentes con labels. Este es uno de los puntos metodolĂłgicamente mĂĄs importantes del mĂłdulo.

10.2. Availability test

`a_t^(f,s) <= c_t`

Toda violaciĂłn implica:

`temporal_availability_breach = 1`

y el score:

`L_time^(f) =

debe satisfacer:

`L_time^(f) = 0`

para una feature canĂłnica de producciĂłn. Cualquier breach real implica `FAIL` crĂ­tico. :contentReference[oaicite:13]{index=13}

- breach temporal directo

La lĂłgica correcta es:

11. Drift temporal

11.1. MotivaciĂłn

Una feature puede ser sana hoy y dejar de ser estable si cambia la estructura del mercado, el universo o su propia definiciĂłn. El drift no siempre invalida la seĂ±al, pero sĂ­ afecta comparabilidad y calibraciĂłn.

11.2. Ventanas de referencia

El drift debe medirse entre una ventana actual y una ventana de referencia vĂĄlidas temporalmente:

`W_ref^(f), W_curr^(f)`

Si cambia la definiciĂłn (`formula_hash`, `lineage_hash`), la comparaciĂłn histĂłrica deja de ser vĂĄlida y debe activarse:

`definition_break = 1`

11.3. MĂ©tricas recomendadas

PSI:

`PSI^(f) = sum_j (p_j,curr - p_j,ref) * log(p_j,curr / p_j,ref)`

KS:

`KS^(f) = sup_x |F_curr^(f)(x) - F_ref^(f)(x)|`

JS:

`JS^(f) =
 0.5 * KL(P_curr^(f) || M^(f))
 + 0.5 * KL(P_ref^(f) || M^(f))`

con

`M^(f) = 0.5 * (P_curr^(f) + P_ref^(f))`

11.4. Score de drift

El score de drift puede agregarse como:

`Q_drift^(f) =
 omega_1 * PSI_tilde^(f)
 + omega_2 * KS_tilde^(f)
 + omega_3 * JS_tilde^(f)`

donde las mĂ©tricas estĂĄn normalizadas y ponderadas por configuraciĂłn.

El drift debe evaluarse ademĂĄs por persistencia, no sĂłlo por valor medio agregado. :contentReference[oaicite:15]{index=15}

12. Dependencias y colinealidad

12.1. CorrelaciĂłn cross-sectional

Para dos features `f_i` y `f_j`:

`rho_ij,t = corr(X_t^(f_i), X_t^(f_j))`

12.2. Colinealidad extrema persistente

No basta con una correlaciĂłn alta aislada. Debe definirse persistencia:

`persist_ij =
 (1/|T_ij|)

12.3. InterpretaciĂłn

La colinealidad no implica siempre descarte automĂĄtico. Debe distinguirse entre:

- duplicidad casi exacta
- colinealidad esperable dentro de la misma familia
- complementariedad econĂłmica pese a correlaciĂłn alta

La acciĂłn recomendada dependerĂĄ tambiĂ©n de lineage, familia y prioridad econĂłmica declarada. El gate tĂ­pico serĂĄ:

13. AgregaciĂłn de scores y lĂłgica de decisiĂłn

La lĂłgica correcta tiene tres niveles:

Nivel 1: veto crĂ­tico
Si falla integridad estructural o leakage temporal real:

Nivel 2: gates por dimensiĂłn
Se evalĂșan cobertura, sanidad numĂ©rica, drift y colinealidad.

Nivel 3: score agregado condicionado

- `pass`
- `warn`
- `fail`
- `shadow_only`

La tabla de decisiĂłn debe mapear:

- `review`
- `drop`
- `shadow_only`
