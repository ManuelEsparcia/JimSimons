Especificación Cuantitativa: features/feature_store.py

1. Propósito del módulo

El objetivo de feature_store.py no es simplemente persistir tablas de features ni servir como almacén pasivo de artefactos intermedios. Su función es más precisa y más importante:

definir la memoria experimental oficial del sistema para snapshots de features versionados, inmutables, auditables y reproducibles.

La pregunta correcta del módulo es:

“¿Podemos garantizar que todo modelo, validación o backtest consume exactamente el mismo snapshot de features que fue publicado, con unicidad lógica, aislamiento temporal, trazabilidad completa y sin posibilidad de lecturas parciales o ambiguas?”

En un sistema cuantitativo serio, la reproducibilidad no consiste solo en guardar el código.
Consiste en poder reconstruir exactamente:
- qué features existían,
- bajo qué definición,
- en qué fechas,
- con qué schema,
- a partir de qué inputs,
- bajo qué configuración,
- y qué snapshot exacto fue leído por cada consumidor downstream.

Si esa capa falla, toda validación posterior pierde fuerza epistemológica, porque el experimento deja de ser reconstruible de forma exacta.

2. Alcance y no-objetivos

2.1 Alcance

El módulo:
- materializa snapshots de features aislados por run_id;
- impone unicidad lógica de la PK (date, symbol);
- separa staging y committed con publicación atómica a nivel lógico;
- valida coherencia entre schema declarado y schema físico;
- versiona diccionario, schema y manifest;
- expone un contrato de lectura reproducible por run_id, rango temporal, columnas y modo;
- gobierna cambios de schema mediante reglas explícitas de compatibilidad;
- y conserva hashes e información de lineage suficientes para auditoría retrospectiva.

2.2 Fuera de alcance

El módulo no:
- calcula features desde datos raw;
- no valida por sí solo ausencia de leakage semántico en el cálculo upstream;
- no decide calidad económica de las señales;
- no corrige errores conceptuales del feature engineering;
- no reemplaza un lineage engine general de todo el stack;
- ni debe convertirse en un data lake genérico sin contrato cuantitativo fuerte.

3. Objeto formal almacenado

Para cada versión κ identificada por un run_id, el módulo define una familia de snapshots:

X^κ = {X_t^κ ∈ ℝ^{|S_t|×|F_κ|}}_{t∈T_κ}

donde:
- S_t es el universo elegible en fecha t,
- F_κ es el conjunto de features disponibles en la versión κ,
- T_κ es el dominio temporal cubierto por la publicación,
- y X_t^κ(s,f) es el valor persistido de la feature f para el símbolo s en la fecha t.

La premisa central del módulo es:

una vez committed, X^κ es inmutable.

Esto implica que cualquier modificación material de contenido, schema, diccionario o inputs debe producir una nueva versión κ’, nunca sobrescribir una versión ya publicada.

4. Estructura física y aislamiento

La organización física mínima recomendada es por run_id, con separación explícita entre datos y metadatos:

run_id/
  data/date=YYYY-MM-DD/
  _meta/dict/
  _meta/schema/
  _meta/manifest/

El propósito de esta estructura no es meramente organizativo.
Su propósito es imponer aislamiento físico suficiente para que:
- dos runs distintos no compartan estado mutable,
- un consumidor pueda reconstruir el snapshot exacto que leyó,
- y la trazabilidad de una versión no dependa de convenciones implícitas.

El aislamiento por run_id es una condición necesaria de reproducibilidad experimental.

5. Publicación staging → committed

Toda versión debe publicarse en dos fases:

5.1 Staging
El snapshot se escribe primero en una zona staging, donde todavía no es visible para consumidores oficiales.

5.2 Validación pre-commit
En staging se ejecutan los checks mínimos:
- unicidad de PK,
- integridad de particiones,
- coherencia schema_declared = schema_physical,
- presencia de metadatos requeridos,
- consistencia de content_hash,
- y completitud del manifest.

5.3 Commit lógico atómico
Solo si todas las validaciones pasan, el snapshot cruza a committed mediante una transición atómica a nivel lógico.

La propiedad clave es:
ningún consumidor oficial debe poder observar un estado intermedio.

Esto elimina partial reads, snapshots incompletos y contaminación por publicaciones fallidas.

6. Clave lógica canónica

La clave lógica mínima y no negociable del snapshot es:

(date, symbol)

Esto significa que:
- no puede existir duplicidad de observaciones para la misma fecha y símbolo dentro de una misma versión;
- cualquier violación de esa unicidad constituye un hard failure estructural;
- y toda semántica adicional de granularidad debe quedar explicitada si el sistema evoluciona hacia intradía u otras dimensiones.

La unicidad de (date, symbol) no es un detalle técnico menor.
Es el ancla que garantiza que el snapshot representa una vista funcional bien definida del espacio cross-sectional por fecha.

7. Schema, diccionario y contrato semántico

Todo snapshot committed debe venir acompañado, como mínimo, de:

- schema físico materializado;
- schema declarado;
- diccionario de features;
- manifest de publicación;
- hashes de configuración y lineage;
- estado de publicación;
- y content_hash del snapshot.

El diccionario de features debe permitir responder, para cada columna:
- nombre canónico,
- tipo,
- definición económica o estadística,
- convención temporal,
- unidades si aplica,
- y notas relevantes de construcción.

Sin este diccionario, el snapshot puede ser físicamente legible pero semánticamente ambiguo.

8. Contrato de lectura

La lectura debe ser explícita y reproducible.

Formato conceptual:

read(run_id, date_range, columns, mode="strict|compatible")

Interpretación mínima:
- run_id identifica la versión exacta a leer;
- date_range define el rango temporal solicitado;
- columns define la proyección deseada;
- mode controla el régimen de compatibilidad.

8.1 Modo strict
El consumidor exige coincidencia exacta con el contrato esperado:
- mismas columnas,
- mismos tipos,
- misma semántica requerida,
- sin tolerar divergencias no registradas.

8.2 Modo compatible
El consumidor tolera únicamente cambios previamente clasificados como backward-compatible bajo reglas versionadas.

El método read_latest_snapshot() debe devolver solo versiones committed.
Nunca debe exponer:
- staging,
- failed,
- quarantined,
- ni estados parciales.

9. Evolución de schema

El módulo debe distinguir explícitamente entre:

9.1 Cambios backward-compatible
Por ejemplo:
- añadir columnas opcionales,
- ampliar metadatos no obligatorios,
- refinar documentación sin romper contrato de lectura existente.

9.2 Cambios condicionalmente compatibles
Cambios que pueden ser compatibles solo para ciertos consumidores o bajo mode="compatible".

9.3 Cambios breaking
Cambios que alteran de forma material:
- nombres,
- tipos,
- cardinalidad esperada,
- semántica,
- o estructura requerida por consumidores existentes.

Estos cambios deben requerir nueva versión explícita y no pueden quedar escondidos bajo heurísticas implícitas.

10. Manifest e integridad de snapshot

Cada snapshot committed debe incluir un manifest con, como mínimo:

- run_id
- publish_success
- store_config_hash
- feature_dict_hash
- lineage_hash
- schema_hash
- content_hash
- timestamp de publicación
- estado final
- notas o warnings relevantes

El manifest no debe verse como una nota auxiliar.
Es el documento mínimo que hace auditable la publicación.

11. content_hash y reproducibilidad

El módulo debe calcular un hash fuerte del snapshot publicado.

Forma conceptual:

content_hash = H(partitions + schema + dict + inputs + metadata crítica)

Su función es:
- detectar corrupción o alteración posterior;
- identificar divergencia entre artefacto declarado y artefacto físico;
- y reforzar la reproducibilidad exacta entre publicación y lectura.

Un snapshot sin content_hash verificable no debería recibir PASS fuerte.

12. Invariantes de implementación

El módulo debe imponer, al menos, estos invariantes:

- ningún snapshot no committed puede ser visible como lectura oficial;
- toda versión committed debe ser inmutable;
- toda versión debe estar aislada por run_id;
- toda PK (date, symbol) debe ser única dentro de la versión;
- schema_declared y schema_physical deben coincidir o registrar explícitamente incompatibilidad;
- todo snapshot committed debe incluir manifest, schema y diccionario;
- read_latest_snapshot() debe ignorar staging y publicaciones fallidas;
- toda decisión de compatibilidad debe venir de reglas versionadas, no de inferencia oportunista;
- toda lectura reproducible debe poder reconstruirse por run_id y metadatos asociados;
- y toda publicación fallida debe dejar evidencia auditable de la causa.

13. Failure modes frecuentes

El módulo debe señalar explícitamente:

13.1 Partial reads
Consumidores observando un snapshot antes de completarse la publicación.

13.2 Duplicidad de PK
Más de una fila para el mismo (date, symbol).

13.3 Divergencia de schema
Schema declarado distinto del schema físico realmente publicado.

13.4 Deriva semántica no registrada
La columna conserva nombre pero cambia significado económico o temporal.

13.5 Commit no verdaderamente atómico
Estados intermedios visibles por errores de implementación.

13.6 Lectura ambigua de latest
Seleccionar una versión incorrecta por presencia de estados staging o failed.

13.7 Lineage incompleto
Snapshot físicamente correcto pero sin capacidad de reconstrucción upstream suficiente.

13.8 Hash inconsistente
Metadatos y contenido físico no corresponden al mismo artefacto real.

14. Gobernanza PASS / WARN / FAIL

14.1 PASS
Un snapshot o una implementación del módulo recibe PASS si:
- la publicación committed es atómica a nivel lógico;
- PK (date, symbol) es única;
- schema y diccionario son consistentes;
- manifest y hashes están completos;
- la lectura por run_id es reproducible;
- y no hay evidencia de estados ambiguos o corrupciones.

14.2 WARN
Recibe WARN si:
- existe trazabilidad parcial pero no completa;
- hay compatibilidad condicional con documentación insuficiente;
- faltan ciertos controles fuertes de integridad o lineage;
- o la implementación es correcta en lo esencial pero todavía no certifica robustez institucional plena.

14.3 FAIL
Recibe FAIL si:
- committed no está bien aislado de staging;
- PK no es única;
- schema y contenido divergen materialmente;
- read_latest puede exponer estados inválidos;
- no existe manifest mínimo suficiente;
- o la versión no puede reconstruirse de forma auditable.

15. Qué ya está muy bien en esta especificación

La especificación actual ya acierta en piezas muy importantes:

- X^κ como familia formal de snapshots;
- aislamiento por run_id;
- staging → committed atómico;
- unicidad canónica de (date, symbol);
- validación explícita schema_declared = schema_physical;
- content_hash como defensa de integridad;
- read_latest solo sobre committed;
- reglas explícitas de schema evolution;
- y contrato de lectura con mode strict|compatible.

Eso sitúa el módulo claramente por encima de un almacén de features estándar de research amateur.

16. Qué falta para rozar el 99.5

Si el objetivo es empujar este módulo hacia nivel “Simons-scale” de infraestructura cuantitativa, faltan sobre todo extensiones de gobernanza avanzada:

16.1 Validación probabilística de snapshot
No solo checks deterministas, sino una certificación del tipo:

P(valid^κ | schema_physical, content_hash, consistency_metrics)

16.2 Verificación causal de lineage
Capturar explícitamente propagación de errores upstream:

upstream_error → snapshot_corruption → IC_degradation

16.3 Compatibilidad dinámica de schema
Formalizar algo del tipo:

P(compatible | schema_v1→v2, consumer_contract)

en lugar de depender solo de taxonomía estática manual.

16.4 Cuantificación de degradación al leer snapshots imperfectos
Medir sensibilidad downstream:

∂IC_model / ∂snapshot_degradation

16.5 Monitorización online
Detección continua de:
- schema drift,
- salud de particiones,
- consistencia entre consumidores,
- y anomalías de publicación.

- check afectado,

- descripción corta,
- y acción recomendada.

18. Esqueleto de implementación

def publish_feature_snapshot(features_df, dictionary, schema_declared, lineage, config):
    run_id = init_run_id(config)
    staging_path = stage_snapshot(features_df, run_id, config)

    pk_report = validate_primary_key(staging_path, keys=["date", "symbol"])
    schema_report = validate_schema(staging_path, schema_declared)
    meta_report = validate_required_metadata(dictionary, lineage, config)

    content_hash = compute_content_hash(
        staging_path,
        schema_declared=schema_declared,
        dictionary=dictionary,
        lineage=lineage,
        config=config
    )

    manifest = build_manifest(
        run_id=run_id,
        store_config_hash=hash_config(config),
        feature_dict_hash=hash_dictionary(dictionary),
        lineage_hash=hash_lineage(lineage),
        content_hash=content_hash
    )

    gate, binding_failure = apply_store_governance(
        pk_report=pk_report,
        schema_report=schema_report,
        meta_report=meta_report,
        manifest=manifest,
        config=config
    )

    if gate == "FAIL":
        mark_snapshot_failed(run_id, binding_failure, manifest, config)
        return build_store_report(run_id, gate, "failed", binding_failure, manifest)

    atomic_commit(run_id, staging_path, manifest, config)

    return build_store_report(
        run_id=run_id,
        gate="PASS",
        status="committed",
        binding_failure=None,
        manifest=manifest
    )

19. MVP vs versión avanzada

19.1 MVP
- aislamiento por run_id;
- staging y committed separados;
- unicidad estricta de PK;
- schema_declared = schema_physical;
- manifest mínimo;
- content_hash reproducible;
- contrato de lectura strict|compatible;
- read_latest solo sobre committed.

19.2 Versión avanzada
- certificación probabilística de validez;
- verificación causal de lineage;
- compatibilidad dinámica por tipo de consumidor;
- monitorización online de drift e integridad;
- auditoría multi-consumidor;
- y scoring de impacto downstream de degradaciones.

20. Interpretación correcta

La lectura correcta de feature_store.py es:

“este módulo garantiza que la capa de features del sistema tiene memoria experimental fuerte, de modo que toda lectura relevante puede reconstruirse como snapshot exacto, inmutable, aislado y auditable.”

La lectura incorrecta es:

“este módulo guarda parquet y algunos metadatos para comodidad del research.”

No está para comodidad.
Está para evitar que el research cuantitativo se vuelva irreproducible.

21. Referencias orientativas

- Apache Iceberg / Delta Lake / Hudi, por patrones de publicación atómica, versionado y schema evolution
- literatura de data lineage y data contracts en sistemas analíticos
- Pearl, por causalidad si se extiende lineage hacia propagación de errores
- Gelman et al., si se formaliza certificación probabilística de validez
- prácticas de feature stores industriales y research platforms institucionales

22. Conclusión

feature_store.py está ya conceptualmente en un nivel altísimo.

La combinación de:
- snapshots X^κ aislados por run_id,
- unicidad no negociable de (date, symbol),
- staging → committed atómico,
- schema evolution explícita,
- read contract reproducible,
- y manifest con hashes de configuración, diccionario y lineage

define una infraestructura de research claramente institucional.

Lo que le falta para rozar techo absoluto no es la base conceptual, que ya es excelente, sino una capa adicional de certificación probabilística, causal lineage explícito y monitorización continua.

Como especificación de memoria experimental cuantitativa, está cerca de excelencia real.
